## Summary

Current `main` still treats `/new` and `/reset` as immediate reset commands in the Control UI, TUI, gateway, config, and docs. I found no confirmation step, timeout-based double-submit flow, or configurable `session.confirmWipe`/`confirmSteps` equivalent.

## Evidence

- **Control UI dispatch is immediate:** `dispatchSlashCommand` sends `/new` and `/reset` straight to `sendChatMessageNow(...)` with no confirmation state or second-step check.
  - file: [ui/src/ui/app-chat.ts:417](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/ui/src/ui/app-chat.ts#L417)
  - command: `nl -ba ui/src/ui/app-chat.ts | sed -n '360,530p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Slash command text assumes immediate execution:** The slash-command executor returns `Starting new session...` for `/new` and `Resetting session...` for `/reset`; there is no pending-confirmation action.
  - file: [ui/src/ui/chat/slash-command-executor.ts:93](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/ui/src/ui/chat/slash-command-executor.ts#L93)

  - file: [src/tui/tui-command-handlers.ts:523](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/tui/tui-command-handlers.ts#L523)
