## Summary

Current main does not show the reported auth-ordering bug: trusted-proxy auth is evaluated before token/password fallback, mixed trusted-proxy+token configs are rejected, and the docs define trusted-proxy as identity-aware proxy auth rather than plain TLS-terminating nginx.

## Evidence

- **Trusted-proxy branch runs before shared-secret auth:** `authorizeGatewayConnectCore` handles `auth.mode === "trusted-proxy"` first and returns before the token/password branches later in the function.
  - file: [src/gateway/auth.ts:415](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/gateway/auth.ts#L415)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Token/password fallback is later and unreachable for trusted-proxy mode:** The shared-token and password branches are below the trusted-proxy branch, so they are not evaluated first when `gateway.auth.mode` is `trusted-proxy`.
  - file: [src/gateway/auth.ts:481](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/gateway/auth.ts#L481)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)

  - file: [src/gateway/auth.ts:240](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/gateway/auth.ts#L240)

## Close Comment

Closing after a Codex review because I could not verify the reported auth-ordering bug against current `main`.

The current gateway auth flow handles `gateway.auth.mode: "trusted-proxy"` before any token/password fallback (`src/gateway/auth.ts:415`), and the token/password branches only appear later in the function (`src/gateway/auth.ts:481`). In addition, OpenClaw rejects mixed `trusted-proxy` + shared-token configs at startup (`src/gateway/auth.ts:240`). The protocol and deployment docs also define this mode as identity-aware proxy auth from request headers, not plain TLS-terminating nginx (`docs/gateway/protocol.md:511`, `docs/gateway/trusted-proxy-auth.md:15`).

I also checked the latest release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`v2026.4.22`), and the same trusted-proxy branch ordering is already present there.

If you still have a reproducer on current `main`, please open a new issue with the exact `gateway.auth` / `gateway.trustedProxies` config, the proxy identity header being injected, whether the caller is an operator or node client, and the returned auth reason from the handshake.
