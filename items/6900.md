## Summary

Keep open. Current main documents a workable foreground path for containerized nodes (`openclaw node run` plus device approval), but it does not implement the requested Docker-friendly install behavior: `openclaw node install` still goes through the user-service path and still fails when `systemctl --user` is unavailable.

## Evidence

- **`node install` is still service-manager based:** The install command resolves a node service and aborts with `Node service check failed: ...` if the service probe throws, so Docker/non-systemd sessions still hit the same failure mode described in the issue.
  - file: [src/cli/node-cli/daemon.ts:89](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/cli/node-cli/daemon.ts#L89)
  - command: `nl -ba src/cli/node-cli/daemon.ts | sed -n '70,150p'`
  - sha: [cc295fb8c9a3](https://github.com/openclaw/openclaw/commit/cc295fb8c9a3e775549838a23fff7dbfb72081bf)
- **Linux service install still requires systemd user services:** The Linux service layer still throws `systemctl not available; systemd user services are required on Linux.` or `systemctl --user unavailable: ...` when user services are not reachable.
  - file: [src/daemon/systemd.ts:414](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/daemon/systemd.ts#L414)

- **Current docs position `node run` as the non-service path:** The node CLI docs still describe `openclaw node install` as a background service flow, explicitly say `Use openclaw node run for a foreground node host (no service)`, and document pairing through `openclaw devices list` / `openclaw devices approve <requestId>`.
  - file: [docs/cli/node.md:77](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/docs/cli/node.md#L77)
