## Summary

Close as `clawhub`: the request is for a custom exec policy engine, and current main already exposes plugin hooks that can inspect tool calls, block them, or require approval with the same approval UX. Project scope explicitly prefers optional capability in plugins rather than expanding core exec-approvals config.

## Evidence

- **Core exec approvals still have no denylist mode:** Current main only defines `ExecSecurity` as `deny | allowlist | full`.
  - file: [src/infra/exec-approvals.ts:22](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/infra/exec-approvals.ts#L22)
- **Core approvals schema is allowlist-only:** Per-agent approvals config exposes `allowlist` only; there is no `denylist` field in the persisted schema.
  - file: [src/infra/exec-approvals.ts:129](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/infra/exec-approvals.ts#L129)
- **Docs/UI still describe only allowlists + ask policy:** User-facing docs list only `deny`, `allowlist`, and `full`, and the Control UI surface is documented as editing allowlists plus ask policy.
  - file: [docs/tools/exec-approvals.md:208](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/docs/tools/exec-approvals.md#L208)
- **Plugin hooks can block tool calls:** The plugin SDK documents `before_tool_call` as a first-class hook where `{ block: true }` is terminal.
  - file: [docs/plugins/sdk-overview.md:205](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/docs/plugins/sdk-overview.md#L205)
- **Plugin hooks can request approval with the same approval UX:** Plugin docs state `before_tool_call` can return `{ requireApproval: true }`, which pauses execution and reuses the approval overlay, chat buttons, and `/approve` flow.
  - file: [docs/plugins/building-plugins.md:182](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/docs/plugins/building-plugins.md#L182)
- **Hook payload includes tool name and params:** The hook contract passes `toolName` plus arbitrary `params`, which is enough for an exec-policy plugin to match command text/arguments and decide block vs approval.
  - file: [src/plugins/hook-types.ts:306](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/plugins/hook-types.ts#L306)
- **Plugin approvals are wired into runtime already:** Runtime plumbing calls `plugin.approval.request` when a `before_tool_call` hook asks for approval, so a plugin can implement deny/approval policies without core schema changes.
  - file: [src/agents/pi-tools.before-tool-call.ts:219](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/agents/pi-tools.before-tool-call.ts#L219)
- **Project scope prefers optional capability as plugins:** The vision doc says core should stay lean and optional capability should usually ship as plugins.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/VISION.md#L54)

## Risks / Open Questions

- A maintainer could still prefer a built-in denylist for host-local policy ergonomics, but that is a product choice rather than a missing extension seam.
- A plugin-based solution is not the same as adding a new core `exec-approvals.json` mode, so users wanting only host-local schema support may still need a community plugin or future first-party plugin.

## Close Comment

ClawSweeper/Codex review: closing this as a ClawHub/plugin-scope request rather than a core exec-approvals change. Current main still keeps core exec approvals intentionally small: `security` is only `deny | allowlist | full`, and the persisted per-agent schema is still allowlist-only (`src/infra/exec-approvals.ts:22`, `src/infra/exec-approvals.ts:129`, `docs/tools/exec-approvals.md:208`). For custom policy like “allow everything except X”, OpenClaw already has the extension seam: `before_tool_call` hooks can inspect tool params, hard-block, or require approval using the same approval UX and `/approve` flow (`docs/plugins/sdk-overview.md:205`, `docs/plugins/building-plugins.md:182`, `src/plugins/hook-types.ts:306`, `src/agents/pi-tools.before-tool-call.ts:219`). That matches the project scope in `VISION.md:54`, which keeps optional capability in plugins rather than growing core config. If someone wants to pursue this, the right path is a ClawHub plugin, not a core `exec-approvals.json` mode.
