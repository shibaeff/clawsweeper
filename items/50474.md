## Summary

Current main already handles local CLI shared-token/password auth by attaching a device identity for gateway calls, which preserves operator scopes for CLI RPCs like status/cron instead of relying on device-less shared-secret scopes. The old device-less path still intentionally clears self-declared scopes, so the issue’s proposed fix of force-granting admin on token/password auth does not match the shipped design.

## Risks / Open Questions

- If the CLI cannot create or reuse `~/.openclaw/identity/device.json` in a restricted/read-only environment, it still falls back to device-less shared-secret auth and can be scope-limited.
- The issue report was against OpenClaw 2026.3.13; this review only shows the behavior is implemented on current main and in the later shipped release v2026.4.22.

## Close Comment

Closing from a Codex review: current `main` already implements the fix path for this report, and that implementation is present in the shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). CLI gateway RPCs route through `src/cli/gateway-rpc.runtime.ts:20`, and `src/gateway/call.ts:196` now auto-loads a local device identity for shared-token/password gateway calls specifically so operator scopes remain available for CLI RPCs like status/cron. Regression coverage is in `src/gateway/call.test.ts:306`. Current gateway behavior still intentionally clears self-declared scopes for device-less shared-secret operator connects (`src/gateway/server.auth.compat-baseline.test.ts:123`), so force-granting `operator.admin` on token/password auth is not the supported design on current main. If this still reproduces on a newer build, please open a fresh issue with the exact version and whether the CLI could create/use the local device identity.
