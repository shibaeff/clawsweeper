## Summary

Keep open. Current `main` still swallows Discord `fetchUser("@me")` startup failures, leaves `botUserId` undefined, and continues startup with downstream mention/self-message logic gated on that optional value.

## Evidence

- **Bot identity fetch still degrades to undefined:** `fetchDiscordBotIdentity()` catches `client.fetchUser("@me")` errors, logs them, and returns `{ botUserId: undefined, botUserName: undefined }` instead of failing startup or retrying.
  - file: [extensions/discord/src/monitor/provider.startup.ts:211](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/discord/src/monitor/provider.startup.ts#L211)
  - command: `nl -ba extensions/discord/src/monitor/provider.startup.ts | sed -n '200,280p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Provider startup still proceeds after missing identity:** `monitorDiscordProvider()` assigns the possibly-undefined identity result and continues wiring voice, message handling, listener registration, and startup status logging; there is no guard that aborts startup when `botUserId` is missing.
  - file: [extensions/discord/src/monitor/provider.ts:998](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/discord/src/monitor/provider.ts#L998)

- **Message routing and mention detection still depend on botUserId:** Preflight self-message suppression and explicit-mention detection both only run when `params.botUserId` is truthy, so an undefined bot identity changes inbound routing behavior exactly in the area reported.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:395](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/discord/src/monitor/message-handler.preflight.ts#L395)

- **Other listeners also rely on botUserId being present:** The debounce-path self-message filter and reaction listener early-exit both short-circuit only when `botUserId` is set, so the undefined identity state is not fully neutralized elsewhere in the Discord monitor stack.
