## Summary

Current `main` already addresses both parts of the report: fallback now treats `LiveSessionModelSwitchError` as a failover condition and continues the model chain, and session-scoped auto failover overrides are cleared so stale `modelOverride` values do not pin subsequent attempts to the wrong model.

## Evidence

- **Fallback now normalizes live model switch errors:** `runWithModelFallback` explicitly catches `LiveSessionModelSwitchError`, wraps it as a `FailoverError`, records the failed candidate, and `continue`s to the next fallback candidate instead of rethrowing.
  - file: [src/agents/model-fallback.ts:921](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/model-fallback.ts#L921)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '918,956p'`
- **Regression tests cover the reported fallback failure mode:** There are dedicated tests asserting that `LiveSessionModelSwitchError` is treated as failover on the last candidate and that the fallback chain continues to the next candidate.
  - file: [src/agents/model-fallback.test.ts:644](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/model-fallback.test.ts#L644)
  - command: `nl -ba src/agents/model-fallback.test.ts | sed -n '644,684p'`
- **Auto failover session override is now self-healed:** `createModelSelectionState` detects direct session entries with `modelOverrideSource === "auto"`, clears the stored override back to the configured default, and resets in-memory `provider`/`model` to the primary before allowlist validation or retry selection.
