## Summary

Keep open. Current `main` still does not implement the framework-level tool interruption requested here, and the issue is recent, specific, and actionable rather than stale or incoherent.

## Evidence

- **Abort wrapper forwards signals but does not interrupt an in-flight tool:** `wrapToolWithAbortSignal()` only combines signals, checks `combined?.aborted` once, and then directly awaits `execute(...)`; there is no framework-level `Promise.race` or equivalent cancellation wrapper around a running tool call.
  - file: [src/agents/pi-tools.abort.ts:48](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/agents/pi-tools.abort.ts#L48)
  - command: `nl -ba src/agents/pi-tools.abort.ts | sed -n '1,120p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **`process poll` still ignores the tool abort signal during its wait loop:** The `process` tool execute signature still names the signal parameter `_signal`, and the `poll` path sleeps in a 250 ms loop until deadline/exit without checking that signal, matching the report's core complaint.
  - file: [src/agents/bash-tools.process.ts:306](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/agents/bash-tools.process.ts#L306)

- **`sessions_send` does not accept or propagate an abort signal:** The `sessions_send` tool execute callback currently only takes `(_toolCallId, args)` and its gateway lookup calls pass `timeoutMs` but no abort signal, so the specific interruption gap discussed in the issue is still present on `main`.
  - file: [src/agents/tools/sessions-send-tool.ts:92](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/agents/tools/sessions-send-tool.ts#L92)
