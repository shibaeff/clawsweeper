## Summary

Keep open. Current `main` still has the stale-data pattern, but this PR is not the right cleanup: it targets an `undefined` success payload that the current gateway contract does not prove, and it also clears `sessionsResult` on every exception, which would regress transient-error behavior. The underlying stale-count report remains plausible, so this should stay open for a better-scoped fix rather than be auto-closed.

## Evidence

- **Current controller still preserves stale sessions on generic errors:** `loadSessionsOnce` only writes `state.sessionsResult` inside `if (res)`, then on non-scope errors it sets `sessionsError` and returns without clearing the previous result. That means stale session counts can remain visible after a failed refresh.
