## Summary

Keep open. Current `main` has a channel-aware voice-note path and an OpenAI-specific Opus fix, but the reported Microsoft/Edge WhatsApp case is still plausible on current code: Microsoft still defaults/falls back to MP3, only infers extensions from the requested format string, and WhatsApp only treats actual `audio/ogg` media as Opus voice notes.

## Evidence

- **Maintainer review on the issue already leaves it open for Microsoft/Edge:** The provided GitHub comment from `steipete` explicitly says the OpenAI-specific WhatsApp bug was fixed separately in #60993, but this issue should stay open because the reported `messages.tts.provider: "microsoft"` path is still a real gap on current `main`.
  - command: `provided GitHub context comment 4318439632`
- **Speech core now requests native voice-note output for supported channels:** `synthesizeSpeechPayload` chooses `target = "voice-note"` for channels that support native voice-note TTS, which is the central routing change the maintainer comment referenced.
  - file: [extensions/speech-core/src/tts.ts:810](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/speech-core/src/tts.ts#L810)
  - command: `nl -ba extensions/speech-core/src/tts.ts | sed -n '800,825p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **OpenAI provider has the WhatsApp/voice-note-specific Opus behavior:** OpenAI resolves `voice-note` requests to `opus` by default and only marks the result `voiceCompatible` when the emitted format is actually `opus`, which matches the separate fix path mentioned in the issue discussion.
  - file: [extensions/openai/speech-provider.ts:83](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/openai/speech-provider.ts#L83)
  - command: `nl -ba extensions/openai/speech-provider.ts | sed -n '70,100p'; nl -ba extensions/openai/speech-provider.ts | sed -n '248,262p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Microsoft provider still defaults and falls back to MP3:** The Microsoft/Edge provider default is `audio-24khz-48kbitrate-mono-mp3`, and `synthesize` retries with that MP3 format when a requested `outputFormat` fails. That means a requested OGG/Opus format is not guaranteed to survive to the produced file.
  - file: [extensions/microsoft/speech-provider.ts:29](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/microsoft/speech-provider.ts#L29)

- **Microsoft TTS helper infers filename extension from the requested format string, not the produced codec/container:** `inferEdgeExtension` maps strings containing `ogg` or `opus` to `.ogg`/`.opus`, but there is no remux/transcode step here. Combined with the MP3 fallback above, that leaves room for mismatched extension vs actual payload.
  - file: [extensions/microsoft/tts.ts:15](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/microsoft/tts.ts#L15)

The existing WhatsApp tests cover the `.ogg` path, and the Microsoft tests shown in-tree still exercise only `target: "audio-file"` with MP3 config.
