## Summary

Keep open. Current `main` does not implement a `tools.exec.shell` config surface, and the PR discussion identified real behavior gaps for PTY and gateway-approval runs that still need design and code changes before this can be considered maintainer-cleanup safe to close.

## Evidence

- **No configurable exec shell on current main:** `getShellConfig` on `main` still takes no override and only applies built-in platform defaults (`SHELL`/`sh` on Unix, PowerShell on Windows). That means the requested custom shell path is not implemented in the execution path today.
  - file: [src/agents/shell-utils.ts:54](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/agents/shell-utils.ts#L54)
  - command: `nl -ba src/agents/shell-utils.ts | sed -n '1,140p'`
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **Config surface still absent on main:** Current config types and runtime schema do not expose `tools.exec.shell`, and exec config resolution does not thread any shell value into tool defaults. This confirms the feature is not already shipped on `main`.
  - file: [src/config/types.tools.ts:242](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/config/types.tools.ts#L242)
