## Summary

Keep open. Current `main` has related fallback visibility, but it does not implement either requested feature end-to-end: there is no `agents.defaults.model.requireApproval` support for approving failover before switching models, and there is no `modelAttribution` config or per-message model attribution output on normal replies.

## Evidence

- **Agent model config has no approval or attribution fields:** `AgentModelSchema` only accepts a string or an object with `primary` and `fallbacks`; there is no `requireApproval` or `modelAttribution` field in the agent model config surface.
  - file: [src/config/zod-schema.agent-model.ts:3](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/config/zod-schema.agent-model.ts#L3)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **User docs only document primary plus fallback models:** The gateway configuration docs show `agents.defaults.model` with `primary` and `fallbacks` only, matching the schema and leaving no documented approval-mode or attribution option.
  - file: [docs/gateway/configuration.md:129](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/gateway/configuration.md#L129)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Repo-wide search found no modelAttribution implementation:** A repo-wide `rg -n 'modelAttribution' src docs ui packages extensions` returned no matches on the checked-out `main`, which is strong evidence that the proposed config/API from the issue comment/PR is not present in current code.
  - command: `rg -n 'modelAttribution' src docs ui packages extensions`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Current fallback behavior is notification after selection, gated by verbose mode:** `agent-runner.ts` resolves the actual `providerUsed`/`modelUsed`, computes a fallback transition, and only appends a fallback notice when `verboseEnabled` is true. There is no approval prompt before switching to the fallback model.
  - file: [src/auto-reply/reply/agent-runner.ts:1278](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/auto-reply/reply/agent-runner.ts#L1278)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Fallback notice is informational, not approval-based:** `buildFallbackNotice()` formats a message like `Model Fallback: <active> (selected <selected>; <reason>)`. This is post-fallback disclosure, not a pre-fallback confirmation flow.
  - file: [src/auto-reply/fallback-state.ts:83](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/auto-reply/fallback-state.ts#L83)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Tests confirm the current feature is a verbose fallback notice:** The e2e test asserts that replies contain `Model Fallback:` only for the notice case and explicitly asserts the opposite otherwise, which matches the current notification-only implementation rather than optional per-message attribution or approval gating.
  - file: [src/auto-reply/reply/agent-runner.runreplyagent.e2e.test.ts:671](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/auto-reply/reply/agent-runner.runreplyagent.e2e.test.ts#L671)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Current visibility lives in status, not in every message:** `status-message.ts` shows the selected model and, when active, a fallback line with the active fallback model and reason. That is useful related functionality, but it is a status surface, not automatic attribution appended to each assistant message.
  - file: [src/status/status-message.ts:851](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/status/status-message.ts#L851)
