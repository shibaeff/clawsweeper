## Summary

Issue #45329 is best closed as implemented on main. Current main contains the TTS reply-media fix that matches the reporter’s later findings: OpenClaw now persists temp TTS outputs into managed outbound media and allows them through reply-media normalization, which is the path that had been silently dropping model-generated voice replies. The remaining bare-`[[tts]]` tagged-mode behavior is not a supported syntax on current main and is documented as requiring `[[tts:key=value]]` or `[[tts:text]]...[[/tts:text]]`.

## Close Comment

Closing from a Codex review because the issue’s later-confirmed failure path is already implemented on current `main`. `CHANGELOG.md:725` records #63511 in the `2026.4.14` release: OpenClaw now persists temp TTS voice outputs into managed outbound media and allows them through reply-media normalization, and the current code does that in `src/auto-reply/reply/reply-media-paths.ts:127` and `src/media/local-roots.ts:30`. Also, tagged mode on current main intentionally requires `[[tts:key=value]]` directives or a `[[tts:text]]...[[/tts:text]]` block (`docs/tools/tts.md:294`, `docs/tools/tts.md:351`), so bare `[[tts]]` is not a supported trigger. If you still hit a new Telegram `sendVoice` transport failure on `v2026.4.14` or later, please open a fresh issue with the exact reply payload and full gateway stack trace so maintainers can track that separately.
