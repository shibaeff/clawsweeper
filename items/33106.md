## Summary

The TrustChain request fits OpenClaw’s supported community-plugin/ClawHub path rather than a core feature: main explicitly keeps optional capabilities out of core, already documents ClawHub/npm distribution for plugins, and exposes plugin hooks/services/routes that can enforce runtime trust policy without changing core.

## Close Comment

Closing this as `clawhub` after a Codex review. The current main branch points optional capability like this to the plugin ecosystem rather than OpenClaw core: `VISION.md:54-60` says optional capability should usually ship as plugins and be maintained in their own repo, `VISION.md:99-107` lists guardrails against core additions when the plugin path already exists, `docs/plugins/building-plugins.md:16-19` tells authors to publish plugins to ClawHub or npm instead of this repository, and `docs/tools/clawhub.md:10-50` documents the native install path for that. The SDK already exposes the extension points a trust layer would need, including hooks/services/routes and runtime gates such as `before_tool_call`, `before_install`, and message hooks in `docs/plugins/sdk-overview.md:90-99` and `docs/plugins/sdk-overview.md:205-215`. Since this issue is proposing an optional trust layer and already points to an external `@trustchain/openclaw` plugin, the maintainers’ supported path here is ClawHub/community-plugin distribution rather than merging this into core.
