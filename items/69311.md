## Summary

Keep open. Current `main` at `9577de2da7dfb6234cf30e3b879940067804e26e` and released `v2026.4.23` still leave the reported bug plausible: `splitMediaFromOutput()` only protects fenced ```/~~~ blocks, trims indentation before `MEDIA:` detection, and still globally folds preserved spacing afterward. The provided timeline also shows an open follow-up PR (`#69312`), so this remains an actionable core bug rather than cleanup noise.

## Evidence

- **Indented code blocks are not part of the media parser's fence guard:** `mayContainFenceMarkers()` only checks for ``` and `~~~`, so CommonMark 4-space/tab indented code blocks never enter the protected span path before directive extraction.
  - file: [src/media/parse.ts:124](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/media/parse.ts#L124)
  - command: `nl -ba src/media/parse.ts | sed -n '120,145p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Leading indentation is stripped before MEDIA detection:** `splitMediaFromOutput()` does `line.trimStart()` and then checks `startsWith("MEDIA:")`; a code line like `    MEDIA: /tmp/example.png` will therefore be treated as a directive candidate instead of preserved text.
  - file: [src/media/parse.ts:446](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/media/parse.ts#L446)
  - command: `nl -ba src/media/parse.ts | sed -n '440,475p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Fence parsing still only understands fenced blocks:** `parseFenceSpans()` matches only fenced markers via `^( {0,3})(`{3,}|~{3,})(.*)$`; there is no indented-code-block handling here, so `splitMediaFromOutput()` has no way to exclude indented code lines.
  - file: [src/markdown/fences.ts:28](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/markdown/fences.ts#L28)
  - command: `nl -ba src/markdown/fences.ts | sed -n '1,80p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Whitespace folding regression is still present:** After line processing, `cleanedText` still applies `.replace(/[ \t]{2,}/g, " ")`, which can flatten indentation on preserved lines whenever a real `MEDIA:` directive was extracted elsewhere in the same output.
  - file: [src/media/parse.ts:589](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/media/parse.ts#L589)

- **A nearby parser already handles indented code blocks:** `normalizeDirectiveWhitespace()` masks both fenced blocks and `(?:^|\n)(?:    |\t)[^\n]*` indented blocks before whitespace normalization, which supports the reporter's claim that the media parser is lagging an existing repo pattern.
  - file: [src/utils/directive-tags.ts:49](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/utils/directive-tags.ts#L49)
