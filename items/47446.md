## Summary

Keep open. Current `main` does not fully implement the Discord proxy behavior this PR targets, but the branch is also no longer the best fix: newer proxy infrastructure on `main` expects NO_PROXY-aware env-proxy handling and proxied HTTPS transport settings via `proxyTls`/`requestTls`, while the PR discussion already surfaced boundary and transport issues that would need a redesign rather than a conservative auto-close.

## Evidence

- **Discord gateway still has no env-proxy fallback on main:** `createDiscordGatewayPlugin()` still only uses the configured/debug proxy URL. When `proxy` is absent it calls `fetchDiscordGatewayMetadataDirect(...)` directly, and when `proxy` is present it only wires `HttpsProxyAgent` for the websocket path. There is no current-main env-proxy fallback for Discord gateway metadata or websocket startup.
