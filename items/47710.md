## Summary

Keep open. Current `main` still has no 408-specific recovery for WhatsApp QR login, and the PR discussion identifies unresolved correctness risks in the proposed fix, so there is not strong evidence to auto-close this PR as implemented, obsolete, or invalid.

## Evidence

- **Current login helper only retries code 515:** `waitForWhatsAppLoginResult` recreates the WhatsApp socket only when `getStatusCode(err) === 515`; every other non-401 failure falls through to a terminal `failed` result. There is no 408 retry path on current `main`.
  - file: [extensions/whatsapp/src/connection-controller.ts:171](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/extensions/whatsapp/src/connection-controller.ts#L171)
  - command: `nl -ba extensions/whatsapp/src/connection-controller.ts | sed -n '150,240p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **QR wait path still treats non-401 errors as fatal:** `waitForWebLogin` checks `login.error` first and, for any non-401 status, resets the active login and returns `WhatsApp login failed: ...`. On current `main`, a 408 therefore ends the QR flow instead of refreshing the socket or surfacing a new QR.
  - file: [extensions/whatsapp/src/login-qr.ts:471](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/extensions/whatsapp/src/login-qr.ts#L471)
  - command: `nl -ba extensions/whatsapp/src/login-qr.ts | sed -n '440,520p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Current design depends on preserving QR callbacks across socket replacement:** `attachLoginWaiter` passes `onQr` into `waitForWhatsAppLoginResult`, and the existing 515 restart path forwards that callback into `createSocket(...)` before calling `onSocketReplaced`. That matches the PR review concern: recreating a socket outside this path without preserving `onQr` would stop fresh QR updates from reaching `qrDataUrl`.
  - file: [extensions/whatsapp/src/login-qr.ts:174](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/extensions/whatsapp/src/login-qr.ts#L174)

- **Tests cover 515 restart and QR refresh, but not 408 recovery:** The WhatsApp QR tests currently verify one restart for status 515 and that rotated QR codes propagate to waiters, but there is no mainline test covering 408 timeout recovery. That makes the proposed behavior change unproven in the checked-out code.
