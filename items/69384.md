## Summary

Current `main` treats this report as documented queue behavior rather than a confirmed stale-branch bug: inbound messages during an active run default to `collect` follow-up handling, with explicit `steer`/`interrupt` controls for newest-message-first behavior.

## Evidence

- **Default queue mode is `collect`:** `resolveQueueSettings()` falls back to `collect` for all channels when no override is set.
  - file: [src/auto-reply/reply/queue/settings.ts:7](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/reply/queue/settings.ts#L7)
  - command: `nl -ba src/auto-reply/reply/queue/settings.ts | sed -n '1,120p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Active runs route new inbound messages into follow-up handling:** When a session is already active, `get-reply-run` marks `followup`/`collect`/`steer-backlog` as follow-up modes and prepares queued follow-up work instead of re-anchoring the in-flight turn.
  - file: [src/auto-reply/reply/get-reply-run.ts:634](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/reply/get-reply-run.ts#L634)
  - command: `nl -ba src/auto-reply/reply/get-reply-run.ts | sed -n '620,700p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Queued follow-ups are explicitly labeled and preserved:** Collect-mode drain builds a new prompt headed `[Queued messages while agent was busy]` and dispatches it as the follow-up run.
  - file: [src/auto-reply/reply/queue/drain.ts:220](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/reply/queue/drain.ts#L220)
  - command: `nl -ba src/auto-reply/reply/queue/drain.ts | sed -n '206,235p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Tests cover collected follow-up behavior:** `queue.collect.test.ts` asserts that two messages arriving while busy are merged into a queued follow-up prompt instead of being treated as a stale-branch answer.
  - file: [src/auto-reply/reply/queue.collect.test.ts:72](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/reply/queue.collect.test.ts#L72)
  - command: `nl -ba src/auto-reply/reply/queue.collect.test.ts | sed -n '70,105p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Docs describe the same semantics and the available override:** The queue docs say default mode is `collect`; `interrupt` aborts the active run and starts the newest message, while Control UI docs say normal follow-ups queue and users can click Steer to inject the queued message into the running turn.
  - file: [docs/concepts/queue.md:25](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/concepts/queue.md#L25)
