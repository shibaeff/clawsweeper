## Summary

Current `main` does not contain this PR's behavior. Cron failure announcements still resolve announce delivery with `job.sessionKey`, even when the job targets a specific `session:` target, and the existing tests do not cover the reported `sessionTarget` failure path. This remains a plausible, targeted bug fix rather than a conservative auto-close candidate.

## Evidence

- **Current main still uses creator sessionKey for failure announce delivery:** In `onEvent`, both the explicit `failureDestination` announce path and the fallback-to-primary-delivery announce path pass `sessionKey: job.sessionKey` into `sendFailureNotificationAnnounce`. There is no `sessionTarget`-derived override in current `main`.

- **Session-targeted cron runs already use the session: target key elsewhere:** The isolated cron run path does special-case `session:` targets and swaps the run session to `assertSafeCronSessionTargetId(job.sessionTarget.slice(8))`, which highlights the mismatch with the failure-alert path still using `job.sessionKey`.
  - file: [src/gateway/server-cron.ts:333](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/gateway/server-cron.ts#L333)
