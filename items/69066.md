## Summary

Keep open. Current `main` does not implement a separate internal service-identity path for gateway children: trusted-proxy still hard-rejects loopback sources, gateway clients still bootstrap with token/password credentials only, and device-identity bypass logic is still coupled to shared auth/trusted-proxy state. The RFC is also only 6 days old and already has an open cross-referenced follow-up bug, so there is not strong evidence for conservative auto-close.

## Evidence

- **Trusted-proxy still fails closed on loopback:** `authorizeTrustedProxy()` rejects any loopback remote address with `trusted_proxy_loopback_source`, and `authorizeGatewayConnectCore()` returns that failure directly for `gateway.auth.mode === "trusted-proxy"`. That means current `main` still does not separate same-host/internal callers from user trusted-proxy auth.

- **Tests still assert the coupled behavior:** Gateway auth tests explicitly assert that local-direct trusted-proxy requests are rejected with `trusted_proxy_loopback_source`, including with valid shared-token credentials and forwarded identity headers. That is the opposite of the RFC's proposed internal-service path.
  - file: [src/gateway/auth.test.ts:953](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/gateway/auth.test.ts#L953)
