## Summary

Current `main` already routes in-process config writes and hot-reload comparisons through the authored source-config snapshot instead of persisting runtime-materialized values, which covers the reported failure mode where an unrelated hot-reload write could clobber `channels.feishu.accounts` credentials.

## Evidence

- **Write path now projects runtime edits back onto source config before persisting:** `writeConfigFile` computes a merge patch from the runtime snapshot to the requested config and applies that patch onto the stored source snapshot before writing. Unchanged authored subtrees are therefore retained instead of being replaced by runtime-materialized state.
  - file: [src/config/io.ts:2043](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/io.ts#L2043)
  - command: `nl -ba src/config/io.ts | sed -n '2043,2057p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Persist candidate explicitly preserves untouched source-config structure:** `resolvePersistCandidateForWrite` builds the persisted payload from `sourceConfig`, projected onto the runtime shape, plus only the computed patch. This is the generic safeguard for unrelated writes such as changing `gateway.auth.rateLimit` without dropping other channel config.
  - file: [src/config/io.write-prepare.ts:143](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/io.write-prepare.ts#L143)

  - file: [src/gateway/config-reload.ts:400](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/config-reload.ts#L400)

  - file: [src/config/io.write-config.test.ts:573](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/io.write-config.test.ts#L573)
