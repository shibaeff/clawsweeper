## Summary

Keep open. Current `main` does not include the proposed serialization fix for gateway config writes, and the PR still has an unresolved review concern around queue-level rejections during gateway drain. There is not strong evidence to auto-close this PR conservatively.

## Evidence

- **Current main still does unsynchronized gateway config writes:** `config.set`, `config.patch`, and `config.apply` each read a snapshot, validate `baseHash`, and call `writeConfigFile(...)` directly. There is no `enqueueCommandInLane(...)` or other serialization in these handlers on `main`. That means the PR's stated fix is not already implemented.
  - file: [src/gateway/server-methods/config.ts:444](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/gateway/server-methods/config.ts#L444)
  - command: `nl -ba src/gateway/server-methods/config.ts | sed -n '440,690p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Gateway request layer does not serialize these writes:** The gateway request dispatcher only rate-limits `config.apply` and `config.patch`; it does not serialize config mutations, and `config.set` is not included in the write-budget set at all.
  - file: [src/gateway/server-methods.ts:42](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/gateway/server-methods.ts#L42)
  - command: `nl -ba src/gateway/server-methods.ts | sed -n '36,145p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Persistence path can revive stale fields from a later writer:** `writeConfigFile` re-reads the current snapshot, then `resolvePersistCandidateForWrite` builds a patch from the current runtime config to the caller-supplied `nextConfig`. If a second request computed `nextConfig` from an older snapshot, that patch can reintroduce stale values instead of surfacing a conflict.
  - file: [src/config/io.write-prepare.ts:143](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/config/io.write-prepare.ts#L143)

- **Queue-rejection review concern is real on current main:** The PR review comment about missing error handling is grounded in current queue behavior: `enqueueCommandInLane(...)` rejects immediately with `GatewayDrainingError` when draining, and `clearCommandLane(...)` rejects queued work with `CommandLaneClearedError`. A lane-based fix needs explicit response handling for those cases.
