## Summary

Keep open. Current `main` still handles `config.set`, `config.patch`, and `config.apply` as independent read/compute/write flows without the PR's gateway write-lane serialization, so the requested fix is not implemented on `main` and the stale-snapshot overwrite concern remains plausible.

## Evidence

- **Gateway config handlers still do unsynchronized read-then-write:** `config.set`, `config.patch`, and `config.apply` each call `readConfigFileSnapshotForWrite()` and later `writeConfigFile(...)` directly inside the handler. There is no serial lane or retry against a refreshed snapshot in these paths.
  - file: [src/gateway/server-methods/config.ts:448](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/server-methods/config.ts#L448)
  - command: `nl -ba src/gateway/server-methods/config.ts | sed -n '444,700p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **No gateway use of command-lane serialization on main:** A repository search for `enqueueCommandInLane(` under `src/gateway` returned no matches, so the mechanism described by PR #43244 is not present in the current gateway code.
  - command: `rg -n "enqueueCommandInLane\(" src/gateway -g'*.ts'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Underlying queue supports the exact serialization primitive the PR wants:** `enqueueCommandInLane` exists, serializes per lane, and can reject with `GatewayDrainingError` / `CommandLaneClearedError`. That makes the absence in `config.ts` meaningful rather than a naming change.
  - file: [src/process/command-queue.ts:242](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/process/command-queue.ts#L242)
  - command: `nl -ba src/process/command-queue.ts | sed -n '228,309p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Config write path does not enforce optimistic conflict rejection at commit time:** `writeConfigFile()` takes a snapshot (or supplied `baseSnapshot`) and computes `previousHash`, but the hash is used for logging/anomaly context rather than rejecting a stale concurrent write. Blocking only covers destructive anomalies like size drops or removing `gateway.mode`.
  - file: [src/config/io.ts:1609](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/config/io.ts#L1609)

Existing tests cover normal `config.set` / `config.patch` / `config.apply` behavior, not the concurrent stale-write race described in the PR.
