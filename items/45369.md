## Summary

Current `main` already routes `openai-completions` fallback attempts through a model-scoped boundary-aware transport that uses the candidate model's own `baseUrl`, so the reported previous-provider endpoint reuse is not the active behavior anymore.

## Evidence

- **Fallback stream selection no longer stays on raw `streamSimple`:** `resolveEmbeddedAgentStreamFn` replaces default/session `streamSimple` fallbacks with `createBoundaryAwareStreamFnForModel(params.model)` for supported APIs, which includes the reported fallback path.
  - file: [src/agents/pi-embedded-runner/stream-resolution.ts:120](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/agents/pi-embedded-runner/stream-resolution.ts#L120)

## Close Comment

Closing from a Codex review. Current OpenClaw no longer leaves `openai-completions` fallback candidates on the generic reused `streamSimple` path: `src/agents/pi-embedded-runner/stream-resolution.ts:120` swaps supported fallbacks onto `createBoundaryAwareStreamFnForModel(params.model)`, `src/agents/provider-transport-stream.ts:13` and `src/agents/provider-transport-stream.ts:83` include `openai-completions` in that transport matrix, and `src/agents/openai-transport-stream.ts:1043` builds a fresh OpenAI client from the candidate model's own `baseUrl`. The released `v2026.4.22` tag (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) already contains that implementation as well. On current main, this report is already implemented/fixed, so there is no remaining maintainer action here.
