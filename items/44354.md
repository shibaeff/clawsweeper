## Summary

Keep open. Current main still uses a cached non-probe health path for `openclaw health --json`, while `openclaw channels status --probe --json` builds live per-account snapshots with runtime and credential-source fields. That means the reported Discord mismatch is still plausible on current main and is not strongly closable as fixed or obsolete.

## Evidence

- **health command does not request probe for --json by default:** `src/commands/health.ts:396`-`415` calls gateway method `health` and only sends `{ probe: true }` when `opts.verbose` is set. Plain `openclaw health --json` therefore uses the non-probe path.
  - file: [src/commands/health.ts:396](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/commands/health.ts#L396)
  - command: `nl -ba src/commands/health.ts | sed -n '396,415p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **gateway health handler serves cached non-probe snapshots:** `src/gateway/server-methods/health.ts:13`-`24` checks `params?.probe === true`; otherwise it can return cached health and refresh with `probe: false` in the background.
  - file: [src/gateway/server-methods/health.ts:13](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/gateway/server-methods/health.ts#L13)

It does not pass runtime state or token-source fields, so summary builders that depend on snapshot fields fall back to defaults.
