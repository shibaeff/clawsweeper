## Summary

Keep open: current `main` still synthesizes the same status-only MCP payload described in the report when an MCP server returns neither `content` nor `structuredContent`, so this is not implemented/fixed on `main` and not safely obsolete.

## Evidence

- **Status-only fallback still exists on main:** `toAgentToolResult()` first uses `result.content`, then `result.structuredContent`, and otherwise fabricates a text payload containing only `{ status, server, tool }`. That matches the issue's observed `status: ok`/server/tool output with no transcript body.
  - file: [src/agents/pi-bundle-mcp-materialize.ts:21](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/agents/pi-bundle-mcp-materialize.ts#L21)
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **The MCP client call is passed through directly:** `callTool()` returns `session.client.callTool(...)` as `CallToolResult` without any intermediate mapping here, so the adapter path is: raw MCP result first, then the fallback materializer above if the result is empty.
  - file: [src/agents/pi-bundle-mcp-runtime.ts:269](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/agents/pi-bundle-mcp-runtime.ts#L269)
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Latest release still has the same result-handling code:** `git diff --stat a9797214338ba31c52c796adbb75afb16e0684a9..1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19 -- src/agents/pi-bundle-mcp-materialize.ts src/agents/pi-bundle-mcp-runtime.ts` returned no output, so the relevant MCP result path is unchanged between `v2026.4.23` and current `main`.
  - command: `git diff --stat a9797214338ba31c52c796adbb75afb16e0684a9..1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19 -- src/agents/pi-bundle-mcp-materialize.ts src/agents/pi-bundle-mcp-runtime.ts`
- **Docs confirm this is the outbound MCP integration surface:** The MCP docs describe `list/show/set/unset` as OpenClaw's client-side registry for other MCP servers that its runtimes consume later, which is the surface implicated by an external `youtube-transcript` MCP server.
  - file: [docs/cli/mcp.md:10](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/docs/cli/mcp.md#L10)

## Risks / Open Questions

- The underlying defect may be in the external `youtube-transcript` MCP server returning no `content`/`structuredContent`; this review did not perform a live MCP reproduction.
- Even if the upstream server is at fault, OpenClaw's synthetic `{status:"ok",server,tool}` fallback can mask that failure and make the behavior look like a core serialization bug.
