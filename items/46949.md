## Summary

Keep open. Current `main` does not implement the PR's core fix. The ACP manager still hard-fails at the concurrent-session limit without any pressure-based reclaim path, and startup identity reconciliation still resolves pending identities without releasing oneshot runtime handles.

## Evidence

- **Startup reconcile still keeps oneshot runtime handles:** `reconcilePendingSessionIdentities()` ensures a runtime handle and reconciles identifiers, but returns immediately afterward. There is no `finally` block to close oneshot runtimes or clear cached state during startup reconciliation.
  - file: [src/acp/control-plane/manager.core.ts:260](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/acp/control-plane/manager.core.ts#L260)

  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Concurrent-session limit still throws immediately at capacity:** `enforceConcurrentSessionLimit()` is still synchronous and only throws `ACP max concurrent sessions reached (...)` once cache size reaches the limit. There is no under-pressure eviction/probing step before rejection.
  - file: [src/acp/control-plane/manager.core.ts:1644](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/acp/control-plane/manager.core.ts#L1644)

## Risks / Open Questions

- The reported `ACP max concurrent sessions reached` failure mode can still happen on current `main` if dormant oneshot runtime handles remain cached.
- Closing this PR as implemented would be incorrect because the mainline code still lacks both the startup oneshot release path and the under-pressure eviction behavior described in the PR.
