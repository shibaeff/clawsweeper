## Summary

Issue #58480 is already fixed on current `main`: WhatsApp creds writes are now serialized through a per-auth-dir queue, written atomically via temp-file rename, and auth/bootstrap reads wait for queued saves before deciding whether repair is needed. The shipped `v2026.4.23` release contains the same production implementation.

## Evidence

- **Queued barrier before auth bootstrap:** `createWaSocket` waits for `waitForCredsSaveQueueWithTimeout(authDir)` before repair, and only then calls `restoreCredsFromBackupIfNeeded`; `creds.update` events enqueue saves instead of writing inline.
  - file: [extensions/whatsapp/src/session.ts:136](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/extensions/whatsapp/src/session.ts#L136)
  - command: `nl -ba extensions/whatsapp/src/session.ts | sed -n '1,260p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **Atomic creds write path:** `writeCredsJsonAtomically` writes to `.creds.<pid>.<uuid>.tmp`, fsyncs, renames onto `creds.json`, chmods, and fsyncs the parent directory. `enqueueCredsSave` maintains the per-auth-dir save queue used by the barrier.
  - file: [extensions/whatsapp/src/creds-persistence.ts:32](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/extensions/whatsapp/src/creds-persistence.ts#L32)
  - command: `nl -ba extensions/whatsapp/src/creds-persistence.ts | sed -n '1,220p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **Ordinary auth reads no longer restore from backup:** `webAuthExists` is a pure read/parse check; restore logic is separate. `readWebAuthStateCore` waits on the save barrier before reading auth state, preventing the reconnect/read race described in the report.
  - file: [extensions/whatsapp/src/auth-store.ts:56](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/extensions/whatsapp/src/auth-store.ts#L56)

- **Regression coverage for the reported failure mode:** Tests cover: no backup restore on ordinary reads; unstable auth state if the save barrier times out; atomic temp-file rename writes; and preservation of valid prior creds if rename fails.
