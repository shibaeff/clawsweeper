## Summary

Current `main` already honors `compat.supportsStore` for OpenAI-compatible completions payloads and does not send `store` when that compat flag is false; the Google AI OpenAI-compatible host is also recognized explicitly. The same implementation is present in release tag `v2026.4.22`, so the issue is already fixed on shipped code.

## Evidence

- **Compat override is honored in completions transport:** `getCompat()` uses `compat.supportsStore` when provided, instead of always using detected defaults.
  - file: [src/agents/openai-transport-stream.ts:1501](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/agents/openai-transport-stream.ts#L1501)

## Close Comment

Closing from a Codex review because current `main` already implements this behavior. In `src/agents/openai-transport-stream.ts:1501`, OpenClaw reads `compat.supportsStore` directly, and in `src/agents/openai-transport-stream.ts:1694` it only adds `store: false` when that flag is enabled. For the reported Google endpoint, `src/agents/provider-attribution.ts:316` classifies `generativelanguage.googleapis.com` as `google-generative-ai`, and the completions compatibility defaults in `src/agents/openai-completions-compat.ts:96` disable store for non-standard/proxy-like routes. Current regression coverage also checks that proxy/custom completions params omit `store` in `src/agents/openai-transport-stream.test.ts:1459`. I also verified the same implementation is present in the shipped `v2026.4.22` tag. If you still hit a 400 on a newer build, please open a fresh issue with the exact model/provider config and a request log excerpt.
