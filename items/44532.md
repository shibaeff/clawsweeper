## Summary

Current `main` does not define an `http_request` tool at all; the shipped HTTP-fetch capability is `web_fetch`, which is documented, registered in the core tool catalog, included in the assembled tool set, and tested as enabled by default. The original report’s own tool list already showed `web_fetch`, so the report no longer matches current OpenClaw behavior and does not show a Docker-specific regression on `main`.

## Evidence

- **Official docs name the HTTP fetch tool `web_fetch`:** The current docs describe `web_fetch` as the HTTP GET content-fetch tool and state it is enabled by default.
  - file: [docs/tools/web-fetch.md:11](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/docs/tools/web-fetch.md#L11)

## Close Comment

Closing from a Codex review. I could not reproduce a current OpenClaw bug here: on `main`, the built-in HTTP fetch tool is `web_fetch`, not `http_request`, and it is documented and tested as enabled by default (`docs/tools/web-fetch.md:11`, `src/agents/tool-catalog.ts:104`, `src/agents/tools/web-tools.enabled-defaults.test.ts:61`). The runtime tool assembly also includes `web_fetch` in the built-in tool set (`src/agents/openclaw-tools.ts:193`). A repo-wide search on `aa21d4ea7e9aab034acbf0b13f8c3575057c5c54` finds no `http_request` symbol at all. Since the original report’s own tool listing already included `web_fetch`, this does not look like a Docker-only regression on current `main`; it looks like a mismatch with outdated/noncanonical docs or naming. If the underlying problem was a different missing HTTP capability, please open a new issue with that exact repro and expected behavior.
