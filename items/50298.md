## Summary

Keep open. Current main partially stabilizes direct `chat.send` by using the caller's `idempotencyKey` as the chat run id, but the requested feature is not implemented for merged/collected followups: the queue still creates a fresh random run id, and the Gateway chat/agent event schemas do not expose any `originalRunIds`/source-run field.

## Evidence

- **Chat events have no original-run field:** `ChatEventSchema` only allows `runId`, `sessionKey`, `seq`, `state`, `message`, `errorMessage`, `errorKind`, `usage`, and `stopReason`. There is no field for original run ids or source run ids.
  - file: [src/gateway/protocol/schema/logs-chat.ts:71](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/gateway/protocol/schema/logs-chat.ts#L71)
  - command: `nl -ba src/gateway/protocol/schema/logs-chat.ts | sed -n '60,95p'`
- **Direct chat.send now reuses the caller run id:** `chat.send` assigns `clientRunId = p.idempotencyKey` and immediately acks with that same `runId`, so direct websocket sends no longer mint a different visible run id.
  - file: [src/gateway/server-methods/chat.ts:2150](https://github.com/openclaw/openclaw/blob/f353a61bab3d1157c706fb71df93aa86911d7a16/src/gateway/server-methods/chat.ts#L2150)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '2148,2258p'`
  - sha: [f353a61bab3d](https://github.com/openclaw/openclaw/commit/f353a61bab3d1157c706fb71df93aa86911d7a16)
- **Merged collect followups still create a brand-new run id:** The collect/followup drain batches queued messages and hands them to the followup runner, which then starts the actual run with `const runId = crypto.randomUUID()`. That means merged execution still gets a fresh run id unrelated to the original queued message ids.
  - file: [src/auto-reply/reply/followup-runner.ts:228](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/auto-reply/reply/followup-runner.ts#L228)
  - command: `nl -ba src/auto-reply/reply/followup-runner.ts | sed -n '220,245p'`
- **Queued followup state keeps only messageId, not a set of run ids:** When a busy session is queued, the constructed `followupRun` stores `messageId: sessionCtx.MessageSidFull ?? sessionCtx.MessageSid`; there is no preserved array/set of original run ids to surface later.
  - file: [src/auto-reply/reply/get-reply-run.ts:682](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/auto-reply/reply/get-reply-run.ts#L682)
  - command: `nl -ba src/auto-reply/reply/get-reply-run.ts | sed -n '676,692p'`
- **Gateway rewrites emitted agent/chat events to one clientRunId only:** `server-chat` maps agent events through `chatLink?.clientRunId ?? evt.runId` before broadcasting. That supports a single client-visible run id, but there is still no payload carrying multiple original run ids for merged runs.
  - file: [src/gateway/server-chat.ts:893](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/gateway/server-chat.ts#L893)
  - command: `nl -ba src/gateway/server-chat.ts | sed -n '890,1000p'`
