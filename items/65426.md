## Summary

Current `main` already fixes this: reply delivery now normalizes workspace-relative `MEDIA:` paths into workspace/sandbox host paths and stages them through the shared outbound attachment loader before channel delivery, with regression tests covering the `./...` path case and escape-path rejection. The fix is present on `main` in commit `453789914bb725d1cd5a3edd7ec5d8b0b45041cd`, but it is newer than the latest tagged release `v2026.4.23`, so no shipped release tag can be confirmed yet.

## Evidence

- **Reply media normalizer resolves workspace-relative paths before staging:** `createReplyMediaPathNormalizer()` rewrites relative `MEDIA:` sources against the agent workspace, then persists them via `resolveOutboundAttachmentFromUrl(...)` instead of passing raw `./...` paths to outbound media loading.
  - file: [src/auto-reply/reply/reply-media-paths.ts:150](https://github.com/openclaw/openclaw/blob/453789914bb725d1cd5a3edd7ec5d8b0b45041cd/src/auto-reply/reply/reply-media-paths.ts#L150)
  - sha: [453789914bb7](https://github.com/openclaw/openclaw/commit/453789914bb725d1cd5a3edd7ec5d8b0b45041cd)
- **Sandbox-aware mapping covers stock gateway reply paths too:** When a sandbox workspace exists, the same normalizer maps `MEDIA:` paths from `/workspace/...` back to the host sandbox workspace before staging, which addresses the stock-gateway path mismatch the issue describes.
  - file: [src/auto-reply/reply/reply-media-paths.ts:172](https://github.com/openclaw/openclaw/blob/453789914bb725d1cd5a3edd7ec5d8b0b45041cd/src/auto-reply/reply/reply-media-paths.ts#L172)
  - sha: [453789914bb7](https://github.com/openclaw/openclaw/commit/453789914bb725d1cd5a3edd7ec5d8b0b45041cd)
- **Normal reply routing now applies the media-path normalizer before channel delivery:** `dispatch-from-config` loads `createReplyMediaPathNormalizer(...)` and applies it to reply payloads before routed delivery, covering normal channel replies such as Telegram gateway/direct-chat delivery.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:386](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/auto-reply/reply/dispatch-from-config.ts#L386)
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Regression test covers the reported `MEDIA:./...` case:** Test asserts that `MEDIA:./out/photo.png` is staged through shared outbound attachment loading as `/tmp/agent-workspace/out/photo.png`, which is the direct fix for the workspace-relative rejection reported here.
  - file: [src/auto-reply/reply/reply-media-paths.test.ts:38](https://github.com/openclaw/openclaw/blob/453789914bb725d1cd5a3edd7ec5d8b0b45041cd/src/auto-reply/reply/reply-media-paths.test.ts#L38)
  - sha: [453789914bb7](https://github.com/openclaw/openclaw/commit/453789914bb725d1cd5a3edd7ec5d8b0b45041cd)
