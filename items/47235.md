## Summary

Current `main` still matches the report closely enough that I would not close this issue. `memory status --deep` still runs a fresh live embedding probe each time, and the human-facing output still reports that probe as `Embeddings: ready|unavailable` alongside persisted index counts without adding the stronger separation the issue asked for.

## Evidence

- **Deep status still does a live probe on every run:** `runMemoryStatus()` calls `manager.probeEmbeddingAvailability()` inside the `--deep` path before rendering status output, so repeated invocations still depend on a fresh runtime probe rather than only persisted index state.
  - file: [extensions/memory-core/src/cli.runtime.ts:695](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/memory-core/src/cli.runtime.ts#L695)
  - command: `nl -ba extensions/memory-core/src/cli.runtime.ts | sed -n '690,708p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Embedding probe is still an actual embed request:** `probeEmbeddingAvailability()` still calls `embedBatchWithRetry(["ping"])` and returns `{ ok: false, error }` on failure, which means transient provider/network issues can still flip the probe result independently of on-disk index contents.
  - file: [extensions/memory-core/src/memory/manager.ts:793](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/memory-core/src/memory/manager.ts#L793)
