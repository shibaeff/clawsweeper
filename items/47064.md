## Summary

Current OpenClaw already uses a unified SecretRef-based env credential contract across config and auth profiles, and `models.json` is now a generated marker file rather than a separate `env:default:VAR`-style user config surface.

## Evidence

- **Config syntax is now `${VAR}` or SecretRef objects:** Gateway config docs show env substitution with `${VAR_NAME}` and the canonical SecretRef object `{ source: "env", provider: "default", id: "VAR" }` for secret-bearing fields such as `models.providers.*.apiKey`.
  - file: [docs/gateway/configuration.md:622](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/docs/gateway/configuration.md#L622)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Canonical SecretRef contract is documented for both config and auth storage:** The configuration reference defines one SecretRef shape and explicitly says `auth-profiles.json` supports `keyRef`/`tokenRef` value-level refs.
  - file: [docs/gateway/configuration-reference.md:635](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/docs/gateway/configuration-reference.md#L635)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Shared parser handles SecretRef objects and `${VAR}` templates:** `coerceSecretRef()` accepts the canonical SecretRef object and also converts `${VAR}` strings into the same env SecretRef structure.
  - file: [src/config/types.secrets.ts:67](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/config/types.secrets.ts#L67)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Auth profiles normalize through the same SecretRef path:** `auth-profiles.json` loading normalizes secret-backed `key`/`token` object values via `coerceSecretRef()` into `keyRef`/`tokenRef`, so auth-profile refs use the same underlying contract.
  - file: [src/agents/auth-profiles/persisted.ts:34](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/auth-profiles/persisted.ts#L34)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Onboarding writes consistent env refs:** Onboarding docs state that ref mode writes `keyRef` for auth-profile-backed providers and writes `models.providers.<id>.apiKey` as the same env SecretRef object for custom providers.
  - file: [docs/cli/onboard.md:78](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/docs/cli/onboard.md#L78)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **`models.json` is generated from source config/auth refs and stores markers, not resolved secrets:** Models docs describe `models.json` as a generated registry that refreshes SecretRef-managed provider credentials from source markers like `ENV_VAR_NAME` and `secretref-env:ENV_VAR_NAME`, rather than a separate user-facing `env:default:VAR` config syntax.
  - file: [docs/concepts/models.md:266](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/docs/concepts/models.md#L266)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Close Comment

Closing from a Codex review. Current OpenClaw already implements a unified env-credential contract rather than the older split described here: `openclaw.json` now documents `${VAR}` substitution and canonical SecretRef objects (`docs/gateway/configuration.md:622`, `docs/gateway/configuration-reference.md:635`), `auth-profiles.json` documents `keyRef`/`tokenRef` using the same SecretRef shape (`docs/gateway/configuration-reference.md:711`), and the auth-profile loader normalizes those refs through the shared `coerceSecretRef()` path (`src/agents/auth-profiles/persisted.ts:34`, `src/config/types.secrets.ts:67`). `models.json` is also documented as a generated marker file that keeps env/source markers instead of resolved secrets (`docs/concepts/models.md:266`). This is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still see a Windows-specific regression on a current build, please open a new issue with exact file contents, the startup command, and the resulting logs.
