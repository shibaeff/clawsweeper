## Summary

Keep open. Current `main` still contains the reported display bug in `status --deep`: the last-heartbeat renderer appends `ago` to a string that already comes from `formatTimeAgo(...)`, and it still substitutes `unknown` when `lastHeartbeat.channel` is absent.

## Evidence

- **Current main still appends a second `ago`:** `buildStatusLastHeartbeatValue` computes `age` with `formatTimeAgo(...)` and then returns ``${age} ago``, which yields strings like `1m ago ago`.
  - file: [src/commands/status.command-sections.ts:130](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/commands/status.command-sections.ts#L130)
  - command: `nl -ba src/commands/status.command-sections.ts | sed -n '120,137p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Current main still falls back to misleading `unknown`:** The same renderer sets `const channel = params.lastHeartbeat.channel ?? "unknown"`, so valid heartbeats without channel metadata still render `unknown` instead of omitting that field.
  - file: [src/commands/status.command-sections.ts:131](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/commands/status.command-sections.ts#L131)
  - command: `nl -ba src/commands/status.command-sections.ts | sed -n '120,137p'`
