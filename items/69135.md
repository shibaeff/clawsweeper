## Summary

Current `main` still appears to have the reported false positive. `gateway probe` will probe both the SSH tunnel target and the configured remote URL when the URLs differ, and `multiple_gateways` still triggers solely from `reachable.length > 1` without collapsing targets that report the same gateway identity. I did not find current tests covering the same-gateway-two-transports case.

## Evidence

- **Probe pass keeps both tunnel and configured remote targets:** When an SSH tunnel is active, the probe prepends the tunnel target and only filters out base targets with the exact same URL. A direct remote URL to the same gateway survives if it differs from the tunnel URL, such as `ws://127.0.0.1:18789` versus `ws://gateway-host:18789`.
  - file: [src/commands/gateway-status/probe-run.ts:113](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/commands/gateway-status/probe-run.ts#L113)
  - command: `nl -ba src/commands/gateway-status/probe-run.ts | sed -n '97,115p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Warning logic does not compare gateway identity:** `buildGatewayStatusWarnings` emits `multiple_gateways` whenever more than one probed target is reachable. It does not inspect `entry.self`, host, IP, or any other identity signal before warning.
  - file: [src/commands/gateway-status/output.ts:58](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/commands/gateway-status/output.ts#L58)

- **Identity extraction exists but is unused for warnings:** The probe captures `self` from presence via `pickGatewaySelfPresence`, which extracts fields like `host` and `ip`, but the warning path above does not use that information to deduplicate same-gateway transports.
  - file: [src/commands/gateway-presence.ts:10](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/commands/gateway-presence.ts#L10)
