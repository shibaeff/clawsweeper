## Summary

Keep open. Current `main` still has the restart window described in #61343: cron execution is run outside the persisted lock, `lastRunAtMs` is only written later during outcome application, startup clears `runningAtMs`, and startup catch-up can rerun a cron job from a stale `lastRunAtMs` value.

## Evidence

- **Phase 1 only persists running marker:** `onTimer` marks due jobs with `runningAtMs` and persists before execution, but does not persist `lastRunAtMs` at that point.
  - file: [src/cron/service/timer.ts:727](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/cron/service/timer.ts#L727)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '724,812p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Job execution completes before outcome persist:** `runDueJob` awaits `executeJobCoreWithTimeout` and returns an in-memory outcome; only after all workers finish does the later locked block call `applyOutcomeToStoredJob(...)` and `persist(state)`, leaving a crash/restart window after successful execution.
  - file: [src/cron/service/timer.ts:752](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/cron/service/timer.ts#L752)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '724,812p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **lastRunAtMs is written only during outcome application:** `applyJobResult` sets `job.state.lastRunAtMs = result.startedAt`, but that function is reached from the post-execution outcome path, not immediately after execution succeeds.
  - file: [src/cron/service/timer.ts:417](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/cron/service/timer.ts#L417)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '404,435p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Startup clears runningAtMs then catch-up can replay from stale lastRunAtMs:** Service startup clears any persisted `runningAtMs` markers and then calls `runMissedJobs(...)`. Separately, `isRunnableJob(...)` allows cron startup replay when `previousRunAtMs > lastRunAtMs`, which matches the duplicate-run path described in the issue if `lastRunAtMs` was never persisted.
  - file: [src/cron/service/ops.ts:104](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/cron/service/ops.ts#L104)

- **Existing regression tests do not cover the restart window:** Current regression tests assert that `applyJobResult(...)` preserves `lastRunAtMs` when next-run computation fails, but they do not simulate a successful execution followed by a restart before the later persist block.

## Risks / Open Questions

- The issue discussion reports a real production duplicate-delivery incident, and the current control flow still matches that failure mode.
- There is no restart-focused regression covering the specific crash window between successful execution and the later persisted outcome block, so this remains easy to miss.
