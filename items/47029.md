## Summary

Current `main` has partial superseding work for this PR, but it does not fully implement the proposed behavior. Feishu config/account inspection now tolerates unresolved SecretRefs, and the newer `gateway-status` path resolves gateway auth secret inputs directly, but `status` command secret targets still exclude `gateway.auth.token/password`, and unauthenticated loopback probes still force `deviceIdentity` to `null` instead of reusing an existing paired identity.

## Evidence

- **Status secret target set still omits local gateway auth:** `STATIC_STATUS_TARGET_IDS` contains only memory-search targets, and `getStatusCommandSecretTargetIds()` returns that list plus channel targets. `gateway.auth.token/password` are only listed under `STATIC_SECURITY_AUDIT_TARGET_IDS`, not status.
  - file: [src/cli/command-secret-targets.ts:34](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/cli/command-secret-targets.ts#L34)
  - command: `nl -ba src/cli/command-secret-targets.ts | sed -n '1,220p'`
- **Loopback probe logic still disables device identity without shared auth:** `probeGateway()` returns `null` for `deviceIdentity` when the URL is loopback and no token/password was supplied, which is the opposite of the PR's proposed loopback-identity reuse.
  - file: [src/gateway/probe.ts:155](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/probe.ts#L155)
  - command: `nl -ba src/gateway/probe.ts | sed -n '130,210p'`
- **Current tests assert unauthenticated loopback probes have no device identity:** The probe test suite expects `deviceIdentity` to be `null` for unauthenticated loopback probes, confirming current `main` intentionally keeps that behavior.
  - file: [src/gateway/probe.test.ts:162](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/probe.test.ts#L162)
  - command: `nl -ba src/gateway/probe.test.ts | sed -n '130,220p'`
- **Feishu inspection path now tolerates unresolved SecretRefs:** Feishu secret resolution returns `undefined` instead of throwing in `inspect` mode, and `resolveFeishuAccount()` is documented as treating unresolved SecretRefs as unavailable for read-only CLI/config snapshots.
  - file: [extensions/feishu/src/accounts.ts:70](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/feishu/src/accounts.ts#L70)
  - command: `nl -ba extensions/feishu/src/accounts.ts | sed -n '50,150p'`
- **Gateway status now resolves gateway auth secrets through a separate surface-specific path:** `gateway-status` uses `resolveGatewayProbeSurfaceAuth()` for local/remote targets, and unresolved auth SecretRefs are emitted as warnings only when the probe is unreachable. This covers part of the PR's motivation, but in a different command path than `getStatusCommandSecretTargetIds()`.
  - file: [src/commands/gateway-status/helpers.ts:162](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/commands/gateway-status/helpers.ts#L162)
  - command: `nl -ba src/commands/gateway-status/helpers.ts | sed -n '162,177p'`

## Risks / Open Questions

- Because current `main` still contradicts the loopback-identity part and the `status` secret-target part, closing as implemented would be inaccurate.
