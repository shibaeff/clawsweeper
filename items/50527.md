## Summary

Keep open. Current `main` does not contain the proposed per-field metadata length checks in `normalizeAgentPayload()`, and mapped hook dispatches still accept request-derived `name`/`sessionKey`/`to`/`model`/`thinking` values without passing through that normalization path. The PR’s stated change would not be fully implemented on `main`, and as scoped in the PR context it also appears incomplete for mapped hooks.

## Evidence

- **Current main still lacks the requested length checks:** `normalizeAgentPayload()` on current `main` only requires `message`, normalizes optional metadata fields, and validates `model` presence when provided. There are no max-length constants or max-length rejections for `name`, `to`, `model`, `thinking`, `agentId`, or `sessionKey`. The only existing length cap in this area is the separate idempotency-key helper.
