## Summary

Current `main` already prevents Feishu streaming cards from merging separate final replies. The fix lives in the reply dispatcher rather than `mergeStreamingText`: each final card reply closes the active streaming session, resets the accumulated stream state, and later finals start a fresh session. Regression tests cover two distinct final replies producing two separate streaming sessions/cards without duplication, and the same implementation is already present in release `v2026.4.22`.

## Evidence

- **Dispatcher closes and resets per final reply:** When a final Feishu card reply arrives on an active stream, the dispatcher merges only the current reply into `streamText`, calls `closeStreaming()`, and records the delivered final text. `closeStreaming()` then nulls the streaming session and clears `streamText`, `lastPartial`, and `reasoningText`, so a later final reply cannot append into the previous card.

## Close Comment

Closing as implemented on `main` after Codex review. Feishu now closes and resets the active streaming card for each final reply in `extensions/feishu/src/reply-dispatcher.ts` (`closeStreaming()` resets the session/state, and final delivery calls it before any later final can reuse that card), and there is a regression test in `extensions/feishu/src/reply-dispatcher.test.ts` that sends two distinct final replies and asserts two separate streaming sessions/cards. I also verified the same implementation is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this report appears fixed in shipped code.
