## Summary

Keep open. Current `main` still lacks the proposed compaction-wait cap and timeout force-clear, so the underlying zombie-state report is not implemented on `main`, and the PR itself still has an unresolved correctness risk around clearing a newer replaced run.

## Evidence

- **Current main still extends compaction wait without an upper bound:** `waitForCompactionRetryWithAggregateTimeout` keeps looping whenever `isCompactionStillInFlight()` stays true; there is no extension cap or absolute timeout beyond the per-window timer.
  - file: [src/agents/pi-embedded-runner/run/compaction-retry-aggregate-timeout.ts:42](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/agents/pi-embedded-runner/run/compaction-retry-aggregate-timeout.ts#L42)
  - command: `nl -ba src/agents/pi-embedded-runner/run/compaction-retry-aggregate-timeout.ts | sed -n '1,120p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Current main timeout path only logs; it does not clear zombie active-run state:** After `abortRun(true)`, the 10-second follow-up timer only checks `activeSession.isStreaming` and logs `embedded run abort still streaming`; it does not remove the session from the active-run registry.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2123](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/agents/pi-embedded-runner/run/attempt.ts#L2123)
  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.ts | sed -n '2090,2165p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Run replacement exists on main, so unconditional force-clear is a real race surface:** `setActiveEmbeddedRun` can replace an existing handle (`reason: run_replaced`), while `clearActiveEmbeddedRun` deliberately guards on handle identity and skips mismatches. That makes the PR review concern concrete: a delayed timer from an older run could clear a newer run if force-clear only keys by `sessionId`.
  - file: [src/agents/pi-embedded-runner/runs.ts:359](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/agents/pi-embedded-runner/runs.ts#L359)

- **PR discussion shows one follow-up addressed tests, but the P1 race remains unresolved:** The PR added a second commit (`35309e0c5b778c4790a90c6bf12d1525a4e0a822`) to address Greptile's missing-test feedback, which is useful progress. But the review thread also contains a Codex P1 warning that `forceClearActiveEmbeddedRun` can delete a newer healthy run after `run_replaced`, and the provided PR diff does not include a fix for that race.
  - sha: [35309e0c5b77](https://github.com/openclaw/openclaw/commit/35309e0c5b778c4790a90c6bf12d1525a4e0a822)
