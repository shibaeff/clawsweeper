## Summary

Current `main` does not match the reported failure mode: there is no 60s global embedded-run timeout that preempts fallback, the default agent timeout is 48 hours, timeout handling is applied per fallback attempt, and current regression tests explicitly cover timeout-classified fallback succeeding on the next model/provider.

## Evidence

- **Default agent timeout is no longer 60s:** `resolveAgentTimeoutSeconds()` defaults to `48 * 60 * 60` seconds (172800s / 48h), not 60s.
  - file: [src/agents/timeout.ts:3](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/agents/timeout.ts#L3)

- **Fallback wraps each attempt:** `agent-command.ts` calls `runWithModelFallback(...)`, and the callback passes `timeoutMs` into each `runAgentAttempt()` invocation. This is per-candidate attempt wiring, not one shared outer timeout around the whole fallback chain.
  - file: [src/agents/agent-command.ts:870](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/agents/agent-command.ts#L870)

  - file: [src/agents/model-fallback.ts:652](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/agents/model-fallback.ts#L652)

## Risks / Open Questions

- This review is based on current `main` static inspection and existing regression tests; I did not run a live reproduction because the task required a byte-for-byte clean checkout and the environment does not have `pnpm` available.
- A user-configured short `agents.defaults.timeoutSeconds` still applies per attempt, so a long fallback chain can exceed that wall-clock value cumulatively; that is different from the reported bug, which claimed the first 60s timeout prevented fallback entirely.

## Close Comment

Closing after Codex review because the reported failure mode does not reproduce against current `main`. The current runtime no longer has the cited 60s default agent timeout: `src/agents/timeout.ts:3` sets the default to 172800s (48 hours), and `docs/concepts/agent-loop.md:166` documents the same. Current regression coverage in `src/agents/model-fallback.run-embedded.e2e.test.ts:433` and `:599` also asserts that timeout-classified failures fall back from the first provider/model to the next one successfully. The specific `describeFailoverError` shadowing pattern mentioned in the issue is also not present in `src/agents/model-fallback.ts:224` on current `main`.
