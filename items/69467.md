## Summary

Current `main` still matches the reported failure mode. `sessionTarget: "current"` is persisted as `session:<live-session-key>`, one-shot jobs still default `deleteAfterRun: true`, non-`isolated` cron runs reuse that live session key, and direct-delivery cleanup still calls `sessions.delete` on that same key. There is no guard limiting that cleanup to ephemeral cron sessions, and current tests only assert the delete behavior rather than protecting live `current/session:*` bindings.

## Evidence
