## Summary

Keep open. Current `main` still lacks a short connection-specific timeout on the Ollama `/api/pull` request, but this PR's proposed `AbortSignal.timeout(30_000)` would time-limit the entire streaming fetch and can abort legitimate long model downloads. No allowed auto-close reason applies with high confidence.

## Evidence

- **Current main still has no pull-specific timeout:** `pullOllamaModelCore` in `extensions/ollama/src/setup.ts` posts to `/api/pull` through `fetchWithSsrFGuard` without `timeoutMs` or a request `signal`; only method/headers/body are passed in `init`. This means the specific 30s timeout from the PR is not already implemented on `main`.
  - file: [extensions/ollama/src/setup.ts:167](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/ollama/src/setup.ts#L167)
  - command: `nl -ba extensions/ollama/src/setup.ts | sed -n '159,176p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **A fetch signal would live for the whole streamed request:** `fetchWithSsrFGuard` builds one abort signal before issuing the request and attaches it to the fetch init; cleanup does not happen until `release()`, which `pullOllamaModelCore` calls only after the response-body read loop finishes. A 30s `AbortSignal.timeout(...)` therefore applies to the whole streaming request, not just initial connect/TLS setup.
  - file: [src/infra/net/fetch-guard.ts:313](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/infra/net/fetch-guard.ts#L313)
  - command: `nl -ba src/infra/net/fetch-guard.ts | sed -n '313,325p'; nl -ba src/infra/net/fetch-guard.ts | sed -n '387,392p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Repo timeout plumbing is stream-oriented, not a 30s connect-only guard:** OpenClaw's guarded Undici path inherits dispatcher `bodyTimeout`/`headersTimeout`, and the global default is 30 minutes (`DEFAULT_UNDICI_STREAM_TIMEOUT_MS = 30 * 60 * 1000`). Tests assert those stream timeouts are what guarded fetches receive when no explicit timeout is passed. That is materially different from the PR's hard 30s abort on the whole request.
  - file: [src/infra/net/undici-global-dispatcher.ts:6](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/infra/net/undici-global-dispatcher.ts#L6)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Local reproduction matches the review concern:** A read-only Node repro against the current runtime showed that a streaming fetch with `AbortSignal.timeout(200)` returned the first chunk, then `reader.read()` failed with `TimeoutError:The operation was aborted due to timeout` before the delayed second chunk arrived. That is the same failure mode the PR review comment describes for large/slow Ollama pulls.
