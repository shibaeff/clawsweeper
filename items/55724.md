## Summary

Keep open. Current `main` already enforces owner-only filtering on `POST /tools/invoke`, but it intentionally treats shared-secret bearer auth as full operator access with owner semantics, so this PR's stricter `senderIsOwner = false` behavior is not implemented on `main` and should not be auto-closed as already-fixed.

## Evidence

- **HTTP invoke handler still gates by computed owner context, not forced non-owner:** `handleToolsInvokeHttpRequest` resolves `senderIsOwner` via `resolveOpenAiCompatibleHttpSenderIsOwner(req, requestAuth)` and then applies `applyOwnerOnlyToolPolicy(tools, senderIsOwner)`. That is materially different from the PR's proposed hardcoded `false` owner context for all authenticated HTTP callers.
