## Summary

Current main already contains the gateway handshake mitigations that match this report: the pre-auth/connect-challenge default is now 10s instead of ~3s, the server and client both read the shared timeout helper, and CLI gateway calls yield one event-loop turn before starting the WebSocket so Windows startup/module-load stalls do not miss `connect.challenge`. The changelog records this as shipped in v2026.3.22.

## Evidence

- **Changelog entry for this exact symptom:** The v2026.3.22 changelog says: "Gateway/WS handshake: raise the default pre-auth handshake timeout to 10 seconds and add OPENCLAW_HANDSHAKE_TIMEOUT_MS as a runtime override so busy local gateways stop dropping healthy CLI connections at 3 seconds."

## Close Comment

Closing as implemented on `main` based on this Codex review. Current OpenClaw already contains the handshake fixes that match this report: `src/gateway/handshake-timeouts.ts` now defaults the pre-auth/connect-challenge window to 10 seconds, `src/gateway/server/ws-connection.ts` uses that shared timeout, and `src/gateway/call.ts` explicitly yields one event-loop turn before starting CLI gateway WebSocket requests to avoid Windows startup/module-load stalls missing `connect.challenge`.
