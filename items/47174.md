## Summary

Keep open. Current `main` still treats `memoryFlushCompactionCount === compactionCount` as the duplicate-run guard, including cycle `0`, and the PR's proposed `compactionCount === 0 => false` behavior would bypass that guard for never-compacted sessions after their first flush.

## Evidence

- **Current guard blocks duplicate flushes by exact compaction-cycle match:** `hasAlreadyFlushedForCurrentCompaction` on `main` computes `compactionCount ?? 0`, reads `memoryFlushCompactionCount`, and returns true when the stored flush cycle equals the current cycle. That includes `0 === 0`.

  - command: `nl -ba src/auto-reply/reply/memory-flush.ts | sed -n '90,130p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Both flush trigger paths depend on that guard:** The normal token-threshold path calls `shouldRunMemoryFlush`, which exits early if `hasAlreadyFlushedForCurrentCompaction` is true, and the transcript-size force-flush path also directly negates the same helper before triggering another flush.
  - file: [src/auto-reply/reply/agent-runner-memory.ts:700](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/auto-reply/reply/agent-runner-memory.ts#L700)

- **Flush persistence records cycle 0 as a real completed flush:** After a flush run, `memoryFlushCompactionCount` is initialized from the active entry's `compactionCount` (falling back to `0`) and then persisted back to the session store. If no compaction-end event happens, a never-compacted session records `memoryFlushCompactionCount = 0`.
