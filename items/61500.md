## Summary

Keep open: current `main` still routes node mode only through `exec`/`system.run`, while the built-in `read`/`write`/`edit` tools remain gateway-local or sandbox-local. I found no node-side `system.read`/`system.write` protocol support, no `tools.fs.host` routing config, and docs still describe split-host nodes as exec-only.

## Evidence

- **Node docs describe split-host nodes as exec-only:** The nodes docs say the gateway forwards `exec` calls to the node host for `host=node`, and define the node host role as executing `system.run`/`system.which` on the node machine.
  - file: [docs/nodes/index.md:57](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/docs/nodes/index.md#L57)
  - command: `nl -ba docs/nodes/index.md | sed -n '55,66p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Documented node system command surface does not include file I/O:** Current docs list macOS/headless node system commands as `system.run`, `system.notify`, `system.which`, and `system.execApprovals.get/set`; there is no documented `system.read` or `system.write` command.
  - file: [docs/nodes/index.md:321](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/docs/nodes/index.md#L321)
