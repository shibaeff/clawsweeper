## Summary

Keep open. Current `main` still has the reported security problem in `src/agents/tools/pdf-native-providers.ts`, but this PR also has a credible unresolved regression risk for proxy-only environments, so there is not strong evidence to close it as implemented or obsolete.

## Evidence

- **Current main still uses raw fetch for Anthropic PDF requests:** `anthropicAnalyzePdf()` builds `baseUrl` from params and calls raw `fetch()` directly, sending the `x-api-key` header to whatever host that URL resolves to. No SSRF guard is applied on current `main`.
  - file: [src/agents/tools/pdf-native-providers.ts:63](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/agents/tools/pdf-native-providers.ts#L63)
  - command: `nl -ba src/agents/tools/pdf-native-providers.ts | sed -n '37,106p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Current main still puts the Gemini API key in the URL:** `geminiAnalyzePdf()` interpolates the API key into `:generateContent?key=...` and then calls raw `fetch()`. Current `main` does not use the safer `x-goog-api-key` header path described in the PR.
  - file: [src/agents/tools/pdf-native-providers.ts:149](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/agents/tools/pdf-native-providers.ts#L149)
  - command: `nl -ba src/agents/tools/pdf-native-providers.ts | sed -n '141,161p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **The vulnerable URL is configurable from model transport:** The PDF tool passes `model.baseUrl` into both native PDF providers, and the unit tests explicitly cover a custom Anthropic base URL. That makes the reported `baseUrl` attack surface plausible on current `main`, not just theoretical.
  - file: [src/agents/tools/pdf-tool.ts:171](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/agents/tools/pdf-tool.ts#L171)
  - command: `nl -ba src/agents/tools/pdf-tool.ts | sed -n '159,190p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **OpenClaw’s guarded-fetch contract warns that strict mode can break proxy-only environments:** `fetchWithTimeoutGuarded()` documents that strict pinned-DNS fetches fail in proxy-only environments and auto-upgrades provider traffic to `TRUSTED_ENV_PROXY` to avoid that. Separate SSRF tests show strict mode does not use `EnvHttpProxyAgent` even when `HTTP_PROXY` is set. That aligns with the PR review comment that `withStrictGuardedFetchMode(...)` here may introduce a real regression.
  - file: [src/media-understanding/shared.ts:300](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/media-understanding/shared.ts#L300)

  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Review discussion already identified the proxy regression risk:** The provided PR discussion includes a Codex review comment on 2026-03-14 calling out that `withStrictGuardedFetchMode` would bypass env-proxy dispatchers and break native PDF calls in proxy-required deployments. No merged follow-up on current `main` resolves either the security issue or that regression concern.

## Risks / Open Questions

- Closing now would leave a plausible SSRF surface and Gemini API-key-in-URL leak unresolved on current `main`.
- Merging the PR as-is still needs maintainer judgment because the strict guarded-fetch change may regress proxy-only deployments for native Anthropic/Gemini PDF calls.
