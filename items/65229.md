## Summary

Keep open. Current `main` still appears to false-warn for env-backed `gateway.auth.token` SecretRefs that resolve through a custom env var, and the PR’s revised pre-resolution approach needs additional doctor-safe handling because the helper currently throws on unresolved refs.

## Evidence

- **Doctor path still skips SecretRef resolution:** `runGatewayAuthHealth` only checks whether `gateway.auth.token` is a SecretRef, then calls `resolveGatewayAuth` on the raw auth config and emits the SecretRef-unavailable warning when `needsToken` stays true. There is no pre-resolution step here on current `main`.
  - file: [src/flows/doctor-health-contributions.ts:118](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/flows/doctor-health-contributions.ts#L118)
  - command: `nl -ba src/flows/doctor-health-contributions.ts | sed -n '110,146p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Resolver only uses OPENCLAW gateway env vars once a SecretRef is present:** `resolveGatewayAuth` drops SecretRef token/password objects by passing `configToken: undefined` / `configPassword: undefined` into credential resolution, and `resolveGatewayCredentialsFromValues` then only reads `OPENCLAW_GATEWAY_TOKEN` / `OPENCLAW_GATEWAY_PASSWORD`. A SecretRef like `{source:"env", id:"GW_TOKEN"}` therefore will not resolve in this doctor path today.
  - file: [src/gateway/auth-resolve.ts:61](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/gateway/auth-resolve.ts#L61)

- **Startup path already has the missing resolution step:** `ensureGatewayStartupAuth` pre-resolves `gateway.auth.token` with `resolveGatewayTokenSecretRefValue(...)` and then feeds the resolved value back through `authOverride` before final auth resolution. That is the behavior the doctor flow is currently missing.
  - file: [src/gateway/startup-auth.ts:160](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/gateway/startup-auth.ts#L160)

## Risks / Open Questions

- Current `main` likely still emits the false-positive doctor warning for env-backed `gateway.auth.token` SecretRefs whose env key is not `OPENCLAW_GATEWAY_TOKEN`.
- The PR’s revised approach needs an unresolved-ref fallback path; otherwise `openclaw doctor` can throw instead of preserving the existing read-only warning behavior.
- The PR branch also carries unrelated drifted file changes, so even a correct doctor fix likely needs rebasing/splitting before landing.
