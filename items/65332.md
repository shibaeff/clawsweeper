## Summary

Close PR #65332 as `cannot_reproduce`: on current `main`, the only `renderMapField` call site normalizes nullish/non-object input to `{}` before rendering, and under the repo’s Node >=22 / ES2023 baseline `{ ...null }` and `{ ...undefined }` are already no-ops rather than throwing.

  - file: [ui/src/ui/views/config-form.node.ts:1009](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/ui/src/ui/views/config-form.node.ts#L1009)

  - file: [ui/src/ui/views/config-form.node.ts:1172](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/ui/src/ui/views/config-form.node.ts#L1172)

  - file: [package.json:1667](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/package.json#L1667)

- **Prior PR discussion already identified the test mismatch:** The existing Codex review comment on this PR correctly pointed out that the added test expects `({ ...value })` with `null` to throw, which does not match current JavaScript behavior on the repo baseline.
