## Close Comment

Closing from a Codex review: current main already implements recovery for this Telegram dead-stop path. In `extensions/telegram/src/polling-session.ts:313`, OpenClaw now detects stalled `getUpdates`, marks the Telegram transport dirty, stops the runner, and restarts polling instead of requiring a full gateway restart. `extensions/telegram/src/monitor.ts:91` also restarts polling after unhandled Telegram network errors, and the behavior is covered by `extensions/telegram/src/polling-session.test.ts:324` and `extensions/telegram/src/monitor.test.ts:721`. The shipped fix is documented in `CHANGELOG.md:484` under v2026.4.18, with additional watchdog hardening in `CHANGELOG.md:388` under v2026.4.22. If this still reproduces on a current release, please open a new issue with `openclaw logs --follow` output around the stall and any Telegram proxy/DNS/network details so maintainers can check for a new regression.
