## Summary

Keep open. Current `main` and the latest shipped release still route `/__openclaw__/canvas/` through canvas-specific auth that only accepts an `Authorization: Bearer ...` header or a node-scoped canvas capability; it does not parse `?token=` and does not fall through to `gateway.auth.mode: "none"` when no bearer header is present. The issue body also appears to confuse the dashboard (`/`) with the canvas host (`/__openclaw__/canvas/`), but there is no evidence that the reported behavior has been fixed on `main` or in `v2026.4.23`.

## Evidence

- **Canvas auth on current main only checks bearer header or canvas capability:** `authorizeCanvasRequest()` reads `getBearerToken(req)`, tries shared-secret auth only when that bearer exists, then falls back to node-scoped `canvasCapability`; otherwise it returns unauthorized. There is no branch that accepts query-string tokens or honors `auth.mode === "none"` when no bearer header is present.
