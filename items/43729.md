## Summary

Keep open. Current `main` has newly added Feishu Bitable CRUD support, but it still does not expose native Bitable attachment upload or record-deletion tools, and `update_record` does not handle local file paths/uploads under the hood.

## Evidence

- **No user-facing delete tools are registered:** `registerFeishuBitableTools` registers `feishu_bitable_get_meta`, `list_fields`, `list_records`, `get_record`, `create_record`, `update_record`, `create_app`, and `create_field`. There is no `feishu_bitable_delete_record` or batch delete tool.
  - file: [extensions/feishu/src/bitable.ts:594](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/feishu/src/bitable.ts#L594)
  - command: `nl -ba extensions/feishu/src/bitable.ts | sed -n '520,780p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **`update_record` is a thin pass-through:** The Bitable update path forwards `fields` directly to `client.bitable.appTableRecord.update(...)` with no file upload step, no local-path handling, and no conversion to attachment `file_token`s.
  - file: [extensions/feishu/src/bitable.ts:432](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/feishu/src/bitable.ts#L432)

- **Existing Feishu file upload path is doc-specific:** The only native Feishu file upload implementation found uploads via `client.drive.media.uploadAll(...)` with `parent_type: "docx_file"` and `parent_node: docToken`, which is specific to Feishu docs rather than Bitable attachments.
  - file: [extensions/feishu/src/docx.ts:839](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/feishu/src/docx.ts#L839)
