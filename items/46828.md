## Summary

Close as `clawhub`: current `main` already exposes shipped plugin seams that can intercept `web_fetch`/`browser`, request operator approval, and forward those approvals across existing channels, while `VISION.md` says optional capability should usually live as plugins/ClawHub rather than add one-off core approval namespaces and config.

## Evidence

- **Vision scopes optional capability to plugins/ClawHub:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, and plugin promotion belongs in ClawHub.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/VISION.md#L54)
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Plugin hooks can gate tool calls with approval:** `before_tool_call` receives `event.toolName`, `event.params`, and agent/session context, and it can return `requireApproval` plus `onResolution` for `allow-once` / `allow-always` / `deny`, which is enough to implement URL-based approval for `web_fetch` and `browser` in a plugin.
  - file: [docs/plugins/hooks.md:107](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/docs/plugins/hooks.md#L107)
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Plugin approvals are already first-class in core:** Current `main` already ships `plugin.approval.request`, `plugin.approval.waitDecision`, and `plugin.approval.resolve`, with forwarded request/resolve handling in gateway server methods.
  - file: [src/gateway/server-methods/plugin-approval.ts:26](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/gateway/server-methods/plugin-approval.ts#L26)
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Plugin approval forwarding is already configurable:** `ApprovalsConfig` already includes independent `approvals.plugin` forwarding, so a plugin can reuse existing approval delivery paths without adding a new core `approvals.http` surface.
  - file: [src/config/types.approvals.ts:29](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/config/types.approvals.ts#L29)
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **No shipped HTTP-specific approval surface on main:** Search on current `main` found no `http.approval`, `approvals.http`, `httpPolicy`, or `withHttpApprovalGate` symbols, so this PR is not already implemented in core; the relevant shipped path is the generic plugin approval seam instead.
  - command: `rg -n "http\.approval|approvals\.http|httpPolicy|withHttpApprovalGate" src docs ui extensions`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
