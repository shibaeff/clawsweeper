## Summary

Current main already provides CA-based TLS trust controls for self-signed/custom model-provider HTTPS endpoints and also auto-applies the system CA bundle where Node startup needs it, so the requested capability is implemented well enough.

## Evidence

- **Provider TLS config surface exists:** Configured model providers can set `request.tls.ca`, `cert`, `key`, `passphrase`, `serverName`, and `insecureSkipVerify` in the public config type.
  - file: [src/config/types.provider-request.ts:18](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/config/types.provider-request.ts#L18)
  - command: `nl -ba src/config/types.provider-request.ts | sed -n '1,120p'`

  - file: [docs/gateway/config-tools.md:410](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/docs/gateway/config-tools.md#L410)
  - command: `nl -ba docs/gateway/config-tools.md | sed -n '400,430p'`

## Risks / Open Questions

- The reported 'trust any cert / disable verification' escape hatch is intentionally not supported on the provider transport path; current main closes the gap via CA/cert management rather than blanket insecure TLS.
- The certificate-management surface is config/docs based rather than a dedicated UI workflow.

## Close Comment

Thanks for the report. A Codex review against current `main` found that this is now implemented well enough to close. OpenClaw now exposes provider-level TLS controls for self-hosted/custom model endpoints through `models.providers.*.request.tls` (`src/config/types.provider-request.ts:18`, `docs/gateway/config-tools.md:410`), and the transport layer applies those values as upstream HTTPS connect options in `src/agents/provider-request-config.ts:545`. Current `main` also auto-applies `NODE_EXTRA_CA_CERTS` on startup when Node needs the system CA bundle (`src/entry.respawn.ts:46`, `docs/help/environment.md:142`). The changelog shows the provider TLS/request-override work shipped in `v2026.4.5` (`CHANGELOG.md:1194`) and the CA-bundle startup fix in `2026.3.13` (`CHANGELOG.md:2381`). Because the requested self-signed certificate trust/management path is present on current main, I’m closing this as implemented.
