## Summary

Keep open. Current `main` still hard-codes Slack thread replies into `:thread:<id>` session keys, still gates Slack thread starter/history context on first-seen session state, and does not expose any `session.threadIsolation` or equivalent config surface in the current schema/docs.

## Evidence

- **Thread suffix still default:** `resolveThreadSessionKeys()` still defaults `useSuffix` to `true`, so thread IDs append `:thread:<normalizedThread>` unless a caller explicitly overrides it.
  - file: [src/routing/session-key.ts:250](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/routing/session-key.ts#L250)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Slack routing still uses thread-scoped session key:** Slack routing passes `canonicalThreadId` into `resolveThreadSessionKeys()` without `useSuffix: false`, and thread history keys still use `sessionKey` for `historyScope === "thread"`.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:106](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L106)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Thread history fetch still gated by prior session timestamp:** Slack thread history is only fetched when `initialHistoryLimit > 0 && !threadSessionPreviousTimestamp`, so shared-session behavior requested in the issue is not implemented here.
  - file: [extensions/slack/src/monitor/message-handler/prepare-thread-context.ts:152](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/slack/src/monitor/message-handler/prepare-thread-context.ts#L152)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Thread starter injection still first-turn only:** `ThreadStarterBody` and `IsFirstThreadTurn` are still suppressed once `threadSessionPreviousTimestamp` exists, matching the issue's described gating that would break shared-session threads.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:689](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/slack/src/monitor/message-handler/prepare.ts#L689)

Slack thread config also only exposes `historyScope`, `inheritParent`, `initialHistoryLimit`, and `requireExplicitMention`.

  - file: [docs/channels/slack.md:517](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/docs/channels/slack.md#L517)
