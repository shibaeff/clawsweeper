## Summary

Current main already has a first-class multi-agent/session inventory surface: `sessions_list`/`sessions.list` aggregate all agent stores and expose `agentId`, model, token/context usage, timestamps, and run status, while spawned-agent lifecycle is persisted in a centralized `subagents/runs.json` registry that other worker processes can read.

## Risks / Open Questions

- This closes the reported gap as stated, but OpenClaw still does not appear to expose generic host/container CPU-RAM telemetry; the current surfaces are session/subagent oriented.

## Close Comment

Closing from a Codex review. Current `main` already implements this well enough through the built-in session and subagent inventory surfaces. `sessions_list` now returns `agentId`, model, token/context usage, timestamps, and run status (`docs/concepts/session-tool.md:27`, `src/agents/tools/sessions-list-tool.ts:233`). Gateway session discovery aggregates all agent stores instead of a single local store (`src/gateway/session-utils.ts:1011`), and the row builder pulls live subagent status plus model/context/cost metadata (`src/gateway/session-utils.ts:1273`). Spawned-agent lifecycle is also persisted in the centralized `subagents/runs.json` registry and merged for cross-worker reads (`src/agents/subagent-registry.store.ts:45`, `src/agents/subagent-registry-state.ts:37`). This is also present in the shipped `v2026.4.22` docs (`docs/cli/sessions.md:29`). If you still need host/container CPU or RAM telemetry rather than OpenClaw session/subagent inventory, please open a new issue scoped to that remaining gap.
