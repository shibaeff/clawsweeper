## Summary

Current `main` does not implement the requested cross-channel delivery-failure notification for agent replies. There is a new Microsoft Teams-specific system-event fallback, but the broader channel reply paths cited in the issue still only log delivery failures instead of injecting a delivery-failure message back into agent context.

## Evidence

- **System events are the mechanism that feeds queued notices back into the next agent turn:** `drainFormattedSystemEvents` drains queued system events and prepends them as `System:` lines to the next prompt, so this is the relevant path for making an agent aware of failed delivery.
  - file: [src/auto-reply/reply/session-system-events.ts:15](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/auto-reply/reply/session-system-events.ts#L15)
  - command: `nl -ba src/auto-reply/reply/session-system-events.ts | sed -n '15,40p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Microsoft Teams has a bespoke delivery-failure system event:** `queueDeliveryFailureSystemEvent` explicitly enqueues a system event saying the previous reply was not delivered and that the user may not have received it.
  - file: [extensions/msteams/src/reply-dispatcher.ts:183](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/msteams/src/reply-dispatcher.ts#L183)

  - file: [extensions/feishu/src/reply-dispatcher.ts:473](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/feishu/src/reply-dispatcher.ts#L473)

## Risks / Open Questions

- Some individual channels may have bespoke recovery or retry behavior beyond the sampled Feishu/Telegram/Microsoft Teams paths, but the requested cross-channel feature is not implemented consistently on `main`.
- This review was read-only and did not exercise live channel integrations, so the conclusion is based on current code/docs paths rather than runtime reproduction against external providers.
