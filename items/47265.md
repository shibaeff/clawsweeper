## Summary

Current main already contains the loopback WebSocket handshake timeout fix and the changelog says it shipped in v2026.3.22. The gateway/server timeout is now 10s, the CLI path yields before opening the socket to avoid starving the connect challenge, and tests/docs were updated to match.

## Evidence

- **Changelog entry:** The release notes for 2026.3.22 explicitly call out this bug class: 'Gateway/WS handshake: raise the default pre-auth handshake timeout to 10 seconds ... so busy local gateways stop dropping healthy CLI connections at 3 seconds.'
  - file: [CHANGELOG.md:2443](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/CHANGELOG.md#L2443)
- **Current timeout default:** Current main sets DEFAULT_PREAUTH_HANDSHAKE_TIMEOUT_MS to 10000 and resolves OPENCLAW_HANDSHAKE_TIMEOUT_MS as the runtime override for the pre-auth handshake.
  - file: [src/gateway/handshake-timeouts.ts:1](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/handshake-timeouts.ts#L1)
- **Gateway server applies timeout:** The WebSocket server arms the pre-auth handshake timer from getPreauthHandshakeTimeoutMsFromEnv() and logs 'handshake timeout' before closing an unauthenticated socket.
  - file: [src/gateway/server/ws-connection.ts:366](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/server/ws-connection.ts#L366)
- **CLI-side mitigation on main:** The CLI gateway call path now yields to the event loop before starting the WebSocket connection, with an inline note that heavy synchronous module loading could otherwise delay the connect.challenge frame and cause handshake timeouts.
  - file: [src/gateway/call.ts:479](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/call.ts#L479)
- **Regression coverage and docs:** Tests cover the timeout/env behavior and silent-handshake closure, and protocol docs now document the preauth/connect-challenge timeout as 10000 ms.
  - file: [src/gateway/server.auth.default-token.suite.ts:80](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/server.auth.default-token.suite.ts#L80)

## Close Comment

Closing as implemented on main after Codex review. This report matches a fix that is already in the codebase and documented as shipped in `v2026.3.22`: `CHANGELOG.md:2443` says the Gateway/WebSocket pre-auth handshake timeout was raised from 3s to 10s so healthy local CLI connections stop getting dropped. Current main reflects that in `src/gateway/handshake-timeouts.ts:1`, the gateway server uses that timeout in `src/gateway/server/ws-connection.ts:366`, and the CLI call path now yields before opening the socket to avoid starving `connect.challenge` handling (`src/gateway/call.ts:479`). Coverage/docs were added as well (`src/gateway/server.auth.default-token.suite.ts:80`, `docs/gateway/protocol.md:493`). If anyone still sees the same failure on a current release newer than `v2026.3.22`, please open a fresh issue with the exact version, platform, and current gateway logs.
