## Summary

Keep open. Current `main` has partial `retry_after` handling for individual Telegram API calls, but it does not implement the requested outbound per-chat send queue, configurable queue depth/rate limiting, or a non-blocking fire-and-drain delivery path. The issue remains materially unimplemented.

## Evidence

- **Telegram config lacks outbound queue controls:** `TelegramAccountSchemaBase` defines `retry` and other Telegram channel settings, but there is no `outboundQueue`, `perChatRateLimit`, or `maxQueueDepth` field. The schema is also closed with `.strict()`, so the proposed config shape is not implemented on `main`.
  - file: [src/config/zod-schema.providers-core.ts:245](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/config/zod-schema.providers-core.ts#L245)
  - command: `nl -ba src/config/zod-schema.providers-core.ts | sed -n '207,252p'; nl -ba src/config/zod-schema.providers-core.ts | sed -n '320,340p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Telegram send path is direct, not queued:** `sendMessageTelegram()` resolves the chat and then calls `api.sendMessage(...)` directly inside the request path. The only module-level `Map` in this file is a client-options cache, not a per-chat outbound queue or limiter.
  - file: [extensions/telegram/src/send.ts:617](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/telegram/src/send.ts#L617)

- **Current retry logic is per-call retry_after backoff only:** `retryAsync()` reads `retryAfterMs` from an error and delays the next retry for that same call. This is retry/backoff behavior, not a shared outbound queue or per-chat serialization mechanism.
  - file: [src/infra/retry.ts:114](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/infra/retry.ts#L114)
