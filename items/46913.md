## Summary

Current `main` already provides a single public `tool_result_persist` hook for OpenClaw-owned tool-result messages, and it is wired through the session-manager append path used by embedded runs and compaction. That gives one central, update-stable interception point to rewrite/redact tool results before they are appended into the OpenClaw transcript/context path. The remaining documented limitation is Codex-native tool-result records, which use a separate middleware path.

## Close Comment

Closing as `implemented_on_main` after a Codex review. Current OpenClaw already exposes a single public `tool_result_persist` hook for OpenClaw-owned tool-result messages (`src/plugins/hook-types.ts:361`, `src/plugins/hooks.ts:828`), and the embedded-run session path wires that hook through `guardSessionManager()` / `installSessionToolResultGuard()` so each `toolResult` is transformed before the underlying append happens (`src/agents/session-tool-result-guard-wrapper.ts:49`, `src/agents/session-tool-result-guard.ts:194`). This is also documented in `docs/concepts/agent-loop.md:101` and covered by regression tests in `src/agents/session-tool-result-guard.tool-result-persist-hook.test.ts:112`. One documented caveat: Codex-native tool-result records are handled through a separate Codex middleware path, not this public hook (`docs/plugins/codex-harness.md:567`).
