## Summary

Current `main` still bypasses `message_sending` for Telegram streaming final delivery. The non-streaming `deliverReplies()` path runs the hook, but the streaming preview-finalization path edits/materializes the preview directly and only emits the internal `message_sent` hook afterward.

## Evidence

- **Hook contract expects outbound delivery interception:** Plugin docs still describe `message_sending` as the outbound rewrite/cancel hook for message delivery.
  - file: [docs/plugins/hooks.md:85](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/docs/plugins/hooks.md#L85)
  - command: `nl -ba docs/plugins/hooks.md | sed -n '80,95p'; nl -ba docs/plugins/hooks.md | sed -n '186,206p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Telegram normal reply delivery runs `message_sending`:** `deliverReplies()` gets the global hook runner, checks `hasHooks("message_sending")`, and calls `runMessageSending(...)` before sending the reply.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:626](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/telegram/src/bot/delivery.replies.ts#L626)
  - command: `nl -ba extensions/telegram/src/bot/delivery.replies.ts | sed -n '620,710p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Streaming finalization can complete without `sendPayload`:** The lane text deliverer finalizes streamed replies by materializing or editing the preview and returns `preview-finalized`; it only falls back to `sendPayload(...)` later when preview finalization is not possible.
  - file: [extensions/telegram/src/lane-delivery-text-deliverer.ts:523](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/telegram/src/lane-delivery-text-deliverer.ts#L523)
  - command: `nl -ba extensions/telegram/src/lane-delivery-text-deliverer.ts | sed -n '520,620p'`
