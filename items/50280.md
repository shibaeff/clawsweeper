## Summary

Current `main` already handles this case: manual cron runs normalize stale `runningAtMs` state before the already-running check, stuck markers are cleared during maintenance normalization, and regression tests cover both timeout cleanup and forced manual reruns after a stale marker.

## Evidence

- **Manual run preflight clears stale markers before blocking on running state:** `inspectManualRunPreflight()` calls `recomputeNextRunsForMaintenance(state)` immediately before checking `job.state.runningAtMs`, with an inline comment stating this prevents stale markers from blocking manual triggers.
  - file: [src/cron/service/ops.ts:510](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/cron/service/ops.ts#L510)

- **Maintenance normalization clears stuck `runningAtMs`:** `normalizeJobStateForMaintenance()` clears `job.state.runningAtMs` when it is older than `STUCK_RUN_MS`, logging `cron: clearing stuck running marker`.
  - file: [src/cron/service/jobs.ts:399](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/cron/service/jobs.ts#L399)
  - command: `nl -ba src/cron/service/jobs.ts | sed -n '392,410p'`

## Close Comment

Closing from a Codex review: this is already implemented on current `main`, and the shipped `v2026.4.22` tree contains the same fix. Manual `cron run` now normalizes cron state before the already-running check, so stale `runningAtMs` markers no longer block forced/manual runs (`src/cron/service/ops.ts:510`). The maintenance normalization path clears stuck running markers (`src/cron/service/jobs.ts:399`), and the result path clears `runningAtMs` when a run finishes or errors (`src/cron/service/timer.ts:416`). There are direct regression tests for both the timeout cleanup case and the stale-marker manual rerun case (`src/cron/service/ops.regression.test.ts:200` and `src/cron/service/ops.regression.test.ts:241`).
