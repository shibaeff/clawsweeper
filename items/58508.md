## Summary

Current main already contains both parts of the fix: explicit-proxy SSRF checks no longer reject operator-configured proxy hosts against the target allowlist, and Telegram media downloads now have a Telegram-specific private-network opt-in plus RFC2544 allowance. The changelog shows this shipped in 2026.4.7, and the issue thread later confirmed updating resolved it.

## Evidence

- **Explicit proxy hostname fix in fetch guard:** The explicit-proxy SSRF path clears the target-scoped hostname allowlist when `allowPrivateProxy` is true, so configured proxy hosts like `localhost` are not rejected just because the target allowlist is `api.telegram.org`.
  - file: [src/infra/net/fetch-guard.ts:174](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/infra/net/fetch-guard.ts#L174)
  - command: `nl -ba src/infra/net/fetch-guard.ts | sed -n '160,260p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Regression test for Telegram via localhost proxy:** Regression coverage uses `https://api.telegram.org/file/...` with an explicit `http://localhost:6152` proxy and expects the guarded fetch to succeed when `allowPrivateProxy` is enabled, matching the proxy-related failure mode described in the issue comments.
  - file: [src/infra/net/fetch-guard.ssrf.test.ts:1084](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/infra/net/fetch-guard.ssrf.test.ts#L1084)

## Risks / Open Questions

- The original report also mentioned Telegram CDN IP classification; current main addresses the proxy-hostname path directly and adds a Telegram-only private-network opt-in for trusted fake-IP environments, but operators still need that opt-in when their network intentionally rewrites Telegram media hosts to non-RFC2544 private/special-use ranges.
