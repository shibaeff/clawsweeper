## Summary

Current main already distinguishes agent-origin inter-session messages from normal human user turns for subagents, and the distinction shipped in v2026.4.22.

## Evidence

- **Provenance model distinguishes user vs inter-session inputs:** OpenClaw defines separate input provenance kinds, including `external_user` and `inter_session`, and persists provenance onto user messages.
  - file: [src/sessions/input-provenance.ts:4](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/sessions/input-provenance.ts#L4)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Nested agent/subagent sends are tagged as inter-session:** Agent-to-agent/subagent messages are sent with `inputProvenance.kind = "inter_session"` plus source session/tool metadata.
  - file: [src/agents/tools/agent-step.ts:32](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/agents/tools/agent-step.ts#L32)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Model-facing transcript gets an explicit prefix:** Before replay to the model, inter-session user messages are rewritten with `[Inter-session message]` plus `sourceSession`/`sourceChannel`/`sourceTool`, so a subagent can tell the message came from another agent session rather than a direct human turn.
  - file: [src/agents/pi-embedded-runner/replay-history.ts:45](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/agents/pi-embedded-runner/replay-history.ts#L45)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Behavior is covered by tests:** Unit coverage asserts that an inter-session user turn is annotated with `[Inter-session message]` and `sourceSession=agent:main:req` before context sanitization.
  - file: [src/agents/pi-embedded-runner.sanitize-session-history.test.ts:432](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/agents/pi-embedded-runner.sanitize-session-history.test.ts#L432)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Subagent announce flow preserves inter-session provenance:** E2E coverage asserts subagent completion messages sent back to the requester carry `inputProvenance.kind = "inter_session"` and the child session key as the source.
  - file: [src/agents/subagent-announce.format.e2e.test.ts:610](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/agents/subagent-announce.format.e2e.test.ts#L610)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Shipped in latest release:** The same provenance and replay-history implementation is present in release tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Risks / Open Questions

- The issue body asks for a webchat-visible human prefix, while the implementation inspected here solves the underlying distinction at the session/provenance and model-context layer rather than by adding a separate UI-only badge.

## Close Comment

Closing as implemented on current OpenClaw. Codex review found that subagent traffic already distinguishes direct user turns from agent-origin turns: `src/sessions/input-provenance.ts:4` defines separate provenance kinds including `external_user` and `inter_session`, nested agent/subagent sends stamp `inter_session` provenance (`src/agents/tools/agent-step.ts:32`), and replay prepends an explicit `[Inter-session message] sourceSession=...` marker before the subagent model sees that turn (`src/agents/pi-embedded-runner/replay-history.ts:45`). There is also test coverage for the annotation path in `src/agents/pi-embedded-runner.sanitize-session-history.test.ts:432` and for subagent announce provenance in `src/agents/subagent-announce.format.e2e.test.ts:610`. This implementation is already present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
