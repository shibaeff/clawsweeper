## Summary

Keep open. Current `main` still does not implement this PR's core Feishu voice-hardening behavior: non-Opus audio still degrades to generic file delivery, and the Feishu streaming/outbound paths shown here still do not strip internal `[[tts]]` markup before delivery.

## Evidence

- **Unsupported Feishu audio still falls back to generic file:** `resolveFeishuOutboundMediaKind()` only treats `.opus`/`.ogg` (or `audio/ogg`/`audio/opus`) as sendable Feishu audio. Other audio formats fall through `detectFileType()` and end up as generic file/media routing instead of voice delivery.
  - file: [extensions/feishu/src/media.ts:539](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/feishu/src/media.ts#L539)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '524,568p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Current test suite locks in MP3-as-file behavior:** `media.test.ts` explicitly asserts that `song.mp3` uploads as `file_type: "stream"` and sends with `msg_type: "file"`, which is the opposite of this PR's transcode-to-Opus goal.
  - file: [extensions/feishu/src/media.test.ts:239](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/feishu/src/media.test.ts#L239)
  - command: `nl -ba extensions/feishu/src/media.test.ts | sed -n '239,260p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Feishu partial streaming path still forwards raw text:** `createFeishuReplyDispatcher()` sends `payload.text` directly to `queueStreamingUpdate()` for partial replies. There is no Feishu-local cleanup for internal `[[tts...]]` tags on this path in current `main`.
  - file: [extensions/feishu/src/reply-dispatcher.ts:536](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/feishu/src/reply-dispatcher.ts#L536)

- **Outbound normalization still does not handle TTS tags:** `normalizeReplyPayloadsForDelivery()` relies on `parseReplyDirectives()`, and `parseReplyDirectives()` only strips reply/audio directive tags plus media markers. It does not remove `[[tts]]` / `[[tts:text]]` markup before delivery.
  - file: [src/infra/outbound/payloads.ts:138](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/infra/outbound/payloads.ts#L138)
