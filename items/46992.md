## Summary

Keep open. Current `main` still documents native Windows `system.run`/`exec` mojibake as an active problem and the runtime still passes raw commands to PowerShell without any UTF-8 initialization. This PR is not already implemented on `main`, and there is not enough basis to call it obsolete. It also still needs maintainer review because the proposed wrapper prepends statements before the user command, which can change PowerShell script semantics for directive-first commands.

## Evidence

- **Docs still describe the bug as current behavior:** `docs/help/faq-first-run.md` has an FAQ titled "Windows exec output shows garbled Chinese text" and tells users to manually run `chcp 65001`, set `[Console]::InputEncoding`, `[Console]::OutputEncoding`, and `$OutputEncoding`, then restart the gateway. That is strong evidence the Windows exec encoding problem is still present on current `main`.

## Risks / Open Questions

- Closing as implemented would be incorrect: current `main` still documents a manual workaround for the exact Windows exec encoding problem.
- Closing as cannot_reproduce would be weak: the runtime path still lacks any UTF-8 wrapper, so the report remains plausible on native Windows.
- The PR should stay open for maintainer judgment because the proposed prepend-style wrapper can change PowerShell semantics for commands that must appear first in script scope, such as `param(...)` or `#requires`.
