## Summary

The request is a useful memory extension idea, but current OpenClaw scope puts memory behavior behind plugin slots and plugin hooks rather than adding more core-integrated memory systems.

## Evidence

- **Vision scope anchors optional capability to plugins and ClawHub:** `VISION.md` says optional capability should usually ship as plugins, plugin promotion belongs in ClawHub, and memory is a special plugin slot with multiple memory options already shipped.
  - file: [VISION.md:55](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/VISION.md#L55)
  - command: `nl -ba VISION.md | sed -n '55,90p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Memory is an explicit exclusive plugin slot:** `src/plugins/slots.ts` maps plugin kind `memory` to the `memory` slot and defaults that slot to `memory-core`, showing memory is designed as swappable plugin-owned functionality, not core-only behavior.
  - file: [src/plugins/slots.ts:12](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/plugins/slots.ts#L12)

- **Docs already present memory as plugin-provided:** `docs/cli/memory.md` says `openclaw memory` is provided by the active memory plugin, defaulting to `memory-core`.
  - file: [docs/cli/memory.md:12](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/docs/cli/memory.md#L12)
  - command: `nl -ba docs/cli/memory.md | sed -n '1,220p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Plugin docs list multiple memory plugins:** `docs/tools/plugin.md` lists bundled memory plugins, including `memory-core` and `memory-lancedb` with auto-recall/capture, reinforcing that memory extensions belong in the plugin surface.
  - file: [docs/tools/plugin.md:109](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/docs/tools/plugin.md#L109)

## Risks / Open Questions

- The issue proposes a Python/DeepSeek-specific implementation outside this repository; closing as ClawHub means maintainers are declining core integration scope, not judging the idea's usefulness.
- If maintainers want an officially owned memory plugin later, they could still reopen scope separately, but current repository guidance points away from core adoption.
