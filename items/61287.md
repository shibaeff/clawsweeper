## Summary

Keep open. Current `main` still mixes cumulative transcript usage with latest-turn totals for session fallback, and the context-window lookup can still fall back from `claude-cli/claude-sonnet-4-6` to the bare Venice `claude-sonnet-4-6` 1,000,000-token entry.

## Evidence

- **Transcript fallback still sums cache/input across turns:** `extractLatestUsageFromTranscriptChunk()` iterates every assistant message in the transcript chunk, adds `input`, `output`, `cacheRead`, and `cacheWrite` to running totals, but overwrites `snapshot.totalTokens` with each message's derived total. That leaves the final snapshot with cumulative cache/input fields but only the latest `totalTokens`, which matches the reported inflated-context bug family.
  - file: [src/gateway/session-utils.fs.ts:444](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/gateway/session-utils.fs.ts#L444)

  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Current tests codify the aggregate/latest mismatch:** The test named `aggregates assistant usage across the full transcript and keeps the latest context snapshot` expects two assistant turns to produce `inputTokens: 4200` and `cacheRead: 1500`, while `totalTokens` remains `3300` from the last turn. That is the same mismatch described in the issue, so it is not fixed on `main`.
  - file: [src/gateway/session-utils.fs.test.ts:731](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/gateway/session-utils.fs.test.ts#L731)
