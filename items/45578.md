## Summary

Current `main` already prevents the Anthropic `tool_use_id`/`tool_result` transcript corruption this PR tries to paper over, and it already has existing `/new` recovery copy for invalid replay/session state. The proposed regex-based rewrite is now obsolete and would still be risky because it matches any error containing `tool_use_id`, including ordinary schema/validation failures.

## Close Comment

Closing from a Codex review. Current `main` already prevents this failure earlier in the pipeline instead of post-processing the raw provider error: `src/agents/session-tool-result-guard.ts:217`, `src/agents/pi-embedded-runner/run/attempt.ts:1627`, and `src/agents/pi-embedded-helpers/turns.ts:167` all repair or drop orphaned `tool_use`/`tool_result` state before replay, with regression coverage in `src/agents/pi-embedded-runner/run/attempt.test.ts:2302` and `src/agents/session-transcript-repair.test.ts:147`. There is also already a `/new` recovery path for replay-invalid state in `src/agents/pi-embedded-helpers/errors.ts:1087`. Given that, I could not justify merging this stale regex rewrite, especially since its bare `tool_use_id` match would also hide ordinary validation/schema errors behind a generic corruption message. If a current `main` build still surfaces a concrete mismatch payload, please open a fresh issue or PR with the exact error text and replay path.
