## Summary

Current main already implements restart recovery for recurring cron jobs, persists runtime state separately from job definitions, records finished runs, and reconciles orphaned cron tasks so the false-positive/silent-drop behavior described for recurring jobs is no longer the current design.

## Evidence

- **Startup clears stale running markers and replays recurring jobs:** `start()` clears persisted `runningAtMs` markers on startup, skips only interrupted one-shot jobs, and immediately runs missed recurring jobs after restart.
  - file: [src/cron/service/ops.ts:93](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/cron/service/ops.ts#L93)

- **Regression test covers interrupted recurring-job restart recovery:** The restart catch-up test verifies that a recurring job interrupted by restart is recovered on first startup, its stale running marker is cleared, and the job is replayed successfully.
  - file: [src/cron/service.restart-catchup.test.ts:118](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/cron/service.restart-catchup.test.ts#L118)

- The implemented fix is strongest for the report's recurring-job case (`every 1 minute`); interrupted one-shot jobs are intentionally treated differently.

## Close Comment

Closing from a Codex review because current `main` already covers the recurring-cron restart case described here. On startup, cron now clears stale `runningAtMs` markers and immediately replays interrupted recurring jobs (`src/cron/service/ops.ts:93`), with regression coverage for that exact path (`src/cron/service.restart-catchup.test.ts:118`). Finished runs are appended to the cron run log used by `openclaw cron runs` (`src/gateway/server-cron.ts:549`), runtime state is persisted in `cron/jobs-state.json` (`src/cron/store.ts:34`), and stale cron-backed tasks are reconciled as `lost` instead of silently looking healthy when the runtime no longer owns them (`src/tasks/task-registry.maintenance.ts:124`). This also shipped in `v2026.4.5` per `CHANGELOG.md:1280`. If you can still reproduce this on `v2026.4.22` or newer, please open a fresh issue with current `openclaw cron runs --id <jobId>` output and task logs.
