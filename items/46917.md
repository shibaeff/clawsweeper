## Summary

Current main already provides a supported config/session opt-out for DM tool-status messages via verbose controls, so the requested capability is implemented well enough.

## Evidence

- **Dispatch uses configured verbose level to gate progress/tool-status delivery:** `dispatchReplyFromConfig` builds `shouldEmitVerboseProgress` from session `verboseLevel`, agent `verboseDefault`, or global `agents.defaults.verboseDefault`, then suppresses working-status / plan-update delivery when that resolves to `off`.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:280](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/auto-reply/reply/dispatch-from-config.ts#L280)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Verbose-off is the runtime default:** `getReplyRun` falls back to `resolvedVerboseLevel ?? "off"`, so tool summaries are not mandatory even when no explicit override is set.
  - file: [src/auto-reply/reply/get-reply-run.ts:803](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/auto-reply/reply/get-reply-run.ts#L803)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Tool-result emission is explicitly disabled when verbose is off:** Helper coverage asserts `createShouldEmitToolResult({ resolvedVerboseLevel: "off" })()` is `false`, and only `on`/`full` enable tool-result forwarding.
  - file: [src/auto-reply/reply/agent-runner-helpers.test.ts:50](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/auto-reply/reply/agent-runner-helpers.test.ts#L50)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Docs expose the config knob:** Agent config docs list `verboseDefault` with values `"off" | "on" | "full"` and document the default as `"off"`.
  - file: [docs/gateway/config-agents.md:363](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/docs/gateway/config-agents.md#L363)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Docs expose the per-session opt-out:** Thinking/verbose docs state `/verbose off` stores a session override, and tool summaries are only sent when verbose is on.
  - file: [docs/tools/thinking.md:73](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/docs/tools/thinking.md#L73)

## Risks / Open Questions

- Current main solves this through agent/session verbose controls, not a separate DM-vs-group-specific config key. If maintainers want independent channel-type toggles later, that would be a new feature request rather than an unimplemented bug.

## Close Comment

Closing after Codex review: current `main` already has a supported way to disable these DM tool-status messages. `dispatchReplyFromConfig` now reads session/agent/global verbose settings (`src/auto-reply/reply/dispatch-from-config.ts:280`) and suppresses verbose progress when that resolves to `off`, while the runtime default is also `off` (`src/auto-reply/reply/get-reply-run.ts:803`). The config surface is documented as `agents.defaults.verboseDefault` with default `"off"` (`docs/gateway/config-agents.md:363`), and per-session opt-out is available via `/verbose off` (`docs/tools/thinking.md:73`). I also verified this is already present in the shipped `v2026.4.22` release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, so there is no remaining maintainer action to take on this request.
