## Summary

Current main already routes Discord voice STT through the shared media-understanding/provider transport, and that transport honors `models.providers.<id>.request.allowPrivateNetwork`. The issue body's described Discord-local bare `fetch()` path does not match the current source. Main also now threads `tools.web.fetch.ssrfPolicy` into media fetches for the broader media path.

## Evidence

- **Discord voice STT no longer uses a Discord-local bare fetch:** `transcribeAudio()` in the Discord voice manager calls `getDiscordRuntime().mediaUnderstanding.transcribeAudioFile(...)`, so voice-channel STT goes through the shared media-understanding runtime rather than a Discord-specific raw `fetch()` path.

## Risks / Open Questions

- `tools.web.fetch.ssrfPolicy` still only exposes `allowRfc2544BenchmarkRange`; a generic `web_fetch` private-network opt-in would be a separate enhancement, but it is not required for the current Discord voice STT path.
- This review was read-only and did not run a live Discord/Groq reproduction, so the decision relies on current source, tests, docs, and commit history rather than an end-to-end network run.
