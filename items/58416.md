## Summary

Current `main` now auto-migrates legacy `plugins.entries.voice-call.config.tts.<provider>` blocks into `tts.providers.<provider>` before config validation, which covers the reported `tts.openai` shape and resolves the startup validation failure on `main`.

## Evidence

- **Strict voice-call TTS schema still only accepts canonical `providers` shape:** The voice-call plugin config uses `TtsConfigSchema` for `plugins.entries.voice-call.config.tts`, and that schema is strict with `providers` as the canonical provider container. This explains the original `must NOT have additional properties` failure for legacy `tts.openai` keys unless compat runs first.
  - file: [src/config/zod-schema.core.ts:500](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/config/zod-schema.core.ts#L500)
  - command: `nl -ba src/config/zod-schema.core.ts | sed -n '500,526p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Voice-call plugin config binds directly to that strict TTS schema:** `VoiceCallConfigSchema` defines `tts: TtsConfigSchema`, so plugin config validation depends on the shared TTS shape.
  - file: [extensions/voice-call/src/config.ts:358](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/voice-call/src/config.ts#L358)
  - command: `nl -ba extensions/voice-call/src/config.ts | sed -n '353,362p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Current main adds runtime migration for legacy voice-call TTS provider keys:** `LEGACY_CONFIG_MIGRATIONS_RUNTIME_TTS` explicitly detects `plugins.entries.voice-call.config.tts.openai|elevenlabs|microsoft|edge` and rewrites them into `plugins.entries.voice-call.config.tts.providers.<provider>`.
  - file: [src/commands/doctor/shared/legacy-config-migrations.runtime.tts.ts:10](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/commands/doctor/shared/legacy-config-migrations.runtime.tts.ts#L10)
  - command: `nl -ba src/commands/doctor/shared/legacy-config-migrations.runtime.tts.ts | sed -n '1,140p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Config read path applies runtime compat before validation:** `resolveLegacyConfigForRead` runs `applyRuntimeLegacyConfigMigrations(resolvedConfigRaw)` and returns the migrated config as `effectiveConfigRaw`, and that migrated config is what the read snapshot validates. That means startup/config-load no longer rejects the legacy voice-call TTS shape.
  - file: [src/config/io.ts:1072](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/config/io.ts#L1072)
  - command: `nl -ba src/config/io.ts | sed -n '1072,1092p'`

- **Docs now promise this compatibility on load:** The voice-call docs state that legacy `tts.<provider>` keys inside plugin config are auto-migrated to `tts.providers.<provider>` on load.
