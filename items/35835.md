## Summary

Keep open. Current `main` still implements `read` multimodal handling as image-only, and the embedded runner/model capability path is also image-only, so the requested audio-file parity is not implemented.

## Evidence

- **`read` wrapper only normalizes image payloads:** `createOpenClawReadTool()` post-processes results through `normalizeReadImageResult()` and `sanitizeToolResultImages()`. The normalization logic looks only for `content` blocks with `type === "image"` and rewrites only `Read image file [...]` headers; there is no parallel audio path.
  - file: [src/agents/pi-tools.read.ts:292](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/pi-tools.read.ts#L292)
  - command: `nl -ba src/agents/pi-tools.read.ts | sed -n '292,349p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Sandbox read MIME detection is image-only:** The sandbox read operations expose only `detectImageMimeType`, and it returns a MIME only when `mime.startsWith("image/")`. Audio MIME types are explicitly not surfaced here, so audio files cannot enter the same attachment path as images.
  - file: [src/agents/pi-tools.read.ts:702](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/pi-tools.read.ts#L702)
  - command: `nl -ba src/agents/pi-tools.read.ts | sed -n '674,718p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Prompt attachment loading is image-only:** The embedded runner has `modelSupportsImages()` and `detectAndLoadPromptImages()`, both dedicated to images. If the model lacks `"image"` input support it returns no attachments, and there is no corresponding `audio` loader for prompt/read references.
  - file: [src/agents/pi-embedded-runner/run/images.ts:460](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/pi-embedded-runner/run/images.ts#L460)
  - command: `nl -ba src/agents/pi-embedded-runner/run/images.ts | sed -n '455,560p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Model input parsing/catalog checks still ignore audio in the agent path:** Agent-side modality parsing collapses model input to `['text']` or `['text','image']` only, and catalog helpers check only for `image`/`document` support. That means even models that may support audio elsewhere in the repo are not wired for native audio input in this `read`/embedded-runner path.
  - file: [src/agents/model-scan.ts:107](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/model-scan.ts#L107)
  - command: `nl -ba src/agents/model-scan.ts | sed -n '100,120p'; nl -ba src/agents/model-catalog.ts | sed -n '220,245p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Tests document only image fallback for `read` media extraction:** `extractToolResultMediaPaths()` has explicit tests for `read` image results falling back from `details.path`, with comments stating the Pi SDK read tool does not include `MEDIA:` but OpenClaw image results do. There is no analogous audio read test coverage.
  - file: [src/agents/pi-embedded-subscribe.tools.media.test.ts:98](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/pi-embedded-subscribe.tools.media.test.ts#L98)
