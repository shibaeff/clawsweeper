## Summary

Current main does not add new internal `gateway:shutdown` or `gateway:pre-restart` events, but it now exposes plugin lifecycle hooks (`gateway_start`/`gateway_stop`) plus session lifecycle hooks (`session_start`/`session_end`) that are a better fit for checkpoint/resume automation under the repo's plugin-first scope rules. The original silent-drop restart concern is also partially addressed by restart draining on main.

## Risks / Open Questions

- `gateway_stop` currently exposes a shutdown reason but not a prebuilt list of active session keys, so a plugin would need to maintain its own active-session registry via `session_start`/`session_end`.
- This closes the request as out-of-scope for core internal hooks, not because the exact proposed internal hook API was implemented.

## Close Comment

Closing from a Codex review. Current `main` points this at the plugin/ClawHub surface rather than a new core internal-hook event. Internal hooks still only document `gateway:startup` (`docs/automation/hooks.md:46`), but the public plugin SDK now exposes `session_start` / `session_end` plus `gateway_start` / `gateway_stop` lifecycle hooks (`docs/concepts/agent-loop.md:103`, `src/plugins/hook-types.ts:385`, `src/plugins/hook-types.ts:491`), and the gateway actually awaits `gateway_stop` before shutdown (`src/gateway/server.impl.ts:909`). The repo vision is also explicit that optional capability should usually ship as plugins and ClawHub, not expand core (`VISION.md:54`). Separately, the original restart-loss concern is partly addressed on `main`: restart now drains active work and rejects new work with an explicit draining error instead of silently dropping it (`src/cli/gateway-cli/run-loop.ts:166`, `src/process/command-queue.ts:228`). A checkpoint/resume solution fits best as a plugin: track active sessions via the session hooks, persist on `gateway_stop`, and resume on `gateway_start`.
