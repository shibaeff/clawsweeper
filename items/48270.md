## Summary

Keep open. Current `main` has a related mitigation for general gateway RPC calls, but `openclaw gateway probe` still goes through `probeGateway`, and that code path does not include the pre-connect event-loop drain this PR adds. The PR therefore still contains distinct, unlanded behavior for the reported probe timeout surface.

## Evidence

- **`gateway probe` still uses `probeGateway` directly:** The gateway-status probe pass calls `probeGateway(...)` for each target, so the user-facing `openclaw gateway probe` command still depends on the dedicated probe path rather than the shared `callGateway` path.
  - file: [src/commands/gateway-status/probe-run.ts:124](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/commands/gateway-status/probe-run.ts#L124)

  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)

That means `main` does not yet contain the probe-path mitigation proposed by the PR.
