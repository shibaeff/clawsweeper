## Summary

Keep open. Current `main` does not implement inbound audio transcription through the `openai-codex` OAuth path: `openai-codex` is image-only in code and docs, while audio transcription only runs through providers that expose `transcribeAudio` such as `openai`'s API-key route.

## Evidence

- **Issue request:** Issue #68539 asks for voice/audio transcription on the `openai-codex` path without `OPENAI_API_KEY`, without OpenAI API billing, and independently of normal Codex message limits.
  - command: `Provided GitHub context for issue #68539`
- **`openai-codex` provider is image-only:** The bundled OpenAI media-understanding provider registers `openai` with `capabilities: ["image", "audio"]` and `transcribeAudio`, but `openai-codex` is registered with `capabilities: ["image"]` only and no audio transcription handler.
  - file: [extensions/openai/media-understanding-provider.ts:21](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/openai/media-understanding-provider.ts#L21)

- **Plugin contract metadata matches image-only `openai-codex`:** The plugin manifest advertises audio media-understanding only for `openai`; `openai-codex` metadata lists only `capabilities: ["image"]` and no default audio model.
  - file: [extensions/openai/openclaw.plugin.json:51](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/openai/openclaw.plugin.json#L51)

## Risks / Open Questions

- The requested behavior depends on an upstream ChatGPT/Codex-authenticated transcription path and quota model that is not represented in current OpenClaw provider contracts or docs.
- Implementing this would require new auth/transport semantics for audio, not a conservative cleanup of existing `openai-codex` support.
