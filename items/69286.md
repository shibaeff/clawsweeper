## Summary

Keep open. Current `main` still does not repair session accounting when compaction completes without a positive `tokensAfter`, so the reported stale `totalTokens`/cost invariant break is not clearly fixed even though the safeguard loop itself has partial mitigation.

## Evidence

- **Compaction counter update only rewrites token cache for positive `tokensAfter`:** `incrementCompactionCount()` updates `totalTokens` only when `tokensAfter != null && tokensAfter > 0`; otherwise it leaves prior `totalTokens`, `totalTokensFresh`, and `estimatedCostUsd` untouched.
  - file: [src/auto-reply/reply/session-updates.ts:257](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/auto-reply/reply/session-updates.ts#L257)
  - command: `nl -ba src/auto-reply/reply/session-updates.ts | sed -n '210,320p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Manual compaction path forwards possibly-missing post-compaction token count:** `handleCompactCommand()` passes `result.result?.tokensAfter` into `incrementCompactionCount()`. If compaction returns no `tokensAfter`, the stale session counters are preserved by the guard above.
  - file: [src/auto-reply/reply/commands-compact.ts:170](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/auto-reply/reply/commands-compact.ts#L170)
