## Summary

Current `main` now treats ACP session identities as pending when `ensureSession` does not return a stable session id yet, persists the ACPX record id, retries stale resume ids without reusing them, and reconciles later-learned agent session ids. That directly covers the Gemini failure mode described here.

## Evidence

- **Pending ACP identities are part of the session contract:** `SessionAcpIdentity` now explicitly stores `acpxRecordId`, `acpxSessionId`, and `agentSessionId`, with `state: "pending" | "resolved"`, so ACP sessions no longer require an immediately usable stable id at spawn time.

## Close Comment

Closing from a Codex review. Current `main` no longer requires Gemini/ACPX to return a stable session id during the initial `ensureSession()` call: ACP session metadata now supports pending identities with `acpxRecordId`, the ensure path preserves that pending state instead of hard-failing, stale resume ids are retried without reuse, and there is explicit Gemini regression coverage for learning the real agent session id later in the run path (`src/config/sessions/types.ts:32`, `src/acp/runtime/session-identity.ts:181`, `src/acp/control-plane/manager.core.ts:1423`, `src/acp/control-plane/manager.test.ts:1541`, `src/acp/control-plane/manager.test.ts:2435`). The original reporter also confirmed on 2026-03-13 that upgrading past 2026.3.8 fixed their Gemini case, so this looks implemented on `main` rather than still actionable as an open bug.
