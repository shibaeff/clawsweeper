## Summary

Keep open. Current `main` still appears vulnerable to the reported cross-midnight log-file stickiness for long-lived loggers: `getLogger()` can rebuild the base logger when the rolling path changes, but `createSubsystemLogger()` lazily caches a child file logger once and keeps reusing it. Many production paths hold subsystem loggers at module scope, so those instances can survive date rollover and keep writing through the stale child logger. I did not find a regression test covering date rollover for an already-created subsystem/child logger.

## Evidence

- **Subsystem logger caches child logger permanently:** `createSubsystemLogger()` stores `fileLogger` in a closure, initializes it once with `getChildLogger({ subsystem: resolvedSubsystem })`, and reuses it on later writes instead of reacquiring after settings/path changes.
  - file: [src/logging/subsystem.ts:327](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/logging/subsystem.ts#L327)

  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Base logger only rotates when callers ask for a fresh logger:** `getLogger()` compares resolved settings and rebuilds `loggingState.cachedLogger` only when invoked again; existing child/sub loggers are not updated here.
  - file: [src/logging/logger.ts:237](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/logging/logger.ts#L237)

## Risks / Open Questions

- This conclusion is based on code inspection rather than a live cross-midnight reproduction, because the review had to remain read-only and avoid artifact-producing test runs.
- The exact stale-write mechanism depends on `tslog` sublogger behavior, but OpenClaw's current caching pattern leaves no visible path that refreshes already-created subsystem/child loggers when the rolling file path changes.
