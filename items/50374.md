## Summary

Keep open. Current `main` has several OpenAI-compatible replay/transport hardenings, but the evidence on disk only shows tool-call ID sanitization, turn-ordering/validation, and related compat shaping. I did not find strong evidence that `main` now strips unsupported extra message keys from outgoing `openai-completions` chat messages for generic strict providers like the report describes.

## Evidence

- **OpenAI-compatible replay policy on main is limited to ID sanitization and turn validation:** `buildOpenAICompatibleReplayPolicy(...)` for `openai-completions` enables `sanitizeToolCallIds`, assistant-first ordering repair, and Gemini/Anthropic turn validation. It does not implement a message-key stripping policy.
  - file: [src/plugins/provider-replay-helpers.ts:12](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/plugins/provider-replay-helpers.ts#L12)
- **Generic fallback policy for strict OpenAI-compatible providers matches the same scope:** The unowned-provider fallback only turns on `sanitizeToolCallIds`, ordering repair, and turn validators for `openai-completions`; there is no strict outgoing-message key filter here either.
  - file: [src/agents/transcript-policy.ts:74](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/transcript-policy.ts#L74)
- **Replay sanitization code does not strip arbitrary top-level message keys:** `sanitizeSessionMessagesImages(...)` rewrites images, tool-call IDs, signatures, and empty assistant text. It does not implement generic removal of unsupported message object fields before transport send.
  - file: [src/agents/pi-embedded-helpers/images.ts:40](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/pi-embedded-helpers/images.ts#L40)
- **Completions transport forwards converted messages without a repo-local strict-key scrubber:** `buildOpenAICompletionsParams(...)` assigns `messages: convertMessages(...)` and forwards them into the `/chat/completions` payload, with only compat-specific shaping like string-content flattening and thought-signature injection around it.
  - file: [src/agents/openai-transport-stream.ts:1705](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/openai-transport-stream.ts#L1705)
- **Current regression coverage is about tool-call IDs and turn repair, not extra key stripping:** OpenAI-compatible tests on `main` assert strict `sanitizeToolCallIds` behavior and assistant-first replay repair for `openai-completions`, which is narrower than the issue request.
  - file: [src/agents/transcript-policy.test.ts:263](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/transcript-policy.test.ts#L263)
- **Replay-history tests likewise cover ID sanitization and assistant-first bootstrap only:** The `sanitize-session-history` tests for `openai-completions` verify tool-call ID sanitization and strict assistant-first bootstrap behavior, not removal of unsupported outgoing message keys like the issue asks for.
  - file: [src/agents/pi-embedded-runner.sanitize-session-history.test.ts:386](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/pi-embedded-runner.sanitize-session-history.test.ts#L386)
- **No repo-local implementation surface for the requested `strictKeys`-style feature was found:** Searches for `strictKeys`, `strip extra message keys`, `sanitize outgoing message`, and similar terms did not find a relevant implementation or docs surface for this request.
