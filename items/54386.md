## Summary

Keep open. Current `main` still has the reported skill-creator behavior, and the PR discussion surfaces real unresolved contract issues around how OpenClaw derives the active workspace path.

  - file: [skills/skill-creator/SKILL.md:274](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/skills/skill-creator/SKILL.md#L274)

- **OpenClaw runtime workspace is profile-aware:** OpenClaw's runtime default workspace resolver does not hard-code `~/.openclaw/workspace`; it uses `OPENCLAW_PROFILE` to select `workspace-<profile>` for non-default profiles.
  - file: [src/agents/workspace-default.ts:6](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/agents/workspace-default.ts#L6)

  - file: [src/infra/home-dir.ts:32](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/infra/home-dir.ts#L32)

  - file: [docs/tools/skills.md:130](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/tools/skills.md#L130)

## Risks / Open Questions

- Closing now would drop a still-valid fix path even though current `main` still requires `--path` and still documents relative examples.
- Merging the PR as-is would risk creating skills in the wrong directory for profiled or `OPENCLAW_HOME`-customized setups unless its fallback is aligned with the runtime workspace contract.
