## Summary

Keep open. Current `main` still resolves `openclaw agent` sessions from `--to`/`--session-id`/`--agent` only, and treats `--channel` as reply-delivery metadata rather than a routing input. I did not find any current-main code or tests that implement lookup by Discord delivery context (`channel` + `to` + `accountId`/thread) before falling back to the main session.

## Evidence

- **CLI still resolves session key without channel context:** `agentViaGatewayCommand` calls `resolveSessionKeyForRequest({ cfg, agentId, to: opts.to, sessionId: opts.sessionId })` and only normalizes `opts.channel` afterward for gateway params, so `--channel` is not part of session selection.
  - file: [src/commands/agent-via-gateway.ts:113](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/commands/agent-via-gateway.ts#L113)
  - command: `nl -ba src/commands/agent-via-gateway.ts | sed -n '86,150p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Session resolver has no delivery-context lookup:** `resolveSessionKeyForRequest` accepts only `to`, `sessionId`, `sessionKey`, and `agentId`. It builds `ctx` as `{ From: opts.to }` and derives a key from that, with no `channel`, `accountId`, or thread/delivery-context matching path.
  - file: [src/agents/command/session.ts:130](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/agents/command/session.ts#L130)
