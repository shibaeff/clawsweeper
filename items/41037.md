## Summary

Keep open. Current `main` still rejects auth profiles that omit `auth.profiles[*].provider`, and the normalization proposed in PR #41037 is not present in the shipped code. The PR appears to address a real current gap rather than something already implemented or obsolete.

## Evidence

- **Auth profile schema still requires explicit provider:** On current main, each `auth.profiles` entry is still validated with a required `provider: z.string()`, so configs that rely on the `<provider>:<name>` key alone still fail schema validation.
  - file: [src/config/zod-schema.ts:454](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/config/zod-schema.ts#L454)
  - command: `nl -ba src/config/zod-schema.ts | sed -n '446,460p'`
  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **No provider backfill before schema validation:** `validateConfigObjectRaw()` strips deprecated keys, runs legacy checks, and then immediately calls `OpenClawSchema.safeParse(normalizedRaw)`. There is no pre-validation step on main that infers `auth.profiles[*].provider` from the profile id.
  - file: [src/config/validation.ts:620](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/config/validation.ts#L620)
  - command: `nl -ba src/config/validation.ts | sed -n '614,647p'`
  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **Regression test from the PR is not present on main:** The current `config.schema-regressions` file ends without the proposed auth-profile-provider regression cases, which is consistent with the fix not having landed.
  - file: [src/config/config.schema-regressions.test.ts:241](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/config/config.schema-regressions.test.ts#L241)
  - command: `nl -ba src/config/config.schema-regressions.test.ts | sed -n '160,260p'`
  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **User-facing docs still encode provider in profile ids:** Docs continue to present auth profile ids as provider-qualified keys such as `anthropic:default` and `<provider>:manual`, which supports the PR's premise that provider identity is already encoded in the profile id.
  - file: [docs/gateway/authentication.md:167](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/docs/gateway/authentication.md#L167)

  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **PR discussion indicates this is a real bug with an accepted fix shape:** The review discussion already includes a maintainer comment (`byungsker`, 2026-03-09) calling this a real usability issue and agreeing that normalization belongs before schema validation. Nothing in current main contradicts that assessment.
