## Summary

Current `main` still appears to have the midnight rolling-log bug from #47280, and this PR addresses the right area but has unresolved review concerns. Conservative cleanup should keep the PR open rather than close it.

## Evidence

- **Current transport still writes to the file path captured at logger construction time:** `buildLogger()` creates the file transport once, initializes state from `settings.file`, and every append/warning write still targets `settings.file` directly. There is no per-write re-resolution of the rolling path.
  - file: [src/logging/logger.ts:402](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/logging/logger.ts#L402)
  - command: `nl -ba src/logging/logger.ts | sed -n '402,448p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Date changes are detected only when rebuilding the cached base logger:** `resolveSettings()` uses `defaultRollingPathForToday()` and `settingsChanged()` compares the resolved `file`, so the base logger can rebuild after midnight, but only when callers go back through `getLogger()`.
  - file: [src/logging/logger.ts:338](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/logging/logger.ts#L338)
  - command: `nl -ba src/logging/logger.ts | sed -n '338,390p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Subsystem loggers cache a child file logger and keep reusing it:** `createSubsystemLogger()` memoizes `fileLogger` on first write with `getChildLogger(...)` and then reuses that instance for later writes. That supports the report that long-lived child/subsystem loggers can keep the old transport alive across midnight.
  - file: [src/logging/subsystem.ts:325](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/logging/subsystem.ts#L325)

- **User-facing docs promise daily rolling log files:** The logging docs say the Gateway writes a rolling file at `/tmp/openclaw/openclaw-YYYY-MM-DD.log`, so continuing to append to yesterday's file is a user-visible behavior bug, not just an internal refactor concern.
  - file: [docs/logging.md:22](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/docs/logging.md#L22)
