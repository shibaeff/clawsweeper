## Summary

Current `main` routes vendor-specific model integrations through provider plugins and ClawHub/plugin distribution rather than new core `src/agents` wiring, so this PR is out of scope in its current form even though the PPIO idea itself is reasonable.

## Evidence

- **Vision scope prefers plugins over new core optional capability:** `VISION.md` says optional capability should usually ship as plugins, providers use code plugins, the preferred path is npm/local extension loading, and plugin promotion belongs in ClawHub.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/VISION.md#L54)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Provider-specific behavior is plugin-owned on current main:** The model-provider docs say most provider-specific logic now lives in provider plugins while core keeps the generic inference loop.
  - file: [docs/concepts/model-providers.md:37](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/concepts/model-providers.md#L37)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **There is an explicit supported path for third-party provider plugins:** The provider-plugin guide describes building a provider plugin with model catalog, API-key auth, dynamic model resolution, and notes that published providers can ship via ClawHub-compatible plugin packaging.
  - file: [docs/plugins/sdk-provider-plugins.md:11](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/plugins/sdk-provider-plugins.md#L11)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Current auth/env discovery is manifest-driven, not new core provider tables:** `src/secrets/provider-env-vars.ts` now builds provider auth env-var candidates from plugin manifests/setup metadata and only keeps a small core fallback list, which is the opposite of adding a new hardcoded provider env var in core.
  - file: [src/secrets/provider-env-vars.ts:76](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/secrets/provider-env-vars.ts#L76)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Bundled provider additions already follow the plugin boundary:** Current bundled providers such as StepFun and Tencent register themselves as provider plugins with manifest metadata and `registerProvider(...)` instead of extending `src/agents` provider internals.
  - file: [extensions/stepfun/index.ts:1](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/stepfun/index.ts#L1)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)

## Risks / Open Questions

- A maintainer could still want an officially bundled PPIO provider plugin later; this close only rejects the current core-targeted shape, not PPIO support in general.
- No PPIO plugin/extension was linked in this PR discussion, so a follow-up would still need an owner to publish and maintain the provider plugin.
