## Summary

Keep open. Current `main` still appears to have the reported gap: doctor stale-plugin repair only removes `plugins.allow`/`plugins.entries`, channel-doctor stale cleanup skips configured channel ids that no longer resolve to a plugin doctor, and config validation still fails on the leftover `channels.<id>` entry.

## Evidence

- **Stale plugin repair only touches plugin surfaces:** `StalePluginSurface` is limited to `"allow" | "entries"`; `scanStalePluginConfig` reads only `cfg.plugins`; and `maybeRepairStalePluginConfig` only removes stale ids from `plugins.allow` and `plugins.entries`, with no `channels.*` cleanup path.
  - file: [src/commands/doctor/shared/stale-plugin-config.ts:8](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/commands/doctor/shared/stale-plugin-config.ts#L8)
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Channel stale cleanup skips unknown configured ids:** `listChannelDoctorEntries` builds entries only when a doctor adapter can be resolved; if none is found it `continue`s. `collectChannelDoctorStaleConfigMutations` iterates only those entries, so a dangling configured channel id with no remaining plugin/doctor produces no cleanup mutation.
  - file: [src/commands/doctor/shared/channel-doctor.ts:216](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/commands/doctor/shared/channel-doctor.ts#L216)
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Validation still rejects dangling channel ids:** Config validation collects allowed channel ids from bundled ids plus discovered plugin manifests, and otherwise emits `channels.<id>: unknown channel id: <id>`. A leftover `channels.openclaw-weixin` entry would still fail validation after the plugin directory is removed.
  - file: [src/config/validation.ts:922](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/config/validation.ts#L922)
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Issue affects an external plugin path that current install recovery does not cover:** The WeChat docs still describe `openclaw-weixin` as an external plugin installed via `openclaw plugins install`. Current invalid-config install recovery is gated by bundled-plugin recovery metadata, so the special recovery path is not a general fix for external plugin reinstalls.
  - file: [src/cli/plugin-install-config-policy.ts:161](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/cli/plugin-install-config-policy.ts#L161)
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **External plugin docs confirm the reported plugin/channel id mapping:** The docs explicitly map `@tencent-weixin/openclaw-weixin` to channel id `openclaw-weixin` and describe Gateway startup discovering that external plugin, matching the issue's reported stale channel reference shape.
  - file: [docs/channels/wechat.md:10](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/docs/channels/wechat.md#L10)
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
