## Summary

Keep open. Current `main` learns and persists Telegram forum `TopicName`, but the session metadata and Control UI label paths still derive dashboard labels from `displayName`/`label`, and Telegram does not map `TopicName` into `ThreadLabel` or `displayName`. The requested forum-topic name labeling is not implemented on `main`.

## Evidence

- **Telegram forum topic names are captured, but only as `TopicName` in inbound context:** `bot-message-context.ts` resolves and caches forum topic names, and the session payload writer exports that value as `TopicName`. This proves the data exists in the Telegram extension, but it is not the session label field the UI consumes.
  - file: [extensions/telegram/src/bot-message-context.ts:164](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/telegram/src/bot-message-context.ts#L164)
  - command: `nl -ba extensions/telegram/src/bot-message-context.ts | sed -n '140,220p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Session metadata still builds group `displayName` from subject/channel/space slugging:** `deriveSessionMetaPatch()` calls `buildGroupDisplayName()` for group sessions, and that helper only considers `subject`, `groupChannel`, `space`, `id`, and `key`, producing values like `telegram:g-...`. There is no `TopicName` input here.
  - file: [src/config/sessions/metadata.ts:166](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/config/sessions/metadata.ts#L166)
  - command: `nl -ba src/config/sessions/metadata.ts | sed -n '1,260p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Session init only promotes `ThreadLabel` into `displayName`:** Current session initialization overwrites `sessionEntry.displayName` only when `ctx.ThreadLabel` is present. Static search found no Telegram path setting `ThreadLabel`, only `TopicName`, so Telegram forum topics do not reach this label path.
  - file: [src/auto-reply/reply/session.ts:669](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/auto-reply/reply/session.ts#L669)
  - command: `rg -n "ThreadLabel|TopicName" src extensions/telegram -g '!**/dist/**' && nl -ba src/auto-reply/reply/session.ts | sed -n '620,700p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Control UI session picker resolves names from `label` and `displayName` only:** `resolveSessionDisplayName()` prefers `row.label`, then `row.displayName`, then a generic parsed fallback. It does not read `subject`, `origin.label`, or Telegram `TopicName` for picker labels.
  - file: [ui/src/ui/chat/session-controls.ts:416](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/ui/src/ui/chat/session-controls.ts#L416)

- **Sessions table also surfaces only `displayName`, not topic metadata:** The sessions view shows the stored `displayName` beside the key when it differs from the key/label. That matches the reported behavior where forum-topic sessions remain hard to distinguish if `displayName` stays at the group slug.
  - file: [ui/src/ui/views/sessions.ts:457](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/ui/src/ui/views/sessions.ts#L457)
