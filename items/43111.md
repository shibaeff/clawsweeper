## Summary

The request is for WeCom-specific trigger behavior, but current `main` only treats WeCom as an external channel plugin. OpenClaw core policy and vision explicitly keep channel-specific behavior in plugins, so this should be tracked in the WeCom plugin rather than `openclaw/openclaw`.

## Evidence

- **Core/plugin scope rule:** Repo policy says core must stay extension-agnostic and that extension-owned behavior stays in the extension.
  - file: [AGENTS.md:41](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/AGENTS.md#L41)
- **Vision scope rule:** Project vision says core stays lean and optional capability should usually ship as plugins maintained in their own repositories.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/VISION.md#L54)
- **WeCom is cataloged as an external plugin:** Official external channel catalog entry for `wecom` points to `@wecom/wecom-openclaw-plugin` and only provides install metadata, not bundled runtime code.
  - file: [scripts/lib/official-external-channel-catalog.json:4](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/scripts/lib/official-external-channel-catalog.json#L4)
- **Docs send users to the external WeCom plugin:** Community plugin docs list WeCom as `@wecom/wecom-openclaw-plugin` with its own upstream repository.
  - file: [docs/plugins/community.md:115](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/docs/plugins/community.md#L115)
- **Latest release only improved WeCom onboarding/catalog display:** Release notes mention showing the official WeCom plugin in the external channel catalog; they do not add bundled WeCom runtime behavior to core.
  - file: [appcast.xml:57](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/appcast.xml#L57)
- **No bundled WeCom runtime in this checkout:** There are no `wecom` or `wechat` runtime files under `src/channels` or `extensions`; only docs/tests/catalog metadata are present.
  - command: `find src/channels extensions -maxdepth 2 \( -iname '*wecom*' -o -iname '*wechat*' \) | sort`

## Risks / Open Questions

- If maintainers want to keep core issues as a demand-tracking mirror for external plugin requests, closing here could hide user demand even though implementation belongs upstream.

## Close Comment

Thanks for the report. I reviewed current `main` in a Codex pass, and this request is in plugin scope rather than OpenClaw core scope. On current `main`, WeCom is only surfaced as an external channel plugin (`scripts/lib/official-external-channel-catalog.json:4`, `docs/plugins/community.md:115`), and the repo rules explicitly keep channel-specific behavior in plugins instead of adding core special cases (`AGENTS.md:41`, `VISION.md:54`). The latest shipped core change here was only onboarding/catalog visibility for the official WeCom plugin (`appcast.xml:57`), not bundled WeCom runtime behavior. Because `requireMention: false` / keyword-trigger behavior is WeCom-plugin-specific, the right home is `@wecom/wecom-openclaw-plugin` / its upstream repo, not `openclaw/openclaw`, so I’m closing this here as plugin-scope.
