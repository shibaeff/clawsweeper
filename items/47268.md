## Summary

Current `main` already implements the reported MSTeams FileConsentCard behavior: the consent-card activity id is captured when the card is sent, and acceptance now replaces that original card in place with `updateActivity`, with regression coverage. The shipped `v2026.4.22` release also contains that fix.

## Close Comment

Codex review: current `main` already implements this. The MSTeams file-consent flow now stores the original consent-card activity id when the FileConsentCard is sent (`extensions/msteams/src/send.ts:178`), and on accept it replaces that original card in place with `context.updateActivity(...)`, only falling back to a new message if the update fails (`extensions/msteams/src/file-consent-invoke.ts:93`). The regression is covered by tests that assert both the in-place update and that no second FileInfo card is sent on the success path (`extensions/msteams/src/monitor-handler.file-consent.test.ts:171`). This behavior is also present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so I’m closing this as already implemented on main. If you can still reproduce it on `v2026.4.22` or newer, please open a fresh issue with current reproduction steps and Teams/plugin version details.
