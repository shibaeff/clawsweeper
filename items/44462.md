## Summary

Current `main` already supports `hooks.mappings[].agentId` end-to-end for webhook routing, with docs, tests, and a changelog entry showing the feature shipped in release `2026.2.9`.

## Evidence

  - file: [src/gateway/hooks-mapping.ts:215](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/gateway/hooks-mapping.ts#L215)

  - file: [src/gateway/server-http.ts:762](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/gateway/server-http.ts#L762)

## Risks / Open Questions

- The issue body's optional idea of templated `agentId` values (for example `"{{workspace}}"`) is not covered by the current mapping implementation; the implemented and documented behavior is direct string `agentId` routing.

## Close Comment

Closing as implemented. Codex review against current `main` found end-to-end `agentId` support for webhook hook mappings: `src/gateway/hooks-mapping.ts:215` normalizes `mappings[].agentId`, `src/gateway/server-http.ts:762` and `src/gateway/server-http.ts:776` validate and route mapped hooks to a target agent, and `src/gateway/server/hooks.ts:58` writes that agent onto the cron job passed to `runCronIsolatedAgentTurn`. This is covered by tests in `src/gateway/hooks-mapping.test.ts:534` and `src/gateway/server.hooks.test.ts:177` / `src/gateway/server.hooks.test.ts:671`, documented in `docs/gateway/configuration-reference.md:440` and `docs/gateway/configuration-reference.md:480`, and the shipped changelog entry is `CHANGELOG.md:5242` under release `2026.2.9`. If you still see default-agent routing on a current build, please open a fresh bug with the exact hook mapping, payload, and OpenClaw version.
