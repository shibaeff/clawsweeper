## Summary

Current `main` no longer schedules the managed memory-core dreaming sweep as a main-session `systemEvent` job. It now reconciles dreaming to an isolated `agentTurn` cron job, which uses the longer agent-turn timeout path instead of the hard-coded 10-minute `systemEvent` timeout. Tests on `main` explicitly cover both the reconciled job shape and successful execution of the managed isolated dreaming cron, so the specific bug reported here is implemented on `main` even though the broader `systemEvent` timeout limitation still exists separately.

## Evidence

- **Dreaming cron now uses isolated agentTurn:** `buildManagedDreamingCronJob` on current `main` creates the managed dreaming job with `sessionTarget: "isolated"`, `payload.kind: "agentTurn"`, and `lightContext: true`, so it no longer takes the main-session `systemEvent` path described in the report.
  - file: [extensions/memory-core/src/dreaming.ts:153](https://github.com/openclaw/openclaw/blob/8e18b3cc20d6c8a5c761315b94303885dd7d2773/extensions/memory-core/src/dreaming.ts#L153)
  - sha: [8e18b3cc20d6](https://github.com/openclaw/openclaw/commit/8e18b3cc20d6c8a5c761315b94303885dd7d2773)
- **AgentTurn jobs get longer timeout path:** Cron timeout policy still uses a 10-minute default for non-agent jobs, but `agentTurn` jobs get a 60-minute safety timeout and honor explicit `timeoutSeconds`. That means the new dreaming job shape avoids the 600s wall reported here.
  - file: [src/cron/service/timeout-policy.ts:8](https://github.com/openclaw/openclaw/blob/8e18b3cc20d6c8a5c761315b94303885dd7d2773/src/cron/service/timeout-policy.ts#L8)
  - sha: [8e18b3cc20d6](https://github.com/openclaw/openclaw/commit/8e18b3cc20d6c8a5c761315b94303885dd7d2773)
- **Reconciliation test locks in new job shape:** The dreaming reconciliation test asserts that the managed job created when enabled is `sessionTarget: "isolated"` with `payload.kind: "agentTurn"` and the dreaming token as the message.
  - file: [extensions/memory-core/src/dreaming.test.ts:415](https://github.com/openclaw/openclaw/blob/8e18b3cc20d6c8a5c761315b94303885dd7d2773/extensions/memory-core/src/dreaming.test.ts#L415)
  - sha: [8e18b3cc20d6](https://github.com/openclaw/openclaw/commit/8e18b3cc20d6c8a5c761315b94303885dd7d2773)
- **Execution test covers isolated dreaming cron:** A dedicated test verifies promotions run when the managed dreaming isolated cron job fires, covering the behavior the issue reported as timing out.
  - file: [extensions/memory-core/src/dreaming.test.ts:1820](https://github.com/openclaw/openclaw/blob/8e18b3cc20d6c8a5c761315b94303885dd7d2773/extensions/memory-core/src/dreaming.test.ts#L1820)
  - sha: [8e18b3cc20d6](https://github.com/openclaw/openclaw/commit/8e18b3cc20d6c8a5c761315b94303885dd7d2773)
- **Issue discussion correctly identified broader systemEvent limitation:** Comments from `frank-bot07`, `desksk`, and others correctly noted that the 600s cap still applies to main-session `systemEvent` jobs in general. Current `main` resolves the memory-core dreaming case by moving it off that path rather than by adding a global `systemEvent` timeout override.
