## Summary

Current main already implements the Docker localhost Control UI pairing path described here. The gateway now classifies Docker-published loopback Control UI traffic as a local browser-container case, allows silent first-pair approval on that path, auto-approves the silent request during the websocket handshake, and ships regression coverage for `127.0.0.1`/`localhost` with Docker bridge remotes like `172.17.0.1`.

## Close Comment

Closing as implemented on current main after a Codex review. The Docker localhost Control UI path described here is now explicitly handled in the gateway pairing logic: Docker-published loopback Control UI traffic is classified as `browser_container_local` in `src/gateway/server/ws-connection/handshake-auth-helpers.ts:167`, first-time `not-paired` requests on that path are eligible for silent local pairing in `src/gateway/server/ws-connection/handshake-auth-helpers.ts:75`, and the websocket handshake auto-approves those silent pairing requests in `src/gateway/server/ws-connection/message-handler.ts:919`. There is also regression coverage for `http://127.0.0.1:18789` / `http://localhost:18789` with Docker bridge remotes like `172.17.0.1` in `src/gateway/server/ws-connection/handshake-auth-helpers.test.ts:110`, and the Docker docs now document the current pairing recovery flow in `docs/install/docker.md:375`. This implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still hit a pairing loop on a current build, please open a fresh issue with the exact topology (localhost vs Tailscale/proxy), the dashboard URL you opened, and current `openclaw devices list` output so it can be triaged as a separate case.
