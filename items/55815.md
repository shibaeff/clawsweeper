## Summary

Keep open. Current `main` still classifies `tool_execution_end` as a hard failure whenever `evt.isError` is true, with no warning-only exemption, while Telegram reaction soft-fail results are still returned as `jsonResult({ ok: false, warning: ... })` payloads under `result.details`. The PR discussion also shows the branch needed follow-up fixes for the real `details.warning` shape, which are not present on `main`.

## Evidence

- **Current main still turns `evt.isError` into a hard tool failure:** `handleToolExecutionEnd()` on `main` computes `const isToolError = isError || isToolResultError(result);` and immediately records `lastToolError` when that is true. There is no warning-only bypass in this code path.
  - file: [src/agents/pi-embedded-subscribe.handlers.tools.ts:816](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-subscribe.handlers.tools.ts#L816)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Telegram reaction soft-fail still returns a warning payload:** The Telegram action runtime returns `jsonResult({ ok: false, warning: reactionResult.warning, ... })` when `reactionResult.ok` is false, so unsupported reactions still flow through as warning-only payloads rather than thrown exceptions.
  - file: [extensions/telegram/src/action-runtime.ts:295](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/action-runtime.ts#L295)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **`jsonResult(...)` wraps the payload in `details`:** `jsonResult(payload)` calls `textResult(...)`, and `textResult` stores the original payload in the result `details` field.
