## Summary

Current main already prevents `sessions_send` inter-session turns from overwriting an established external delivery route. The original hardcoded `webchat` injection remains, but reply routing now preserves the persisted external channel/to for `inter_session` provenance, and the changelog shows this shipped in v2026.4.9.

## Risks / Open Questions

- The implementation fixes the reported routing flip in downstream session-routing logic rather than removing the `webchat` hardcode in `sessions_send`, so adjacent source-channel bugs can still exist as separate issues.

## Close Comment

Closing from a Codex review: current `main` already fixes this behavior. `sessions_send` still injects `channel: INTERNAL_MESSAGE_CHANNEL` in `src/agents/tools/sessions-send-tool.ts:275-287`, but the session reply pipeline now treats `inter_session` provenance specially. `src/auto-reply/reply/session.ts:553-579` passes `isInterSession` into `resolveLastChannelRaw` / `resolveLastToRaw`, and `src/auto-reply/reply/session-delivery.ts:113-120` plus `:155-160` preserve the persisted external route instead of letting `webchat` overwrite Discord/Telegram/etc. The regression is covered by `src/auto-reply/reply/session-delivery.test.ts:4-39`, and the changelog shows it shipped in `v2026.4.9` (`CHANGELOG.md:1019`). If someone can still reproduce this on `v2026.4.9` or newer, that would be a new routing regression and should be opened with fresh repro details.
