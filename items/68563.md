## Summary

Current `main` already filters final reply media against media that was sent earlier during tool execution, and that duplicate-media suppression was shipped in `v2026.4.23`, so the reported double-send path is already implemented in core.

## Evidence

- **Final payload builder receives tool-sent media history:** `runResult.messagingToolSentMediaUrls` is passed into `buildReplyPayloads`, so final replies are filtered using media already emitted during tool execution before channel delivery.
