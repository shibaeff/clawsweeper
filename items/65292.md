## Summary

Current `main` no longer depends on provider-prefix claiming for Codex harness activation. The bundled Codex harness can now be selected explicitly via `embeddedHarness.runtime: "codex"`, custom/remote app-server WebSocket transport is documented, and Codex thread start/resume forwards non-`codex` provider ids through as `modelProvider` instead of hardcoding the legacy provider path.

## Evidence

- **Forced harness selection no longer depends on provider claim matching:** `src/agents/harness/selection.ts` selects a non-`auto` runtime by harness id alone. When `embeddedHarness.runtime` is `"codex"`, OpenClaw returns the registered Codex harness without consulting `supports(...)`, so activation no longer requires the model ref to use the legacy `codex/*` prefix.
  - file: [src/agents/harness/selection.ts:105](https://github.com/openclaw/openclaw/blob/453789914bb725d1cd5a3edd7ec5d8b0b45041cd/src/agents/harness/selection.ts#L105)
  - command: `nl -ba src/agents/harness/selection.ts | sed -n '70,125p'`
  - sha: [453789914bb7](https://github.com/openclaw/openclaw/commit/453789914bb725d1cd5a3edd7ec5d8b0b45041cd)
- **Codex harness still only auto-claims legacy `codex` refs in auto mode:** `extensions/codex/harness.ts` keeps `DEFAULT_CODEX_HARNESS_PROVIDER_IDS = new Set(["codex"])`, which means the old provider-claim path is compatibility-only. This matches the new architecture: explicit runtime selection replaces provider-prefix claiming for native Codex runs.

That is the core behavior needed for backends like `codex-lb` once the Codex runtime is explicitly selected.

## Risks / Open Questions

- This review closes the provider-prefix activation request based on current main’s explicit runtime-selection design. It does not prove compatibility with every third-party Codex-compatible backend implementation.
- The latest tagged release in the prompt (`v2026.4.23`) predates the main-branch changes cited here, so the documented fix path appears to be on `main` but not yet in a released tag.
