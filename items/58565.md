## Summary

Keep open. Current `main` still intentionally treats generic HTTP 401 invalid/expired-key failures as transient `auth` cooldowns rather than permanent disablement, so the issue's requested 401-vs-429/529 differentiation is not implemented end-to-end.

## Evidence

- **Docs still group auth failures with cooldowns:** The failover doc says auth and rate-limit failures both put a profile into cooldown and rotate to the next profile.
  - file: [docs/concepts/model-failover.md:120](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/docs/concepts/model-failover.md#L120)
  - command: `nl -ba docs/concepts/model-failover.md | sed -n '118,132p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **401/403 only become permanent when message matches a narrow permanent-auth classifier:** HTTP 401/403 map to `auth_permanent` only when `isAuthPermanentErrorMessage(message)` matches; otherwise they stay `auth`. 429 maps to `rate_limit`, 529 maps to `overloaded`.
  - file: [src/agents/pi-embedded-helpers/errors.ts:621](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/agents/pi-embedded-helpers/errors.ts#L621)

## Risks / Open Questions

- The exact long-latency reproduction in the issue may be reduced on current `main` because assistant failover now advances to the next profile immediately after marking failure, but dead/invalid keys are still not persistently disabled unless the message matches the narrow `auth_permanent` patterns.
- Closing as implemented would be incorrect because the repo currently codifies the opposite behavior for generic 401 invalid-key cases in both production code and tests.
