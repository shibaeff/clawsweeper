## Summary

Keep open. Current `main` still generates Feishu synthetic reaction message IDs with the `:reaction:<emoji>:<uuid>` suffix and then uses that unmodified ID as the current/reply/read target in downstream Feishu API paths, so the reported 400 remains plausible on `9153598e659daf5a52c479731f5b4ab8a5bf16b7`.

## Evidence

- **Synthetic reaction IDs still include the suffix:** `resolveReactionSyntheticEvent()` returns `message_id: `${messageId}:reaction:${emoji}:${uuid()}`` for reaction events.
  - file: [extensions/feishu/src/monitor.account.ts:164](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/feishu/src/monitor.account.ts#L164)

  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Inbound parsing preserves the suffixed ID as the message context ID:** `parseFeishuMessageEvent()` copies `event.message.message_id` straight into `ctx.messageId` with no normalization.
  - file: [extensions/feishu/src/bot.ts:120](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/feishu/src/bot.ts#L120)
