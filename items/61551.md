## Summary

Current `main` still has the overwrite path described in #61551. Gateway startup activates a scoped, gateway-bindable registry and initializes the global hook runner, but `ensureRuntimePluginsLoaded()` later resolves plugins with a different load context and can still fall through to `loadOpenClawPlugins()`. Because `activatePluginRegistry()` still calls `initializeGlobalHookRunner(registry)` unconditionally, a runtime reload can replace the global runner during message processing. The 2026-04-21 issue comment reporting multiple post-startup `createHookRunner` calls remains consistent with the current code; the separate Telegram bypass discussion does not invalidate this bug.

## Evidence

- **Global hook runner is still overwritten on every activating load:** `initializeGlobalHookRunner()` always assigns both `state.registry` and `state.hookRunner = createHookRunner(registry, ...)` with no guard or merge behavior, so any later activating registry load replaces the existing global runner.
  - file: [src/plugins/hook-runner-global.ts:32](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/plugins/hook-runner-global.ts#L32)
  - command: `nl -ba src/plugins/hook-runner-global.ts | sed -n '1,220p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Runtime helper still loads via compatibility check, not a pinned startup registry:** `ensureRuntimePluginsLoaded()` now calls `resolveRuntimePluginRegistry()` with only `config`, `workspaceDir`, and optional `allowGatewaySubagentBinding`; it does not pass the gateway startup scope (`onlyPluginIds`) or startup activation metadata.
  - file: [src/agents/runtime-plugins.ts:5](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/agents/runtime-plugins.ts#L5)
  - command: `nl -ba src/agents/runtime-plugins.ts | sed -n '1,220p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Gateway startup uses a different plugin-load context than runtime ensure:** `loadGatewayPlugins()` computes `pluginIds` and then calls `loadOpenClawPlugins()` with `activationSourceConfig`, `autoEnabledReasons`, `workspaceDir`, `onlyPluginIds: pluginIds`, and `runtimeOptions.allowGatewaySubagentBinding: true`. That startup registry key is therefore more specific than the later runtime helper call.
  - file: [src/gateway/server-plugins.ts:477](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/gateway/server-plugins.ts#L477)

- **Compatibility fast path still depends on matching cache-key inputs that differ between startup and runtime:** The loader cache key includes `activationMetadataKey`, `onlyPluginIds`, and `runtimeSubagentMode`. `getCompatibleActivePluginRegistry()` only reuses the active registry when that computed cache key matches, with one narrow fallback for default-vs-gateway-bindable mode. That does not cover the gateway-startup vs runtime-helper mismatch in scoped plugin ids and activation metadata.
  - file: [src/plugins/loader.ts:780](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/plugins/loader.ts#L780)

- **Activating a new registry still reinitializes the global runner:** Whenever an activating load occurs, `activatePluginRegistry()` calls `setActivePluginRegistry(...)` and then `initializeGlobalHookRunner(registry)`. So if `resolveRuntimePluginRegistry()` misses the compatibility fast path and loads, the overwrite still happens.

- **Issue discussion matches current code shape:** The 2026-04-21 comment reports three `createHookRunner` calls over live traffic and a watchdog restoring `message_sending` registrations. Current `main` still has unconditional runner replacement plus the startup/runtime load-context mismatch above, so that report still aligns with the code. The commenter also notes #26422 is a separate Telegram-path bypass, not a reason to close #61551.

## Risks / Open Questions

- A runtime plugin load during agent/message processing can still replace the gateway's global hook runner and drop previously reachable hooks.
- The recent `ensureRuntimePluginsLoaded()` refactor reduces duplication but does not prove the startup registry is reused for gateway traffic, because the cache-key inputs still differ.
- Closing this now would likely collapse two separate problems into the Telegram bypass issue and lose the still-open global-runner overwrite defect.
