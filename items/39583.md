## Summary

Keep open. Current `main` does not implement the proposed stringified-content parsing, and the PR discussion shows the latest patch still has an unresolved regression in its detection heuristic.

## Evidence

- **Current main still returns string content unchanged:** `extractTextFromChatContent` only sanitizes and normalizes when `content` is a string; it does not attempt to parse JSON-stringified content-block arrays, so the behavior described in the PR is not implemented on `main`.
  - file: [src/shared/chat-content.ts:44](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/shared/chat-content.ts#L44)
  - command: `nl -ba src/shared/chat-content.ts | sed -n '1,220p'`
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **Array-only extraction means parsed non-text arrays would collapse to null:** The array path only keeps blocks where `type === "text"`; arrays with no text blocks return `null`. That matches the later PR review concern that a broad `"type" in item` guard could wrongly convert arbitrary typed JSON arrays into dropped content.
  - file: [src/shared/chat-content.ts:54](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/shared/chat-content.ts#L54)

- **User-facing callers still depend on this helper:** `safeBody()` uses `extractTextFromChatContent` to coerce conversation bodies for prompts. If a vLLM/OpenAI-compatible response body arrives as a JSON-stringified block array, current `main` would still pass the raw JSON string through here.
  - file: [src/gateway/agent-prompt.ts:14](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/gateway/agent-prompt.ts#L14)
