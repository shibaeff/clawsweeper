## Summary

Current `main` already implements queued-message buffering/coalescing for busy sessions. OpenClaw defaults inbound traffic to `collect`, waits through a configurable debounce window, then merges same-thread queued messages into a single follow-up turn while preserving separate routing for different destinations.

## Risks / Open Questions

- Current behavior is deterministic queue batching/coalescing, not semantic relevance scoring inside one thread. If maintainers want model-based split/merge decisions, that would be a narrower follow-up request rather than this general queued-message-merging request.

## Close Comment

Closing as already implemented. Codex review of current `main` found OpenClaw already has queued-message buffering/coalescing for this case: `collect` is the default queue mode, `messages.queue.debounceMs` waits for a quiet window, and the follow-up drain merges queued same-thread messages into a single follow-up turn while preserving separate routing when the destination differs (`docs/concepts/queue.md:25`, `src/auto-reply/reply/queue/settings.ts:23`, `src/auto-reply/reply/queue/drain.ts:159`, `src/auto-reply/reply/queue.collect.test.ts:57`). The queue docs are dated 2026-01-16 and the 2026.1.9 release notes already mention queued-message support (`CHANGELOG.md:6706`). If there is still a product gap, it would be a narrower request for semantic split/merge decisions within one thread, not the general busy-session message merging requested here.
