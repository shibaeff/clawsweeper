## Summary

Keep open. Current `main` still gates `/stop` only with `rejectUnauthorizedCommand`, so the reported internal-scope gap looks real, but this PR is not safe to close out as-is because it swaps out the existing sender-authorization gate instead of layering the internal `operator.admin` scope check on top.

## Evidence

- **Current `/stop` handler still uses only sender authorization:** `handleStopCommand` on current `main` calls `rejectUnauthorizedCommand(params, "/stop")` and does not call `requireGatewayClientScopeForInternalChannel`, so there is no extra `operator.admin` gate for internal gateway clients.
  - file: [src/auto-reply/reply/commands-session-abort.ts:136](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/auto-reply/reply/commands-session-abort.ts#L136)
  - command: `nl -ba src/auto-reply/reply/commands-session-abort.ts | sed -n '129,150p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Internal scope gate is additive, not a replacement:** `requireGatewayClientScopeForInternalChannel` returns `null` for non-internal channels, so using it alone would skip the existing unauthorized-sender block on normal chat surfaces. It is designed to add an internal-channel scope check, not replace `rejectUnauthorizedCommand`.
  - file: [src/auto-reply/reply/command-gates.ts:47](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/auto-reply/reply/command-gates.ts#L47)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Gateway internal chat messages arrive pre-authorized:** Gateway `chat.send` builds internal message contexts with `Provider`/`Surface` set to the internal channel and `CommandAuthorized: true`, carrying scopes separately in `GatewayClientScopes`. That makes an explicit internal scope gate relevant for `/stop`.
  - file: [src/gateway/server-methods/chat.ts:2318](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server-methods/chat.ts#L2318)

That supports the PR's underlying bug report, but not this exact patch.

## Risks / Open Questions

- Current `main` still appears to permit internal gateway callers without `operator.admin` to issue `/stop`, because internal messages are marked `CommandAuthorized: true` and `/stop` has no extra internal scope check.
- Merging this PR as written would likely regress external-channel safety by removing the existing unauthorized-sender gate for `/stop` on non-internal chat surfaces.
- I did not run tests in this read-only review, so the conclusion is based on source, docs, and the PR discussion rather than executable proof.
