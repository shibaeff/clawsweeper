## Summary

Current `main` already reclaims stale session lock files instead of leaving the session permanently blocked after an abnormal shutdown, and gateway startup also performs stale lock cleanup.

## Evidence

- **Lock acquisition now reclaims stale locks:** `acquireSessionWriteLock` records PID/start time in the lock file, marks dead/recycled/too-old locks as stale, removes stale `.lock` files on contention, and retries acquisition instead of failing permanently.
  - file: [src/agents/session-write-lock.ts:325](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/agents/session-write-lock.ts#L325)
  - command: `nl -ba src/agents/session-write-lock.ts | sed -n '300,590p'`
- **Session store writes use the same stale-lock-aware path:** Session store locking goes through `acquireSessionWriteLock({ sessionFile: storePath, ... })`, so stale `agents/<agentId>/sessions/*.lock` files for `sessions.json` are also reclaimed on the next write attempt.
  - file: [src/config/sessions/store.ts:576](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/config/sessions/store.ts#L576)
  - command: `nl -ba src/config/sessions/store.ts | sed -n '548,590p'`
- **Gateway startup cleans stale session locks:** Gateway sidecar startup enumerates agent session directories and calls `cleanStaleLockFiles(... removeStale: true)` before normal runtime resumes.
  - file: [src/gateway/server-startup-post-attach.ts:149](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/gateway/server-startup-post-attach.ts#L149)
  - command: `nl -ba src/gateway/server-startup-post-attach.ts | sed -n '145,175p'`

  - file: [src/agents/session-write-lock.test.ts:229](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/agents/session-write-lock.test.ts#L229)
  - command: `nl -ba src/agents/session-write-lock.test.ts | sed -n '225,390p'`
