## Summary

Current `main` has exec-policy/approvals merging and visibility, but it does not implement a distinct immutable operator-policy layer that is loaded separately and protected from runtime weakening. The reported feature remains materially unimplemented.

## Evidence

- **Current docs describe only editable config + approvals merge:** The approvals docs say `openclaw exec-policy` synchronizes local `tools.exec.*` config with local `~/.openclaw/exec-approvals.json`, and that the effective result is derived from requested policy plus the host approvals file. There is no third immutable policy source described.
  - file: [docs/cli/approvals.md:23](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/cli/approvals.md#L23)
  - command: `nl -ba docs/cli/approvals.md | sed -n '1,120p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Applying exec policy still writes mutable local files:** `applyLocalExecPolicy` computes the next policy, then calls `saveExecApprovals(nextApprovals)` and `replaceConfigFile(...)`. That is an editable sync path, not an immutable startup-only overlay.
  - file: [src/cli/exec-policy-cli.ts:337](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/cli/exec-policy-cli.ts#L337)
  - command: `nl -ba src/cli/exec-policy-cli.ts | sed -n '320,370p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Approvals CLI also writes approvals state directly:** The local approvals write path is a direct `saveExecApprovals(file)` call. There is no lock/overlay enforcement layer in this path.
  - file: [src/cli/exec-approvals-cli.ts:95](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/cli/exec-approvals-cli.ts#L95)
  - command: `nl -ba src/cli/exec-approvals-cli.ts | sed -n '80,110p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Config includes are normal deep merges, not locked precedence:** `$include` content is merged with sibling keys via `deepMerge(included, rest)`, and sibling keys override included values. That supports modular config, but not an immutable policy source that later writes cannot weaken.
  - file: [src/config/includes.ts:131](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/config/includes.ts#L131)
  - command: `nl -ba src/config/includes.ts | sed -n '120,170p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Runtime config writes remain allowed for operator admins:** `config.patch` accepts arbitrary object patches, validates them, and writes via `writeConfigFile(validated.config, writeOptions)`. The scope mapping shows `config.patch` requires `operator.admin`, but there is no extra immutable-policy check that rejects weakening locked settings.
  - file: [src/gateway/server-methods/config.ts:471](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/gateway/server-methods/config.ts#L471)

## Risks / Open Questions

- Part of the request is adjacent to existing exec-policy visibility, so it is easy to over-credit the current implementation.
- I could not use authenticated `gh pr view` in this environment, so I did not inspect PR #37282 directly; this decision is based on `main` code/docs and read-only Git inspection.
- A deployment can make config externally read-only (for example Nix-managed), but that is not the same as a first-class immutable operator-policy feature in OpenClaw itself.
