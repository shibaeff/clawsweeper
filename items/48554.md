## Summary

Keep open. Current `main` has a FormData normalization path for runtime-dispatcher fetches, but the proxy-fetch wrapper used by media transcription still forwards multipart bodies straight to `undici.fetch` without normalization, and there is still no proxy-wrapper regression test covering `globalThis.FormData` under `HTTP_PROXY`/`HTTPS_PROXY`.

## Evidence

- **Proxy wrapper still passes request body through unchanged:** `makeProxyFetch()` and `resolveProxyFetchFromEnv()` still call `undiciFetch(..., { ...(init as Record<string, unknown>), dispatcher: ... })` with no FormData normalization step.
  - file: [src/infra/net/proxy-fetch.ts:25](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/infra/net/proxy-fetch.ts#L25)
  - command: `nl -ba src/infra/net/proxy-fetch.ts | sed -n '1,200p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Audio transcription still opts into proxy fetch from env:** The media runner still resolves `fetchFn` via `resolveProxyFetchFromEnv()` before calling provider audio transcription code, so proxied audio requests still traverse the proxy wrapper path.
  - file: [src/media-understanding/runner.entries.ts:571](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/media-understanding/runner.entries.ts#L571)
  - command: `nl -ba src/media-understanding/runner.entries.ts | sed -n '565,610p'`

- **Guarded fetch prefers caller-supplied proxy wrapper over runtime-normalized fetch:** When a non-global `fetchImpl` is supplied, `fetchWithSsrFGuard()` uses that fetch implementation directly instead of `fetchWithRuntimeDispatcher()`.   - file: [src/infra/net/runtime-fetch.ts:21](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/infra/net/runtime-fetch.ts#L21)
