## Summary

Keep open. Current `main` still stores the full cumulative `chatStream` into `chatStreamSegments` when a tool event starts, while chat deltas keep `chatStream` cumulative and the renderer displays stored segments plus the live stream separately. That matches the duplicate-bubble regression this PR is trying to fix. The PR discussion also found and corrected an initial 3+ text/tool-cycle bug in the proposed patch, but no equivalent fix is present on `main` yet.

## Evidence

- **Current main still appends full cumulative stream text:** When a new tool stream entry starts, `handleAgentEvent` commits `host.chatStream` verbatim into `chatStreamSegments`; there is no delta-slicing logic on `main`. That is the exact behavior the PR is trying to change.
  - file: [ui/src/ui/app-tool-stream.ts:502](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/ui/src/ui/app-tool-stream.ts#L502)
  - command: `nl -ba ui/src/ui/app-tool-stream.ts | sed -n '470,525p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Chat delta handling keeps `chatStream` cumulative:** On every `chat` delta event, the client assigns `state.chatStream = next`, where `next` is the extracted assistant text from the latest payload. The client is not storing only the incremental suffix here.
  - file: [ui/src/ui/controllers/chat.ts:679](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/ui/src/ui/controllers/chat.ts#L679)
