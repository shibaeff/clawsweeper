## Summary

Keep open. Current `main` does not already implement Claude CLI credential adoption/write-back for Anthropic OAuth, and the PR still needs rework against the newer profile-scoped external-auth architecture before it would be safe to land or close as obsolete.

## Evidence

- **Current main only maps external CLI bootstrap to MiniMax:** `EXTERNAL_CLI_SYNC_PROVIDERS` contains only `minimax-portal:minimax-cli`, and `readExternalCliBootstrapCredential()` resolves by exact `profileId`. There is no Claude/Anthropic entry on `main`, so the PR’s requested Claude CLI sync is not already implemented.
  - file: [src/agents/auth-profiles/external-cli-sync.ts:74](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/auth-profiles/external-cli-sync.ts#L74)
  - command: `nl -ba src/agents/auth-profiles/external-cli-sync.ts | sed -n '1,220p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **OAuth refresh path only consults mapped external providers:** `oauth.ts` wires refresh bootstrap through `readManagedExternalCliCredential()`, and `oauth-manager.ts` only adopts an external credential returned by that adapter before falling through to network refresh. Because Claude is not mapped in `external-cli-sync.ts`, current `main` still will not adopt Claude CLI credentials during Anthropic refresh.
  - file: [src/agents/auth-profiles/oauth-manager.ts:404](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/auth-profiles/oauth-manager.ts#L404)
  - command: `nl -ba src/agents/auth-profiles/oauth-manager.ts | sed -n '180,470p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Current store/runtime code preserves no-keychain-prompt paths:** Secrets/runtime loads explicitly pass `allowKeychainPrompt: false`. Any Claude CLI sync added at store-load time now has to honor that contract; otherwise it can introduce unexpected macOS Keychain access in read-only or non-interactive flows.
  - file: [src/agents/auth-profiles/store.ts:282](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/auth-profiles/store.ts#L282)
  - command: `rg -n "allowKeychainPrompt" src/agents -S && sed -n '260,420p' src/agents/auth-profiles/store.ts`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **The PR still carries unresolved design risk against current architecture:** The latest PR head (`a8e37630fa657e4fccf16ffa519aefc7600bafb9`) fixed the initial macOS Keychain/cache-review comments, but the remaining review feedback still points out that the PR’s refresh adoption/write-back is keyed broadly on `cred.provider === "anthropic"`. That is at odds with current `main`, which treats external CLI auth as profile-scoped and identity-gated, so closing this PR would discard still-useful work rather than reflect a fix already present on `main`.
  - file: [src/agents/auth-profiles/oauth.ts](https://github.com/openclaw/openclaw/blob/a8e37630fa657e4fccf16ffa519aefc7600bafb9/src/agents/auth-profiles/oauth.ts)
