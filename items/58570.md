## Summary

Keep open. Current `main` still does not emit the requested warning-level structured log when an inbound message is dropped because a resolved channel route is denied. The Slack path now has verbose/debug diagnostics, but that is not equivalent to the issue’s requested observability.

## Evidence

- **Explicit deny still blocks inbound routing:** The Slack monitor tests still assert that an explicitly denied channel is blocked even when group policy is open, so the allow:false-style drop path remains real on current main.
  - file: [extensions/slack/src/monitor/monitor.test.ts:266](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/slack/src/monitor/monitor.test.ts#L266)
  - command: `nl -ba extensions/slack/src/monitor/monitor.test.ts | sed -n '266,280p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Message drop path is verbose-only:** When `ctx.isChannelAllowed(...)` is false, the inbound message handler logs `slack: drop message (channel not allowed)` via `logVerbose(...)` and returns `null`; it does not call `logger.warn` or emit a structured warning payload.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:199](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/slack/src/monitor/message-handler/prepare.ts#L199)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare.ts | sed -n '199,207p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Channel-deny diagnostic is still verbose/debug, not warning:** The channel-policy helper logs `slack: drop channel <id> (...)` with match metadata through `logVerbose(...)` before returning `false`. This adds some diagnostics, but still not a warning-level structured log with a machine-readable reason.
  - file: [extensions/slack/src/monitor/context.ts:360](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/slack/src/monitor/context.ts#L360)
  - command: `nl -ba extensions/slack/src/monitor/context.ts | sed -n '341,364p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **`logVerbose` does not provide normal warning logging:** `logVerbose` returns early unless console verbose is enabled or file log level is `debug`; when it does log, it uses `getLogger().debug(...)`, not warn-level logging.
  - file: [src/globals.ts:6](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/globals.ts#L6)
  - command: `nl -ba src/globals.ts | sed -n '6,23p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Docs confirm verbose/debug-only visibility:** Gateway logging docs state that `--verbose` only changes console verbosity and that verbose-only file details require `logging.level=debug` or `trace`, which means default incident logs still would not show this drop as a warning.
  - file: [docs/gateway/logging.md:35](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/docs/gateway/logging.md#L35)

## Risks / Open Questions

- Current evidence is from the Slack inbound path; other channel plugins may behave differently, but there is no strong evidence of a gateway-wide structured warning implementation for this class of denial.
- A maintainer could decide the new verbose/debug lines are sufficient and narrow the issue instead, but they do not match the requested warning-level structured observability closely enough for a conservative close.
