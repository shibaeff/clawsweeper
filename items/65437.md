## Summary

Keep open. Current `main` still routes Slack group `/…` commands through the generic control-command gate before elevated permissions are resolved, so `tools.elevated.allowFrom.slack` alone does not authorize `/elevated on` when `channels.slack.allowFrom` and `channels.slack.channels.<id>.users` are unset. The docs also still describe only a Discord fallback, not a Slack one.

## Evidence

- **Slack inbound preflight blocks unauthorized control commands before reply directive handling:** `extensions/slack/src/monitor/message-handler/prepare.ts` computes `commandAuthorized` from Slack owner/channel allowlists and immediately returns `null` when `commandGate.shouldBlock` is true for a control command.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:409](https://github.com/openclaw/openclaw/blob/453789914bb725d1cd5a3edd7ec5d8b0b45041cd/extensions/slack/src/monitor/message-handler/prepare.ts#L409)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare.ts | sed -n '395,430p'`
  - sha: [453789914bb7](https://github.com/openclaw/openclaw/commit/453789914bb725d1cd5a3edd7ec5d8b0b45041cd)
- **Generic command gate rejects any control command when no authorizer matched:** `src/channels/command-gating.ts` sets `shouldBlock = allowTextCommands && hasControlCommand && !commandAuthorized`, so a Slack `/elevated` message is dropped as soon as neither `channels.slack.allowFrom` nor `channels.slack.channels.<id>.users` authorizes it.
  - file: [src/channels/command-gating.ts:37](https://github.com/openclaw/openclaw/blob/453789914bb725d1cd5a3edd7ec5d8b0b45041cd/src/channels/command-gating.ts#L37)
  - command: `nl -ba src/channels/command-gating.ts | sed -n '31,44p'`
  - sha: [453789914bb7](https://github.com/openclaw/openclaw/commit/453789914bb725d1cd5a3edd7ec5d8b0b45041cd)
- **Elevated permissions are resolved later in the reply pipeline:** `src/auto-reply/reply/get-reply-directives.ts` does not call `resolveElevatedPermissions` until after directive parsing, which is downstream of Slack preflight; a message dropped in Slack preflight never reaches this elevated allowlist check.
  - file: [src/auto-reply/reply/get-reply-directives.ts:377](https://github.com/openclaw/openclaw/blob/453789914bb725d1cd5a3edd7ec5d8b0b45041cd/src/auto-reply/reply/get-reply-directives.ts#L377)
  - command: `nl -ba src/auto-reply/reply/get-reply-directives.ts | sed -n '372,386p'`
  - sha: [453789914bb7](https://github.com/openclaw/openclaw/commit/453789914bb725d1cd5a3edd7ec5d8b0b45041cd)
- **Current Slack audit text matches the reported failure mode:** `extensions/slack/src/security-audit.ts` warns that when Slack slash/native commands are enabled but neither an owner allowlist nor any `channels.<id>.users` allowlist is configured, `/…` commands are rejected for everyone, and recommends pairing or configuring `channels.slack.allowFrom` / `channels.slack.channels.<id>.users`.
  - file: [extensions/slack/src/security-audit.ts:79](https://github.com/openclaw/openclaw/blob/453789914bb725d1cd5a3edd7ec5d8b0b45041cd/extensions/slack/src/security-audit.ts#L79)
  - command: `nl -ba extensions/slack/src/security-audit.ts | sed -n '79,88p'`
  - sha: [453789914bb7](https://github.com/openclaw/openclaw/commit/453789914bb725d1cd5a3edd7ec5d8b0b45041cd)
- **Docs still document only a Discord elevated fallback:** `docs/tools/elevated.md` lists a Discord fallback for `tools.elevated.allowFrom.discord` but does not document any Slack equivalent, which aligns with the issue's docs-gap claim rather than showing the issue is already resolved.
  - file: [docs/tools/elevated.md:84](https://github.com/openclaw/openclaw/blob/453789914bb725d1cd5a3edd7ec5d8b0b45041cd/docs/tools/elevated.md#L84)
