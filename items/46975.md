## Summary

Issue #46975 is already fixed on current `main` and the fix is present in release `v2026.4.22`: the Discord model picker keeps provider and model separate, then reconstructs and dispatches the full `provider/model` ref, so nested OpenRouter ids do not lose the `openrouter/` prefix.

## Close Comment

Closing from a Codex review because this is already fixed on `main`, and the same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The Discord picker now keeps the provider in component state and uses only the model suffix as the select value (`extensions/discord/src/monitor/model-picker.ts:438`), then reconstructs and dispatches the full `provider/model` ref when you select and submit (`extensions/discord/src/monitor/native-command-ui.ts:672`, `extensions/discord/src/monitor/native-command-ui.ts:733`). That prevents nested OpenRouter ids such as `openrouter/deepseek/deepseek-v3.2` from losing the `openrouter/` prefix and being rejected as disallowed. The current picker flow is also covered by Discord model-picker tests (`extensions/discord/src/monitor/model-picker.test.ts:466`, `extensions/discord/src/monitor/native-command.model-picker.test.ts:306`).
