## Summary

Current `main` already routes Feishu subagent completion messages back to the originating Feishu DM/topic conversation, including preserving the original topic `threadId`, and it has targeted regression tests for that behavior. The same implementation is present in release `v2026.4.22`.

## Risks / Open Questions

- I did not run a live Feishu reproduction in this read-only review, so confidence comes from current code plus targeted tests rather than an end-to-end channel run.
- `docs/tools/subagents.md` still says Discord is the only supported thread-binding channel, so docs may lag Feishu's current-conversation completion routing behavior.

## Close Comment

Closing as `implemented_on_main` after a Codex review. Current OpenClaw already preserves Feishu subagent completion routing back to the originating conversation/thread: the Feishu hooks store `deliveryTo` and `deliveryThreadId` when the subagent is spawned and reuse that data when resolving the completion target (`extensions/feishu/src/subagent-hooks.ts:288`, `extensions/feishu/src/thread-bindings.ts:17`, `extensions/feishu/src/subagent-hooks.ts:359`). There is also a regression test that explicitly checks a Feishu topic-thread completion returns to `chat:oc_group_chat` with `threadId: "om_topic_root"` rather than leaking to the group main timeline (`extensions/feishu/src/subagent-hooks.test.ts:115`). The same implementation is already present in shipped release `v2026.4.22` at commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`.
