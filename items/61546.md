## Summary

Keep open. Current `main` still accepts overflowing `heartbeat.every` values and still passes the resulting delay straight into Node's `setTimeout`, so the reported 1ms reschedule loop remains plausible on current code.

## Evidence

- **Scheduler still uses an unbounded timeout delay:** `scheduleNext()` computes `delay = Math.max(0, nextDue - now)` and passes it directly to `setTimeout(..., delay)` with no clamp or re-arm logic for values above Node's timer limit.
  - file: [src/infra/heartbeat-runner.ts:1389](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/infra/heartbeat-runner.ts#L1389)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '1360,1435p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Heartbeat interval parsing has no upper bound:** `resolveHeartbeatIntervalMs()` returns any positive `parseDurationMs(...)` result. It does not cap or reject values above Node's 32-bit timeout maximum.
  - file: [src/infra/heartbeat-summary.ts:61](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/infra/heartbeat-summary.ts#L61)
  - command: `nl -ba src/infra/heartbeat-summary.ts | sed -n '1,180p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Config validation still allows overflowing durations:** `HeartbeatSchema` only checks that `every` parses as a duration string. There is no validation for oversized heartbeat intervals.
  - file: [src/config/zod-schema.agent-runtime.ts:46](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/config/zod-schema.agent-runtime.ts#L46)
  - command: `nl -ba src/config/zod-schema.agent-runtime.ts | sed -n '1,120p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Current Node runtime still clamps these values to 1ms:** On the local Node `v24.14.1` runtime, `setTimeout` with `99999h` and `999h`-equivalent millisecond delays emitted `TimeoutOverflowWarning` and reported that the timeout duration was set to `1`, matching the issue's failure mode.
  - command: `node -v; node -e 'const ms=99999*3600000; const t=setTimeout(()=>{},ms); console.log(ms); clearTimeout(t);'; node -e 'const ms=999*3600000; const t=setTimeout(()=>{},ms); console.log(ms); clearTimeout(t);'`
- **Issue discussion still indicates an outstanding fix:** The provided timeline cross-references PR #61638 (`fix(config): reject heartbeat.every values that overflow Node.js setTimeout`) as open, and a later comment on 2026-04-22 reports the bug still reproducing on `openclaw 2026.4.21` with `999h`.

## Risks / Open Questions

- Users can still configure a long-but-positive heartbeat interval that degrades into a 1ms loop, flooding logs and starving normal channel processing.
- The docs mention `0m` disables heartbeat, but the product still does not guard or clearly warn against oversized nonzero intervals, so the misconfiguration remains easy to repeat.
