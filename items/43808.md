## Summary

Keep open. Current `main` already carries most of the internal cron `delivery.threadId` plumbing, but the PR still contains useful work that is not on `main`: Gateway cron validation/schema still does not accept `delivery.threadId`, and failure-target deduplication still ignores thread context.

## Evidence

- **Internal cron delivery already threads explicit threadId:** `resolveCronDeliveryPlan()` reads `delivery.threadId`, `run.ts` forwards it into delivery-target resolution, and `resolveDeliveryTarget()` passes it as `explicitThreadId` for target resolution.
  - file: [src/cron/delivery-plan.ts:50](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/cron/delivery-plan.ts#L50)

  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Create/update normalization already preserves threadId internally:** `coerceDelivery()` preserves parsed `threadId`, and `mergeCronDelivery()` keeps existing `threadId` plus applies patched `threadId` values.
  - file: [src/cron/normalize.ts:278](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/cron/normalize.ts#L278)
