## Summary

Keep open. Current `main` already threads `delivery.threadId` through cron announce runtime, but the PR still covers two real gaps that are not on `main`: the gateway cron schema does not accept/round-trip `delivery.threadId`, and failure-destination deduplication still ignores thread context.

## Evidence

- **Delivery plan carries threadId on main:** `resolveCronDeliveryPlan()` reads `delivery.threadId` and includes it in the returned `CronDeliveryPlan`, so the core runtime part of the bug is already present on `main`.
  - file: [src/cron/delivery-plan.ts:29](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/cron/delivery-plan.ts#L29)
  - command: `nl -ba src/cron/delivery-plan.ts | sed -n '29,68p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Isolated announce delivery forwards threadId:** `resolveCronDeliveryContext()` passes `deliveryPlan.threadId` into `resolveDeliveryTarget()`, which is the actual announce-delivery path for isolated cron jobs.
  - file: [src/cron/isolated-agent/run.ts:329](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/cron/isolated-agent/run.ts#L329)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '320,340p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Gateway cron schema still omits threadId:** `CronDeliverySharedProperties` does not define `threadId`, and the add/update/job schemas reuse `CronDeliverySchema` / `CronDeliveryPatchSchema` with `additionalProperties: false`. That means gateway `cron.add`, `cron.update`, and job payload round-trips still do not accept or expose `delivery.threadId`.
  - file: [src/gateway/protocol/schema/cron.ts:180](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/gateway/protocol/schema/cron.ts#L180)
  - command: `nl -ba src/gateway/protocol/schema/cron.ts | sed -n '180,229p' && nl -ba src/gateway/protocol/schema/cron.ts | sed -n '256,316p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Failure-target dedupe still ignores thread context:** `isSameDeliveryTarget()` compares only mode/channel/`to`/`accountId`; it never compares the primary delivery's `threadId`. A topic-routed Telegram announce target can therefore still suppress a same-chat failure destination incorrectly.
  - file: [src/cron/delivery-plan.ts:183](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/cron/delivery-plan.ts#L183)

- **PR discussion matches the remaining missing pieces:** The PR author's Mar 12 follow-up said the last fixes were gateway cron schema round-trip plus failure-fallback handling. Those exact pieces are still missing from current `main`, so the PR remains materially useful even though some runtime plumbing has since landed separately.
  - sha: [656eb1ddb60c](https://github.com/openclaw/openclaw/commit/656eb1ddb60cb9b22ba8cec6469747ea86851a3b)

## Risks / Open Questions

- Closing this PR would leave gateway clients unable to create, update, or list cron jobs with `delivery.threadId` through the typed protocol.
- Closing this PR would also leave the failure-notification suppression bug in place for topic-routed Telegram announce jobs that use explicit `delivery.threadId`.
