## Summary

Close as `clawhub`. Current `main` already provides the core enforcement primitives and Feishu knowledge/doc access, while the remaining behavior requested here is a workflow-specific retrieval/abstain policy that `VISION.md` explicitly pushes toward plugins/ClawHub rather than OpenClaw core.

## Risks / Open Questions

- A ClawHub implementation still needs a concrete retrieval strategy for Feishu KB content, since `feishu_wiki` currently supports browse/get flows but not full text search.
- No OpenClaw core change can truly prove 'never use model prior knowledge'; that behavior is inherently prompt/workflow constrained, so expectations should be set accordingly.

## Close Comment

Closing this as a ClawHub-scope request after Codex review against current `main` (`68315792677f9628d1370d6869366442f1d3ecad`). OpenClaw already has the core pieces needed here: hard tool policy can block web tools like `web_search`/`web_fetch` (`docs/gateway/config-tools.md:16-49`), agents can be locked down to specific skills or no skills (`docs/gateway/config-agents.md:41-58`), and Feishu knowledge/doc access already lives in plugin-owned tools (`extensions/feishu/src/wiki.ts:174-231`, `extensions/feishu/src/docx.ts:1418-1593`, plus the matching Feishu skills). The remaining behavior in this issue is a specialized 'query Feishu KB/docs, abstain if missing, use a canned fallback' workflow, and `VISION.md:54-70` says optional capability should usually ship as plugins and new skills should go to ClawHub first. That makes this a better fit for a Feishu KB-only ClawHub skill/plugin than for new OpenClaw core surface.
