## Summary

Current `main` already contains explicit ACPX recovery logic for the reported `queue owner unavailable` / stale-resume-session failure mode, and the changelog shows it shipped in `v2026.4.22`. The code now retries ACP session init without persisted resume ids, clears stale persistent resume state on `Resource not found`, and prepares a fresh session instead of leaving the ACP session dead.

## Evidence

- **Changelog entry matches the reported failure mode:** `CHANGELOG.md` explicitly lists two ACPX/runtime fixes: repairing `queue owner unavailable` session recovery and retrying dead-session repair without `--resume-session` when the ACPX session id is stale.
  - file: [CHANGELOG.md:1651](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/CHANGELOG.md#L1651)

- **Session init now retries without stale persisted ids:** The ACP manager catches failed persistent-session resume init, strips stale `acpxSessionId` / `agentSessionId`, and re-runs `ensureSession()` without a persisted resume id.
  - file: [src/acp/control-plane/manager.core.ts:1448](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/acp/control-plane/manager.core.ts#L1448)

## Close Comment

Closing from a Codex review: this issue appears implemented on current `main` and shipped in `v2026.4.22`. The changelog calls out the exact ACPX/runtime fixes for `queue owner unavailable` recovery and stale `--resume-session` fallback (`CHANGELOG.md:1651`, `CHANGELOG.md:1652`). The current ACP manager now retries persistent ACP session init without persisted resume ids after resume-init failure (`src/acp/control-plane/manager.core.ts:1448`) and clears stale persistent resume state before preparing a fresh session when the backend reports `Resource not found` (`src/acp/control-plane/manager.core.ts:1699`). Regression coverage was added for dead-status re-ensure plus the stale-session `Resource not found` recovery path (`src/acp/control-plane/manager.test.ts:792`, `src/acp/control-plane/manager.test.ts:1148`, `src/acp/control-plane/manager.test.ts:2101`). If you still see this on `v2026.4.22` or newer, please open a new issue with a fresh `/acp status` snapshot and logs from that version so maintainers can check whether it is a new regression.
