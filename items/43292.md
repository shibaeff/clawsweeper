## Summary

Keep open. Current `main` still contains both reported gaps: the reminder guard only treats matching `sessionKey` cron jobs as coverage, and compaction retry still zeros `successfulCronAdds` before the final attempt result is emitted.

## Evidence

- **Reminder guard still ignores isolated-session coverage:** `hasSessionRelatedCronJobs` only accepts `cronStorePath` and `sessionKey`, and returns true only for enabled jobs whose `job.sessionKey === params.sessionKey`. There is no `agentId` fallback for isolated jobs.
  - file: [src/auto-reply/reply/agent-runner-reminder-guard.ts:29](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/auto-reply/reply/agent-runner-reminder-guard.ts#L29)

  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Reply pipeline still relies on that session-only guard when `successfulCronAdds === 0`:** The reply path computes `coveredByExistingCron` only via `hasSessionRelatedCronJobs({ cronStorePath, sessionKey })`; it does not pass `agentId` or any isolated-cron identifier.
  - file: [src/auto-reply/reply/agent-runner.ts:1388](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/auto-reply/reply/agent-runner.ts#L1388)
