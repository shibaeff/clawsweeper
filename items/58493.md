## Summary

Current `main` already restores full operator scopes and owner semantics for shared-secret bearer auth on `/v1/chat/completions`, with matching unit/e2e coverage and docs. This appears fixed on `main` but not yet in a tagged release.

## Evidence

- **Compat HTTP path uses the OpenAI-specific scope resolver:** `handleOpenAiHttpRequest` now routes `/v1/chat/completions` through `handleGatewayPostJsonEndpoint` with `resolveOperatorScopes: resolveOpenAiCompatibleHttpOperatorScopes`, rather than relying on the generic trusted-HTTP scope fallback.
  - file: [src/gateway/openai-http.ts:505](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/gateway/openai-http.ts#L505)
  - command: `nl -ba src/gateway/openai-http.ts | sed -n '499,516p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Shared-secret bearer auth restores full operator scopes:** `resolveOpenAiCompatibleHttpOperatorScopes` explicitly returns `CLI_DEFAULT_OPERATOR_SCOPES` for shared-secret gateway methods, which includes `operator.write`; `resolveOpenAiCompatibleHttpSenderIsOwner` also marks that path as owner.
  - file: [src/gateway/http-utils.ts:232](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/gateway/http-utils.ts#L232)
  - command: `nl -ba src/gateway/http-utils.ts | sed -n '232,266p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Unit coverage matches the reported regression:** Request-context tests assert that shared-secret bearer auth plus a narrower `x-openclaw-scopes` header still resolves to the full default operator scope set, including `operator.write`.
  - file: [src/gateway/http-utils.request-context.test.ts:128](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/gateway/http-utils.request-context.test.ts#L128)
  - command: `nl -ba src/gateway/http-utils.request-context.test.ts | sed -n '128,156p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **End-to-end chat completions test covers bearer auth behavior:** The OpenAI HTTP e2e test starts a token-auth gateway, sends `Authorization: Bearer secret` with a narrower `x-openclaw-scopes: operator.approvals`, gets HTTP 200, and verifies the resulting agent call is treated as owner-backed.
  - file: [src/gateway/openai-http.test.ts:1211](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/gateway/openai-http.test.ts#L1211)
  - command: `nl -ba src/gateway/openai-http.test.ts | sed -n '1211,1239p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Docs now state the intended contract explicitly:** The `/v1/chat/completions` docs say shared-secret token/password auth restores the full default operator scope set and owner semantics, and ignores narrower `x-openclaw-scopes` values on that path.
  - file: [docs/gateway/openai-http-api.md:51](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/docs/gateway/openai-http-api.md#L51)
