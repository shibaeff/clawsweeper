## Summary

Keep open. Current `main` still leaves `storeLocks` entries resident indefinitely in `src/cron/service/locked.ts`, and the proposed cleanup/test coverage from PR #43390 is not present.

## Evidence

- **Current implementation still leaks per-store entries:** `src/cron/service/locked.ts:3-21` defines a module-global `Map<string, Promise<void>>`, writes `storeLocks.set(storePath, keepAlive)`, and returns without any `finally`/`delete` path to prune resolved entries.
  - file: [src/cron/service/locked.ts:3](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cron/service/locked.ts#L3)
  - command: `nl -ba src/cron/service/locked.ts | sed -n '1,120p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Map key is an arbitrary service store path:** `src/cron/service/state.ts:40-44,135-145` shows `storePath` is injected into each cron service state, so long-running processes can accumulate distinct keys over time if multiple store paths are used.
  - file: [src/cron/service/state.ts:40](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cron/service/state.ts#L40)
