## Summary

The request is a useful conditional-skill workflow, but current main already steers this kind of optional behavior into plugins/bundles distributed through ClawHub rather than adding new core SKILL.md schema and file-operation hook semantics.

## Evidence

- **Project direction is plugin-first for optional capability:** OpenClaw vision says core should stay lean, optional capability should usually ship as plugins, and if something useful cannot be built as a plugin the preferred path is extending the plugin API rather than adding one-off core behavior.

## Risks / Open Questions

- Current main does not provide a built-in `SKILL.md` `paths` field, so an author would need to implement the file-pattern matching logic inside a plugin or bundle hook.
- Codex-style hook packs are supported in bundle form; Claude `hooks.json` automation is not the documented execution path on main, so the practical packaging target is a native plugin or Codex-style bundle.
