## Summary

Current `main` already contains a Windows Scheduled Task startup fallback that stops treating the `schtasks` "not yet run" state as a hard duplicate-start condition when there is evidence the gateway is already launching or listening. The new logic caps that wait at 15 seconds, checks for verified gateway listeners and matching task processes, and has regression tests for the exact startup-progress cases described in #61579.

## Evidence

- **Fallback wait is bounded and only triggers when `schtasks` remains stuck with no startup evidence:** `shouldFallbackScheduledTaskLaunch` recognizes the Windows `0x41303` not-yet-run code, polls for at most 15 seconds, and only falls back when the state never changes and no listener/process evidence appears.
  - file: [src/daemon/schtasks.ts:216](https://github.com/openclaw/openclaw/blob/f191dd3d536ceb8b499fd0e68ff7f1c4f98baf50/src/daemon/schtasks.ts#L216)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '210,225p;664,810p'`
  - sha: [f191dd3d536c](https://github.com/openclaw/openclaw/commit/f191dd3d536ceb8b499fd0e68ff7f1c4f98baf50)
- **Runtime detection now treats a verified gateway listener as running even if `schtasks` still reports stale state:** `readScheduledTaskRuntime` consults `resolveListenerBackedScheduledTaskRuntime`, which returns `running` when a verified gateway listener is on the configured port, instead of leaving the service in a false stopped/not-yet-run state.
  - file: [src/daemon/schtasks.ts:431](https://github.com/openclaw/openclaw/blob/f191dd3d536ceb8b499fd0e68ff7f1c4f98baf50/src/daemon/schtasks.ts#L431)

  - sha: [f191dd3d536c](https://github.com/openclaw/openclaw/commit/f191dd3d536ceb8b499fd0e68ff7f1c4f98baf50)
- **Regression tests cover startup progress and already-starting task cases:** Tests assert that OpenClaw does not relaunch when `schtasks` shows startup progress after `/Run`, does not relaunch when the scheduled-task process is already starting, and reports a fallback-launched gateway as running while `schtasks` still says not-yet-run.
  - file: [src/daemon/schtasks.startup-fallback.test.ts:258](https://github.com/openclaw/openclaw/blob/f191dd3d536ceb8b499fd0e68ff7f1c4f98baf50/src/daemon/schtasks.startup-fallback.test.ts#L258)

- **User-facing docs now describe quick Windows fallback behavior:** The Windows docs say Scheduled Tasks are tried first, then OpenClaw falls back to a per-user Startup-folder launcher, and that wedged `schtasks` calls now abort quickly instead of hanging.
