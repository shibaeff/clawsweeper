## Summary

Keep open. Current `main` still hardcodes inline dreaming output to `memory/YYYY-MM-DD.md`, and the shipped `memory-core` config surface only exposes storage mode toggles, not a configurable inline target such as `memory/daily-memory/{date}.md`. The recent default change to `dreaming.storage.mode: "separate"` is a partial workaround, but it does not implement the issue's requested inline-path alignment.

## Evidence

- **Inline path is still hardcoded:** `resolveDailyMemoryPath()` always returns `path.join(workspaceDir, "memory", `${isoDay}.md`)`, and `writeDailyDreamingPhaseBlock()` uses that path whenever inline storage is enabled. No config-controlled relative path or target selector is consulted.

## Risks / Open Questions

- The default switch to `dreaming.storage.mode: "separate"` may make the feature less urgent for some users, but it does not satisfy operators who explicitly want inline blocks merged into `memory/daily-memory/{date}.md`.
- Public docs currently summarize only `enabled` and `frequency` for dreaming, while the schema also exposes `storage` knobs; that docs/config mismatch makes the surface harder to reason about, but it still does not show the requested path-setting exists.
