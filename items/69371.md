## Summary

Keep open. The report is recent and coherent, and current `main` does not show an ARM64-specific stabilization pass for the `status`/`sessions`/`message send` paths. Some related pieces already exist (cron retries, compaction controls, `sessions.send` run IDs), but the requested ARM-aware timeout handling, idle-context compaction triggers, and `message send` background/retry surface are not implemented well enough to treat this as fixed or obsolete.

## Evidence

- **Pi support is still a documented target:** The public Raspberry Pi install guide still presents Pi 4/5 as a normal OpenClaw host and recommends compile-cache/performance tweaks, which makes ARM64 stability a current supported surface rather than an out-of-scope niche.
  - file: [docs/install/raspberry-pi.md:10](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/install/raspberry-pi.md#L10)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Status still has short internal timeout caps:** `openclaw status` exposes `--timeout` with a 10s default, but the overview scan hard-caps the `channels.status` request payload timeout to 8s and the outer gateway call to 2.5s/5s. That is not an ARM-aware relaxation and is consistent with the report's concern about slow edge devices.
  - file: [src/commands/status.scan-overview.ts:77](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/commands/status.scan-overview.ts#L77)

- **Compaction knobs exist, but not the requested idle-context trigger settings:** Current compaction config exposes `reserveTokens`, `keepRecentTokens`, `maxHistoryShare`, `timeoutSeconds`, `memoryFlush`, and related safeguards, but there is no `contextHighMark` or periodic idle context-check field matching the issue request.
