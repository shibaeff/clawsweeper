## Summary

Keep open. Current `main` has fixes for stale auto-fallback session state on `/reset` and `/new`, but the status/session code still reads persisted session model state, and I did not find strong evidence that a plain gateway restart refreshes fallback display/state the way #61318 reports.

## Evidence

- **`openclaw status` sessions are built from persisted session model state:** The status summary builds session rows by calling `resolveSessionModelRef(cfg, entry, ...)` for each stored session entry, then displays that resolved model in the Sessions section.
  - file: [src/commands/status.summary.ts:195](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/commands/status.summary.ts#L195)
  - command: `nl -ba src/commands/status.summary.ts | sed -n '186,232p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Persisted runtime/override model wins over current defaults:** `resolveSessionModelRef()` first resolves the configured default, then prefers `entry.modelProvider`/`entry.model` and `providerOverride`/`modelOverride` via `resolvePersistedSelectedModelRef()`. That means stale persisted session model state can still drive status output unless something clears/recomputes it.
  - file: [src/gateway/session-utils.ts:1102](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/gateway/session-utils.ts#L1102)
