## Summary

Keep open. Current main still has UTF-8-only decoding in the exec paths, while only the node-host/system.run path has Windows codepage-aware decoding. The docs also still describe garbled Chinese exec output on latest OpenClaw as an active problem with a workaround and tracker reference.

## Evidence

- **runExec still hardcodes UTF-8:** `runExec` builds `execFile` options with `encoding: "utf8"` for both numeric and object options, including on Windows.
  - file: [src/process/exec.ts:131](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/process/exec.ts#L131)
  - command: `nl -ba src/process/exec.ts | sed -n '130,155p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **exec supervisor child adapter still decodes with default Buffer.toString():** The non-PTY exec path converts stdout/stderr chunks via `chunk.toString()` with no Windows codepage detection or override.
  - file: [src/process/supervisor/adapters/child.ts:111](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/process/supervisor/adapters/child.ts#L111)
  - command: `nl -ba src/process/supervisor/adapters/child.ts | sed -n '111,120p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Windows codepage-aware decoding exists only in node-host path:** `src/node-host/invoke.ts` maps Windows code pages (including 936=>`gbk`) and decodes captured buffers with `TextDecoder(encoding)`, showing the repo has a fix pattern for `system.run`-style execution but not for the exec runtime above.
  - file: [src/node-host/invoke.ts:36](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/node-host/invoke.ts#L36)
  - command: `nl -ba src/node-host/invoke.ts | sed -n '30,70p;108,168p;188,215p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Docs still treat mojibake in exec output as current issue:** FAQ documents that native Windows `system.run`/`exec` can render Chinese as mojibake, gives a manual `chcp 65001` workaround, and points users to a tracker issue if it still reproduces on latest OpenClaw.
  - file: [docs/help/faq-first-run.md:395](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/docs/help/faq-first-run.md#L395)
  - command: `nl -ba docs/help/faq-first-run.md | sed -n '395,420p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)

## Risks / Open Questions

- I could not run a native Windows reproduction in this Linux checkout, so the review is code-and-docs based rather than an end-to-end Windows repro.
- The Windows exec tool now prefers PowerShell (`src/agents/shell-utils.ts:54`), which may reduce the problem for some commands, but the child-process decoding path still lacks explicit codepage handling.
