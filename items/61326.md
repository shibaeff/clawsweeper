## Summary

Current `main` already resets the WhatsApp watchdog window on reconnect, includes an explicit regression test for that behavior, and the fix is documented as shipped in `v2026.4.5`.

## Evidence

- **Release note matches the reported symptom:** The `2026.4.5` changelog explicitly says the WhatsApp fix resets watchdog timeouts after reconnect so quiet chats no longer fall into reconnect loops, which matches this report's infinite 499-loop symptom.
  - file: [CHANGELOG.md:1288](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/CHANGELOG.md#L1288)

  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Each reconnect gets a new watchdog baseline:** `createLiveConnection()` initializes a fresh connection with `startedAt = Date.now()` and `lastInboundAt = null`, so reconnects do not inherit the previous idle baseline.
  - file: [extensions/whatsapp/src/connection-controller.ts:103](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/whatsapp/src/connection-controller.ts#L103)
  - command: `nl -ba extensions/whatsapp/src/connection-controller.ts | sed -n '103,116p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Reconnect path replaces the live connection and restarts timers:** `openConnection()` closes any existing connection, creates a new live connection object, registers it, and starts timers for that new connection.
  - file: [extensions/whatsapp/src/connection-controller.ts:348](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/whatsapp/src/connection-controller.ts#L348)
  - command: `nl -ba extensions/whatsapp/src/connection-controller.ts | sed -n '339,375p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Watchdog now measures idleness from the current connection:** The watchdog computes staleness from `connection.lastInboundAt ?? connection.startedAt`, so after reconnect the timeout window restarts from the new connection's start time.
  - file: [extensions/whatsapp/src/connection-controller.ts:563](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/whatsapp/src/connection-controller.ts#L563)
  - command: `nl -ba extensions/whatsapp/src/connection-controller.ts | sed -n '563,579p'`

- **Regression test covers the reported failure mode:** The WhatsApp e2e test `gives a reconnected listener a fresh watchdog window` asserts that a reconnected listener does not immediately fall back into another watchdog-triggered reconnect.
  - file: [extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts:258](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts#L258)
