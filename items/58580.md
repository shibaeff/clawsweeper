## Summary

Keep open. Current main adds stuck-session diagnostics and channel health auto-restarts, but I did not find any implementation that auto-clears or recreates an individual stuck conversation session after prolonged message-level unresponsiveness.

## Evidence

- **Stuck-session handling is warning-only:** `diagnostics.stuckSessionWarnMs` is documented and typed only as a threshold for emitting stuck-session warnings while a session stays in `processing` state; no reset or clear action is described.
  - file: [docs/gateway/configuration-reference.md:814](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/docs/gateway/configuration-reference.md#L814)
  - command: `rg -n "stuckSessionWarnMs|session\.stuck" docs/gateway/configuration-reference.md docs/logging.md src/config/types.base.ts`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Runtime logs stuck sessions but does not recover them:** The diagnostic heartbeat checks processing sessions older than the threshold and calls `logSessionStuck(...)`; that path emits a warning event and returns, with no session reset or store mutation.
  - file: [src/logging/diagnostic.ts:456](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/logging/diagnostic.ts#L456)
  - command: `nl -ba src/logging/diagnostic.ts | sed -n '440,463p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Automatic health recovery is channel-scoped, not session-scoped:** The gateway health monitor evaluates channel/account transport health and restarts the channel via `stopChannel`/`startChannel`. This addresses stale sockets, not a corrupted per-chat session entry.
  - file: [src/gateway/channel-health-monitor.ts:158](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/gateway/channel-health-monitor.ts#L158)
  - command: `nl -ba src/gateway/channel-health-monitor.ts | sed -n '127,168p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Existing automatic session rollover is inactivity-based:** `initSessionState()` only rolls to a new `sessionId` when the existing session is stale by configured reset policy (daily/idle) on the next inbound message. That is different from the report's case of continued inbound traffic against a silently broken session.
  - file: [src/auto-reply/reply/session.ts:433](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/auto-reply/reply/session.ts#L433)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '433,463p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Docs confirm the only built-in automatic session reset is idle expiry:** Session management docs describe daily reset and idle expiry creating a new `sessionId` after inactivity. They do not document any health-check-driven per-session auto-clear.
  - file: [docs/reference/session-management-compaction.md:132](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/docs/reference/session-management-compaction.md#L132)
