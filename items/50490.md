## Summary

Current `main` still appears to have the reported Feishu bug/design gap. `/activation mention` is accepted and persisted for group sessions, but Feishu group-message gating still relies only on config-derived `requireMention`/`groupPolicy`, so switching activation at runtime would not stop unmentioned group messages from reaching the agent.

## Evidence

- **`/activation` still reports success and only updates session state:** The shared activation handler parses `/activation`, writes `sessionEntry.groupActivation`, marks `groupActivationNeedsSystemIntro`, persists the session entry, and replies `Group activation set to ...`. There is no channel-specific guard here preventing Feishu from using the command.
  - file: [src/auto-reply/reply/commands-session.ts:201](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/auto-reply/reply/commands-session.ts#L201)
  - command: `nl -ba src/auto-reply/reply/commands-session.ts | sed -n '198,242p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Activation state is used for prompt behavior, not Feishu ingress filtering:** Group activation is consumed when building the group intro text, producing prompt lines like `Activation: trigger-only ...`. This supports the reporter comment that activation affects model instructions/session behavior rather than the Feishu routing gate.
  - file: [src/auto-reply/reply/groups.ts:237](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/auto-reply/reply/groups.ts#L237)

- **Feishu still hard-gates on `requireMention` before reply generation:** In the Feishu bot, group messages call `resolveFeishuReplyPolicy(...)`, then drop only when `requireMention && !ctx.mentionedBot`. This decision happens in the Feishu ingress path before the shared reply flow and does not reference session activation.
