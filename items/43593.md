## Summary

Current main already handles the LaunchAgent-specific gateway path/credential discovery that would cause the macOS menu bar widget to stay stuck on Idle/No Activity. The implementation is also present in release v2026.4.22.

## Evidence

- **Menu bar gateway log path now comes from the installed LaunchAgent plist:** `GatewayLaunchAgentManager.launchdGatewayLogPath()` prefers `StandardOutPath` / `StandardErrorPath` from the gateway LaunchAgent plist instead of blindly using the legacy `/tmp/openclaw/openclaw-gateway.log` fallback.

## Close Comment

Closing from a Codex review because current `main` already implements the LaunchAgent-aware behavior this report asks for. The macOS app now reads the installed gateway LaunchAgent plist for `StandardOutPath` / `StandardErrorPath` and uses that resolved path when refreshing gateway logs (`apps/macos/Sources/OpenClaw/Launchctl.swift:41`, `apps/macos/Sources/OpenClaw/GatewayLaunchAgentManager.swift:88`, `apps/macos/Sources/OpenClaw/GatewayProcessManager.swift:172`). It also falls back to LaunchAgent-provided gateway token/password when resolving the local control endpoint (`apps/macos/Sources/OpenClaw/GatewayEndpointStore.swift:43`, `apps/macos/Sources/OpenClaw/GatewayEndpointStore.swift:81`, `apps/macos/Sources/OpenClaw/GatewayEndpointStore.swift:149`), with coverage in `apps/macos/Tests/OpenClawIPCTests/GatewayEndpointStoreTests.swift:29`. This implementation is already present in release `v2026.4.22` at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`. If you can still reproduce on `v2026.4.22` or newer, please open a fresh issue with the generated LaunchAgent plist details, `openclaw gateway status --json`, and the exact menu bar state you see.
