## Summary

Keep open. Current `main` still lacks the requested Feishu outbound delivery-result logs, and the PR as described is not conservative to close out because it appears to misclassify some successful fallback direct sends as replies.

## Evidence

- **Current main still ends outbound handling at dispatch-summary logs:** `extensions/feishu/src/bot.ts:1237` logs only `dispatch complete (queuedFinal=..., replies=...)`, and `extensions/feishu/src/comment-handler.ts:250` does the same for drive comments. There is no per-delivery success log on current `main`.
  - file: [extensions/feishu/src/bot.ts:1237](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/feishu/src/bot.ts#L1237)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '1228,1242p'; nl -ba extensions/feishu/src/comment-handler.ts | sed -n '244,256p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Reply dispatcher on main sends text/card/media without success-side logging:** `extensions/feishu/src/reply-dispatcher.ts:322-360` and `:428-471` show chunked text/card sends and media sends, but those paths only await `sendMessageFeishu` / `sendStructuredCardFeishu` / `sendMediaFeishu`; they do not log delivery results or returned `messageId`s.
  - file: [extensions/feishu/src/reply-dispatcher.ts:322](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/feishu/src/reply-dispatcher.ts#L322)
  - command: `nl -ba extensions/feishu/src/reply-dispatcher.ts | sed -n '280,540p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Feishu send path can convert a reply attempt into a direct send:** `extensions/feishu/src/send.ts:126-179` falls back from `im.message.reply` to `im.message.create` for withdrawn/not-found non-thread reply targets. `extensions/feishu/src/send.reply-fallback.test.ts:231-252` covers that behavior explicitly.
  - file: [extensions/feishu/src/send.ts:126](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/feishu/src/send.ts#L126)
  - command: `nl -ba extensions/feishu/src/send.ts | sed -n '103,180p'; nl -ba extensions/feishu/src/send.reply-fallback.test.ts | sed -n '228,280p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **PR as provided infers log mode from reply intent, not actual send result:** The provided PR patch/context logs `mode: sendReplyToMessageId ? "reply" : "send"`. Given the fallback behavior in `send.ts`, that would log `reply` even when Feishu actually accepted a top-level direct send after reply fallback, which weakens the observability this PR is trying to add.
  - file: [extensions/feishu/src/reply-dispatcher.ts](https://github.com/openclaw/openclaw/blob/261abb654cef0a2bd3d35e9b4a19f6568163c1da/extensions/feishu/src/reply-dispatcher.ts)
