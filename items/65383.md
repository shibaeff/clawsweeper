## Summary

Keep open. Current `main` still lacks any iMessage default/named watcher dedupe, so the reported double-watcher problem does not look implemented already. At the same time, the PR shape changes `plugin.config.isEnabled` and `resolveOutboundTarget` in ways that affect broader surfaces than watcher startup, which matches the unresolved review concerns in the discussion.

## Evidence

- **Current main has no iMessage watcher-dedupe logic:** `resolveIMessageAccount` only computes `enabled` and `configured`, and `listEnabledIMessageAccounts` filters solely on `account.enabled`. There is no watcher-key comparison or default-account suppression on current `main`.

For iMessage startup, `startIMessageGatewayAccount` launches the provider from the account's `cliPath`/`dbPath`, so two configured accounts that share the same backend would still both start on current `main`.
