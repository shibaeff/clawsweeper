## Summary

Current `main` already contains a targeted Control UI fix for stale `chat.history` snapshots that would otherwise replace the visible transcript on re-render/final refresh, plus regression tests and docs. The symptom still reported on `v2026.4.23` is consistent with the pre-fix release behavior.

## Evidence

- **Current main preserves optimistic transcript tail during stale history refresh:** `loadChatHistory` now captures `previousMessages`, computes `preserveOptimisticTailMessages(...)`, and assigns that merged result instead of blindly replacing the UI transcript with the latest `chat.history` payload. That directly matches the report that a newly rendered answer disappears on the next re-render when the backend history briefly lags.
  - file: [ui/src/ui/controllers/chat.ts:175](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/ui/src/ui/controllers/chat.ts#L175)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '120,330p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Latest shipped release lacked the stale-snapshot preservation:** In `v2026.4.23`, `loadChatHistory` assigned `state.chatMessages = messages.filter(...)` directly, with no optimistic-tail preservation. That older behavior fits the issue report and explains why reporters still reproduced on `2026.4.23`.

That lines up with the stale-history replacement path fixed on `main`, not a backend session-loss bug.
