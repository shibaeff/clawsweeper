## Summary

Keep open. Current `main` still has the same split-source behavior: `openclaw nodes list` reads only `node.pair.list`, while `openclaw nodes status` reads `node.list`, and `node.list` marks a node as paired when either device pairing or node pairing exists. I found no evidence on `main` of a fix that would make the two commands agree for this case.

## Evidence

- **`nodes list` still uses `node.pair.list`:** `src/cli/nodes-cli/register.status.ts:325` calls `callGatewayCli("node.pair.list", ...)` and reports `Pending`/`Paired` from that payload, so `openclaw nodes list` is still driven by the node-pairing store.
  - file: [src/cli/nodes-cli/register.status.ts:325](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/cli/nodes-cli/register.status.ts#L325)
  - command: `nl -ba src/cli/nodes-cli/register.status.ts | sed -n '315,390p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **`nodes status` still uses `node.list`:** `src/cli/nodes-cli/register.status.ts:125` calls `callGatewayCli("node.list", ...)` and computes `Known / Paired / Connected` from that separate response, so `openclaw nodes status` is still sourced differently from `nodes list`.
  - file: [src/cli/nodes-cli/register.status.ts:125](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/cli/nodes-cli/register.status.ts#L125)
  - command: `nl -ba src/cli/nodes-cli/register.status.ts | sed -n '100,185p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Gateway methods still return different backing data:** `src/gateway/server-methods/nodes.ts:575` returns `node.pair.list` from `listNodePairing()`, while `src/gateway/server-methods/nodes.ts:707-717` builds `node.list` from `listDevicePairing()`, `listNodePairing()`, and live connected nodes. That preserves the discrepancy the issue describes.
  - file: [src/gateway/server-methods/nodes.ts:707](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/server-methods/nodes.ts#L707)

- **`node.list` still treats device pairing as paired:** `src/gateway/node-catalog.ts:118` sets `paired: Boolean(devicePairing ?? nodePairing)`, and `src/gateway/node-catalog.ts:143-156` includes node-role device pairings in the catalog. So `nodes status` can still show `paired: true` even when `node.pair.list` has no paired-node entry.
  - file: [src/gateway/node-catalog.ts:118](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/node-catalog.ts#L118)
