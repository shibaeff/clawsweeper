## Summary

Current `main` already mitigates the reported simultaneous Discord startup burst by delaying later Discord accounts, but it does not expose the requested `channels.discord.connectStagger` config surface. The issue is therefore only partially addressed and is not strong enough to close as implemented.

## Evidence

- **Discord startup is now staggered on main:** `extensions/discord/src/channel.ts:191` computes a per-account startup delay, and `extensions/discord/src/channel.ts:718` applies that delay before starting later Discord accounts. The delay is hardcoded via `DISCORD_ACCOUNT_STARTUP_STAGGER_MS = 10_000` at `extensions/discord/src/channel.ts:119`.
  - file: [extensions/discord/src/channel.ts:119](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/discord/src/channel.ts#L119)

  - file: [extensions/discord/src/channel.test.ts:348](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/discord/src/channel.test.ts#L348)

## Risks / Open Questions

- The hardcoded 10-second stagger likely resolves the original overload scenario for many multi-bot setups, so maintainers may decide the remaining ask is reduced to configurability rather than a startup-stability bug.
- Because the current implementation changes behavior unconditionally instead of adding the requested `channels.discord.connectStagger` knob, closing as `implemented_on_main` would overstate what shipped.
