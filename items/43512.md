## Summary

Keep open. Current `main` still routes Feishu control commands through the normal per-chat sequential queue except for `/stop`, so the PR's broader fix is not already implemented. The PR discussion also identified destructive-command buffer/queue invalidation work that is not present on `main`.

## Evidence

- **Control commands skip debounce but still hit the sequential queue:** `createFeishuMessageReceiveHandler()` marks control commands as non-debounced, but `onFlush()` still dispatches through `dispatchFeishuMessage()`, which always calls `enqueue(sequentialKey, task)`. That means immediate control commands still wait on the same chat lane unless the sequential key changes.
  - file: [extensions/feishu/src/monitor.message-handler.ts:185](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/feishu/src/monitor.message-handler.ts#L185)
  - command: `nl -ba extensions/feishu/src/monitor.message-handler.ts | sed -n '161,339p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **`/stop` is special-cased, `/status` is not:** `getFeishuSequentialKey()` only moves abort requests onto `:control` and `/btw` onto `:btw`; everything else stays on the base chat key.
  - file: [extensions/feishu/src/sequential-key.ts:19](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/feishu/src/sequential-key.ts#L19)
  - command: `nl -ba extensions/feishu/src/sequential-key.ts | sed -n '1,40p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Current tests assert `/status` remains on the normal chat lane:** `sequential-key.test.ts` expects `/status` to resolve to `feishu:default:oc_dm_chat`, while only `/stop` resolves to `feishu:default:oc_dm_chat:control`. That is direct evidence the broader queue-bypass request is still open on `main`.
  - file: [extensions/feishu/src/sequential-key.test.ts:29](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/feishu/src/sequential-key.test.ts#L29)

- **No buffer-drop API exists on main:** The current inbound debouncer only returns `{ enqueue, flushKey }`. The PR discussion converged on dropping buffered work for destructive commands, but that capability is not present in `main`'s debouncer implementation.
  - file: [src/auto-reply/inbound-debounce.ts:235](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/inbound-debounce.ts#L235)

- **PR discussion identified unresolved destructive-command semantics:** Review comments converged on dropping pending debounced text and invalidating already-queued chat work for destructive commands like `/stop` and `/new`. That discussion remains relevant because current `main` only adds a separate `/stop` sequential lane and does not implement the broader command handling proposed in the PR.

## Risks / Open Questions

- The PR branch targets the older `extensions/feishu/src/monitor.account.ts` path, while current `main` moved this logic into `extensions/feishu/src/monitor.message-handler.ts` and `extensions/feishu/src/sequential-key.ts`; it likely needs a rebase/rewrite rather than a straight merge.
- Current `main` appears to improve `/stop` specifically, but `/new`, `/reset`, `/status`, `/help`, and other control commands can still be serialized behind the base Feishu chat lane.
