## Summary

Keep open. Current `main` does not implement this PR’s Feishu control-command queue bypass completely: `/status` is still expected to use the normal per-chat lane, and the debounce/queued-work dropping added by the PR is absent.

## Evidence

- **Feishu still keeps `/status` on the normal chat lane:** `getFeishuSequentialKey()` only special-cases abort-like text and `/btw`; otherwise it returns the base per-chat key. The matching test explicitly expects `/status` to resolve to `feishu:default:oc_dm_chat`, not a control/bypass lane.
  - file: [extensions/feishu/src/sequential-key.ts:19](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/feishu/src/sequential-key.ts#L19)
  - command: `nl -ba extensions/feishu/src/sequential-key.ts | sed -n '1,220p'; nl -ba extensions/feishu/src/sequential-key.test.ts | sed -n '1,220p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Inbound Feishu messages still flow through the debouncer and then `dispatchFeishuMessage()`:** `createFeishuMessageReceiveHandler()` always calls `inboundDebouncer.enqueue(event)`. Control commands are excluded from debouncing via `shouldDebounce`, but `onFlush` still routes them to `dispatchFeishuMessage()`, which enqueues work on the sequential queue instead of directly invoking the handler the way PR #43512 proposed.
  - file: [extensions/feishu/src/monitor.message-handler.ts:183](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/feishu/src/monitor.message-handler.ts#L183)
  - command: `nl -ba extensions/feishu/src/monitor.message-handler.ts | sed -n '159,337p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **The destructive-command buffer drop from the PR is not on main:** `createInboundDebouncer()` currently returns only `{ enqueue, flushKey }`; there is no `dropKey()` API on `main`, so the PR’s logic for discarding buffered text before destructive commands is not present.
  - file: [src/auto-reply/inbound-debounce.ts:141](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/auto-reply/inbound-debounce.ts#L141)

- **The queued-work invalidation from the PR is also absent:** Feishu’s `createSequentialQueue()` is still a plain promise chain with no `clear()` or generation invalidation, so the PR’s later fix for cancelling already-queued chat work was not merged either.
  - file: [extensions/feishu/src/sequential-queue.ts:1](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/feishu/src/sequential-queue.ts#L1)

## Risks / Open Questions

- Closing as implemented would be incorrect: current `main` still leaves `/status` on the normal Feishu chat lane, so the PR’s stated broader control-command behavior is not present.
- Current `main` also lacks the PR’s `dropKey`/queue-clear protections, so destructive-command races with buffered or already-queued text remain materially different from the PR branch.
