## Summary

Keep open. Current `main` does not implement a trusted inbound `sender_name`/`sender_display_name` field for agents. The prompt builder explicitly keeps human names out of the trusted inbound metadata payload and instead serializes sender identity into user-role `untrusted metadata` blocks, so the request remains unimplemented on `main` at `028b6c9b13c960b6f59a7643aff4f6de04e07f71`.

## Evidence

- **Trusted inbound metadata omits sender identity fields:** `buildInboundMetaSystemPrompt` only emits stable routing fields (`schema`, `account_id`, `channel`, `provider`, `surface`, `chat_type`, `response_format`) and comments state that sender names belong in untrusted context blocks.
  - file: [src/auto-reply/reply/inbound-meta.ts:135](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/auto-reply/reply/inbound-meta.ts#L135)
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Current tests assert `sender_id` is absent from trusted payload:** The system-prompt tests verify `sender_id` is `undefined` in the trusted inbound metadata payload, so current `main` does not even keep `sender_id` there, let alone a trusted resolved sender name.
  - file: [src/auto-reply/reply/inbound-meta.test.ts:114](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/auto-reply/reply/inbound-meta.test.ts#L114)
