## Summary

Current main already exposes a public plugin seam for Telegram callback interception before agent fallback, so deterministic callback->webhook routing is better scoped as a ClawHub/plugin solution than bundled OpenClaw config.

## Risks / Open Questions

- Public docs for Telegram interactive handlers appear sparse, so the plugin path is real on main but not especially discoverable yet.
- This closes the bundled-config request, not the underlying user need; a packaged ClawHub/plugin example would still help users adopt the existing seam.

## Close Comment

The Telegram callback path already gives plugins first shot at callback payloads in `extensions/telegram/src/bot-handlers.runtime.ts:1381`, returns early when handled at `extensions/telegram/src/bot-handlers.runtime.ts:1427`, and only falls through to the synthetic agent-message path later at `extensions/telegram/src/bot-handlers.runtime.ts:1762`. Closing this cleanup item on that basis; if someone wants to package this as a ClawHub/plugin workflow, the needed callback hook already exists on current `main`.
