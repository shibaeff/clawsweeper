## Summary

Keep open. Current main still appears to send Telegram native approval traffic through a Gateway client that switches local connections to `wss://127.0.0.1:<port>` when `gateway.tls.enabled=true`, but that approval-client path does not pass a TLS fingerprint or other explicit trust material into `GatewayClient`. Since `GatewayClient` only relaxes verification when `tlsFingerprint` is set, the reported self-signed-certificate failure and 1-second retry loop remain plausible on current main rather than already fixed.

## Evidence

  - file: [src/infra/approval-handler-bootstrap.ts:17](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/infra/approval-handler-bootstrap.ts#L17)

- **Local TLS changes loopback URL to WSS:** Gateway connection resolution uses `wss` for the default local loopback URL whenever `config.gateway.tls.enabled === true`, so first-party local clients will target `wss://127.0.0.1:<port>` under that config.
  - file: [src/gateway/connection-details.ts:36](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/gateway/connection-details.ts#L36)

- **Approval client omits TLS fingerprint:** `createOperatorApprovalsGatewayClient()` resolves the gateway URL/auth and constructs `GatewayClient` with URL, token, and password, but does not pass `tlsFingerprint` or CA-related options. This is the client path used by native approval handling.

  - file: [src/gateway/client.ts:225](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/gateway/client.ts#L225)
