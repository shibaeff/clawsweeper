## Summary

Current `main` already implements the requested self-compaction path via the gateway session RPC: `sessions.compact` performs manual compaction on a target session, interrupts an active run first, is exposed in the gateway method list/protocol docs, and has regression coverage proving it compacts `agent:main:main` and updates session state. This is not a `session_status(action:"compact")` tool action, but it satisfies the issue’s proposed gateway-RPC solution.

## Risks / Open Questions

- There is still no dedicated `session_status(action:"compact")` surface; the supported implementation on `main` is the gateway RPC/CLI path (`sessions.compact`).
- `sessions.compact` is classified as an admin-scoped gateway method, so the practical agent path is the gateway call route rather than a lightweight session tool.

## Close Comment

Closing from a ClawSweeper/Codex review because current `main` already implements the requested capability, just under the gateway session RPC rather than `session_status`. `src/gateway/server-methods/sessions.ts:1492` adds `sessions.compact`; it interrupts an active run first at `src/gateway/server-methods/sessions.ts:1552` and then calls manual compaction with `trigger: "manual"` at `src/gateway/server-methods/sessions.ts:1571`. There is also regression coverage in `src/gateway/server.sessions.gateway-server-sessions-a.test.ts:1515`, which proves `sessions.compact` compacts `agent:main:main` and updates `compactionCount`. So the issue’s proposed gateway-RPC path is already present on `main`, even though it landed as `sessions.compact` instead of a `session_status(action: "compact")` tool action.
