## Summary

Current `main` still reinjects the reported Discord thread metadata/context on follow-up turns, so there is not enough evidence to close this as fixed or obsolete.

## Evidence

- **Discord inbound context still sets volatile per-turn fields on every thread message:** `processDiscordMessage` always populates `InboundHistory`, `MessageSid`, `ReplyToId`, `ThreadStarterBody`, and `Timestamp` into the finalized inbound context for Discord messages; there is no first-turn/session guard around those assignments.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:477](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/discord/src/monitor/message-handler.process.ts#L477)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.process.ts | sed -n '430,540p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Discord thread starter is still fetched for any thread message:** When `threadChannel` is present, the code resolves the Discord thread starter and stores it in `threadStarterBody` without checking whether this is the first turn in the session.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:383](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/discord/src/monitor/message-handler.process.ts#L383)
