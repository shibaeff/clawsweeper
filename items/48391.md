## Summary

Keep open. Current `main` does not implement the PR’s cron read-model changes or richer `cron.run` acknowledgment, and the review thread surfaced an unresolved semantics gap around reporting queued work as `running` before the job actually acquires the cron lane.

## Evidence

- **`cron.list` still returns raw persisted jobs:** `list()` only sorts and returns stored jobs; it does not project any read-only running/completed fields such as `currentStatus` or `lastCompleted*`.
  - file: [src/cron/service/ops.ts:161](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/cron/service/ops.ts#L161)
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Manual runs keep prior status fields in-place while marking a job running:** `prepareManualRun()` sets `runningAtMs` and clears `lastError`, but it does not preserve `lastRunStatus`/`lastStatus` into separate completed-status fields. That matches the PR’s reported ambiguity: a running job can still expose the previous completed status.
  - file: [src/cron/service/ops.ts:565](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/cron/service/ops.ts#L565)
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **`cron.run` ack is still the old generic enqueue shape:** The public run result type still models enqueue responses as only `{ ok: true; enqueued: true; runId: string }`, and `enqueueRun()` returns exactly that shape on current `main`.
  - file: [src/cron/service/state.ts:165](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/cron/service/state.ts#L165)
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Protocol schema still has no separate read-only cron state schema:** `CronJobStateSchema` is the only cron job state schema here and `CronJobPatchSchema` still uses `Type.Partial(CronJobStateSchema)`. Current `main` therefore does not contain the PR’s split between writable persisted state and read-only derived state.
  - file: [src/gateway/protocol/schema/cron.ts:231](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/gateway/protocol/schema/cron.ts#L231)
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Gateway tests still assert the old enqueue response:** Current gateway coverage still expects `cron.run` to return only `{ ok: true, enqueued: true, runId }`, which confirms the richer in-progress ack from this PR is not present on `main`.
  - file: [src/gateway/server.cron.test.ts:263](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/gateway/server.cron.test.ts#L263)
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)

## Risks / Open Questions

- The PR’s review thread identified a real semantics question: returning `currentStatus: "running"` from `enqueueRun()` would be optimistic before the queued job actually sets `runningAtMs`, so the proposed API may still need refinement even though the underlying problem remains unsolved on `main`.
