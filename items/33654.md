## Summary

Current `main` already surfaces model fallback events to users: fallback state is persisted for `/status`, lifecycle fallback events are emitted, the Control UI shows a visible fallback indicator, and verbose chat replies include inline `Model Fallback` notices. The changelog records this behavior as shipped in `2026.2.21`.

## Evidence

- **Fallback state persisted for status/reporting:** `runReplyAgent` records `fallbackNoticeSelectedModel`, `fallbackNoticeActiveModel`, and `fallbackNoticeReason` into the session store when the active runtime model differs from the selected model.
  - file: [src/auto-reply/reply/agent-runner.ts:1286](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/auto-reply/reply/agent-runner.ts#L1286)
  - command: `nl -ba src/auto-reply/reply/agent-runner.ts | sed -n '1278,1315p'`
- **Lifecycle event emitted on fallback transition:** On fallback, `runReplyAgent` emits a lifecycle event with `phase: "fallback"`, selected/active model refs, reason summary, and attempt summaries; it also emits `fallback_cleared` when the runtime returns to the selected model.
  - file: [src/auto-reply/reply/agent-runner.ts:1493](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/auto-reply/reply/agent-runner.ts#L1493)
  - command: `nl -ba src/auto-reply/reply/agent-runner.ts | sed -n '1460,1555p'`
