## Summary

Current `main` already handles same-provider session model overrides during fallback resolution, so the reported 429 followup retry path no longer drops back to `agents.defaults.model.fallbacks` just because the session model differs from the configured primary.

## Evidence

- **Current fallback resolution keeps same-provider session overrides in the fallback chain:** `resolveFallbackCandidates()` now always adds the requested provider/model first and, when the session override stays on the same provider, returns the configured fallback chain instead of bailing out when the model differs from the configured primary.
  - file: [src/agents/model-fallback.ts:397](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/agents/model-fallback.ts#L397)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '397,471p'`
- **Regression tests cover the reported case:** Tests now assert that a session model different from the configured primary still uses fallback candidates on retry, including same-provider model-version differences.
  - file: [src/agents/model-fallback.test.ts:1288](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/agents/model-fallback.test.ts#L1288)
  - command: `nl -ba src/agents/model-fallback.test.ts | sed -n '1288,1406p'`
- **Followup retry path still routes through the shared fallback helper with the session run model:** The followup execution path calls `runWithModelFallback()` using `followupRun.run.provider` and `followupRun.run.model`, so the fixed candidate resolution applies to followup retries.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:859](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/auto-reply/reply/agent-runner-execution.ts#L859)

  - file: [src/auto-reply/reply/followup-runner.ts:198](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/auto-reply/reply/followup-runner.ts#L198)
