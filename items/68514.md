## Summary

Keep open. Current `main` and the latest shipped release still pass Feishu post markdown through unchanged, so the reported URL-boundary fix is not implemented on main. The PR also still has an unresolved correctness concern from review: its markdown-link masking regex does not handle nested brackets, which can rewrite already-linked content.

## Evidence

  - file: [extensions/feishu/src/send.ts:432](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/feishu/src/send.ts#L432)

- **Latest release also lacks the fix:** The same `buildFeishuPostMessagePayload` implementation in release `v2026.4.23` also writes `messageText` directly, so this behavior was not already shipped.
  - file: [extensions/feishu/src/send.ts:432](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/extensions/feishu/src/send.ts#L432)

- **PR still has an unresolved correctness concern:** The review discussion on PR #68514 includes a follow-up Codex finding after the force-push: the masking regex for existing markdown links does not support nested brackets (for example `[foo [bar]](https://example.com/a_b)`), so the bare-URL rewrite can mutate already-linked content. That means the PR is not a clear candidate for conservative auto-close even though the main bug remains plausible.

## Risks / Open Questions

- Merging the PR as-is may regress valid existing markdown links with nested brackets unless the masking logic is tightened.
