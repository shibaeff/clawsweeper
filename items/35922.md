## Summary

Current `main` already has the version-resolution path this PR is trying to add, and the branch’s other two fixes are also already present on `main` in newer form.

## Evidence

- **`openclaw --version` already routes through the shared resolver:** `src/entry.version-fast-path.ts:35-49` loads `./version.js` for root `--version` and prints that resolved version, so the CLI no longer depends on ad hoc path probing from unrelated modules.
  - file: [src/entry.version-fast-path.ts:35](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/entry.version-fast-path.ts#L35)
- **Current resolver already falls back to bundled build metadata:** `src/version.ts:7-18` defines package/build-info candidates, and `src/version.ts:68-87` prefers package metadata, then `build-info.json`, then bundled/env fallbacks. This already covers packaged installs without needing the PR’s extra `../../../../package.json` probes.
  - file: [src/version.ts:7](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/version.ts#L7)
- **Release packaging always writes and ships `dist/build-info.json`:** `scripts/build-all.mjs:72-86` always runs `scripts/write-build-info.ts`, that script writes `dist/build-info.json` from package version metadata, and `package.json:23-30` includes `dist/` in the published package files.
  - file: [scripts/build-all.mjs:72](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/scripts/build-all.mjs#L72)
- **Homebrew layout does not need deeper package.json probes from `dist/version.js`:** A direct path-resolution check against a Homebrew-style install root showed `../package.json` from `.../openclaw/dist/version.js` already resolves to `.../openclaw/package.json`, while `./build-info.json` resolves to `.../openclaw/dist/build-info.json`.
  - command: `node - <<'EOF'
const path = require('node:path');
const base = '/opt/homebrew/Cellar/openclaw/2026.4.22/libexec/lib/node_modules/openclaw/dist/version.js';
for (const c of ['../package.json','../../package.json','../../../package.json','./package.json','./build-info.json']) {
  console.log(c, '=>', path.normalize(path.join(path.dirname(base), c)));
}
EOF`
- **The branch’s other commits are already represented on `main`:** `src/config/types.discord.ts:61-64` already includes `autoThread?: boolean`, and BlueBubbles debounce now normalizes message text before trimming in `extensions/bluebubbles/src/monitor-debounce.ts:105-112`, covering the null-text case in a newer implementation.
  - file: [src/config/types.discord.ts:61](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/config/types.discord.ts#L61)

## Close Comment

Closing from a Codex review because current `main` already covers the version-resolution problem this PR was aiming at. `openclaw --version` now goes through `src/entry.version-fast-path.ts:35-49`, which imports the shared resolver in `src/version.ts`; that resolver already falls back from `package.json` to `build-info.json` (`src/version.ts:7-18`, `src/version.ts:68-87`), and release packaging always writes and ships `dist/build-info.json` (`scripts/build-all.mjs:72-86`, `scripts/write-build-info.ts:37-47`, `package.json:23-30`). I also checked the Homebrew-style path shape directly: from `.../openclaw/dist/version.js`, the existing `../package.json` candidate already lands on the package root, so the extra deep parent probes in this PR are not needed.
