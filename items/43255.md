## Summary

Current main already implements queued manual cron execution with persisted finished-run history, and that implementation is present in the latest release tag v2026.4.22.

## Evidence

- **Gateway `cron.run` uses the queued manual-run path:** `src/gateway/server-methods/cron.ts:357` routes `cron.run` to `context.cron.enqueueRun(...)`, returning the enqueue result instead of claiming synchronous completion.
  - file: [src/gateway/server-methods/cron.ts:357](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/server-methods/cron.ts#L357)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Queued manual runs actually execute and persist completion state:** `src/cron/service/ops.ts:548-731` shows the flow: preflight, persist `runningAtMs`, emit `started`, execute outside the lock, emit `finished`, persist updated job state, and return `{ ok: true, enqueued: true, runId }` from `enqueueRun`.
  - file: [src/cron/service/ops.ts:548](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cron/service/ops.ts#L548)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Finished cron runs are written to run-log files consumed by `cron.runs`:** `src/gateway/server-cron.ts:549-577` appends each `finished` cron event to `cron/runs/<jobId>.jsonl`, including status, summary, delivery status, session info, and timing.
  - file: [src/gateway/server-cron.ts:549](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/server-cron.ts#L549)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Integration test covers the reported symptom directly:** `src/gateway/server.cron.test.ts:814-905` force-runs a job, waits for the `finished` event, then asserts `cron.runs` returns an entry for that job with the expected summary and delivery status. `src/gateway/server.cron.test.ts:949-1002` also verifies `cron.run` returns immediately while background execution still completes.
  - file: [src/gateway/server.cron.test.ts:814](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/server.cron.test.ts#L814)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **This behavior shipped in the latest release:** The latest release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` already contains the same queued-manual-run code and docs note that `openclaw cron run` returns `{ ok: true, enqueued: true, runId }` and `cron runs` should be used to follow the outcome (`docs/cli/cron.md:41`).
  - file: [docs/cli/cron.md:41](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/docs/cli/cron.md#L41)

## Close Comment

Closing as implemented on current `main` and already shipped in `v2026.4.22` (release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). Codex review of the current code shows `cron.run` goes through the queued manual-run path (`src/gateway/server-methods/cron.ts:357`), that path persists and executes the run (`src/cron/service/ops.ts:548-731`), and finished runs are appended to the per-job run log used by `cron.runs` (`src/gateway/server-cron.ts:549-577`). The integration test at `src/gateway/server.cron.test.ts:814-905` specifically force-runs a job and then verifies that `cron.runs` contains the finished entry, which matches the behavior requested here. If you still see the old enqueue-without-execution behavior on `v2026.4.22` or newer, please open a fresh issue with exact version, job JSON, gateway logs around the `cron.run` call, and whether the job is `main` or `isolated`.
