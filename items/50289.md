## Summary

Keep open. Current main no longer matches the report's per-message config-cache-miss theory, but the large `commands.ownerAllowFrom` performance concern is not clearly fixed: command authorization still rebuilds filtered owner lists and does linear membership checks on each authorization call.

## Evidence

- **Owner allowlist is still rebuilt per authorization call:** `resolveOwnerAuthorizationState()` calls `resolveOwnerAllowFromList()` for both config and context owner lists. `resolveOwnerAllowFromList()` still loops every entry, trims/slices strings, filters by provider prefix, and then normalizes the resulting array.
  - file: [src/auto-reply/command-auth.ts:270](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/auto-reply/command-auth.ts#L270)

  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Authorization still uses linear `includes()` scans:** `resolveCommandAuthorization()` still checks `ownerState.ownerList.includes(candidate)` and `ownerState.ownerCandidatesForCommands.includes(candidate)`. The command allowlist path also uses `commandsAllowFromList.includes(candidate)`, so large lists remain O(n) on lookup.
  - file: [src/auto-reply/command-auth.ts:679](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/auto-reply/command-auth.ts#L679)

That means the owner-list work is still in an active message-processing path, not an offline/admin-only path.
