## Summary

Current `main` already refreshes managed gateway service metadata during `openclaw update` by rerunning the gateway install path before restart, and the same implementation is present in release `v2026.4.23`.

## Evidence

- **Update flow refreshes managed service before restart:** `updateCommand` enables service-env refresh when a managed gateway service is loaded, and `maybeRestartService` invokes `refreshGatewayServiceEnv` before the restart step.
  - file: [src/cli/update-cli/update-command.ts:1382](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cli/update-cli/update-command.ts#L1382)
  - command: `nl -ba src/cli/update-cli/update-command.ts | sed -n '258,286p;742,776p;1382,1422p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Refresh path is equivalent to `openclaw gateway install --force`:** `refreshGatewayServiceEnv` runs `gateway install --force` from the updated install root when possible, otherwise falls back to `runDaemonInstall({ force: true })`.
  - file: [src/cli/update-cli/update-command.ts:258](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cli/update-cli/update-command.ts#L258)

- **Systemd install rewrites and reloads the unit:** The systemd installer overwrites the unit file, then runs `systemctl --user daemon-reload`, `enable`, and `restart`, which is the metadata refresh the issue requested.
  - file: [src/daemon/systemd.ts:432](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/daemon/systemd.ts#L432)
