## Summary

Keep open. Current `main` appears to have reduced Control UI polling pressure, but the core `sessions.list` path still eagerly materializes every session and performs per-session subagent/store work before applying `limit`, so the reported scale problem is not safely obsolete.

## Evidence

- **`sessions.list` still loads the full combined store on every request:** The gateway handler loads the combined session store and immediately passes it to `listSessionsFromStore` for each `sessions.list` call.
  - file: [src/gateway/server-methods/sessions.ts:579](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/gateway/server-methods/sessions.ts#L579)
  - command: `nl -ba src/gateway/server-methods/sessions.ts | sed -n '575,595p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **`limit` is applied only after row construction and sorting:** `listSessionsFromStore` filters entries, maps every remaining entry through `buildGatewaySessionRow`, sorts the full row list, and only then slices by `opts.limit`. That means `limit` does not bound the expensive work.
  - file: [src/gateway/session-utils.ts:1483](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/gateway/session-utils.ts#L1483)
  - command: `nl -ba src/gateway/session-utils.ts | sed -n '1461,1565p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **Each session row does additional per-session traversal:** `buildGatewaySessionRow` resolves subagent state and child sessions for every row. `resolveChildSessionKeys` iterates the controller runs and then scans `Object.entries(store)`, which compounds listing cost as session count grows.
  - file: [src/gateway/session-utils.ts:1273](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/gateway/session-utils.ts#L1273)
  - command: `nl -ba src/gateway/session-utils.ts | sed -n '1235,1565p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **Subagent display lookup itself is linear over in-memory runs:** `getSessionDisplaySubagentRunByChildSessionKey` walks `subagentRuns.values()` to find the latest match for one child session key, so repeated row building adds more scale-sensitive work under heavy subagent usage.
  - file: [src/agents/subagent-registry-read.ts:50](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/agents/subagent-registry-read.ts#L50)
  - command: `nl -ba src/agents/subagent-registry-read.ts | sed -n '20,80p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **Control UI polling pressure seems partially improved, not eliminated issue-wide:** Current UI polling code only sets intervals for nodes/logs/debug. Sessions now subscribe to `sessions.subscribe` and refresh on `sessions.changed`, which suggests the old always-on sessions polling concern was reduced, but this does not fix the core listing path above.
  - file: [ui/src/ui/app-polling.ts:15](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/ui/src/ui/app-polling.ts#L15)
