## Summary

Current main does not appear to implement Mattermost wake-on-reply-to-bot behavior. The Mattermost mention gate still requires an explicit mention in `oncall` mode, while Telegram has separate reply-to-bot implicit-mention logic.

## Evidence

- **Mattermost gate still enforces mention-only `oncall` behavior:** `evaluateMattermostMentionGate()` only treats actual mentions, authorized control commands, or `onchar` triggers as wake conditions. `threadRootId` is accepted in the input type but never used to bypass mention gating.
  - file: [extensions/mattermost/src/mattermost/monitor-gating.ts:25](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/mattermost/src/mattermost/monitor-gating.ts#L25)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor-gating.ts | sed -n '1,140p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Inbound Mattermost path passes thread metadata but still drops on missing mention:** The websocket handler computes `threadRootId` from `post.root_id`, passes it into `evaluateMattermostMentionGate()`, and immediately returns on `dropReason === "missing-mention"`. There is no fetch/compare of the parent/root post author before that drop.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:1367](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/mattermost/src/mattermost/monitor.ts#L1367)
  - command: `sed -n '1348,1460p' extensions/mattermost/src/mattermost/monitor.ts | nl -ba -v1348`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Tests still model unmentioned thread replies as allowed only in `onmessage`:** Current tests explicitly allow unmentioned thread replies in `chatmode: "onmessage"`, and separately assert that `chatmode: "oncall"` rejects unmentioned channel posts with `dropReason: "missing-mention"`. There is no corresponding oncall reply-to-bot allowance.
  - file: [extensions/mattermost/src/mattermost/monitor.test.ts:123](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/mattermost/src/mattermost/monitor.test.ts#L123)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor.test.ts | sed -n '95,160p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Docs still describe Mattermost `oncall` as mention-only:** The user-facing docs say `oncall` responds only when the bot is `@mentioned` in channels, with no reply-to-bot exception documented.
  - file: [docs/channels/mattermost.md:116](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/docs/channels/mattermost.md#L116)
  - command: `nl -ba docs/channels/mattermost.md | sed -n '114,145p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Telegram has explicit reply-to-bot implicit mention logic:** Telegram computes `replyToBotMessage` and feeds `reply_to_bot` into its implicit-mention policy, which is the parity behavior requested by this issue and is absent from Mattermost.
