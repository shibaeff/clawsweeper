## Summary

Issue #43018 is implemented on current `main` and in release `v2026.4.22`: custom providers configured with `api: "openai-responses"` now use the Responses transport with the configured `baseUrl`, and embedded/session-managed runs keep proxy/custom endpoints off the hardcoded native OpenAI websocket path.

## Close Comment

Closing from a Codex review. Current `main` already implements this well enough, and the same behavior is present in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The Responses transport now uses the configured provider `baseUrl` and calls `client.responses.create(...)` instead of routing custom `api: "openai-responses"` providers through chat/completions (`src/agents/openai-transport-stream.ts:639`). Embedded/session-managed runs also re-resolve the provider stream each turn and keep non-public/custom `baseUrl` values off the hardcoded native OpenAI websocket path (`src/agents/pi-embedded-runner/run/attempt.ts:1302`, `src/agents/pi-embedded-runner/run/attempt.thread-helpers.ts:41`). The docs and regression tests cover the supported custom-provider shape and proxied Responses routes as well (`docs/gateway/config-tools.md:356`, `src/plugins/provider-runtime.test.ts:1448`, `src/agents/openai-ws-stream.test.ts:1877`). If you are still seeing a remaining proxy-specific or payload-specific Responses bug on `v2026.4.22` or newer, please open a new focused issue with the exact provider config, outbound path/log evidence, and whether the failure is transport selection vs. payload/stream parsing.
