## Summary

Current `main` already treats Feishu channel config edits as channel-only hot reloads and contains launchd restart logic/tests that avoid the bootout-driven SIGTERM/restart behavior described in the report.

## Evidence

- **Feishu config is declared hot-reloadable:** The Feishu plugin registers `channels.feishu` as a reload prefix, so config edits are handled as channel reload work rather than a gateway restart trigger.
  - file: [extensions/feishu/src/channel.ts:603](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/feishu/src/channel.ts#L603)
  - command: `nl -ba extensions/feishu/src/channel.ts | sed -n '590,610p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Gateway reload planner maps channel config to hot reload:** `buildGatewayReloadPlan` turns channel/plugin reload prefixes into `restart-channel:*` actions, while only `gateway`, `plugins`, `discovery`, and `canvasHost` paths force a full gateway restart.
  - file: [src/gateway/config-reload-plan.ts:126](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/config-reload-plan.ts#L126)
  - command: `nl -ba src/gateway/config-reload-plan.ts | sed -n '117,176p'`

- **Hot reload handler restarts only the affected channel:** When `plan.restartChannels` is set, the reload handler logs `restarting <channel> channel`, stops/starts that channel, and then logs `config hot reload applied`; it does not request a gateway restart on this path.
  - file: [src/gateway/server-reload-handlers.ts:166](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/server-reload-handlers.ts#L166)

## Close Comment

Closing from a Codex review because current `main` already implements the behavior this issue asks for. Feishu config changes are registered as hot-reloadable channel config in `extensions/feishu/src/channel.ts:603`, the gateway reload planner routes those paths to channel-only reload actions in `src/gateway/config-reload-plan.ts:126`, and the reload handler restarts just the affected channel and then logs `config hot reload applied` in `src/gateway/server-reload-handlers.ts:166` rather than scheduling a gateway restart.
