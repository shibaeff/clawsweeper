## Summary

Keep open. Current `main` has partial Slack conversation-binding support for existing bound threads, but it does not provide the requested Slack `channels.slack.threadBindings` config surface or ACP thread-spawn parity/documentation that the issue asks for.

## Evidence

- **Slack config schema still lacks `threadBindings`:** `SlackAccountSchema` includes `thread` and other Slack keys, but no `threadBindings` object or `spawnAcpSessions` / `spawnSubagentSessions` fields, so `channels.slack.threadBindings.*` is not part of the current validated Slack config surface.
  - file: [src/config/zod-schema.providers-core.ts:897](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/config/zod-schema.providers-core.ts#L897)
  - command: `nl -ba src/config/zod-schema.providers-core.ts | sed -n '897,980p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Slack plugin only gets generic current-conversation binding support:** `createChatChannelPlugin()` injects only `conversationBindings.supportsCurrentConversationBinding: true` unless a plugin supplies overrides. The Slack plugin instantiation shown here does not add a Slack-specific `conversationBindings` override, so there is no evidence of Slack child-thread placement parity like Discord/Matrix-style `defaultTopLevelPlacement: "child"`.
  - file: [src/plugin-sdk/core.ts:728](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/plugin-sdk/core.ts#L728)
  - command: `nl -ba src/plugin-sdk/core.ts | sed -n '728,748p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Thread-binding policy defaults to `current` without a plugin override:** Thread-binding spawn policy derives top-level placement from `conversationBindings.defaultTopLevelPlacement`; if absent it falls back to `"current"`. Combined with the Slack plugin evidence above, current main does not show the requested Slack-specific ACP thread-spawn model/parity surface.
  - file: [src/channels/thread-bindings-policy.ts:62](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/channels/thread-bindings-policy.ts#L62)
  - command: `nl -ba src/channels/thread-bindings-policy.ts | sed -n '62,76p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Slack does route already-bound thread replies:** Slack message preparation resolves runtime conversation bindings for thread replies using `(channel=slack, conversationId=thread_ts, parentConversationId=baseConversationId)`, so part of the requested behavior already exists once a binding record is present.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:82](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L82)
