## Summary

Keep open. Current `main` does not contain the KimiCodingPlan malformed-stream workaround; the checked-out code still only surfaces the out-of-order `message_start`/`message_stop` failure as a generic invalid streaming response, and the specific synthetic `message_end` handling appears only in the still-open PR #58412 patch.

## Evidence

- **Kimi Coding still uses Anthropic streaming path with only wrapper-level tweaks:** The bundled Kimi provider is configured as `api: "anthropic-messages"`, and its stream wrapper only rewrites tagged tool-call text and thinking payloads. There is no Kimi-specific recovery for malformed `message_start`/`message_stop` ordering in current `main`.
  - file: [extensions/kimi-coding/provider-catalog.ts:20](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/kimi-coding/provider-catalog.ts#L20)

- **Current message handlers lack the synthetic-close state added by the proposed fix:** `handleMessageStart` on current `main` only resets assistant state and fires typing start. `EmbeddedPiSubscribeState` also has no `inAssistantMessage` or `staleSyntheticMessageEndTexts` fields, and `handleMessageEnd` has no synthetic/stale-end skip path.
