## Summary

Current `main` still lacks a live inbound-message path for TUI `--session` mode. The TUI listens only for `chat`, `chat.side_result`, and `agent` websocket events, while transcript/inbound session updates are emitted as `session.message`; those user messages are only rendered when history is reloaded. The manual-refresh duplication note looks partially obsolete because history reload now clears and rebuilds the chat log before replaying messages.

## Evidence

- **TUI event loop ignores transcript events:** `client.onEvent` only routes `chat`, `chat.side_result`, and `agent`. There is no `session.message` handling in the TUI event loop.
  - file: [src/tui/tui.ts:1034](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/tui/tui.ts#L1034)
  - command: `nl -ba src/tui/tui.ts | sed -n '1034,1043p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **No TUI subscription for live session-message events:** A repo search in `src/tui` found no `session.message` or `sessions.messages.subscribe` usage, so the TUI is not wired to the gateway's transcript-event channel.
  - command: `rg -n "session\.message|sessions\.messages\.subscribe" src/tui`
