## Summary

Keep open. Current `main` still validates xhigh support through active plugin-registry state, then falls back and persists the downgraded thinking level in `agent-command`. I did not find a main-branch fix or regression test that specifically covers the reported cold-start `thinkingDefault: "xhigh"` downgrade for `openai-codex/gpt-5.4`.

## Evidence

- **Registry-dependent xhigh lookup:** Provider thinking hooks are resolved only from the global active plugin registry. If that registry is not populated yet, the generic thinking helpers see no provider-specific xhigh support.
  - file: [src/plugins/provider-thinking.ts:42](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/plugins/provider-thinking.ts#L42)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **openai-codex exposes xhigh through provider profile:** `openai-codex` advertises `xhigh` for matching model ids via `resolveThinkingProfile`; `gpt-5.4` is covered by that provider-owned profile path.
  - file: [extensions/openai/openai-codex-provider.ts:493](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/openai/openai-codex-provider.ts#L493)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Agent startup still downgrades and persists unsupported thinking:** `agent-command` resolves the default thinking level, checks support with `isThinkingLevelSupported({ provider, model, level })`, computes a fallback when unsupported, and writes that fallback back into the session entry when the stored thinking level matches the pre-fallback value.
  - file: [src/agents/agent-command.ts:784](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/agents/agent-command.ts#L784)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Generic thinking support check still depends on provider hooks:** `resolveThinkingProfile()` still builds provider-specific thinking levels from provider hooks first and falls back to the base profile otherwise; the later support check in `agent-command` does not pass catalog data into that lookup.
  - file: [src/auto-reply/thinking.ts:133](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/thinking.ts#L133)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **No focused regression test for this path:** The current model-selection tests cover hydrating runtime catalog metadata when `reasoning` is missing, but they stop at a generic default of `"medium"`; they do not cover configured `thinkingDefault: "xhigh"` plus startup support validation and session persistence.
  - file: [src/auto-reply/reply/model-selection.test.ts:112](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/reply/model-selection.test.ts#L112)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)

## Risks / Open Questions

- This review is based on code inspection, not a live runtime reproduction; `pnpm` is unavailable in this environment, so I did not execute a session-start repro.
- If some other startup path now guarantees the active provider registry before `agent-command` runs, that guarantee is not obvious from the inspected files and is not covered by a targeted regression test.
