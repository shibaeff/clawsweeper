## Summary

Current `main` does not implement the requested Telegram persisted-state diagnosis/recovery flow. Telegram status warnings only cover group/privacy/membership conditions, read-only session inspection silently treats malformed JSON as an empty store, and there is no Telegram-specific reset command. There is partial runtime hardening for invalid persisted Telegram update-offset state, but that is not the status/doctor recovery UX requested here.

## Evidence

- **Telegram status collector has no persisted-state checks:** `collectTelegramStatusIssues` only reports privacy-mode, wildcard/unresolved group config, and unreachable-group runtime warnings; it does not inspect Telegram state files, JSON validity, file size, or recovery actions.
  - file: [extensions/telegram/src/status-issues.ts:83](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/telegram/src/status-issues.ts#L83)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Status UI only surfaces plugin issues and missing configured files:** `status --all` derives channel warnings from plugin `collectStatusIssues`, missing configured paths, credential availability, and link state. There is no branch for Telegram persisted session/update-offset corruption or reset guidance.
  - file: [src/commands/status-all/channels.ts:240](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/commands/status-all/channels.ts#L240)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Read-only session status hides malformed JSON:** `readSessionStoreReadOnly` returns `{}` for empty, malformed, or schema-mismatched JSON, and `status.summary` uses that helper directly. That means status will silently treat a bad session store as empty instead of diagnosing it.
  - file: [src/config/sessions/store-read.ts:10](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/config/sessions/store-read.ts#L10)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Status summary reads session stores without surfacing parse errors:** `getStatusSummary` caches `readSessionStoreReadOnly(storePath)` and then builds rows from the returned object; no warning path is emitted when the store was unreadable or malformed.
  - file: [src/commands/status.summary.ts:176](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/commands/status.summary.ts#L176)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Telegram runtime now ignores invalid persisted update-offset state:** `readTelegramUpdateOffset` returns `null` on invalid JSON/state, and the polling monitor proceeds with `lastUpdateId = null` rather than failing startup. This mitigates one class of bad persisted Telegram state, but it is silent and does not add operator-facing recovery guidance.
  - file: [extensions/telegram/src/update-offset-store.ts:48](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/telegram/src/update-offset-store.ts#L48)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **No Telegram reset-session command exists on main:** Repo search found only ACP `--reset-session`; no `openclaw telegram reset-session` or comparable Telegram recovery command/docs were present.
  - file: [docs/cli/status.md:13](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/docs/cli/status.md#L13)

  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)

## Risks / Open Questions

- The exact persisted JSON artifact from the report is not identifiable from the issue alone; some variants may now be mitigated by the update-offset hardening.
- Malformed generic session-store JSON is still hidden from status by silent `{}` fallback, so operator diagnosis remains weak even if runtime avoids a hard failure.
- No first-class Telegram recovery/reset workflow was found, so manual cleanup still appears to be the only path for state-artifact recovery.
