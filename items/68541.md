## Summary

Keep open. Current `main` and the latest release still apply symmetric jitter after deriving a `retryAfterMs`-based delay, so the server-supplied lower bound is not preserved. The issue also has an active, focused, mergeable follow-up PR (#68543), while the alternate PR (#68600) was explicitly closed as superseded by that work rather than merged.

## Evidence

- **Current main still undercuts `retryAfterMs`:** `applyJitter` still uses symmetric jitter (`(fraction * 2 - 1) * jitter`), and `retryAsync` still calls it after computing a `retryAfterMs`-based `baseDelay`, then clamps only to `minDelayMs`/`maxDelayMs`. There is no floor re-applied for the server hint.
