## Summary

Keep open. Current main clearly has a claim-check path for gateway chat attachments, but I did not find strong evidence that inbound Discord/channel transcript persistence from #1210 is fixed end-to-end. The strongest concrete evidence still points the other way: the issue remained reproducible after the Phase 1 workaround, and the supplied timeline still references an open fix PR (#50914) rather than a landed main-branch resolution.

## Evidence

- **Issue asks for a storage-level fix beyond the existing workaround:** Issue #1210 says the current workaround is only runtime trimming via `limitHistoryImages()` and explicitly proposes switching transcript storage from inline base64 to media references. That means the original request was not satisfied by Phase 1 alone. ([github.com](https://github.com/openclaw/openclaw/issues/1210))
- **Current main shows targeted offload logic for gateway chat attachments:** Current main `src/gateway/chat-attachments.ts` defines `OffloadedRef`, replaces large attachments with `media://inbound/<id>`, and states that callers must persist `offloadedRefs` separately for transcript media metadata. That is evidence of a specific gateway attachment path, not proof that inbound Discord/channel transcript persistence is already fixed everywhere. ([raw.githubusercontent.com](https://raw.githubusercontent.com/openclaw/openclaw/refs/heads/main/src/gateway/chat-attachments.ts))
  - file: `src/gateway/chat-attachments.ts`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Current main has hydration support for media references:** Current main `src/agents/pi-embedded-runner/run/images.ts` detects `[media attached: media://inbound/<id>]` markers and resolves them back into image blocks by loading the stored file. That shows the reference-and-hydrate mechanism exists, but it still does not show that the inbound Discord path from #1210 persists transcripts this way instead of inline base64. ([raw.githubusercontent.com](https://raw.githubusercontent.com/openclaw/openclaw/refs/heads/main/src/agents/pi-embedded-runner/run/images.ts))
  - file: `src/agents/pi-embedded-runner/run/images.ts`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Related transcript-bloat issue was not treated as broadly solved:** Related issue #6202 explicitly says the Phase 1 fix only covered inbound channel images and that read/tool-result images were still being persisted as inline base64 in session JSONL. That makes closing #1210 unsafe without stronger evidence of a broader landed fix. ([github.com](https://github.com/openclaw/openclaw/issues/6202))
- **Task context still points to unresolved fix work:** The supplied issue timeline references open PR #50914, titled `fix(agents): strip inline image data before session JSONL persistence`, which is another signal that maintainers still considered this unresolved rather than already implemented on `main`.
