## Summary

Current `main` already clears persisted auto-fallback model overrides before the next run and skips stored session overrides when an explicit heartbeat model is configured, which addresses the reported heartbeat session getting stuck on fallback indefinitely.

## Evidence

- **Auto fallback is now treated as transient:** `createModelSelectionState` clears direct `modelOverrideSource: "auto"` session overrides, persists that cleanup back to the session store, and resets the in-memory provider/model to the configured primary before the next attempt.
  - file: [src/auto-reply/reply/model-selection.ts:173](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/auto-reply/reply/model-selection.ts#L173)
  - command: `nl -ba src/auto-reply/reply/model-selection.ts | sed -n '140,220p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Explicit heartbeat model now beats stored session fallback:** When a heartbeat run resolved `heartbeat.model`, model selection skips any stored session override instead of inheriting a persisted fallback model from prior runs.
  - file: [src/auto-reply/reply/model-selection.ts:242](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/auto-reply/reply/model-selection.ts#L242)

- **Dedicated tests cover the exact regression:** The regression tests assert that auto-failover overrides are cleared, the configured primary is retried, and preloaded fallback selections are reset instead of sticking across turns.
