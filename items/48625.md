## Summary

Keep open. Current `main` still rejects arbitrary host `/tmp` image paths in sandboxed mode, so the underlying request is not implemented, and the submitted PR bypasses the existing alias/symlink boundary checks that current sandboxed media reads rely on.

## Evidence

- **Sandboxed image reads rely on prior path validation:** In current `main`, sandboxed image inputs are resolved through `resolveSandboxedBridgeMediaPath(...)`, then passed to `loadWebMedia(..., { sandboxValidated: true, readFile: createSandboxBridgeReadFile(...) })`. That path only stays safe if the resolver preserves sandbox boundary checks.
  - file: [src/agents/tools/image-tool.ts:463](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/agents/tools/image-tool.ts#L463)
  - command: `nl -ba src/agents/tools/image-tool.ts | sed -n '420,500p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Current main intentionally limits temp-path allowance to OpenClaw-managed tmp roots:** `resolveSandboxedMediaSource` only accepts tmp media through `resolveAllowedTmpMediaPath`, which requires the resolved path to live under `resolvePreferredOpenClawTmpDir()` and then calls `assertNoPathAliasEscape(..., boundaryLabel: "tmp root")`. It does not allow arbitrary `/tmp/*` host paths.
  - file: [src/agents/sandbox-paths.ts:172](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/agents/sandbox-paths.ts#L172)

  - file: [src/agents/sandbox-paths.test.ts:201](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/agents/sandbox-paths.test.ts#L201)

Review comments correctly note that this skips the existing alias/symlink protection and would permit symlink-based sandbox escape via `/tmp`.

## Risks / Open Questions

- Merging the PR as written would weaken the sandbox by allowing temp-path reads without the alias/symlink checks current code applies at the sandbox/tmp-root boundary.
- The original report may still describe a real usability gap for temp-file images, but it needs a different implementation that preserves `assertNoPathAliasEscape`-level safety instead of bypassing it.
