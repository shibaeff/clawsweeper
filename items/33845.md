## Summary

Keep open. Current `main` still uses an in-memory-only Slack thread participation cache, so the reported restart regression is still plausible and not implemented on `main` or the latest local `v2026.4.22` tag.

## Evidence

- **Slack cache is still memory-only on main:** `extensions/slack/src/sent-thread-cache.ts` only uses `resolveGlobalDedupeCache`; `recordSlackThreadParticipation()` writes to the in-memory dedupe cache and `clearSlackThreadParticipationCache()` only clears memory. There is no disk/state-dir persistence path in this file.
  - file: [extensions/slack/src/sent-thread-cache.ts:1](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/slack/src/sent-thread-cache.ts#L1)
  - command: `nl -ba extensions/slack/src/sent-thread-cache.ts | sed -n '1,260p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Mention gating depends on that cache for thread follow-ups:** `prepare.ts` treats `hasSlackThreadParticipation(account.accountId, message.channel, message.thread_ts)` as an implicit mention kind (`bot_thread_participant`) used by `resolveInboundMentionDecision()`. If the cache is empty after restart, non-mention follow-ups in an existing Slack thread can be skipped again.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:323](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/slack/src/monitor/message-handler/prepare.ts#L323)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare.ts | sed -n '300,460p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Replies repopulate the cache only after delivery:** `dispatch.ts` records thread participation only after a reply is actually delivered. Combined with the in-memory-only cache, a gateway restart drops the participation state until the bot is explicitly re-engaged and replies again.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:995](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/slack/src/monitor/message-handler/dispatch.ts#L995)

- **Latest local release tag shows the same implementation:** The local `v2026.4.22` tag still contains the same `sent-thread-cache.ts` with no persistence logic, so there is no shipped fix in the current release line either.
