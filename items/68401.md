## Summary

Keep open. Current `main` still serializes session compaction on the session lane, marks the run as compacting during preflight compaction, rejects steering new messages into an active run while compacting, and falls back to queued follow-up handling rather than concurrent processing. I found no landed background/offloaded compaction path that would satisfy this request.

## Evidence

- **Preflight compaction explicitly enters a compacting phase:** `runPreflightCompactionIfNeeded` sets `replyOperation` to `preflight_compacting` and then awaits `compactEmbeddedPiSession(...)`, so the session is intentionally in a compacting state before the next reply turn proceeds.

- **Reply registry classifies preflight compaction as compacting:** `isReplyRunCompacting` returns true for `preflight_compacting` and `memory_flushing`, confirming that the gateway treats these phases as compacting/busy rather than normal streaming.
  - file: [src/auto-reply/reply/reply-run-registry.ts:160](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/auto-reply/reply/reply-run-registry.ts#L160)

That directly contradicts the requested behavior of continuing live message processing during compaction.

The user-facing control UI docs also still say that while a run is active, normal follow-ups queue.
