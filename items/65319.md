## Summary

Keep open. Current `main` only fixes this class of `plugins.allow` contradiction when the owning plugin publishes manifest `commandAliases`, and the current published `lossless-claw` manifest still does not advertise `lcm` or `/lossless`. That means `openclaw lcm` or `openclaw /lossless` can still fall through to the generic `plugins.allow excludes "<command>"` guidance instead of consistently pointing at `lossless-claw`.

## Evidence

- **CLI fallback still tells users to allow the command token itself when no alias owner is known:** `resolveMissingPluginCommandMessage(...)` first asks `resolveManifestCommandAliasOwner(...)` for a parent plugin. If that returns nothing, the fallback at lines 123-130 emits `plugins.allow excludes "<command>"` and tells the user to add that command token directly.
  - file: [src/cli/run-main.ts:90](https://github.com/openclaw/openclaw/blob/30aa7e0d4d225650b58c2fba79e71bffa5627b08/src/cli/run-main.ts#L90)
  - sha: [30aa7e0d4d22](https://github.com/openclaw/openclaw/commit/30aa7e0d4d225650b58c2fba79e71bffa5627b08)
- **Alias ownership is manifest-only:** `resolveManifestCommandAliasOwnerInRegistry(...)` only scans manifest `commandAliases` entries. It does not infer ownership from runtime-registered commands or context-engine ids, so a plugin command without manifest alias metadata is invisible to this diagnostic path.
  - file: [src/plugins/manifest-command-aliases.ts:63](https://github.com/openclaw/openclaw/blob/30aa7e0d4d225650b58c2fba79e71bffa5627b08/src/plugins/manifest-command-aliases.ts#L63)
  - sha: [30aa7e0d4d22](https://github.com/openclaw/openclaw/commit/30aa7e0d4d225650b58c2fba79e71bffa5627b08)
- **Config validation only redirects when manifest alias metadata exists:** `plugins.allow` validation checks `resolveManifestCommandAliasOwner(...)` before emitting a warning. If no alias owner is found, it keeps the generic stale `plugin not found` path instead of redirecting to the parent plugin id.
  - file: [src/config/validation.ts:1074](https://github.com/openclaw/openclaw/blob/30aa7e0d4d225650b58c2fba79e71bffa5627b08/src/config/validation.ts#L1074)
  - sha: [30aa7e0d4d22](https://github.com/openclaw/openclaw/commit/30aa7e0d4d225650b58c2fba79e71bffa5627b08)
- **Published lossless-claw metadata still lacks the alias data this fix depends on:** The current upstream `openclaw.plugin.json` for `@martian-engineering/lossless-claw` (npm latest `0.9.2`, gitHead `8149b76f237c1cc46e307a3163be15ca8c07a53a`) contains no `commandAliases` entry for `lcm` or `/lossless`, so current `main` still has no manifest-level mapping back to `lossless-claw` for this plugin.
  - command: `curl -fsSL https://raw.githubusercontent.com/Martian-Engineering/lossless-claw/main/openclaw.plugin.json | rg -n 'commandAliases|lcm|lossless'`
  - sha: [8149b76f237c](https://github.com/openclaw/openclaw/commit/8149b76f237c1cc46e307a3163be15ca8c07a53a)

## Risks / Open Questions

- I did not run a live CLI reproduction because this review was read-only and the external `lossless-claw` plugin is not part of the checkout.
- If `lossless-claw` publishes manifest `commandAliases` in a later release, the issue may become obsolete without another core change.
