## Summary

Keep open. Current `main` does not implement the PR's per-subsystem shutdown timeout behavior before channel/plugin and server teardown, and the PR discussion contains unresolved correctness/regression concerns that still need maintainer judgment rather than auto-closure.

## Evidence

- **Current shutdown path still awaits early subsystems without per-subsystem timeout:** `createGatewayCloseHandler` on current `main` still awaits `bonjourStop`, `tailscaleCleanup`, `canvasHost.close`, `canvasHostServer.close`, `stopChannel`, and `stopGmailWatcher` directly before WebSocket/HTTP shutdown. A hung stop in those earlier calls would still block later teardown.
  - file: [src/gateway/server-close.ts:116](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-close.ts#L116)
  - command: `nl -ba src/gateway/server-close.ts | sed -n '1,240p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current main only has targeted timeout protection later in shutdown:** Current `main` has timeout races for bundle MCP disposal, WebSocket close, and HTTP close, but not for the earlier subsystem stop/close calls this PR is trying to protect.
  - file: [src/gateway/server-close.ts:145](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-close.ts#L145)

That matches the PR review concern that timing out HTTP close too early could let restart proceed while the listener is still draining.

- Closing as `implemented_on_main` would be incorrect because current `main` still lacks the proposed per-subsystem timeout wrapper for the early shutdown steps.
- Closing as `cannot_reproduce` would be weak: the current code path still has a plausible serial-blocking failure mode if one early subsystem stop hangs.
- Auto-closing this PR would discard potentially useful reliability work even though reviewer feedback suggests it needs revision rather than abandonment.
