## Summary

Current `main` does not contain this PR’s Slack wake/thread-routing behavior. Slack block actions still enqueue plain system events without an immediate heartbeat wake, and Slack system-event session resolution still ignores `threadTs`, so the bug described by the PR is not implemented on `main`.

## Evidence

- **`enqueueSystemEvent()` has no wake hook on current main:** `SystemEventOptions` only includes `sessionKey`, `contextKey`, `deliveryContext`, and `trusted`, and `enqueueSystemEvent()` just queues the event and returns. There is no `wake` option and no call to `requestHeartbeatNow()` in the current implementation.
  - file: [src/infra/system-events.ts:36](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/infra/system-events.ts#L36)
  - command: `nl -ba src/infra/system-events.ts | sed -n '1,180p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Slack block actions still do not request an immediate wake:** The block-action handler resolves a session key and calls `enqueueSystemEvent()` with only `sessionKey` and `contextKey`. The call site does not pass any wake hint, so interactive actions still rely on the normal heartbeat cadence.
  - file: [extensions/slack/src/monitor/events/interactions.block-actions.ts:630](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/slack/src/monitor/events/interactions.block-actions.ts#L630)

There is no `threadTs` parameter and no `resolveThreadSessionKeys()` call, so thread-scoped interactive actions still cannot resolve to a thread session key.
