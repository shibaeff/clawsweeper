## Summary

Keep open. Current main appears to have fixed the Feishu voice-bubble path for TTS voice-note delivery by marking Feishu as a transcoded voice-note channel and converting likely audio files to Ogg/Opus when `audioAsVoice` is set, but the Microsoft per-agent/per-message voice-override part of the request is still not implemented. The issue is mixed, and the latest maintainer comment also says it is only partially fixed and not closeable as complete.

## Evidence

- **Feishu now transcodes voice-intent audio to native audio bubbles:** `sendMediaFeishu` now calls `prepareFeishuVoiceMedia`, which transcodes likely audio inputs to 48kHz Ogg/Opus via `ffmpeg` when `audioAsVoice` is true, then routes them as Feishu `audio` messages instead of file attachments.
  - file: [extensions/feishu/src/media.ts:601](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/src/media.ts#L601)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Speech core marks Feishu TTS replies for channel-side transcoding:** Speech core treats `feishu` as a native voice-note channel and as the only transcoded voice-note channel, so non-Opus provider output still gets `audioAsVoice: true` on Feishu voice-note replies.
