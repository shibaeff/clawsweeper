## Summary

Current `main` does not match the reported Feishu bootstrap failure path: startup bot-info/token timeouts are treated as non-fatal, account startup continues, background bot-identity recovery is scheduled, and the pinned Feishu SDK version uses auto-reconnect with unlimited retry count rather than a single retry-and-give-up loop.

## Evidence

- **Startup timeout is non-fatal:** `fetchBotIdentityForMonitor()` treats timeout errors as a logged warning and returns an empty identity so startup can continue instead of failing the Feishu account bootstrap.
  - file: [extensions/feishu/src/monitor.startup.ts:67](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/feishu/src/monitor.startup.ts#L67)
  - command: `nl -ba extensions/feishu/src/monitor.startup.ts | sed -n '45,74p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Account startup proceeds after missing bot identity:** `monitorSingleAccount()` only starts background recovery when `botOpenId` is missing, then continues into `monitorWebSocket()`/`monitorWebhook()` instead of aborting the account start.
  - file: [extensions/feishu/src/monitor.account.ts:424](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/feishu/src/monitor.account.ts#L424)
  - command: `nl -ba extensions/feishu/src/monitor.account.ts | sed -n '411,459p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Background recovery retries are built in:** Feishu bot-identity recovery retries on a backoff schedule of 60s, 120s, 300s, 600s, and 900s, and logs success/failure without requiring a manual gateway restart.
  - file: [extensions/feishu/src/monitor.bot-identity.ts:10](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/feishu/src/monitor.bot-identity.ts#L10)
  - command: `nl -ba extensions/feishu/src/monitor.bot-identity.ts | sed -n '8,86p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Timeout path is covered by tests:** The Feishu startup test explicitly covers a probe timeout and asserts startup still advances to the next account while logging the timeout, which matches the intended current behavior.
  - file: [extensions/feishu/src/monitor.startup.test.ts:123](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/feishu/src/monitor.startup.test.ts#L123)
  - command: `nl -ba extensions/feishu/src/monitor.startup.test.ts | sed -n '123,157p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Pinned upstream WS client retries instead of giving up once:** OpenClaw pins `@larksuiteoapi/node-sdk` `^1.61.1`. Reviewing that version's published source shows `WSConfig` defaults `autoReconnect: true` and `reconnectCount: -1`, and `WSClient.start()` enters `reConnect(true)` with continued reconnect attempts rather than a one-shot retry path.
  - file: [extensions/feishu/package.json:7](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/feishu/package.json#L7)

## Risks / Open Questions

- No live Feishu credentials were available in this review, so the conclusion is based on current code, tests, and upstream SDK source rather than an end-to-end network reproduction.
- The broader restart/recovery concerns mentioned in #68766, #69254, and #69304 may still warrant separate design work, but they are not strong evidence that this specific current-main Feishu bootstrap failure still reproduces.
