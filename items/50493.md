## Summary

Keep open. Current `main` still replaces the global hook runner on every registry activation, so the reported overwrite path is not implemented on `main`, and this PR's guard would also block an intentional second full gateway plugin reload.

## Evidence

  - file: [src/plugins/loader.ts:1820](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/plugins/loader.ts#L1820)

  - file: [src/agents/pi-embedded-runner/run.ts:307](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/pi-embedded-runner/run.ts#L307)

A blanket idempotency guard on the global hook runner would freeze the earlier registry across that intentional reload.
  - file: [src/gateway/server.impl.ts:769](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/gateway/server.impl.ts#L769)

- **Setup-runtime loads do not expose full hook registration, so later reloads can legitimately add hooks:** Channel plugins enter `setup-runtime` mode during deferred startup, but `buildPluginApi` only exposes `registerHook`/typed `on(...)` in `full` mode. That means the later full reload can be the first time some hooks are registered, so preserving the earlier runner is not generally safe.
