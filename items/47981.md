## Summary

Keep open. Current `main` still computes status reachability from `gatewayProbe?.ok === true`, so scope-limited probes are still treated as unreachable in the `status`/`status --all` pipeline even though the shared gateway helper and docs classify `missing scope: operator.read` as reachable-but-degraded.

## Evidence

- **Current main still uses strict probe success for status reachability:** `resolveGatewayProbeSnapshot` sets `gatewayReachable` from `gatewayProbe?.ok === true`, so a probe that connected successfully but failed the read RPC with `missing scope: operator.read` is still marked unreachable in the shared status snapshot.
  - file: [src/commands/status.scan.shared.ts:131](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/commands/status.scan.shared.ts#L131)
  - command: `nl -ba src/commands/status.scan.shared.ts | sed -n '110,170p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **OpenClaw already has the correct reachability helper:** `isProbeReachable(probe)` explicitly treats scope-limited probe failures as reachable by returning `probe.ok || isScopeLimitedProbeFailure(probe)`. The helper test covers the exact `missing scope: operator.read` case and expects it to be reachable.
  - file: [src/commands/gateway-status/helpers.ts:279](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/commands/gateway-status/helpers.ts#L279)

  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Docs say this is degraded reachability, not failure:** The gateway docs state that `Read probe: limited - missing scope: operator.read` is reported as degraded reachability, not full failure. Troubleshooting says the same: connect worked, but detail RPC is scope-limited.
  - file: [docs/cli/gateway.md:227](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/docs/cli/gateway.md#L227)
  - command: `nl -ba docs/cli/gateway.md | sed -n '220,238p'; nl -ba docs/help/troubleshooting.md | sed -n '24,34p'`

That means current `main` still surfaces the scope-limited case as unreachable across status output instead of reachable-but-limited.
  - file: [src/commands/status.scan-overview.ts:84](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/commands/status.scan-overview.ts#L84)

I do not see equivalent logic on current `main`, which is consistent with this PR not having landed yet.
