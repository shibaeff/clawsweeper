## Summary

Current `main` still appears to have the reported watchdog behavior, and this PR should stay open. `executePreparedCliRun()` always computes and passes `noOutputTimeoutMs` to the supervisor even when prompt input is sent on stdin, while the child adapter closes stdin immediately after writing that input. That means a long-running silent CLI process can still be killed by the no-output watchdog even though it is not waiting for interactive input. The PR is directionally correct, but the review discussion also identified an unresolved empty-stdin edge case and misleading `?? 0` timeout text, so this is not a strong auto-close candidate.
