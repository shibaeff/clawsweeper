## Summary

This is a coherent integrity-monitoring idea, but on current `main` it fits OpenClaw’s extension model better than core: `VISION.md` says optional capability should usually ship as plugins and new skills should go to ClawHub first, while the workspace and config docs already expose the seams needed for startup checks, periodic checks, notifications, and versioned baselines around `AGENTS.md`/`SOUL.md`/related files. ([raw.githubusercontent.com](https://raw.githubusercontent.com/openclaw/openclaw/main/docs/concepts/agent-workspace.md))

## Evidence

- **Scope anchor points away from core:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, and new skills should go to ClawHub first. That is the repo’s stated policy for where features like this belong.

## Risks / Open Questions

- A ClawHub skill/plugin can monitor and announce drift, but it does not by itself create a stronger host-level trust boundary. If maintainers later want enforced signed-bootstrap semantics in core, that would be a separate product/security decision.

## Close Comment

Closing as `clawhub` after a ClawSweeper/Codex review against current main (`db17b8c`). The idea is useful, but current main already points this kind of workspace-integrity automation out of core: `VISION.md:4` says optional capability should usually ship as plugins and `VISION.md:6` says new skills should go to ClawHub first; `docs/concepts/agent-workspace.md:5` defines the exact workspace files you want to protect, `docs/concepts/agent-workspace.md:6` already provides `BOOT.md` and `HEARTBEAT.md` hooks, `docs/concepts/agent-workspace.md:11` recommends private git backup/versioning for those files, and `docs/tools/skills.md:6` documents plugin/skill delivery plus ClawHub. That makes this a better fit as a ClawHub skill/plugin that hashes `AGENTS.md`/`SOUL.md`/`USER.md`/`IDENTITY.md`/`TOOLS.md`, stores a baseline, and alerts on drift, rather than adding new core surface area.
