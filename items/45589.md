## Summary

Keep open. Current `main` fixes part of the report: model fallback now walks an ordered candidate chain across providers, and the default LLM idle timeout is 120s when no explicit timeout is configured. But the Discord path still serializes inbound work per session/channel key through a keyed async queue, with a default inbound-worker timeout of 30 minutes. That means a slow/stuck run can still block later messages for the same Discord session, so the reported channel-unresponsive failure mode is not implemented away on `main`.
