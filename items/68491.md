## Summary

Keep open. Current `main` still appears to have the reported Feishu duplicate-final race: `closeStreaming()` clears streaming state without recording the streamed text in `deliveredFinalTexts`, so a later `final` delivery can still bypass the streaming path and resend the same content. The newer Feishu dispatcher/tests added on `main` after this PR opened do not cover the `onIdle` -> `final` sequence called out in the PR and review discussion.

## Evidence

- **Current main still clears streaming without marking the final text as delivered:** `closeStreaming()` awaits startup/queue, closes the active streaming card, then nulls `streaming` and clears `streamText`, but it never adds the streamed text to `deliveredFinalTexts`. That leaves no dedupe marker for a later `final` payload after `onIdle` closes streaming.

  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Duplicate-final guard still depends on `deliveredFinalTexts` after streaming is already inactive:** `deliver()` skips duplicate final text only when `deliveredFinalTexts.has(text)` is true. In the streaming branch, the set is updated only when `info?.kind === "final"` arrives while `streaming?.isActive()` is still true. If `onIdle` already called `closeStreaming()`, execution falls through the non-streaming path and can resend the same text.
