## Summary

The reported `gateway closed (1000 normal closure): no close reason` symptom matches OpenClaw's old pre-auth WebSocket handshake-timeout path, and current main contains the shipped fixes for that path, including a longer default timeout and a Windows-targeted client-side mitigation.

## Evidence

- **Reported error matches the old handshake-timeout close path:** The gateway pre-auth handshake timer closes the socket with the default 1000/no-reason close when the client does not finish connect in time, which lines up with the issue's `gateway closed (1000 normal closure): no close reason` message.
  - file: [src/gateway/server/ws-connection.ts:366](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/server/ws-connection.ts#L366)
  - command: `nl -ba src/gateway/server/ws-connection.ts | sed -n '366,379p'`
- **Current main increased the default handshake timeout:** Current main sets `DEFAULT_PREAUTH_HANDSHAKE_TIMEOUT_MS = 10_000`, so healthy local CLI connections are no longer dropped on the old shorter default.
  - file: [src/gateway/handshake-timeouts.ts:1](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/handshake-timeouts.ts#L1)
  - command: `nl -ba src/gateway/handshake-timeouts.ts | sed -n '1,45p'`
- **Current main added a Windows-specific mitigation before connect:** `callGateway()` now yields to the event loop before opening the WebSocket, with an inline note that Windows large dist bundles could starve `connect.challenge` processing and cause handshake timeouts.
  - file: [src/gateway/call.ts:479](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/call.ts#L479)
  - command: `nl -ba src/gateway/call.ts | sed -n '466,500p'`
- **Changelog shows the fix shipped after 2026.3.13:** The 2026.3.22 changelog explicitly says Gateway/WS handshake now uses a 10 second pre-auth timeout so busy local gateways stop dropping healthy CLI connections at 3 seconds.
  - file: [CHANGELOG.md:2443](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/CHANGELOG.md#L2443)
  - command: `nl -ba CHANGELOG.md | sed -n '2428,2444p'`

## Close Comment

Closing from a Codex review because this looks implemented on current `main`. The issue was reported against OpenClaw 2026.3.13, and the current codebase now hardens the exact failure path behind `gateway closed (1000 normal closure): no close reason`: the pre-auth handshake timeout in `src/gateway/server/ws-connection.ts:366` closes with the default 1000/no-reason close, current main raises that default timeout to 10s in `src/gateway/handshake-timeouts.ts:1`, and `src/gateway/call.ts:479` adds a Windows-specific event-loop yield to avoid `connect.challenge` starvation. The changelog entry for release 2026.3.22 also calls out this handshake fix directly (`CHANGELOG.md:2443`). If this is still reproducible on a current release, please open a fresh issue with current version, gateway logs around startup/connect, and whether the gateway is foreground, Scheduled Task, or Startup-folder managed on Windows.
