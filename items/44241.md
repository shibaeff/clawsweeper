## Summary

Current `main` does not have the save path described in the issue. Memory flushes are now routed through a dedicated `memoryFlushWritePath` and an append-only `write` tool that persists directly to `memory/YYYY-MM-DD.md`; `systemPromptReport` is only prompt-diagnostics metadata. The timeout branch in `runEmbeddedPiAgent` affects the returned payload, not a later memory-file save stage, so the reported root cause no longer matches current or latest-release code.

## Close Comment

Closing as `cannot_reproduce` based on a Codex review of current `main` and the latest shipped tag I could inspect. On current OpenClaw, memory flush persistence is handled inside the memory-triggered run itself: `src/auto-reply/reply/agent-runner-memory.ts:766` passes `trigger: "memory"` plus `memoryFlushWritePath`, `src/agents/pi-tools.ts:638` restricts the run to memory-flush-safe tools, and `src/agents/pi-tools.read.ts:564` appends directly to the allowed `memory/YYYY-MM-DD.md` file during the `write` tool call. `systemPromptReport` is only prompt/context diagnostics metadata (`src/agents/system-prompt-report.ts:98`, `src/config/sessions/types.ts:459`), not the durable memory content being flushed. The timeout branch in `src/agents/pi-embedded-runner/run.ts:1742` only changes the returned payload when no payloads exist; it is not guarding a later memory-file save stage in current code. I also checked the tagged `v2026.4.22` source and found the same structure. If this still reproduces on current `main` or a recent release, please open a new issue with the exact version/commit, timeout logs, and whether the memory `write` tool ran before the timeout.
