## Summary

Keep open. Current `main` still lacks the blockquote-prefix continuation handling this PR adds, and the option is used in shipped Slack/Signal formatters, so this remains a real user-facing bug rather than an obsolete PR.

## Evidence

- **Current `main` drops quote prefixes on soft/hard continuation lines:** `renderTokens` handles `softbreak` and `hardbreak` by appending only `"\n"`. With `blockquotePrefix` set, continuation lines inside a blockquote are not re-prefixed.
  - file: [src/markdown/ir.ts:634](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/markdown/ir.ts#L634)
  - command: `nl -ba src/markdown/ir.ts | sed -n '580,660p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Current `main` also drops prefixes after paragraph breaks inside blockquotes:** `appendParagraphSeparator` emits `"\n\n"` and has no blockquote-depth/prefix handling, so a multi-paragraph blockquote cannot preserve `> ` on the next paragraph.
  - file: [src/markdown/ir.ts:265](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/markdown/ir.ts#L265)
  - command: `nl -ba src/markdown/ir.ts | sed -n '260,273p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **The affected option is used by shipped channel formatters:** Slack and Signal both call `markdownToIR(..., { blockquotePrefix: "> " })`, so the missing continuation-prefix logic affects real outbound rendering surfaces.
  - file: [extensions/slack/src/format.ts:121](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/extensions/slack/src/format.ts#L121)
