## Summary

Keep open. Current `main` does not implement the requested behavior, and the PR's `.passthrough()` change would not be sufficient because OpenClaw still normalizes and exposes plugin-owned settings only through `plugins.entries.<id>.config`. No allowed close reason applies.

## Evidence

- **Schema still rejects extra top-level plugin-entry keys:** `PluginEntrySchema` on current `main` allows only `enabled`, `hooks`, `subagent`, and `config`, then ends with `.strict()`, so provider-specific top-level keys are still invalid.
  - file: [src/config/zod-schema.ts:158](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/config/zod-schema.ts#L158)
  - command: `nl -ba src/config/zod-schema.ts | sed -n '158,190p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Normalization drops unknown top-level entry keys:** Plugin entry normalization copies only `enabled`, `hooks`, `subagent`, and `config` into the normalized entry object. Any extra top-level keys would be discarded even if schema validation were relaxed.
  - file: [src/plugins/config-normalization-shared.ts:60](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/plugins/config-normalization-shared.ts#L60)
  - command: `nl -ba src/plugins/config-normalization-shared.ts | sed -n '60,140p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Runtime validates and passes only `entry.config`:** Plugin loading validates `entry?.config` against the manifest schema and passes that validated value as `pluginConfig` to the plugin API, so top-level passthrough keys would still not reach plugin runtime.
  - file: [src/plugins/loader.ts:2371](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/plugins/loader.ts#L2371)
  - command: `nl -ba src/plugins/loader.ts | sed -n '2334,2384p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Public SDK contract documents plugin config under `.config`:** The SDK docs state that `api.pluginConfig` comes from `plugins.entries.<id>.config`, matching the runtime code path rather than arbitrary top-level plugin entry keys.
  - file: [docs/plugins/sdk-overview.md:256](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/docs/plugins/sdk-overview.md#L256)
  - command: `nl -ba docs/plugins/sdk-overview.md | sed -n '248,262p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Validation paths are anchored to `.config`:** Plugin-schema validation reports are rooted at `plugins.entries.<pluginId>.config`, which reinforces that plugin-defined settings are expected inside the `config` subobject.
  - file: [src/config/validation.ts:691](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/config/validation.ts#L691)
  - command: `nl -ba src/config/validation.ts | sed -n '680,708p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)

## Risks / Open Questions

- Closing this PR as `implemented_on_main` or `cannot_reproduce` would be wrong: current `main` still rejects the reported top-level keys.
- Merging the PR as-is would likely create a silent misconfiguration path, because relaxed schema acceptance would not change normalization/runtime behavior that only reads `plugins.entries.<id>.config`.
