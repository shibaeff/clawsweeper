## Summary

Keep open. Current `main` still enforces plugin-owned settings under `plugins.entries.<id>.config`, and this PR's `.passthrough()` change would not carry extra top-level keys through normalization or runtime loading. As written, it would turn a hard validation error into a silent misconfiguration path rather than actually implementing provider-specific top-level entry keys.

## Evidence

- **Root plugin entry schema stays strict:** `PluginEntrySchema` on current `main` only allows `enabled`, `hooks`, `subagent`, and `config`, then calls `.strict()`. Extra top-level keys are still rejected at the root config schema boundary.
  - file: [src/config/zod-schema.ts:158](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/config/zod-schema.ts#L158)
  - command: `nl -ba src/config/zod-schema.ts | sed -n '158,182p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Normalization would still drop passthrough keys:** Plugin entry normalization only preserves `enabled`, `hooks`, `subagent`, and `config`; arbitrary top-level keys are not copied into normalized plugin state. That means accepting unknown keys at parse time would not make them available later.
  - file: [src/plugins/config-normalization-shared.ts:132](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/plugins/config-normalization-shared.ts#L132)
  - command: `nl -ba src/plugins/config-normalization-shared.ts | sed -n '132,140p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Runtime validates and injects only `entry.config`:** Plugin loading validates `entry?.config` against the plugin manifest schema and passes that validated object as `pluginConfig`. The SDK helper also reads only `plugins.entries.<id>.config`. Top-level passthrough keys would still be ignored by runtime/plugin APIs.
  - file: [src/plugins/loader.ts:2525](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/plugins/loader.ts#L2525)
  - command: `nl -ba src/plugins/loader.ts | sed -n '2520,2545p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **SDK contract documents config under `.config`:** Repo docs explicitly define plugin config as `plugins.entries.<plugin-id>.config`, and `api.pluginConfig` is documented as coming from that nested object rather than from arbitrary top-level entry keys.
  - file: [docs/plugins/manifest.md:514](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/plugins/manifest.md#L514)
  - command: `nl -ba docs/plugins/manifest.md | sed -n '508,525p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Prior PR review already identified the same gap:** The PR discussion includes a review noting that switching to `.passthrough()` would not fix runtime behavior because downstream normalization/loading only keep `enabled` and `config`, creating a silent misconfiguration path instead of a real fix.
  - sha: [5b33b7a88206](https://github.com/openclaw/openclaw/commit/5b33b7a88206077425524b4874f7520a18741719)
