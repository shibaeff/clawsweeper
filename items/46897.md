## Summary

Current `main` already carries the loopback-probe fix: authenticated local `openclaw status` probes attach device identity again, so read/detail RPCs are no longer scope-limited by the shared-auth hardening path. The same logic is present in the shipped `v2026.4.22` release, and current tests cover the exact local authenticated probe scenario.

## Close Comment

Thanks for the report. A Codex review checked current `main` and the shipped `v2026.4.22` release. The local loopback probe now explicitly keeps authenticated token/password probes device-bound in `src/gateway/probe.ts:148`, which avoids the shared-auth scope stripping that produced `missing scope: operator.read` on `status` detail RPCs. This is covered by `src/gateway/probe.auth.integration.test.ts:29` and `src/gateway/probe.test.ts:129`, which assert that authenticated probes against `ws://127.0.0.1` can fetch `health`/`status`/`config.get` and attach device identity. Since that implementation is already present on `main` and in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), I’m closing this as fixed. If you still hit the warning on a newer build, please open a fresh issue with the exact version and `openclaw status --all` output.
