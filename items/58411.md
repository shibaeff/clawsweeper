## Summary

Current `main` still lacks a `sessions_spawn` equivalent to `/acp spawn ... --bind here` for ACP sessions. The tool schema only exposes `thread`, and the Discord ACP path still resolves that to child-thread placement rather than binding the current thread/conversation in place.

## Evidence

- **`sessions_spawn` has no current-bind parameter:** `SessionsSpawnToolSchema` defines `thread` but no `bindTo`/`bind` field, and the ACP execution path forwards only `thread` into `spawnAcpDirect`.
  - file: [src/agents/tools/sessions-spawn-tool.ts:100](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/tools/sessions-spawn-tool.ts#L100)
  - command: `nl -ba src/agents/tools/sessions-spawn-tool.ts | sed -n '85,125p'; nl -ba src/agents/tools/sessions-spawn-tool.ts | sed -n '230,270p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **ACP spawn defaults to child placement when available:** `resolvePlacementWithoutChannelPlugin` returns `child` whenever the adapter supports it, and `prepareAcpThreadBinding` uses that placement when the channel does not override it.
  - file: [src/agents/acp-spawn.ts:281](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/acp-spawn.ts#L281)
