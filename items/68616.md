## Summary

Keep open. The report is recent, current docs/code still imply browser should remain available to subagents unless a separate policy removes it, and I found no targeted fix on `main` that would justify closing as resolved.

## Evidence

- **Subagent docs still promise non-session tools by default:** `docs/tools/subagents.md` says sub-agents do not get session tools by default, and the depth-policy section says other session/system tools remain denied. It does not document any browser-specific exclusion for spawned subagents.
  - file: [docs/tools/subagents.md:69](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/docs/tools/subagents.md#L69)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Current subagent deny list does not exclude browser:** `resolveSubagentToolPolicyForSession()` only auto-denies `gateway`, `agents_list`, `session_status`, `cron`, `sessions_send`, plus session-control tools for leaf roles. `browser` is not in the built-in subagent deny lists, so current `main` does not show this as intentional behavior.
  - file: [src/agents/pi-tools.policy.ts:30](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/pi-tools.policy.ts#L30)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Control UI tools panel is runtime/session scoped:** The Control UI docs say 'Available Right Now' is driven by `tools.effective(sessionKey=...)`, and the handler derives channel/session context from the stored session before calling `resolveEffectiveToolInventory()`. That means a spawned-session mismatch here is still a live runtime surface, not just stale docs/config text.
  - file: [docs/web/webchat.md:42](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/docs/web/webchat.md#L42)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **`tools.effective` uses stored session context:** The `tools.effective` handler loads the target session, derives `messageProvider` from delivery/session metadata, and passes that into `resolveEffectiveToolInventory()`. Current `main` therefore still has session-specific tool-surface logic that could affect spawned subagents.
  - file: [src/gateway/server-methods/tools-effective.ts:75](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/gateway/server-methods/tools-effective.ts#L75)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **No targeted browser-subagent fix found in local history:** Local history since the issue date shows `fix(subagents): retire stale unended runs` and `fix(agents): keep reply tool snapshots aligned`, but nothing that clearly fixes browser inheritance for spawned subagents.
  - command: `git log --oneline --since='2026-04-18 00:00:00' --grep='subagent\|browser\|tools.effective\|reply tool' --all -- src docs ui | sed -n '1,40p'`
