## Summary

Current `main` still appears to match the reported bug. Docker sandbox pruning removes the runtime container and registry entry, but there is no code path that deletes the corresponding sandbox workspace directory under the sandbox workspace root.

## Evidence

- **Prune path only removes runtime and registry entry:** `pruneSandboxRegistryEntries` prunes by calling `removeRuntime(entry)` and then `remove(entry.containerName)`. There is no filesystem cleanup hook for sandbox workspace directories in the container prune path.
  - file: [src/agents/sandbox/prune.ts:38](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/sandbox/prune.ts#L38)
  - command: `nl -ba src/agents/sandbox/prune.ts | sed -n '38,79p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Docker backend removal only runs docker rm:** The Docker sandbox backend manager's `removeRuntime` implementation only executes `docker rm -f <container>`. It does not remove any host workspace directory.
  - file: [src/agents/sandbox/docker-backend.ts:143](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/sandbox/docker-backend.ts#L143)
  - command: `nl -ba src/agents/sandbox/docker-backend.ts | sed -n '143,149p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Registry does not track workspace directory:** `SandboxRegistryEntry` stores container/session/image metadata only. There is no `workspaceDir` or sandbox workspace path recorded, so current prune logic has no persisted directory target to delete for Docker entries.
  - file: [src/agents/sandbox/registry.ts:8](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/sandbox/registry.ts#L8)
  - command: `nl -ba src/agents/sandbox/registry.ts | sed -n '8,18p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Sandbox workspaces are still created under sandbox workspace root:** When sandbox workspace access is not `rw`, the sandbox uses `sandboxWorkspaceDir`, derived from `cfg.workspaceRoot` and the session scope key, and ensures that directory exists. That is the directory family the report says accumulates.
  - file: [src/agents/sandbox/context.ts:38](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/sandbox/context.ts#L38)

  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)

  - file: [src/agents/sandbox/prune.ts:38](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/sandbox/prune.ts#L38)
