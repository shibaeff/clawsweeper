## Summary

Current main already contains targeted gateway handshake and local-loopback pairing fixes that match the failure modes reported in #45504, with corresponding tests and shipped changelog entries.

## Evidence

- **Pre-auth handshake timeout raised:** Current main defaults the pre-auth handshake timeout to 10 seconds and keeps an explicit env override, matching the changelog note about busy local gateways dropping healthy CLI connections.
  - file: [src/gateway/handshake-timeouts.ts:1](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/handshake-timeouts.ts#L1)
- **CLI connect path now yields before WS start:** The gateway call path explicitly yields one event-loop turn before opening the WebSocket so heavy synchronous startup work does not starve the connect challenge and trigger handshake timeouts.
  - file: [src/gateway/call.ts:479](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/call.ts#L479)
- **Local loopback pairing is allowed silently:** Local/non-remote pairing requests now auto-allow the not-paired, role-upgrade, and scope-upgrade paths instead of forcing a disconnect-driven recovery flow.
  - file: [src/gateway/server/ws-connection/handshake-auth-helpers.ts:75](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/server/ws-connection/handshake-auth-helpers.ts#L75)
- **Silent local pairing is wired into connect handling:** When a connect needs pairing, the message handler now passes the local-loopback decision into the actual pairing request so eligible local requests are approved silently instead of failing the session.
  - file: [src/gateway/server/ws-connection/message-handler.ts:919](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/server/ws-connection/message-handler.ts#L919)
- **Regression coverage exists for direct local and shared-secret loopback paths:** Tests assert silent local pairing for direct local not-paired/upgrade cases and for shared-secret loopback metadata-upgrade cases, covering the local pairing regressions described in the issue thread.
  - file: [src/gateway/server/ws-connection/handshake-auth-helpers.test.ts:82](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/server/ws-connection/handshake-auth-helpers.test.ts#L82)
- **Shipped fix notes are recorded in releases after the report:** The changelog records the relevant shipped fixes: handshake-timeout hardening in 2026.3.22, loopback pairing treatment in 2026.4.20, and CLI/gateway teardown stabilization in 2026.4.22.
  - file: [CHANGELOG.md:2440](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/CHANGELOG.md#L2440)
- **Logs fallback was added for local loopback pairing deadlocks:** Docs now describe that `openclaw logs` falls back to the local log file when a local loopback gateway asks for pairing, which aligns with the related `logs --follow` symptom reported in the issue comments.
  - file: [docs/logging.md:73](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/docs/logging.md#L73)

## Close Comment

Closing after a Codex review of current `main`. The local-loopback CLI/gateway regression family reported here is already implemented in the current codebase: `src/gateway/handshake-timeouts.ts:1` raises the pre-auth handshake timeout to 10s, `src/gateway/call.ts:479` yields before the CLI WebSocket connect so `connect.challenge` handling is not starved, and `src/gateway/server/ws-connection/handshake-auth-helpers.ts:75` plus `src/gateway/server/ws-connection/message-handler.ts:919` now allow the local loopback pairing paths instead of failing them with disconnect-driven recovery. The shipped release notes also record this fix family in `CHANGELOG.md:2440` (`2026.3.22`), `CHANGELOG.md:390` (`2026.4.20`), and `CHANGELOG.md:151` (`2026.4.22`). On that basis, this issue is no longer actionable against current `main`.
