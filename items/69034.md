## Summary

Current main already implements `MEDIA:` parsing and Telegram attachment delivery; the reported `.txt` repro also conflicts with the documented host-local attachment policy, while broader local-path regressions are tracked separately in #69183.

## Evidence

- **Reply text is normalized for MEDIA directives before channel delivery:** `parseReplyDirectives()` calls `splitMediaFromOutput()` and returns parsed `mediaUrls`/`mediaUrl` from assistant reply text, so final reply text is not handed to Telegram as raw text-only output.
  - file: [src/auto-reply/reply/reply-directives.ts:16](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/auto-reply/reply/reply-directives.ts#L16)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **MEDIA parser extracts standalone MEDIA lines into attachment URLs:** `splitMediaFromOutput()` explicitly recognizes lines whose trimmed start is `MEDIA:`, validates local/remote targets, removes them from visible text, and returns `mediaUrls`/`mediaUrl` for downstream delivery.
  - file: [src/media/parse.ts:446](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/media/parse.ts#L446)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Telegram outbound adapter forwards parsed media to sendMessageTelegram:** `sendTelegramPayloadMessages()` resolves payload media URLs and sends them through Telegram's outbound send path; `telegramOutbound.sendMedia` and `sendPayload` both pass `mediaUrl` plus local-root/read-file context onward.
  - file: [extensions/telegram/src/outbound-adapter.ts:75](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/telegram/src/outbound-adapter.ts#L75)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Telegram send path uploads attachments via Telegram Bot API media methods:** When `mediaUrl` is present, `sendMessageTelegram()` loads the media and chooses `sendAnimation`, `sendPhoto`, `sendVideo`, `sendVoice`, `sendAudio`, or `sendDocument` instead of text-only delivery.
  - file: [extensions/telegram/src/send.ts:814](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/telegram/src/send.ts#L814)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Main includes tests covering MEDIA normalization and Telegram media delivery:** Tests assert MEDIA extraction from outbound payloads, direct Telegram media sends with local roots, and Telegram media upload behavior such as `sendPhoto` retries.
  - file: [src/infra/outbound/payloads.test.ts:525](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/infra/outbound/payloads.test.ts#L525)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Docs and media loader policy do not treat generic host-local .txt files as sendable attachments:** User docs say outbound `MEDIA:` is supported, but host-local sends are limited to media plus safe document types; the local-media guard allows PDF/Office/CSV/Markdown and rejects generic plaintext-like host files.
  - file: [docs/start/openclaw.md:191](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/docs/start/openclaw.md#L191)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)

## Risks / Open Questions

- A separate local outbound media allowlist/staging problem may still exist in some setups; the timeline cross-reference to #69183 is consistent with a broader local-path regression, but that is different from the specific claim that Telegram lacks `MEDIA:` parsing/upload support.
