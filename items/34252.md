## Summary

Keep open. Current main still treats native Ollama as a single bundled provider id (`ollama`) in manifest, discovery, docs, and runtime provider ownership. There is partial groundwork for custom local providers and provider-specific base URLs, but not enough evidence that `ollama-jetson`/multiple native Ollama providers are supported end-to-end on main.

## Evidence

- **Bundled Ollama manifest exports only one provider/auth surface:** The plugin manifest lists only `providers: ["ollama"]`, `syntheticAuthRefs: ["ollama"]`, and `providerAuthEnvVars.ollama`, with no aliases for alternate provider ids.
  - file: [extensions/ollama/openclaw.plugin.json:2](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/ollama/openclaw.plugin.json#L2)
  - command: `nl -ba extensions/ollama/openclaw.plugin.json | sed -n '1,120p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Discovery is hard-coded to models.providers.ollama:** The Ollama discovery/shared config path reads `config.models.providers.ollama`, resolves auth with `OLLAMA_PROVIDER_ID = "ollama"`, and returns only that provider shape.
  - file: [extensions/ollama/src/discovery-shared.ts:5](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/ollama/src/discovery-shared.ts#L5)
  - command: `nl -ba extensions/ollama/src/discovery-shared.ts | sed -n '1,180p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Runtime plugin ownership still matches exact provider ids:** Provider ownership resolution matches a plugin only when the requested provider id equals one of the manifest provider ids or CLI backends. A custom id like `ollama-jetson` would not map to the bundled Ollama plugin.
  - file: [src/plugins/providers.ts:367](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/plugins/providers.ts#L367)

- **Ollama stream registration depends on resolving a provider-owned runtime hook:** Simple completion setup calls `registerProviderStreamForModel`, which first asks `resolveProviderStreamFn` for the model's exact provider id. Without provider ownership, OpenClaw does not register the Ollama stream hook for that custom provider.

  - file: [docs/providers/ollama.md:151](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/providers/ollama.md#L151)
