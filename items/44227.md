## Summary

Current main already changed channel health evaluation so quiet Discord channels are no longer restarted from stale app-event timestamps alone; that behavior shipped in v2026.4.22.

## Evidence

- **Release note for the shipped fix:** The v2026.4.22 changelog explicitly says stale-socket recovery now uses provider-proven transport activity instead of inbound app-event freshness, and calls out Discord among the channels affected.
  - file: [CHANGELOG.md:266](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/CHANGELOG.md#L266)
  - command: `nl -ba CHANGELOG.md | sed -n '220,280p'`
- **Current health policy no longer treats stale app events as socket failure:** `evaluateChannelHealth` now only runs stale-socket detection when `lastTransportActivityAt` is present, with an inline comment that app-level events are not socket liveness for quiet Slack/Discord workspaces.
  - file: [src/gateway/channel-health-policy.ts:115](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/gateway/channel-health-policy.ts#L115)
  - command: `nl -ba src/gateway/channel-health-policy.ts | sed -n '1,240p'`
- **Regression test covers the quiet-channel case:** There is a test asserting that stale app events without transport activity are ignored and the channel remains healthy, matching the restart pattern described in this issue.
  - file: [src/gateway/channel-health-policy.test.ts:139](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/gateway/channel-health-policy.test.ts#L139)
  - command: `nl -ba src/gateway/channel-health-policy.test.ts | sed -n '120,230p'`

## Risks / Open Questions

- If someone can still reproduce repeated Discord code-1005 disconnects on v2026.4.22 or newer with fresh diagnostics, that would be a new reproducible bug rather than the stale-socket false positive fixed here.

## Close Comment

Closing as implemented on main after Codex review. The stale-socket path that was restarting quiet Discord channels has been changed: `src/gateway/channel-health-policy.ts:115` now bases stale-socket recovery on `lastTransportActivityAt` instead of stale inbound app events, and `src/gateway/channel-health-policy.test.ts:139` covers the exact quiet-channel case by keeping it healthy. This shipped in `v2026.4.22`, documented in `CHANGELOG.md:266`. If you still see repeat Discord disconnects on `v2026.4.22` or newer, please open a new issue with a fresh diagnostics export and the Discord gateway logs around the disconnect so maintainers can investigate that newer path separately.
