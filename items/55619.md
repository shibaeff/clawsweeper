## Summary

Keep open. Current `main` does not implement the Feishu reconnect/PingInterval patch, the bundled Feishu SDK version still has the underlying `PingInterval` dereference and built-in reconnect timing, and the PR discussion raised unresolved correctness risks in the proposed monkey-patch approach.

## Evidence

- **Current main still instantiates the SDK WS client directly:** `createFeishuWSClient` on current `main` just returns `new feishuClientSdk.WSClient(...)`; there is no Feishu-specific reconnect/backoff or `PingInterval` guard in-tree.
  - file: [extensions/feishu/src/client.ts:222](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/src/client.ts#L222)
  - command: `nl -ba extensions/feishu/src/client.ts | sed -n '218,237p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Bundled dependency is still the affected SDK line:** The Feishu plugin currently depends on `@larksuiteoapi/node-sdk` `^1.61.1`, so current `main` is still relying on the upstream WS client behavior unless OpenClaw adds its own mitigation.
  - file: [extensions/feishu/package.json:6](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/package.json#L6)
  - command: `nl -ba extensions/feishu/package.json | sed -n '1,12p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Upstream SDK still has the reported behavior:** In `@larksuiteoapi/node-sdk@1.61.1`, `handleControlData` unconditionally multiplies `PingInterval`, and `reConnect(false)` already applies `reconnectNonce` plus repeated `reconnectInterval` timers before retrying. An extra wrapper sleep would stack on top of that logic unless it fully replaces the SDK path.
  - command: `curl -L --silent https://unpkg.com/@larksuiteoapi/node-sdk@1.61.1/es/index.js | nl -ba | sed -n '85378,85605p'`
- **OpenClaw shutdown paths rely on `wsClient.close()` canceling reconnect work:** Feishu monitor cleanup and global monitor stop both call `wsClient.close()`. That makes the PR review concern material: any custom backoff timer held outside the SDK instance could survive shutdown and reopen the socket after cleanup.
  - file: [extensions/feishu/src/monitor.transport.ts:102](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/src/monitor.transport.ts#L102)
  - command: `nl -ba extensions/feishu/src/monitor.transport.ts | sed -n '99,117p'; nl -ba extensions/feishu/src/monitor.state.ts | sed -n '107,160p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **PR review already surfaced unresolved P1-level concerns:** The provided PR discussion includes repeated review comments on three substantive issues in the patch as proposed: backoff counter reset after successful runtime reconnects, canceling the pre-reconnect timer on `close()`, and avoiding stacked custom backoff plus the SDK's own reconnect delay. That is enough maintainer-risk to avoid conservative auto-close.
  - sha: [6edc66571374](https://github.com/openclaw/openclaw/commit/6edc665713745b21bfb72f6922f4eba13f34db3d)

- Merging the PR unchanged also looks risky because the proposed monkey patch is not yet proven against the SDK's existing reconnect lifecycle and shutdown semantics.
