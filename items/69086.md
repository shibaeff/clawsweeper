## Summary

Keep open. Current `main` still has the session-history guard bug described in the issue, and there is no dedicated retry-prompt override surface in the attempt-execution path. The issue discussion also says PR #69109 carried the guard fix but was closed unmerged, which matches the current code state.

## Evidence

- **Session-history guard still returns true on any assistant record:** `jsonlFileHasAssistantMessage` scans JSONL lines and returns `true` immediately on the first parsed record whose `message.role` is `assistant`. It does not track whether a newer `user` message appeared later in the file, so an older answered turn still marks the session as having history for a newer unanswered turn.
  - file: [src/agents/command/attempt-execution.helpers.ts:25](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/command/attempt-execution.helpers.ts#L25)
  - command: `nl -ba src/agents/command/attempt-execution.helpers.ts | sed -n '20,125p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Fallback retry path still relies on that broad guard:** On fallback retries, `agent-command` computes `sessionHasHistory` as `!isNewSession || (await attemptExecutionRuntime.sessionFileHasContent(sessionFile))`, and `runAgentAttempt` feeds that into `resolveFallbackRetryPrompt`. That means the broad transcript check still controls retry prompt behavior on new sessions.
  - file: [src/agents/agent-command.ts:918](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/agent-command.ts#L918)
