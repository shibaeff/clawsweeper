## Summary

Keep open. I did not find high-confidence evidence that current `main` fixes the reported Feishu inbound filename mojibake. The current code still trusts download metadata first, and the test coverage only proves ASCII header parsing or fallback when metadata is missing, not the reported UTF-8-to-Latin-1 corruption case.

## Evidence

- **Inbound save path still prefers download metadata over payload filename:** Feishu inbound media handling passes `result.fileName || mediaKeys.fileName` to `saveMediaBuffer()`. If the SDK/header-derived filename is mojibaked, it overrides the message payload filename instead of falling back to it.
  - file: [extensions/feishu/src/bot-content.ts:434](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/extensions/feishu/src/bot-content.ts#L434)
  - command: `nl -ba extensions/feishu/src/bot-content.ts | sed -n '430,460p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Filename decoder only handles RFC 5987 `filename*=` explicitly:** `decodeDispositionFileName()` decodes `filename*=UTF-8''...`, but plain `filename="..."` is returned verbatim. There is no recovery path for classic UTF-8 bytes misread as Latin-1 mojibake.
  - file: [extensions/feishu/src/media.ts:149](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/extensions/feishu/src/media.ts#L149)
