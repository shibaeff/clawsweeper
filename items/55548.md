## Summary

Current `main` does not contain this fix. Telegram `callback_query` handling still overwrites the turn message id with `callback.id`, the Telegram inbound context has no separate callback-source message field, and downstream reply/threading code still derives `currentMessageId` only from `MessageSid`/`MessageSidFull`. The PR remains relevant; the second PR commit also matches the review feedback about declaring and propagating `CallbackMessageId` rather than leaving it as an untyped payload field.

## Evidence

There is no separate source-message id passed alongside it, so the turn still identifies the callback query itself instead of the Telegram message being edited.
