## Summary

Keep open: current `main` appears to have restored 24h duplicate-heartbeat suppression, but the plugin-channel heartbeat fallback gap still remains in the heartbeat delivery path when plugin normalization is unavailable.

## Evidence

- **Heartbeat path still falls back to `no-target` after re-entering plugin-dependent resolution:** `resolveHeartbeatDeliveryTarget()` first recovers a session target, but then unconditionally calls `resolveOutboundTarget()` and converts any failure into `{ channel: "none", reason: "no-target" }`. In the plugin-registry-empty case described by the issue, this still depends on plugin lookup succeeding.
  - file: [src/infra/outbound/targets.ts:173](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/infra/outbound/targets.ts#L173)
  - command: `nl -ba src/infra/outbound/targets.ts | sed -n '84,245p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Session-level fallback exists independently of loaded plugin parsing:** `resolveSessionDeliveryTarget()` can recover `lastChannel`/`lastTo` from stored session delivery context without needing plugin-owned target parsing to succeed, which matches the issue's claim that the fallback exists at the session layer but is incomplete end-to-end for heartbeat routing.
  - file: [src/infra/outbound/targets-session.ts:120](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/infra/outbound/targets-session.ts#L120)

- **Regression coverage only proves session fallback, not heartbeat fallback, with registry unavailable:** There is a test for `resolveSessionDeliveryTarget()` keeping raw plugin-owned targets when the plugin registry is unavailable, but no matching heartbeat-target regression alongside it in the current file.
  - file: [src/infra/outbound/targets.test.ts:346](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/infra/outbound/targets.test.ts#L346)
