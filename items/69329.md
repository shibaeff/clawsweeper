## Summary

Current `main` confirms the report: `exec`/`process` completion is intentionally process-lifecycle based. Because the requested artifact-gated closure is a workflow-specific policy for an out-of-tree consumer, and repo docs/vision push that kind of guardrail into plugins/skills unless a concrete generic API gap appears, this is better closed as ClawHub/plugin scope rather than kept as a core bug.

## Evidence

- **Exec outcome is lifecycle-only:** `buildExecExitOutcome(...)` derives `completed` vs `failed` entirely from `RunExit.reason` and exit code; there is no artifact existence or path-validation check in this decision point.
  - file: [src/agents/bash-tools.exec-runtime.ts:480](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/agents/bash-tools.exec-runtime.ts#L480)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Finished-session state persists that lifecycle verdict immediately:** `runExecProcess(...)` builds the outcome, then immediately calls `markExited(session, ..., outcome.status, ...)` before notifying/persisting the finished session; no post-run artifact-gating seam exists in this path.
  - file: [src/agents/bash-tools.exec-runtime.ts:840](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/agents/bash-tools.exec-runtime.ts#L840)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Documented exec/process contract is process-status confirmation:** The background exec docs describe `exec` and `process` in terms of running/completed state, logs, and quiet-success confirmation via `process`, not durable-artifact validation.
  - file: [docs/gateway/background-process.md:25](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/gateway/background-process.md#L25)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Project vision prefers plugin scope for optional workflow policy:** `VISION.md` says core stays lean, optional capability should usually ship as plugins, and missing extensibility should be handled by extending the plugin API rather than adding one-off core behavior.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/VISION.md#L54)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Workflow-specific guardrails are explicitly directed to plugins/skills:** Task Flow docs say reusable workflows should be packaged as a ClawHub skill or plugin and that workflow-specific guardrails should stay there unless the plugin API lacks a needed generic capability. Plugin docs also show tool/hook plugins as the extension path.
  - file: [docs/automation/taskflow.md:93](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/automation/taskflow.md#L93)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Plugin docs provide the extension mechanism:** The plugin quick-start documents tool/hook plugins and states external plugins should be published via ClawHub or npm rather than added to the repo.
  - file: [docs/plugins/building-plugins.md:16](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/plugins/building-plugins.md#L16)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)

- There is no linked existing ClawHub plugin in the issue thread yet, so the workflow-specific verification still needs to be packaged by the reporter or another plugin author.
