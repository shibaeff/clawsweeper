## Summary

Current `main` partially mitigates duplicate-gateway crash loops, but it does not implement this PR's probe-based distinction between a healthy existing OpenClaw gateway and a foreign listener. The exit code still depends on `GatewayLockError` message text, and the final `EADDRINUSE` path labels any listener as "another gateway instance", so the PR's stricter `exit 0 only for confirmed healthy OpenClaw / exit 1 for foreign or failed probe` behavior is not present on `main`.

## Evidence

- **Exit code is still message-based, not probe-based:** `src/cli/gateway-cli/run.ts:341-348` defines `isHealthyGatewayLockError` as a substring check on the error message, and `src/cli/gateway-cli/run.ts:689-706` exits with code 0 whenever that matcher returns true.
  - file: [src/cli/gateway-cli/run.ts:341](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/cli/gateway-cli/run.ts#L341)
  - command: `nl -ba src/cli/gateway-cli/run.ts | sed -n '320,730p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **`EADDRINUSE` is not verified against an OpenClaw health probe:** `src/gateway/server/http-listen.ts:49-58` throws `GatewayLockError("another gateway instance is already listening on ws://...")` for any final `EADDRINUSE` after retries. That path does not inspect whether the existing listener is actually OpenClaw versus a foreign process.
  - file: [src/gateway/server/http-listen.ts:49](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/gateway/server/http-listen.ts#L49)

There is no corresponding current-main coverage for the PR's foreign-process or failed-probe exit-1 cases.
