## Summary

Current `main` already implements automatic cron retry backoff plus failure notification and run-history support. The shipped surface is not the exact per-job `retry` object proposed in the issue, but the underlying request is covered well enough to close conservatively.

## Evidence

- **Recurring cron backoff is documented on main:** `docs/cli/cron.md` explicitly states that recurring jobs use exponential retry backoff after consecutive errors: `30s -> 1m -> 5m -> 15m -> 60m`, then return to the normal schedule after the next successful run.
  - file: [docs/cli/cron.md:39](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/docs/cli/cron.md#L39)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Backoff schedule is implemented in code:** `src/cron/service/jobs.ts` defines the default cron error backoff schedule as `[30000, 60000, 5m, 15m, 60m]`, and `errorBackoffMs(...)` selects the delay from consecutive error count.
  - file: [src/cron/service/jobs.ts:41](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cron/service/jobs.ts#L41)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Scheduler applies retries after failures:** `src/cron/service/timer.ts` increments `consecutiveErrors`, schedules one-shot retries for transient failures from `cron.retry`, and applies error backoff to enabled recurring jobs by moving `nextRunAtMs` forward after failures.
  - file: [src/cron/service/timer.ts:433](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cron/service/timer.ts#L433)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Failure notification surface exists:** `src/config/types.cron.ts` exposes `cron.retry`, `failureAlert`, and `failureDestination`; `src/cron/service.failure-alert.test.ts` verifies alerts after consecutive failures and cooldown behavior; `src/cron/delivery.failure-notify.test.ts` verifies resolved delivery of failure notifications.
  - file: [src/config/types.cron.ts:6](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/config/types.cron.ts#L6)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Run attempts are persisted and inspectable:** Docs point users to `openclaw cron runs --id <job-id>`, and `src/cron/run-log.ts` defines persisted per-job run log entries under `cron/runs/<jobId>.jsonl` including status, error, delivery, and timing fields.
  - file: [src/cron/run-log.ts:18](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cron/run-log.ts#L18)

## Risks / Open Questions

- The implementation shape differs from the issue's proposed per-job `retry` block: recurring backoff is built in, one-shot retry tuning is under global `cron.retry`, and notifications are handled through `failureAlert` / `failureDestination` rather than `notifyOnFinalFailure`.
