## Summary

Current main already provides plugin-level pre-tool guard and approval seams, and the project vision explicitly keeps optional policy/governance capability out of core. This request is better scoped as a ClawHub/community plugin, not an OpenClaw core feature.

## Close Comment

Closing after a ClawSweeper/Codex review. Current `main` already exposes the extension seams needed for this, and the repo vision keeps optional governance/policy layers out of OpenClaw core. `VISION.md:54` says optional capability should usually ship as plugins. `docs/plugins/building-plugins.md:184` documents `before_tool_call` guards that can terminal-block a tool call or require approval, with approval UX through the overlay, Telegram buttons, Discord interactions, or `/approve`. That behavior is enforced in `src/agents/pi-tools.before-tool-call.ts:212`, and plugin-defined approval transport is already built into the gateway in `src/gateway/server-methods/plugin-approval.ts:26` and `docs/gateway/protocol.md:373`. OpenClaw also already has built-in host exec approvals at the execution boundary in `docs/tools/exec-approvals.md:10`. Given that scope and the existing plugin API, this belongs in ClawHub/community-plugin land rather than as another core-native guardrail layer. If a plugin author hits a missing seam, a focused issue for that specific API gap would be the right follow-up.
