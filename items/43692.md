## Summary

The proposal is useful, but current main treats this class of behavior tuning as plugin/ClawHub territory rather than a new core framework. Core already exposes model-family prompt overlays, provider-owned system-prompt contributions, and pluggable context engines, while VISION.md explicitly says optional capability should usually ship as plugins and avoids heavy orchestration layers in core.

## Close Comment

Closing as `clawhub` after a Codex review.

Current `main` points this kind of behavior-control work toward plugins/ClawHub rather than a new first-class core framework. `VISION.md:54-70` says optional capability should usually ship as plugins and new skills should go to ClawHub first, and `VISION.md:99-107` explicitly avoids heavy orchestration layers in core. The repo already exposes the seams this MVP would need: context-engine plugins can own per-run assembly and inject `systemPromptAddition` (`docs/concepts/context-engine.md:14-16`, `67-100`), provider plugins can add model-family prompt guidance through `resolveSystemPromptContribution` (`docs/concepts/system-prompt.md:13-28`, `docs/plugins/sdk-provider-plugins.md:475`), and current `main` already ships a shared GPT-5 family behavior overlay (`src/agents/gpt5-prompt-overlay.ts:50-131`, `docs/gateway/config-agents.md:471-491`).

Given that shape, per-model / per-lane behavior sheets look like a good plugin or ClawHub package pattern layered on existing hooks, not a new OpenClaw core concept. If you want to continue this, the most actionable next step is a plugin/ClawHub proposal that names the smallest missing core seam, if any.
