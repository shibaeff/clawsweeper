## Summary

Keep open. Current `main` still enqueues Slack `block_action` system events against a non-thread-aware session-key path, while Slack inbound message routing still creates `:thread:` session keys for threaded conversations. The existing interaction test suite also codifies the missing `threadTs` argument, so the reported threaded wake mismatch remains plausible and unaddressed on `main`.

## Evidence

- **Block-action enqueue still omits threadTs when resolving the target session:** `enqueueSlackBlockActionEvent` includes `threadTs` in the event payload, but the actual `resolveSlackSystemEventSessionKey(...)` call passes only `channelId`, `channelType`, and `senderId`, so the queued system event cannot target a thread-specific session key from this path.
  - file: [extensions/slack/src/monitor/events/interactions.block-actions.ts:630](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/slack/src/monitor/events/interactions.block-actions.ts#L630)
  - command: `nl -ba extensions/slack/src/monitor/events/interactions.block-actions.ts | sed -n '618,640p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **The Slack system-event session-key helper does not accept threadTs:** The `SlackMonitorContext` type and implementation for `resolveSlackSystemEventSessionKey` accept only `channelId`, `channelType`, and `senderId`. There is no thread parameter to incorporate a Slack thread into system-event routing.
  - file: [extensions/slack/src/monitor/context.ts:79](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/slack/src/monitor/context.ts#L79)
  - command: `nl -ba extensions/slack/src/monitor/context.ts | sed -n '70,110p'; nl -ba extensions/slack/src/monitor/context.ts | sed -n '172,214p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Slack inbound message routing still creates thread-scoped session keys:** Regular Slack message preparation derives `canonicalThreadId` from `threadTs` and feeds it into `resolveThreadSessionKeys(...)`, so threaded conversations can live on `...:thread:<ts>` sessions. That makes a non-thread-aware system-event route a real mismatch, not just a theoretical one.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:80](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L80)

- **Current tests encode the missing threadTs argument:** The interaction test asserts that `resolveSlackSystemEventSessionKey` is called with `{ channelId, channelType, senderId }` only, even when the payload contains `threadTs: "100.100"`. That means the present test suite still treats the non-thread-aware routing as correct behavior.
  - file: [extensions/slack/src/monitor/events/interactions.test.ts:342](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/slack/src/monitor/events/interactions.test.ts#L342)
