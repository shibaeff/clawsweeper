## Summary

Keep open. Current `main` still restores from `.bak` without checking for redacted-placeholder pollution, and backup rotation still copies the current config into `.bak` unchanged, so the issue's reported fixation path remains plausible on `b84e57fca38deacfa9834a82f7e270b32562ae1b`. The repo now has placeholder-pollution guards for `.last-good` snapshots, but that safeguard is not applied to the `.bak` restore path named in this issue.

## Evidence

- **`maybeRecoverSuspiciousConfigRead` restores `.bak` after parse/gateway checks only:** The async recovery path reads and parses `${configPath}.bak`, verifies only that the backup fingerprint has `gatewayMode`, then copies the backup over the main config. There is no call here to `collectPollutedSecretPlaceholders` or any equivalent redacted-secret validation before restore.

## Risks / Open Questions

- A polluted `.bak` can still be auto-restored into the live config, reintroducing redacted secret placeholders and causing persistent auth failures.
- `maintainConfigBackups` can still spread a polluted config into the active backup ring, reducing the chance of clean recovery from recent backups.
- There is now a `.last-good` safeguard, but it does not close the narrower `.bak` restore/rotation hole reported in this issue.
