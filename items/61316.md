## Summary

Current `main` does not include this hardening. `parseBooleanValue` still treats custom truthy/falsy lists case-sensitively and still throws on malformed non-iterable option values, so this PR addresses a real unfixed bug and is not a conservative close candidate.

## Evidence

- **Current implementation is still case-sensitive for custom lists:** `parseBooleanValue` lowercases the input string, but custom `truthy`/`falsy` values are inserted into `Set`s without lowercasing. `['YES']` therefore will not match normalized `'yes'`.
  - file: [src/utils/boolean.ts:23](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/utils/boolean.ts#L23)
  - command: `nl -ba src/utils/boolean.ts | sed -n '1,80p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Current implementation still throws on malformed non-iterable option objects:** The code constructs `new Set(falsy)` / `new Set(truthy)` for non-default overrides with no `Array.isArray` guard. A direct read-only reproduction showed `parseBooleanValue("no", { falsy: { not: "an array" } })` raising `TypeError: object is not iterable`.
  - file: [src/utils/boolean.ts:29](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/utils/boolean.ts#L29)
  - command: `node -e 'const normalize=v=>typeof v==="string"?(v.trim().toLowerCase()||undefined):undefined; const DEFAULT_TRUTHY=["true","1","yes","on"]; const DEFAULT_FALSY=["false","0","no","off"]; function parseBooleanValue(value, options={}){ if(typeof value==="boolean") return value; if(typeof value!=="string") return undefined; const normalized=normalize(value); if(!normalized) return undefined; const truthy=options.truthy ?? DEFAULT_TRUTHY; const falsy=options.falsy ?? ## Risks / Open Questions

- Part of the PR description is stale against current `main`: a malformed string override like `{ truthy: "YES" }` now returns `undefined` instead of throwing, because `new Set("YES")` is iterable. The remaining crash case is malformed non-iterable inputs such as plain objects.
- The unresolved review note is valid polish: the proposed implementation still preserves non-string entries in the temporary `Set`, so the branch likely needs a small follow-up before merge.
