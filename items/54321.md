## Summary

Current `main` already has graceful config/cron reload behavior: config changes are watched and classified into hot-reload vs restart-required paths, safe sections like `cron`, `hooks`, and `channels.*` hot-apply in process, cron store changes are re-read without a gateway restart, and restart-required changes are deferred until active replies/tasks drain instead of immediately killing in-flight work.

## Evidence

- **Config writes feed directly into reload evaluation:** The managed gateway config reloader subscribes to in-process config writes, captures the new runtime/source config snapshot, and schedules reload evaluation immediately rather than requiring a manual process restart.
  - file: [src/gateway/config-reload.ts:400](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/gateway/config-reload.ts#L400)

- **Reload planner marks cron/hooks/models as hot-reloadable:** The reload plan classifies `hooks`, `agents.defaults.*` model/heartbeat settings, `models`, and `cron` as `hot` changes with explicit reload actions, which matches the issue's core ask for config and cron changes without downtime.
  - file: [src/gateway/config-reload-plan.ts:46](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/gateway/config-reload-plan.ts#L46)

## Risks / Open Questions

- This appears to be implemented on `main` after the latest shipped release `v2026.4.23`, so released builds may still show the older restart-heavy behavior.
- OpenClaw still intentionally requires a real restart for unsafe surfaces like `gateway.*`, `plugins`, `discovery`, and `canvasHost`; the implementation is graceful deferral plus hot reload where safe, not universal live mutation.
- There is no dedicated `openclaw gateway reload` subcommand in the inspected CLI/docs; the underlying requested behavior is handled by the config watcher/write path instead.
