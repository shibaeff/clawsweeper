## Summary

Current main already contains the Control UI session-selection safeguards that address this report: the chat selector is bound to the active `sessionKey`, the active key is forced into the option list even if `sessions.list` omits it, and gateway snapshot defaults canonicalize bare `main` aliases without overwriting an explicit non-main selection. The same code is present in the shipped `v2026.4.22` tag.

## Evidence

- **Selector uses the active session key:** `ui/src/ui/chat/session-controls.ts:23-63` renders the chat-session `<select>` with `.value=${state.sessionKey}` and marks the matching `<option>` as selected, so the UI no longer has an independent selector state that can drift from the chat pane.

## Close Comment

Closing from a Codex review because current `main` already implements the behavior needed here, and the same code is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

Concrete evidence:
- `ui/src/ui/chat/session-controls.ts:36-60` binds the conversation selector directly to `state.sessionKey` and marks the matching option selected.
- `ui/src/ui/chat/session-controls.ts:498-507` always adds the active `sessionKey` back into the selector options even if `sessions.list` omits it, which prevents the selector from visually falling back to another chat like Telegram.
- `ui/src/ui/app-gateway.ts:180-250` normalizes legacy/bare `main` aliases to the canonical `sessionDefaults.mainSessionKey` while preserving an explicitly selected non-main session, and `ui/src/ui/app-gateway.ts:609-610` applies that normalization from the gateway snapshot.

That is the mismatch described in this report: the page content and selector no longer have separate effective session identities on current main / `v2026.4.22`. If you still see this on a newer build, the next useful step would be a fresh issue with exact current version, the current session keys shown in the selector, and a new screenshot/video.
