## Summary

Keep open. Current `main` still advertises `event.messages` as user-visible output for hook events, but the `message:received` path only constructs an internal hook event and passes it into `triggerInternalHook(...)` inside a fire-and-forget call. It never keeps the event object or routes `event.messages` afterward. A different command-hook path does explicitly read `hookEvent.messages` and send them, which makes the reported gap on `message:received` look real and still unresolved.
