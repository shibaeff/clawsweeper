## Summary

Keep open. The original 4x/second symptom was corrected by the reporter, but the reopened issue describes a real current-main behavior: the cron scheduler intentionally clamps its wake interval to 60 seconds instead of sleeping until the next scheduled job.

## Evidence

- **Reporter clarified the real issue:** The issue discussion first retracts the 4x/second claim, then immediately reopens with a narrower report: the scheduler polls every 60 seconds even when the next job is hours away, and asks for wake-on-next-job behavior instead.
  - command: `GitHub issue context provided in task for #61334`
- **Current scheduler intentionally clamps wakeups to 60 seconds:** `armTimer` computes the next due time with `nextWakeAtMs(state)`, but then caps the timeout with `Math.min(..., MAX_TIMER_DELAY_MS)` where `MAX_TIMER_DELAY_MS = 60_000`. The surrounding comment explicitly says to wake at least once a minute to avoid drift and recover from process pauses or wall-clock jumps.
  - file: [src/cron/service/timer.ts:43](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/cron/service/timer.ts#L43)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '620,672p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Current tests lock in the 60-second cap:** `timer.regression.test.ts` asserts that far-future schedules still arm a 60,000 ms timeout, which means current main deliberately preserves the polling behavior the reopened issue is asking to change.
  - file: [src/cron/service/timer.regression.test.ts:34](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/cron/service/timer.regression.test.ts#L34)
  - command: `nl -ba src/cron/service/timer.regression.test.ts | sed -n '33,69p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **The scheduler still tracks the real next scheduled job:** `nextWakeAtMs` returns the minimum `nextRunAtMs` across enabled jobs, so the limiting factor is not lack of schedule knowledge; it is the deliberate clamp in `armTimer`.
  - file: [src/cron/service/jobs.ts:535](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/cron/service/jobs.ts#L535)

## Risks / Open Questions

- A future wake-on-next-job change would need to preserve the existing safeguards for clock jumps, paused processes, and long-running jobs that current code and tests rely on.
