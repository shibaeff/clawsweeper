## Summary

Keep open. Current `main` still has no subagent-announce fallback for oversize completion messages: direct completion delivery only retries a narrow set of transient transport errors, then returns a failed direct delivery; when no queue fallback is available, tests expect the announce to fail; cleanup then retries three times and gives up, which matches the issue report's silent-loss behavior.

## Evidence

- **No oversize-specific fallback in direct announce delivery:** `isTransientAnnounceDeliveryError()` only treats availability/network-style failures as retryable and does not classify message-length errors; `sendSubagentAnnounceDirectly()` then returns `{ delivered: false, path: "direct" }` on any unhandled error instead of truncating or sending a minimal completion notice.
  - file: [src/agents/subagent-announce-delivery.ts:126](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/agents/subagent-announce-delivery.ts#L126)
  - command: `nl -ba src/agents/subagent-announce-delivery.ts | sed -n '120,210p'; nl -ba src/agents/subagent-announce-delivery.ts | sed -n '432,569p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Completion-mode dispatch still fails outright when queue fallback is unavailable:** Completion announces are direct-first. If direct delivery fails and the fallback queue cannot deliver, dispatch preserves the direct failure. The integration-style test also explicitly expects `runSubagentAnnounceFlow()` to return `false` for this case.
  - file: [src/agents/subagent-announce-dispatch.test.ts:90](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/agents/subagent-announce-dispatch.test.ts#L90)

  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Registry cleanup still retries then gives up after three failed announce attempts:** `finalizeSubagentCleanup()` increments `announceRetryCount`, and on `give-up` it logs `Subagent announce give up (...)` and completes cleanup anyway. The restart test covers exactly three retries followed by terminal cleanup.

## Risks / Open Questions

- Telegram now has outbound chunking support elsewhere in the repo, so some specific channel/path combinations may no longer hit this exact failure mode; but the subagent announce path itself still lacks an oversize fallback and still has a tested hard-fail path when direct delivery errors.
- Repository history is shallow in this checkout, so I could not tie the current behavior to a broader post-issue fix attempt or release note.
