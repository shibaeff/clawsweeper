## Summary

Current main already treats memory as a pluggable slot, and repo scope docs say optional capabilities like alternative backends should live as external plugins rather than new core integrations. PowerMem now exists as a standalone OpenClaw plugin (`memory-powermem`) with documented install/config flow, so this request fits the ClawHub/community plugin path instead of OpenClaw core.

## Evidence

- **Vision scope says optional capability should ship as plugins:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, plugin authors should host/maintain them in their own repositories, and memory is a special plugin slot with one active memory plugin at a time.
  - file: `VISION.md:278`
  - sha: `ef88cabe3947bf97494a0407d5ef7d56c83cc7c5`
- **Main docs define memory as plugin-owned:** `docs/concepts/memory.md` says `memory_search` and `memory_get` are provided by the active memory plugin, not hardwired core behavior.
  - file: `docs/concepts/memory.md:268`
  - sha: `ef88cabe3947bf97494a0407d5ef7d56c83cc7c5`
- **Current main already exposes the memory plugin seam:** `extensions/memory-core/index.ts` registers `memory-core` as `kind: "memory"` and calls `api.registerMemoryCapability(...)`, showing the intended extension point already exists on main.
  - file: `extensions/memory-core/index.ts:430`
  - sha: `ef88cabe3947bf97494a0407d5ef7d56c83cc7c5`
- **Plugin install flow is first-class on main:** `docs/tools/plugin.md` documents installing external plugins with `openclaw plugins install <package>` / `openclaw plugins install clawhub:<pkg>`, and lists memory plugins under the plugin system rather than as bespoke core integrations.
