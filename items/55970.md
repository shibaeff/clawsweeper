## Summary

Keep open. Current `main` still drops explicit reasons for failed inbound Telegram document downloads, and this PR's proposed fix has an unresolved multi-entry debounce gap already called out in review.

## Evidence

- **Current main still collapses getFile failures to null:** `resolveTelegramFileWithRetry` returns `null` both for Bot API file-too-large errors and for exhausted download retries, so downstream code cannot preserve a user-visible reason for document failures.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:112](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/bot/delivery.resolve-media.ts#L112)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '112,140p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Document failures are dropped before body rendering:** `resolveMedia` returns `null` when `getFile` fails, which means the inbound handler has no structured fallback payload to carry a document-unavailable reason forward.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:397](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/bot/delivery.resolve-media.ts#L397)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '397,416p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Main body builder has no unavailable-text channel:** `TelegramMessageContextOptions` on `main` has no `mediaUnavailableText` field, and `resolveTelegramInboundBody` only renders text/location plus the media placeholder. With no user text, a failed document degrades to placeholder-only output.
  - file: [extensions/telegram/src/bot-message-context.types.ts:18](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/bot-message-context.types.ts#L18)
  - command: `nl -ba extensions/telegram/src/bot-message-context.types.ts | sed -n '12,24p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Rendered body is still placeholder-only on empty-text failures:** `resolveTelegramInboundBody` falls back to `placeholder` when `rawBody` is empty and never appends an explicit failure reason on current `main`. That matches the bug described in the PR body.
  - file: [extensions/telegram/src/bot-message-context.body.ts:176](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/bot-message-context.body.ts#L176)
  - command: `nl -ba extensions/telegram/src/bot-message-context.body.ts | sed -n '176,192p'`
