## Summary

Keep open. Current `main` still accepts Claude permission replies based only on `request_id`, without checking that the reply came from the same session that originated the permission request. This PR is addressing a real, still-unfixed security bug; it should not be auto-closed. The PR discussion also contains actionable review feedback for two small hardening follow-ups, but those do not negate the underlying fix.

## Evidence

- **Permission request schema still has no session binding:** On current `main`, `ClaudePermissionRequest` only stores `toolName`, `description`, and `inputPreview`, and `ClaudePermissionRequestSchema` only requires `request_id`, `tool_name`, `description`, and `input_preview`. There is no `sessionKey`/`session_key` field to bind approvals to the originating session.
  - file: [src/mcp/channel-shared.ts:107](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/mcp/channel-shared.ts#L107)
  - command: `nl -ba src/mcp/channel-shared.ts | sed -n '100,130p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **MCP server cannot forward an originating session key:** The notification handler on current `main` forwards only `requestId`, `toolName`, `description`, and `inputPreview` into `handleClaudePermissionRequest`. Even if an upstream sender knew the session, this code path drops it before storage.
  - file: [src/mcp/channel-server.ts:42](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/mcp/channel-server.ts#L42)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Approval replies are still accepted by request ID alone:** `handleClaudePermissionRequest` stores pending Claude permissions keyed only by `requestId`, and `handleSessionMessageEvent` resolves a `yes/no <id>` reply whenever `pendingClaudePermissions.has(requestId)` is true. The inbound `sessionKey` is parsed but never compared to the stored request, so cross-session spoofing remains possible on current `main`.

There is no corresponding negative test asserting that a different session must be rejected, so current test coverage does not invalidate the reported bug.

## Risks / Open Questions

- Current `main` appears to leave a cross-session approval spoofing path open anywhere another session can send `yes/no <request_id>` for a live pending permission.
- If this PR is revived, it should likely also address the review's two hardening points: non-empty `session_key` validation and trimming/normalization consistency.
