## Summary

Keep open. Current `main` still lacks the PR's pre-seeding fix for abort reply prefix templates, so the reported bug remains plausible and the PR is not redundant.

## Evidence

- **Prefix context on main still starts without model/provider/thinking data:** `createReplyPrefixContext()` only seeds `identityName`; `model`, `modelFull`, `provider`, and `thinkingLevel` are filled later only inside `onModelSelected()`.
  - file: [src/channels/reply-prefix.ts:31](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/channels/reply-prefix.ts#L31)
  - command: `nl -ba src/channels/reply-prefix.ts | sed -n '1,120p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Fast-abort exits before normal model-selection flow:** In `dispatch-from-config.ts`, the fast-abort branch formats the abort text and returns immediately when `fastAbort.handled` is true. That path does not run reply generation first, so it has no obvious opportunity to populate the prefix context via `onModelSelected()`.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:612](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/auto-reply/reply/dispatch-from-config.ts#L612)

## Risks / Open Questions

- Closing this PR would drop a still-relevant user-facing fix: fast-abort replies can still leak literal `{model}` / `{thinkingLevel}` tokens when `messages.responsePrefix` uses templates.
- `main` also does not contain the PR's focused regression test file, so there is no current guard for this abort-prefix path.
