## Summary

Keep open. Current `main` still has the reported Discord `thread-reply` attachment gap: `filePath`/`path` are not mapped through the plugin handler or runtime for `thread-reply`, while `send` does support them.

## Evidence

- **`thread-reply` handler still ignores `path` and `filePath`:** In `extensions/discord/src/actions/handle-action.guild-admin.ts`, the `thread-reply` branch reads only `media` into `mediaUrl` before dispatching the action. There is no fallback for `path` or `filePath`, so those params are dropped on current `main`.
  - file: [extensions/discord/src/actions/handle-action.guild-admin.ts:364](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/discord/src/actions/handle-action.guild-admin.ts#L364)
  - command: `nl -ba extensions/discord/src/actions/handle-action.guild-admin.ts | sed -n '360,392p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Discord runtime still requires `mediaUrl` for `threadReply`:** In `extensions/discord/src/actions/runtime.messaging.ts`, the `threadReply` case reads only `mediaUrl` and passes that into `sendMessageDiscord`. It does not read `path` or `filePath`, so a caller that sends only those params still loses the attachment.
  - file: [extensions/discord/src/actions/runtime.messaging.ts:527](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/discord/src/actions/runtime.messaging.ts#L527)
  - command: `nl -ba extensions/discord/src/actions/runtime.messaging.ts | sed -n '520,555p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Core attachment hydration does not cover `thread-reply`:** `src/infra/outbound/message-action-params.ts` only hydrates attachment params for `sendAttachment`, `setGroupIcon`, and `upload-file`. `thread-reply` is excluded, so there is no earlier normalization step that would convert `filePath` into a `mediaUrl` for Discord.
  - file: [src/infra/outbound/message-action-params.ts:374](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/infra/outbound/message-action-params.ts#L374)
  - command: `nl -ba src/infra/outbound/message-action-params.ts | sed -n '374,400p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **`send` already supports the fallback chain the issue requests:** In contrast, `extensions/discord/src/actions/handle-action.ts` reads `media`, then `path`, then `filePath` for `send`. That makes the inconsistency reported in the issue real and still present on `main`.
  - file: [extensions/discord/src/actions/handle-action.ts:84](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/discord/src/actions/handle-action.ts#L84)
  - command: `nl -ba extensions/discord/src/actions/handle-action.ts | sed -n '70,90p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **No current test coverage for `thread-reply` attachment aliases:** A repo search for `threadReply` / `thread-reply` in Discord action tests returned no matching attachment-focused test, so there is no evidence this path was fixed and locked down by tests on current `main`.
  - command: `rg -n 'threadReply|thread-reply' extensions/discord/src/actions/*.test.ts extensions/discord/src/**/*.test.ts`
