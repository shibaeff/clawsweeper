## Summary

Current `main` already addresses both parts of #44258: Discord accounts are staggered at startup, and the Discord gateway metadata 503/plain-text failure path is treated as transient instead of crashing the gateway. The shipped `v2026.4.22` tag contains the same implementation.

## Risks / Open Questions

- The stagger is hard-coded at 10s on current `main`, not user-configurable as suggested in the issue. That is a possible follow-up enhancement, but the startup-overflow/crash bug described here is already addressed.
- This was a code-and-history review only; I did not run a live multi-bot Discord reproduction because the task required read-only inspection.

## Close Comment

Closing as implemented on `main` after Codex review. The reported startup flood is already mitigated in `extensions/discord/src/channel.ts:119` and `extensions/discord/src/channel.ts:716`, where Discord accounts are staggered by 10 seconds per account before connect. The crash path is also handled now: `extensions/discord/src/monitor/gateway-plugin.ts:64` treats Discord 503/plain-text `reset reason: overflow` gateway metadata failures as transient and falls back instead of letting invalid JSON bubble through the Carbon register path, and the transient classification is covered in `src/infra/unhandled-rejections.test.ts:133`. The same code is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
