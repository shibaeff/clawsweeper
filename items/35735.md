## Summary

Current `main` still does not ship CLI `openclaw sessions rm` / `openclaw sessions clear`. The PR covers a real gap, but it is not safely closeable because the feature is unimplemented on `main` and the proposed mutation path still needs maintainer review against current session-store maintenance behavior.

## Evidence

- **CLI surface on current main:** `registerStatusHealthSessionsCommands` defines `openclaw sessions` and `openclaw sessions cleanup`, but no `rm` or `clear` subcommands. The corresponding CLI tests only cover list forwarding and `cleanup` forwarding.

  - command: `nl -ba src/cli/program/register.status-health-sessions.ts | sed -n '124,224p'; nl -ba src/cli/program/register.status-health-sessions.test.ts | sed -n '189,249p'`
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **User-facing docs still omit delete commands:** The shipped sessions CLI docs document listing sessions and `sessions cleanup` only. There is no published `sessions rm`/`sessions clear` surface on current `main`.
  - file: [docs/cli/sessions.md:8](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/docs/cli/sessions.md#L8)

- **Current store-write semantics make the PR non-trivial to merge safely:** `updateSessionStore()` always persists through `saveSessionStoreUnlocked()`. In maintenance `enforce` mode, that save path prunes stale entries, caps entry count, and applies disk-budget cleanup on every write. A naive CLI `rm`/`clear` implementation can therefore remove more than the explicitly targeted sessions and diverge from dry-run output, which matches the open PR review concerns rather than resolving them.
