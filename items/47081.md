## Summary

Current `main` already has a Feishu-specific session-routing fix that preserves scoped conversation inheritance across restart/bootstrap paths, and the repo docs show normal chat-session persistence is gateway-backed rather than dependent on the optional `session-memory` hook.

## Close Comment

Closing from a Codex review because current `main` already contains the Feishu session-routing fix for restart/bootstrap paths. The shipped changelog entry in `CHANGELOG.md:1546` says Feishu scoped inheritance is preserved across bootstrap/model-override/restart/tool-policy paths, and the current implementation now resolves Feishu conversation ancestry through the plugin-owned session surface (`extensions/feishu/src/session-conversation.ts:27` and `extensions/feishu/src/channel.ts:1135`). Core also has pre-bootstrap bundled-plugin fallback logic in `src/channels/plugins/session-conversation.ts:153`, with regression coverage in `src/channels/plugins/session-conversation.test.ts:81` and `src/channels/plugins/session-conversation.bundled-fallback.test.ts:78`. Also, the optional `session-memory` hook is not the source of normal chat-session persistence; docs show regular session transcripts are gateway-backed (`docs/concepts/messages.md:67`) and `session-memory` only exports memory snapshots on `/new` and `/reset` (`docs/automation/hooks.md:156`). If you still see loss on a current release newer than 2026.4.2, please open a fresh issue with exact Feishu account mode, `groupSessionScope`, whether the chat is DM/group/topic, and the session key/log lines before and after reconnect.
