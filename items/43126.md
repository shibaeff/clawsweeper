## Summary

Current main already implements the requested behavior well enough: periodic heartbeats run in a single main session per agent, `heartbeat.target: "last"` is delivery routing only, forced subagent-session heartbeats are redirected back to main, and busy target-session lanes are skipped/retried instead of running in parallel.

## Evidence

## Risks / Open Questions

- This close only covers the reported `heartbeat.target: "last"` multi-session behavior; broader duplicate outbound delivery problems may still exist under other causes, including the separately open #60207.
- Automatic pruning of old normal sessions is not established by this evidence; session accumulation as a housekeeping concern appears separate from the heartbeat-target bug.

## Close Comment

Codex review: this looks implemented on current OpenClaw. The current heartbeat docs define heartbeats as periodic turns in the main session, and `target: "last"` only controls where delivery goes (`docs/gateway/heartbeat.md:13-25`). In code, heartbeat session resolution falls back to the main session for forced subagent keys (`src/infra/heartbeat-runner.ts:245-372`), `target: "last"` is handled as outbound target resolution from one session entry rather than a scan across many sessions (`src/infra/outbound/targets.ts:84-135`), and the runner skips/retries when the target session lane is already busy instead of letting overlapping heartbeats answer in parallel (`src/infra/heartbeat-runner.ts:782-794`). Those guards are covered by dedicated regression tests in `src/infra/heartbeat-runner.subagent-session-guard.test.ts:12-80` and `src/infra/heartbeat-runner.skips-busy-session-lane.test.ts:64-99`. I also verified the protections are present in the shipped release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`v2026.4.22`).
