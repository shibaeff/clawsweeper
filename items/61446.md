## Summary

Keep open. Current `main` still only checks nested `gemini-cli-core` paths and recurses downward from `gemini-cli`, so the hoisted npm global sibling layout described in PR #61446 is not implemented on `main` and is still missing direct test coverage.

## Evidence

- **Known-path lookup is still nested-only:** `readGeminiCliCredentialsFromKnownPaths` on current `main` checks only `geminiCliDir/node_modules/@google/gemini-cli-core/...` paths. It does not check the sibling hoisted layout `dirname(geminiCliDir)/gemini-cli-core/...` that this PR adds.
  - file: [extensions/google/oauth.credentials.ts:170](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/google/oauth.credentials.ts#L170)
  - command: `nl -ba extensions/google/oauth.credentials.ts | sed -n '167,190p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Fallback traversal cannot reach a sibling hoisted package:** `extractGeminiCliCredentials` falls back to `findGeminiCliCredentialsInTree(geminiCliDir, 10)`, and that helper only descends into child directories of `geminiCliDir`. A sibling `@google/gemini-cli-core` directory is therefore outside the search tree.
  - file: [extensions/google/oauth.credentials.ts:73](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/google/oauth.credentials.ts#L73)

- **Current tests do not cover the hoisted sibling layout:** The existing credential tests cover nested known-path, npm shim, bundled npm, and Homebrew layouts, but there is no test for a hoisted sibling `@google/gemini-cli-core` layout like the one described in the PR.
