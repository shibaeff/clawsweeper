## Summary

Keep open. Current `main` still persists recent tool-result image blocks, so this PR addresses a real gap rather than something already implemented or obsolete.

## Evidence

- **No built-in strip in persistence wrapper:** On `main`, `guardSessionManager()` only forwards the optional plugin `tool_result_persist` hook into `transformToolResultForPersistence`; there is no default core transform that removes base64 image payloads before transcript writes.
  - file: [src/agents/session-tool-result-guard-wrapper.ts:49](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/session-tool-result-guard-wrapper.ts#L49)
  - command: `nl -ba src/agents/session-tool-result-guard-wrapper.ts | sed -n '1,240p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Current truncation ignores image payload size:** `getToolResultTextLength()` counts only text blocks, and `truncateToolResultMessage()` explicitly keeps non-text blocks as-is. A tool result with a huge `{ type: "image", data: ... }` block is therefore not reduced by the current live tool-result cap.
  - file: [src/agents/pi-embedded-runner/tool-result-truncation.ts:226](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/pi-embedded-runner/tool-result-truncation.ts#L226)
