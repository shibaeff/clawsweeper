## Summary

Keep open. Current `main` does not implement the PR's dedicated cron-owned nested lane: cron-triggered embedded work still routes onto the shared `nested` lane, and gateway startup/hot reload still only apply `cron.maxConcurrentRuns` to `CommandLane.Cron`. The PR discussion also shows the contributor already addressed the earlier review concern about widening shared `nested` behavior and the unrealistic hot-reload test path, so this remains a substantive, unresolved fix rather than stale cleanup.
