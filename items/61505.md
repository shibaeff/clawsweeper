## Summary

Keep open. Current `main` adds Discord READY watchdogs and a regression test for a missing gateway handle, but that is only evidence that the startup hang is now detected/restarted rather than silently accepted. I could not prove from the current repo that the underlying Discord/Carbon readiness race reported here is fully fixed end-to-end.

## Evidence

- **Current startup path still depends on immediate gateway lookup after Carbon client construction:** `createDiscordMonitorClient` constructs the Carbon client and immediately reads `client.getPlugin("gateway")` to start supervision. That means the related async registration concern from the cross-referenced issue is not disproven by the current code alone.
