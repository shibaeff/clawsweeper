## Summary

Keep open. Current `main` still shows the selected auth profile label from the session override, but usage/quota lookup is resolved only by provider and generic auth-order selection, so the reported weekly usage can still come from a different OAuth credential.

## Evidence

- **Status text uses session override for auth label:** `buildStatusText()` derives `selectedModelAuth` via `resolveModelAuthLabel({ sessionEntry, ... })`, and `resolveModelAuthLabel()` passes `sessionEntry.authProfileOverride` as `preferredProfile` into auth-profile ordering. That matches the issue's observation that the displayed active OAuth profile is correct.
  - file: [src/status/status-text.ts:165](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/status/status-text.ts#L165)

  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Status usage lookup ignores session-selected profile:** The same status builder calls `loadProviderUsageSummary({ providers: [currentUsageProvider], agentDir })` without passing any explicit auth/profile selection. That means usage is fetched for the provider, not the session's chosen auth profile.
  - file: [src/status/status-text.ts:203](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/status/status-text.ts#L203)
