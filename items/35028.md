## Summary

Keep open. Current main has ACP transcript/session-file hardening, but the ensured-identity path still marks stable ACP session ids as pending, so the report’s identity-reconcile failure mode is not clearly fixed on `main`. A linked PR that explicitly says it closes #35028 is still open and unmerged.

## Evidence

- **Ensure path still forces pending identity:** `createIdentityFromEnsure()` still writes ensured ACP identities with `state: "pending"` even when the runtime handle already carries stable session ids. That matches the issue’s `identity.state: 'pending'` symptom.
  - file: [src/acp/runtime/session-identity.ts:181](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/acp/runtime/session-identity.ts#L181)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Startup reconcile path is still active for pending stable ids:** Startup reconciliation still scans ACP sessions, and it specifically retries sessions whose identity is pending but already has a stable session id. That is the same path behind the reported `acp startup identity reconcile ...` warnings.
  - file: [src/acp/control-plane/manager.core.ts:250](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/acp/control-plane/manager.core.ts#L250)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Gateway still logs reconcile counts on startup:** Gateway startup still invokes ACP pending-identity reconciliation and logs `checked/resolved/failed` counts, so the reported startup noise remains a live behavior whenever identities stay pending.
  - file: [src/gateway/server-startup-post-attach.ts:303](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/gateway/server-startup-post-attach.ts#L303)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Main now persists transcript files for ACP spawns:** `spawnAcpDirect()` resolves/persists the session transcript file during ACP spawn initialization, and the ACP spawn tests assert those transcript-resolution calls. This suggests the older `transcriptPath/sessionFile null` symptom may be partly improved, but it does not prove the issue is fully fixed.
  - file: [src/agents/acp-spawn.ts:485](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/acp-spawn.ts#L485)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Gateway creates session transcripts on session creation:** `sessions.patch` now ensures a transcript file exists and persists `sessionFile` into the session entry when creating a session. That reduces the chance of `transcriptPath` staying null, but it is separate from the unresolved ACP identity bug above.
  - file: [src/gateway/server-methods/sessions.ts:876](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/gateway/server-methods/sessions.ts#L876)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
