## Summary

Current `main` does not show a hot-reload path that rewrites existing session model selection from `agents.defaults.model.primary`, and per-turn reply selection still re-resolves `channels.modelByChannel` for sessions without an explicit override. The report does not match the current code path.

## Evidence

- **Hot reload scope:** `applyHotReload()` handles `agents.defaults.model.*` by resetting model catalog cache and updating runtime services such as heartbeat/hooks; it does not touch session-store entries or session model override fields.
  - file: [src/gateway/server-reload-handlers.ts:107](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/gateway/server-reload-handlers.ts#L107)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Per-turn channel override resolution:** Reply selection re-resolves `channels.modelByChannel` on each turn and applies that model whenever the session does not already have an explicit `modelOverride`/`providerOverride`. That means an existing Telegram session should still honor its channel pin after a config hot reload.
  - file: [src/auto-reply/reply/get-reply.ts:315](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/reply/get-reply.ts#L315)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Only persisted explicit overrides outrank channel mapping:** Stored override resolution only looks at persisted `providerOverride`/`modelOverride` on the session or parent session. It does not derive a session override from the global default model.
  - file: [src/auto-reply/reply/stored-model-override.ts:27](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/reply/stored-model-override.ts#L27)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Session reuse preserves stored override fields only:** Session initialization reuses persisted override fields from the session store when reusing an existing session; there is no path here that synthesizes a new override from `agents.defaults.model.primary` during reload.
  - file: [src/auto-reply/reply/session.ts:481](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/reply/session.ts#L481)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Release parity:** The same relevant `get-reply` and reload-handler logic is present in the latest release tag `v2026.4.23`, so this conclusion is not based on unreleased `main`-only behavior.
