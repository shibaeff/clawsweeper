## Summary

Current `main` no longer matches the report: setup now sources Gemini CLI OAuth grouping from manifest metadata, and Gemini CLI model handling is provider-specific with forward-compat coverage instead of inheriting the direct Google provider's model-id normalization.

## Evidence

- **Manifest still declares Gemini CLI OAuth in the Google group:** The bundled Google plugin manifest declares the `google-gemini-cli` auth choice with `groupId: "google"`, `groupLabel: "Google"`, and the expected choice metadata.
  - file: [extensions/google/openclaw.plugin.json:24](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/google/openclaw.plugin.json#L24)
  - command: `nl -ba extensions/google/openclaw.plugin.json | sed -n '1,60p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Current setup flow prefers manifest auth choices over runtime wizard entries:** `resolveProviderSetupFlowContributions()` now reads manifest `providerAuthChoices` first and suppresses duplicate runtime choices. That makes setup visibility come from the manifest metadata even though the runtime Gemini CLI provider still omits `groupId` in its `wizard.setup` block.
  - file: [src/flows/provider-flow.ts:125](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/flows/provider-flow.ts#L125)
  - command: `nl -ba src/flows/provider-flow.ts | sed -n '110,210p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Docs describe the manifest-first setup behavior:** The internal architecture docs explicitly state that provider setup now uses manifest `providerAuthChoices` first, then falls back to runtime wizard choices for compatibility.
  - file: [docs/plugins/architecture-internals.md:72](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/docs/plugins/architecture-internals.md#L72)
  - command: `nl -ba docs/plugins/architecture-internals.md | sed -n '68,84p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Gemini CLI does not inherit Google's normalizeModelId hook:** Provider hook resolution matches the requested provider plugin, and model-id normalization is taken only from that matched plugin. The direct `google` provider registers `normalizeGoogleModelId`, but `google-gemini-cli` does not register any `normalizeModelId` hook in its provider definition.
  - file: [src/plugins/provider-hook-runtime.ts:155](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/plugins/provider-hook-runtime.ts#L155)

  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Current Google Gemini CLI path has forward-compat model coverage:** `resolveGoogleGeminiForwardCompatModel()` contains explicit Gemini CLI handling, and tests cover current CLI cases like `gemini-2.5-flash-lite` and `gemini-3.1-pro-preview` for `google-gemini-cli`.
  - file: [extensions/google/provider-models.ts:118](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/google/provider-models.ts#L118)

Current `main` resolves the setup visibility through the manifest-first path instead of relying on the runtime `wizard.setup` metadata alone.
