## Summary

Keep open. Current main still relies on process-local/in-memory inbound dedupe for Telegram, Slack, and shared reply dispatch, and subagent announce idempotency is still keyed by `childRunId`. That does not strongly support closing a live duplicate-delivery report, especially when the issue has multiple recent reproductions on current-era releases.

## Evidence

- **Subagent announce idempotency is still run-scoped:** `buildAnnounceIdFromChildRun` still returns `v1:<childSessionKey>:<childRunId>`, so resumed/remapped runs produce fresh announce identities instead of a stable logical completion identity.
  - file: [src/agents/announce-idempotency.ts:1](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/announce-idempotency.ts#L1)
- **Tests still codify distinct announce keys per run ID:** Current e2e coverage explicitly expects separate idempotency keys for `run-1` and `run-2`, which matches the run-scoped behavior reported in the issue comments rather than proving a stable dedupe fix.
  - file: [src/agents/subagent-announce.format.e2e.test.ts:1696](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/subagent-announce.format.e2e.test.ts#L1696)
- **Telegram channel dedupe is in-memory only:** Telegram builds update keys from `update_id` / `chatId:message_id` and backs `createTelegramUpdateDedupe()` with `createDedupeCache({ ttlMs: 5 * 60_000, maxSize: 2000 })`; `bot-core` then uses `recentUpdates`, `pendingUpdateKeys`, and `activeHandledUpdateKeys`, all process-local structures.
  - file: [extensions/telegram/src/bot-updates.ts:35](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/telegram/src/bot-updates.ts#L35)
- **Shared inbound dedupe is also in-memory and is released on failure:** The global inbound dedupe cache uses `resolveGlobalDedupeCache`, and `dispatch-from-config` releases the claim when dispatch throws before completion. That means replay after retry/error paths is still an intentional current-main behavior in some failure modes.
  - file: [src/auto-reply/reply/inbound-dedupe.ts:21](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/auto-reply/reply/inbound-dedupe.ts#L21)
- **Slack message dedupe is still a short-lived in-memory cache:** Slack monitor context uses `createDedupeCache({ ttlMs: 60_000, maxSize: 500 })` for seen messages, with `markMessageSeen` / `releaseSeenMessage` managing a local cache rather than durable replay protection.
  - file: [extensions/slack/src/monitor/context.ts:155](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/slack/src/monitor/context.ts#L155)
- **No obvious post-release fix landed in the touched surfaces:** `git log a9797214338ba31c52c796adbb75afb16e0684a9..HEAD -- extensions/telegram src/agents src/auto-reply/reply extensions/slack` showed only unrelated recent commits in the shallow checkout, with no clear duplicate-message or durable-dedupe fix to cite for this issue.
  - command: `git log --oneline a9797214338ba31c52c796adbb75afb16e0684a9..HEAD -- extensions/telegram src/agents src/auto-reply/reply extensions/slack`
