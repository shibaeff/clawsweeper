## Summary

Current main already implements persistent workspace files, session persistence, and durable memory writes for the reported SOUL.md/MEMORY.md workflow; the remaining macOS-specific concern is documented as a permissions/TCC setup issue rather than a missing core feature.

## Close Comment

Closing from a Codex review because current `main` already implements the behavior this issue asks for. On `41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00`, OpenClaw has a persistent workspace at `~/.openclaw/workspace` with first-class `SOUL.md`/`MEMORY.md` handling (`src/agents/workspace.ts:17`, `src/agents/workspace.ts:342`), uses that workspace as the tool working directory, and persists session transcripts under `~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl` (`docs/concepts/agent.md:15`, `docs/concepts/agent.md:78`). Durable memory is file-backed and can be written into `MEMORY.md` via the bundled memory flow (`docs/concepts/memory.md:9`, `extensions/memory-core/src/cli.ts:152`). For macOS specifically, the current docs describe the remaining folder-access issue as a TCC/permissions setup problem and recommend using the OpenClaw workspace path to avoid Desktop/Documents/Downloads grants (`docs/platforms/mac/permissions.md:41`). Because the requested capability is already present on current main, this is being closed as implemented on main. If this still reproduces on a current build, please open a new issue with the exact failing path, whether sandboxing is enabled, and the concrete read/write error text.
