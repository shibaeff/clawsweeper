## Summary

Current `main` and shipped `v2026.4.23` already validate legacy CLI session reuse through binding-based checks and clear stale Claude CLI session state, so the scalar `--resume` bypass/persistent poisoning described in #71632 is no longer present.

## Evidence

- **Binding-based reuse validation blocks stale legacy resume:** `getCliSessionId()` delegates to `getCliSessionBinding()`, and `resolveCliSessionReuse()` invalidates reuse when auth profile, auth epoch, system prompt hash, or MCP hashes change. That means legacy scalar session IDs are no longer resumed blindly when the current boot/runtime state differs.

  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Legacy-scalar invalidation is explicitly regression-tested:** The shipped test suite covers the exact legacy-binding case and expects reuse to be invalidated when auth/prompt/MCP state changes, contradicting the reported scalar bypass on current code.
