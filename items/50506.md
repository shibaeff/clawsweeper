## Summary

Keep open. Current `main` does not implement this PR, and the proposal still has an unresolved behavioral tradeoff around `Cache-Control: no-store` on single-use media URLs.

## Evidence

- **Current media route still only sets nosniff:** `attachMediaRoutes()` sets `X-Content-Type-Options` but none of the PR's added `X-Frame-Options`, `Referrer-Policy`, `Content-Security-Policy`, or `Cache-Control` headers are present on `main`.
  - file: [src/media/server.ts:49](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/media/server.ts#L49)
  - command: `nl -ba src/media/server.ts | sed -n '39,100p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Media URLs are externally handed out:** Hosted media is exposed as `https://${hostname}/media/${saved.id}`, so this route is a real externally consumed surface, not just an internal helper.
  - file: [src/media/host.ts:54](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/media/host.ts#L54)

## Risks / Open Questions

- `/media/:id` on `main` still lacks the proposed framing/CSP/referrer hardening.
- Merging the PR as-is could regress legitimate reopen/retry flows for ephemeral media URLs because the file is deleted after first successful fetch.
- The proposed CSP still needs product/security review because `style-src 'unsafe-inline'` weakens the otherwise restrictive sandboxing stance.
