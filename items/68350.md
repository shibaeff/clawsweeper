## Summary

Keep open. Current `main` still lacks the PR's reply-routing idempotency plumbing and Discord target normalization, so it is not implemented already, and the PR discussion identifies a credible P1 regression in the proposed Telegram dedupe path for multi-chunk replies.

## Evidence

- **`routeReply` on `main` still has no reply-routing idempotency field:** Current `RouteReplyParams` does not include `idempotencyKey`, and the `deliverOutboundPayloads` call from `routeReply` does not pass one through the outbound path or transcript mirror.
  - file: [src/auto-reply/reply/route-reply.ts:41](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/auto-reply/reply/route-reply.ts#L41)
  - command: `nl -ba src/auto-reply/reply/route-reply.ts | sed -n '1,260p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Channel outbound context on `main` does not carry `idempotencyKey`:** `ChannelOutboundContext` in current `main` has no `idempotencyKey` field, so the PR's outbound-idempotency threading is not already present in the shipped core channel contract.
  - file: [src/channels/plugins/outbound.types.ts:16](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/channels/plugins/outbound.types.ts#L16)
  - command: `nl -ba src/channels/plugins/outbound.types.ts | sed -n '1,120p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Discord action handler on `main` still forwards raw `to` for `send`/`poll`:** Current `handleDiscordMessageAction` reads `to` directly for `send` and `poll`; it does not call `parseDiscordTarget(..., { defaultKind: "channel" })`, so the PR's normalization rescue is also not already implemented.
  - file: [extensions/discord/src/actions/handle-action.ts:70](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/discord/src/actions/handle-action.ts#L70)
  - command: `nl -ba extensions/discord/src/actions/handle-action.ts | sed -n '1,140p'`

- **Current delivery runtime sends long text as multiple chunked sends:** `deliverOutboundPayloadsCore` plans text units and then loops over them, calling `handler.sendText(...)` once per chunk. That makes the PR review concern material: a single shared Telegram dedupe key at the outbound base context would be reused across chunk sends.
  - file: [src/infra/outbound/deliver.ts:823](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/infra/outbound/deliver.ts#L823)

- **PR discussion already identified a concrete P1 regression in the proposed Telegram cache:** The PR review comments flag that commit `7f5f4427331cb92ce4b2ab513dfc650e65f67391` dedupes Telegram sends by `(accountId, chatId, idempotencyKey)` and would therefore drop chunk 2+ of long replies because every chunk shares the same routed idempotency key.
  - file: [extensions/telegram/src/send.ts:319](https://github.com/openclaw/openclaw/blob/7f5f4427331cb92ce4b2ab513dfc650e65f67391/extensions/telegram/src/send.ts#L319)
