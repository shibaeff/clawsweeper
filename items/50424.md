## Summary

Current main already contains Telegram-specific fixes for the reported pattern: polling no longer treats recent non-`getUpdates` send activity as a stall/restart condition, and non-idempotent Telegram sends no longer retry on ambiguous post-connect failures that can duplicate messages. The first stable release that documents the watchdog-side fix is `v2026.4.5`.

## Evidence

- **Stable release note for the restart-side fix:** `CHANGELOG.md` places the Telegram polling watchdog fix under release `2026.4.5`: “keep the watchdog from aborting long-running reply delivery by treating recent non-polling API activity as bounded liveness instead of a hard stall.”
  - file: [CHANGELOG.md:1485](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/CHANGELOG.md#L1485)
  - command: `nl -ba CHANGELOG.md | sed -n '1185,1194p'; nl -ba CHANGELOG.md | sed -n '1483,1487p'`
- **Current main liveness logic:** `TelegramPollingLivenessTracker.detectStall()` now suppresses stall detection when either recent API activity or a recent in-flight API call is within the threshold, which directly covers sends happening while polling is otherwise quiet.
  - file: [extensions/telegram/src/polling-liveness.ts:110](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/telegram/src/polling-liveness.ts#L110)
  - command: `nl -ba extensions/telegram/src/polling-liveness.ts | sed -n '1,220p'`
- **Regression coverage for send-caused restarts:** There is an explicit Telegram regression test asserting that a `sendMessage` call does not trigger a polling stall restart: `does not trigger stall restart when non-getUpdates API calls are active`.
  - file: [extensions/telegram/src/polling-session.test.ts:695](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/telegram/src/polling-session.test.ts#L695)
  - command: `nl -ba extensions/telegram/src/polling-session.test.ts | sed -n '690,860p'`
- **Duplicate-send guard on outbound retries:** Current main restricts non-idempotent Telegram send retries to pre-connect failures only via `isSafeToRetrySendError`, and the regression test `does not retry generic grammY failed-after envelopes for non-idempotent sends` verifies ambiguous network failures are not retried into duplicate visible messages.
  - file: [extensions/telegram/src/send.ts:599](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/telegram/src/send.ts#L599)

## Close Comment

Closing this as implemented on current `main` after a Codex review. The current Telegram polling code now treats recent non-`getUpdates` API activity as liveness instead of forcing a stall restart (`extensions/telegram/src/polling-liveness.ts:110`, with regression coverage at `extensions/telegram/src/polling-session.test.ts:695`). The outbound send path also no longer retries ambiguous non-idempotent send failures that can create duplicate visible messages (`extensions/telegram/src/send.ts:599`, `extensions/telegram/src/send.test.ts:1124`). The restart-side fix is documented in the stable changelog for `v2026.4.5` at `CHANGELOG.md:1485`, so there is strong evidence this report’s behavior is already addressed on maintained builds. If you still hit duplicate Telegram sends on a current release, please open a fresh issue with exact reproduction steps, OpenClaw version, Telegram account mode (polling/webhook), and logs around the restart.
