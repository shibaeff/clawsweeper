## Summary

Keep open. Current `main` does not implement the reported fix well enough: non-poll tool outcomes still hash the full `details` object, so repeated `exec` calls with the same command/output but changing volatile fields do not form a stable no-progress streak, and `genericRepeat` is still warn-only rather than escalating at `criticalThreshold`.

## Evidence

- **Non-poll outcomes still hash full details:** `hashToolOutcome()` special-cases only known poll tools. For all other tools, including `exec`, it hashes `{ details, text }` verbatim, so volatile exec metadata remains part of the result hash.
  - file: [src/agents/tool-loop-detection.ts:205](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/tool-loop-detection.ts#L205)
  - command: `nl -ba src/agents/tool-loop-detection.ts | sed -n '205,259p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **No-progress breaker requires identical args and identical result hash:** `getNoProgressStreak()` only counts history entries for the same `toolName` and same `argsHash`, and stops when `resultHash` changes. If exec result hashes differ because `details.durationMs`/`pid`/similar fields change, the breaker never accumulates.
  - file: [src/agents/tool-loop-detection.ts:288](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/tool-loop-detection.ts#L288)
  - command: `nl -ba src/agents/tool-loop-detection.ts | sed -n '288,316p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Generic repeat is still warning-only:** For non-poll tools, `detectToolCallLoop()` has only a warning path for `genericRepeat`; there is no critical/blocking path at `criticalThreshold` for repeated identical exec calls.
  - file: [src/agents/tool-loop-detection.ts:541](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/tool-loop-detection.ts#L541)
  - command: `nl -ba src/agents/tool-loop-detection.ts | sed -n '541,559p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Quick hash reproduction shows volatile fields change the outcome hash:** A direct `node -e` reproduction of the same stable-stringify/digest pattern used here produced different hashes for two otherwise identical exec-style outcomes that differed only in volatile fields like `durationMs`/`pid` (`equal: false`).
  - command: `node -e 'const {createHash}=require("node:crypto"); const stable=v=>v===null||typeof v!=="object"?JSON.stringify(v):Array.isArray(v)?`[${v.map(stable).join(",")}]`:`{${Object.keys(v).sort().map(k=>`${JSON.stringify(k)}:${stable(v[k])}`).join(",")}}`; const digest=v=>createHash("sha256").update(stable(v)).digest("hex"); const text=`{"id":2,"name":"Loki"}`; const hash=details=>digest({details,text}); const a=hash({status:"completed",exitCode:0,durationMs:136,pid:11,cwd:"/tmp"}); const b=hash({status:"completed",exitCode:0,durationMs:141,pid:12,cwd:"/tmp"}); console.log(JSON.stringify({a,b,equal:a===b},null,2));'`

The broken hashing therefore affects real tool-loop enforcement, not dead code.

## Risks / Open Questions

- The broader 'creative retry, same failure result' case from later comments also remains open, because current no-progress tracking is keyed to the same `argsHash`.
- Even if volatile exec fields were normalized, the current generic-repeat path would still not block at `criticalThreshold`; only the global no-progress breaker can hard-stop non-poll repeats.
