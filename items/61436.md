## Summary

Current `main` does not implement path-specific skill loading. The issue’s token-cost rationale is partly outdated because OpenClaw now injects a bounded skill catalog rather than eagerly loading every `SKILL.md` body, but there is still no `paths:` frontmatter support or file-pattern/file-access-based eligibility logic. This remains a plausible core feature request rather than a cleanup close.

## Evidence

- **No `paths:` field in parsed skill model:** Skill frontmatter is stored as a flat `Record<string, string>`, and the parsed metadata/invocation surface only resolves install/os/requires/homepage/skillKey plus invocation flags. There is no typed or parsed `paths` field.
  - file: [src/agents/skills/types.ts:68](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/agents/skills/types.ts#L68)
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Frontmatter resolver lacks path-pattern handling:** `resolveOpenClawMetadata()` only maps `always`, `emoji`, `homepage`, `skillKey`, `primaryEnv`, `os`, `requires`, and `install`; `resolveSkillInvocationPolicy()` only handles `user-invocable` and `disable-model-invocation`. No file-glob/path eligibility is parsed.
  - file: [src/agents/skills/frontmatter.ts:187](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/agents/skills/frontmatter.ts#L187)
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Eligibility is runtime/config based, not file-context based:** `shouldIncludeSkill()` filters by per-skill enablement, bundled allowlist, OS, bins, env, and config truthiness via `evaluateRuntimeEligibility()`. The eligibility context has only remote platform/bin data, not accessed-file paths or glob matching.
  - file: [src/agents/skills/config.ts:73](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/agents/skills/config.ts#L73)
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Current main mitigates prompt bloat but still catalogs eligible skills:** `resolveWorkspaceSkillPromptState()` still builds a prompt from all eligible skills, then applies compact/truncated formatting under prompt budgets. This reduces token use, but it is not path-specific loading.
  - file: [src/agents/skills/workspace.ts:768](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/agents/skills/workspace.ts#L768)
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **System prompt reads at most one skill on demand:** The prompt instructs the model to scan the `<available_skills>` descriptions first and read only one matching `SKILL.md` up front. That means the issue body overstates current behavior, but the requested `paths:` feature is still absent.
  - file: [src/agents/system-prompt.ts:162](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/agents/system-prompt.ts#L162)
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Docs document only coarse-grained visibility controls:** User-facing docs document agent skill allowlists and load-time gating (`requires`, OS, env, config), but they do not document any `paths:` frontmatter or file-pattern activation.
  - file: [docs/tools/skills-config.md:54](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/docs/tools/skills-config.md#L54)
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)

## Risks / Open Questions

- The issue body claims all skills are loaded on every request; current main instead injects a bounded catalog and reads skill bodies lazily, so the problem statement is directionally right but technically overstated.
- Because the report is a feature request, not a stale bug report, there is no strong cleanup basis for closing it without an explicit product/scope decision.
