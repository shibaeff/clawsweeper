## Summary

Keep open. Current `main` still has a plausible lost-update race for concurrent `config.patch`/`config.apply` requests: gateway requests run concurrently, each config write handler validates against one snapshot and later writes a full config derived from that snapshot, and the write path does not reject if the file changed in between.

## Evidence

- **Gateway requests can overlap on one connection:** The WebSocket handler dispatches each request in a detached async task (`void (async () => { await handleGatewayRequest(...) })()`), so a second config write can begin before the first finishes.
  - file: [src/gateway/server/ws-connection/message-handler.ts:1504](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/server/ws-connection/message-handler.ts#L1504)
  - command: `nl -ba src/gateway/server/ws-connection/message-handler.ts | sed -n '1500,1516p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **config.patch computes from one snapshot, then writes later:** `config.patch` reads `readConfigFileSnapshotForWrite()`, checks `baseHash`, merges against `snapshot.config`, validates, and finally calls `writeConfigFile(validated.config, writeOptions)`. There is no second conflict check between the merge/validate step and the final write.
  - file: [src/gateway/server-methods/config.ts:475](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/server-methods/config.ts#L475)
  - command: `nl -ba src/gateway/server-methods/config.ts | sed -n '471,580p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **config.apply has the same stale-write shape:** `config.apply` also snapshots once, checks `baseHash` once, and later writes `parsed.config` directly via `writeConfigFile(parsed.config, writeOptions)`.
  - file: [src/gateway/server-methods/config.ts:633](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/server-methods/config.ts#L633)
  - command: `nl -ba src/gateway/server-methods/config.ts | sed -n '629,653p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Write path does not enforce the original base hash:** `writeConfigFile` re-reads the current snapshot (`readConfigFileSnapshotInternal`) to prepare persistence, but it does not compare that snapshot to the handler's earlier hash or raise a conflict before persisting the caller-provided config.
  - file: [src/config/io.ts:1609](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/config/io.ts#L1609)
  - command: `nl -ba src/config/io.ts | sed -n '1603,1618p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)

## Risks / Open Questions

- A concurrent `config.patch`/`config.apply` pair that starts from the same hash can still race and let the later stale full write clobber the earlier update.
- The newer `baseHash` guard reduces stale retries after a write completes, but it does not close the in-flight overlap window described in the issue.
