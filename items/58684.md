## Summary

Current `main` does not already implement this memory-write guard, so there is no strong basis to close as fixed or obsolete. The PR’s intent still matches current docs, but the proposed implementation is not closure-safe because current trigger semantics use `user`/`manual`, and the main tool assembly still leaves other mutating paths unguarded.

## Risks / Open Questions

- The underlying data-integrity concern may still exist for writable non-main sessions or host-mode runs.
- The PR as written appears bypassable because current main still exposes separate `apply_patch` and sandboxed `write`/`edit` paths.
- The proposed `human` trigger check would likely regress legitimate interactive runs on current main, which uses `user` and `manual`.
