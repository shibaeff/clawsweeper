## Summary

Keep open. Current `main` mitigates the retry storm with bounded `LiveSessionModelSwitchError` retries and cron error backoff, but it does not implement the issue's requested semantics: isolated cron still derives state from the provided `sessionKey`, preserves prior per-session overrides on forced-new sessions, and documents persisted cron-session model overrides plus retry-on-switch as intended behavior.

## Evidence

- **Isolated cron still resolves from the provided session key:** `runCronIsolatedAgentTurn` uses `input.sessionKey` (or `cron:<job.id>`) as the base session key, resolves an agent session key from it, and passes that key into `resolveCronSession(..., forceNew: input.job.sessionTarget === "isolated")`. An isolated job bound to an existing session key still reads that session entry first.
  - file: [src/cron/isolated-agent/run.ts:459](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/cron/isolated-agent/run.ts#L459)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '455,493p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Forced-new isolated sessions intentionally preserve prior overrides:** `resolveCronSession` loads `store[params.sessionKey]` and then builds the new session entry with `...entry` even when `forceNew` created a fresh `sessionId`. The inline comment explicitly says to preserve existing per-session overrides when rolling to a new session.

The CLI docs likewise say cron persists the switched provider/model before retrying.
