## Summary

Current main clearly has auto-compaction, tool-pair-preserving compaction boundaries, and pre-compaction memory flush. But #7052 is broader than that: it asks for proactive preservation before a session freezes and calls out reset/orphaned-tool-result recovery. Current main still appears to have unresolved reset/recovery gaps, so this is not strong enough for an `implemented_on_main` close.

## Evidence

- **Auto-compaction is present on main:** Current docs say OpenClaw compacts older conversation into a persisted summary so the chat can continue, runs auto-compaction by default when the session nears or exceeds the context window, retries after overflow, and can run a silent memory flush before compaction.
  - file: `docs/concepts/compaction.md:2`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Compaction preserves tool-use/result pairs:** The compaction deep dive says chunk boundaries keep assistant tool calls paired with matching `toolResult` entries, and auto-compaction triggers both on overflow recovery and on threshold maintenance (`contextTokens > contextWindow - reserveTokens`).
  - file: `docs/reference/session-management-compaction.md:17`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Compaction and memory-flush config exists:** The config reference exposes `agents.defaults.compaction` with `reserveTokensFloor`, `notifyUser`, and `memoryFlush` settings, which confirms the proactive compaction feature family is implemented and configurable on current main.
  - file: `docs/gateway/configuration-reference.md:109`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Reset still looks like a separate unresolved gap:** Current docs describe `/new` and `/reset` as creating a new `sessionId`, and the startup-context section covers post-reset startup injection, not a pre-reset agentic save/checkpoint path.
  - file: `docs/reference/session-management-compaction.md:10`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Current issue tracker still treats pre-reset preservation as open work:** Open issue #45608 explicitly describes the compaction-time `runMemoryFlushIfNeeded` path as existing, but says `/new`, `/reset`, and daily reset still lack the same agentic pre-reset flush. That matches the unresolved part of #7052 rather than a fully implemented mainline fix.
- **Current issue tracker still has reset/compaction recovery failures:** Open issue #40295 reports that timed-out compaction can still block `/new` and `/reset` on the same session lane, which means the 'prevent brain freeze / clean recovery' goal from #7052 is not fully achieved on current main.
- **Tool-pair corruption is not fully retired:** Open issues #37834 and #42112 show the orphaned tool-use/tool-result corruption family still exists in some failure paths, so the reset/orphan-cleanup portion of #7052 is not clearly finished well enough to close conservatively.
