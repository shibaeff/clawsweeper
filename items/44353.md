## Summary

Keep open. Current `main` still appears to treat the reported Bedrock provider-level validation failure as a non-failover error, so the configured fallback chain would not reliably trigger for the issue’s example.

## Evidence

- **Fallback only happens for classified failover errors:** `src/agents/pi-embedded-runner/run.ts:1408-1517` only escalates prompt-side failures into `FailoverError` when `classifyFailoverReason(...)` or `isFailoverErrorMessage(...)` recognizes the provider error; otherwise it surfaces the prompt error directly.
  - file: [src/agents/pi-embedded-runner/run.ts:1408](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/agents/pi-embedded-runner/run.ts#L1408)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '1390,1575p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Current Bedrock plugin hook is narrow:** `extensions/amazon-bedrock/register.sync.runtime.ts:419-426` classifies only `ThrottlingException` as `rate_limit` and `ModelNotReadyException` as `overloaded`. It does not classify the issue’s reported Bedrock message about raw model IDs no longer supporting on-demand throughput / requiring an inference profile ID.
  - file: [extensions/amazon-bedrock/register.sync.runtime.ts:419](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/amazon-bedrock/register.sync.runtime.ts#L419)

- **Generic provider-side fallback patterns do not cover the reported error:** `src/agents/pi-embedded-helpers/provider-error-patterns.ts:54-76` only ships generic Bedrock-specific failover patterns for throttling and model-not-ready, plus separate context-overflow handling. Search of failover code found no matcher for `Invocation of model ID ... on-demand throughput is not supported` / `inference profile` wording.
