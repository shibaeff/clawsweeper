## Summary

Keep open. Current `main` still treats Telegram group/topic sessions as non-main sessions for runtime policy, and sandbox docs explicitly describe group/channel sessions as sandboxed under `agents.defaults.sandbox.mode: "non-main"`. Elevated access is configured only per provider allowlist, not per Telegram topic/session type, so the requested capability is not already implemented and the issue is coherent.

## Evidence

- **Runtime policy keeps non-DM Telegram topic sessions on their own session key:** `resolveRuntimePolicySessionKey()` only rewrites main-session aliases for `ChatType === "direct"`; non-direct sessions return the original session key unchanged, so a Telegram group topic stays a non-main session for sandbox policy.
  - file: [src/auto-reply/reply/runtime-policy-session-key.ts:107](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/auto-reply/reply/runtime-policy-session-key.ts#L107)

  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Non-main sessions are sandboxed by runtime policy:** `resolveSandboxRuntimeStatus()` uses `shouldSandboxSession()`, which sandboxes every session whose comparable key differs from the agent main session when mode is `non-main`. Topic sessions therefore remain sandboxed unless config changes.
