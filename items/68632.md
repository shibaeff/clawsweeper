## Summary

Current `main` already handles env-backed Telegram `SecretRef` values in the `agents list` read-only/status path and downgrades unresolved secret reads to non-fatal `not configured` status. The same implementation is present in the shipped `v2026.4.23` release, so the reported crash path does not match current OpenClaw behavior.

## Evidence

- **`agents list` uses read-only account inspection and catches unresolved SecretRef errors:** `agentsListCommand` builds provider status through `buildProviderStatusIndex`, which prefers `inspectAccount` for read-only status surfaces and converts unresolved SecretRef resolution failures into `state: "not configured"` instead of aborting the command.
  - file: [src/commands/agents.commands.list.ts:102](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/commands/agents.commands.list.ts#L102)
  - command: `nl -ba src/commands/agents.commands.list.ts | sed -n '70,130p'; nl -ba src/commands/agents.providers.ts | sed -n '54,110p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Telegram read-only inspection resolves env SecretRefs from process env:** `inspectTelegramAccount` explicitly inspects env-backed SecretRefs in read-only paths, resolves allowed env refs from `process.env`, and marks unavailable refs as `configured_unavailable` instead of throwing.
  - file: [extensions/telegram/src/account-inspect.ts:67](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/telegram/src/account-inspect.ts#L67)
  - command: `nl -ba extensions/telegram/src/account-inspect.ts | sed -n '40,120p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Telegram runtime token resolution no longer uses a strict unresolved path for env refs:** `resolveTelegramToken` first calls `resolveSecretInputString(..., mode: "inspect")`, then resolves env-backed SecretRefs via `resolveEnvSecretRefValue`; only non-env SecretRefs stay strict. That does not match the report's older `normalizeResolvedSecretInputString`/`assertSecretInputResolved` stack.
  - file: [extensions/telegram/src/token.ts:52](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/telegram/src/token.ts#L52)
  - command: `nl -ba extensions/telegram/src/token.ts | sed -n '20,110p'; rg -n "normalizeResolvedSecretInputString|assertSecretInputResolved" src extensions packages -g '!**/dist/**'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Regression tests cover env-backed SecretRefs in both runtime and read-only status paths:** Tests assert that Telegram runtime token resolution returns the env-backed SecretRef value from `process.env`, and that read-only status inspection also resolves env templates without crashing.
  - file: [extensions/telegram/src/token.test.ts:232](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/telegram/src/token.test.ts#L232)
