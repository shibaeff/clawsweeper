## Summary

Keep open. Current `main` does not implement filename override plumbing for the generic message send flow, so this PR's core request is still unmet; at the same time, the PR as provided is too broad and carries unresolved risk rather than providing strong close evidence.

## Evidence

- **Current main still drops filename in send flow:** `handleSendAction` accepts `media`/`mediaUrl`/`path`/`filePath` aliases, but the eventual `executeSendAction(...)` call forwards only message/media/reply/thread fields and no `filename`, so caller-provided attachment names are not threaded through the normal send path.

- **Outbound payload/context types on main have no filename field:** The reply payload and outbound channel context types on `main` do not carry a `filename`, which matches the missing plumbing above and shows the request is not already implemented in current core delivery types.
