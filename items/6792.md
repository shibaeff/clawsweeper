## Summary

Keep open. Current `main` does not implement manifest-level `configPatch` for plugin install. The only `configPatch` support I found is the existing provider-auth return shape, while the manifest type, manifest docs, and plugin install flow still lack any corresponding manifest field or install-time merge/apply path.

## Evidence

- **Manifest type has no `configPatch` field:** In current `main`, `PluginManifest` enumerates manifest fields such as `id`, `configSchema`, `enabledByDefault`, `autoEnableWhenConfiguredProviders`, `channels`, `providers`, `providerDiscoveryEntry`, and related metadata, but no `configPatch` field appears.
  - file: `src/plugins/manifest.ts:10`
  - command: `GitHub current-main source inspection`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Manifest docs do not document install-time config patching:** `docs/plugins/manifest.md` describes `openclaw.plugin.json` as pre-runtime metadata for discovery, config validation, onboarding/setup hints, and UI hints, and explicitly says not to use it for npm install metadata. There is no `configPatch` entry or install-time merge behavior documented.
  - file: `docs/plugins/manifest.md:4`
  - command: `GitHub current-main docs inspection`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Plugin install flow has no `configPatch` handling:** Current install surfaces show no manifest `configPatch` plumbing: `src/plugins/install.ts` contains no `configPatch` references, and the CLI path in `src/cli/plugins-cli.ts` exposes `plugins install` without any corresponding apply/merge output such as the issue’s proposed 'Applied config patch' step.
  - file: `src/plugins/install.ts`
  - command: `GitHub current-main source inspection`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Existing `configPatch` support is still provider-auth only:** `src/plugins/types.ts` still defines `ProviderAuthResult.configPatch?: Partial<OpenClawConfig>`, which matches the issue’s prior-art note and reinforces that the missing piece is manifest-level support, not a general config-patch primitive.
  - file: `src/plugins/types.ts:4`
  - command: `GitHub current-main source inspection`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
- **Issue-linked PR does not establish shipped behavior:** The issue timeline references closed PR #43421 (`Plugins: apply manifest config patches on install`), but the current `main` source/docs above do not contain the feature, so there is no strong basis to close this as already implemented.
  - command: `Issue timeline review`
  - sha: `db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`
