## Summary

Keep open. Current `main` already lets plugins declare some SecretRef-capable config paths, but secret providers themselves are still a core-only `env|file|exec` union and the supported core SecretRef surface remains explicitly enumerated in core/docs. The feature request for plugin-implemented secret-ref providers that can broadly cover core schemas is not implemented on `main`.

## Evidence

## Risks / Open Questions

- The request mixes two distinct changes: plugin-defined secret provider backends and broader SecretRef coverage for core config surfaces. Those should likely be scoped separately before implementation.
- Some plugin-owned config SecretRef support already exists, so closing as implemented would overstate current behavior and miss the unresolved core-provider extensibility gap.
