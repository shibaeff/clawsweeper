## Summary

Keep open. Current `main` does not implement this PR, and the submitted refactor appears to weaken the managed-collection repair path rather than prove an existing fix. The PR needs maintainer review or revision, not conservative cleanup closure.

## Evidence

- **Current main still forces full collection rebuild during repair:** `rebuildManagedCollectionsForRepair()` on `main` still unconditionally removes and re-adds every managed collection before logging success. That means the PR's proposed `await this.ensureCollections()` swap is not already implemented upstream.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:998](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/memory-core/src/memory/qmd-manager.ts#L998)
  - command: `nl -ba extensions/memory-core/src/memory/qmd-manager.ts | sed -n '988,1018p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Current tests intentionally require remove+add on repair scenarios:** The `main` test suite still asserts that null-byte and duplicate-document repair paths remove both managed collections and then add them back. The PR patch changed those expectations to `removeCalls === []`, which conflicts with the behavior currently codified on `main`.
  - file: [extensions/memory-core/src/memory/qmd-manager.test.ts:1460](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/memory-core/src/memory/qmd-manager.test.ts#L1460)
  - command: `nl -ba extensions/memory-core/src/memory/qmd-manager.test.ts | sed -n '1299,1468p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **ensureCollections skips healthy-looking listed collections:** On `main`, `ensureCollections()` consults `listCollectionsBestEffort()` and skips remove/add whenever `shouldRebindCollection()` returns false for matching path and pattern. Replacing the dedicated repair loop with `ensureCollections()` can therefore turn a repair into a no-op when QMD still reports collection metadata normally.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:479](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/memory-core/src/memory/qmd-manager.ts#L479)

  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **PR discussion already identified the same regression risk:** The provided PR review discussion includes both Greptile and Codex review comments calling out that delegating repair to `ensureCollections()` may be a no-op for duplicate-document constraint failures when `qmd collection list --json` still returns matching metadata. Greptile also flagged unquoted YAML output for `name` and `path` in the proposed `index.yml` writer.
