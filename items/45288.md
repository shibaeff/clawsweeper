## Summary

Keep open. Current `main` does not implement the reported gateway-status SSRF validation, and the open PR should not be closed as already fixed. The proposed PR also has a credible unresolved gap for IPv4-mapped IPv6 metadata addresses.

## Evidence

- **Current main still accepts arbitrary ws/wss hosts:** On `main` at `028b6c9b13c960b6f59a7643aff4f6de04e07f71`, `normalizeWsUrl()` only trims the string and checks for `ws://` / `wss://`, then returns it unchanged. `resolveTargets()` uses that helper for both explicit probe URLs and `cfg.gateway.remote.url`, so there is no hostname/IP SSRF filtering in the current implementation.
  - file: [src/commands/gateway-status/helpers.ts:74](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/commands/gateway-status/helpers.ts#L74)
  - command: `nl -ba src/commands/gateway-status/helpers.ts | sed -n '70,95p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Shared SSRF helper already covers the missing address classes:** `isPrivateIpAddress()` in the shared SSRF module rejects special-use IPv4, blocked IPv6, and IPv4-mapped IPv6 by calling `extractEmbeddedIpv4FromIpv6()`. That is the repo's existing mechanism for catching metadata-style addresses beyond plain dotted IPv4.
  - file: [src/infra/net/ssrf.ts:137](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/infra/net/ssrf.ts#L137)
  - command: `nl -ba src/infra/net/ssrf.ts | sed -n '137,199p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Repo tests already treat IPv4-mapped metadata addresses as blocked:** `src/infra/net/ssrf.test.ts` includes `::ffff:169.254.169.254` and `0:0:0:0:0:ffff:a9fe:a9fe` in `privateIpCases`, which supports the review feedback that any gateway-status SSRF fix should also block those forms.
  - file: [src/infra/net/ssrf.test.ts:21](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/infra/net/ssrf.test.ts#L21)
  - command: `nl -ba src/infra/net/ssrf.test.ts | sed -n '1,40p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **The PR's proposed guard has an unresolved bypass shape:** The PR discussion flags that a `169.254.` string-prefix check misses IPv4-mapped IPv6 URLs. Local URL parsing matches that concern: `node -e "console.log(new URL('ws://[::ffff:169.254.169.254]:80').hostname)"` prints `[::ffff:a9fe:a9fe]`, so a plain dotted-prefix filter would not catch it.
  - command: `node -e "console.log(new URL('ws://[::ffff:169.254.169.254]:80').hostname)"`

## Risks / Open Questions

- Closing this PR would discard an unfixed security issue: current `main` still has no gateway-status hostname/IP validation.
- Treating the PR as effectively done would be unsafe because the proposed patch, as reviewed, does not fully cover IPv4-mapped IPv6 metadata endpoints.
