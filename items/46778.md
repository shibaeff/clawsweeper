## Summary

Keep open. Current `main` still deduplicates Feishu inbound messages by `message_id` only, and I did not find a content-based fallback or coverage for the reported same-content/different-`message_id` case.

## Evidence

- **Persistent dedupe remains message-id only:** `claimUnprocessedFeishuMessage`, `finalizeFeishuMessageProcessing`, `recordProcessedFeishuMessage`, and `hasProcessedFeishuMessage` all normalize and store only `messageId`; there is no alternate key derived from sender, timestamp, or content.
  - file: [extensions/feishu/src/dedup.ts:44](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/feishu/src/dedup.ts#L44)
  - command: `nl -ba extensions/feishu/src/dedup.ts | sed -n '1,141p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **In-flight replay guard is also keyed only by message_id:** `resolveEventDedupeKey()` builds the in-memory processing claim key as `${namespace}:${messageId}`. If Feishu delivers the same logical message with a different `message_id`, this guard will not treat it as a duplicate.
  - file: [extensions/feishu/src/processing-claims.ts:6](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/feishu/src/processing-claims.ts#L6)
  - command: `nl -ba extensions/feishu/src/processing-claims.ts | sed -n '1,59p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Debounce path does not suppress same-content different-id duplicates:** The inbound debounce merge helper explicitly dedupes entries only by `message.message_id`. On flush, surviving entries are combined by joining their text with `"\n"` and dispatching the last event, so two copies of the same text with different ids would still flow through rather than being collapsed as one message.
  - file: [extensions/feishu/src/monitor.message-handler.ts:111](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/feishu/src/monitor.message-handler.ts#L111)
  - command: `nl -ba extensions/feishu/src/monitor.message-handler.ts | sed -n '111,299p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Current regression coverage only proves same-id dedupe:** The Feishu bot test covers concurrent duplicate suppression for the same image `message_id`. I did not find a test that exercises identical inbound content arriving under different `message_id` values.
  - file: [extensions/feishu/src/bot.test.ts:2699](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/feishu/src/bot.test.ts#L2699)
  - command: `nl -ba extensions/feishu/src/bot.test.ts | sed -n '2699,2729p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
