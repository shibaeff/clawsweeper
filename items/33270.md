## Summary

Keep open. Current `main` already has the Discord `autoPresence` runtime health signaling described in the issue, but it does not implement a Discord `self-profile`/`set-profile` action or the proposed `channels.discord.actions.selfProfile` gate, so the request is only partially landed.

## Evidence

- **Discord discovery exposes presence only:** Discord message-tool discovery adds `set-presence` behind the `presence` action gate and does not add `set-profile`.
  - file: [extensions/discord/src/channel-actions.ts:153](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/discord/src/channel-actions.ts#L153)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Discord runtime has no self-profile handler:** The Discord action dispatcher only routes messaging, guild, moderation, and `setPresence`; there is no `setProfile` or self-profile branch.
  - file: [extensions/discord/src/actions/runtime.ts:55](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/discord/src/actions/runtime.ts#L55)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Message-action entrypoint maps only set-presence:** The external Discord message-action handler recognizes `set-presence` and forwards it to `setPresence`; there is no corresponding `set-profile` mapping.
  - file: [extensions/discord/src/actions/handle-action.ts:301](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/discord/src/actions/handle-action.ts#L301)

  - file: [docs/channels/discord.md:856](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/channels/discord.md#L856)
