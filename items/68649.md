## Summary

Current `main` has lower-level idle-timeout plumbing for remote media reads, but the PDF tool path does not enable it, and the `/new`/`/restart` recovery paths still do not clear session queues or abort the active embedded run. The reported zombie-session failure mode is therefore not clearly fixed on `main` and should stay open.

## Evidence

- **Idle timeout exists but is opt-in:** `fetchRemoteMedia` only applies stalled-body protection when `readIdleTimeoutMs` is provided; the timeout is optional in the fetch options and only passed into `readResponseWithLimit` through that field.
  - file: [src/media/fetch.ts:45](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/media/fetch.ts#L45)
  - command: `nl -ba src/media/fetch.ts | sed -n '38,57p'; nl -ba src/media/fetch.ts | sed -n '223,233p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Web media loader does not expose or pass the idle timeout:** `WebMediaOptions` has no `readIdleTimeoutMs`, and the remote-URL branch calls `fetchRemoteMedia` with `url`, `maxBytes`, `ssrfPolicy`, `dispatcherPolicy`, and `trustExplicitProxyDns`, but no stalled-read timeout.
  - file: [src/media/web-media.ts:42](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/media/web-media.ts#L42)

- **PDF tool remote fetch path still goes through loader without timeout:** For remote PDFs, `createPdfTool()` loads the document through `loadWebMediaRaw(...)` with `maxBytes`, `localRoots`/`ssrfPolicy`, but no fetch idle-timeout parameter. That means the PDF tool still uses the untimed remote-media path.
