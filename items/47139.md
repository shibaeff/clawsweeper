## Summary

Current `main` already preserves workspace-installed skills as `source: "openclaw-workspace"` with `bundled: false`, and the Control Panel groups those entries under Workspace. The reported bundled/workspace mix-up is already covered by shipped code and a regression test.

## Evidence

- **Workspace loader tags workspace skills correctly:** `loadWorkspaceSkillEntries` loads the workspace skills directory with `source: "openclaw-workspace"`, so ClawHub-installed skills living in the workspace path carry the workspace source into status rendering.
  - file: [src/agents/skills/workspace.ts:574](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/skills/workspace.ts#L574)
  - command: `nl -ba src/agents/skills/workspace.ts | sed -n '572,577p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Status builder no longer marks workspace overrides as bundled:** `buildWorkspaceSkillStatus` sets `bundled` only when `skillSource === "openclaw-bundled"` or when the source is `unknown` and the name matches a bundled skill. A `source` of `openclaw-workspace` does not become bundled.
  - file: [src/agents/skills-status.ts:190](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/skills-status.ts#L190)
  - command: `git show v2026.4.22:src/agents/skills-status.ts | nl -ba | sed -n '188,194p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Regression test covers the reported scenario:** The test `does not mark an overridden workspace skill as bundled by bundled name alone` asserts `source === "openclaw-workspace"` and `bundled === false` for a workspace skill that shares a name with a bundled skill.
  - file: [src/agents/skills.buildworkspaceskillstatus.test.ts:133](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/skills.buildworkspaceskillstatus.test.ts#L133)
  - command: `git show v2026.4.22:src/agents/skills.buildworkspaceskillstatus.test.ts | nl -ba | sed -n '133,158p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Control Panel grouping uses workspace source:** The UI groups `openclaw-workspace` entries into `Workspace Skills`; with the current status payload (`bundled: false` for workspace overrides), the Control Panel renders them in the expected group.
  - file: [ui/src/ui/views/skills-grouping.ts:10](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/ui/src/ui/views/skills-grouping.ts#L10)
  - command: `git show v2026.4.22:ui/src/ui/views/skills-grouping.ts | nl -ba | sed -n '9,27p'`

## Risks / Open Questions

- `ui/src/ui/views/skills-grouping.ts` still prefers `skill.bundled` over `skill.source` for malformed payloads, so a future regression in status generation could reintroduce the symptom. The current backend path and regression test prevent that combination for workspace overrides.

## Close Comment

Closing from a Codex review because this is already fixed on current `main`, and the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) contains the relevant behavior. Workspace-installed skills are loaded with `source: "openclaw-workspace"` in `src/agents/skills/workspace.ts:574`, and `buildWorkspaceSkillStatus` only marks a skill as bundled when the source is `openclaw-bundled` or `unknown` in `src/agents/skills-status.ts:190`. There is also a regression test in `src/agents/skills.buildworkspaceskillstatus.test.ts:133` that explicitly covers a workspace skill overriding a bundled skill name and asserts `source === "openclaw-workspace"` and `bundled === false`. With that payload, the Control Panel groups the entry under Workspace via `ui/src/ui/views/skills-grouping.ts:10`. If someone can still reproduce this on a newer build, they should open a fresh issue with the exact version and a current screenshot/status payload.
