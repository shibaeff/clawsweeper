## Summary

Keep open. Current `main` still encodes the Teams personal-DM media gap the report describes: DM fallback paths are gated on HTML `<attachment>` tags, while the docs still say personal DM attachments/images work with Teams bot file APIs and RSC-only setups.

## Evidence

- **DM fallback is still HTML-gated on main:** `resolveMSTeamsInboundMedia` only enters the Bot Framework DM fallback when `hasHtmlFileAttachment` is true, and only enters the Graph fallback when that same HTML gate is true and the conversation is not treated as a Bot Framework personal chat. A DM payload without `<attachment id="...">` tags still has no fallback path here.
  - file: [extensions/msteams/src/monitor-handler/inbound-media.ts:62](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/msteams/src/monitor-handler/inbound-media.ts#L62)
  - command: `nl -ba extensions/msteams/src/monitor-handler/inbound-media.ts | sed -n '1,220p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Tests lock in the skip behavior for DM messages without attachment tags:** The current test suite explicitly expects a personal DM whose HTML lacks `<attachment>` tags to skip both the Bot Framework and Graph media fallbacks. That matches the reported inline-image/attachment gap rather than disproving it.
  - file: [extensions/msteams/src/monitor-handler/inbound-media.test.ts:235](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/msteams/src/monitor-handler/inbound-media.test.ts#L235)
  - command: `nl -ba extensions/msteams/src/monitor-handler/inbound-media.test.ts | sed -n '1,340p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Graph fallback for `a:` DMs depends on Graph chat-id resolution:** The message handler first tries to translate `a:` DM conversation IDs into a Graph chat ID before calling `resolveMSTeamsInboundMedia`. That path is best-effort and failure is only logged, so RSC-only setups cannot rely on it to recover missing DM media.
  - file: [extensions/msteams/src/monitor-handler/message-handler.ts:554](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/msteams/src/monitor-handler/message-handler.ts#L554)
  - command: `nl -ba extensions/msteams/src/monitor-handler/message-handler.ts | sed -n '552,602p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Graph chat-id resolution requires Graph token/API access:** `resolveGraphChatId` calls `tokenProvider.getAccessToken(GRAPH_SCOPE)` and queries Graph `/me/chats`. That is a Graph-dependent recovery path, not an RSC-only DM attachment path.
  - file: [extensions/msteams/src/graph-upload.ts:293](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/msteams/src/graph-upload.ts#L293)
