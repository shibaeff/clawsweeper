## Summary

Current `main` already implements Telegram TTS/native voice-message delivery: Telegram is treated as a native voice-note channel, `/tts audio` preserves `audioAsVoice`, Telegram delivery routes compatible audio through `sendVoice`, and MP3/Ogg/M4A are explicitly accepted as voice-compatible. The same behavior is present in the shipped `v2026.4.22` tag.

## Close Comment

Closing from a Codex review because current OpenClaw already implements this behavior. Telegram is treated as a native voice-note TTS channel in `extensions/speech-core/src/tts.ts:594` and successful TTS replies preserve `audioAsVoice` in `extensions/speech-core/src/tts.ts:1171`. The `/tts audio` command also forwards that flag in `src/auto-reply/reply/commands-tts.ts:167`. On the Telegram side, delivery routes compatible audio through `sendVoice` rather than `sendAudio` in `extensions/telegram/src/bot/delivery.replies.ts:349` and `extensions/telegram/src/send.ts:924`, and shared media compatibility explicitly allows MP3/Ogg/M4A in `src/media/audio.ts:4`, so Edge/Microsoft MP3 output still takes the voice-message path. There is regression coverage in `extensions/telegram/src/send.test.ts:1332` and `extensions/telegram/src/bot/delivery.test.ts:435`, and the same behavior is present in the shipped `v2026.4.22` tag. If you still see file-attachment behavior on a current build, please open a fresh issue with the exact provider/channel config and a minimal reproduction.
