## Summary

The request is a valid integration idea, but current main already treats web search/fetch providers as plugin capabilities and routes search setup through the plugin registry. Under the repo’s scope rules, Parallel belongs as a ClawHub/community plugin rather than another bundled core integration.

## Risks / Open Questions

- Closing as scope means there is still no maintained Parallel plugin today; someone would need to publish and support it externally.
- The current setup evidence is strongest for web_search; a Parallel plugin would still need to implement its own web_fetch provider metadata and UX.

## Close Comment

Closing this as a ClawHub/plugin-scope request after Codex review of current `main`. OpenClaw already treats `web_search` and `web_fetch` as plugin capabilities rather than core-only integrations: the repo vision says optional capability should usually ship as plugins and that the bar for adding optional plugins to core is intentionally high (`VISION.md:54-60`). The plugin SDK already supports external providers registering both `registerWebFetchProvider(...)` and `registerWebSearchProvider(...)` and publishing that work to ClawHub (`docs/plugins/sdk-provider-plugins.md:612-689`). Current search setup also pulls provider choices from the runtime plugin registry and auto-enables the selected plugin (`src/flows/search-setup.ts:86-100`, `src/flows/search-setup.ts:200-209`), so a Parallel integration fits the existing plugin path instead of needing a bundled core addition.
