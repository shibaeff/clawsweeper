## Summary

Keep open. Current main does not implement `browser.autoVisualInput`, and the proposed PR shape is not strong enough for conservative cleanup: it would add an unsupported `VISUAL_INPUT:` marker, it would read screenshot bytes from a path returned by the gateway even though `openclaw browser` supports remote gateways, and it would mislabel some screenshot payloads because browser screenshots can be normalized to JPEG server-side.

## Evidence

- **Current screenshot CLI only emits MEDIA path and supports jpeg output:** On current main, `openclaw browser screenshot` accepts `--type <png|jpeg>` and, in non-JSON mode, only logs `MEDIA:${shortenHomePath(result.path)}`. There is no existing visual-input directive path here, so the PR would be introducing new output semantics rather than matching an existing contract.
  - file: [extensions/browser/src/cli/browser-cli-inspect.ts:18](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/browser/src/cli/browser-cli-inspect.ts#L18)
  - command: `nl -ba extensions/browser/src/cli/browser-cli-inspect.ts | sed -n '1,80p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Screenshot RPC returns a saved file path, not inline bytes:** The browser screenshot route normalizes the image, saves it via `saveMediaBuffer`, and returns JSON containing an absolute `path`. A CLI-side `readFile(result.path)` assumes that returned path is readable from the CLI host.
  - file: [extensions/browser/src/browser/routes/agent.snapshot.ts:139](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/browser/src/browser/routes/agent.snapshot.ts#L139)
  - command: `nl -ba extensions/browser/src/browser/routes/agent.snapshot.ts | sed -n '132,175p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Remote gateway/browser-node flows are first-class:** The browser CLI docs explicitly support controlling another machine, and the browser gateway request path can proxy browser-node results before returning them. That means screenshot paths are not guaranteed to be local CLI paths, so a PR that reads `result.path` from the CLI side is unsafe for remote setups.
  - file: [docs/cli/browser.md:4](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/docs/cli/browser.md#L4)
  - command: `nl -ba docs/cli/browser.md | sed -n '1,30p'; nl -ba extensions/browser/src/gateway/browser-request.ts | sed -n '174,232p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Main can transcode screenshots to JPEG regardless of requested type:** `normalizeBrowserScreenshot()` may return `{ contentType: "image/jpeg" }`, and the route uses that normalized content type when saving the screenshot. A hardcoded `VISUAL_INPUT:image/png` would therefore be wrong not only for `--type jpeg`, but also for oversized PNG captures normalized to JPEG.
  - file: [extensions/browser/src/browser/screenshot.ts:35](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/browser/src/browser/screenshot.ts#L35)

- **Current agent/output pipeline recognizes MEDIA, not VISUAL_INPUT:** The system prompt documents `MEDIA:<path-or-url>` as the attachment directive, and the media parser extracts only `MEDIA:` lines from text. I found no `VISUAL_INPUT` support on current main, so the PR's new marker is not wired into the existing output pipeline.
