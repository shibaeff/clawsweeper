## Summary

Keep open. Current `main` still uses the non-blocking/cached context-token lookup in the cron persistence path and the read-only status path, so the behavior described in PR #48334 is not implemented on `main`. The review discussion also surfaced follow-up correctness concerns in the proposed status-repair logic, so this is not a conservative cleanup close.

## Evidence

- **Cron persistence still uses sync cached lookup + 200k fallback:** `finalizeCronRun()` still computes `contextTokens` from `prepared.agentCfg?.contextTokens ?? lookupContextTokens(modelUsed, { allowAsyncLoad: false }) ?? persisted entry ?? DEFAULT_CONTEXT_TOKENS`. That is the cold-cache fallback path described in the PR, and it persists the result back onto the session entry.
