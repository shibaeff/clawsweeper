## Summary

Keep open. Current `main` does not contain the Discord teardown drain/managed-identity work from this PR, so it is not `implemented_on_main`. The PR also has an unresolved correctness finding on duplicate bot IDs overwriting the proposed managed-identity map, so it is not a safe conservative auto-close.

## Evidence

- **Current teardown still stops immediately:** `monitorDiscordProvider()` still only calls `deactivateMessageHandler?.()` in `finally` and then tears down gateway/thread bindings. There is no idle-drain step or managed bot identity cleanup on current `main`, so the PR's core behavior is not already shipped.

The shared `KeyedAsyncQueue` also has no `waitForIdle()` helper on current `main`, so the PR's drain mechanism is absent rather than superseded.
