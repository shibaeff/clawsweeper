## Summary

Current `main` does not implement the reported fix. Cron delivery routing still only accepts string `delivery.channel` values, so an object-shaped value like `{ id: "telegram" }` will be normalized away and delivery can still fall back to `last`, matching the Feishu/Telegram misroute described in #46899.

## Evidence

- **Current routing only accepts string channels:** `resolveCronDeliveryPlan()` calls `normalizeChannel()`, and `normalizeChannel()` only lowercases plain string input. Non-string values return `undefined`, so announce mode falls back to `last`.
  - file: [src/cron/delivery-plan.ts:21](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/cron/delivery-plan.ts#L21)
  - command: `nl -ba src/cron/delivery-plan.ts | sed -n '1,180p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Persisted cron jobs are normalized on load:** Cron store loading runs every persisted job through `normalizeCronJobInput(raw)` before execution, so malformed persisted `delivery.channel` values do affect runtime behavior.
  - file: [src/cron/service/store.ts:39](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/cron/service/store.ts#L39)
