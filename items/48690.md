## Summary

Keep open. Current `main` still has no task-level lane timeout, so the reported lane-jam failure mode is not implemented away, and the PR still contains useful work. The review discussion also surfaced real timeout-budget contract questions for agent and cron callers, so this is not a high-confidence cleanup close.

## Evidence

- **Current queue still blocks forever on a never-settling task:** `drainLane()` only clears `activeTaskIds` and re-enters `pump()` after `await entry.task()` resolves or rejects. There is no timeout or third completion path, so a hung promise still leaves the lane occupied indefinitely.
  - file: [src/process/command-queue.ts:189](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/process/command-queue.ts#L189)
  - command: `nl -ba src/process/command-queue.ts | sed -n '155,225p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **No timeout API exists on current enqueue surface:** `enqueueCommandInLane()` still accepts only `warnAfterMs` and `onWait`; there is no `taskTimeoutMs` or equivalent caller-controlled lane timeout on `main`. A direct search for `CommandLaneTaskTimeoutError`, `taskTimeoutMs`, and `lane task timed out` returned no matches in the queue code/tests.
  - file: [src/process/command-queue.ts:242](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/process/command-queue.ts#L242)
