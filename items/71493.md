## Summary

Keep open. Current `main` still uses first-match LAN address selection for `gateway.bind=lan` without de-prioritizing Docker/bridge interfaces, and the issue is brand new with an active linked fix PR (#71503) rather than an implementation already on `main`.

## Evidence

- **Dashboard/status LAN picker still falls back to any external IPv4:** `pickPrimaryLanIPv4()` still delegates to `pickMatchingExternalInterfaceAddress(..., { family: "IPv4", preferredNames: ["en0", "eth0"] })`. If those preferred names are absent, selection falls through to the first matching external IPv4 in the interface snapshot, with no bridge/container-interface filtering.
  - file: [src/gateway/net.ts:23](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/gateway/net.ts#L23)
  - command: `nl -ba src/gateway/net.ts | sed -n '1,40p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Underlying helper returns first matching external address:** `pickMatchingExternalInterfaceAddress()` iterates `Object.entries(snapshot)` and, after optional preferred-name checks, returns `addresses.find((entry) => matches(entry.address))?.address`. There is no logic here to de-prioritize `docker0`, `br-*`, `veth*`, `cni*`, `podman*`, or `virbr*` interfaces.
  - file: [src/infra/network-interfaces.ts:71](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/infra/network-interfaces.ts#L71)
