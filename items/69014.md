## Summary

Current `main` already routes `sessions_send` nested work onto per-session nested lanes instead of one shared `nested` lane, which directly addresses the reported cross-agent head-of-line blocking.

## Evidence

- **Changelog records the fix:** The changelog entry for `2026.4.19-beta.2` says nested agent work is now scoped per target session so a long-running nested run on one session no longer blocks unrelated sessions across the gateway (`#67785`).
  - file: [CHANGELOG.md:682](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/CHANGELOG.md#L682)

  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Nested lane helper is per-session:** `resolveNestedAgentLaneForSession()` now returns `nested:<sessionKey>` when a session key is present, rather than sending all nested work through one shared `nested` lane.
  - file: [src/agents/lanes.ts:18](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/lanes.ts#L18)

- **sessions_send uses the per-session nested lane:** The main `sessions_send` dispatch path passes `lane: resolveNestedAgentLaneForSession(resolvedKey)` into the gateway agent call.
  - file: [src/agents/tools/sessions-send-tool.ts:281](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/tools/sessions-send-tool.ts#L281)

- **A2A follow-up steps also use per-session nested lanes:** The follow-up reply and announce steps both call `runAgentStep()` with `resolveNestedAgentLaneForSession(...)`, so the nested A2A flow no longer shares one global nested lane across unrelated sessions.
  - file: [src/agents/tools/sessions-send-tool.a2a.ts:95](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/tools/sessions-send-tool.a2a.ts#L95)
