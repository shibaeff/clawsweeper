## Summary

Keep open. Current `main` has Telegram transport-level polling-stall recovery and session-store cleanup tooling, but the reported failure is narrower: a Telegram session can stop replying while the channel probe still looks healthy. I did not find code or docs on `main` showing session-level stall detection, session-state inspection for this condition, or an automatic recovery path that matches the issue request. The existing workaround from the discussion (`openclaw sessions cleanup --enforce` plus `/new`) is documented as store/transcript maintenance, not as a proven fix for a stuck Telegram session loop.

## Evidence
