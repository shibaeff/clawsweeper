## Summary

Keep open. Current `main` still creates backup tarballs by remapping only archive entry paths, while `backup verify` validates only `entry.path` values and never inspects hardlink `linkpath` targets. Upstream `tar@7.5.13` computes hardlink targets relative to `cwd`, so `..` link targets are still plausible for workspace content outside the current directory. The linked backup/workspace issues in the timeline are consistent with this surface still being unsettled.

## Evidence

- **Backup creation rewrites only entry paths:** `createBackupArchive()` calls `tar.c()` with `portable: true` and `preservePaths: true`; its `onWriteEntry` callback assigns only `entry.path = remapArchiveEntryPath(...)` and does not touch `entry.linkpath` or disable hardlink emission.
  - file: [src/infra/backup-create.ts:353](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/infra/backup-create.ts#L353)
  - command: `nl -ba src/infra/backup-create.ts | sed -n '330,380p'`
  - sha: [443b837bd521](https://github.com/openclaw/openclaw/commit/443b837bd521b0485c87bb7359f370a4ccec10fa)
- **Verifier ignores hardlink targets:** `backup verify` collects only `entry.path` from `tar.t()`, then normalizes and validates only those paths. No code reads or validates `entry.linkpath`, so a tarball can pass verification while containing an unsafe hardlink target.
  - file: [src/commands/backup-verify.ts:169](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/commands/backup-verify.ts#L169)
  - command: `nl -ba src/commands/backup-verify.ts | sed -n '165,250p'`
  - sha: [443b837bd521](https://github.com/openclaw/openclaw/commit/443b837bd521b0485c87bb7359f370a4ccec10fa)
- **Docs overstate restore safety relative to this case:** The backup docs say `openclaw backup verify` rejects traversal-style archive paths and checks manifest payload presence. That matches the implementation, but it does not cover hardlink target traversal, which is the bug reported here.
  - file: [docs/cli/backup.md:25](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/docs/cli/backup.md#L25)
  - command: `nl -ba docs/cli/backup.md | sed -n '1,120p'`
  - sha: [443b837bd521](https://github.com/openclaw/openclaw/commit/443b837bd521b0485c87bb7359f370a4ccec10fa)
- **Tests do not cover hardlink-linkpath traversal:** Current verification tests cover unsafe archive paths and duplicate entry paths, but there is no test for a hardlink entry whose `linkpath` contains `..` and still passes `backup verify`.
  - file: [src/commands/backup-verify.test.ts:300](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/commands/backup-verify.test.ts#L300)
  - command: `nl -ba src/commands/backup-verify.test.ts | sed -n '240,340p'`
  - sha: [443b837bd521](https://github.com/openclaw/openclaw/commit/443b837bd521b0485c87bb7359f370a4ccec10fa)
- **Upstream tar still derives hardlink targets relative to cwd:** In `tar@7.5.13`, `onWriteEntry` runs before header encoding, and hardlink entries set `this.linkpath = path.relative(this.cwd, linkpath)`. With OpenClaw not overriding `cwd` or rewriting `entry.linkpath`, `..` hardlink targets remain possible when backing up paths outside the current working directory.
  - command: `curl -L --max-time 20 https://raw.githubusercontent.com/isaacs/node-tar/v7.5.13/src/write-entry.ts | nl -ba | sed -n '180,340p'`

## Risks / Open Questions

- I did not run a live macOS `bsdtar` extraction because this review was constrained to read-only inspection, so the conclusion is based on current source and upstream dependency behavior rather than a fresh end-to-end repro.
- The current backup implementation was introduced very recently on `main`, so maintainers may still choose a different fix shape than the reporter suggested (rewrite hardlinks, dereference them, or extend verification).
