## Summary

Keep open. Current `main` still fires `gateway:startup` from startup handoff timing, not from channel transport readiness. Telegram startup only becomes `connected` after polling succeeds, so the reported race remains plausible, and a recent issue comment reports the same symptom on a newer release.

## Evidence

- **`gateway:startup` still fires on a fixed timer after `startChannels()`:** `startGatewaySidecars()` schedules the internal `gateway:startup` hook with `setTimeout(..., 250)` immediately after `startChannels()` returns; there is no readiness/connected check before dispatch.
  - file: [src/gateway/server-startup-post-attach.ts:276](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-startup-post-attach.ts#L276)
  - command: `nl -ba src/gateway/server-startup-post-attach.ts | sed -n '275,290p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Gateway startup waits for channel start handoff, not channel readiness:** `startChannels()` only awaits `startChannel(plugin.id)` for each plugin. That is the barrier used by startup sidecars before `onSidecarsReady()` and later hook execution paths proceed.
  - file: [src/gateway/server-channels.ts:598](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-channels.ts#L598)
  - command: `nl -ba src/gateway/server-channels.ts | sed -n '598,618p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **`startChannel()` marks channels running before the long-lived account task is actually ready:** The channel manager sets `running: true`, creates the `startAccount(...)` task, stores it, then resolves the startup gate in `finally` without awaiting task completion. That means channel startup can report done before transport readiness is proven.
  - file: [src/gateway/server-channels.ts:401](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-channels.ts#L401)
  - command: `nl -ba src/gateway/server-channels.ts | sed -n '398,520p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Telegram only flips to connected after polling succeeds:** Telegram `startAccount` delegates to the monitor, and the polling status publisher sets `connected: false` on polling start and only sets connected state on `notePollSuccess()`. So Telegram adapter readiness happens after the startup hook barrier.
  - file: [extensions/telegram/src/polling-status.ts:11](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/polling-status.ts#L11)
  - command: `nl -ba extensions/telegram/src/channel.ts | sed -n '865,919p' && nl -ba extensions/telegram/src/polling-status.ts | sed -n '9,35p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Recent public repro exists after the original report:** Issue comment from `argus-as` on 2026-04-24 reports the same startup-drop behavior on OpenClaw `2026.4.21`, including a first outbound Telegram message reported as sent but not delivered.
