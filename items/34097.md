## Summary

Current `main` treats first-class mcporter/MCP client integration as out of OpenClaw core scope and points optional integrations like this to the plugin/ClawHub path. The issue itself already demonstrates a viable plugin-based solution.

## Close Comment

Closing as a ClawHub/plugin-scoped request after a Codex review. Current `main` explicitly keeps first-class MCP runtime out of OpenClaw core when `mcporter` already provides the bridge path (`VISION.md:72-83`, `VISION.md:99-106`). The current MCP docs also clarify that `openclaw mcp list/show/set/unset` only manage OpenClaw-owned `mcp.servers` config rather than auto-discovering external mcporter registries (`docs/cli/mcp.md:10-20`, `docs/cli/mcp.md:356-383`). For optional capabilities like this, the project directs authors to community plugins published on ClawHub or npm, and the plugin SDK already supports registering tools via `api.registerTool(...)` (`docs/plugins/community.md:9-22`, `docs/plugins/building-plugins.md:95-135`). Since this issue already includes a working MCP bridge plugin proof of concept, the maintainers’ current scope points to shipping that as a plugin rather than adding native core integration.
