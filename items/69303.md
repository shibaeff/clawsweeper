## Summary

Keep open. Current `main` still releases the inbound dedupe claim on the catch path without writing a durable dedupe mark, and the test suite explicitly preserves retry/replay of the same `MessageSid` after a dispatch failure.

## Evidence

- **Catch path still drops the dedupe claim instead of persisting it:** `dispatchReplyFromConfig` commits the dedupe key on success, but in the `catch` block it still calls `releaseInboundDedupe(...)`. That matches the reported replay path rather than a fix.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:1183](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/auto-reply/reply/dispatch-from-config.ts#L1183)
  - command: `nl -ba src/auto-reply/reply/dispatch-from-config.ts | sed -n '1178,1200p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Release path still does not stamp the durable cache:** `commitInboundDedupe` calls `cache.check(key, ...)` before clearing `inFlight`, while `releaseInboundDedupe` only deletes the in-flight key. There is still no third terminal state such as the proposed `poisoned` state.
  - file: [src/auto-reply/reply/inbound-dedupe.ts:114](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/auto-reply/reply/inbound-dedupe.ts#L114)
