## Summary

Current main already ships generic webhook ingress for wake and agent/session routing, covering the requested external webhook-to-session automation without adding the exact `/hooks/event` shape from the issue.

## Evidence

- **Generic hook endpoints exist in runtime:** Gateway HTTP handling already accepts `POST /hooks/agent` and generic mapped `POST /hooks/<name>` requests, validates optional `sessionKey`, and dispatches the hook run.
  - file: [src/gateway/server-http.ts:663](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/server-http.ts#L663)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)

  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)

## Risks / Open Questions

- The implementation uses `/hooks/agent`, `/hooks/wake`, and mapping-driven `/hooks/<name>` routes rather than the exact `/hooks/event` request shape proposed in the issue.

## Close Comment

Closing from a Codex review because current `main` already implements this capability and it was shipped in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The repo now has generic webhook ingress via `POST /hooks/agent`, `POST /hooks/wake`, and mapping-based `POST /hooks/<name>` routes, with documented `sessionKey` routing controls in `docs/gateway/configuration-reference.md:468` and runtime handling in `src/gateway/server-http.ts:663` plus `src/gateway/server/hooks.ts:43`. The behavior is also covered by gateway tests for request and mapping session keys in `src/gateway/server.hooks.test.ts:322` and `src/gateway/server.hooks.test.ts:335`. This does not match the exact `/hooks/event` payload proposed here, but it covers the same webhook-to-session/agent automation use case on released OpenClaw builds.
