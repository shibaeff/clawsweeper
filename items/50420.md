## Summary

Keep open. Current `main` still has a live chat path where assistant finals are emitted or synthesized without `usage`/`model` metadata, and the Control UI footer intentionally renders nothing when both are missing. That matches the reported inconsistent status bar behavior rather than proving a fix on `main`.

## Evidence

- **Renderer hides footer metadata without usage/model:** `extractGroupMeta()` returns `null` when an assistant group has neither `usage` nor `model`, and `renderMessageMeta()` renders nothing for `null`. The footer therefore disappears for otherwise normal assistant replies that arrive without those fields.
  - file: [ui/src/ui/chat/grouped-render.ts:392](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/ui/src/ui/chat/grouped-render.ts#L392)
  - command: `nl -ba ui/src/ui/chat/grouped-render.ts | sed -n '380,499p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **UI still synthesizes bare assistant messages from stream text:** On `chat` final events with no usable `message`, `handleChatEvent()` appends a synthetic assistant message containing only `role`, `content`, and `timestamp`. That synthetic message has no `usage` or `model`, so the footer cannot render.
  - file: [ui/src/ui/controllers/chat.ts:456](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/ui/src/ui/controllers/chat.ts#L456)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '451,470p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Behavior is intentional and still covered by tests:** The current test suite explicitly expects streamed text to be persisted when a final event carries no message, confirming this no-metadata path remains part of current behavior.
  - file: [ui/src/ui/controllers/chat.test.ts:178](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/ui/src/ui/controllers/chat.test.ts#L178)
  - command: `nl -ba ui/src/ui/controllers/chat.test.ts | sed -n '178,204p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Gateway still emits final chat payloads without usage/model metadata:** `server-chat.ts` builds `state: "final"` chat payloads whose `message` only includes `role`, text `content`, and `timestamp`. No `usage`, `cost`, `context`, or `model` fields are attached on that path.
  - file: [src/gateway/server-chat.ts:826](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/gateway/server-chat.ts#L826)
  - command: `nl -ba src/gateway/server-chat.ts | sed -n '826,840p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)

## Risks / Open Questions

- There may be other chat-final paths that do include metadata, so the inconsistency can depend on which gateway path produced the reply.
- A later history reload could backfill metadata for some sessions, but the current live-update path still allows footerless assistant messages on `main`.
