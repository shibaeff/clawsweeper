## Summary

Keep open. Current `main` still has a plausible Control UI bug path: when the browser is reconnecting with a cached device token and hits `AUTH_DEVICE_TOKEN_MISMATCH`, the browser client does not clear the stale cached device token and does not classify that error as terminal, so it can keep reconnecting until the gateway rate-limit path trips. Parts of the report are outdated or overstated (`instanceId` churn does not match current Control UI wiring), but the core stale-device-token loop remains credible on `main`.

## Evidence

- **Browser mismatch path still auto-reconnects:** `GatewayBrowserClient` only suppresses reconnect after the one-shot `AUTH_TOKEN_MISMATCH` shared-token retry path. A browser-side `AUTH_DEVICE_TOKEN_MISMATCH` falls through to `scheduleReconnect()`, so the client can continue retrying after a stale device-token failure.
  - file: [ui/src/ui/gateway.ts:341](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/ui/src/ui/gateway.ts#L341)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '334,350p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Stale browser device token is only cleared when shared-token fallback exists:** The browser client clears cached device auth on `AUTH_DEVICE_TOKEN_MISMATCH` only when `plan.selectedAuth.canFallbackToShared` is true. In `selectConnectAuth`, that flag is only set when there is both a stored device token and an explicit gateway token, so device-token-only Control UI sessions keep the stale cached token.
  - file: [ui/src/ui/gateway.ts:503](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/ui/src/ui/gateway.ts#L503)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '500,618p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Control UI still prefers explicit shared auth over cached device auth:** Current browser auth selection does not prefer stale local device auth over the dashboard token: `resolvedDeviceToken` is only used when there is no explicit shared token/password, and `canFallbackToShared` is derived from `explicitGatewayToken`. That means the issue's precedence claim is not fully accurate, but it also highlights that the remaining bug is specifically the device-token-only reconnect path.
  - file: [ui/src/ui/gateway.ts:588](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/ui/src/ui/gateway.ts#L588)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '588,618p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Node client already has the stale-token cleanup that the browser client lacks:** The Node gateway client explicitly clears persisted device auth on `1008` `device token mismatch` closes when device-token auth was active, and there is a regression test for that behavior. That asymmetry makes the missing browser-side cleanup look like a real gap rather than intended behavior.
  - file: [src/gateway/client.ts:315](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/client.ts#L315)

So the report's incrementing `instanceId` values are not explained by current Control UI reconnect logic alone.
