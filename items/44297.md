## Summary

Keep open. Current `main` still treats Slack external arg-menu registration failure as a silent verbose-only fallback, with no visible channels-status/doctor signal for operators.

## Evidence

- **Verbose-only fallback remains:** `registerSlackMonitorSlashCommands` catches `app.options()` failure, flips `supportsExternalArgMenus = false`, and only calls `logVerbose(...)`; there is no non-verbose warning or status emission in that path.
  - file: [extensions/slack/src/monitor/slash.ts:817](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/slack/src/monitor/slash.ts#L817)
  - command: `nl -ba extensions/slack/src/monitor/slash.ts | sed -n '780,860p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Slack status surface has no issue collector for this degradation:** Slack's status adapter is created without a `collectStatusIssues` hook, so `channels status` has no plugin-owned warning path for this fallback even though the generic status pipeline supports it.
  - file: [extensions/slack/src/channel.ts:488](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/slack/src/channel.ts#L488)

  - file: [extensions/slack/src/doctor.ts:50](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/slack/src/doctor.ts#L50)
