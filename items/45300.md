## Summary

Current `main` already hardens Discord preflight against the reported failure mode by REST-hydrating inbound messages when raw mention tokens are present but Carbon did not populate mention metadata, then applying mention gating with the account-specific bot identity. The latest tagged release does not appear to contain that code yet, so this looks fixed on `main` but not in `v2026.4.22`.

## Close Comment

Closing from a Codex review: current `main` already implements the missing-mention fallback behind this report. In `extensions/discord/src/monitor/message-handler.preflight.ts:327`, Discord preflight now REST-hydrates inbound messages when raw mention tokens like `<@...>` are present but the event payload has no populated mention metadata, then merges the fetched `mentions` back into the message before mention gating runs. The actual `no-mention` drop still happens later in the same file (`extensions/discord/src/monitor/message-handler.preflight.ts:917` onward), after mention state is resolved from that hydrated payload. Multi-account routing is also account-scoped here: `extensions/discord/src/monitor/provider.ts:998` fetches `botUserId` per account and passes it into the message handler at `extensions/discord/src/monitor/provider.ts:1049`, so mention checks use the owning bot identity instead of one shared id. There is regression coverage on `main` in `extensions/discord/src/monitor/message-handler.preflight.test.ts:598` asserting that a message with raw bot mention syntax and empty `mentionedUsers` is accepted after hydration. I could not confirm this implementation in release `v2026.4.22`, so this looks fixed on `main` but not yet attributable to a shipped release here.
