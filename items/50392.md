## Summary

Keep open. Current `main` does not include this PR's core `/bash` exec-policy inheritance change, so there is no strong basis to close it as already implemented or obsolete. The reported behavior is still plausible from code inspection: `/bash` builds `createExecTool(...)` without forwarding configured `tools.exec` policy, while the exec tool itself falls back to approval defaults when those fields are omitted.

## Evidence

- **`/bash` still omits exec policy fields on current main:** In `handleBashChatCommand`, the `createExecTool(...)` call only forwards `timeoutSec`, `sessionKey`, and notify/elevated settings. It does not pass `security`, `ask`, `pathPrepend`, `safeBins`, `safeBinTrustedDirs`, `safeBinProfiles`, or `approvalRunningNoticeMs` on current `main`.
  - file: [src/auto-reply/reply/bash-command.ts:350](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/auto-reply/reply/bash-command.ts#L350)

  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Missing fields materially affect behavior:** `createExecTool` resolves `security` and `ask` from `defaults?.security` / `defaults?.ask`, and if omitted falls back to `loadExecApprovals().defaults` before host defaults. That means `/bash` not threading configured `tools.exec` policy can still change approval behavior.
