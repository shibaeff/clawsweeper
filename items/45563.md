## Summary

Current main already contains shipped Kimi web_search auth and request-flow fixes that address the reported 401 class, especially the Moonshot host mismatch path that caused valid Moonshot CN keys to authenticate against the wrong endpoint.

## Evidence

- **Current runtime inherits Moonshot host for Kimi search:** `resolveKimiBaseUrl` now reuses `models.providers.moonshot.baseUrl` when the Kimi web_search base URL is unset, instead of always forcing the default `.ai` host.
  - file: [extensions/moonshot/src/kimi-web-search-provider.runtime.ts:98](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/moonshot/src/kimi-web-search-provider.runtime.ts#L98)
  - command: `nl -ba extensions/moonshot/src/kimi-web-search-provider.runtime.ts | sed -n '75,120p'`
- **Regression tests cover the `.cn` auth-host case:** Tests assert that Kimi web_search inherits `https://api.moonshot.cn/v1` from Moonshot chat config and keeps explicit overrides, which is the concrete protection against the 401 host-mismatch bug.
  - file: [extensions/moonshot/src/kimi-web-search-provider.test.ts:36](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/moonshot/src/kimi-web-search-provider.test.ts#L36)
  - command: `nl -ba extensions/moonshot/src/kimi-web-search-provider.test.ts | sed -n '1,90p'`
- **Docs describe the exact 401 mitigation:** The Kimi search docs state that when chat uses `https://api.moonshot.cn/v1`, OpenClaw reuses that host for Kimi web_search so Moonshot China-console keys do not hit the international endpoint by mistake, which 'often returns HTTP 401'.
  - file: [docs/tools/kimi-search.md:64](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/docs/tools/kimi-search.md#L64)
  - command: `nl -ba docs/tools/kimi-search.md | sed -n '50,80p'`
- **Shipped changelog entry for the auth fix:** The changelog for `2026.4.2` says: Kimi `web_search` now inherits the native Moonshot chat `baseUrl` (`.ai` / `.cn`) when unset so China console keys authenticate on the same host as chat.
  - file: [CHANGELOG.md:1383](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/CHANGELOG.md#L1383)
  - command: `nl -ba CHANGELOG.md | sed -n '1360,1400p'`
- **Earlier shipped Kimi web_search repair:** The changelog for `2026.4.1` also notes Kimi web_search flow repairs: verbatim `$web_search` replay, thinking disablement for `kimi-k2.5`, and region/model setup prompts so bundled Kimi web_search works again.
  - file: [CHANGELOG.md:1659](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/CHANGELOG.md#L1659)
  - command: `nl -ba CHANGELOG.md | sed -n '1640,1685p'`

## Close Comment

Closing from a Codex review: current `main` already carries the Kimi `web_search` auth/flow fixes for this issue class. `extensions/moonshot/src/kimi-web-search-provider.runtime.ts:98` now inherits the native Moonshot chat `baseUrl` for Kimi search when unset, `extensions/moonshot/src/kimi-web-search-provider.test.ts:36` covers the `.cn` case, and `docs/tools/kimi-search.md:64` documents this specifically as avoiding the HTTP 401 that happens when Moonshot China-console keys hit the international endpoint by mistake. This was shipped in `v2026.4.2` (`CHANGELOG.md:1383`), with additional Kimi web_search flow repairs already shipped in `v2026.4.1` (`CHANGELOG.md:1659`). If you still see a 401 on `2026.4.2` or newer, please open a fresh issue with the exact `models.providers.moonshot.baseUrl`, `plugins.entries.moonshot.config.webSearch.*` values (redacting secrets), and the full failing response body so maintainers can check for a new provider-side regression.
