## Summary

Keep open. Current `main` does not contain this fix, and the submitted approach still has unresolved correctness risk from scanning full-session history for `NO_REPLY` tool results.

## Evidence

- **Current main still uses assistant/tool-media payload count only:** `runEmbeddedPiAgent` computes `payloadCount` from `payloadsWithToolMedia` and later returns only `payloadsWithToolMedia`; there is no current-main path that synthesizes a silent `NO_REPLY` payload from a tool result before incomplete-turn handling.
  - file: [src/agents/pi-embedded-runner/run.ts:1871](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/pi-embedded-runner/run.ts#L1871)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '1760,1878p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Incomplete-turn suppression only covers silent assistant text:** `resolveIncompleteTurnPayloadText` exits early when `assistantTexts` already contain a silent reply token, but otherwise a zero-payload empty final assistant turn still falls into the incomplete-turn path. There is no corresponding current-main exemption for a successful tool result whose text is exact `NO_REPLY`.
  - file: [src/agents/pi-embedded-runner/run/incomplete-turn.ts:170](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/pi-embedded-runner/run/incomplete-turn.ts#L170)
  - command: `nl -ba src/agents/pi-embedded-runner/run/incomplete-turn.ts | sed -n '150,330p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **`messagesSnapshot` is the session snapshot, not just the current turn:** Attempt results persist `messagesSnapshot` from `snapshotSelection.messagesSnapshot`, and that snapshot is sourced from `activeSession.messages.slice()`. That means any helper that scans `messagesSnapshot` without a turn boundary is reading the full session transcript.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2677](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/pi-embedded-runner/run/attempt.ts#L2677)
  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.ts | sed -n '2660,2705p'`

- **Current-turn assistant logic already applies a turn boundary:** `findCurrentAttemptAssistantMessage` slices `messagesSnapshot` starting at `prePromptMessageCount` before searching backward. That supports the PR review concern: a new helper that scans raw `messagesSnapshot` for tool results without an equivalent boundary can accidentally reuse a stale prior-turn `NO_REPLY`.
  - file: [src/agents/pi-embedded-runner/run/attempt.context-engine-helpers.ts:104](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/pi-embedded-runner/run/attempt.context-engine-helpers.ts#L104)

- **PR discussion itself leaves the scan-scope edge unresolved:** The review thread on PR #68453 flagged that the new helper scans the full session history and could synthesize a false silent success from an older `NO_REPLY` tool result. The author replied that the concern is fair, that the patch is intentionally scoped to fresh-session isolated cron, and that tightening the precondition would be follow-up work rather than something already solved in this PR.
