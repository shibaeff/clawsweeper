## Summary

Current `main` has outbound reply tags and channel/config-based auto-threading, but it does not implement an inbound free-text directive that strips suffixes like `reply in thread`/`rit` and forces `replyToId` on the response. The issue is still a real, unshipped feature request.

## Evidence

- **Inbound directive parser only recognizes slash-style directives:** `parseInlineDirectives()` composes inbound parsing from `/think`, `/verbose`, `/trace`, `/fast`, `/reasoning`, `/elevated`, `/exec`, `/status`, `/model`, and `/queue`. There is no reply/thread directive in the inbound parser.
  - file: [src/auto-reply/reply/directive-handling.parse.ts:74](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/auto-reply/reply/directive-handling.parse.ts#L74)
  - command: `nl -ba src/auto-reply/reply/directive-handling.parse.ts | sed -n '1,220p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Directive matching requires a `/` prefix:** The directive helpers match `/(name)` tokens via regex and expose only think/verbose/trace/fast/notice/elevated/reasoning/status extractors. This does not cover plain trailing phrases like `reply in thread`, `rit`, `thread`, or `in thread`.
  - file: [src/auto-reply/reply/directives.ts:23](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/auto-reply/reply/directives.ts#L23)
  - command: `nl -ba src/auto-reply/reply/directives.ts | sed -n '1,220p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Reply threading is driven by outbound tags or existing reply mode policy:** `resolveReplyThreadingForPayload()` applies implicit threading from `replyToMode` and parses outbound `[[reply_to_*]]` tags from assistant text. There is no inbound suffix-stripping path that converts user text into forced `replyToId`.
  - file: [src/auto-reply/reply/reply-payloads-base.ts:25](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/auto-reply/reply/reply-payloads-base.ts#L25)
  - command: `nl -ba src/auto-reply/reply/reply-payloads-base.ts | sed -n '1,140p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Docs still describe reply control as outbound tags:** The rich output protocol documents `[[reply_to_current]]` / `[[reply_to:<id>]]` as assistant output metadata. No inbound `reply in thread` user directive is documented there.
  - file: [docs/reference/rich-output-protocol.md:9](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/docs/reference/rich-output-protocol.md#L9)
  - command: `nl -ba docs/reference/rich-output-protocol.md | sed -n '1,80p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Current user-facing docs expose config-based threading, not the requested inbound trigger:** Channel docs describe `replyToMode` plus manual `[[reply_to_*]]` tags. Example: Telegram documents explicit reply tags and `channels.telegram.replyToMode`; Slack documents `channels.slack.replyToMode` plus manual reply tags. That matches current config/output behavior, not the requested inbound free-text trigger.
  - file: [docs/channels/telegram.md:460](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/docs/channels/telegram.md#L460)
