## Summary

Keep open. Current `main` still routes Discord voice-channel ingress through the normal Discord agent path (`messageChannel: "discord"`) without a voice-specific spoken-output contract, while the general TTS system hint remains active and the `tts` tool is only suppressed for `messageProvider: "voice"`. That leaves the reported `tts` + silent/empty final payload behavior plausible on `7b5307acfc347a4e6d6647156600ee4b8d6eaa37`, and I did not find a main-branch fix or release note closing this specific Discord voice reply gap.
