## Summary

Issue #43706 is implemented on current `main`: Control UI gateway tokens are now stored in session-scoped browser storage keyed by gateway URL, reloaded on startup for the current tab session, and reused across reconnects; the gateway client also does a bounded trusted device-token retry on `AUTH_TOKEN_MISMATCH` to avoid restart/rebuild reconnect churn. The report was against `v2026.3.8`; the changelog shows both relevant fixes shipped in `v2026.3.11`.

## Risks / Open Questions

- Session-scoped storage is intentionally not long-term `localStorage` persistence. If the reporter meant a full fresh app/tab session with no tokenized dashboard bootstrap URL, that remains a different UX request than the reconnect/rebuild bug described here.

## Close Comment

Closing as implemented on current `main`. Codex review: the Control UI now reloads the gateway token from session-scoped storage keyed by gateway URL (`ui/src/ui/storage.ts:154-230`), reconnects with that saved token (`ui/src/ui/app-gateway.ts:279-282`, `ui/src/ui/gateway.ts:334-350`), and also does a bounded trusted device-token retry on `AUTH_TOKEN_MISMATCH` to avoid reconnect churn after gateway restarts/rebuilds (`ui/src/ui/gateway.ts:466-490`). The changelog shows these fixes shipped in `v2026.3.11` (`CHANGELOG.md:2710-2720`), while this report was filed against `v2026.3.8`. If you still see this on a current build, please open a new issue with the exact app surface and whether the app/tab was fully relaunched, because that would be a narrower follow-up than the behavior originally reported.
