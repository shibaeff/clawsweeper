## Summary

Keep open. Current `main` added restart deferral plus a new main-session restart recovery path, but the recovery replay is still launched with `deliver: false`, and the agent delivery path only sends externally when `deliver === true`. That means the reported Discord-visible gap is not clearly fixed: an interrupted turn may be resumed internally, but there is no proven automatic recovery/apology message or resumed answer posted back to the Discord channel.

## Evidence

- **Config/tool restarts are only deferred, not prevented:** Gateway restarts now wait for pending work to drain, but the deferral still times out and restarts anyway after `gateway.reload.deferralTimeoutMs` (default 5 minutes). So an in-flight turn can still be interrupted under timeout or non-deferral restart paths.

## Risks / Open Questions

- A later uninspected path could still layer external delivery on top of the internal recovery replay, but the current recovery and delivery code paths do not show that.
- The new restart deferral logic reduces incidence for config-driven reloads, so the bug may be narrower now, but it does not provide the user-visible recovery behavior requested in the issue.
