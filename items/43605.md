## Summary

Keep open. Current `main` still treats sessions as gateway-local state, `sessions_send` resolves targets only through the current gateway's session store, and the CLI remote path still validates `--agent` against local config before calling the gateway. I found no shipped cross-gateway federation/peer routing implementation on `main` at `28fc03c386e14db502429bbe7e062cbe90439ca1`.

## Evidence

- **`sessions_send` still resolves labels through the current gateway only:** The tool builds `sessions.resolve` params from the requested label/agent and calls the current gateway; failures return `No session found with label ...`. There is no remote peer or gateway-routing branch here.
  - file: [src/agents/tools/sessions-send-tool.ts:149](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/tools/sessions-send-tool.ts#L149)

  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)

In `src/gateway/sessions-resolve.ts`, both sessionId and label lookup scan the gateway's combined local session store via `loadCombinedSessionStoreForGateway(cfg)`; there is no peer/federation lookup.
