## Summary

Current `main` already blocks memory-index fallback activation when `memorySearch.fallback` is `"none"`, and the same guard is present in the shipped `v2026.4.22` release.

## Evidence

## Close Comment

Closing from a Codex review: current `main` already prevents this behavior. The memory-index fallback path now goes through `resolveMemoryFallbackProviderRequest()`, which returns `null` when `memorySearch.fallback === "none"` (`extensions/memory-core/src/memory/manager-provider-state.ts:82`), and `activateFallbackProvider()` exits immediately when that happens (`extensions/memory-core/src/memory/manager-sync-ops.ts:1072`). That means index builds no longer silently switch to Gemini when the configured embedding backend is unreachable and fallback is disabled. The same guard is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still hit a wrong-model store on a newer build, please open a fresh issue with the exact version, config, and `openclaw memory index --verbose` logs.
