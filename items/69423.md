## Summary

Keep open. Current `main` still has the unguarded Anthropic tool conversion path the report describes, plugin tool registration still accepts tools without schema validation, and the latest release commit shows the same behavior. I did not find a merged fix or a current regression test covering malformed plugin tool schemas.

## Evidence

- **Current Anthropic adapter still dereferences `tool.parameters` without a guard:** `convertAnthropicTools(...)` still does `const parameters = tool.parameters as Record<string, unknown>` and then reads `parameters.properties` / `parameters.required` for every tool. A plugin tool with no `parameters` object will still throw before the request is sent.
  - file: [src/agents/anthropic-transport-stream.ts:394](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/agents/anthropic-transport-stream.ts#L394)
  - command: `nl -ba src/agents/anthropic-transport-stream.ts | sed -n '394,410p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Plugin registration still stores tools without schema validation:** `registerTool(...)` records the tool factory and names but does not validate that the registered tool has a well-formed `parameters` schema or emit a diagnostic for missing schema keys.
  - file: [src/plugins/registry.ts:371](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/plugins/registry.ts#L371)

## Risks / Open Questions

- A single malformed third-party plugin tool can still break every Anthropic-model run instead of failing that tool in isolation.
- Users still lack an early validation/error surface at plugin registration time, so diagnosis remains likely to happen only after runtime failure.
- No current regression test appears to lock in the expected defensive behavior for malformed plugin schemas.
