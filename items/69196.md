## Summary

Keep open. The report is still plausible on current `main`: Control UI dictation still depends entirely on browser `SpeechRecognition`/`webkitSpeechRecognition`, the UI still drops runtime STT errors without showing the user anything, and the reporter's follow-up narrowed the failure to Chrome returning `not-allowed` from that browser API on the local Control UI origin. I found no evidence of a fix on `main` after `v2026.4.23`, and the gateway already sends the same-origin microphone permission header the reporter said did not resolve the deeper failure.

## Evidence

- **Reporter narrowed failure to browser speech recognition, not basic mic permission:** Issue comments say Chrome mic permission is granted, the microphone works on other sites, `window.SpeechRecognition`/`window.webkitSpeechRecognition` exist on the OpenClaw page, but manually starting `webkitSpeechRecognition()` on `http://127.0.0.1:18789` fails with `ERROR not-allowed`. That makes this a still-actionable browser/STT-path bug report rather than missing reproduction detail.
  - command: `Provided GitHub context comments 4282690312 and 4282714796`
- **Current main still uses browser Web Speech API as the only dictation path:** `getSpeechRecognitionCtor()` resolves only `SpeechRecognition` or `webkitSpeechRecognition`, and `startStt()` instantiates that browser API directly. There is no alternate plain-audio capture or fallback STT path in this module.
