## Summary

Keep open. Current `main` does not implement this PR’s per-turn Mattermost behavior: the Mattermost monitor still uses one reusable draft preview post, resets only local partial-text state on `onAssistantMessageStart`, and documents Mattermost as a single-preview surface. That means there is not strong evidence for `implemented_on_main` or `cannot_reproduce`, and the author’s abandonment note alone is not an allowed close reason.

## Evidence

- **Mattermost monitor still uses one draft preview for the whole run:** `createMattermostDraftStream(...)` is created once per inbound message, `onPartialReply` always calls `draftStream.update(...)`, and `onAssistantMessageStart` only resets `lastPartialText`; it does not rotate to a fresh preview post.

- **Draft stream implementation reuses the same Mattermost post id:** `sendOrEditStreamMessage` edits `streamPostId` when present and only creates a new post when no preview exists. A `forceNewMessage()` helper exists, but it is not wired into the Mattermost monitor path.
