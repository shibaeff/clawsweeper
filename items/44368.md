## Summary

Telegram large-media inbound handling is already implemented on current main and was shipped in v2026.4.22: oversized Bot API downloads are treated as non-fatal, inbound handlers catch media download failures instead of killing the polling loop, and polling now has explicit stall detection plus status updates.

## Evidence

- **Large Telegram files are treated as non-fatal:** `isFileTooBigError` recognizes Telegram's `file is too big` response, `isRetryableGetFileError` refuses to retry it, and `resolveTelegramFileWithRetry` returns `null` instead of throwing so the inbound path can continue.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:52](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/telegram/src/bot/delivery.resolve-media.ts#L52)

- **Inbound handler isolates media download failures:** The main inbound message handler wraps `resolveMedia(...)` in `try/catch`, replies with a user-facing failure message on fetch errors, and returns without crashing the polling session.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1111](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/telegram/src/bot-handlers.runtime.ts#L1111)

## Risks / Open Questions

- The oversize-file skip path logs through verbose diagnostics rather than a loud default warning, so if a user reports silent failures on v2026.4.22+ it may indicate a different remaining path rather than this original regression.

## Close Comment

Closing as implemented on main. Codex review found that Telegram inbound media handling now treats Bot API `file is too big` failures as non-fatal and skips the attachment instead of letting the error unwind (`extensions/telegram/src/bot/delivery.resolve-media.ts:52`). The inbound handler also catches media download failures locally (`extensions/telegram/src/bot-handlers.runtime.ts:1111`), and long-polling now has an explicit stall watchdog plus polling status updates/restarts (`extensions/telegram/src/polling-session.ts:313`, `extensions/telegram/src/polling-status.ts:9`). This exact behavior is covered by tests for oversized media and stalled polling restarts (`extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts:243`, `extensions/telegram/src/polling-session.test.ts:324`) and is already present in the shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still reproduce inbound polling death on `v2026.4.22` or newer, please open a fresh issue with current logs and exact media details because that would likely be a different path.
