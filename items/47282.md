## Summary

Current `main` already contains the heartbeat scheduler fix and regression coverage for the reported "fires once, then stops" behavior, and the changelog shows it shipped in version `2026.3.28`.

## Evidence

- **Interval timer is explicitly re-armed after runs:** `startHeartbeatRunner()` computes the next due heartbeat, schedules a timer via `setTimeout(...)`, and always calls `scheduleNext()` from the `finally` path unless the wake layer is handling a short retry for `requests-in-flight`. That prevents the interval loop from dying after a single tick.
  - file: [src/infra/heartbeat-runner.ts:1368](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/infra/heartbeat-runner.ts#L1368)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '1348,1560p'`
- **Unexpected runner errors no longer stop future heartbeats:** Unhandled `runOnce()` exceptions are caught, the agent schedule is advanced, and the outer `finally` still re-arms the scheduler. This matches the report's symptom of heartbeats stopping after one interrupted cycle.
  - file: [src/infra/heartbeat-runner.ts:1527](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/infra/heartbeat-runner.ts#L1527)
