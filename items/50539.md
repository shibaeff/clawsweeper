## Summary

Current main does not add an internal `gateway:shutdown` HOOK.md event, but it already exposes shutdown lifecycle handling through the plugin SDK (`gateway_stop`) and the project vision steers optional lifecycle automation toward plugins/ClawHub rather than expanding core internal hooks.

## Risks / Open Questions

- This close assumes the plugin SDK is the preferred extensibility layer even for operator-facing shutdown notifications; if maintainers specifically want HOOK.md parity for lifecycle events, the issue could still be reopened as a core ergonomics request.

## Close Comment

Closing this as a Codex review because the shutdown lifecycle extension point already exists on current `main` at the plugin layer, which is the repo’s preferred place for this kind of optional automation. `src/plugins/hook-types.ts:82-83,758-765` defines `gateway_stop`, and `src/gateway/server.impl.ts:910-918` runs that hook during gateway shutdown. By contrast, the internal HOOK.md system still only exposes `gateway:startup` (`docs/automation/hooks.md:38-50,154-159`), and the SDK docs explicitly steer gateway-owned lifecycle work toward plugin hooks instead of internal `gateway:startup` hooks (`docs/plugins/sdk-overview.md:243`). Given `VISION.md:54-70` (`Core stays lean; optional capability should usually ship as plugins` / `New skills should be published to ClawHub first`), this request fits a ClawHub or community plugin better than expanding OpenClaw core’s internal hook surface.
