## Summary

Current `main` already addresses the reported fallback-loop failure mode: live model switches are now limited to explicit user-requested switches, `LiveSessionModelSwitchError` during fallback is downgraded to a failover so the chain continues, and the outer runner has a bounded retry guard instead of looping back to the overloaded primary.

## Evidence

- **Live switch pending is explicit-only:** `applyModelOverrideToSessionEntry` sets `liveModelSwitchPending` only when `markLiveSwitchPending` is passed, so ordinary auto/fallback model updates do not arm a live switch back to primary.
  - file: [src/sessions/model-overrides.ts:117](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/sessions/model-overrides.ts#L117)
  - command: `nl -ba src/sessions/model-overrides.ts | sed -n '90,145p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **Code comment documents old bug class removed:** `shouldSwitchToLiveModel` is explicitly documented as a user-initiated switch path and says it replaces an older approach that could not distinguish user switches from system-initiated fallback rotations.
  - file: [src/agents/live-model-switch.ts:124](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/agents/live-model-switch.ts#L124)
  - command: `nl -ba src/agents/live-model-switch.ts | sed -n '123,143p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **Fallback chain now continues past switch errors:** `runWithModelFallback` catches `LiveSessionModelSwitchError`, converts it to a `FailoverError`, records the failed candidate, and `continue`s to the next fallback candidate instead of re-throwing into an outer loop.
  - file: [src/agents/model-fallback.ts:921](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/agents/model-fallback.ts#L921)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '880,980p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Regression tests cover the loop break:** Tests now assert both that the fallback chain continues past `LiveSessionModelSwitchError` and that repeated switch conflicts are bounded instead of looping forever.
  - file: [src/agents/model-fallback.test.ts:644](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/agents/model-fallback.test.ts#L644)
  - command: `nl -ba src/agents/model-fallback.test.ts | sed -n '630,710p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **Outer retry loop is bounded:** `agent-runner-execution` counts `LiveSessionModelSwitchError` retries and stops after `MAX_LIVE_SWITCH_RETRIES`, with a dedicated regression test for the former death-loop scenario.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1401](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/auto-reply/reply/agent-runner-execution.ts#L1401)

## Risks / Open Questions

- The fix is present on current `main`, but the latest tagged release provided in context (`v2026.4.23`) predates these changes, so released users may still hit the older behavior until a newer release ships.
- Current `main` still allows an active attempt to be interrupted by a user-requested live switch; the evidence here is that this no longer collapses fallback into an infinite restore loop.
