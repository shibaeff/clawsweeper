## Summary

Close PR #43580 as `clawhub`: current `main` has already moved TTS to a generic speech-provider plugin architecture, so a Deepgram Aura integration belongs as a provider plugin extension (likely alongside the existing `deepgram` extension) rather than as new core-special-case branches.

## Evidence

- **Vision is plugin-first and ClawHub-first for optional capabilities:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, and plugin promotion belongs in ClawHub; the bar for adding optional plugins to core is intentionally high.

- **Gateway docs say bundled speech providers are plugin-owned:** Current docs explicitly state that bundled speech providers are plugin-owned and should be enabled as plugins, with `microsoft` given as the example TTS provider plugin.
  - file: [docs/gateway/config-agents.md:1304](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/gateway/config-agents.md#L1304)

  - file: [docs/tools/tts.md:348](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/tools/tts.md#L348)

- **Current code already lists and validates TTS providers dynamically:** Gateway `tts.status`, `tts.setProvider`, and `tts.providers` all operate on registered speech providers via `listSpeechProviders(...)`/`getSpeechProvider(...)` instead of hard-coded provider branches.
  - file: [src/gateway/server-methods/tts.ts:37](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/gateway/server-methods/tts.ts#L37)

Extending that plugin with a speech provider fits the current architecture better than adding core-first Deepgram branches.
