## Summary

Current `main` has partial primitives for this request: durable multi-step Task Flow state, background-task progress notifications, subagent completion announce, and ACP parent-stream relays. But it does not yet show a first-class step graph/orchestrator on `main` with explicit per-step lifecycle, retries/timeouts, and deviation handling of the kind requested here, so there is not strong evidence to close this conservatively.

## Evidence

- **Task Flow exists on main as a durable orchestration substrate:** The docs describe Task Flow as durable multi-step flow orchestration with managed/mirrored modes and CLI inspection/cancel commands.
  - file: [docs/automation/taskflow.md:10](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/docs/automation/taskflow.md#L10)
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Current Task Flow record is coarse-grained, not a structured step state machine:** `TaskFlowRecord` stores one flow status plus `currentStep`, `blockedTaskId`, `stateJson`, and `waitJson`. There is no built-in per-step object model here for step-specific timeout/retry policy or step event kinds.
  - file: [src/tasks/task-flow-registry.types.ts:24](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/tasks/task-flow-registry.types.ts#L24)
