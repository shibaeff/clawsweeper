## Summary

Keep open. Current main does not contain this PR's additive/default-merging behavior, and the repo is internally inconsistent about the intended contract for `logging.redactPatterns`, so there is not strong evidence for a conservative close.

## Evidence

- **Current implementation still replaces defaults when patterns are supplied:** `resolvePatterns` uses `value?.length ? value : DEFAULT_REDACT_PATTERNS`, so any configured/user-supplied patterns replace the built-in set instead of merging with it.
  - file: [src/logging/redact.ts:75](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/logging/redact.ts#L75)

  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Config-based tool redaction still forwards resolved config directly:** `redactToolDetail` calls `resolveConfigRedaction()` and then passes that object into `redactSensitiveText(detail, resolved)`, so there is no additive merge path in current main.
  - file: [src/logging/redact.ts:166](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/logging/redact.ts#L166)
