## Summary

Current `main` no longer unconditionally schedules a direct `SIGUSR1` from `config.patch`; the handler now gates that through reload planning, and tests cover the hybrid-mode handoff to the config reloader. But the browser plugin is still registered as restart-required (`restartPrefixes: ["browser"]`), while the docs still say `browser` hot-applies without restart. Because the issue is specifically about `browser.profiles.*` being hot-reloadable, the code/docs evidence is mixed and not strong enough for a conservative close.

## Risks / Open Questions

- The specific `browser.profiles.*` behavior may still be user-visible as a restart-required change, because current code classifies all `browser` config under restart prefixes.
- Docs and code disagree on whether `browser` is hot-reloadable, so closing would risk hiding a real bug or at least a real behavior/docs mismatch.
- No live reproduction was run in this read-only review, and `pnpm` was unavailable in the checkout environment, so the conclusion relies on static code/tests/docs inspection only.
