## Summary

Keep open. Current `main` still relies on process-local iMessage echo caches with no startup hydration, persistence, or startup fence, so the reported restart-specific replay path is not clearly implemented away.

## Evidence

- **Echo cache is recreated fresh on monitor startup:** `monitorIMessageProvider` constructs new `sentMessageCache` and `selfChatCache` maps each time the iMessage monitor starts. There is no restore-from-disk, startup snapshot, or rowid/grace-window logic in this path.
  - file: [extensions/imessage/src/monitor/monitor-provider.ts:132](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/imessage/src/monitor/monitor-provider.ts#L132)
  - command: `nl -ba extensions/imessage/src/monitor/monitor-provider.ts | sed -n '120,150p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Echo cache is strictly in-memory with TTL maps:** The cache implementation is just three in-memory `Map`s with TTL cleanup. It has `remember`/`has` only; nothing persists state across restarts or seeds a startup fence.
  - file: [extensions/imessage/src/monitor/echo-cache.ts:45](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/imessage/src/monitor/echo-cache.ts#L45)
  - command: `nl -ba extensions/imessage/src/monitor/echo-cache.ts | sed -n '1,115p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Cache population only happens after outbound sends in the current process:** `deliverReplies` writes to the echo cache only after `sendMessageIMessage(...)` returns for the current runtime. That covers live sends, but not pre-restart outbound rows already present in the Messages DB.
  - file: [extensions/imessage/src/monitor/deliver.ts:57](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/imessage/src/monitor/deliver.ts#L57)
  - command: `nl -ba extensions/imessage/src/monitor/deliver.ts | sed -n '35,75p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Self-chat `is_from_me` rows still dispatch when there is no cache hit:** For true self-chat, inbound processing drops the message only if `hasIMessageEchoMatch(...)` succeeds; otherwise it proceeds as a dispatchable inbound message. That matches the issue's described failure mode when the cache is cold after restart.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:230](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/imessage/src/monitor/inbound-processing.ts#L230)
