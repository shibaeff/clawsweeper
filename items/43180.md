## Summary

Keep open. Current `main` does not show a landed fix for bundled-hook discovery, and the issue is still plausible. Artifact inspection weakens one theory (`HOOK.md` missing from npm packages) but does not disprove the Windows `pnpm add -g` regression itself.

## Evidence

- **Bundled-hook resolution on main is still the simple pre-hardening logic:** `src/hooks/bundled-dir.ts:5` only checks `OPENCLAW_BUNDLED_HOOKS_DIR`, `process.execPath` sibling `hooks/bundled`, and `path.dirname(import.meta.url) + '/bundled'`, then falls back to `src/hooks/bundled`. There is no package-root or pnpm/global-layout handling in current `main`.
  - file: [src/hooks/bundled-dir.ts:5](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/hooks/bundled-dir.ts#L5)
  - command: `nl -ba src/hooks/bundled-dir.ts | sed -n '1,48p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Hooks CLI still depends on bundled hook discovery for `hooks list`:** `src/hooks/workspace.ts:252` resolves bundled hooks through `resolveBundledHooksDir()`, and `src/hooks/workspace.ts:262` loads them into the workspace hook entries. If bundled discovery fails, `openclaw hooks list` can legitimately end up empty.
  - file: [src/hooks/workspace.ts:252](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/hooks/workspace.ts#L252)
  - command: `nl -ba src/hooks/workspace.ts | sed -n '242,289p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Build intent still targets `dist/bundled` plus copied `HOOK.md` metadata:** `tsdown.config.ts:98` emits bundled hook handlers as `bundled/<hook>/handler`, and `tsdown.config.ts:263` includes those entries in the unified dist graph. `scripts/copy-hook-metadata.ts:13` copies `HOOK.md` into `dist/bundled/<hook>/HOOK.md`. That means source/build inspection alone does not prove the reported Windows failure is gone.
  - file: [tsdown.config.ts:98](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/tsdown.config.ts#L98)

Its API payload shows it would have changed `src/hooks/bundled-dir.ts` and added `src/hooks/bundled-dir.test.ts`, but current `main` has neither change nor that test file.

## Risks / Open Questions

- No direct Windows `pnpm add -g openclaw` reproduction was run here, so closing as `cannot_reproduce` would be too aggressive.
- Because the packaged tarballs already contain bundled hook metadata, the real regression may be a Windows/pnpm path-resolution nuance that source inspection alone does not eliminate.
