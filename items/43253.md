## Summary

Keep open. Current main fixed runtime/frontmatter parsing, but `skill-creator` still scaffolds an unquoted YAML-sensitive `description` and its packaging validator still rejects that scaffold when PyYAML is installed, so this PR remains relevant rather than obsolete.

## Evidence

- **Scaffold still emits the YAML-sensitive placeholder:** `init_skill.py` still generates `description: [TODO: ... file types, or tasks ...]` without quotes, which is the exact shape this PR proposes to fix.
  - file: [skills/skill-creator/scripts/init_skill.py:25](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/skills/skill-creator/scripts/init_skill.py#L25)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Normal packaging path still enforces validation first:** `package_skill.py` always calls `validate_skill()` before creating a `.skill` archive, so a broken scaffold blocks the advertised authoring flow.
  - file: [skills/skill-creator/scripts/package_skill.py:56](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/skills/skill-creator/scripts/package_skill.py#L56)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Validator still takes the strict PyYAML path:** `quick_validate.py` imports PyYAML when available, uses `yaml.safe_load()` for frontmatter, and later requires `description` to be a string; that combination is what turns the unquoted scaffold value into a validation failure.
  - file: [skills/skill-creator/scripts/quick_validate.py:83](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/skills/skill-creator/scripts/quick_validate.py#L83)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Current scaffold fails validation on current main:** In this environment, importing the current `quick_validate.py` and validating a temp skill built from the current template returned `(False, 'Description must be a string, got list')`. That reproduces the authoring/package failure the PR addresses.
  - command: `python3 -c 'import importlib.util, pathlib, tempfile, shutil; spec=importlib.util.spec_from_file_location("qv", str(pathlib.Path("skills/skill-creator/scripts/quick_validate.py"))); m=importlib.util.module_from_spec(spec); spec.loader.exec_module(m); d=pathlib.Path(tempfile.mkdtemp()); s=d/"demo"; s.mkdir(); (s/"SKILL.md").write_text("---\nname: demo\ndescription: [TODO: Complete and informative explanation of what the skill does and when to use it. Include WHEN to use this skill - specific scenarios, file types, or tasks that trigger it.]\n---\n# Demo\n", encoding="utf-8"); print(m.validate_skill(s)); shutil.rmtree(d)'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Runtime parsing was fixed separately, not the skill-creator flow:** Current main/release added a merged YAML-plus-line frontmatter parser that prefers inline values containing `:` when YAML interprets them as structured. That explains why runtime loading is more tolerant now, but it does not change `skill-creator`'s template or packaging validator.
