## Summary

Current `main` has landed some targeted config-form fixes, but the broader request for a clearer, beginner-friendly configuration UI is not implemented well enough to close. The docs still position configuration around `openclaw onboard`, `openclaw configure`, direct config editing, and a schema-driven form with Raw JSON as the escape hatch, and recent 2026 Control UI issues show the config editor is still fragile.

## Evidence

- **Docs still describe config as CLI/file-first plus schema form + Raw editor:** `docs/gateway/configuration.md` says new users should start with `openclaw onboard`, shows `openclaw configure` and `openclaw config set`, and describes the Control UI as a live-schema form with a Raw JSON editor escape hatch rather than a dedicated guided configuration UX.
  - file: `docs/gateway/configuration.md:260`
  - sha: `ef88cabe3947bf97494a0407d5ef7d56c83cc7c5`
- **Only a narrow config-form fix landed on main:** PR #31866 merged on 2026-03-02 with land commit `49f8232` / merge commit `534168a`. Its scope was specific: handle `SecretInput` unions so `models.providers` stays editable in the form. That addresses one schema edge case, not the broader UI reorganization and beginner guidance requested here.
  - sha: `49f8232`
- **Recent open issue shows config UI is still brittle:** Issue #59330, opened 2026-04-02 and still open in the fetched GitHub state, reports the Control UI config editor can get stuck in Form-only mode with `Raw mode disabled (snapshot cannot safely round-trip raw text)`. That is strong evidence the area is still not stable enough to call this request implemented.
- **Additional April 2026 bug traffic hit the same surface:** Issue #62098 (filed against OpenClaw 2026.4.5) reported the Control UI dreaming toggle writing config to the wrong path and creating orphaned config. Even though that specific bug was later closed, it shows the configuration UI remained under repair well after this feature request was opened.

## Risks / Open Questions

- Local shell commands were unavailable in this environment (`bwrap: loopback: Failed RTM_NEWADDR: Operation not permitted`), so this review relied on GitHub-rendered docs/history/issues instead of a local UI run.
- This is a broad UX request; maintainers may eventually want to replace it with a more narrowly scoped tracking issue, but the current evidence does not support closing it as already done or obsolete.
