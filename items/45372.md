## Summary

Current `main` already contains the Telegram transport and polling changes that address this IPv6-fallback failure mode: it now keeps a sticky IPv4 fallback, escalates to a pinned Telegram API IP when needed, preserves that transport across polling restarts, and logs/retries stalled or recoverable polling failures instead of silently wedging.

## Evidence

- **Sticky IPv4 and pinned-IP fallback are implemented in Telegram transport:** The Telegram transport now builds a 3-step fallback chain: default dispatcher, sticky forced-IPv4 dispatcher, then a pinned `api.telegram.org` IP fallback. Successful fallback attempts become sticky for subsequent requests.
