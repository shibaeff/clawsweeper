## Summary

Keep open. Current `main` ships `active-memory` as a bundled plugin that users enable in config, but there is still no evidence of a published or bundled `@openclaw/active-memory` npm-spec alias. The reported scoped install failure remains consistent with current code and docs.

## Evidence

- **Bundled plugin exists, but docs enable it via config rather than an install command:** The Active Memory docs describe it as an optional plugin-owned feature and the quick start tells users to enable `plugins.entries.active-memory` in `openclaw.json`; there is no npm/ClawHub install step on that page.
  - file: [docs/concepts/active-memory.md:10](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/concepts/active-memory.md#L10)
  - command: `nl -ba docs/concepts/active-memory.md | sed -n '1,120p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Plugin overview does not list Active Memory as an installable official package:** The public plugin overview separates installable npm plugins from core shipped plugins. Its memory section lists `memory-core` and `memory-lancedb`, but not `active-memory`, which reinforces that users do not currently have a documented `@openclaw/active-memory` install path.
  - file: [docs/tools/plugin.md:90](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/tools/plugin.md#L90)
  - command: `nl -ba docs/tools/plugin.md | sed -n '1,140p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Bundled-plugin npm fallback only works when a bundled source has an npm spec or package name:** Bundled source lookup derives `npmSpec` only from the bundled candidate's package manifest (`openclaw.install.npmSpec` or package name). Without that metadata, scoped npm-spec fallback cannot resolve to a bundled plugin.
  - file: [src/plugins/bundled-sources.ts:58](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/plugins/bundled-sources.ts#L58)
  - command: `nl -ba src/plugins/bundled-sources.ts | sed -n '1,220p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **CLI special-cases bare plugin ids, not arbitrary missing scoped specs:** `resolveBundledInstallPlanBeforeNpm` intercepts only bare names like `active-memory`, while `resolveBundledInstallPlanForNpmFailure` falls back only if the missing npm spec matches a bundled plugin's recorded `npmSpec`. That means `openclaw plugins install @openclaw/active-memory` still has no proven bundled alias path on current `main`.
  - file: [src/cli/plugin-install-plan.ts:48](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/cli/plugin-install-plan.ts#L48)

- **Install command uses that bundled-plan logic before/after npm resolution:** The CLI install flow first redirects bare bundled ids, then later falls back from npm only when `resolveBundledInstallPlanForNpmFailure` succeeds. There is no separate hardcoded Active Memory alias in the install command.
  - file: [src/cli/plugins-install-command.ts:484](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/cli/plugins-install-command.ts#L484)
