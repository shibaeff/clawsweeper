## Summary

Current `main` does not implement this feature. The config schema has no opt-in mirror flag, the session routing code preserves existing external routes rather than adding a second mirrored delivery, and gateway tests explicitly assert that UI/webchat sends stay on `webchat` even when `deliver: true` and a prior external route exists.

## Evidence

- **No opt-in config key exists:** `SessionSchema` exposes `dmScope`, `sendPolicy`, `agentToAgent`, `threadBindings`, and related session settings, but there is no `mirrorWebchatToLastExternalChannel` or equivalent mirror toggle.
  - file: [src/config/zod-schema.session.ts:27](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/config/zod-schema.session.ts#L27)
  - command: `nl -ba src/config/zod-schema.session.ts | sed -n '1,140p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Gateway behavior still keeps UI sends on webchat:** The test `chat.send does not inherit external delivery context for UI clients on main sessions when deliver is enabled` seeds a Telegram last route, then asserts the dispatched context remains `OriginatingChannel: "webchat"` with `ExplicitDeliverRoute: false`. That is the opposite of opt-in mirroring to the last external channel.
  - file: [src/gateway/server-methods/chat.directive-tags.test.ts:1038](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/gateway/server-methods/chat.directive-tags.test.ts#L1038)
  - command: `nl -ba src/gateway/server-methods/chat.directive-tags.test.ts | sed -n '1038,1088p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Session routing preserves existing external route instead of mirroring:** `resolveLastChannelRaw` and `resolveLastToRaw` explicitly preserve the persisted external route when a turn originates from `webchat`/other non-deliverable sources. The adjacent tests assert webchat access preserves `lastChannel`/`lastTo` for external-channel sessions and main sessions, rather than duplicating replies to both surfaces.
