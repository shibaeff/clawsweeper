## Summary

Keep open. Current `main` still does not implement a general per-IP HTTP request limiter for gateway traffic, and the submitted PR has an unresolved reverse-proxy correctness issue called out in review.

## Evidence

- **HTTP server only accepts auth limiter on main:** `createGatewayHttpServer()` currently exposes only `rateLimiter?: AuthRateLimiter`; there is no `requestRateLimiter` option and no early per-request flood check before stage dispatch.
  - file: [src/gateway/server-http.ts:887](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/gateway/server-http.ts#L887)
  - command: `nl -ba src/gateway/server-http.ts | sed -n '884,950p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Startup builds auth throttling only:** Gateway startup creates limiters only from `cfgAtStart.gateway?.auth?.rateLimit`, so current main still limits failed auth attempts rather than all inbound HTTP requests.
  - file: [src/gateway/server.impl.ts:440](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/gateway/server.impl.ts#L440)
  - command: `nl -ba src/gateway/server.impl.ts | sed -n '430,470p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Runtime wiring has no general request limiter plumbing:** `createGatewayRuntimeState()` accepts and forwards only `rateLimiter` into `createGatewayHttpServer()`. There is no parallel request-level limiter threaded through the runtime state on main.
  - file: [src/gateway/server-runtime-state.ts:65](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/gateway/server-runtime-state.ts#L65)
  - command: `nl -ba src/gateway/server-runtime-state.ts | sed -n '50,90p'; nl -ba src/gateway/server-runtime-state.ts | sed -n '183,210p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Public docs still describe auth-only rate limiting:** The configuration reference documents `gateway.auth.rateLimit` as an optional failed-auth limiter keyed by client IP and auth scope. There is no shipped docs surface for a general per-IP gateway request limiter.
  - file: [docs/gateway/configuration-reference.md:379](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/docs/gateway/configuration-reference.md#L379)

- **PR discussion identified unresolved reverse-proxy bug:** The provided PR review comments on head `5cc6be71ac87a88e39e5f3a0df86e80241336976` both flag the same issue: keying the limiter by `req.socket.remoteAddress` would either bypass the limiter for same-host reverse proxies or collapse all proxied users into one bucket. Current auth throttling already uses `resolveRequestClientIp(...)` before rate-limit checks, which matches that review feedback.
  - file: [src/gateway/auth.ts:367](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/gateway/auth.ts#L367)

## Risks / Open Questions

- Closing as implemented would be incorrect because `main` still ships only failed-auth throttling, not the general request limiter proposed here.
- Merging the PR as submitted would be unsafe for reverse-proxy deployments unless the limiter uses proxy-aware client IP resolution instead of the raw socket peer address.
