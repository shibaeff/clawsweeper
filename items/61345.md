## Summary

Keep open. Current `main` still unconditionally sends any disconnected Control UI state to the login gate, so the slow-reconnect bug described by PR #61345 is not implemented on `main` and remains plausible from code inspection.

## Evidence

- **Current main still hard-gates on any disconnect:** `renderApp()` returns the login gate for every `!state.connected` state, with no distinction between transient reconnects and true auth/config failures.
  - file: [ui/src/ui/app-render.ts:665](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/ui/src/ui/app-render.ts#L665)
  - command: `nl -ba ui/src/ui/app-render.ts | sed -n '640,690p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Transient reconnect states do exist in current main:** Gateway close handling sets `connected = false` before reconnect flows, and seq-gap handling explicitly marks a reconnecting state. That means a slow reconnect can still hit the unconditional login-gate branch above.
  - file: [ui/src/ui/app-gateway.ts:342](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/ui/src/ui/app-gateway.ts#L342)
