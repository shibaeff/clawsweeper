## Summary

Keep open. The PR solves a real UX gap, but the proposed `models auth login --profile` flag conflicts with the existing global CLI `--profile` root option. On current `main`, root parsing strips any `--profile` token anywhere in argv and applies it to `OPENCLAW_PROFILE` before command dispatch, so this PR's new flag would be unreachable in real CLI execution and could redirect state/config writes to a different OpenClaw profile. The follow-up type/docs gaps noted in discussion are secondary but real.

## Evidence

  - file: [src/cli/profile.ts:17](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/cli/profile.ts#L17)

  - file: [src/cli/profile.test.ts:63](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/cli/profile.test.ts#L63)

## Risks / Open Questions

- Merging as-is would create a user-facing CLI flag that does not work in real execution, because the root parser consumes it before Commander sees the subcommand option.
- The same invocation could also unintentionally switch the active OpenClaw profile/state directory via `OPENCLAW_PROFILE`, so auth writes may land in the wrong config scope.
- Even after renaming or exempting the flag, the provider auth SDK surface should explicitly declare `opts.profile` and refresh the `ProviderAuthContext.opts` documentation.
