## Summary

Keep open. Current `main` still falls back to exact-command `=command:<hash>` durable approvals when `allow-always` cannot derive a reusable pattern, and interpreter-like executables such as Python are explicitly excluded from executable-level `allow-always` persistence. That matches the core complaint that approving one `python ...` invocation does not generalize to later argument variants. The separate claim that `exec-approvals.json` is wiped on startup does not match current code.

## Evidence

- **Interpreter-like commands are excluded from reusable allow-always persistence:** `collectAllowAlwaysPatterns` returns early for interpreter-like executables unless `strictInlineEval` is enabled and the invocation is non-inline. That means a normal `python script.py ...` approval does not persist a reusable executable/path allowlist entry.

  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Python is in the interpreter-like set:** The interpreter inline-eval specs explicitly include `python`, `python2`, `python3`, `pypy`, and `pypy3`, so the interpreter guard applies to the issue's Python example.
  - file: [src/infra/exec-inline-eval.ts:36](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/infra/exec-inline-eval.ts#L36)
