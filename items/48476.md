## Summary

Keep open. Current `main` does not contain the PR’s launchd retry or `launchctl load` fallback work, and the proposed fallback still has unresolved correctness risks in the PR discussion around sudo/root targeting and legacy `launchctl load` semantics.

## Evidence

- **Current install path still fails fast on GUI-domain bootstrap errors:** `bootstrapLaunchAgentOrThrow()` still does `launchctl bootstrap gui/<uid>` once and immediately throws the GUI-session error when that bootstrap fails. There is no retry-after-bootout path and no `launchctl load` fallback on current `main`.

- **Current GUI-domain matcher does not cover the original issue string:** `isUnsupportedGuiDomain()` only matches `domain does not support specified action` / `bootstrap failed: 125`. It does not match `Could not find domain for user gui: <uid>`, which the PR explicitly adds support for.
  - file: [src/daemon/launchd.ts:386](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/daemon/launchd.ts#L386)

## Risks / Open Questions

- Closing this PR would discard a still-live launchd install gap: current `main` still has no recovery path for the reported GUI-domain/bootstrap failures.
- Merging or closing as resolved would be premature while the PR discussion still leaves open whether the `launchctl load` fallback is safe under `sudo`, headless sessions, and later `kickstart`/status flows.
