## Summary

Keep open. Current `main` still forwards arbitrary `firecrawl_scrape` URLs to Firecrawl without any local SSRF validation, so the core security fix in this PR is not implemented on `main`. The cache-growth part of the PR description is stale because shared cache writes are already capped centrally, but that does not make the SSRF fix obsolete.

## Evidence

- **Current main still forwards caller-supplied scrape URLs without a guard:** `runFirecrawlScrape()` builds the Firecrawl request body with `url: params.url` and sends it via `postFirecrawlJson(...)`. There is no `assertFirecrawlUrlAllowed` helper, no `isBlockedHostnameOrIp` import, and no URL validation before the request is made.

## Risks / Open Questions

- Current `main` still appears to permit Firecrawl scrape requests for arbitrary caller-controlled URLs, so closing this PR would leave that hardening gap unaddressed.
- The PR description overstates the cache fix; maintainers should review the branch for the SSRF change specifically rather than assuming both reported problems still exist on `main`.
- Even if merged, the proposed guard is a static hostname/IP check; review comments correctly note it would not resolve DNS-rebinding or public-hostname-to-private-IP cases in self-hosted Firecrawl deployments.
