## Summary

Keep open. Current `main` still contains the reported bug in the Linux/WSL systemd user-service path: the classifier still treats `Failed to connect to bus: No such file or directory` as `missing_systemctl`, and `assertSystemdAvailable` still emits the old `systemctl not available` message instead of a user-bus/WSL-specific error. The PR remains relevant and is not implemented on `main`.

## Evidence

- **Classifier still misorders the overlapping matchers:** On current `main`, `isSystemctlMissingDetail()` still matches the loose `no such file or directory` substring, and `classifySystemdUnavailableDetail()` still checks that matcher before `isSystemdUserBusUnavailableDetail()`. That means the WSL detail string described in the PR (`Failed to connect to bus: No such file or directory`) still resolves to `missing_systemctl` instead of `user_bus_unavailable`.
