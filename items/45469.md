## Summary

Keep open. Current `main` still has an unbounded reconnect loop in `GatewayClient.scheduleReconnect()`, so the reported max-retry bug is not implemented away on `main` and cannot be conservatively closed.

## Evidence

- **Current reconnect logic is still unbounded:** `scheduleReconnect()` only checks `this.closed`, clears `tickTimer`, doubles `backoffMs` up to 30s, and always schedules `setTimeout(() => this.start(), delay)`. There is no retry counter or retry ceiling.
  - file: [src/gateway/client.ts:837](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/client.ts#L837)
  - command: `nl -ba src/gateway/client.ts | sed -n '836,850p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Reconnect state resets only on successful hello:** On successful connect, the code resets `backoffMs` to 1000ms, which implies reconnect attempts continue until success rather than stopping after a bounded number of failures.
  - file: [src/gateway/client.ts:541](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/client.ts#L541)
  - command: `nl -ba src/gateway/client.ts | sed -n '536,568p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Close path still auto-reconnects after socket close:** The websocket `close` handler flushes pending errors and then calls `this.scheduleReconnect()` unless the failure matches specific auth-pause cases. Unreachable-gateway failures still fall into the auto-reconnect path.
  - file: [src/gateway/client.ts:302](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/client.ts#L302)
  - command: `nl -ba src/gateway/client.ts | sed -n '289,338p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Tests cover only bounded auth-specific stop cases:** Current tests assert no auto-reconnect for `AUTH_TOKEN_MISSING` and untrusted token mismatch cases, but there is no test asserting a general max retry limit for unreachable gateways.
  - file: [src/gateway/client.test.ts:829](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/client.test.ts#L829)
  - command: `nl -ba src/gateway/client.test.ts | sed -n '829,850p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Protocol docs only describe a bounded device-token retry:** Docs say trusted clients may attempt one bounded retry for `AUTH_TOKEN_MISMATCH`, and if that retry fails clients should stop reconnect loops. That is narrower than the reported issue and does not show a general reconnect ceiling shipped on `main`.
