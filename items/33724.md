## Summary

Keep open. Current `main` does not implement user-configurable backup path/count/trigger settings for config backups. The issue’s premise about backups living in a workspace root no longer matches current code, because backups follow the active config file path, but the requested configurability is still absent.

## Evidence

- **Backup count and path remain hardcoded:** `CONFIG_BACKUP_COUNT` is fixed at 5, and backup files are always derived as `${configPath}.bak` / `${configPath}.bak.N`; there is no user-configurable backup directory or retention setting in this path.
  - file: [src/config/backup-rotation.ts:3](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/config/backup-rotation.ts#L3)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Backups are triggered on config writes:** Config writes call `maintainConfigBackups(configPath, ...)` only when the config file already exists, so current main backs up on write paths rather than on arbitrary config interaction.
  - file: [src/config/io.ts:1836](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/config/io.ts#L1836)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Config location is already overrideable:** The active config path can already be moved with `OPENCLAW_CONFIG_PATH`, and docs tell users to point writes at the real file via that variable. Because backups are adjacent to `configPath`, backup placement follows that override indirectly.
  - file: [src/config/paths.ts:102](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/config/paths.ts#L102)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Docs reflect active-config-path behavior:** The config docs state that `config file` prints the active config path resolved from `OPENCLAW_CONFIG_PATH` or the default location, which matches the current code path for where adjacent backups live.
  - file: [docs/cli/config.md:390](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/cli/config.md#L390)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **No config schema surface for backup knobs:** A repo search for backup path/count/trigger config entries in `src/config/schema.help.ts`, `src/config/schema.base.generated.ts`, and `docs/cli/config.md` returned no matches, so there is no documented or generated user-facing setting for this request.
  - command: `rg -n "backup.*path|backup.*count|trigger.*backup|config.*backup" src/config/schema.help.ts src/config/schema.base.generated.ts docs/cli/config.md`

## Risks / Open Questions

- The issue title is narrower than the body: path placement is already indirectly controllable through `OPENCLAW_CONFIG_PATH`, but count/trigger customization is still missing. Closing would therefore be too aggressive.
- Linked PRs could not be inspected through `gh` in this environment because GitHub CLI authentication was unavailable; this decision is based on current `main`, docs, and local tests/code only.
