## Summary

Current `main` still contains the reported no-model-aliases bug for slash skill resolution, and the PR is not superseded. Recent discussion also confirms the impact extends beyond `/skill ...` to custom workspace slash skills. There is also an unresolved review concern that the proposed fallback change would introduce redundant workspace-skill rescans when no skills exist, so this is not conservative-cleanup close material.

## Evidence

- **Directive path still gates skill discovery on configured model aliases:** `resolveReplyDirectives` computes `rawAliases` as `[]` when no model aliases are configured, then only loads workspace skill commands when `rawAliases.length > 0`. On current `main`, that means slash skill commands are never preloaded in the no-alias configuration described by the PR.
  - file: [src/auto-reply/reply/get-reply-directives.ts:239](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/reply/get-reply-directives.ts#L239)
  - command: `nl -ba src/auto-reply/reply/get-reply-directives.ts | sed -n '216,266p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Empty upstream skill list is passed straight into inline handling:** `get-reply.ts` forwards `skillCommands` from directive resolution into `handleInlineActions`, and `handleInlineActions` currently treats any passed array as authoritative. With the upstream no-alias path yielding `[]`, slash-skill resolution sees an empty list and falls through instead of dispatching the workspace skill.
  - file: [src/auto-reply/reply/get-reply-inline-actions.ts:233](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/reply/get-reply-inline-actions.ts#L233)
  - command: `nl -ba src/auto-reply/reply/get-reply-inline-actions.ts | sed -n '227,255p'; nl -ba src/auto-reply/reply/get-reply.ts | sed -n '435,545p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Recent PR discussion confirms current-user impact:** A commenter on 2026-04-24 reported reproducing the same no-alias failure in a custom workspace skill using `disable-model-invocation: true`, `command-dispatch: tool`, and `command-arg-mode: raw`, with behavior fixed only after adding a model alias. That matches the current code path above and broadens the affected surface beyond `/skill ...`.
- **Proposed patch has an unresolved maintainer-relevant tradeoff:** The existing Codex review on the PR noted that changing the inline fallback to reload on `params.skillCommands.length === 0` would rescan workspace skills on each `/skill ...` or unknown slash command in workspaces with zero skills, adding avoidable I/O and latency. That makes this an active review item, not a cleanup close candidate.

## Risks / Open Questions

- Closing this PR would leave the current no-model-aliases slash-skill bug unaddressed on `main`.
- The latest discussion indicates the bug affects custom workspace slash commands that dispatch directly to tools, so the impact is wider than the original `/skill` report.
- Merging the PR as-is may need a follow-up for the repeated zero-skill rescan path called out in review.
