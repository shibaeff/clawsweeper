## Summary

Keep open. Current `main` does not implement this PR, and the proposed one-line change would hide a transport error that current code/tests intentionally preserve; that false negative can skip service removal during uninstall.

## Evidence

- **Current main still throws for unresolved `is-enabled` probe failures:** `isSystemdServiceEnabled` only returns `false` for missing `systemctl` or disabled/not-found unit states. Any other detail still throws `systemctl is-enabled unavailable: ...`, so current main has not adopted the PR's non-fatal bus-unavailable behavior.
  - file: [src/daemon/systemd.ts:663](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/daemon/systemd.ts#L663)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '646,667p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Main already handles the common bus-unavailable case via fallback:** Before surfacing an error, `execSystemctlUser` retries `systemctl --user` through `--machine <user>@ --user` when the direct user bus is unavailable and not a permission-denied case. That means current main already narrowed the failure mode, and the PR's blanket `false` return would apply after retries are exhausted.
  - file: [src/daemon/systemd.ts:357](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/daemon/systemd.ts#L357)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '357,398p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Tests on main explicitly expect bus-unavailable to reject here:** Current tests assert that `isSystemdServiceEnabled` rejects when `is-enabled` cannot connect to the bus without fallback, and also when both the direct and machine-scope checks report bus unavailability. That codifies the opposite behavior from this PR.
  - file: [src/daemon/systemd.test.ts:268](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/daemon/systemd.test.ts#L268)
  - command: `nl -ba src/daemon/systemd.test.ts | sed -n '268,310p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Returning `false` here can skip uninstall work:** The uninstall flow calls `service.isLoaded()`, and if it gets `false` it logs that the gateway service is not loaded and returns without calling `stop` or `uninstall`. Collapsing a DBus transport failure into `false` can therefore leave an installed systemd unit behind.
  - file: [src/commands/uninstall.ts:63](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/commands/uninstall.ts#L63)
  - command: `nl -ba src/commands/uninstall.ts | sed -n '55,84p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **PR discussion already identified the same regression risk:** The provided PR review discussion notes that treating bus-unavailable as `false` would cause `stopAndUninstallService` to skip stop/uninstall and misreport the service as not loaded. That concern matches the current source flow on `main`.
