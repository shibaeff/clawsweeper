## Summary

Current `main` already implements persistent Feishu inbound replay dedupe across restarts/reconnects, warms that state on monitor startup, filters already-processed messages before dispatch, and includes a regression test that asserts one reply across duplicate replay. The replay-dedupe work is also documented in the `2026.4.22` changelog, so this report against `2026.3.8` appears already fixed in a later release.
