## Summary

Keep open. Current `main` has config-specific and `sessions_history` redaction markers, but the shared tool-output redaction path still returns plain masked strings with no machine-readable marker, so the issue’s requested generic behavior is not implemented on `main`.

## Evidence

- **Shared redaction API still returns plain strings:** `redactSensitiveText` and `redactToolDetail` both return `string` values only; there is no structured redaction flag, field list, or stable marker added by the shared redaction path.
  - file: [src/logging/redact.ts:152](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/logging/redact.ts#L152)
  - command: `nl -ba src/logging/redact.ts | sed -n '1,190p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Tests lock in unmarked masked output:** Current tests assert outputs like `OPENAI_API_KEY=sk-123…cdef` and `{"token":"abcdef…ghij"}` with no appended marker or metadata, matching the issue’s complaint about transparent redaction.
  - file: [src/logging/redact.test.ts:12](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/logging/redact.test.ts#L12)
  - command: `nl -ba src/logging/redact.test.ts | sed -n '8,65p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Config APIs have a narrow sentinel-based mitigation:** `config.get` returns `redactConfigSnapshot(...)`, and config write handlers restore `__OPENCLAW_REDACTED__` placeholders from the on-disk snapshot. That protects config RPC round-trips, but it is explicitly scoped to gateway config responses and writes.
  - file: [src/config/redact-snapshot.ts:88](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/config/redact-snapshot.ts#L88)
  - command: `nl -ba src/config/redact-snapshot.ts | sed -n '86,110p'; nl -ba src/gateway/server-methods/config.ts | sed -n '196,214p;390,420p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **One tool exposes redaction metadata, but only per-tool:** `sessions_history` tracks whether content was redacted and returns `contentRedacted` in its JSON result. That shows a partial/tool-specific pattern, not the generic machine-readable marker for all agent-visible redacted tool output requested here.
  - file: [src/agents/tools/sessions-history-tool.ts:38](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/tools/sessions-history-tool.ts#L38)

## Risks / Open Questions

- Agents can still receive ordinary redacted tool/file/terminal text that looks like real data because the shared redaction layer does not annotate that redaction occurred.
- The newer config sentinel and `sessions_history` flag reduce specific corruption paths, but they do not satisfy the broader cross-tool contract this issue asks for.
