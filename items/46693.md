## Summary

Keep open. Current `main` still drops explicit `cacheRetention` for `openai`/`openai-codex` before the OpenAI transport can translate it, so the PR addresses a real gap rather than a stale or already-landed fix.

## Evidence

- **Current resolver excludes OpenAI families:** `resolveCacheRetention` only recognizes Anthropic-family models and Google prompt-cache-eligible models. When neither condition matches it returns `undefined`, so explicit `cacheRetention` from config never reaches OpenAI/OpenAI-Codex transports.
  - file: [src/agents/pi-embedded-runner/prompt-cache-retention.ts:17](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/agents/pi-embedded-runner/prompt-cache-retention.ts#L17)
  - command: `nl -ba src/agents/pi-embedded-runner/prompt-cache-retention.ts | sed -n '1,120p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Tests on main still encode the old behavior:** The existing cache-retention test suite explicitly treats `openai` as a non-Anthropic provider where cache-retention behavior is not applied, and there are no replacement tests for OpenAI/OpenAI-Codex passthrough in this area.
  - file: [src/agents/pi-embedded-runner/extra-params.cache-retention-default.test.ts:126](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/agents/pi-embedded-runner/extra-params.cache-retention-default.test.ts#L126)

- **OpenAI transport already supports the downstream mapping:** The OpenAI transport consumes `options.cacheRetention`; for responses it emits `prompt_cache_retention` when retention is `long` and the base URL is `api.openai.com`, and for completions it also threads cache options into the payload.
