## Summary

Current `main` already includes a native-approval bootstrap that waits for channel runtime-context registration, retries failed startup, and is wired into gateway channel lifecycle. That matches the race described in #69012 and aligns with the issue timeline’s linked PR #70649. The fix appears to be on `main` after `v2026.4.23`, so this is implemented on `main` but not in the latest tagged release.

## Evidence

- **Current main starts approval bootstrap as a lifecycle helper, not as an eager Telegram-only startup path:** `startChannelApprovalHandlerBootstrap(...)` is invoked from gateway channel startup before `startAccount(...)`, so the bootstrap can observe later runtime-context registration instead of assuming the native handler is immediately ready.
  - file: [src/gateway/server-channels.ts:401](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/gateway/server-channels.ts#L401)

- **Bootstrap now waits for runtime-context registration and retries failed native-handler startup:** The bootstrap watches `approval.native` runtime-context events, starts only after a `registered` event, and retries failures after 1s instead of failing once on initial boot.
  - file: [src/infra/approval-handler-bootstrap.ts:17](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/infra/approval-handler-bootstrap.ts#L17)
