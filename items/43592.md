## Summary

Current `main` already surfaces inbound user turns over Gateway WebSocket, but through the session transcript event stream (`session.message` / `sessions.changed`) rather than a new `chat` event state.

## Risks / Open Questions

- The implementation uses `session.message` / `sessions.changed` rather than adding `chat` events with `state: "user"`, so clients written specifically for the proposed `chat` shape would need to consume the newer session-stream events instead.

## Close Comment

Closing as implemented on `main` after Codex review. The gateway now surfaces inbound user turns over WebSocket through the session transcript stream rather than a new `chat` state: `chat.send` eagerly emits a user transcript update (`src/gateway/server-methods/chat.ts:2330`, `src/gateway/server-methods/chat.ts:2489`), the runtime fans transcript updates out as `session.message` / `sessions.changed` events (`src/gateway/server-session-events.ts:83`, `src/gateway/server-runtime-subscriptions.ts:54`), and clients can subscribe with `sessions.messages.subscribe` (`src/gateway/server-methods/sessions.ts:608`, `docs/gateway/protocol.md:341`). Coverage also exists for the user-turn emission and websocket routing (`src/gateway/server-methods/chat.directive-tags.test.ts:1647`, `src/gateway/session-message-events.test.ts:518`). This is present on current `main` in commit `d4e93e791bc5093ea14666bf4d8202b45a5132b4`; it does not appear to be in the latest tagged release yet.
