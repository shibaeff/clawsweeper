## Summary

Keep open. Current `main` does not implement this PR's hardening: `extensions/nextcloud-talk/src/signature.ts` still does a manual XOR comparison and returns early on length mismatch, so the reported timing side-channel concern remains relevant.

## Evidence

- **Current Nextcloud Talk verifier still has early length return:** `verifyNextcloudTalkSignature` returns `false` immediately when `signature.length !== expected.length`, then uses a hand-rolled XOR loop instead of `crypto.timingSafeEqual`. That is the exact behavior the PR aims to replace.
  - file: [extensions/nextcloud-talk/src/signature.ts:28](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/nextcloud-talk/src/signature.ts#L28)
  - command: `nl -ba extensions/nextcloud-talk/src/signature.ts | sed -n '1,120p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Current tests do not cover the proposed hardening surface:** The existing Nextcloud Talk test only checks valid generation/verification, empty signature rejection, and one short invalid signature case inside `core.test.ts`; the dedicated `signature.test.ts` added by the PR is not present on `main`.
  - file: [extensions/nextcloud-talk/src/core.test.ts:82](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/nextcloud-talk/src/core.test.ts#L82)

## Risks / Open Questions

- The Nextcloud Talk webhook verifier still leaks whether an attacker-supplied signature has the expected length via the early return path in `extensions/nextcloud-talk/src/signature.ts:28`.
