## Summary

Keep open. Current `main` still has the reported gap: `gateway.auth.mode: "trusted-proxy"` is enforced as header-based auth, while the internal gateway client still authenticates with connect-payload token/password/device fields over a loopback self-connection. The docs now clarify that trusted-proxy is for browser/non-loopback proxy traffic, but there is no implementation on `main` that gives manager/subagent/backend self-connections a supported identity path in trusted-proxy mode.

## Evidence

- **Trusted-proxy mode still rejects missing identity headers:** `assertGatewayAuthConfigured` still treats trusted-proxy as mutually exclusive with shared token auth, and `authorizeGatewayConnect` takes the trusted-proxy branch before token/password checks. Missing proxy identity still returns `trusted_proxy_user_missing`, matching the report. See `src/gateway/auth.ts:240`, `src/gateway/auth.ts:415`, and `src/gateway/auth.test.ts:720`.
  - file: [src/gateway/auth.ts:240](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/auth.ts#L240)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Internal gateway client still uses loopback plus connect-payload auth:** The client still defaults to `ws://127.0.0.1:18789` and sends only `token`, `bootstrapToken`, `deviceToken`, and `password` in `connect.params.auth`; it does not provide trusted-proxy identity headers. See `src/gateway/client.ts:224` and `src/gateway/client.ts:447`. The protocol docs also state trusted-proxy auth is satisfied from request headers instead of `connect.params.auth.*` at `docs/gateway/protocol.md:509`.
  - file: [src/gateway/client.ts:224](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/gateway/client.ts#L224)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Current docs describe trusted-proxy as non-loopback/browser-facing auth, not an internal self-connection path:** The docs now scope `gateway.auth.mode: "trusted-proxy"` to non-loopback reverse-proxy traffic and explicitly describe it as reverse-proxy auth for browser clients. See `docs/gateway/configuration-reference.md:306` and `docs/web/webchat.md:69`. That clarifies the limitation, but it does not implement the missing backend/self-connection behavior reported here.
  - file: [docs/gateway/configuration-reference.md:306](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/docs/gateway/configuration-reference.md#L306)
