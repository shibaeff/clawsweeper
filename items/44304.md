## Summary

Current main already supports Slack custom sender identity for routed agent replies: built-in Slack delivery resolves the routed agent identity, passes it into reply delivery, and `chat.postMessage` includes `username` plus `icon_url`/`icon_emoji` when `chat:write.customize` is available. The docs describe this as shipped behavior for the active agent identity, which covers the reported dedicated-channel-per-agent use case.

## Risks / Open Questions

- The current solution is agent-scoped, not an additional Slack-channel-only override. If one agent must present different Slack names/icons in different channels, that narrower use case is still distinct.

## Close Comment

Closing after Codex review: current `main` already implements this well enough for the reported setup. Built-in Slack reply routing resolves the routed agent identity in `extensions/slack/src/monitor/message-handler/dispatch.ts:222` and passes it into delivery at `extensions/slack/src/monitor/message-handler/dispatch.ts:486`, while `extensions/slack/src/send.ts:118` forwards `username` plus `icon_url`/`icon_emoji` to Slack `chat.postMessage` when `chat:write.customize` is available. The shipped docs also call this out in `docs/channels/slack.md:380` as using the active agent identity for outgoing Slack messages. This behavior is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still need a separate Slack-only override for the same agent across different channels, please open a new issue focused on that narrower gap.
