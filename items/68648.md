## Summary

Keep open. Current `main` does not already contain this change, and the proposed `expectFinal: false` switch appears to introduce a real regression: `sendExecApprovalFollowup` would treat the gateway's initial `accepted` ack as success and lose the existing direct-message fallback when the resumed agent turn later fails.

## Evidence

- **Current main still waits for the final agent result:** `sendExecApprovalFollowup` resumes the session with `callGatewayTool("agent", ...)` using `{ expectFinal: true }`, and only falls back to direct delivery if that call throws. That means current behavior intentionally waits for terminal success/error before deciding whether to send the fallback message.
  - file: [src/agents/bash-tools.exec-approval-followup.ts:218](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/bash-tools.exec-approval-followup.ts#L218)
  - command: `nl -ba src/agents/bash-tools.exec-approval-followup.ts | sed -n '150,270p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Gateway `accepted` is only an ack, not proof the resumed run succeeded:** The gateway sends an immediate `{ status: "accepted" }` response before reactivating/dispatching the run, then later emits a second response for final success or final error. So a caller that stops at `accepted` can miss later dispatch/runtime failures.
  - file: [src/gateway/server-methods/agent.ts:901](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/gateway/server-methods/agent.ts#L901)
  - command: `nl -ba src/gateway/server-methods/agent.ts | sed -n '870,925p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Final errors are delivered after acceptance:** `dispatchAgentRunFromGateway(...).catch(...)` sends a later error response for failed runs, and the code comment explicitly says the second response exists so clients with `expectFinal` can wait for it.
  - file: [src/gateway/server-methods/agent.ts:287](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/gateway/server-methods/agent.ts#L287)
  - command: `nl -ba src/gateway/server-methods/agent.ts | sed -n '250,360p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Client semantics make the PR's one-line change behaviorally significant:** The gateway client ignores `status === "accepted"` only when `pending.expectFinal` is true. Flipping this call site to `expectFinal: false` would therefore resolve on the ack instead of waiting for the later terminal frame.
  - file: [src/gateway/client.ts:774](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/gateway/client.ts#L774)

- **Existing tests cover the current fallback-on-failure contract:** Current tests assert that when session resume fails, `sendExecApprovalFollowup` sends a sanitized direct followup instead of silently succeeding. The PR adds accepted-only coverage, but it does not preserve coverage for post-accept terminal failures.
