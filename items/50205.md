## Summary

Keep open. Current `main` still does not implement configurable Gemini/Vertex request labels: the provider config schema rejects unknown keys like `requestLabels`, the Google transport only builds/sends `contents` plus existing generation/tool fields, and the bundled Google plugin exposes no label-related config surface.

## Evidence

- **Provider config schema still rejects extra keys:** `ModelProviderSchema` is strict and only allows `baseUrl`, `apiKey`, `auth`, `api`, `injectNumCtxForOpenAICompat`, `headers`, `authHeader`, `request`, and `models`. There is no `requestLabels` field, so `models.providers.google-vertex.requestLabels` would still be unrecognized.
  - file: [src/config/zod-schema.core.ts:346](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/config/zod-schema.core.ts#L346)
  - command: `nl -ba src/config/zod-schema.core.ts | sed -n '330,360p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Gemini request body builder has no label injection:** `buildGoogleGenerativeAiParams(...)` creates the outbound body with `contents`, optional `cachedContent`, `generationConfig`, `systemInstruction`, `tools`, and `toolConfig` only. There is no `labels` field sourced from config or added for Google/Vertex requests.
  - file: [extensions/google/transport-stream.ts:441](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/google/transport-stream.ts#L441)
