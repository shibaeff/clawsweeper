## Summary

Current `main` does not contain this change, and the PR as proposed is not safe to close or merge as-is. The generated config schema still publishes `minimum: 1` for both keys, `channelMaxRestartsPerHour: 0` would disable restarts rather than make them unlimited, and hard-coding `channelStaleEventThresholdMinutes` to `min(5)` would regress configs that intentionally use faster health checks with a smaller but still valid stale threshold.

## Evidence

- **Current schema still rejects zero for restart cap and uses dynamic stale-threshold validation:** `main` still has `channelStaleEventThresholdMinutes: z.number().int().min(1)` and `channelMaxRestartsPerHour: z.number().int().min(1)`, plus a `superRefine` that enforces `channelStaleEventThresholdMinutes >= channelHealthCheckMinutes` using the effective interval rather than a hard-coded `5`.
