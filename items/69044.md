## Summary

Keep open. Current `main` still dereferences the return value of `loadChannelPlugin()` without a null/undefined guard, so the reported crash path remains plausible and the linked fix does not appear to be present in the inspected checkout.

## Evidence

- **Unsafe bundled loader path still exists:** `getBundledChannelPluginForRoot()` assigns `const plugin = entry.loadChannelPlugin();` and immediately reads `plugin.id` and `plugin.meta` when building `normalizedPlugin`. There is no null/undefined guard before those property reads, so a malformed runtime return would still throw here on current `main`.
  - file: [src/channels/plugins/bundled.ts:548](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/channels/plugins/bundled.ts#L548)
  - command: `nl -ba src/channels/plugins/bundled.ts | sed -n '530,565p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Reported config-read call chain is still reachable:** `collectChannelLegacyConfigRules()` still calls `getBootstrapChannelPlugin(channelId)` while resolving legacy config rules, which matches the issue's claim that config reads can hit the bundled channel loader during commands like `openclaw cron --help`.
  - file: [src/channels/plugins/legacy-config.ts:92](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/channels/plugins/legacy-config.ts#L92)
