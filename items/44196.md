## Summary

Current main already handles the reported session-corruption loop during replay: Anthropic-compatible transcripts get tool-use/tool-result pairing repair and turn validation before provider calls, aborted/error assistant turns do not synthesize new tool results, and timeout cleanup hardening shipped in 2026.3.7 specifically to stop follow-up-turn poisoning after idle timeouts.

## Close Comment

Closing from a Codex review because current `main` already covers the reported failure mode well enough. Before replaying session history, the embedded runner sanitizes transcripts in memory (`src/agents/pi-embedded-runner/replay-history.ts:489`), repairs/drops orphaned tool-use/tool-result spans (`src/agents/session-transcript-repair.ts:416`), and validates Anthropic turns by stripping dangling `tool_use` blocks (`src/agents/pi-embedded-helpers/turns.ts:167`). The persistence guard also avoids creating synthetic tool results for aborted/error assistant turns because that exact pattern causes later API 400s (`src/agents/session-tool-result-guard.ts:217`), with regression coverage in `src/agents/session-tool-result-guard.test.ts:489`, `src/agents/pi-embedded-helpers.validate-turns.test.ts:377`, and `src/agents/session-transcript-repair.test.ts:228`. This also appears to have shipped in the 2026.3.7 release (`CHANGELOG.md:2967`).
