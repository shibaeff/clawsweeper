## Summary

Keep open. Current `main` still does not implement a subcommand-help fast path, so this PR is not obsolete, but the proposed shape conflicts with current loader and pairing contracts in ways that need design or code changes before conservative maintainer cleanup could close it.

## Evidence

- **Current main only fast-paths root help:** `runCli()` has a dedicated root-help fast path, then falls through to the normal CLI bootstrap for other invocations. There is no current-main equivalent of a subcommand-help fast path before plugin/runtime registration.
  - file: [src/cli/run-main.ts:70](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/cli/run-main.ts#L70)
  - command: `nl -ba src/cli/run-main.ts | sed -n '1,280p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Plugin CLI registration still uses the full runtime loader:** `registerPluginCliCommands()` loads registration entries via `loadPluginCliRegistrationEntriesWithDefaults()`, which resolves to `loadPluginCliCommandRegistryWithContext()` and then `loadOpenClawPlugins(...)`, not the metadata-only registry. That means `openclaw <plugin> --help` still pays full plugin registration on current `main`.
  - file: [src/plugins/cli-registry-loader.ts:106](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/plugins/cli-registry-loader.ts#L106)

  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Main explicitly preserves full-loader compatibility for legacy plugin CLI:** The test suite currently documents that runtime CLI command registration stays on the full plugin loader for legacy plugins, which cuts against switching all help invocations to metadata-only registration without a compatibility plan.
  - file: [src/plugins/cli.test.ts:268](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/plugins/cli.test.ts#L268)
  - command: `nl -ba src/plugins/cli.test.ts | sed -n '200,340p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Metadata-only capture intentionally skips plugins without dedicated `cli-metadata` entrypoints:** The loader tests confirm that bundled non-channel full entries without a dedicated `cli-metadata` entry are omitted during CLI metadata capture. That supports the review concern that a blanket help-only metadata path could regress plugin-owned help for plugins that only expose CLI from their full runtime entry.
  - file: [src/plugins/loader.cli-metadata.test.ts:390](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/plugins/loader.cli-metadata.test.ts#L390)

That matches the PR review warning that short-circuiting pairing help before plugin registration can render incomplete channel help.

## Risks / Open Questions

- A help-only metadata path needs a compatibility story for plugins that still expose CLI only from their full runtime entry.
- `pairing --help` cannot skip plugin registration unless pairing help text stops depending on `listPairingChannels()` at registration time.
- Any memory-core lazy-runtime refactor must keep `register(api)` synchronous or change the loader contract first.
