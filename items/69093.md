## Summary

Keep open. Current `main` still has the exact `Dirent.isSymbolicLink()` -> `readlinkSync()` path the PR is trying to harden, `run-node` still invokes the runtime postbuild path that reaches this code, and the current test suite does not contain the proposed Windows false-positive regression. The report is concrete and the draft PR appears to address a real gap rather than an already-shipped fix.

## Evidence

- **Current main still trusts `Dirent.isSymbolicLink()` before `readlinkSync()`:** `appendDirectoryFingerprint()` on current `main` branches on `entry.isSymbolicLink()` and immediately calls `fs.readlinkSync(fullPath)` without re-checking with `lstatSync()`. That is the exact crash surface described in the PR.
  - file: [scripts/stage-bundled-plugin-runtime-deps.mjs:452](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/scripts/stage-bundled-plugin-runtime-deps.mjs#L452)
  - command: `nl -ba scripts/stage-bundled-plugin-runtime-deps.mjs | sed -n '452,476p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **The runtime-postbuild path is still live in source runs:** `runRuntimePostBuild()` still calls `stageBundledPluginRuntimeDeps()`, and `run-node` still executes `runRuntimePostBuild()`. So a bad fingerprint walk can still affect normal source-run flows rather than being dead code.
  - file: [scripts/runtime-postbuild.mjs:84](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/scripts/runtime-postbuild.mjs#L84)
