## Summary

Keep open. Current `main` still does not implement interactive custom Feishu account IDs during onboarding, and the PR discussion already identifies blocking bugs in the proposed change: the selected account id is not propagated into the config-writing path, so second-account onboarding would still not land on an independent Feishu account.

## Evidence

- **Feishu wizard disables account-id prompting:** The Feishu setup wizard overrides generic channel setup behavior by returning `false` from `resolveShouldPromptAccountIds`, and its `resolveAccountIdForConfigure` only chooses the explicit override or the current default account. That means interactive Feishu onboarding on `main` still does not prompt for a new named account id.

  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Resolved account id is still dropped before config writes:** `finalize` receives wizard state but does not thread `accountId` into either `runEditFlow` or `runNewAppFlow`, so any custom account choice would be lost before configuration is written.
  - file: [extensions/feishu/src/setup-surface.ts:562](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/feishu/src/setup-surface.ts#L562)

That matches the PR review concern that independent multi-bot setup is not actually achieved.
