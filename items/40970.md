## Summary

Keep open. Current `main` still treats `logging.file` as a literal path, so the requested `YYYY-MM-DD` expansion is not implemented. The proposed PR also looks incomplete against current `main`, because another shipped resolver still returns the configured path unchanged for status/reporting surfaces.

## Evidence

- **Current logger behavior is literal:** `resolveSettings()` still assigns `cfg?.file` directly with no `YYYY-MM-DD` substitution, so a configured wildcard path would remain literal on current `main`.
  - file: [src/logging/logger.ts:392](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/logging/logger.ts#L392)

- **No current test covers wildcard expansion:** `logger-settings.test.ts` only covers the Vitest fast path and config-read behavior; there is no current assertion that `logging.file` expands `YYYY-MM-DD`.
  - file: [src/logging/logger-settings.test.ts:58](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/logging/logger-settings.test.ts#L58)
