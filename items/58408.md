## Summary

Current `main` contains a WhatsApp listener-state sharing fix that matches the reported symptom: inbound traffic can own the live socket while outbound CLI resolves the active listener through a shared global registry, with a dedicated runtime-boundary regression test covering the light/heavy module split.

## Evidence

- **Shared listener registry on global symbol:** `extensions/whatsapp/src/connection-controller-registry.ts:11` stores controllers in `globalThis[Symbol.for("openclaw.whatsapp.connectionControllerRegistry")]`, so separate runtime module instances share the same active-listener state.
