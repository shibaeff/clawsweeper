## Summary

Keep open. Current main appears to have fixed part of the original Discord slash-command routing problem by sending native commands to the resolved/bound agent session and enforcing channel guards before dispatch, but the requested feature is not implemented: `/new` and `/reset` still do not support `@agent` targeting or any other agent-selection argument, and there is no smart-default reset selection logic in the current command surface.

## Evidence

- **User-facing command contract:** The current docs still describe `/new [model]` and `/reset` only; there is no documented `/new @agent` or `/reset @agent` form.
  - file: [docs/tools/slash-commands.md:91](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/docs/tools/slash-commands.md#L91)
- **Built-in command definitions:** `reset` and `new` are defined with `acceptsArgs: true` only. They have no structured `agent` argument, no choices, and no mentionable-selector field.
  - file: [src/auto-reply/commands-registry.shared.ts:663](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/auto-reply/commands-registry.shared.ts#L663)
- **Discord native slash option shape:** When a command only has `acceptsArgs: true` and no explicit arg definitions, Discord registers a single string option named `input`, so there is no native agent-target selector for `/new` or `/reset`.
  - file: [extensions/discord/src/monitor/native-command.ts:678](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/discord/src/monitor/native-command.ts#L678)
- **Routing/session targeting on main:** Current main routes native commands through the resolved route and `commandTargetSessionKey`, and a Discord test asserts `/status` targets the routed `agent:qwen:discord:channel:...` session. This supports that part of the original wrong-agent symptom was addressed, but it does not implement explicit agent targeting for reset/new.
  - file: [extensions/discord/src/monitor/native-command.ts:1115](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/discord/src/monitor/native-command.ts#L1115)
- **Regression proof for routed Discord native commands:** The Discord native-command test expects the routed session key `agent:qwen:discord:channel:1478836151241412759`, showing current main intentionally routes native slash commands to the resolved agent/session.
  - file: [extensions/discord/src/monitor/native-command.plugin-dispatch.test.ts:744](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/discord/src/monitor/native-command.plugin-dispatch.test.ts#L744)

## Risks / Open Questions

- Part of the original report may already be obsolete: current native-command routing and guard checks likely fixed the wrong-agent and channel-parity symptoms, but the issue title and requested behavior still describe an unimplemented feature.
- Closing as implemented would be too aggressive because there is still no concrete `@agent` targeting surface for `/new` or `/reset` on current main.
