## Summary

Current `main` already covers this installed-plugin duplicate path: installed plugins recorded in `plugins.installs` get explicit duplicate-precedence handling, same-directory duplicates are suppressed, and plugin auto-enable no longer re-emits changes once the plugin is already enabled and allowlisted. The shipped release tag `v2026.4.22` includes the relevant fix commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`fix: allow installed plugins through allowlist`).

## Close Comment

Closing from a Codex review because this is already implemented on current `main` and shipped in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The current code now treats `plugins.installs` as explicit installed-plugin provenance in duplicate resolution (`src/plugins/manifest-registry.ts:453`), suppresses false duplicate warnings when candidates resolve to the same physical directory (`src/plugins/manifest-registry.ts:625`), and avoids re-auto-enabling already-enabled allowlisted plugins in doctor auto-enable (`src/config/plugin-auto-enable.shared.ts:759`). There is regression coverage for the installed-duplicate path in `src/plugins/loader.test.ts:5828` and for same-path duplicate-warning suppression in `src/plugins/manifest-registry.test.ts:922`. If anyone still hits this on a newer build, please open a fresh issue with the current `openclaw version`, exact warning text, and the relevant `plugins.allow` / `plugins.entries` / `plugins.installs` config.
