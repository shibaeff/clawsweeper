## Summary

Keep open. Current `main` still has the reported case-sensitive `memory/` checks in the memory-core extension, so the underlying bug is not implemented on `main` and the PR remains relevant even though it now targets pre-move paths and likely needs a rebase.

## Evidence

- **Current temporal-decay logic is still case-sensitive:** `isEvergreenMemoryPath` only accepts `MEMORY.md` exactly, then requires `normalized.startsWith("memory/")` and tests the dated-file regex against the original string. Uppercase or mixed-case `MEMORY/` paths are still rejected on current `main`.

- **Current default memory-path gate is still case-sensitive:** `isDefaultMemoryPath` still returns `normalized.startsWith("memory/")`, so `MEMORY/...` and `Memory/...` are not treated as default memory paths on current `main`. That path gate is used when resolving workspace reads.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:90](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/memory-core/src/memory/qmd-manager.ts#L90)
