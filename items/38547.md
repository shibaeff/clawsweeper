## Summary

Keep open. Current `main` does not implement native remote session delegation for `sessions_spawn`/`sessions_send` via `gatewayUrl`; the session tools stay on the local gateway/session path, while remote-gateway overrides exist only on other tools.

## Evidence

- **`sessions_spawn` has no remote-gateway params and dispatches to local spawn helpers:** The current tool schema includes `task`, `agentId`, `model`, `thinking`, `cwd`, `thread`, `mode`, `sandbox`, `context`, etc., but no `gatewayUrl`/`gatewayToken`. Its execute path calls `spawnAcpDirect(...)` or `spawnSubagentDirect(...)` directly with local session context.
  - file: [src/agents/tools/sessions-spawn-tool.ts:100](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/agents/tools/sessions-spawn-tool.ts#L100)
  - sha: [76a4c167f7ef](https://github.com/openclaw/openclaw/commit/76a4c167f7ef65040fd5b00d206c59b8bb28fd49)
- **`sessions_send` also lacks remote-gateway params and uses local `callGateway`/`agent`:** The schema only accepts `sessionKey`, `label`, `agentId`, `message`, and `timeoutSeconds`. Execution resolves a local `gatewayCall` and starts the nested run with `method: "agent"` against that gateway caller.
  - file: [src/agents/tools/sessions-send-tool.ts:37](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/agents/tools/sessions-send-tool.ts#L37)
  - sha: [76a4c167f7ef](https://github.com/openclaw/openclaw/commit/76a4c167f7ef65040fd5b00d206c59b8bb28fd49)
- **Remote gateway override plumbing exists, but session tools do not use it:** `src/agents/tools/gateway.ts` defines shared `gatewayUrl`/`gatewayToken` override handling, and tools like `nodes` expose those params in their schemas. The session tools do not.
  - file: [src/agents/tools/gateway.ts:19](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/agents/tools/gateway.ts#L19)
  - sha: [76a4c167f7ef](https://github.com/openclaw/openclaw/commit/76a4c167f7ef65040fd5b00d206c59b8bb28fd49)
- **Current docs still describe a single gateway as the owner of sessions/state:** Remote-access docs say one gateway host owns sessions, auth profiles, channels, and state; remote machines connect to that gateway rather than delegating session execution across gateways.
  - file: [docs/gateway/remote.md:10](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/docs/gateway/remote.md#L10)
  - sha: [76a4c167f7ef](https://github.com/openclaw/openclaw/commit/76a4c167f7ef65040fd5b00d206c59b8bb28fd49)
- **Session-tool docs do not document any remote-gateway delegation option:** The documented `sessions_spawn` options cover runtime/model/thread/sandbox/context and describe local sub-agent session behavior, with no `gatewayUrl`/remote execution path.
  - file: [docs/concepts/session-tool.md:101](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/docs/concepts/session-tool.md#L101)
  - sha: [76a4c167f7ef](https://github.com/openclaw/openclaw/commit/76a4c167f7ef65040fd5b00d206c59b8bb28fd49)
- **Security docs treat remote agent spawning as sensitive, not as an implemented session-tool feature:** Gateway HTTP hard-denies `sessions_spawn` by default and explicitly notes that spawning agents remotely is an RCE surface.
  - file: [docs/gateway/tools-invoke-http-api.md:104](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/docs/gateway/tools-invoke-http-api.md#L104)
  - sha: [76a4c167f7ef](https://github.com/openclaw/openclaw/commit/76a4c167f7ef65040fd5b00d206c59b8bb28fd49)
- **Checkout remained clean:** `git status --short` produced no output before and after inspection.
  - command: `git status --short`
