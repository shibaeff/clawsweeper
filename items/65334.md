## Summary

Current `main` already fixes the reported cosmetic version mismatch: plugin compatibility checks now prefer the runtime `VERSION` over a stale `OPENCLAW_VERSION` env value, and the shipped `v2026.4.23` tag contains both that code path and a regression test for the stale-env case.

## Evidence

- **Compatibility host version now prefers runtime VERSION for process env:** `resolveCompatibilityHostVersion()` uses `runtime-first` ordering when reading the real process environment, so a stale `OPENCLAW_VERSION` no longer overrides the actual runtime version during compatibility checks.
  - file: [src/version.ts:138](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/version.ts#L138)
  - command: `nl -ba src/version.ts | sed -n '138,151p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Plugin loader warnings use that compatibility resolver:** Plugin manifest loading still computes `currentHostVersion` via `resolveCompatibilityHostVersion(env)` before emitting the `plugin requires OpenClaw >=... but this host is ...` diagnostic, so the warning path now receives the corrected host version.
  - file: [src/plugins/manifest-registry.ts:624](https://github.com/openclaw/openclaw/blob/30aa7e0d4d225650b58c2fba79e71bffa5627b08/src/plugins/manifest-registry.ts#L624)
