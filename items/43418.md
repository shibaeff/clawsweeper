## Summary

Current `main` already implements the requested universal pre-LLM interception hook as `before_dispatch`, and the shipped release `v2026.4.22` contains it.

## Risks / Open Questions

- The shipped API is not identical to the issue sketch: it uses `handled`/`text` rather than `block`/`replyText`, and current metadata is split across event/context types.
- Current `before_dispatch` types do not include `messageId`; that appears to be follow-up scope rather than evidence that the core hook is missing.

## Close Comment

Closing after Codex review: current `main` already implements the requested pre-LLM interception hook as `before_dispatch`. The hook is part of the public plugin contract in `src/plugins/hook-types.ts:83` and `src/plugins/hook-types.ts:237`, the runner executes it with first-claim-wins semantics in `src/plugins/hooks.ts:701`, and the universal dispatch path calls it before normal model dispatch in `src/auto-reply/reply/dispatch-from-config.ts:682`. When a plugin returns `{ handled: true, text?: ... }`, OpenClaw short-circuits the normal dispatch path and can send the handled reply instead, with regression coverage in `src/auto-reply/reply/dispatch-from-config.test.ts:3271`. This was also shipped in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) and recorded in `CHANGELOG.md:2049`. The final API shape differs slightly from the proposal here, but the requested universal pre-LLM interception hook is implemented.
