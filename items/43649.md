## Summary

Keep open. I did not find strong evidence that `main` fixed #43649, and the current Web UI chat-history path still has an unguarded render failure mode consistent with the report: persisted history is accepted mostly raw in `ui/src/ui/controllers/chat.ts`, then every message is normalized during render in `ui/src/ui/chat/build-chat-items.ts`, where `ui/src/ui/chat/message-normalizer.ts` still dereferences `item.type` inside `m.content.flatMap(...)` without first proving each history block is an object. That makes a malformed saved transcript a plausible current cause of the reported вЂњclick sessions/chat -> Loading data -> UI freezes until the session is deletedвЂќ behavior.

## Evidence

- **Raw history is passed through with minimal filtering:** `loadChatHistory` stores `chat.history` results directly into `state.chatMessages` after only filtering NO_REPLY / synthetic repair messages; it does not sanitize nested content blocks before the UI renders them.
  - file: [ui/src/ui/controllers/chat.ts:179](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/ui/src/ui/controllers/chat.ts#L179)

  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Chat rendering normalizes every persisted message:** `buildChatItems` calls `normalizeMessage(msg)` for each history entry while building the chat view, so any malformed stored message shape reaches the renderer on page load.
  - file: [ui/src/ui/chat/build-chat-items.ts:215](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/ui/src/ui/chat/build-chat-items.ts#L215)
