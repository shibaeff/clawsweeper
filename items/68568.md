## Summary

Current `main` and the shipped `v2026.4.23` release already cover the duplicate-final concern in the Telegram lane deliverer, while this PR's extra archived-preview short-circuit would regress existing fallback/media/error behavior that current tests explicitly protect.

## Evidence

- **Same-text final preview edits are already treated as delivered:** `tryEditPreviewMessage` treats Telegram's `message is not modified` response as a successful final delivery and does not fall back to `sendPayload`, which already covers the normal 'final text matches visible preview text' case without creating a duplicate message.
  - file: [extensions/telegram/src/lane-delivery-text-deliverer.ts:250](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/telegram/src/lane-delivery-text-deliverer.ts#L250)
  - command: `nl -ba extensions/telegram/src/lane-delivery-text-deliverer.ts | sed -n '225,309p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Fallback sends still clean up consumed archived previews:** When an archived preview cannot be finalized by edit, current `main` sends the replacement payload and then deletes the consumed archived preview. That preserves the non-duplicate behavior the PR says it must not change.
  - file: [extensions/telegram/src/lane-delivery-text-deliverer.ts:461](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/telegram/src/lane-delivery-text-deliverer.ts#L461)

Those are the exact paths the PR review comments flagged as regressions.

That means matching text alone is not sufficient to suppress a final send when the payload also carries media or an error, which matches the review concerns on this PR.
