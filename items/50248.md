## Summary

Keep open. Current main contains a partial fix for new cron/isolated session rollovers, but the `sessions cleanup --fix-missing` path still trusts persisted `sessionFile` metadata and does not fall back to the canonical `<sessionId>.jsonl` transcript for already-misaligned store entries.

## Evidence

- **Cleanup still prunes from resolved `sessionFile` only:** `pruneMissingTranscriptEntries()` resolves one transcript path for each entry and deletes the entry if that path does not exist. There is no second check for the canonical transcript derived from `sessionId` before pruning.
  - file: [src/commands/sessions-cleanup.ts:139](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/commands/sessions-cleanup.ts#L139)
  - command: `nl -ba src/commands/sessions-cleanup.ts | sed -n '139,158p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Path resolver still prefers persisted `sessionFile`:** `resolveSessionFilePath()` returns `entry.sessionFile` when it is a valid in-store path, and only falls back to `<sessionId>.jsonl` when the stored path is invalid/outside the sessions dir. It does not fall back when the stored file is simply missing or mismatched to `sessionId`.
  - file: [src/config/sessions/paths.ts:263](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/config/sessions/paths.ts#L263)
  - command: `nl -ba src/config/sessions/paths.ts | sed -n '263,278p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Main has the rollover-side fix:** Fresh cron/isolated sessions now clear inherited `sessionFile` on new-session rollover, which matches one of the issue's proposed remedies and should prevent new stale pointers from being created in this path.
  - file: [src/cron/isolated-agent/session.ts:68](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/cron/isolated-agent/session.ts#L68)
  - command: `nl -ba src/cron/isolated-agent/session.ts | sed -n '68,89p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Main also recomputes transcript path when `sessionId` changes:** `resolveAndPersistSessionFile()` now avoids reusing the persisted transcript path when the logical session rotates to a new `sessionId`, and persists the recomputed path.
  - file: [src/config/sessions/session-file.ts:21](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/config/sessions/session-file.ts#L21)
  - command: `nl -ba src/config/sessions/session-file.ts | sed -n '21,40p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)

  - file: [src/config/sessions/sessions.test.ts:400](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/config/sessions/sessions.test.ts#L400)

It does not document a `sessions cleanup --fix-missing` canonical-path fallback.

## Risks / Open Questions

- Already-corrupted session stores from pre-fix releases can still hit false `prune-missing` decisions because cleanup currently follows the persisted `sessionFile` first.
- I did not run a live reproduction because this review was constrained to read-only inspection and the checkout does not have `pnpm` available (`pnpm docs:list` failed with `pnpm: command not found`).
- If maintainers consider the rollover fix sufficient for this issue, the remaining cleanup hardening may need to be split into a separate issue instead of closing this one directly.
