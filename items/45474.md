## Summary

Keep open. Current `main` still has the same false-healthy shape the report describes: WhatsApp is marked `healthy` as soon as the socket connects, `lastInboundAt` stays `null` until a real inbound message is processed, and the stale/dead-session watchdog does not trip until the 30-minute timeout. There is also still no WhatsApp-specific `probeAccount`, so `channels status --probe` cannot actively verify message delivery for this channel.

## Evidence

- **Health flips to healthy on connect before any inbound traffic:** `noteConnected()` sets `healthState = "healthy"`; `lastInboundAt` starts as `null` and is only updated by `noteInbound()`. That means a newly connected but non-delivering session can still look healthy.
  - file: [extensions/whatsapp/src/auto-reply/monitor-state.ts:36](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/whatsapp/src/auto-reply/monitor-state.ts#L36)
- **Dead-session detection is delayed by a 30 minute watchdog:** `monitorWebChannel()` defaults `messageTimeoutMs` to `30 * 60 * 1000`, and the controller watchdog only marks the channel stale / forces reconnect after that window.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:183](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/whatsapp/src/auto-reply/monitor.ts#L183)
- **Watchdog uses connect time when no inbound has ever arrived:** The controller computes staleness from `lastInboundAt ?? startedAt`, so a session with `lastInboundAt: null` remains apparently healthy until the timeout expires.
  - file: [extensions/whatsapp/src/connection-controller.ts:563](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/whatsapp/src/connection-controller.ts#L563)
- **`channels status --probe` only probes channels that implement `probeAccount`:** Gateway status calls `plugin.status.probeAccount` only when that hook exists. A repo search for `probeAccount` / `auditAccount` under `extensions/whatsapp` returned no hits, so WhatsApp has no active probe implementation here.
  - file: [src/gateway/server-methods/channels.ts:199](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/server-methods/channels.ts#L199)
  - command: `rg -n "probeAccount\s*:|auditAccount\s*:" extensions/whatsapp`
- **The report's UX bug is still present on main:** Current conflict/logged-out runtime errors still tell operators to relink with `openclaw channels login --channel web`, even though the documented/current CLI uses `--channel whatsapp`.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:463](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/whatsapp/src/auto-reply/monitor.ts#L463)

## Risks / Open Questions

- Some of the user's 440/401 behavior may still depend on external WhatsApp/Baileys state, but the current code does not provide strong evidence that the false-healthy inbound-drop path is fixed.
