## Summary

Keep open. Current `main` still does not treat `application/x-cfb` or `application/msword` as binary in the media-understanding path, and current regression coverage only locks in `application/vnd.*` Office cases, not legacy `.doc` containers.

## Evidence

- **Current main still lacks the requested binary MIME entries:** `isBinaryMediaMime()` on current `main` only treats images/audio/video, `application/octet-stream`, archive mimes, `+zip`, and `application/vnd.*` as binary. `application/x-cfb` and `application/msword` are absent.
  - file: [src/media-understanding/apply.ts:309](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/media-understanding/apply.ts#L309)
  - command: `nl -ba src/media-understanding/apply.ts | sed -n '309,350p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Non-binary file mimes still flow into text-coercion heuristics:** The attachment pipeline skips a file only if `isBinaryMediaMime(normalizedRawMime)` returns true; otherwise it continues into UTF-16/UTF-8 text heuristics and MIME override logic. That matches the failure mode described in the PR body.
