## Summary

Current `main` already includes the fix for this false-positive rate-limit path: cooldown windows with no recorded failure history now resolve to `unknown` instead of `rate_limit`, which matches the issue's symptom of showing `API rate limit reached` without an upstream provider request.

## Evidence

- **Changelog entry:** `CHANGELOG.md` lists this exact fix under release `2026.3.11`: "Agents/cooldowns: default cooldown windows with no recorded failure history to `unknown` instead of `rate_limit`, avoiding false API rate-limit warnings while preserving cooldown recovery probes."
  - file: [CHANGELOG.md:2738](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/CHANGELOG.md#L2738)
  - command: `nl -ba CHANGELOG.md | sed -n '2687,2745p'`
- **Current implementation:** `resolveProfilesUnavailableReason` now falls back to `unknown` when an active cooldown has no recorded failure counts, with an inline note that the previous behavior caused false `rate limit reached` warnings.
  - file: [src/agents/auth-profiles/usage.ts:281](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/agents/auth-profiles/usage.ts#L281)
