## Summary

Keep open. Current `main` does not implement the requested behavior: the Control UI still bootstraps gateway auth from `#token=`/legacy `?token=` links or manual entry, intentionally stores the token in `sessionStorage` rather than `localStorage`, and I found no OpenClaw-side Paperclip hire-flow code that could auto-populate `adapterConfig.headers["x-openclaw-token"]` for `openclaw_gateway` agents.

## Evidence

- **Control UI token storage is intentionally session-scoped, not localStorage-backed:** `loadSettings()` reads the gateway token from `sessionStorage`, and `persistSettings()` writes only non-secret UI settings to `localStorage` while persisting the token separately in session storage.
  - file: [ui/src/ui/storage.ts:154](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/ui/src/ui/storage.ts#L154)
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Tests assert tokens are not persisted in localStorage:** The storage test explicitly verifies that saving settings with a token leaves `localStorage` without a `token` field and keeps the auth token only in session storage.
  - file: [ui/src/ui/storage.node.test.ts:230](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/ui/src/ui/storage.node.test.ts#L230)
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **URL bootstrap still relies on `#token=` / legacy `?token=` import:** `applySettingsFromUrl()` imports `token` from the URL fragment or query string, applies it to settings, and strips it from the URL; there is no `localStorage` token bootstrap path here.
  - file: [ui/src/ui/app-settings.ts:187](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/ui/src/ui/app-settings.ts#L187)
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Current docs still tell users to use a tokenized dashboard URL or paste the token manually:** The dashboard/help/troubleshooting docs say dashboard URL tokens are kept in `sessionStorage` for the current tab and tell users to paste the token into Control UI settings when auth is missing.
  - file: [docs/web/dashboard.md:31](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/docs/web/dashboard.md#L31)
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **The shipped design explicitly rejects long-lived localStorage token persistence:** The changelog entry for `2026.3.11` says OpenClaw moved dashboard auth tokens to session-scoped browser storage specifically to avoid restoring long-lived localStorage token persistence.
  - file: [CHANGELOG.md:2713](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/CHANGELOG.md#L2713)
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **No OpenClaw-side Paperclip hire-flow integration code found:** Repo search for `Paperclip` and `openclaw_gateway` found changelog/test-script/completion references, but no current source/UI code in this repo that would own Paperclip agent creation or inject `x-openclaw-token` headers during a Paperclip "Hire Agent" flow.
  - command: `rg -n "Paperclip|openclaw_gateway" . --glob '!**/node_modules/**' --glob '!**/dist/**'`
