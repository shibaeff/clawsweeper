## Summary

Issue #35059 is a useful security idea, but it fits the ClawHub/community skill-plugin surface rather than OpenClaw core. Current main’s project scope explicitly pushes new skills to ClawHub first, and core already has built-in security-audit/code-safety hooks for installed skills and gateway-backed installs.

## Risks / Open Questions

- Current main does not pre-scan the `openclaw skills install <slug>` ClawHub download path before extraction; the existing built-in coverage is post-install audit plus gateway-backed install scanning, so the security need itself is real even if the right home is ClawHub.
- If maintainers later want registry-side moderation or a mandatory pre-install scan in core/ClawHub service, this issue’s scope decision may need to be revisited.

## Close Comment

Closing as a `clawhub` item after Codex review. The request is coherent and useful, but it fits the OpenClaw project’s external skill/plugin scope rather than core. `VISION.md:66-70` says new skills should be published to ClawHub first, and `VISION.md:99-101` explicitly lists new core skills that can live on ClawHub as out of scope for core. The current docs also position ClawHub as the publication surface for skills/plugins (`docs/tools/clawhub.md:10-16`, `docs/tools/clawhub.md:61-71`).

Current main already has core security primitives here: `openclaw security audit --deep` scans non-bundled installed skills for dangerous/suspicious code via `skills.code_safety` (`src/security/audit-extra.async.ts:949-1005`), and gateway-backed skill installs already run a built-in dangerous-code scanner (`docs/tools/skills.md:131-137`). Given that this proposal is itself a standalone scanner from another repo, the right path is to ship and maintain it as a ClawHub skill/plugin or adjacent community tool rather than merge it into OpenClaw core.
