## Summary

Current main already avoids the reported undercount in the real memory-flush path. The flush runner computes a projected token count as prompt tokens + last output tokens + current prompt estimate, reads transcript usage when near the threshold so missing output tokens can change the decision, and passes that projected count into `shouldRunMemoryFlush`. The same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Close Comment

Closing from a Codex review because current OpenClaw already handles this in the live memory-flush path. `src/auto-reply/reply/agent-runner-memory.ts:112` computes the projected next-context size as prompt tokens + last output tokens + the new prompt estimate, `src/auto-reply/reply/agent-runner-memory.ts:588` reads transcript usage when the session is close enough to threshold that missing output tokens could change the result, and `src/auto-reply/reply/agent-runner-memory.ts:673` / `src/auto-reply/reply/agent-runner-memory.ts:704` pass that projected total into `shouldRunMemoryFlush`. I also checked the shipped `v2026.4.22` release commit (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and it contains the same implementation. The helper in `memory-flush.ts` still has a fallback path, but the production caller no longer relies on bare cached `totalTokens` in the reported near-threshold case.
