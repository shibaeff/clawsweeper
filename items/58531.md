## Summary

Current `main` already handles the reported persisted-selection/failover loop: fallback normalizes `LiveSessionModelSwitchError` instead of rethrowing into an outer retry loop, auto failover overrides are cleared on the next turn, and regression tests/docs cover the overload-loop scenario. The shipped `v2026.4.23` tag already contains this behavior.

## Evidence

- **Fallback loop no longer rethrows live-switch conflicts:** `runWithModelFallback` treats `LiveSessionModelSwitchError` as a failover condition so the chain continues or surfaces a normalized failure instead of restarting the outer loop with the conflicting persisted model.

- **Regression test matches the reported infinite overload loop:** The test explicitly documents the old behavior as restarting with the overloaded model indefinitely, and now asserts the error is not rethrown as `LiveSessionModelSwitchError` on the last candidate.
  - file: [src/agents/model-fallback.test.ts:459](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/model-fallback.test.ts#L459)

- **Per-turn selection healing clears auto failover overrides:** Current session model selection treats auto failover overrides as transient, clears them, and retries the configured primary on the next turn instead of letting persisted auto state pin the session back to the fallback/old model.

- **Regression test covers auto-override clearing:** The model-selection test asserts that an auto failover override is cleared and the configured primary is retried, which is the intended antidote to persisted live-selection pinning across turns.
  - file: [src/auto-reply/reply/model-selection.test.ts:635](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/auto-reply/reply/model-selection.test.ts#L635)
