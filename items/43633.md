## Summary

Current main already implements the Feishu/Lark routing behaviors this issue reports: inbound Feishu dispatch passes the runtime account id and group/direct peer kind into route resolution, legacy `lark` channel ids normalize to `feishu`, and routing explicitly supports `group`/`channel` peer-kind compatibility with coverage in tests and docs.

## Close Comment

Closing from a Codex review because current main already implements this routing path. Feishu inbound dispatch now passes the active `account.accountId` and `peer.kind: "group"` / `"direct"` into `resolveAgentRoute()` (`extensions/feishu/src/bot.ts:590`, `extensions/feishu/src/comment-handler.ts:136`). Routing also normalizes legacy `channel: "lark"` bindings to `feishu` (`src/channels/ids.ts:46`, `extensions/feishu/package.json:34`) and treats `group` and `channel` peer kinds as compatible (`src/routing/peer-kind-match.ts:3`, `src/routing/resolve-route.test.ts:730`). The current docs/tests also lock the account behavior, including that omitted `accountId` means default-account-only (`src/routing/resolve-route.test.ts:451`, `docs/concepts/multi-agent.md:245`). This implementation is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this issue is no longer actionable on main. If someone still reproduces it on `v2026.4.22` or newer, they should open a fresh issue with the exact binding config and current logs.
