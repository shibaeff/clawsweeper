## Summary

Current main already implements this cleanup via generic model compatibility metadata (`compat.supportsStore`) and centralized provider capability resolution, with tests and docs covering Azure and other non-supporting providers.

## Risks / Open Questions

## Close Comment

Closing as implemented on main based on this Codex review. Current OpenClaw no longer needs an Azure-only `store` special case: `src/agents/openai-responses-payload-policy.ts:101` resolves behavior through provider capabilities using `model.compat`, and `src/agents/provider-attribution.ts:681` computes the Responses `store` gates from `compat.supportsStore` rather than a hardcoded provider check. Coverage exists for both Azure and other OpenAI-compatible providers in `src/agents/pi-embedded-runner-extraparams.test.ts:3137` and `src/agents/openai-responses-payload-policy.test.ts:62`, and the behavior is documented in `docs/providers/openai.md:671`. This is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so there is no remaining maintainer cleanup item to track here.
