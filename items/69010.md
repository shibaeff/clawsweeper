## Summary

Current `main` still preserves persisted session delivery context over the live gateway request, still omits `runContext.currentChannelId` in this path, and still carries tests that assert the old preserved-session behavior. This PR appears to address a real, still-open gateway routing bug rather than something already fixed or obsolete.

## Evidence

- **Current main still makes stored delivery context authoritative:** `agent.ts` still computes `effectiveDelivery` with `mergeDeliveryContext(deliveryFields.deliveryContext, requestDeliveryHint)`, so the stored session context is primary. In `mergeDeliveryContext`, conflicting channels keep the primary channel/to/accountId/threadId, which means stale persisted route data still wins over the current request in this call site.
  - file: [src/gateway/server-methods/agent.ts:675](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/gateway/server-methods/agent.ts#L675)

  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Session patching still preserves old route fields:** After normalizing `effectiveDeliveryFields`, the session patch still falls back to `entry?.lastTo`, `entry?.lastAccountId`, and `entry?.lastThreadId` when the new delivery fields are absent. That is the exact stale-route persistence shape this PR is trying to correct.
