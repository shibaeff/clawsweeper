## Summary

Current OpenClaw already implements the requested behavior: iMessage DM policy defaults to `pairing`, `dmPolicy: "open"` is now explicitly documented as requiring `allowFrom: ["*"]`, invalid `open` configs are rejected by schema validation, and the iMessage setup flow auto-adds `"*"` when operators choose `open`. The same behavior is present in the shipped `v2026.4.23` release snapshot.

## Evidence

- **iMessage config rejects invalid open mode:** Top-level and per-account iMessage config validation now requires `allowFrom` to include `"*"` when `dmPolicy` is `open`, and requires a non-empty allowlist for `allowlist` mode.
  - file: [src/config/zod-schema.providers-core.ts:1362](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/config/zod-schema.providers-core.ts#L1362)

- **Setup flow keeps pairing default and repairs open mode:** The iMessage setup helper reads `pairing` as the default DM policy, and when `open` is selected it patches config to merge `"*"` into `allowFrom` automatically.
  - file: [extensions/imessage/src/setup-core.ts:123](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/imessage/src/setup-core.ts#L123)

- **Regression tests cover this exact contract:** Dedicated tests assert that iMessage accepts `dmPolicy: "open"` only with `allowFrom: ["*"]`, rejects `open` without the wildcard, and defaults DM policy to `pairing`.
  - file: [extensions/imessage/src/config-schema.test.ts:4](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/imessage/src/config-schema.test.ts#L4)

- The iMessage status surface still distinguishes setup/configuration from DM authorization, so operators can still misunderstand pairing vs open behavior even though the contract is now documented and validated.
