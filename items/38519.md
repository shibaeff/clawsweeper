## Summary

Keep open. Current `main` has lower-level browser download capture in the control API/CLI, plus proxy file-path remapping for `download.path`, but the agent-facing `browser` tool still does not expose `download` or `waitfordownload` actions. That means the reported first-class agent workflow gap is not fully implemented on `main`.

## Evidence

- **Agent browser tool schema omits download actions:** `BROWSER_TOOL_ACTIONS` only includes `status`, `start`, `stop`, `profiles`, `tabs`, `open`, `focus`, `close`, `snapshot`, `screenshot`, `navigate`, `console`, `pdf`, `upload`, `dialog`, and `act`. There is no `download` or `waitfordownload` action in the agent-facing schema.
  - file: [extensions/browser/src/browser-tool.schema.ts:18](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/extensions/browser/src/browser-tool.schema.ts#L18)
  - command: `nl -ba extensions/browser/src/browser-tool.schema.ts | sed -n '1,220p'`
  - sha: [76a4c167f7ef](https://github.com/openclaw/openclaw/commit/76a4c167f7ef65040fd5b00d206c59b8bb28fd49)
- **Agent browser tool executor has no download handling:** The `switch (action)` implementation handles the same action set and falls through to `Unknown action` in the default case. There is no `case "download"` or `case "waitfordownload"`, so standard agent browser calls cannot request download capture.
  - file: [extensions/browser/src/browser-tool.ts:466](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/extensions/browser/src/browser-tool.ts#L466)

- **Download capture exists only in lower-level control routes:** The browser control server does implement `POST /wait/download` and `POST /download`, returning `{ ok: true, targetId, download: result }`. This shows partial implementation, but at the route/API layer rather than the normal agent browser tool surface.

## Risks / Open Questions

- The existence of `/download` and `waitfordownload` at the browser-control/CLI layer makes this look implemented at a glance, but the normal agent-facing `browser` tool still cannot invoke those flows directly.
- Even where download capture exists, docs still describe download interception as a managed-browser feature, so behavior is not universally available across browser modes.
