## Summary

Keep open. Current `main` has newer LINE multi-account/webhook lifecycle code and some LINE webhook fixes, but I did not find strong evidence that issue #50306's specific symptom (named LINE webhook paths returning 404 until a config save/reload) is already fixed on `main` or obsolete enough to close conservatively.

## Evidence

- **Current docs still describe the same multi-account LINE shape reported by the issue:** The LINE docs explicitly support multiple accounts with per-account `webhookPath`, which matches the reporter's setup of two LINE channels with two webhook paths.
  - file: [docs/channels/line.md:95](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/docs/channels/line.md#L95)

- **LINE webhook routes are registered dynamically and only removed on stop/abort:** `monitorLineProvider()` registers the account webhook route and unregisters it only from `stopHandler()`, which runs on explicit stop or abort. That makes the reported 404 symptom about route disappearance plausible, but this file alone does not prove a fix for that regression.
  - file: [extensions/line/src/monitor.ts:305](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/line/src/monitor.ts#L305)

- **If a LINE route exists, verification should answer 200 on GET:** The current webhook handler returns `200 OK` for `GET` and `204` for `HEAD`. A LINE Developers verify request getting `404` therefore still points to the route not being present, not just a signature/verification bug.
