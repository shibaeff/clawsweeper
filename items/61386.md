## Summary

Current `main` still has the overflow path this PR fixes. `scheduleNext()` passes the raw heartbeat delay into `setTimeout` with no clamp, Node 24 still overflows a 30-day timeout to 1 ms, and heartbeat duration parsing still accepts day-based values, so this remains a real bug rather than stale cleanup.

## Evidence

- **Current scheduler is still unclamped:** `scheduleNext()` computes `const delay = Math.max(0, nextDue - now)` and passes it straight to `setTimeout(...)`; there is no `2^31 - 1` guard on current `main`.
  - file: [src/infra/heartbeat-runner.ts:1389](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/infra/heartbeat-runner.ts#L1389)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '1368,1390p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Heartbeat config validation still routes through generic duration parsing:** `HeartbeatSchema` accepts `every` by calling `parseDurationMs(val.every, { defaultUnit: "m" })`; it does not impose an upper bound before the scheduler sees the parsed value.
  - file: [src/config/zod-schema.agent-runtime.ts:46](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/config/zod-schema.agent-runtime.ts#L46)
  - command: `nl -ba src/config/zod-schema.agent-runtime.ts | sed -n '18,58p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Duration parser supports day units:** `parseDurationMs` includes `d` in `DURATION_MULTIPLIERS`, so `30d` is a valid parsed heartbeat interval on current `main`.
  - file: [src/cli/parse-duration.ts:8](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/cli/parse-duration.ts#L8)
  - command: `sed -n '1,40p' src/cli/parse-duration.ts`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Node still reproduces the overflow behavior described in the PR:** On the repo host's Node `v24.14.1`, `setTimeout(..., 2592000000)` yields `_idleTimeout === 1` and emits `TimeoutOverflowWarning`, matching the PR's reported failure mode for a 30-day heartbeat interval.
  - command: `node -e "const t=setTimeout(()=>{},2592000000); console.log(process.version); console.log(t._idleTimeout); clearTimeout(t);"`
- **PR discussion aligns with current behavior:** The PR body and Greptile summary both describe the same 32-bit overflow and proposed clamp; local inspection of `main` and the Node reproduction both support that diagnosis, so the PR still appears relevant.
