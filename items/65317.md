## Summary

Keep open. Current `main` still routes `functions.write` through a pinned mutation helper that requires `python3`/`python`, while the default-image bootstrap path can still create `openclaw-sandbox:bookworm-slim` by tagging plain `debian:bookworm-slim`. That leaves the reported shell-write-succeeds but `functions.write`-fails mismatch still credible on current `main`, so there is not strong evidence to close this as fixed or obsolete.

## Evidence

- **`functions.write` still depends on the pinned Python helper:** `SandboxFsBridge.writeFile()` resolves the target and then calls `buildPinnedWritePlan(...)`; that plan shells out to a helper that searches for `python3`/`python` and exits with `sandbox pinned mutation helper requires python3 or python` when neither exists.
  - file: [src/agents/sandbox/fs-bridge.ts:74](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/agents/sandbox/fs-bridge.ts#L74)
  - command: `nl -ba src/agents/sandbox/fs-bridge.ts | sed -n '1,220p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Current helper error matches the issue exactly:** `buildPinnedMutationPlan()` still emits the exact stderr text from the report after probing absolute Python candidates and `command -v python3` / `python`.
  - file: [src/agents/sandbox/fs-bridge-mutation-helper.ts:293](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/agents/sandbox/fs-bridge-mutation-helper.ts#L293)
  - command: `nl -ba src/agents/sandbox/fs-bridge-mutation-helper.ts | sed -n '250,380p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Default-image auto-bootstrap still uses plain Debian, not the repo sandbox Dockerfile:** When the configured image is the default and missing locally, `ensureDockerImage()` still does `docker pull debian:bookworm-slim` and `docker tag debian:bookworm-slim openclaw-sandbox:bookworm-slim`. That path can produce a sandbox image without the helper runtime dependencies even though file tools assume they exist.
  - file: [src/agents/sandbox/docker.ts:289](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/agents/sandbox/docker.ts#L289)

- **Repo build path for the default sandbox image expects Python to be present:** The checked-in default sandbox Dockerfile installs `python3`, and `scripts/sandbox-setup.sh` builds `openclaw-sandbox:bookworm-slim` from that Dockerfile. This conflicts with the plain-Debian fallback above and supports the reported runtime mismatch.
  - file: [Dockerfile.sandbox:3](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/Dockerfile.sandbox#L3)

That workaround matches the current helper dependency and the default-image mismatch found in `main`.
