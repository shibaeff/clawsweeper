## Summary

Current main already avoids Feishu card splits across tool-call boundaries by forcing Feishu reply dispatchers to disable generic block streaming and by reusing one Card Kit streaming session that updates the same card until final close. The shipped v2026.4.22 tag contains the same implementation and changelog entry.

## Evidence

- **Feishu reply dispatcher disables generic block streaming:** The Feishu reply dispatcher returns `replyOptions.disableBlockStreaming: true`, so the core block-streaming path that would emit separate reply blocks is turned off for Feishu replies.
  - file: [extensions/feishu/src/reply-dispatcher.ts:494](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/feishu/src/reply-dispatcher.ts#L494)
  - command: `nl -ba extensions/feishu/src/reply-dispatcher.ts | sed -n '340,540p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Single streaming card session is updated, not recreated, during a reply:** The dispatcher starts one `FeishuStreamingSession`, feeds block/final text into that active session, and only closes it on final/idle. While active, reply text is merged into the current stream instead of sending new cards.
  - file: [extensions/feishu/src/reply-dispatcher.ts:391](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/feishu/src/reply-dispatcher.ts#L391)

- **Streaming card implementation patches the existing card content/settings:** `FeishuStreamingSession.update()` issues PUT updates to the existing card element, and `close()` PATCHes card settings to end streaming mode. That matches the issue's requested reuse of one card across mid-turn updates.
  - file: [extensions/feishu/src/streaming-card.ts:297](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/feishu/src/streaming-card.ts#L297)

## Risks / Open Questions

- The Feishu docs still describe `channels.feishu.blockStreaming` as enabled by default in `docs/channels/feishu.md:245,399`, which can be confusing now that reply dispatch forces `disableBlockStreaming` for this path.

## Close Comment

Closing as implemented on current `main`. Codex review found that Feishu reply dispatch now forces `disableBlockStreaming: true` in `extensions/feishu/src/reply-dispatcher.ts:494`, which prevents generic block replies from splitting one turn into multiple bubbles, and the same dispatcher keeps a single `FeishuStreamingSession` open across the reply (`extensions/feishu/src/reply-dispatcher.ts:391-425`) while `extensions/feishu/src/streaming-card.ts:297-444` updates and closes that existing card with Card Kit PUT/PATCH calls. This is also covered by regression tests in `extensions/feishu/src/reply-dispatcher.test.ts:278-326`. The shipped `v2026.4.22` release already contains this behavior and includes the related changelog entry for `#38258`, so there is not an open maintainer action left on `main` for this report.
