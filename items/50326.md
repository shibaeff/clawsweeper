## Summary

Keep open. Current `main` still appears to drop the implicit Telegram reply target for normal message delivery, so the reported `replyToMode: "all"` behavior in group/forum topics is not clearly fixed on `main` or in the latest release.

## Evidence

- **Delivery path still ignores ambient/default reply target:** `deliverReplies` derives `replyToId` only from `reply.replyToId`; there is no fallback to a message-level/default reply target before text/media send helpers run.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:679](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/telegram/src/bot/delivery.replies.ts#L679)
  - command: `nl -ba extensions/telegram/src/bot/delivery.replies.ts | sed -n '675,690p;720,755p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **General Telegram dispatch computes trigger message id but never passes it into final delivery:** The dispatch path computes `draftReplyToMessageId` from `msg.message_id` for previews, but `deliveryBaseOptions` has no `replyToId`, and the normal `sendPayload` call forwards raw payloads to `deliverReplies` unchanged.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:331](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/telegram/src/bot-message-dispatch.ts#L331)

- **Only native commands inject the triggering Telegram message id:** The native-command path wraps payloads with `replyToId: String(msg.message_id)` when absent, which highlights that the general message path lacks the same fallback.
  - file: [extensions/telegram/src/bot-native-commands.ts:956](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/telegram/src/bot-native-commands.ts#L956)
