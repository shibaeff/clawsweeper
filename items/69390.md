## Summary

Current `main` now propagates explicit/global Undici stream timeouts into the guarded HTTP/1.1 dispatchers used by Ollama requests, with regression tests for inherited `headersTimeout`/`bodyTimeout`; the latest tagged release `v2026.4.23` still lacked that wiring, so the fix is implemented on `main` but not tied to a shipped release from this review.

## Evidence

- **Issue timeline already pointed at the matching fix:** The provided timeline cross-references PR #69433 (`fix(infra/net): propagate global stream timeout onto pinned http/1.1 dispatchers`), which matches the root cause described in the issue body (`fetch failed | Headers Timeout Error` before first token).
- **Released code still missed timeout propagation:** In tag `v2026.4.23`, `fetchWithSsrFGuard` created env/direct/pinned dispatchers without forwarding any timeout, so pinned HTTP/1.1 dispatcher requests could still hit Undici's own timeout instead of the configured OpenClaw timeout.
  - file: [src/infra/net/fetch-guard.ts:340](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/infra/net/fetch-guard.ts#L340)
  - command: `git show v2026.4.23:src/infra/net/fetch-guard.ts | nl -ba | sed -n '330,395p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Current main forwards timeout into guarded dispatchers:** On current `main`, `fetch-guard` resolves a dispatcher timeout from either explicit `timeoutMs` or the global Undici timeout bridge, then passes it into `createHttp1EnvHttpProxyAgent`, `createPolicyDispatcherWithoutPinnedDns`, and `createPinnedDispatcher`. That is the exact behavior needed for slow Ollama first-token latency.
  - file: [src/infra/net/fetch-guard.ts:29](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/infra/net/fetch-guard.ts#L29)
  - command: `nl -ba src/infra/net/fetch-guard.ts | sed -n '1,80p' && nl -ba src/infra/net/fetch-guard.ts | sed -n '330,420p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Global timeout is now bridged into guarded fetch:** `ensureGlobalUndiciStreamTimeouts` now stores `_globalUndiciStreamTimeoutMs` before dispatcher setup, and `fetch-guard` reads that bridge. In `v2026.4.23`, the same function did not expose that timeout to guarded fetch.
  - file: [src/infra/net/undici-global-dispatcher.ts:124](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/infra/net/undici-global-dispatcher.ts#L124)
  - command: `nl -ba src/infra/net/undici-global-dispatcher.ts | sed -n '118,130p' && git show v2026.4.23:src/infra/net/undici-global-dispatcher.ts | nl -ba | sed -n '108,125p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Regression tests cover the reported failure mode:** Current tests explicitly assert that guarded direct dispatchers inherit the configured global stream timeout and set both `headersTimeout` and `bodyTimeout` to `1_900_000`, which directly covers the reported ~5 minute headers-timeout gap.
  - file: [src/infra/net/fetch-guard.ssrf.test.ts:1049](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/infra/net/fetch-guard.ssrf.test.ts#L1049)
