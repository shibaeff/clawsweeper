## Summary

Keep open. Current `main` still collapses ACP backend errors to `end_turn` except for `errorKind === "refusal"`, so the reported gap is real and not implemented on `main`.

## Evidence

- **Translator still maps most ACP errors to end_turn:** `state === "error"` reads `payload.errorKind`, but only `"refusal"` maps to `stopReason: "refusal"`; everything else resolves as `"end_turn"`.
  - file: [src/acp/translator.ts:967](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/acp/translator.ts#L967)
  - command: `nl -ba src/acp/translator.ts | sed -n '967,970p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Prompt response surface preserves only stopReason:** `finishPrompt` resolves the ACP prompt with `{ stopReason }` only, with no structured backend error payload carried through to clients.
  - file: [src/acp/translator.ts:1025](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/acp/translator.ts#L1025)

## Risks / Open Questions

- ACP clients still cannot distinguish timeout, rate-limit, or server-error conditions from normal end-of-turn completion.
- Retry and UX logic built on ACP `stopReason` remain lossy for backend failures.
- Closing this as already fixed would hide an acknowledged gap that current code, tests, and docs still preserve.
