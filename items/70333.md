## Summary

Keep open. Current `main` still persists auto-enable results into config during gateway startup and `doctor`, even though current reload code now has a startup-hash guard that appears to reduce the older self-restart churn. The issue's requested behavior is therefore not implemented on `main`.

## Evidence

- **Gateway startup still writes auto-enable output back to config:** `loadGatewayStartupConfigSnapshot` computes `applyPluginAutoEnable(...)` and, when there are changes, calls `writeConfigFile(autoEnable.config)`. That is the persistence path the issue is asking to remove.
  - file: [src/gateway/server-startup-config.ts:120](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/gateway/server-startup-config.ts#L120)

  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Auto-enable materialization still injects `enabled: true`:** `registerPluginEntry` materializes built-in channel/plugin entries by writing `enabled: true` into `channels.<id>` or `plugins.entries.<id>`, so omitted keys are still synthesized into the config object.
  - file: [src/config/plugin-auto-enable.shared.ts:611](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/config/plugin-auto-enable.shared.ts#L611)
