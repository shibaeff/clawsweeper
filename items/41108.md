## Summary

Keep open. Current `main` still lacks the PR's MS Teams thread-reply implicit-mention fix: mention detection only checks cached `replyToId`, and outbound caching still records only bot reply IDs, not the thread-root ID carried in `conversation.id;messageid=`.

## Evidence

- **Implicit mention check is still replyToId-only on main:** `implicitMentionKinds` is populated only when `replyToId` exists and `wasMSTeamsMessageSent(conversationId, replyToId)` is true. There is no fallback to `extractMSTeamsConversationMessageId(activity.conversation?.id)` here.
  - file: [extensions/msteams/src/monitor-handler/message-handler.ts:958](https://github.com/openclaw/openclaw/blob/33d5ebbff7bceb782b314de14ebd9a213ba31869/extensions/msteams/src/monitor-handler/message-handler.ts#L958)
  - command: `nl -ba extensions/msteams/src/monitor-handler/message-handler.ts | sed -n '950,970p'`
  - sha: [33d5ebbff7bc](https://github.com/openclaw/openclaw/commit/33d5ebbff7bceb782b314de14ebd9a213ba31869)
- **Main does not cache the thread-root ID on send:** The outbound `onSentMessageIds` hook records only returned bot message IDs. It does not also record the parsed `conversationMessageId`, which the PR describes as necessary because Teams encodes the thread root in `conversation.id;messageid=`.
  - file: [extensions/msteams/src/monitor-handler/message-handler.ts:821](https://github.com/openclaw/openclaw/blob/33d5ebbff7bceb782b314de14ebd9a213ba31869/extensions/msteams/src/monitor-handler/message-handler.ts#L821)
