## Summary

Current `main` already contains a shipped fix for plugin webhook routes being lost across plugin-registry churn, including the exact startup-pin/live-route behavior this report asks for.

## Evidence

## Close Comment

Closing from a Codex review because this is already implemented on current `main` and was shipped in `2026.3.22`. The gateway now pins plugin webhook routes to the startup registry and resolves both plugin-route auth and dispatch from that same live HTTP-route registry (`src/gateway/server-runtime-state.ts:108`, `src/gateway/server-runtime-state.ts:152`, `src/plugins/runtime.ts:109`, `src/plugins/http-registry.ts:26`, `src/gateway/server/plugins-http.ts:69`). There is direct regression coverage for the exact “active registry changed later, but the webhook route still works” case in `src/plugins/http-registry.test.ts:208` and `src/gateway/server/plugins-http.test.ts:300`. The shipped changelog entry is `CHANGELOG.md:2281` under `2026.3.22`. If you still hit 404s on a current build, please open a new issue with the exact version, current Google Chat config, and fresh gateway logs.
