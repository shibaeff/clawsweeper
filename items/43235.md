## Summary

Keep open. Current `main` still routes inbound embedded agent runs through a shared global lane by default, and the documented/configured concurrency control remains a single `agents.defaults.maxConcurrent` setting rather than per-agent lane isolation.

## Evidence

- **Queue docs still describe a shared global main lane:** The command-queue docs say each session run is queued into a global lane (`main` by default) and capped by `agents.defaults.maxConcurrent`.
  - file: [docs/concepts/queue.md:21](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/docs/concepts/queue.md#L21)
  - command: `nl -ba docs/concepts/queue.md | sed -n '15,35p'`

- **Global lane resolver still defaults to `main`:** `resolveGlobalLane()` only special-cases `cron` to `nested`; otherwise an omitted/blank lane becomes `CommandLane.Main`.
  - file: [src/agents/pi-embedded-runner/lanes.ts:8](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/agents/pi-embedded-runner/lanes.ts#L8)

- **Gateway concurrency is still configured only for shared named lanes:** Gateway startup applies concurrency caps for `cron`, `main`, and `subagent`; `main` is sourced from `resolveAgentMaxConcurrent(cfg)` with no per-agent lane map.

  - file: [src/config/agent-limits.ts:9](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/config/agent-limits.ts#L9)

## Risks / Open Questions

- There may be operator workarounds that set a custom `lane` on specific RPC/plugin-triggered runs, but that does not satisfy the issue's request for first-class per-agent lane isolation for normal inbound multi-agent traffic.
