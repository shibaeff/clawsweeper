## Summary

Issue #43329 is already implemented on current main and in release v2026.4.22: Twilio outbound `conversation` calls with streaming enabled now defer the initial message until stream connect, preserve `initialMessage` until playback succeeds, and include regression coverage for the exact scenario described.

## Evidence

- **Current main defers Twilio conversation greeting until stream connect:** `CallManager.maybeSpeakInitialMessageOnAnswered()` now returns early for Twilio + streaming + stream-connect-capable providers, instead of eagerly speaking on `answered`.
  - file: [extensions/voice-call/src/manager.ts:298](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/voice-call/src/manager.ts#L298)

## Close Comment

Codex review: this looks already fixed on current `main`, and the fix is also present in `v2026.4.22`. The current manager defers outbound Twilio `conversation` greetings until stream connect when streaming is enabled (`extensions/voice-call/src/manager.ts:298`), and `speakInitialMessage()` now keeps `metadata.initialMessage` until playback succeeds instead of deleting it first (`extensions/voice-call/src/manager/outbound.ts:303`). There is also direct regression coverage for the Twilio streaming case in `extensions/voice-call/src/manager.notify.test.ts:180`, and the shipped docs describe the same startup behavior in `docs/plugins/voice-call.md:451`. Closing as implemented on main from this Codex review.
