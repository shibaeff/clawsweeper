## Summary

Current `main` still appears to have the reported gap. Ollama implicit discovery was refactored into the bundled Ollama plugin, but that path still only treats `models.providers.ollama` as explicit config; a non-default remote provider using `api: "ollama"` would not suppress the ambient local probe.

## Evidence

- **Current discovery logic only checks the default `ollama` provider key:** `resolveOllamaDiscoveryResult()` reads `config.models?.providers?.ollama` into `explicit` and derives `hasMeaningfulExplicitConfig` only from that entry before deciding whether to call `buildProvider(...)`. There is no scan for other explicit providers whose `api` is `"ollama"`.
  - file: [extensions/ollama/src/discovery-shared.ts:102](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/extensions/ollama/src/discovery-shared.ts#L102)
  - command: `nl -ba extensions/ollama/src/discovery-shared.ts | sed -n '94,157p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Ambient discovery still probes the default local Ollama endpoint:** `buildOllamaProvider()` resolves the API base from the configured base URL or the default and warns when unreachable unless `quiet` is set. Without an explicit `models.providers.ollama` entry, the current discovery path still falls back to the local default probe.
