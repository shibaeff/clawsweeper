## Summary

Keep open. Current `main` does not already implement this PR's behavior: ambient Ollama discovery still keys only off `models.providers.ollama`, so a separate explicit provider using `api: "ollama"` would not suppress the implicit local probe or its warning path.

## Evidence

- **Current discovery logic still ignores alternate `api: "ollama"` providers:** `extensions/ollama/src/discovery-shared.ts:102-143` reads only `config.models?.providers?.ollama`, derives `hasMeaningfulExplicitConfig` from that one entry, then calls `buildProvider(explicit?.baseUrl, ...)`. There is no scan of other explicit providers, no remote-provider short-circuit, and no 127/8 loopback helper.
  - file: [extensions/ollama/src/discovery-shared.ts:102](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/ollama/src/discovery-shared.ts#L102)

  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **The warning path still exists on failed implicit discovery:** `extensions/ollama/src/provider-models.ts:260-268` still probes the resolved Ollama base URL and emits `console.warn("Ollama could not be reached ...")` when the fetch fails and `quiet` is false. In the reported remote-provider scenario, `quiet` would still be false because a real `OLLAMA_API_KEY` is present while no explicit `models.providers.ollama` entry exists.
  - file: [extensions/ollama/src/provider-models.ts:260](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/ollama/src/provider-models.ts#L260)
