## Summary

Current `main` already implements the Telegram single-preview finalization path and guards against duplicate final resend behavior. The code finalizes streamed replies by editing/materializing the existing preview, avoids duplicate fallback sends on ambiguous final-edit errors, and has regression tests plus changelog entries covering the exact symptom.

## Evidence

- **Final reply edits/retains preview instead of duplicating send:** `extensions/telegram/src/lane-delivery-text-deliverer.ts` treats preview-final `message is not modified` as delivered, retains the existing preview on ambiguous/network-after-send cases to avoid duplicate fallback sends, and only falls back to `sendPayload(...)` when the preview truly cannot be finalized safely.
  - file: [extensions/telegram/src/lane-delivery-text-deliverer.ts:250](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/telegram/src/lane-delivery-text-deliverer.ts#L250)

  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Finalization prefers preview/materialize before normal send:** For final text replies, the deliverer first tries preview finalization/materialization and returns `preview-finalized`/`preview-retained`; only after that does it stop the lane and call the normal send path. ## Close Comment
