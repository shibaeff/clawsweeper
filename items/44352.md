## Summary

Keep open. Current `main` still depends on the runtime SQLite build supporting FTS5, still reports `fts unavailable` when `CREATE VIRTUAL TABLE ... USING fts5` fails, and I found no code or release evidence that the Debian-specific failure was fixed on `main` or shipped in `v2026.4.22`.

## Evidence

- **Current schema path still requires runtime FTS5 support:** `ensureMemoryIndexSchema()` still executes `CREATE VIRTUAL TABLE ... USING fts5(...)` and captures the thrown error as `ftsError` if that statement fails, so the reported `no such module: fts5` path still exists on current `main`.
  - file: [src/memory-host-sdk/host/memory-schema.ts:61](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/memory-host-sdk/host/memory-schema.ts#L61)
  - command: `nl -ba src/memory-host-sdk/host/memory-schema.ts | sed -n '1,220p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Current manager still warns when hybrid search is enabled and FTS init fails:** `MemoryIndexManager.ensureSchema()` records `result.ftsError` and logs `fts unavailable: ...` whenever `this.fts.enabled` is true. The only suppression is when hybrid search is disabled, which does not resolve the reporter's Debian/FTS5 runtime failure when hybrid search is on.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:348](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/memory-core/src/memory/manager-sync-ops.ts#L348)

- **Current search path degrades around missing FTS, but does not fix it:** When an embedding provider exists but FTS is unavailable, `manager.search()` falls back to vector-only results. That is graceful degradation, not evidence that the missing-FTS5 runtime issue was fixed.
  - file: [extensions/memory-core/src/memory/manager.ts:391](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/memory-core/src/memory/manager.ts#L391)
