## Summary

Keep open. Current `main` does not implement `toolsOverride` or any equivalent context-engine return value for per-turn tool-schema filtering. Main now has adjacent seams (`availableTools` into `assemble()`, plus run-level `toolsAllow`), but those do not satisfy this PR's requested behavior.

## Evidence

- **No `toolsOverride` implementation on main:** A repo search on current `main` found no `toolsOverride` field and no `wrapStreamFnWithToolsOverride` helper anywhere under `src` or `docs`. That means the PR's feature has not already landed under the same names.
  - command: `rg -n "toolsOverride|wrapStreamFnWithToolsOverride" src docs`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **`AssembleResult` still lacks a tool-override return field:** On current `main`, `AssembleResult` only exposes `messages`, `estimatedTokens`, and optional `systemPromptAddition`; there is no field for returning a filtered tool list from the context engine.
  - file: [src/context-engine/types.ts:6](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/context-engine/types.ts#L6)
  - command: `nl -ba src/context-engine/types.ts | sed -n '1,20p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Runner only consumes assembled messages and system-prompt additions:** The current assemble call path passes `availableTools` into `assemble()`, then applies only `assembled.messages` and `assembled.systemPromptAddition`. There is no stream wrapper or other hook here that filters the tool schemas after assembly.

## Risks / Open Questions

- The PR is not obsolete, but it is partially overlapped by newer `availableTools` and `toolsAllow` seams, so it likely needs rebasing and API rethinking rather than blind merge.
- The proposed PR behavior still needs maintainer review around contract semantics: an empty override, prompt-cache effects, and whether hiding schemas while leaving executors callable is acceptable.
