## Summary

Close. Current OpenClaw core already contains a generic transcript-persistence fix for pre-created session files dropping the first inbound user message, which matches the reported Gateway-received-but-not-session-persisted symptom.

## Close Comment

Closing from a Codex review because current OpenClaw already contains the core fix for this failure mode. `src/agents/pi-embedded-runner/session-manager-init.ts:7-14` documents the SessionManager persistence quirk where a pre-created session file with no assistant message can drop the initial user turn, and `src/agents/pi-embedded-runner/session-manager-init.ts:44-51` resets that state so the first user+assistant turn is persisted correctly. The runner applies that guard on every embedded turn in `src/agents/pi-embedded-runner/run/attempt.ts:988-1047` before the model run starts. I also verified the same code is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). DingTalk itself is a community plugin (`docs/plugins/community.md:52-61`), so this report no longer matches current OpenClaw core behavior. If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with current logs plus the exact DingTalk plugin version and session key involved.
