## Summary

Current main already treats this as a bounded context-usage notice, not a fullscreen overlay, and it suppresses the notice when token totals are stale. The same implementation is present in the shipped release `v2026.4.22`, so the reported behavior is already fixed well enough to close conservatively.

## Evidence

- **UI suppresses stale token-based notices:** `getContextNoticeViewModel()` returns `null` when `totalTokensFresh === false`, and computes the warning from `session.totalTokens` vs `contextTokens` rather than cumulative `inputTokens`. That directly matches the report's later diagnosis that the UI was showing a bogus 100% context warning.

## Close Comment

Closing as `implemented_on_main` after a Codex review. The current Web UI already renders this as a bounded `context-notice` pill with a fixed 16x16 SVG icon (`ui/src/ui/chat/context-notice.ts:99`, `ui/src/styles/chat/layout.css:168`), so the triangle is no longer able to expand across the chat surface. The token mismatch described in the issue comments is also handled now: the UI suppresses the warning when `totalTokensFresh === false` and uses fresh `totalTokens` rather than cumulative `inputTokens` (`ui/src/ui/chat/context-notice.ts:53`, `src/auto-reply/reply/session-usage.ts:157`, `src/gateway/session-utils.ts:1330`). There is a regression test covering the exact case where large accumulated `inputTokens` should not show the warning and confirming the SVG stays 16x16 (`ui/src/ui/chat/context-notice.test.ts:24`). I also checked the shipped release `v2026.4.22` at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, and the same implementation is present there.
