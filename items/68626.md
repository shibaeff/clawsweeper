## Summary

Keep open. Current `main` does not implement the proposed plugin-owned approval-resolution seam: `requireApproval` still exposes only post-decision `onResolution`, and the tool-call path resolves/waits in core before invoking that callback. This looks like a live, in-scope SDK/RFC request rather than something already implemented, obsolete, incoherent, or better closed as ClawHub.

## Evidence

- **`requireApproval` has no `onRegistered`/resolver handle:** The current hook contract exposes `title`, `description`, timeout/severity fields, `pluginId`, and `onResolution` only. There is no callback that receives an approval ID or bound resolver.
  - file: [src/plugins/hook-types.ts:346](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/plugins/hook-types.ts#L346)
  - command: `nl -ba src/plugins/hook-types.ts | sed -n '331,365p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Core waits for the approval decision, then notifies the plugin afterward:** `runBeforeToolCallHook` calls `plugin.approval.request`, then `plugin.approval.waitDecision`, derives the final resolution, and only then calls `safeOnResolution(...)`. That matches the issue's claim that `onResolution` is observational rather than a plugin-controlled resolve path.
