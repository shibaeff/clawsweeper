## Summary

Keep open. Current `main` still treats multi-agent work as isolated sessions, and the only shipped group fan-out path explicitly says shared context is not supported yet. The request is coherent, recent, and describes a real product gap rather than something already implemented, unreproducible, or clearly ClawHub-only.

## Evidence

- **Multi-agent routing is explicitly isolated:** The current docs define multi-agent routing as multiple isolated agents, each with its own workspace, state directory, and session history.
  - file: [docs/concepts/multi-agent.md:8](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/concepts/multi-agent.md#L8)

  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Subagents are isolated by default:** `sessions_spawn` is documented as creating an isolated child session by default; `context: "fork"` only copies the requester transcript into the child, not a live shared session among agents.
  - file: [docs/concepts/session-tool.md:101](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/concepts/session-tool.md#L101)

- **Shipped group broadcast still uses separate agent sessions:** The WhatsApp broadcast implementation builds a distinct session key per `agentId` and processes each target agent separately, which is fan-out across isolated sessions rather than one shared conversation pool.
  - file: [extensions/whatsapp/src/auto-reply/monitor/broadcast.ts:15](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/whatsapp/src/auto-reply/monitor/broadcast.ts#L15)

  - file: [docs/channels/broadcast-groups.md:420](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/channels/broadcast-groups.md#L420)

## Risks / Open Questions

- Closing as `implemented_on_main` would be incorrect because current docs and code still describe isolated sessions and explicitly mark shared context as future work.
- Closing as `clawhub` is not well supported by the repo's own docs: the existing group-broadcast surface and its future-enhancement notes place this gap inside OpenClaw's channel/session model, not clearly outside core scope.
