## Summary

Keep open. Current `main` still builds fresh `claude-cli` runs with the system prompt on argv via `--append-system-prompt`, and that system prompt still includes injected workspace bootstrap files. That matches the Windows argv-length failure described in #69158, and there is no touched-surface diff from `v2026.4.23` showing a shipped fix.

## Evidence

- **Anthropic CLI backend still uses argv system prompts:** The bundled `claude-cli` backend is configured with `input: "stdin"` for the user prompt, but it still sets `systemPromptArg: "--append-system-prompt"` and enables the Claude live-session path.
  - file: [extensions/anthropic/cli-backend.ts:57](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/anthropic/cli-backend.ts#L57)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Fresh Claude live sessions still append system prompt to argv:** `shouldUseClaudeLiveSession()` routes `claude-cli` through the live-session path, and `buildClaudeLiveArgs()` appends the system prompt flag/value on fresh sessions before spawning `[backend.command, ...args]`. There is no Windows guard or stdin/file fallback for the system prompt here.
  - file: [src/agents/cli-runner/claude-live-session.ts:75](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/cli-runner/claude-live-session.ts#L75)

The spawn test explicitly asserts that workspace files like `AGENTS.md`, `SOUL.md`, `IDENTITY.md`, and `USER.md` are embedded in the Claude CLI system prompt.
