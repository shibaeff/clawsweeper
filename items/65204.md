## Summary

Keep open. Current `main` still appears to reproduce the reported config-shape failure in code: when a managed profile sets both `cdpPort` and a bare `cdpUrl` like `http://127.0.0.1`, `resolveProfile()` prefers the URL and overwrites the port with `80`, which matches the issue’s observed `Failed to start Chrome CDP on port 80`. The docs now warn that local managed profiles should not set `cdpUrl`, but that guidance is not enforced or repaired in code.

## Evidence

- **`resolveProfile()` still overwrites `cdpPort` from a bare profile `cdpUrl`:** For non-`existing-session` profiles, `resolveProfile()` starts with `profile.cdpPort`, but if `profile.cdpUrl` is present it parses that URL and replaces `cdpPort` with the parsed port. For `http://127.0.0.1` that becomes `80`, matching the report.

- **Launch failure message uses the resolved `profile.cdpPort`:** The Chrome startup error string is built from `profile.cdpPort`, so once the config above resolves to `80`, current `main` would still emit the same `Failed to start Chrome CDP on port 80` symptom.
