## Summary

The requested metadata-only TTS flow is useful, but current main intentionally keeps the built-in `tts` tool as an auto-delivery primitive. OpenClaw already exposes lower-level TTS runtime seams for plugin/skill code, and the repo vision pushes channel-specific orchestration like native voice-note delivery out of core and into plugins/ClawHub.

## Close Comment

Closing from a Codex review. This is a useful capability, but current `main` keeps the built-in `tts` tool as an auto-delivery primitive rather than a raw synthesis API: `src/agents/tools/tts-tool.ts:65` explicitly says audio is delivered automatically, and successful tool media is then queued for outbound send in `src/agents/pi-embedded-subscribe.handlers.tools.ts:536`. The motivating use case here is channel-specific post-processing/native voice-note delivery, and the repo scope points that kind of orchestration to plugins/skills instead of expanding core tool behavior (`VISION.md:54`, `VISION.md:68`, `docs/pi.md:130`). There is already a lower-level plugin seam for this: `api.runtime.tts.textToSpeech(...)` plus `TtsResult.audioPath` / `outputFormat` / `voiceCompatible` (`src/plugin-sdk/tts-runtime.types.ts:112`). On that basis, this is better handled as a plugin or ClawHub skill/tool rather than a new mode on the core `tts` tool.
