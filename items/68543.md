## Summary

Keep open. Current `main` still applies symmetric jitter to `retryAfterMs` delays in `retryAsync`, and the existing tests only cover `retryAfterMs` with `jitter: 0`, so the reported lower-bound bug is still plausible and not implemented on `main`.

## Evidence

- **Current retry logic still uses symmetric jitter unconditionally:** `applyJitter` still has the old two-argument signature and always computes a symmetric offset in `[-jitter, +jitter]` via `(generateSecureFraction() * 2 - 1) * jitter`. There is no positive-only mode or special handling for server-supplied lower bounds.
  - file: [src/infra/retry.ts:61](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/infra/retry.ts#L61)
  - command: `nl -ba src/infra/retry.ts | sed -n '1,240p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **`retryAsync` still jitters `retryAfterMs` below the server hint:** When `retryAfterMs` is present, `baseDelay` is derived from `Math.max(retryAfterMs, minDelayMs)`, then clamped to `maxDelayMs`, then passed through symmetric `applyJitter`, and finally only clamped back to `[minDelayMs, maxDelayMs]`. Nothing restores a `retryAfterMs` floor after jitter.
  - file: [src/infra/retry.ts:114](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/infra/retry.ts#L114)
