## Summary

The report is about `clawhub install` behavior and China-specific mirror/proxy handling for the ClawHub registry flow, not an OpenClaw core bug. Current `main` documents native OpenClaw skill installs via `openclaw skills install`, and the in-repo install path talks to ClawHub archive/search endpoints directly rather than cloning from GitHub.

## Close Comment

Closing from a Codex review because this request targets the separate ClawHub product/CLI rather than OpenClaw core. Current `main` documents native skill installs as `openclaw skills install <slug>` (`docs/cli/skills.md:12`) and the in-repo install path fetches skill metadata and archives from ClawHub API endpoints (`src/infra/clawhub.ts:516`, `src/infra/clawhub.ts:538`, `src/infra/clawhub.ts:616`) instead of cloning from GitHub. The docs also describe `clawhub` as a separate CLI (`docs/tools/clawhub.md:10`) and route ClawHub issues to the separate `openclaw/clawhub` repo (`CONTRIBUTING.md:213`). A mirror/proxy change for `npx clawhub install find-skills` should be tracked there rather than in `openclaw/openclaw`. If this is still needed, please open it in `openclaw/clawhub` with the exact failing command, region/network details, and any 429 or clone error output.
