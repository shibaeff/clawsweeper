## Summary

Keep open. The duplicate-warning problem is real, but this PR's approach is not safe on current main: non-bundled manifest channel owners still require explicit trust/enablement, so skipping `plugins.entries` would regress workspace plugins and does not fully solve restrictive-allowlist cases for global plugins.

## Evidence

- **Current auto-enable path still relies on explicit plugin entries:** `materializePluginAutoEnableCandidatesInternal` only skips when a plugin entry is already enabled; otherwise it calls `registerPluginEntry()` and `ensurePluginAllowlisted()`. That means the current design treats `plugins.entries.<id>.enabled` as the trust/activation mechanism for non-bundled auto-enable candidates.
  - file: [src/config/plugin-auto-enable.shared.ts:759](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/config/plugin-auto-enable.shared.ts#L759)
  - command: `nl -ba src/config/plugin-auto-enable.shared.ts | sed -n '752,792p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Workspace plugins are disabled by default unless explicitly trusted/enabled:** Effective activation returns `workspace-disabled-by-default` when origin is `workspace`, the plugin is not allowlisted, and `plugins.entries.<id>.enabled !== true`. Skipping the entry for workspace channel plugins would leave them disabled.
  - file: [src/plugins/config-activation-shared.ts:188](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/plugins/config-activation-shared.ts#L188)
