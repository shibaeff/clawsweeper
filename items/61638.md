## Summary

Keep open. Current `main` still lacks the overflow guard, and this PR as proposed introduces a real validation regression by returning early after an invalid `heartbeat.every`, which skips `activeHours` validation in the same parse pass.

## Evidence

- **Current main still accepts oversized heartbeat durations:** `HeartbeatSchema.superRefine()` only checks that `heartbeat.every` parses; it does not cap the parsed millisecond value before the config is accepted.
  - file: [src/config/zod-schema.agent-runtime.ts:46](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/config/zod-schema.agent-runtime.ts#L46)

  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Resolved heartbeat interval is passed through without an upper bound:** `resolveHeartbeatIntervalMs()` returns any positive parsed duration and does not reject values above Node's timer ceiling.
  - file: [src/infra/heartbeat-summary.ts:44](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/infra/heartbeat-summary.ts#L44)

  - file: [src/infra/heartbeat-runner.ts:1390](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/infra/heartbeat-runner.ts#L1390)

## Risks / Open Questions

- Closing this PR would discard a real bug fix while `main` still lacks any guard against oversized heartbeat delays.
- Merging the PR as-is would regress multi-field config validation by hiding `activeHours` errors when `heartbeat.every` is invalid.
- The PR's current test coverage misses the exact combined-error case called out in review, so the regression could slip through unchanged.
