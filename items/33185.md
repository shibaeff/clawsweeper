## Summary

Keep open. Current main still has the image-tool/media-understanding lookup path that can throw `Unknown model` for provider-prefixed custom image models, and I did not find a shipped fix or regression test that closes out the reported behavior with high confidence.

## Evidence

- **Image runtime still uses bare registry lookup:** `resolveImageRuntime` normalizes the provider/model pair, then does only `modelRegistry.find(resolvedRef.provider, resolvedRef.model)` before throwing `Unknown model`. There is no fallback lookup for provider-prefixed custom model ids on this path.
  - file: [src/media-understanding/image.ts:107](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/media-understanding/image.ts#L107)
  - command: `nl -ba src/media-understanding/image.ts | sed -n '98,118p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Configured imageModel refs are split into provider + model before execution:** `runWithImageModelFallback` resolves `agents.defaults.imageModel.primary/fallbacks` through `resolveModelRefFromString`, so a configured ref like `kimchi/vision-1` is executed as provider `kimchi`, model `vision-1`. That makes the bare lookup above miss when the registry stores the model id with the provider prefix.
  - file: [src/agents/model-fallback.ts:342](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/model-fallback.ts#L342)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '342,375p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **A fallback helper exists elsewhere, but not in image runtime:** `resolveModelFromRegistry` in shared media-tool code explicitly retries with `${provider}/${model}` when the first lookup misses, and a dedicated test covers provider-prefixed custom ids. The image runtime does not use this helper.
  - file: [src/agents/tools/media-tool-shared.ts:418](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/tools/media-tool-shared.ts#L418)
  - command: `nl -ba src/agents/tools/media-tool-shared.ts | sed -n '418,438p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Image tool config already preserves provider-prefixed custom ids:** The image-tool tests explicitly cover a custom provider whose image-capable model id is `kimchi/vision-1` and assert that config resolution keeps that prefixed id. I did not find a matching runtime regression test in `src/media-understanding/image.test.ts` for the actual lookup path.
  - file: [src/agents/tools/image-tool.test.ts:711](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/tools/image-tool.test.ts#L711)
