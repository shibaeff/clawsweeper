## Summary

Current `main` still lacks the runtime-bootstrap behavior this PR is proposing for memory public-artifact reads, and the bridge/status call sites still depend on that path. I do not have strong evidence that the report is obsolete or already implemented, so this PR should stay open for maintainer judgment.

## Evidence

- **Artifact listing still returns only registered capability state:** On current `main`, `listActiveMemoryPublicArtifacts` reads `memoryPluginState.capability?.capability.publicArtifacts?.listArtifacts(params)` directly and sorts the result. It does not use `cfg` to resolve/load the runtime registry first, so a fresh process with no registered memory capability will still return `[]`.
  - file: [src/plugins/memory-state.ts:274](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/plugins/memory-state.ts#L274)
  - command: `nl -ba src/plugins/memory-state.ts | sed -n '240,320p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Bridge/status surfaces still call the unfixed helper:** `memory-wiki` status and bridge import both call `listActiveMemoryPublicArtifacts({ cfg: ... })` on current `main`, so they still depend on the non-self-healing path above.
  - file: [extensions/memory-wiki/src/status.ts:223](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/extensions/memory-wiki/src/status.ts#L223)
