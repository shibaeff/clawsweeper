## Summary

Keep open. Current `main` still has the concurrent-start race in `VoiceCallWebhookServer.start()`, so this PR is not implemented on main; at the same time, the proposed patch appears to need follow-up hardening before a conservative maintainer close would be justified.

## Evidence

- **Current main only guards already-listening servers:** `start()` returns early only when `this.server?.listening` is true. Two overlapping callers before the first `listen()` callback both fall through into new server creation, so the reported race still exists on current `main`.
  - file: [extensions/voice-call/src/webhook.ts:435](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/voice-call/src/webhook.ts#L435)
  - command: `nl -ba extensions/voice-call/src/webhook.ts | sed -n '431,525p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **No concurrent-start regression test on main:** The current tests cover sequential `start()` reuse, restart after `stop()`, and `stop()` before start, but there is no `Promise.all([start(), start()])` coverage on `main`.
  - file: [extensions/voice-call/src/webhook.test.ts:921](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/voice-call/src/webhook.test.ts#L921)
  - command: `nl -ba extensions/voice-call/src/webhook.test.ts | sed -n '921,970p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **PR hardening concern remains plausible:** The serve config schema accepts any positive integer port, so a cached `startPromise` implementation would need to handle synchronous `server.listen()` throws such as an out-of-range port rather than only async `error` events.
  - file: [extensions/voice-call/src/config.ts:77](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/voice-call/src/config.ts#L77)
  - command: `nl -ba extensions/voice-call/src/config.ts | sed -n '74,90p'`
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Repository has a safer existing start/stop pattern elsewhere:** `exec-approval-channel-runtime` already serializes concurrent `start()` calls with `startPromise`, clears it in `finally`, and waits for in-flight startup inside `stop()`. That is stronger than the PR summary's reset-on-success/error/stop approach and suggests the voice-call fix likely still needs refinement before closure.
  - file: [src/infra/exec-approval-channel-runtime.ts:268](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/infra/exec-approval-channel-runtime.ts#L268)

## Risks / Open Questions

- Closing as implemented_on_main would be incorrect: current `main` still lacks the shared in-flight start guard and concurrent regression test.
- Closing as cannot_reproduce would be weak: the race follows directly from the current `this.server?.listening` guard and absence of any pre-listen serialization.
- If the PR lands as summarized in the GitHub context, it likely still needs review for synchronous `listen()` throws and `stop()` interaction with an in-flight start.
