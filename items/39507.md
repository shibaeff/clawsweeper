## Summary

Keep open. Current `main` and the latest shipped release still force Feishu thread replies whenever inbound `root_id`/`thread_id` is present, even when `replyInThread` is configured as `"disabled"`. That conflicts with the documented config semantics. The PR stays relevant, and the review-raised topic-scope regression was already addressed in the PR's follow-up commit.

## Evidence

- **Config docs say disabled means inline replies:** The Feishu config schema documents `replyInThread: "disabled"` as normal inline replies and `"enabled"` as creating/continuing a topic thread.
  - file: [extensions/feishu/src/config-schema.ts:132](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/extensions/feishu/src/config-schema.ts#L132)
  - command: `nl -ba extensions/feishu/src/config-schema.ts | sed -n '1,240p'`

- **Current main overrides disabled when thread metadata exists:** `resolveFeishuGroupSession` computes `threadReply` from inbound `threadId`/`rootId`, then ORs that into `replyInThread`, so an inbound quote/thread reply turns `replyInThread` on even if config is `"disabled"`.
  - file: [extensions/feishu/src/bot-content.ts:68](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/extensions/feishu/src/bot-content.ts#L68)
