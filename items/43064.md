## Summary

Keep open. Current `main` still has the reported gap: heartbeat replies use the raw configured `responsePrefix` string and do not wire the normal reply-prefix template context/model-selection flow before prepending it.

## Evidence

- **Heartbeat path reads the raw configured prefix:** `runHeartbeatOnce()` resolves `responsePrefix` directly from `resolveEffectiveMessagesConfig(...)` and stores the template string as-is.
  - file: [src/infra/heartbeat-runner.ts:836](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/infra/heartbeat-runner.ts#L836)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '832,1118p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Heartbeat reply call omits model-selection/template-context hooks:** The heartbeat `replyOpts` only contain heartbeat/runtime flags, then `getReplyFromConfig(ctx, replyOpts, cfg)` is called without `onModelSelected`, `responsePrefix`, or any response-prefix context provider.
  - file: [src/infra/heartbeat-runner.ts:1070](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/infra/heartbeat-runner.ts#L1070)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '832,1118p'`

- **Heartbeat normalization prepends the raw prefix literally:** `normalizeHeartbeatReply()` checks `responsePrefix` and prepends ``${responsePrefix} ${finalText}`` directly, so placeholders like `{model}` would remain literal in heartbeat alerts.
  - file: [src/infra/heartbeat-runner.ts:496](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/infra/heartbeat-runner.ts#L496)

That machinery is not used by the heartbeat runner.
