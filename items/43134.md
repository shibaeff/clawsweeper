## Summary

Close. Current `main` already handles the Windows scheduled-task restart no-op/queued case by detecting a `schtasks /Run` restart that never actually launches and falling back to directly spawning the gateway task script.

## Evidence

- **CLI `gateway restart` uses the Windows scheduled-task restart path:** The Windows service registry wires `openclaw gateway restart` to `restartScheduledTask` on `win32`.
  - file: [src/daemon/service.ts:199](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/daemon/service.ts#L199)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Restart path retries through the scheduled-task helper and completes:** `restartScheduledTask` still ends the task, waits for the port to clear, then calls `runScheduledTaskOrThrow(...)` and returns `{ outcome: "completed" }`.
  - file: [src/daemon/schtasks.ts:986](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/daemon/schtasks.ts#L986)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **No-op/queued launches now fall back to a direct spawn:** `shouldFallbackScheduledTaskLaunch` watches for the task staying in the `not-yet-run` state (`0x41303`) with no launch evidence, and `runScheduledTaskOrThrow` then calls `launchFallbackTaskScript`, which directly spawns the gateway task script detached on Windows.
  - file: [src/daemon/schtasks.ts:664](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/daemon/schtasks.ts#L664)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Direct fallback launcher exists on current main:** `launchFallbackTaskScript` reads the task command and spawns it directly, bypassing the stuck scheduled-task queue state.
  - file: [src/daemon/schtasks.ts:316](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/daemon/schtasks.ts#L316)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Regression test covers the queued/no-op restart case:** There is an explicit test named `relaunches the task script when restart sees a scheduled-task run no-op`, which exercises `restartScheduledTask(...)` and asserts the fallback spawn path is used.
  - file: [src/daemon/schtasks.startup-fallback.test.ts:418](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/daemon/schtasks.startup-fallback.test.ts#L418)

## Close Comment

Closing from a Codex review because current `main` already implements the requested fix. On Windows, `openclaw gateway restart` goes through `restartScheduledTask` (`src/daemon/service.ts:199`), and that restart path now detects the scheduled-task `/Run` no-op case and falls back to directly spawning the gateway task script when the task stays in the `not-yet-run` state with no launch evidence (`src/daemon/schtasks.ts:316`, `src/daemon/schtasks.ts:664`, `src/daemon/schtasks.ts:986`). There is also a regression test for this exact scenario: `relaunches the task script when restart sees a scheduled-task run no-op` (`src/daemon/schtasks.startup-fallback.test.ts:418`). I could not tie the fix to a shipped release tag from the local checkout, so `fixedRelease` is left unknown.
