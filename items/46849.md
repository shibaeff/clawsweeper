## Summary

Current `main` already replaced `openclaw gateway status` with a dedicated read-only daemon-status path that bypasses plugin preload and does not enter the gateway run/respawn lifecycle. The implementation shipped in `v2026.4.22` at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`.

## Close Comment

Thanks for the report. Codex review on current `main` found that this path has already been replaced by a dedicated read-only implementation. `openclaw gateway status` now routes directly to `runDaemonStatus` (`src/cli/program/routed-command-definitions.ts:79`), and the new route is explicitly covered as a no-plugin-preload path (`src/cli/program/routes.test.ts:103`). The status handler itself only gathers service/runtime/config/port state and optionally performs RPC probes (`src/cli/daemon-cli/status.ts:7`, `src/cli/daemon-cli/status.gather.ts:339`, `src/cli/daemon-cli/probe.ts:26`); it no longer enters the gateway run/respawn lifecycle that could create a nested `openclaw-gateway` child. This implementation shipped in `v2026.4.22` via commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, so I’m closing this as already implemented on main. If you can still reproduce the same hang on `v2026.4.22` or newer, please open a new issue with the exact version, command, and fresh logs.
