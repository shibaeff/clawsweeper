## Summary

Current main already implements a configurable, hot-reloadable stale-socket threshold, but under `gateway.channelStaleEventThresholdMinutes` rather than the issue's proposed `gateway.staleEventThresholdMinutes`. The changelog records it in release `v2026.3.22`, and current main also further reduces false positives by using provider-proven transport activity instead of inbound app-event freshness.

## Risks / Open Questions

- The shipped config key is `gateway.channelStaleEventThresholdMinutes`, not the exact alias proposed in the issue.
- The current schema requires the threshold to be at least 1 minute, so the issue body's proposed `0` value for disabling only stale-socket detection is not supported as written.

## Close Comment

Closing from a Codex review because this capability is already implemented on current OpenClaw. The supported config key is `gateway.channelStaleEventThresholdMinutes` (rather than the earlier proposed `gateway.staleEventThresholdMinutes`): it is defined in `src/config/zod-schema.ts:747`, wired into gateway startup in `src/gateway/server-runtime-services.ts:36`, and hot-reloadable via `src/gateway/config-reload-plan.ts:49` plus `src/gateway/server-reload-handlers.ts:149`. It is also documented in `docs/gateway/health.md:35`, and the changelog records it in the `2026.3.22` release at `CHANGELOG.md:2251`. Current main goes further and now bases stale-socket checks on provider-proven transport activity instead of inbound app-event freshness (`src/gateway/channel-health-policy.ts:115`), which directly addresses the low-traffic false-positive case described here. If you still want the old alias name or a `0` value that disables only stale-socket detection, please open a focused follow-up issue for that exact behavior.
