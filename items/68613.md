## Summary

The report is about Straico-specific provider behavior on a non-bundled integration. Current main intentionally keeps vendor-specific provider routing, auth/catalog setup, transport normalization, and custom request behavior in provider plugins, with plugin distribution/promotion routed through ClawHub rather than adding one-off core logic.

## Evidence

- **Vision scope: optional capabilities should be plugins:** The project vision says core stays lean, optional capability should usually ship as plugins, and plugin promotion belongs in ClawHub.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/VISION.md#L54)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Provider behavior is plugin-owned:** The model-provider docs state that provider-specific logic lives in provider plugins, including transport/config normalization, auth mapping, tool-schema cleanup, failover classification, and other vendor behavior.
  - file: [docs/concepts/model-providers.md:40](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/docs/concepts/model-providers.md#L40)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Published extension path for custom/OpenAI-compatible providers:** The provider-plugin guide explicitly says it is the path when you want to add an OpenAI-compatible proxy or custom LLM to OpenClaw.
  - file: [docs/plugins/sdk-provider-plugins.md:1](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/docs/plugins/sdk-provider-plugins.md#L1)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Generic proxy escape hatch is limited:** Current docs only document vendor-specific body overrides via `params.extra_body` / `params.extraBody` for OpenAI-compatible `openai-completions` proxies, not a Straico-specific selector contract on the Responses path.
  - file: [docs/concepts/model-providers.md:631](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/docs/concepts/model-providers.md#L631)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **OpenAI-compatible route docs confirm the same limit:** The OpenAI provider docs describe proxy-compatible routes as looser compat paths and only call out `params.extra_body` for OpenAI-compatible Completions proxies.
  - file: [docs/providers/openai.md:796](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/docs/providers/openai.md#L796)

## Risks / Open Questions

- This review did not validate against a live Straico account, so it cannot rule out a broader generic OpenAI-compat regression if Straico later becomes fully standard on the same endpoints.
- If maintainers want Straico promoted as an officially bundled provider, this scope judgment would need explicit product/ownership approval rather than conservative cleanup.
