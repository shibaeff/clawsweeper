## Summary

Keep open. Current main still advertises `openclaw acp client` as an interactive prompt-debug flow, the checked-in client/translator code can wait indefinitely for a terminal ACP event, and I found no post-release ACP source change on `main` that would make this report obsolete or clearly fixed.

## Evidence

- **Docs still present the feature as interactive and supported:** The current CLI docs still say the built-in ACP client 'spawns the ACP bridge and lets you type prompts interactively,' with `openclaw acp client` as the documented sanity-check path.
  - file: [docs/cli/acp.md:102](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/docs/cli/acp.md#L102)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Interactive client waits on each prompt until the bridge settles:** `runAcpClientInteractive()` sends every entered line through `client.prompt(...)` and only prints a result / reprompts after that promise resolves or rejects. If the bridge never delivers a terminal response, the UX is an indefinite hang.
  - file: [src/acp/client.ts:202](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/acp/client.ts#L202)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Bridge prompt path has no independent timeout while connected:** `AcpGatewayAgent.prompt()` stores the prompt as pending and sends `chat.send` with `{ timeoutMs: null }`; completion happens later in `handleChatEvent()` only when the gateway emits `final`, `aborted`, or `error`. There is no separate timeout for a still-connected but silent run.
  - file: [src/acp/translator.ts:722](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/acp/translator.ts#L722)
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Current ACP tests focus on disconnect/reconnect edge cases, not this interactive regression:** The checked-in ACP stop-reason tests cover reconnect grace windows and missed-final reconciliation. I did not find matching end-to-end regression coverage for `runAcpClientInteractive` in `src/acp/client.test.ts`, so the reported CLI hang is not disproven by current tests.
  - file: [src/acp/translator.stop-reason.test.ts:84](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/acp/translator.stop-reason.test.ts#L84)
  - command: `rg -n 'runAcpClientInteractive|client\.prompt|OpenClaw ACP client' src/acp/client.test.ts`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **No ACP source fix is present on current main beyond the latest release:** `git diff --name-only a9797214338ba31c52c796adbb75afb16e0684a9..7f6452897e25543c9263b67aabf140379d1c0b0c -- src/acp src/cli/acp-cli.ts` returned no files, so current `main` does not contain a source-level ACP/client change that would justify closing this as already fixed.
