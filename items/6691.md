## Summary

Current `main` already fixes the token-cost/functionality problem behind #6691 via progressive skill disclosure: the runtime injects only a compact `<available_skills>` catalog, instructs the agent to read at most one matching `SKILL.md` on demand, and documents that skill bodies load only when the skill triggers. The shipped interface is different from the issue’s proposed trigger frontmatter or `/load`/`/unload` commands, but the requested outcome is implemented well enough on current `main`. ([github.com](https://github.com/openclaw/openclaw/blob/main/docs/tools/skills.md))

## Evidence

- **System prompt uses on-demand skill reads:** `buildSkillsSection` tells the agent to scan `<available_skills>`, read exactly one applicable `SKILL.md`, and read none if no skill clearly applies. That is the current context-aware skill use path. 
  - file: `src/agents/system-prompt.ts:2502`
- **Prompt cost is now a compact skills catalog:** The skills docs say OpenClaw injects a compact XML list of available skills into the system prompt, with fixed small per-skill overhead, instead of carrying full skill bodies every turn. That directly addresses the token-cost complaint in this issue. ([github.com](https://github.com/openclaw/openclaw/blob/main/docs/tools/skills.md))
  - file: `docs/tools/skills.md:488`
- **Skill body loads only when triggered:** The bundled `skill-creator` guidance documents the three-level loading model: metadata is always in context, the `SKILL.md` body loads when the skill triggers, and bundled resources load only as needed. ([github.com](https://github.com/openclaw/openclaw/blob/main/skills/skill-creator/SKILL.md))
  - file: `skills/skill-creator/SKILL.md:330`
- **Session snapshot is about the eligible skill list:** The same docs describe session snapshots as reusing the eligible skills list, with refresh on watcher, remote-node, or allowlist changes. Combined with the compact prompt format above, the remaining snapshot behavior is not the old “full skill text every turn” model described in #6691. ([github.com](https://github.com/openclaw/openclaw/blob/main/docs/tools/skills.md))
  - file: `docs/tools/skills.md:466`

## Risks / Open Questions

- The exact API proposed in #6691 was not what shipped: current `main` solves the problem through progressive disclosure rather than trigger-frontmatter or explicit session load/unload commands. If maintainers still want those exact controls, that would be a separate narrower enhancement, not a reason to keep this issue open. ([github.com](https://github.com/openclaw/openclaw/blob/main/docs/tools/skills.md))

## Close Comment

Closing as implemented on current `main` (`db17b8cc4a31dccbb5f451bbb4c406d08c87e5b9`) after a ClawSweeper/Codex review. The current runtime no longer has the skill-token behavior described in this report: `src/agents/system-prompt.ts:2502` instructs the agent to scan `<available_skills>` and read one matching `SKILL.md` on demand, `docs/tools/skills.md:488` documents that only a compact XML skills list is injected into the prompt, and `skills/skill-creator/SKILL.md:330` documents the progressive-disclosure model explicitly (metadata always in context, `SKILL.md` body only when triggered). That appears to resolve the token-cost vs functionality tradeoff this issue was filed for, even though the shipped solution differs from the proposed trigger-frontmatter or `/load`/`/unload` API.
