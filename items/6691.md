## Summary

Keep open. Current `main` does not implement the requested context-aware dynamic skill loading/unloading. The codebase supports static agent allowlists, explicit per-session/per-run `skillFilter` inputs, and `disable-model-invocation`, but I found no trigger metadata, conversation-intent classifier, auto-unload policy, or `/load`/`/unload` session commands matching issue #6691.

## Evidence

- **Skill filtering is explicit name-based filtering only:** `filterSkillEntries` applies filtering only when a caller provides `skillFilter`, then keeps entries whose skill name is in that list. There is no prompt-text or conversation-context analysis in this path.
  - file: [src/agents/skills/workspace.ts:101](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/agents/skills/workspace.ts#L101)
  - sha: [cc295fb8c9a3](https://github.com/openclaw/openclaw/commit/cc295fb8c9a3e775549838a23fff7dbfb72081bf)
- **Snapshots store manual/static filters, not dynamic trigger state:** `buildWorkspaceSkillSnapshot` stores only a normalized `skillFilter`, and the workspace build options define `skillFilter` as 'only include skills with these names'.
  - file: [src/agents/skills/workspace.ts:719](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/agents/skills/workspace.ts#L719)
  - sha: [cc295fb8c9a3](https://github.com/openclaw/openclaw/commit/cc295fb8c9a3e775549838a23fff7dbfb72081bf)
- **Run path populates skill filtering from agent config:** The agent command path resolves `skillFilter` from `resolveAgentSkillsFilter(cfg, sessionAgentId)` before building the snapshot. That is static config-driven filtering, not context-aware auto-loading.
  - file: [src/agents/agent-command.ts:573](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/agents/agent-command.ts#L573)
  - sha: [cc295fb8c9a3](https://github.com/openclaw/openclaw/commit/cc295fb8c9a3e775549838a23fff7dbfb72081bf)
- **Session-level filtering is documented as caller-provided:** Reply/session options describe `skillFilter` as 'If provided, only load these skills for this session', and session snapshot types persist that normalized filter.
  - file: [src/auto-reply/get-reply-options.types.ts:149](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/src/auto-reply/get-reply-options.types.ts#L149)
  - sha: [cc295fb8c9a3](https://github.com/openclaw/openclaw/commit/cc295fb8c9a3e775549838a23fff7dbfb72081bf)
- **Docs describe static allowlists and hide/show frontmatter only:** The skills docs document `agents.defaults.skills` and `agents.list[].skills` as static allowlists, and the supported frontmatter keys include `user-invocable` and `disable-model-invocation` but not trigger metadata like `triggers`, `dynamicLoading`, or `auto-unload-after`.
  - file: [docs/tools/skills.md:44](https://github.com/openclaw/openclaw/blob/cc295fb8c9a3e775549838a23fff7dbfb72081bf/docs/tools/skills.md#L44)
  - sha: [cc295fb8c9a3](https://github.com/openclaw/openclaw/commit/cc295fb8c9a3e775549838a23fff7dbfb72081bf)
- **Targeted repo search found no dynamic-loading metadata/config implementation:** A targeted search for `dynamicLoading|trigger-confidence|autoLoadTriggers|auto-unload-after|disable-model-invocation|user-invocable` returned only the existing `disable-model-invocation` / `user-invocable` frontmatter surfaces, not context-aware dynamic loading fields.
