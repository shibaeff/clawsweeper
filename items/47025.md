## Summary

Current main already ships the chat-after-restart behavior through the first-class Gateway restart continuation pipeline, so this PR's exec-specific restart breadcrumb path is no longer needed.

- Current main solves this via the supported Gateway tool + restart-sentinel continuation path, not by intercepting arbitrary exec calls to `openclaw gateway restart`; if maintainers still want raw exec interception as a fallback, that would be a separate enhancement rather than an unmerged bugfix blocker.

## Close Comment

Closing from a Codex review because current main already ships the post-restart reply flow this PR was aiming for, through the supported Gateway restart continuation pipeline rather than shell-command interception. The Gateway tool now tells agents to use `continuationMessage` for restart follow-ups and writes the restart sentinel with the session routing before restart emission (`src/agents/tools/gateway-tool.ts:357`). Session-scoped restarts also default to an automatic success continuation (`src/infra/restart-sentinel.ts:97`), and startup wake consumes that sentinel to send the restart notice and resume the original session after boot (`src/gateway/server-restart-sentinel.ts:307`). Agent guidance also now says to use first-class tools instead of equivalent CLI/exec commands and notes that restarts ping the last active session automatically (`src/agents/system-prompt.ts:729`, `src/agents/system-prompt.ts:770`). This behavior is already called out in `CHANGELOG.md:296` and `CHANGELOG.md:305`, and it is present in release `v2026.4.22`.
