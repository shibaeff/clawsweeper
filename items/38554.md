## Summary

Current main already fixes this Feishu MP4 reply regression: outbound MP4/video payloads are routed as native Feishu `msg_type: "media"` instead of `"file"`, including reply and reply-in-thread paths, and the changelog says that fix shipped in v2026.3.7.

## Evidence

- **MP4/video routing now uses native media messages:** `resolveFeishuOutboundMediaKind()` maps `.mp4`/video MIME types to `{ fileType: "mp4", msgType: "media" }`, and `sendMediaFeishu()` forwards that routing into the send path.
  - file: [extensions/feishu/src/media.ts:524](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/extensions/feishu/src/media.ts#L524)

  - sha: [76a4c167f7ef](https://github.com/openclaw/openclaw/commit/76a4c167f7ef65040fd5b00d206c59b8bb28fd49)
- **Reply path preserves `msg_type: "media"`:** `sendFileFeishu()` uses the passed `msgType` for both normal sends and `client.im.message.reply(...)`, including `reply_in_thread`, so MP4 replies no longer fall back to Feishu file messages.
