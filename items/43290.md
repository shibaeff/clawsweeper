## Summary

Current `main` already implements the core of this request: browser availability checks now reuse an already-running CDP target instead of immediately launching another Chromium process, Playwright CDP connections are cached per `cdpUrl`, and attach-only / existing-session modes are documented and supported.

## Evidence

- **Availability flow reuses an existing browser before launch:** `ensureBrowserAvailable()` first probes the configured CDP endpoint, gives a pre-existing loopback browser a retry window, and only calls `launchOpenClawChrome()` after those checks fail. ## Close Comment

Closing from a Codex review because current `main` already covers this behavior. The browser availability flow in `extensions/browser/src/browser/server-context.availability.ts:222` now prefers a reachable existing CDP target and only falls back to `launchOpenClawChrome()` after the retry path fails; the regression test at `extensions/browser/src/browser/server-context.ensure-browser-available.waits-for-cdp-ready.test.ts:90` specifically locks the pre-existing loopback reuse case and verifies no new browser launch happens. On top of that, `extensions/browser/src/browser/pw-session.ts:428` caches Playwright CDP connections by normalized `cdpUrl`, so concurrent browser-tool calls reuse the same CDP connection instead of reconnecting per session. The requested attach-only / connect-only surface is also already documented in `src/config/schema.help.ts:269`, `docs/tools/browser.md:159`, and `docs/cli/browser.md:213`. I verified this on `main` at `41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00`; I could not confirm a released tag for it yet, so `fixedRelease` is left null.
