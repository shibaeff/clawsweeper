## Summary

Keep open. Current `main` still hard-codes exec command display truncation at 120 characters and only exposes `/verbose` as an on/off/full control; the proposed `verboseLimit` feature is not implemented on `main`, and the PR discussion already contains unresolved TUI correctness findings against the submitted patch.

## Evidence

- **Exec detail is still hard-coded to 120 chars on main:** `compactRawCommand` still defaults to `120`, and `resolveExecDetail` still calls it without any session-configurable length input, so current `main` has no shipped per-session verbose-limit path.
  - file: [src/agents/tool-display-exec.ts:377](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/tool-display-exec.ts#L377)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **TUI still only supports `/verbose <on|off>`:** The TUI command handler still prints `usage: /verbose <on|off>` and only patches `verboseLevel`; there is no `/verbose limit <n>` handling on `main`.
  - file: [src/tui/tui-command-handlers.ts:395](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/tui/tui-command-handlers.ts#L395)

## Risks / Open Questions

- If this PR is revived, the TUI session-switch and session-refresh paths need explicit tests for setting, clearing, and carrying `verboseLimit` across sessions.
- The web UI path would also need verification that any new `verboseLimit` patch refreshes the active session snapshot before tool cards re-render.
