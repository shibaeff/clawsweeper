## Summary

Current `main` and the shipped `v2026.4.23` release both restore Control UI image attachments when `chat.send` fails, and the WebSocket client rejects pending requests on disconnect so that restore path runs. The report matches an older `2026.4.15` bundle, not current behavior.

## Evidence

- **Current main restores attachments after failed send:** `sendChatMessageNow()` checks the `sendChatMessage()` result and restores `host.chatAttachments` from `opts.previousAttachments` when the send is not successful.
  - file: [ui/src/ui/app-chat.ts:178](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/ui/src/ui/app-chat.ts#L178)
  - command: `nl -ba ui/src/ui/app-chat.ts | sed -n '162,205p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Composer clear is paired with previousAttachments handoff:** `handleSendChat()` still clears the composer immediately, but passes the original attachments as `previousAttachments` into the send helper so they can be restored on failure.
  - file: [ui/src/ui/app-chat.ts:381](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/ui/src/ui/app-chat.ts#L381)
  - command: `nl -ba ui/src/ui/app-chat.ts | sed -n '318,399p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Disconnects reject pending RPCs:** `GatewayBrowserClient` flushes and rejects all pending requests on WebSocket close, and `request()` also rejects immediately when the socket is not open, so a connection failure surfaces as a failed `chat.send` instead of silently dropping the request.
  - file: [ui/src/ui/gateway.ts:334](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/ui/src/ui/gateway.ts#L334)

  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **The same fix shipped in v2026.4.23:** The `v2026.4.23` release snapshot already contains the same `previousAttachments` restore logic in `ui/src/ui/app-chat.ts`, so this behavior is not limited to unreleased `main`.
  - file: [ui/src/ui/app-chat.ts:168](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/ui/src/ui/app-chat.ts#L168)

## Risks / Open Questions

- There is no targeted `app-chat` test in the current tree that explicitly asserts attachment restoration on failed send, so a future refactor could regress this path without a focused test.
- A narrower transport edge case where the browser never surfaces a WebSocket close/error could still leave the UI waiting, but that is different from the reported 'attachments are lost on connection failure' behavior.
