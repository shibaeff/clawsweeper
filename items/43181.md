## Summary

Current `main` already implements Telegram polling-stall mitigation for long-running work by tracking recent/in-flight non-`getUpdates` API activity, only forcing a restart when both poll liveness and overall Telegram API liveness are stale. The behavior is covered by regression tests and documented as shipped in the `v2026.4.20` release notes.

## Evidence

- **Polling liveness now tracks non-poll API activity:** `TelegramPollingLivenessTracker` records both successful API activity and the newest in-flight non-`getUpdates` API call start time, which is the key mitigation for false polling-stall restarts during long-running work.

## Close Comment

Closing from a Codex review. Current `main` already has Telegram polling-stall mitigation for this class of long-running work: `extensions/telegram/src/polling-liveness.ts:37` tracks recent/in-flight non-`getUpdates` API activity, and `extensions/telegram/src/polling-liveness.ts:92` only forces a restart when both `getUpdates` liveness and overall Telegram API liveness are stale. Regression coverage in `extensions/telegram/src/polling-session.test.ts:695` onward specifically verifies that recent or in-flight non-`getUpdates` work does not trigger the watchdog restart.
