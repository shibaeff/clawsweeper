## Summary

Keep open. Current `main` does not implement this behavior, and the PR as written changes established semantics without updating the affected tests/callers.

## Evidence

- **Current main still skips replies for existing requests:** `issuePairingChallenge` returns early when `created` is false, so repeated `/start` messages do not resend the pairing reply on current `main`. That means the request is not already implemented.
  - file: [src/pairing/pairing-challenge.ts:31](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/pairing/pairing-challenge.ts#L31)
  - command: `nl -ba src/pairing/pairing-challenge.ts | sed -n '1,120p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Existing requests still carry a reusable code:** `upsertChannelPairingRequest` returns `{ code, created: false }` for an existing pending request. Swapping the guard from `!created` to `!code` would therefore resend on repeats and change the function result path, not just unblock a missing-code edge case.
  - file: [src/pairing/pairing-store.ts:576](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/pairing/pairing-store.ts#L576)
  - command: `nl -ba src/pairing/pairing-store.ts | sed -n '530,610p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Core test suite currently locks the old behavior:** The unit test `does not send a reply when request already exists` expects `sendPairingReply` not to run and the result to be `{ created: false }` when `created: false` comes back from the store. The one-line PR would break that test unchanged.
  - file: [src/pairing/pairing-challenge.test.ts:94](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/pairing/pairing-challenge.test.ts#L94)
  - command: `nl -ba src/pairing/pairing-challenge.test.ts | sed -n '70,150p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Channel-level behavior also expects no repeat creation event:** Channel tests and callers treat this as a first-time event. For example, Discord has `skips pairing reply when pairing request already exists`, and Telegram uses `onCreated` for a new pairing-request log entry. The proposed change would need coordinated updates to those semantics, not a cleanup close.
  - file: [extensions/discord/src/monitor/dm-command-decision.test.ts:82](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/discord/src/monitor/dm-command-decision.test.ts#L82)
  - command: `nl -ba extensions/discord/src/monitor/dm-command-decision.test.ts | sed -n '70,95p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **`onCreated` currently means new request creation:** Telegram's pairing path logs `"telegram pairing request"` inside `onCreated`, so firing that callback on every repeated `/start` would duplicate first-creation telemetry unless the callback contract is changed or re-guarded.
  - file: [extensions/telegram/src/dm-access.ts:102](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/extensions/telegram/src/dm-access.ts#L102)
