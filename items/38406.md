## Summary

Keep open. Current `main` only embeds the original filename into the staged outbound media path; it does not show evidence that Signal receives a clean basename without the `---<uuid>` suffix, so this PR's stated behavior is not clearly implemented on `main`.

## Evidence

- **Outbound attachment resolver returns saved path directly:** `resolveOutboundAttachmentFromUrl()` passes `media.fileName` into `saveMediaBuffer()` and then returns `saved.path` unchanged. There is no symlink alias, renamed handoff path, or other Signal-specific basename preservation in this function.
  - file: [src/media/outbound-attachment.ts:23](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/media/outbound-attachment.ts#L23)
  - command: `nl -ba src/media/outbound-attachment.ts | sed -n '1,80p'`
  - sha: [76a4c167f7ef](https://github.com/openclaw/openclaw/commit/76a4c167f7ef65040fd5b00d206c59b8bb28fd49)
- **Saved media IDs still include UUID in the basename:** `buildSavedMediaId()` formats stored files as `{sanitizedOriginal}---{uuid}{ext}` when `originalFilename` is present. If Signal uses the path basename as the displayed filename, current `main` would still expose the UUID suffix rather than the exact original filename.
  - file: [src/media/store.ts:271](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/media/store.ts#L271)

- **Signal sender still hands signal-cli the raw staged path:** `extensions/signal/src/send.ts` sets `attachments = [resolved.path]` and forwards that array directly in RPC params. There is no extraction of the original filename or alternate attachment encoding on the current path.
  - file: [extensions/signal/src/send.ts:153](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/extensions/signal/src/send.ts#L153)

I did not find a Signal test that proves recipients see the clean original filename.
