## Summary

Keep open. Current `main` still does a single pass across fallback candidates, exposes only `{ primary, fallbacks }` model config, and intentionally rethrows context-overflow errors instead of compacting to fit a smaller fallback model. The only retry I found is a one-time outer retry of the entire fallback cycle for transient HTTP failures, which does not implement the requested per-candidate retry/backoff or `safe`/`strict` context mode.

## Evidence

## Risks / Open Questions

- Adding safe fallback-to-smaller-context behavior would need explicit coordination with the embedded runner's preemptive compaction path so the fallback model's actual context window drives compaction decisions.
- The current contract deliberately keeps context overflow inside compaction/retry logic; changing that boundary risks regressions in fallback summaries, cooldown decisions, and prompt-precheck behavior.
