## Summary

Keep open. Current `main` still has the GitHub Copilot device-flow behavior this PR is trying to fix, and the newer extension-owned auth path means the PR likely needs a rebase/port rather than closure.

## Evidence

- **Current poll loop still fires immediately and ignores server-provided slow_down interval:** `pollForAccessToken` posts to the token endpoint at the top of the loop before any initial sleep. On `authorization_pending` it sleeps `params.intervalMs`, and on `slow_down` it sleeps `params.intervalMs + 2000`; there is no parsed `interval` field on the error response and no first-poll delay timestamp logic.
  - file: [extensions/github-copilot/login.ts:68](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/extensions/github-copilot/login.ts#L68)
  - command: `nl -ba extensions/github-copilot/login.ts | sed -n '68,116p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **No --wait support in the current login implementation:** `githubCopilotLoginCommand` only accepts `{ profileId?: string; yes?: boolean }`, always renders the clack `note()` with URL/code, and starts polling immediately after that. There is no confirmation prompt or plain-text URL/code path for WSL/remote terminals.
  - file: [extensions/github-copilot/login.ts:119](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/extensions/github-copilot/login.ts#L119)
