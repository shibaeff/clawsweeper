## Summary

The issue proposes bundling a third-party memory plugin into OpenClaw core, but current main explicitly steers optional capabilities and community-maintained plugins to the external plugin path (ClawHub/npm) rather than new core inclusion. That policy is especially strong for memory plugins, where OpenClaw already supports external memory-slot plugins and states it wants to converge toward one recommended default path instead of expanding bundled memory variants.

## Close Comment

Closing after a Codex review because this belongs on the community plugin path, not in OpenClaw core. Current main says optional capability should usually ship as plugins, plugin authors should host and maintain plugins in their own repos, and the bar for adding optional plugins to core is intentionally high (`VISION.md:54-64`). The community plugin docs also direct third-party plugins to ClawHub/npm for discovery and installation instead of core inclusion (`docs/plugins/community.md:9-16`, `docs/plugins/community.md:129-149`). OpenClaw already supports external memory plugins through the `plugins.slots.memory` seam (`docs/plugins/manifest.md:684`, `src/plugins/memory-state.ts:143-151`). Since `memory-lancedb-pro` is a third-party memory plugin maintained in its own repository, the appropriate path is ClawHub/npm distribution rather than bundling it into OpenClaw.
