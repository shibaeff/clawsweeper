## Summary

Keep open. Current `main` still persists a per-endpoint TLS fingerprint on Android and hard-fails reconnects when the certificate changes, with no in-app re-trust or pin-reset path for the same `stableId`. That matches the reported reverse-proxy certificate-rotation failure closely enough that conservative cleanup should not auto-close it.

## Evidence

- **Stored pin is always preferred and discovery cannot replace it:** `resolveTlsParamsForEndpoint` uses any saved fingerprint as the required expected fingerprint, and all branches set `allowTOFU = false`. If discovery only hints TLS, the app still requires TLS but does not trust the advertised TXT fingerprint.
