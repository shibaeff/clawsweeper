## Summary

Keep open. Current `main` still appears to reject explicit `chrome-extension://...` origins because `src/gateway/origin-check.ts:14-27` normalizes `new URL(origin).origin`, and Node returns `"null"` for Chrome extension URLs, so the allowlist comparison at `src/gateway/origin-check.ts:45-52` cannot match a configured `chrome-extension://<id>` entry. That matches the report and the commenter’s `"*" works` workaround.

## Evidence

- **Origin parser collapses extension origins to URL.origin:** `src/gateway/origin-check.ts:14-27` parses the incoming Origin with `new URL(trimmed)` and stores `url.origin`. For `chrome-extension://pfhemcnpfilapbppdkfemikblgnnikdp`, Node reports `origin: "null"`, `host: "pfhemcnpfilapbppdkfemikblgnnikdp"`, `protocol: "chrome-extension:"`, so the canonicalized origin string is not the configured extension origin.
  - file: [src/gateway/origin-check.ts:14](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/gateway/origin-check.ts#L14)
  - command: `node -e 'const u=new URL("chrome-extension://pfhemcnpfilapbppdkfemikblgnnikdp"); console.log(JSON.stringify({origin:u.origin,host:u.host,hostname:u.hostname,protocol:u.protocol}))'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Allowlist match only checks parsedOrigin.origin:** `src/gateway/origin-check.ts:45-52` builds a normalized allowlist and accepts only `"*"` or `parsedOrigin.origin`. With extension origins parsing to `"null"`, an explicit `chrome-extension://<id>` entry will not match, while `"*"` still will.
  - file: [src/gateway/origin-check.ts:45](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/gateway/origin-check.ts#L45)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Handshake path uses this check for browser/operator UI connects:** `src/gateway/server/ws-connection/message-handler.ts:473-497` passes `configSnapshot.gateway?.controlUi?.allowedOrigins` into `checkBrowserOrigin(...)` and emits the exact `origin not allowed` handshake error when that check fails.
  - file: [src/gateway/server/ws-connection/message-handler.ts:473](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/gateway/server/ws-connection/message-handler.ts#L473)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Docs promise explicit full-origin allowlisting:** `docs/web/control-ui.md:387-402` and `docs/gateway/configuration-reference.md:319-320` describe `gateway.controlUi.allowedOrigins` as an explicit full-origin allowlist for browser websocket connects. Current handling of `chrome-extension://...` does not satisfy that contract.
  - file: [docs/web/control-ui.md:387](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/docs/web/control-ui.md#L387)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Regression coverage is missing for extension schemes:** `src/gateway/origin-check.test.ts:5-93` covers http/https allowlist, wildcard, fallback, and invalid origins, but there is no case for `chrome-extension://...`, so this path is currently unguarded.
  - file: [src/gateway/origin-check.test.ts:5](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/gateway/origin-check.test.ts#L5)
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
