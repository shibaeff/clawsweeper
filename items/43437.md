## Summary

Current main already contains shipped Telegram fixes that match the reported partial-health DM failure pattern: stale ACP DM bindings are cleaned on startup before routing can reuse them, polling transport stale-socket recovery now rebuilds and disposes transports explicitly, and Telegram fetch policy no longer mutates process-global undici dispatcher settings. The HEARTBEAT disappearance noted in the issue is also expected when heartbeat OK messages are suppressed.

## Close Comment

Closing from a Codex review because current `main` already includes the Telegram fixes that match this report well enough. Startup now cleans up stale ACP Telegram bindings before restart routing can reuse them (`extensions/telegram/src/thread-bindings.ts:443`, covered by `extensions/telegram/src/thread-bindings.test.ts:337`), polling transport rebuilds explicitly close stale transports and publish successful `getUpdates` liveness (`extensions/telegram/src/polling-transport-state.ts:31`, `extensions/telegram/src/polling-status.ts:20`), and Telegram fetch policy is no longer mutating process-global undici dispatcher settings (`extensions/telegram/src/fetch.test.ts:821`). These Telegram polling/ACP fixes were shipped in `v2026.4.18` (`CHANGELOG.md:460`, especially `CHANGELOG.md:484`-`486`). Also, the reported `HEARTBEAT` disappearance is expected with default heartbeat visibility because `HEARTBEAT_OK` is suppressed unless `showOk` is enabled (`src/config/types.channels.ts:4`). If this still reproduces on a build newer than `v2026.4.18`, please open a new issue with a current diagnostics export and the exact Telegram account/binding config so it can be checked as a fresh regression.
