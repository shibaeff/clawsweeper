## Summary

Keep open: current `main` only handles Windows code-page decoding in the buffered node-host path, while the streamed exec/supervisor paths targeted by this PR still use plain `Buffer.toString()`. The PR discussion also identifies unresolved regressions in the proposed patch, so this is neither implemented on `main` nor ready for conservative cleanup.

## Evidence

- **Current main already has only partial Windows decode support:** `src/node-host/invoke.ts` contains Windows code-page parsing and buffered output decoding, and `runCommand()` applies that logic only after collecting complete stdout/stderr buffers. `src/node-host/invoke.sanitize-env.test.ts` covers the GBK decode case. This matches the PR discussion's cross-reference to prior work, but it is limited to node-host capture rather than the streamed process paths in this PR.
  - file: [src/node-host/invoke.ts:108](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/node-host/invoke.ts#L108)
  - command: `nl -ba src/node-host/invoke.ts | sed -n '108,165p'; nl -ba src/node-host/invoke.ts | sed -n '200,264p'; nl -ba src/node-host/invoke.sanitize-env.test.ts | sed -n '60,95p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **`runCommandWithTimeout()` still decodes streamed output with plain UTF-8 coercion:** In current `main`, `src/process/exec.ts` appends `child.stdout` and `child.stderr` chunks via `d.toString()` at the streamed execution path. There is no shared Windows console decoder there, so the behavior described by this PR is not implemented on `main`.
  - file: [src/process/exec.ts:340](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/process/exec.ts#L340)
