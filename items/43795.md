## Summary

Keep open. Current `main` still has the reported failure mode on the embedded OpenAI-compatible response path: assistant message parsing still assumes `item.content` is iterable, so a provider returning string/object/null content can still throw before failover surfaces the original provider error.

## Evidence

- **Current main still routes completed provider responses through the unsafe parser:** On `response.completed`, the websocket stream builds the final assistant message via `buildAssistantMessageFromResponse(event.response, ...)`, so malformed provider payloads still hit this parser on the embedded agent path described in the issue.
  - file: [src/agents/openai-ws-stream.ts:1100](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/openai-ws-stream.ts#L1100)
  - command: `nl -ba src/agents/openai-ws-stream.ts | sed -n '1088,1110p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Parser still assumes assistant `item.content` is an array:** `buildAssistantMessageFromResponse()` still calls `item.content?.some(...)` and then iterates `for (const part of item.content ?? [])` without an `Array.isArray` guard. If a third-party OpenAI-compatible provider returns string/object/null content, this path can still throw instead of degrading safely.
  - file: [src/agents/openai-ws-message-conversion.ts:507](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/openai-ws-message-conversion.ts#L507)
