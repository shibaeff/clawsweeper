## Summary

Keep open. Current `main` still uses plain UTF-8 decoding in the affected exec/supervisor paths, while the repo docs still describe garbled Chinese Windows exec output as a current workaround-only problem. That is not strong evidence for auto-closing this PR.

## Evidence

- **`runExec()` on current main still forces UTF-8:** `runExec()` passes `encoding: "utf8"` into `execFileAsync` and returns that decoded text directly, so it does not use the repo's Windows code-page-aware decoding helper in this path.
  - file: [src/process/exec.ts:136](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/process/exec.ts#L136)
  - command: `nl -ba src/process/exec.ts | sed -n '120,170p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **`runCommandWithTimeout()` still decodes streaming output with `toString()`:** The timeout-based spawn path still appends `d.toString()` for stdout/stderr chunks, which is exactly the kind of unconditional UTF-8 decode the PR is trying to replace for Windows console output.
  - file: [src/process/exec.ts:340](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/process/exec.ts#L340)

- **Supervisor child/PTY adapters still lack Windows code-page handling:** `createChildAdapter()` uses `chunk.toString()` for stdout/stderr, and the PTY adapter forwards `chunk` directly. Current main therefore still has non-UTF-8 Windows output paths outside `src/node-host/invoke.ts`.
  - file: [src/process/supervisor/adapters/child.ts:111](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/process/supervisor/adapters/child.ts#L111)
