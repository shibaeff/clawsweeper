## Summary

Keep open. Current `main` still does not validate explicit gateway agent IDs from `x-openclaw-agent-id` / `x-openclaw-agent` or `model` selectors against configured agents before routing `/v1/chat/completions` and `/v1/responses`, so the PR’s underlying bug is not implemented on `main` and the contract mismatch remains.

## Evidence

## Risks / Open Questions

- If this PR is revived, the validation logic should preserve documented explicit `main` compatibility examples such as `docs/gateway/openresponses-http-api.md:321` and account for the current configurable default-agent model (`openclaw/default` vs literal `main`).
- Current `main` uses the same unchecked `resolveAgentIdForRequest` path in `/v1/embeddings` (`src/gateway/embeddings-http.ts:271`), so maintainers may want to decide whether agent-id rejection should be consistent across all compat HTTP endpoints.
