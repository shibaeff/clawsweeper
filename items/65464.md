## Summary

Current `main` already handles transient cron tool-call failures followed by a later successful payload/delivery: intermediate error payloads are treated as non-fatal when a later success exists, the isolated runner returns terminal `status: "ok"` without an `error`, and cron state persistence records the terminal status/delivery outcome. The same implementation is present in released `v2026.4.23`.

## Evidence

- **Transient error payloads are ignored once a later success exists:** `resolveCronPayloadOutcome()` marks error payloads non-fatal when there is a later non-error payload after the last error, which directly matches the report's "first message() fails, retry succeeds" sequence.
  - file: [src/cron/isolated-agent/helpers.ts:150](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/cron/isolated-agent/helpers.ts#L150)

- **Isolated cron runner converts that non-fatal sequence into terminal ok status:** `runCronIsolatedAgentTurn()` uses `hasFatalErrorPayload` to decide terminal status. When the helper says the later success makes the earlier error non-fatal, the returned run outcome is `status: "ok"` and omits `error` while preserving delivery metadata.
  - file: [src/cron/isolated-agent/run.ts:814](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/cron/isolated-agent/run.ts#L814)

- **Cron state persistence follows terminal result, not intermediate tool failures:** `applyJobResult()` writes `lastRunStatus`/`lastStatus` from the final run result, stores delivery state separately, and resets `consecutiveErrors` on non-error runs. Because the runner's success path omits `error`, `lastError` is cleared on a recovered run.
