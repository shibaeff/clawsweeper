## Summary

Current `main` already implements the requested strict execution behavior via the embedded Pi `strict-agentic` execution contract, including retrying plan-only/filler turns, treating `update_plan` alone as non-progress, and surfacing an explicit blocked state when the model still does not act.

## Evidence

- **User-facing config and behavior are documented:** The docs describe `agents.defaults.embeddedPi.executionContract: "strict-agentic"` and say it no longer treats plan-only turns as success, retries with an act-now steer, auto-enables `update_plan`, and surfaces an explicit blocked state.
  - file: [docs/providers/openai.md:739](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/docs/providers/openai.md#L739)
  - command: `nl -ba docs/providers/openai.md | sed -n '736,764p'`
- **Public config type includes the strict execution contract:** Agent defaults expose `executionContract?: EmbeddedPiExecutionContract` with `strict-agentic` described as keeping GPT-5 OpenAI/OpenAI Codex runs acting until a real blocker.
  - file: [src/config/types.agent-defaults.ts:289](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/config/types.agent-defaults.ts#L289)
  - command: `nl -ba src/config/types.agent-defaults.ts | sed -n '284,296p'`
- **Runtime enforces the contract:** `resolveEffectiveExecutionContract` activates `strict-agentic` for supported GPT-5 OpenAI/OpenAI Codex runs, and the embedded Pi runner retries planning-only turns with an act-now steer before returning an explicit blocked terminal payload when retries are exhausted.
  - file: [src/agents/execution-contract.ts:46](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/agents/execution-contract.ts#L46)

  - file: [src/agents/pi-embedded-runner/run.incomplete-turn.test.ts:158](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/agents/pi-embedded-runner/run.incomplete-turn.test.ts#L158)

## Risks / Open Questions

- The implemented solution is a native strict execution contract, not a general-purpose `validateAssistantTurn` hook; closing relies on the issue explicitly accepting a strict-mode option as a valid solution.

## Close Comment

Closing after Codex review because current `main` already implements this request well enough. OpenClaw now has a first-class embedded Pi `strict-agentic` execution contract (`agents.defaults.embeddedPi.executionContract: "strict-agentic"`), documented in `docs/providers/openai.md:739` and typed in `src/config/types.agent-defaults.ts:289`. In the runtime, supported GPT-5 OpenAI/OpenAI Codex runs use that contract, planning-only/filler turns are retried with an act-now steer, and repeated non-actionable turns end in an explicit blocked payload instead of being accepted as successful progress (`src/agents/execution-contract.ts:46`, `src/agents/pi-embedded-runner/run.ts:1827`, `src/agents/pi-embedded-runner/run.ts:1936`). Tests also cover that `update_plan` alone does not count as progress and that repeated plan-only turns surface the blocked state (`src/agents/pi-embedded-runner/run.incomplete-turn.test.ts:158`, `src/agents/pi-embedded-runner/run.incomplete-turn.test.ts:594`). The changelog shows this shipped in `v2026.4.10` (`CHANGELOG.md:860-881`).
