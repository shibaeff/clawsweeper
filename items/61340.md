## Summary

Keep open. Current `main` has a partial mitigation for systemd installs by moving state-dir `.env` values into `EnvironmentFile`, but the report is not fully resolved: gateway install plans still carry auth-profile/config-derived secrets such as `OPENAI_API_KEY`, the systemd writer still renders non-`.env` managed values inline in the unit, reinstall still copies the previous unit to `.bak` before overwrite, and doctor still recommends `gateway install --force` as the cleanup path for embedded gateway tokens. The related open RFC #69066 in the issue timeline also points to ongoing work in this area rather than a completed cleanup.

## Evidence

- **Gateway install plan still persists auth-profile env secrets into service environment:** `buildGatewayInstallEnvironment` merges `collectAuthProfileServiceEnvVars(...)` into the managed service environment. The corresponding test asserts `OPENAI_API_KEY` and `ANTHROPIC_TOKEN` are present in `plan.environment`, so those values are still eligible to reach service installers.

## Risks / Open Questions

- Closing this as implemented would miss that only state-dir `.env` values were moved to `EnvironmentFile`; auth-profile/config-derived secrets can still be written inline.
- Closing would also miss the remaining `.bak` exposure path during reinstall of older secret-bearing units.
- Doctor guidance still points affected users at the reinstall flow, so the reported workflow is still user-relevant on current `main`.
