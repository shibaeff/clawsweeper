## Summary

Current `main` already reports cron jobs with `delivery.mode = "none"` as `not-requested`, and the regression is covered by tests. The same code is present in release `v2026.4.22`, so the issue is implemented on shipped builds newer than the reporter's `2026.4.15` test.

## Evidence

  - file: [src/cron/delivery-plan.ts:57](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/cron/delivery-plan.ts#L57)

## Close Comment

Closing from a Codex review: current `main` already handles this case correctly. `src/cron/delivery-plan.ts:57` marks `delivery.mode = "none"` as `requested: false`, and `src/cron/service/timer.ts:261` returns `lastDeliveryStatus = "not-requested"` before any `delivered === false` path can produce `"not-delivered"`. The regression is also covered in `src/cron/service.persists-delivered-status.test.ts:176`. I also verified the same logic is present in the shipped `v2026.4.22` release, so this appears fixed in builds newer than `2026.4.15`. If someone can still reproduce on `v2026.4.22` or later, please open a fresh issue with the exact version and exported cron job JSON.
