## Summary

Current `main` does not auto-populate `SessionEntry.label` from messaging-provider chat/group names, and the Feishu inbound path still feeds the chat ID into group metadata. The sessions UI can show a secondary `displayName`, but the actual Label column remains whatever was manually set, so the reported problem is not implemented on `main`.

## Evidence

- **Feishu group inbound context still uses chat ID, not chat name:** The Feishu bot builds inbound session context with `GroupSubject: isGroup ? ctx.chatId : undefined`, so group session metadata is seeded from the raw chat ID rather than a human-readable group name.
  - file: [extensions/feishu/src/bot.ts:996](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/feishu/src/bot.ts#L996)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '976,1002p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Session recording preserves label only from existing/manual state:** When recording inbound session state, `label` is set from `persistedLabel ?? baseEntry?.label`; the metadata patch runs afterward, so provider-derived metadata does not write `SessionEntry.label`.
  - file: [src/auto-reply/reply/session.ts:643](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/auto-reply/reply/session.ts#L643)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '638,676p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Derived metadata writes displayName and origin.label, not label:** `deriveSessionOrigin` stores the resolved conversation label under `origin.label`, and `deriveSessionMetaPatch` merges only `groupPatch` plus `origin`; it does not populate `patch.label`.
  - file: [src/config/sessions/metadata.ts:63](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/config/sessions/metadata.ts#L63)
  - command: `nl -ba src/config/sessions/metadata.ts | sed -n '52,93p'; nl -ba src/config/sessions/metadata.ts | sed -n '181,205p'`
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Gateway/UI still treat label as separate from displayName:** Gateway rows derive `displayName` from group metadata and fall back to `entry.label` only later, while the sessions page renders `displayName` under the key and binds the editable Label column to `row.label`. That means a friendly secondary name may appear, but the label field itself still stays empty unless manually set.
  - file: [src/gateway/session-utils.ts:1256](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/gateway/session-utils.ts#L1256)

  - file: [src/config/sessions/group.ts:57](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/config/sessions/group.ts#L57)

## Risks / Open Questions

- No live Feishu account reproduction was run in this read-only review, so this conclusion is from current code paths rather than an end-to-end session capture.
- Some providers may already set `ConversationLabel` or `displayName` on their own paths, but the issue asks for auto-populating the session `label` across messaging providers, and that broader behavior is not present on `main`.
