## Summary

Current `main` does not implement IPv6 gateway bind support for the reported LAN/bind-mode case. The gateway still resolves `lan` and container `auto` binds to IPv4 `0.0.0.0`, `tailnet` uses Tailnet IPv4 only, and `custom` bind paths are explicitly restricted to valid IPv4 addresses.

## Evidence

- **Gateway bind resolver is IPv4-only for LAN/auto/tailnet/custom:** `resolveGatewayBindHost` hardcodes `lan` to `0.0.0.0`, uses Tailnet IPv4 for `tailnet`, and only accepts `custom` hosts when `isValidIPv4(host)` passes; otherwise it falls back to `0.0.0.0`.
  - file: [src/gateway/net.ts:287](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/gateway/net.ts#L287)
  - command: `nl -ba src/gateway/net.ts | sed -n '287,338p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Only loopback gets dual-stack listen handling:** `resolveGatewayListenHosts` adds `::1` only when the resolved bind host is `127.0.0.1`; every non-loopback bind returns a single host unchanged, so `lan` does not become a dual-stack listener.
  - file: [src/gateway/net.ts:398](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/gateway/net.ts#L398)
  - command: `nl -ba src/gateway/net.ts | sed -n '398,409p'`

- **Runtime config rejects non-IPv4 custom bind hosts:** Gateway startup throws unless `gateway.customBindHost` is a valid IPv4 address, with the explicit error `gateway.bind=custom requires a valid IPv4 customBindHost`.
  - file: [src/gateway/server-runtime-config.ts:67](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/gateway/server-runtime-config.ts#L67)
  - command: `nl -ba src/gateway/server-runtime-config.ts | sed -n '67,80p'`
