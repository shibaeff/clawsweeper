## Summary

Keep open. Current main already has a schema guard that rejects unsupported provider-model `input` values such as `audio`, but I did not find strong evidence that the issue's full acceptance criteria are complete: the current checkout does not show targeted invalid-modality tests for `models.providers.*.models[*].input`, and the config docs/help do not document that field's validation expectations.

## Evidence

- **Schema guard exists:** `ModelDefinitionSchema` restricts `models.providers.*.models[*].input` to `"text" | "image"`, so `audio` would be rejected during config validation.
  - file: [src/config/zod-schema.core.ts:307](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/config/zod-schema.core.ts#L307)
- **Type surface is not fully aligned:** Other current-main type surfaces still allow `"document"` for model input metadata, which makes the overall modality contract look broader than the config schema and weakens an `implemented_on_main` close.
  - file: [src/plugin-sdk/provider-catalog-shared.ts:20](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/plugin-sdk/provider-catalog-shared.ts#L20)
- **Related catalog type also allows document:** `ModelInputType` includes `"document"`, again showing the current modality story is not a cleanly closed text/image-only validation path end to end.
  - file: [src/agents/model-catalog.types.ts:1](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/model-catalog.types.ts#L1)
- **Tests found are positive-path only:** The checked-in model-config tests exercise valid `input` metadata like `["text"]`, `["image"]`, and `["text", "image"]`, but I did not find targeted tests for invalid provider-model modality config such as `audio` under this schema path.
  - file: [src/agents/models-config.merge.test.ts:61](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/models-config.merge.test.ts#L61)
  - command: `rg -n 'models\.providers\.\*\.models\[\]\.input|input:\s*\[\"audio\"|unsupported.*audio|invalid modality' src docs test -g '*.ts' -g '*.md' -g '*.test.ts'`
- **Docs/help do not cover the input field expectation:** Current docs/help describe `models.providers.*.models` generically, but I did not find a field-level help/doc entry for `models.providers.*.models[].input` explaining the allowed modalities or CI/validation expectation.
  - file: [docs/gateway/config-tools.md:416](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/docs/gateway/config-tools.md#L416)
  - command: `rg -n 'models\.providers\.\*\.models\[\]\.input' src/config/schema.base.generated.ts src/config/schema.help.ts docs/gateway/config-tools.md docs/concepts/model-providers.md`

## Risks / Open Questions

- Closing as implemented would likely drop follow-up work for the missing invalid-modality test coverage and docs/help called out in the issue.
- The current schema/type mismatch around `document` means the model-input contract is not cleanly settled enough for a conservative maintainer close.
