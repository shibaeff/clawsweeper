## Summary

Current main already routes isolated cron announce delivery through the real outbound delivery path and only marks `delivered=true` after outbound sends succeed. The explicit Telegram cron path is covered by regression tests that verify `delivery: { mode: "announce", channel: "telegram", to }` sends via `sendMessageTelegram` and does not use the older subagent announce flow.

## Evidence

- **Cron delivery now uses actual outbound send results:** `dispatchCronDelivery` sends requested cron announce payloads through `deliverOutboundPayloads(...)`, then sets `delivered = deliveryResults.length > 0 && !hadPartialFailure`, so cron delivery status is tied to outbound send results instead of a successful internal agent call.
  - file: [src/cron/isolated-agent/delivery-dispatch.ts:602](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cron/isolated-agent/delivery-dispatch.ts#L602)
- **Code comment documents the fix intent:** The cron delivery path explicitly states that text-only cron runs are finalized and then sent through the real outbound adapter so `delivered=true` reflects an actual channel send instead of internal announce routing.
  - file: [src/cron/isolated-agent/delivery-dispatch.ts:822](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cron/isolated-agent/delivery-dispatch.ts#L822)
- **Explicit Telegram cron regression coverage:** The Telegram announce regression helper runs `runCronIsolatedAgentTurn` with `delivery: { mode: "announce", channel: "telegram", to }`, expects `res.delivered === true`, asserts `runSubagentAnnounceFlow` was not called, and verifies a real `sendMessageTelegram(...)` call.
  - file: [src/cron/isolated-agent.direct-delivery-core-channels.test.ts:211](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cron/isolated-agent.direct-delivery-core-channels.test.ts#L211)
- **Shipped release contains the corrected cron path:** The `v2026.4.22` release tag already contains the same direct cron delivery block in `src/cron/isolated-agent/delivery-dispatch.ts`, so this behavior is not only on current `main` but also present in the latest shipped release.
  - command: `git show v2026.4.22:src/cron/isolated-agent/delivery-dispatch.ts | nl -ba | sed -n '791,853p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Risks / Open Questions

- No live Telegram reproduction was run in this read-only review; the decision is based on current code, shipped tag contents, and regression coverage.
- The older subagent announce helper in `src/agents/subagent-announce-delivery.ts` still reports success from a successful gateway agent call, but explicit isolated cron announce delivery no longer goes through that path.

## Close Comment

Closing from a Codex review. Current `main` already implements this specific cron fix: isolated cron announce delivery now goes through the real outbound delivery path in `src/cron/isolated-agent/delivery-dispatch.ts:602`, and `delivered=true` is only set after outbound send results come back cleanly (`src/cron/isolated-agent/delivery-dispatch.ts:630`). The cron path also explicitly avoids the old internal announce-routing semantics (`src/cron/isolated-agent/delivery-dispatch.ts:822`). The explicit Telegram case is covered in `src/cron/isolated-agent.direct-delivery-core-channels.test.ts:211`, where `delivery: { mode: "announce", channel: "telegram", to }` is verified to call `sendMessageTelegram(...)` and not `runSubagentAnnounceFlow` (`src/cron/isolated-agent.direct-delivery-core-channels.test.ts:218`). This behavior is already present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this issue is implemented on main and released.
