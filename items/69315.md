## Summary

Keep open. Current `main` still does TTS inline before final reply delivery, and the public config only exposes `messages.tts.mode: "final" | "all"` plus `auto` modes, not an async/background delivery mode. The report's chunk-by-chunk description is partly outdated because default TTS mode is now `final`, but the requested non-blocking behavior is still not implemented.

## Evidence

- **Public TTS config has no async/background delivery option:** `src/config/types.tts.ts` defines `TtsMode` as only `"final" | "all"`, and `TtsConfig.mode` is documented as applying TTS to final replies only or all replies. `docs/tools/tts.md` matches that public surface and does not document any async/background delivery mode.
  - file: [src/config/types.tts.ts:3](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/config/types.tts.ts#L3)

  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Final reply path still awaits TTS before routing the reply:** In `src/auto-reply/reply/dispatch-from-config.ts`, `sendFinalPayload()` calls `await maybeApplyTtsToReplyPayload(...)` before `routeReplyToOriginating(...)` or `dispatcher.sendFinalReply(...)`. That is still synchronous inline delivery, not text-first/audio-later.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:683](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/auto-reply/reply/dispatch-from-config.ts#L683)
