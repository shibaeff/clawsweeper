## Summary

Keep open. Current `main` still has the normalization gap the PR describes, so there is not strong evidence to close it as already implemented or obsolete.

## Evidence

- **Core usage normalization still stops on zero-valued `input_tokens`:** `normalizeUsage` uses a nullish chain for input selection: `raw.input ?? raw.inputTokens ?? raw.input_tokens ?? raw.promptTokens ?? raw.prompt_tokens ...`. If a provider returns `input_tokens: 0` and `prompt_tokens: N`, the `0` wins and the later prompt token alias is never read, matching the bug described in the PR.
  - file: [src/agents/usage.ts:114](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/agents/usage.ts#L114)
  - command: `nl -ba src/agents/usage.ts | sed -n '100,151p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **CLI JSONL usage path still lacks `prompt_tokens`/`completion_tokens` fallback:** `toCliUsage` only reads positive `input_tokens`/`inputTokens` and `output_tokens`/`outputTokens`. It does not consider `prompt_tokens` or `completion_tokens`, so OpenAI-compatible CLI responses using those fields still lose usage data.
  - file: [src/agents/cli-output.ts:123](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/agents/cli-output.ts#L123)
  - command: `nl -ba src/agents/cli-output.ts | sed -n '113,146p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Assistant message conversion still depends on `normalizeUsage`:** OpenAI WS message conversion calls `normalizeUsage(response.usage)` and then builds assistant usage from that normalized result, so the zero-vs-prompt alias bug still affects surfaced usage on current `main`.
  - file: [src/agents/openai-ws-message-conversion.ts:598](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/agents/openai-ws-message-conversion.ts#L598)
  - command: `nl -ba src/agents/openai-ws-message-conversion.ts | sed -n '596,618p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Current tests cover prompt/completion aliases only when the primary fields are absent:** `usage.test.ts` covers `prompt_tokens` and `completion_tokens` by themselves, but there is no regression case for mixed payloads like `input_tokens: 0` with `prompt_tokens: N`. `cli-output.test.ts` likewise has no JSONL coverage for prompt/completion aliases.
  - file: [src/agents/usage.test.ts:58](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/agents/usage.test.ts#L58)

- **Simple reproduction of the nullish-chain behavior returns the wrong field:** Evaluating the same selection pattern used in `normalizeUsage` with `{ input_tokens: 0, prompt_tokens: 44 }` returns `0`, confirming the later alias is ignored.
