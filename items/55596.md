## Summary

Keep open: current `main` still renders code-mode markdown tables with UTF-16 string length instead of display width, so CJK/wide-grapheme columns remain misaligned on the Telegram and Discord paths that default to `tableMode: "code"`. The proposed PR is still addressing a live bug rather than something already implemented or obsolete.

## Evidence

- **Code-table renderer still pads by `.length`:** `renderTableAsCode` computes column widths from `cell?.text.length` and pads with `widths[i] - (cell?.text.length ?? 0)`, so wide CJK and emoji graphemes are treated as 1 code unit cell width instead of terminal/display width.
  - file: [src/markdown/ir.ts:521](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/markdown/ir.ts#L521)
  - command: `nl -ba src/markdown/ir.ts | sed -n '509,578p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Width-aware helper exists elsewhere but is not used here:** `visibleWidth()` already measures grapheme/display width, including CJK and emoji, in the shared terminal helper. That makes the markdown code-table path divergence concrete rather than hypothetical.
  - file: [src/terminal/ansi.ts:115](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/terminal/ansi.ts#L115)
  - command: `nl -ba src/terminal/ansi.ts | sed -n '88,119p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Telegram still routes outbound markdown tables through this path:** Telegram send resolves a markdown table mode for the channel and renders outbound text with that mode. The formatter passes `tableMode` into `markdownToIR`, so the broken code-table padding still affects Telegram when tables are rendered as code blocks.
  - file: [extensions/telegram/src/send.ts:659](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/send.ts#L659)

- **Discord still routes outbound markdown tables through this path:** Discord send resolves a table mode, then calls `convertMarkdownTables`, which uses `markdownToIRWithMeta(..., { tableMode: effectiveMode })` and renders code tables from the same markdown IR path.
  - file: [extensions/discord/src/send.outbound.ts:158](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/send.outbound.ts#L158)

- **Current-main reproduction still misaligns CJK rows:** Using the current `renderTableAsCode` width logic with current `visibleWidth()` on the PR's CJK-style sample produced line widths `[43,49,52]`, proving the rendered rows do not align on current `main`. The renderer should produce equal visible widths per row for a fixed-width code table.
