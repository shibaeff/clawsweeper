## Summary

Keep open. Current `main` still hard-codes a 30-minute WhatsApp inbound-silence watchdog, still forces reconnects from that timer, and still does not expose `messageTimeoutMs` or `watchdogCheckMs` through the supported config schema/docs or the normal gateway startup path.

## Evidence

- **Internal watchdog defaults still hard-coded:** `monitorWebChannel` still reads only internal tuning overrides and defaults to `30 * 60 * 1000` for `messageTimeoutMs` plus `60 * 1000` for `watchdogCheckMs` before constructing the connection controller.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:183](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/whatsapp/src/auto-reply/monitor.ts#L183)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor.ts | sed -n '120,210p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Watchdog still force-reconnects on idle time alone:** The connection controller timer compares `Date.now() - (lastInboundAt ?? startedAt)` to `this.messageTimeoutMs` and immediately `forceClose`s with status `499` once the threshold is exceeded.
  - file: [extensions/whatsapp/src/connection-controller.ts:565](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/whatsapp/src/connection-controller.ts#L565)
