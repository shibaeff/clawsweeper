## Summary

Keep open. Current `main` does not contain the proposed `application/msword` / `application/x-cfb` binary-classification change, and the PR’s own review discussion identified that its added test would stay green even without the production fix. That is not strong enough evidence for any cleanup close reason.

## Evidence

- **Current main still lacks the proposed MIME additions:** `isBinaryMediaMime()` on current `main` treats images, audio, video, octet-stream, common archives, `+zip`, and most `application/vnd.*` payloads as binary, but it does not include `application/msword` or `application/x-cfb`.
  - file: [src/media-understanding/apply.ts:309](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/media-understanding/apply.ts#L309)
  - command: `nl -ba src/media-understanding/apply.ts | sed -n '309,340p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Default file MIME allowlist explains the weak regression test:** Default file extraction allowlist is `text/plain`, `text/markdown`, `text/html`, `text/csv`, `application/json`, and `application/pdf`. That means a test using default config can skip `application/msword` / `application/x-cfb` later at the allowlist gate even if `isBinaryMediaMime()` is unchanged.
  - file: [src/media/input-files.ts:103](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/media/input-files.ts#L103)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Current tests have no landed coverage for legacy .doc / CFB binary classification:** Current `apply.test.ts` covers ZIP-like binary payloads, audio binaries, UTF-16 text, PDF handling, and configured `allowedMimes`, but there is no landed test for `application/msword` or `application/x-cfb` on `main`.
