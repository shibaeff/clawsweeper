## Summary

Current `main` already computes `nextRunAtMs` for one-shot `at` jobs when they are created, recomputes scheduler state on add, and immediately rearms the timer against the minimum `nextWakeAtMs`. The shipped `v2026.4.22` release also contains this behavior and a status regression check for one-shot wake times.

## Evidence

- **One-shot jobs get `nextRunAtMs` at creation:** `createJob` assigns `job.state.nextRunAtMs = computeJobNextRunAtMs(job, now)`, and `computeJobNextRunAtMs` returns the parsed `at`/`atMs` timestamp for enabled one-shot jobs.
  - file: [src/cron/service/jobs.ts:272](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/cron/service/jobs.ts#L272)

  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Scheduler add path persists and rearms immediately:** `add()` pushes the new job, recomputes next runs, persists, then calls `armTimer(state)`, and logs `schedulerNextWakeAtMs` from the current state.
  - file: [src/cron/service/ops.ts:249](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/cron/service/ops.ts#L249)

## Risks / Open Questions

- There is not an exact checked-in regression test matching the reporter's mixed recurring-plus-earlier-one-shot scenario, but the current add/rearm/min-wake code path makes that scenario work as described.

## Close Comment

Closing from a Codex review because current `main` already implements this behavior. One-shot `at` jobs get `state.nextRunAtMs` when created (`src/cron/service/jobs.ts:272-293, 606`), `cron add` persists and immediately rearms the scheduler (`src/cron/service/ops.ts:249-271`), and `nextWakeAtMs()` selects the minimum scheduled wake across all enabled jobs (`src/cron/service/jobs.ts:535-548`). The same implementation is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and there is a current test asserting one-shot wake reporting (`src/cron/service.skips-main-jobs-empty-systemevent-text.test.ts:111-126`). If you can still reproduce this on `v2026.4.22` or newer, please open a fresh issue with exact `cron add` commands or stored job JSON plus `openclaw cron status` output from the same run.
