## Summary

Keep open. Current main has related Telegram session-route work, but I could not verify that the reported plain-group persistence bug is fixed. The explicit coverage on main is for DMs and forum-topic groups, while the issue reports a plain Telegram group session (`agent:main:telegram:group:-5160837533`) losing history across a gateway restart.

## Evidence

- **Plain groups still skip `updateLastRoute`:** Telegram inbound context only attaches `updateLastRoute` when the chat is not a group or when a group has a resolved thread id. For a plain group with no topic/thread id, `updateLastRoute` stays undefined.
  - file: [extensions/telegram/src/bot-message-context.session.ts:398](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/telegram/src/bot-message-context.session.ts#L398)
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Focused regression test covers DMs and forum topics, not plain groups:** The dedicated `updateLastRoute` regression test asserts DM topics, regular DMs, forum-topic groups, and the forum General topic. It does not cover a non-topic Telegram group like the one in the report.
  - file: [extensions/telegram/src/bot-message-context.dm-topic-threadid.test.ts:60](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/telegram/src/bot-message-context.dm-topic-threadid.test.ts#L60)
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Inbound metadata write and route persistence are separate paths:** `recordInboundSession` always records inbound session metadata first, then only calls `updateLastRoute` if an update object is present. That means current main proves metadata recreation, but not the restart/persistence path described in the issue.
  - file: [src/channels/session.ts:43](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/channels/session.ts#L43)
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Only direct-session recreation is explicitly regression-tested:** The current Telegram recreation regression test covers a deleted direct session reappearing after the next inbound message. I did not find an analogous group-session recreation test on main.
  - file: [src/gateway/session-utils.telegram-recreate.test.ts:66](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/session-utils.telegram-recreate.test.ts#L66)
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
