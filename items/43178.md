## Summary

Current main already changed Telegram stall handling from a full gateway SIGUSR1 restart to an in-process polling-runner restart, with a higher default watchdog threshold and configurable per-account overrides. That matches the substance of the report closely enough to close as implemented.

## Evidence

- **Polling watchdog now restarts only the Telegram polling cycle:** On stall, the Telegram session marks the transport dirty, stops the runner/bot, and schedules a polling-cycle restart; it does not call any gateway SIGUSR1 restart path from the Telegram code.
  - file: [extensions/telegram/src/polling-session.ts:313](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/telegram/src/polling-session.ts#L313)
  - command: `nl -ba extensions/telegram/src/polling-session.ts | sed -n '220,420p'`
- **Telegram extension no longer contains a SIGUSR1 watchdog path:** A repo search found no `SIGUSR1`, `scheduleGatewaySigusr1Restart`, `polling loop died`, or `auto-restarting` references under `extensions/telegram`.
  - command: `rg -n "SIGUSR1|scheduleGatewaySigusr1Restart|polling loop died|auto-restarting" extensions/telegram -g '*.ts'`

## Risks / Open Questions

- I did not run a live reproduction because this review had to remain read-only and artifact-free.

## Close Comment

Closing this as implemented on current `main` after a Codex review. Telegram polling stalls no longer route through a full gateway `SIGUSR1` restart path; the watchdog now stops and restarts only the Telegram polling runner (`extensions/telegram/src/polling-session.ts:313`, `extensions/telegram/src/polling-session.ts:357`). OpenClaw also raised the default stall threshold to 120s and made it configurable with `channels.telegram.pollingStallThresholdMs`, including per-account overrides (`src/config/types.telegram.ts:155`, `extensions/telegram/src/monitor.ts:230`, `extensions/telegram/src/config-schema.test.ts:56`, `docs/channels/telegram.md:260`). The changelog entry for `v2026.4.22` calls out this exact Telegram polling change (`CHANGELOG.md:388`). If you still see session drops on a current build, please open a fresh issue with new logs from that build, since that would be a different bug than the old full-gateway watchdog restart behavior.
