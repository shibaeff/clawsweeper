## Summary

Current main already implements Slack implicit reply-to-bot thread detection for channel messages, including recovery when Slack omits `thread_ts`, and this behavior is covered by targeted tests and documented in the channel docs.

## Evidence

  - file: [docs/channels/groups.md:228](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/docs/channels/groups.md#L228)

  - file: [docs/channels/slack.md:492](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/docs/channels/slack.md#L492)

## Close Comment

Closing from a Codex review because this is already implemented on current `main`, and the same behavior is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). In `extensions/slack/src/monitor/message-handler/prepare.ts:327`, Slack thread replies where `parent_user_id === ctx.botUserId` are treated as the `reply_to_bot` implicit-mention path. In `extensions/slack/src/monitor/thread-resolution.ts:81`, Slack events that arrive with `parent_user_id` but no `thread_ts` are recovered via `conversations.history` instead of being dropped. The current tests cover both the room-thread case and the default non-explicit-mention case in `extensions/slack/src/monitor.tool-result.test.ts:478` and `extensions/slack/src/monitor/message-handler/prepare.test.ts:1176`, matching the docs in `docs/channels/groups.md:228` and `docs/channels/slack.md:492`. This report does not match current main anymore.
