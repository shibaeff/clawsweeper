## Summary

Current main already addresses this well enough: chat history filters out the synthetic transcript-repair tool result before rendering, OpenAI/Codex replay paths no longer synthesize missing tool results, and teardown now waits for idle or clears pending tool-call state to avoid restart/long-command races.

## Evidence

- **History view hides the synthetic repair marker:** `loadChatHistory()` filters `toolResult` entries whose text matches the synthetic transcript-repair message before storing `chatMessages`.
  - file: [ui/src/ui/controllers/chat.ts:61](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/ui/src/ui/controllers/chat.ts#L61)
- **History filtering is covered by regression tests:** Tests verify the synthetic transcript-repair tool result is removed from visible history while real tool results remain.
  - file: [ui/src/ui/controllers/chat.test.ts:565](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/ui/src/ui/controllers/chat.test.ts#L565)
- **OpenAI/Codex replay no longer synthesizes missing tool results:** Synthetic tool-result insertion is limited to Anthropic/Gemini/Bedrock APIs; OpenAI-compatible transports are excluded.
  - file: [src/agents/transport-message-transform.ts:3](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/agents/transport-message-transform.ts#L3)
- **Replay-policy regression test for OpenAI-compatible transports:** Test asserts OpenAI-compatible transport replay keeps only the assistant and user messages instead of inserting a synthetic tool result.
  - file: [src/agents/transport-message-transform.test.ts:24](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/agents/transport-message-transform.test.ts#L24)
- **Default transcript policy disables synthetic tool results unless a provider opts in:** The default transcript policy sets `allowSyntheticToolResults` to `false`, and only Google/Anthropic-style fallbacks enable it.
  - file: [src/agents/transcript-policy.ts:47](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/agents/transcript-policy.ts#L47)
- **Restart/teardown cleanup now avoids synthetic flush races:** Cleanup waits for agent idle and, on timeout, can clear pending tool calls without persisting synthetic results; embedded-runner teardown uses that path.
  - file: [src/agents/pi-embedded-runner/wait-for-idle-before-flush.ts:43](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/src/agents/pi-embedded-runner/wait-for-idle-before-flush.ts#L43)
- **Changelog documents the shipped user-facing history fix:** The changelog entry for 2026.4.12 says WebChat hides synthetic transcript-repair tool results from history reloads.
  - file: [CHANGELOG.md:713](https://github.com/openclaw/openclaw/blob/76a4c167f7ef65040fd5b00d206c59b8bb28fd49/CHANGELOG.md#L713)

## Risks / Open Questions

- I did not run a live gateway reproduction in this read-only review, so this conclusion is based on current code, tests, and changelog evidence.
- The directly evidenced user-facing hide is for WebChat/Control UI history reloads; if the reporter was seeing the marker in a different surface, that surface would need separate confirmation.

## Close Comment

Closing as `implemented_on_main` based on a Codex review of current `main`. The current WebChat history path explicitly filters the synthetic transcript-repair tool result before rendering (`ui/src/ui/controllers/chat.ts:61`, covered by `ui/src/ui/controllers/chat.test.ts:565`), and current replay handling no longer synthesizes missing tool results for OpenAI/Codex-style transports (`src/agents/transport-message-transform.ts:3`, covered by `src/agents/transport-message-transform.test.ts:24`). Current teardown also waits for idle or clears pending tool-call state to avoid restart/long-command flush races (`src/agents/pi-embedded-runner/wait-for-idle-before-flush.ts:43`). There is also a shipped changelog entry for hiding these markers from history reloads in `CHANGELOG.md:713`. If you can still reproduce this on current `main`, please open a new issue with the exact surface (WebChat/TUI/etc.), provider/model, and whether it appears during live streaming or only after history reload.
