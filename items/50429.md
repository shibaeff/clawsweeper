## Summary

Keep open. Current `main` still drops non-streaming Feishu card replies when `sendStructuredCardFeishu()` throws, so this is not implemented on `main`. PR #50429 is directionally correct, but the provided review context still shows unresolved correctness risks around multi-chunk duplicate delivery and `block` preview fallback behavior.

## Evidence

- **Current main has no card-to-text fallback:** `deliver()` sends card replies through `sendChunkedTextReply({ useCard: true, ... })` and directly awaits `sendStructuredCardFeishu(...)` with no surrounding `try/catch`, so a Feishu card send failure still aborts reply delivery instead of retrying as plain text.
  - file: [extensions/feishu/src/reply-dispatcher.ts:428](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/feishu/src/reply-dispatcher.ts#L428)
  - command: `nl -ba extensions/feishu/src/reply-dispatcher.ts | sed -n '318,470p'`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Send layer does not downgrade card failures:** `sendStructuredCardFeishu()` only builds a structured card and returns `sendCardFeishu(...)`; there is no fallback in the Feishu send helper, so dispatcher-level handling is still required on `main`.
  - file: [extensions/feishu/src/send.ts:661](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/feishu/src/send.ts#L661)
