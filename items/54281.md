## Summary

Keep open. Current `main` still appears to have the bug described in PR #54281: gateway startup and hot reload set concurrency for `cron`, `main`, and `subagent`, but not `nested`, while cron-triggered inner work is explicitly remapped onto the `nested` lane. That means this is not implemented on `main`, not stale, and not a close candidate for conservative cleanup.

## Evidence

- **Startup lane setup still omits nested concurrency:** `applyGatewayLaneConcurrency` on current `main` sets `CommandLane.Cron`, `CommandLane.Main`, and `CommandLane.Subagent`, but does not call `setCommandLaneConcurrency(CommandLane.Nested, ...)`, so the nested lane keeps the default concurrency of 1.
  - file: [src/gateway/server-lanes.ts:6](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server-lanes.ts#L6)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Hot reload path has the same omission:** Current hot-reload logic directly reapplies lane concurrency for `Cron`, `Main`, and `Subagent` only. It likewise does not update `Nested`, so changing `cron.maxConcurrentRuns` at runtime still leaves the nested bottleneck in place.
  - file: [src/gateway/server-reload-handlers.ts:221](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server-reload-handlers.ts#L221)

- **Cron work is intentionally rerouted onto the nested lane:** `resolveGlobalLane` contains an explicit rule that when the caller lane is `cron`, inner operations must run on `CommandLane.Nested` to avoid deadlock. That makes the missing nested concurrency setting load-bearing, and it matches the PR's root-cause description and discussion.
