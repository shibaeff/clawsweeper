## Summary

Keep open. Current `main` and the latest tagged release still overwrite `memoryFlushCompactionCount` with the post-increment compaction count, while the dedup gate suppresses flushes whenever that stored value equals `compactionCount`. That matches the PR's reported bug, so this is not implemented on `main` and is still actionable. The PRtags duplicate reference to #51421 is useful corroboration that maintainers are seeing the same bug report from multiple contributors.

## Evidence

- **Current main still writes the post-increment count back into `memoryFlushCompactionCount`:** `runMemoryFlushIfNeeded` initializes `memoryFlushCompactionCount` from the current entry, then after compaction calls `incrementCompactionCount(...)` and overwrites `memoryFlushCompactionCount = nextCount`. That is exactly the behavior the PR says is wrong.
