## Summary

Current `main` still appears to have the Mattermost multi-turn edit-in-place batching bug. The Mattermost path explicitly disables core block streaming, which turns off boundary flush wiring, and the remaining preview logic does not start a fresh draft post at assistant-message boundaries.

## Evidence

- **Mattermost still disables core block streaming for draft preview mode:** The Mattermost monitor passes `disableBlockStreaming: true` while relying on `onPartialReply` updates for the draft preview path.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:1742](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/extensions/mattermost/src/mattermost/monitor.ts#L1742)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **disableBlockStreaming turns block streaming off entirely:** `get-reply-directives` resolves `disableBlockStreaming: true` to `resolvedBlockStreaming = "off"`, so `blockStreamingEnabled` becomes false for this Mattermost path.
  - file: [src/auto-reply/reply/get-reply-directives.ts:438](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/auto-reply/reply/get-reply-directives.ts#L438)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Text-only block replies are not delivered when block streaming is off:** `createBlockReplyDeliveryHandler` only enqueues or directly sends text block replies when `blockStreamingEnabled` is true; with streaming off, only media-only payloads get a direct fallback.
  - file: [src/auto-reply/reply/reply-delivery.ts:151](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/auto-reply/reply/reply-delivery.ts#L151)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Boundary flush callback is not wired for the Mattermost path:** `onBlockReplyFlush` is only provided when `params.blockStreamingEnabled && blockReplyPipeline`. Because Mattermost disables block streaming, message-end/tool-start flushes are not connected for that flow.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1229](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/auto-reply/reply/agent-runner-execution.ts#L1229)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Core expects boundary flushes, but Mattermost preview does not start a new draft message:** The embedded-session tests explicitly expect `onBlockReplyFlush` before `tool_execution_start` and at `message_end`. In Mattermost, `onAssistantMessageStart` only resets `lastPartialText`; it does not force a new preview post, and `forceNewMessage` has no call sites under `extensions/mattermost/src/mattermost`.
  - file: [src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.calls-onblockreplyflush-before-tool-execution-start-preserve.test.ts:9](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.calls-onblockreplyflush-before-tool-execution-start-preserve.test.ts#L9)
  - command: `rg -n "forceNewMessage" extensions/mattermost/src/mattermost`
