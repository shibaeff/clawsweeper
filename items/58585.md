## Summary

Current `main` still preserves prior model state when `sessionTarget: "isolated"` creates a fresh cron session, and it still prefers `subagents.model` over the normal agent default. That matches the reported bug rather than resolving it.

## Evidence

- **Fresh isolated cron sessions still inherit prior session model fields:** `resolveCronSession` builds the new session entry with `...entry` and, on `isNewSession`, only clears delivery routing fields plus `sessionFile`. It does not clear `model`, `modelProvider`, `modelOverride`, or `providerOverride`.
  - file: [src/cron/isolated-agent/session.ts:68](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/cron/isolated-agent/session.ts#L68)
  - command: `nl -ba src/cron/isolated-agent/session.ts | sed -n '68,89p'`
  - sha: [3de44fe59305](https://github.com/openclaw/openclaw/commit/3de44fe593057b508d76f65abb36d58e0936ef1d)
- **Tests explicitly lock in preserving stale model overrides across `forceNew`:** Current tests expect `forceNew: true` and stale-session rollovers to preserve `modelOverride` and `providerOverride`, and earlier in the file also preserve `model` from the prior entry.
  - file: [src/cron/isolated-agent/session.test.ts:64](https://github.com/openclaw/openclaw/blob/3de44fe593057b508d76f65abb36d58e0936ef1d/src/cron/isolated-agent/session.test.ts#L64)
