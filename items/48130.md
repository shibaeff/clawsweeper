## Summary

Keep open. Current `main` and the latest release still contain the broken unquoted Windows executable-path regex in the browser extension, so this PR addresses a live bug rather than an already-fixed or obsolete report. The PR likely needs rebasing because the code moved from `src/browser` to `extensions/browser`, but there is not strong evidence to close it.

## Evidence

- **Current main still uses the buggy unquoted Windows regex:** `detectDefaultChromiumExecutableWindows()` still reads the registry command, expands env vars, and passes it to `extractWindowsExecutablePath()`, which uses `const unquoted = command.match(/([^\\s]+\\.exe)/i);`. That character class excludes backslashes, so it cannot capture a full Windows path.
  - file: [extensions/browser/src/browser/chrome.executables.ts:299](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/extensions/browser/src/browser/chrome.executables.ts#L299)

  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Reasonable reproduction still fails on current main behavior:** Evaluating the current regexes against representative Windows registry command strings shows the failure persists: an unquoted command like `C:\Program Files\Google\Chrome\Application\chrome.exe --single-argument %1` does not yield the full executable path; the current unquoted regex only matches `chrome.exe` because it stops at backslashes.
  - file: [extensions/browser/src/browser/chrome.executables.ts:451](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/extensions/browser/src/browser/chrome.executables.ts#L451)
