## Summary

Current `main` changes the default npm install path in the way this report was failing: the one-line installer still funnels into global `npm install -g openclaw`, but commit `9895ecead3e5fad179c11a7a741f720fd59bea10` removed `node-llama-cpp` from published package metadata, updated docs to treat it as an operator-installed optional runtime, and added release checks/tests to prevent it from being reintroduced. That is a strong match for an EC2 install dying during the global npm step.

## Evidence

- **Installer and raw npm use the same failing path:** `run_npm_global_install()` in `scripts/install.sh` shells out to `npm --loglevel ... install -g "$spec"`. The issue body failed inside that step, and the reporter's follow-up comment says plain `npm i -g openclaw` was also immediately `Killed`, so both reproductions converge on the same global npm install path.
  - file: [scripts/install.sh:680](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/scripts/install.sh#L680)
  - command: `nl -ba scripts/install.sh | sed -n '675,704p'`
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Fix commit removed default llama runtime install:** Commit `9895ecead3e5fad179c11a7a741f720fd59bea10` deletes the `node-llama-cpp` peer dependency block from `package.json`, so current `main` no longer asks npm to bring that native local-embeddings runtime into a default `openclaw` install.
  - file: [package.json](https://github.com/openclaw/openclaw/blob/9895ecead3e5fad179c11a7a741f720fd59bea10/package.json)

  - file: [scripts/openclaw-npm-release-check.ts:272](https://github.com/openclaw/openclaw/blob/9895ecead3e5fad179c11a7a741f720fd59bea10/scripts/openclaw-npm-release-check.ts#L272)

  - file: [test/openclaw-npm-release-check.test.ts:526](https://github.com/openclaw/openclaw/blob/9895ecead3e5fad179c11a7a741f720fd59bea10/test/openclaw-npm-release-check.test.ts#L526)
