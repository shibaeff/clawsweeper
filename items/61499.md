## Summary

Current `main` still routes `openai-completions` runs through OpenClaw's boundary-aware transport, and that transport still recognizes only `function_call` and `tool_calls`. A backend that returns `finish_reason: "tool_call"` would still fall into the generic `Provider finish_reason: ...` error path, so this is not implemented/fixed on `main` and should stay open.

## Evidence

- **Current mapper still misses singular `tool_call`:** `mapStopReason()` in the OpenAI completions transport maps `function_call` and `tool_calls` to `toolUse`, then falls through to the generic error path for any other finish reason. `tool_call` is not handled.
