## Summary

Current `main` still lacks the PR's abort-aware queue behavior. Cron timeouts create and propagate an `AbortSignal`, but `runEmbeddedPiAgent()` does not thread that signal into `enqueueCommandInLane()`, and `enqueueCommandInLane()` itself has no `signal` option or queued-entry abort removal. The report remains plausible on current code, so this PR should stay open for maintainer review rather than auto-close.

## Evidence

- **Queue API on main is not abort-aware:** `enqueueCommandInLane()` on current `main` only accepts `warnAfterMs` and `onWait`; it pushes queued entries directly and provides no `AbortSignal` handling for entries that have not dequeued yet.
  - file: [src/process/command-queue.ts:242](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/process/command-queue.ts#L242)

- **Embedded runner still enqueues without passing abort:** `runEmbeddedPiAgent()` computes `enqueueGlobal` and `enqueueSession` from `enqueueCommandInLane(globalLane, task, opts)` and `enqueueCommandInLane(sessionLane, task, opts)`. Although `params.abortSignal` exists later in the function, it is not threaded into either queue call on current `main`.
