## Summary

Keep open. Current `main` still lacks the PR's core fix: queued command-lane entries are not abort-aware, while cron timeouts still create and thread an `AbortSignal` into isolated runs. That means the reported stale queued-entry failure mode is still plausible on `main`, so there is not strong evidence to close this PR as already implemented or obsolete.

## Evidence

- **Queue API still has no abort support:** On current `main`, `enqueueCommandInLane()` only accepts `warnAfterMs` and `onWait`, then unconditionally pushes the entry into the lane queue and drains it. There is no `signal` option, no early-abort rejection, and no queued-entry removal path.

- **Embedded runner does not thread abort into queueing:** `runEmbeddedPiAgent()` still builds `enqueueGlobal` and `enqueueSession` as plain wrappers around `enqueueCommandInLane(..., opts)` without adding `params.abortSignal`, so a caller timeout cannot remove a still-queued entry.
