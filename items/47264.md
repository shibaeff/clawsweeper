## Summary

Keep open. Current `main` still has the reported LINE multi-account bug: `accounts.default` is ignored during account resolution, named accounts without explicit `enabled: true` resolve disabled, and gateway startup then fails before registering webhook routes. The latest release tag `v2026.4.22` contains the same resolver logic, so this is not implemented on main or in the latest release.

## Evidence

- **`accounts.default` is ignored:** `resolveLineAccount` only loads `accountConfig` for non-default IDs, so credentials stored under `channels.line.accounts.default` are never read for the default account.
  - file: [extensions/line/src/accounts.ts:100](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/line/src/accounts.ts#L100)
- **Configured named accounts default to disabled:** For named accounts, `enabled` falls back to `false` unless the per-account config sets `enabled`. That means a configured account like `twgreen` in the report will not start unless explicitly enabled.
  - file: [extensions/line/src/accounts.ts:128](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/line/src/accounts.ts#L128)
- **Gateway startup fails before webhook registration when credentials are missing:** `startAccount` throws if the resolved account has an empty token or secret. Because `accounts.default` is ignored, the default LINE account remains unconfigured and the webhook route never registers.
  - file: [extensions/line/src/gateway.ts:21](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/line/src/gateway.ts#L21)
- **Account enumeration still includes `default` from the `accounts` map:** `listLineAccountIds` adds every key from `channels.line.accounts`, including `default`, so the gateway will try to start a `default` account that `resolveLineAccount` cannot actually resolve from `accounts.default`.
  - file: [extensions/line/src/accounts.ts:159](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/extensions/line/src/accounts.ts#L159)
- **Docs still advertise multi-account config without per-account `enabled`:** The LINE docs show a multi-account example with only credentials and `webhookPath`, no `enabled: true`, which conflicts with the current resolver behavior for named accounts.
  - file: [docs/channels/line.md:95](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/docs/channels/line.md#L95)
- **Latest release still contains the same broken resolver logic:** Inspecting the release tag shows the same `resolveLineAccount` implementation that ignores `accounts.default` and defaults named accounts to disabled.
  - file: [extensions/line/src/accounts.ts:100](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/line/src/accounts.ts#L100)
  - command: `git show v2026.4.22:extensions/line/src/accounts.ts | nl -ba | sed -n '92,130p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Risks / Open Questions

- I did not run a live gateway reproduction because this review had to remain read-only, but the static code path is sufficient to explain the reported 404 and doctor symptoms.
- There is also a docs/runtime mismatch for named LINE accounts: the docs imply configured named accounts should work without explicit `enabled: true`, but current code does not.
