## Summary

Keep open. Current `main` still routes non-idempotent Telegram sends through the strict retry predicate only, so the PR's proposed regex-only `421` addition in `src/infra/retry-policy.ts` would not fix the user-visible send path it targets. Main also has no `421` coverage in the Telegram send-specific retry predicates/tests.

## Evidence

- **Strict mode bypasses fallback regex:** `resolveChannelApiShouldRetry` returns the custom predicate unchanged when `strictShouldRetry` is true, so edits to `CHANNEL_API_RETRY_RE` do not affect strict callers.
  - file: [src/infra/retry-policy.ts:24](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/src/infra/retry-policy.ts#L24)
  - command: `nl -ba src/infra/retry-policy.ts | sed -n '1,140p'`
  - sha: [151befb90b90](https://github.com/openclaw/openclaw/commit/151befb90b903f52dce9f4951991d39b9df8b358)
- **Telegram send path uses strict retry mode:** Telegram non-idempotent outbound sends are created with `shouldRetry: (err) => isSafeToRetrySendError(err) || isTelegramRateLimitError(err)` and `strictShouldRetry: true`, so they never consult the fallback retry regex.
  - file: [extensions/telegram/src/send.ts:606](https://github.com/openclaw/openclaw/blob/151befb90b903f52dce9f4951991d39b9df8b358/extensions/telegram/src/send.ts#L606)
