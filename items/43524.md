## Summary

Current `main` still appears to have the reported gap. Implicit daily/scheduled session rollover creates a new session without setting `resetTriggered`, while the fallback hook emission path and the bundled `session-memory` hook still only run for explicit `/new` or `/reset` command flows.

## Evidence

- **Implicit stale-session rollover is not treated as a reset command:** `initSessionState()` archives the previous session on stale daily/scheduled rollover, but `resetTriggered` remains false and the new-session branch only preserves reset-specific behavior when `resetTriggered` is true.
  - file: [src/auto-reply/reply/session.ts:477](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/auto-reply/reply/session.ts#L477)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '470,535p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Reset-hook fallback still only fires for explicit slash commands:** `maybeEmitMissingResetHooks()` returns unless `resetTriggered` is true, then further requires `command.commandBodyNormalized` to match `^/(new|reset)`. That excludes ordinary messages arriving after a stale daily reset boundary.
