## Summary

Current `main` already fixes the reported auth-profile/custom-model failure by ensuring generated `models.json` persists non-secret `apiKey` markers for auth-profile- or SecretRef-backed providers instead of leaving providers without `apiKey`, and the fix was shipped in `v2026.4.14`.

## Evidence

- **Missing provider apiKey is backfilled before writing models.json:** `normalizeProviders()` detects providers with models but no configured `apiKey`, resolves credentials from auth profiles, and passes the result into `resolveMissingProviderApiKey()` before `models.json` is written.
  - file: [src/agents/models-config.providers.normalize.ts:92](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/models-config.providers.normalize.ts#L92)
  - command: `nl -ba src/agents/models-config.providers.normalize.ts | sed -n '92,110p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Auth-profile credentials become persisted markers, not blank apiKey fields:** `resolveMissingProviderApiKey()` writes either the env-var name or the resolved profile marker into the provider config when models exist and `apiKey` is otherwise missing, which prevents Pi `ModelRegistry` from seeing a provider with models and no `apiKey`.
  - file: [src/agents/models-config.providers.secret-helpers.ts:271](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/models-config.providers.secret-helpers.ts#L271)
