## Summary

Keep open. Current `main` still lacks the requested cleanup, and the PR discussion surfaced unresolved correctness gaps in the proposed fix.

## Evidence

- **Current `cron.remove` only removes the job record:** On `main` at `0bef73d151bd0a8d547b28bf16c3aa2e21078e0f`, `remove()` filters the job list, persists, re-arms the timer, and emits `removed`; it does not delete cron run logs or purge session-store entries.
  - file: [src/cron/service/ops.ts:333](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/cron/service/ops.ts#L333)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '330,365p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Existing reaper does not handle deleted-job base sessions:** `session-reaper.ts` documents that the base session (`cron:<jobId>`) is kept, and `sweepCronRunSessions()` only deletes keys matching cron run sessions. That means current `main` does not address the ghost base-session symptom described in the PR.
  - file: [src/cron/session-reaper.ts:1](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/cron/session-reaper.ts#L1)
  - command: `nl -ba src/cron/session-reaper.ts | sed -n '1,140p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Main code already uses `sessionStorePath` fallback elsewhere:** The cron timer reaper resolves store paths via `resolveSessionStorePath` when available, but explicitly falls back to `sessionStorePath` otherwise. That matches the PR review concern that the proposed `cron.remove` cleanup path was incomplete if it only used `resolveSessionStorePath`.
  - file: [src/cron/service/timer.ts:820](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/cron/service/timer.ts#L820)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '818,846p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **Docs still describe retention-only cleanup for cron artifacts:** The session-management docs say cron run sessions are pruned by `cron.sessionRetention` and run logs are pruned by size/line retention settings. There is no current documented behavior that deleting a job removes its session artifacts immediately.
  - file: [docs/reference/session-management-compaction.md:97](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/docs/reference/session-management-compaction.md#L97)
  - command: `nl -ba docs/reference/session-management-compaction.md | sed -n '97,112p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **PR discussion found unresolved issues in the proposed patch:** The supplied review comments flagged at least three non-trivial issues in the PR itself: missing dedup protection when collecting purged sessions, skipping cleanup when only `sessionStorePath` is configured, and detached async cleanup that can return before sessions/logs are actually removed. No follow-up commit addressing those comments is present in the provided PR timeline.
  - sha: [9717195e4e89](https://github.com/openclaw/openclaw/commit/9717195e4e8980347ef3cbdbd3c1b9f337c58088)
