## Summary

Keep open. Current `main` still defaults persisted auto-generated gateway auth to `"token"`, and the proposed `"none"` default conflicts with shipped docs/security guidance while creating a real regression for daemon installs on non-loopback binds when `gateway.auth.mode` is unset.

## Evidence

- **Current main persists token mode, not none:** `maybePersistAutoGeneratedGatewayInstallToken()` still writes `mode: baseConfig.gateway?.auth?.mode ?? "token"` when saving an auto-generated token.
  - file: [src/commands/gateway-install-token.ts:74](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/commands/gateway-install-token.ts#L74)
  - command: `nl -ba src/commands/gateway-install-token.ts | sed -n '1,180p'`

- **Daemon install always takes the persist-generated-token path:** `openclaw gateway install` calls `resolveGatewayInstallToken()` with `autoGenerateWhenMissing: true` and `persistGeneratedToken: true`, so this PR would affect normal install flows, not just a niche helper path.
  - file: [src/cli/daemon-cli/install.ts:136](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/cli/daemon-cli/install.ts#L136)

## Risks / Open Questions

- Merging this PR as written would let daemon install persist `gateway.auth.mode: "none"` on installs where token auth is still required, including non-loopback deployments with unset auth mode.
- It would also put current code out of sync with shipped docs, FAQ, and security-audit expectations that shared-secret auth is the default, even on loopback.
- There may still be a valid UX problem for purely local loopback setups, but it likely needs a narrower fix that distinguishes safe local-only onboarding from broader install flows.
