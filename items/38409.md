## Summary

Current `main` already carries the Slack DM routing/threading path that keeps message-tool sends inside the active DM thread by using the concrete DM channel id (`D…`) for live replies and tool-context inference.

## Close Comment

Closing from Codex review: current `main` already has the DM routing/threading path needed here, and the latest shipped code I checked (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, release `v2026.4.22`) includes it too. The relevant pieces are `extensions/slack/src/monitor/message-handler/prepare.ts:761`, which routes live Slack DM replies to the concrete `channel:D…` target while keeping persisted session metadata user-scoped, `extensions/slack/src/threading-tool-context.ts:26`, which falls back to `NativeChannelId` for DM tool context, and `src/infra/outbound/message-action-normalization.ts:42`, which infers the message-tool target from that current channel id. There is also direct test coverage for the DM reply target and DM `NativeChannelId` fallback in `extensions/slack/src/monitor/message-handler/prepare.test.ts:371` and `extensions/slack/src/threading-tool-context.test.ts:157`. On that basis, this looks implemented on `main` rather than still open work. If you can still reproduce it on a current build, please open a fresh issue with the exact `message` tool payload and whether the send used an explicit `target` such as `user:...` versus the inferred current-chat target.
