## Summary

Keep open. The issue comments narrowed the original report to a macOS background-auth/keychain problem, but current `main` still does not show a strong end-to-end resolution for isolated cron runs: auth-profile/keychain failures are still low-signal at runtime, the manual doctor path is only a diagnostic, and the cron docs still do not document the macOS headless keychain requirement/workaround.

## Evidence

- **Isolated cron runtime stays non-interactive for auth:** Embedded cron runs resolve the auth store with `allowKeychainPrompt: false`, so background isolated runs still cannot rely on interactive keychain prompts.
  - file: [src/agents/pi-embedded-runner/run.ts:409](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/agents/pi-embedded-runner/run.ts#L409)
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Auth-profile failures remain debug-only in provider auth resolution:** `resolveApiKeyForProvider` and related auth checks catch profile lookup failures and only emit `log.debug`, not warn/error, which matches the reporter complaint that the real keychain failure is easy to miss outside debug logs.
  - file: [src/agents/model-auth.ts:535](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/agents/model-auth.ts#L535)
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)

  - file: [src/commands/doctor-claude-cli.ts:246](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/commands/doctor-claude-cli.ts#L246)
