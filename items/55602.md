## Summary

Keep open. Current `main` does not implement `%DATE%` expansion for `logging.file`, and the PR discussion records unresolved correctness concerns around retention/pruning and config-load behavior, so there is not strong evidence for any conservative auto-close reason.

## Evidence

- **Current main uses configured log path verbatim:** `resolveSettings()` reads `cfg?.file` directly and only falls back to the default rolling path when `logging.file` is unset; there is no `%DATE%` placeholder expansion on current `main`.
  - file: [src/logging/logger.ts:376](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/logging/logger.ts#L376)
  - command: `nl -ba src/logging/logger.ts | sed -n '320,400p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Configured-path helper also returns the literal path:** `resolveConfiguredLogFilePath()` returns `config?.logging?.file` unchanged, which matches the absence of placeholder support elsewhere in main.
  - file: [src/logging/log-file-path.ts:39](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/logging/log-file-path.ts#L39)
  - command: `nl -ba src/logging/log-file-path.ts | sed -n '1,120p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Rolling/pruning logic is still hard-coded to OpenClaw's default dated basename:** File pruning only runs when the basename matches `openclaw-YYYY-MM-DD.log`, and deletion only targets files starting with `openclaw-` and ending in `.log`. That means the requested custom dated-path behavior is not present on `main`.
  - file: [src/logging/logger.ts:417](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/logging/logger.ts#L417)
  - command: `nl -ba src/logging/logger.ts | sed -n '400,620p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Docs still describe `logging.file` as a stable path override:** The public-facing docs say the default log file is `openclaw-YYYY-MM-DD.log` and tell users to set `logging.file` for a stable path, which is consistent with current behavior and inconsistent with `%DATE%` support already having shipped.
  - file: [docs/gateway/configuration-reference.md:846](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/gateway/configuration-reference.md#L846)
  - command: `nl -ba docs/gateway/configuration-reference.md | sed -n '830,865p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **PR discussion shows unresolved correctness risks in the proposed patch:** The review thread included multiple P1/P2 comments after the initial implementation: custom pruning broadening could delete unrelated dated logs in shared directories, `%DATE%` at the start or middle of a basename breaks prune matching, and fallback config loading needed extra guards for `config schema`. Those are substantive maintainer-review issues, not evidence for auto-closing the PR.
