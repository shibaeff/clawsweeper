## Summary

Current `main` intentionally routes MiniMax image understanding through `MiniMax-VL-01` and keeps `MiniMax-M2.7` as a text-only chat model in the API-key/onboarding path, so the reported `Model does not support images: minimax/MiniMax-M2.7` failure path is addressed on `main`. This appears implemented after the latest shipped release, so it is fixed on `main` but not yet attributable to `v2026.4.23`.

## Evidence

- **Image tool now pairs MiniMax primaries with the dedicated vision model:** `resolveImageModelConfigForTool()` falls back to `minimax/MiniMax-VL-01` (or `minimax-portal/MiniMax-VL-01`) whenever the primary provider is MiniMax, instead of trying to use the chat model as the image model.
  - file: [src/agents/tools/image-tool.ts:126](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/agents/tools/image-tool.ts#L126)
  - command: `nl -ba src/agents/tools/image-tool.ts | sed -n '111,167p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **MiniMax API-key model definitions are explicitly text-only:** `buildMinimaxModelDefinition()` now hard-codes `input: ["text"]` for MiniMax chat models, which prevents `MiniMax-M2.7` from being treated as an image-capable tool model in onboarded/configured API-key setups.
  - file: [extensions/minimax/model-definitions.ts:64](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/minimax/model-definitions.ts#L64)

- **Regression tests cover the reported mismatch:** Tests verify that a MiniMax primary model pairs with `MiniMax-VL-01`, and that configured `MiniMax-M2.7` chat metadata is not reused as the image model.
