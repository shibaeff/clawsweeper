## Summary

Current main does not match the reported failure mode: `chat.send` requires and echoes a non-empty run id, terminal failures are surfaced through gateway dedupe/`agent.wait`, and the log-size cap path only suppresses file writes.

## Evidence

- **Protocol requires non-empty run identifiers:** `ChatSendParamsSchema` requires a non-empty `idempotencyKey`, and chat event schemas use `NonEmptyString` for `runId`, so a WebSocket `chat.send` success payload with `runId: ""` is not valid current-main gateway protocol.
  - file: [src/gateway/protocol/schema/logs-chat.ts:35](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/gateway/protocol/schema/logs-chat.ts#L35)
  - command: `nl -ba src/gateway/protocol/schema/logs-chat.ts | sed -n '35,73p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **`chat.send` acks with the caller's idempotency key:** The handler sets `clientRunId = p.idempotencyKey` and immediately responds with `{ runId: clientRunId, status: "started" }`, so the reported empty `runId` response does not match current main after validation succeeds.
  - file: [src/gateway/server-methods/chat.ts:2150](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/gateway/server-methods/chat.ts#L2150)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '2145,2260p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Terminal failures are observable after the initial ack:** Async `chat.send` failures are recorded as `status: "error"` in gateway dedupe, and `agent.wait` reads terminal snapshots from `chat:<runId>` / `agent:<runId>` entries, so current main provides a follow-up failure signal for a started run.
  - file: [src/gateway/server-methods/agent-wait-dedupe.ts:65](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/gateway/server-methods/agent-wait-dedupe.ts#L65)
  - command: `nl -ba src/gateway/server-methods/agent-wait-dedupe.ts | sed -n '65,120p'; nl -ba src/gateway/server-methods/chat.ts | sed -n '2712,2744p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Log-cap path only suppresses file writes:** When the log file exceeds `maxFileBytes`, the logger emits one warning and returns early from the transport; the path is wrapped in `try/catch` with `// never block on logging failures`, and there is no gateway spawn/error hook here.
  - file: [src/logging/logger.ts:183](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/logging/logger.ts#L183)
  - command: `nl -ba src/logging/logger.ts | sed -n '176,205p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Docs describe `chat.send` as non-blocking with follow-up status lookup:** The user-facing docs say `chat.send` immediately acks with `{ runId, status: "started" }`, and the gateway protocol docs point callers to `agent.wait` for the terminal snapshot, matching the current code path instead of the reported silent-success-without-run-id behavior.
  - file: [docs/web/control-ui.md:152](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/docs/web/control-ui.md#L152)
