## Summary

Keep open. Current `main` still unconditionally remaps cron work onto the nested lane, so this PR is addressing a live bug rather than obsolete or already-landed behavior.

## Evidence

- **Current lane resolver still forces cron -> nested:** `resolveGlobalLane()` on current `main` trims the lane and unconditionally returns `CommandLane.Nested` when the cleaned value is `CommandLane.Cron`; there is no top-level vs inner-operation distinction yet.
  - file: [src/agents/pi-embedded-runner/lanes.ts:8](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/pi-embedded-runner/lanes.ts#L8)
  - command: `nl -ba src/agents/pi-embedded-runner/lanes.ts | sed -n '1,220p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Top-level dispatch still uses the unconditional remap:** `runEmbeddedPiAgent()` resolves `globalLane` with `resolveGlobalLane(params.lane)` before queueing the run, so top-level cron dispatch is still routed through the remapped lane on current `main`. `compactEmbeddedPiSession()` does the same, which matches the PR's claim that both call sites need different handling.
  - file: [src/agents/pi-embedded-runner/run.ts:251](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/pi-embedded-runner/run.ts#L251)

- **Compaction path also lacks an inner-only flag:** The queued compaction entrypoint still calls `resolveGlobalLane(params.lane)` with no options, so current `main` has no way to preserve cron for top-level dispatch while still remapping nested follow-up work.

The public config docs also advertise `cron.maxConcurrentRuns`. ## Risks / Open Questions

- I did not run tests in this read-only review, and `pnpm` is not available in this environment, so this decision is based on code/docs inspection rather than an executed reproduction.
- Changing top-level cron dispatch back to `CommandLane.Cron` should still be checked against the older deadlock scenario referenced in `src/agents/pi-embedded-runner/lanes.test.ts:16`; the PR's proposed `inner` split appears to address that, but maintainers should confirm the follow-up/compaction paths are the only nested cases that require remapping.
