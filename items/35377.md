## Summary

Current `main` already implements inbound audio/voice-message transcription through the shared `tools.media.audio` pipeline, including provider/CLI STT options, transcript injection into reply context, and voice-note handling in channel flows such as Feishu and Telegram.

## Risks / Open Questions

- The issue text names iFlytek specifically, but the implemented feature is the generic STT pipeline rather than a bundled iFlytek provider. A follow-up provider-specific request could still be valid if iFlytek support is desired.
- This review was read-only and `pnpm` was unavailable in the checkout, so the conclusion is based on current code, docs, tests, and release-tag inspection rather than a live channel reproduction.

## Close Comment

Closing as already implemented. Codex review of current `main` found that inbound voice/audio transcription is already part of OpenClaw’s shared media pipeline: `docs/nodes/audio.md:12` describes automatic audio transcription and transcript injection, `docs/nodes/audio.md:163` documents preflight transcription for mention-gated voice notes, and `src/media-understanding/audio-preflight.ts:10` contains the corresponding implementation. The Feishu channel already routes inbound `audio` messages into that pipeline (`extensions/feishu/src/bot-content.ts:307` and `extensions/feishu/src/bot-content.ts:329`), and Telegram has regression coverage showing DM voice notes become transcribed body text (`extensions/telegram/src/bot-message-context.body.test.ts:149`). I also checked the shipped `v2026.4.22` tag, and the same audio-transcription docs/code are present there. If you still want a specific additional provider such as iFlytek, please open a narrower follow-up for that provider integration rather than the already-shipped STT feature itself.
