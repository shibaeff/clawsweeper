## Summary

Keep open. Current `main` still injects the reported volatile Discord thread context on follow-up turns, so the issue is not implemented-on-main or obsolete.

## Evidence

- **Discord inbound processing still records volatile thread/message fields every turn:** `processDiscordMessage` populates `InboundHistory`, `MessageSid`, `ReplyToId`, `ThreadStarterBody`, and `Timestamp` on every inbound Discord message before finalizing the context payload.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:477](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/extensions/discord/src/monitor/message-handler.process.ts#L477)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.process.ts | sed -n '360,530p'`
  - sha: [aa21d4ea7e9a](https://github.com/openclaw/openclaw/commit/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54)
- **Prompt context builder still serializes the reported metadata:** `buildInboundUserContextPrefix` emits `message_id`, `reply_to_id`, `timestamp`, `Thread starter (untrusted, for context)`, and `Chat history since last reply (untrusted, for context)` blocks from the inbound context.
  - file: [src/auto-reply/reply/inbound-meta.ts:191](https://github.com/openclaw/openclaw/blob/aa21d4ea7e9aab034acbf0b13f8c3575057c5c54/src/auto-reply/reply/inbound-meta.ts#L191)
