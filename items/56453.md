## Summary

Keep open: current `main` still derives implicit `openclaw agent --to ...` sessions through `main` when `--agent` is omitted, and the current tests still lock in `agent:main:main` for that path. I found no evidence that PR #56453's fix has already landed on `main`.

## Evidence

- **Current session-key helper still hardcodes `DEFAULT_AGENT_ID`:** `resolveSessionKey()` only accepts `(scope, ctx, mainKey?)` and builds both direct and group implicit keys with `DEFAULT_AGENT_ID`, so non-`main` configured defaults are not threaded into implicit `--to` session derivation.
  - file: [src/config/sessions/session-key.ts:29](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/session-key.ts#L29)
  - command: `nl -ba src/config/sessions/session-key.ts | sed -n '1,220p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current request path still selects the store before any default-agent-aware derivation:** `resolveSessionKeyForRequest()` computes `storeAgentId` from `explicitSessionKey` or `requestedAgentId ?? normalizeAgentId(undefined)`, then loads that store before deriving the `--to` session key. With omitted `--agent`, that falls back to `main` and then calls `resolveSessionKey(scope, ctx, mainKey)` without a default-agent override.
  - file: [src/agents/command/session.ts:145](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/command/session.ts#L145)
