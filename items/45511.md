## Summary

Current main already has the WhatsApp outbound-listener fix, with regression coverage and a shipped changelog entry matching this failure mode.

## Evidence

- **Process-global listener registry:** WhatsApp connection controllers are stored on `globalThis[Symbol.for("openclaw.whatsapp.connectionControllerRegistry")]`, which makes the active listener shared across split runtime chunks instead of module-local.
  - file: [extensions/whatsapp/src/connection-controller-registry.ts:11](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/whatsapp/src/connection-controller-registry.ts#L11)
- **Outbound send resolves the live listener:** WhatsApp outbound send resolves the controller at call time via `getRegisteredWhatsAppConnectionController(...).getActiveListener()` and only throws `No active WhatsApp Web listener` when that shared registry is actually empty.
  - file: [extensions/whatsapp/src/send.ts:39](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/whatsapp/src/send.ts#L39)
- **Duplicate-module regression test:** Regression test imports the registry module twice with separate cache-busting URLs and verifies both module instances see the same registered controller.
  - file: [extensions/whatsapp/src/connection-controller-registry.test.ts:11](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/whatsapp/src/connection-controller-registry.test.ts#L11)
- **Light/heavy runtime boundary regression test:** Regression test stages separate bundled light and heavy WhatsApp runtime modules and verifies `light.getActiveWebListener(accountId)` sees the listener registered by the heavy runtime path.
  - file: [src/plugins/runtime-plugin-boundary.whatsapp.test.ts:125](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/plugins/runtime-plugin-boundary.whatsapp.test.ts#L125)
- **Shipped release note:** The changelog entry under `2026.3.23` says WhatsApp outbound sends now keep the active Web listener on a direct process-global symbol so `openclaw message send --channel whatsapp` stops failing after connect.
  - file: [CHANGELOG.md:2140](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/CHANGELOG.md#L2140)
- **Exact fix note in changelog:** The specific fix entry is recorded in the 2026.3.23 release notes.
  - file: [CHANGELOG.md:2197](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/CHANGELOG.md#L2197)

## Close Comment

Closing as implemented on `main` after a Codex review. The current code keeps the WhatsApp connection-controller registry on `globalThis[Symbol.for("openclaw.whatsapp.connectionControllerRegistry")]` in `extensions/whatsapp/src/connection-controller-registry.ts`, and outbound WhatsApp sends resolve the active listener from that shared registry at send time in `extensions/whatsapp/src/send.ts`. There is dedicated regression coverage for duplicate module instances in `extensions/whatsapp/src/connection-controller-registry.test.ts` and for the light/heavy bundled runtime split in `src/plugins/runtime-plugin-boundary.whatsapp.test.ts`. The shipped changelog entry for `2026.3.23` in `CHANGELOG.md` also matches this exact failure mode: it says `openclaw message send --channel whatsapp` stops failing after connect because the active Web listener is kept on a process-global symbol. If you still hit this on a current release, please open a fresh issue with the exact OpenClaw version, install method, and current gateway/channel logs.
