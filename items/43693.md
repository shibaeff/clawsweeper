## Summary

Model Hub support is a reasonable request, but current `main` explicitly treats provider-specific behavior as plugin-owned and already exposes plugin-defined onboarding, auth env-var, and dynamic catalog hooks. This PR is better closed as a ClawHub/provider-plugin contribution than kept open as a core-repo provider addition.

## Evidence

- **Plugin-first project scope:** The repo vision says core should stay lean, optional capability should usually ship as plugins, and plugin promotion belongs in ClawHub; the bar for adding optional plugins to core is intentionally high.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/VISION.md#L54)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Providers are plugin-owned:** The model-provider docs say most provider-specific logic lives in provider plugins, including onboarding, model catalogs, auth env-var mapping, transport/config normalization, failover classification, and related behavior.
  - file: [docs/concepts/model-providers.md:37](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/concepts/model-providers.md#L37)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Public provider-plugin guide covers this exact feature shape:** The provider-plugin guide explicitly documents building a model provider with a model catalog, API key auth, manifest auth/env-var metadata, CLI flag wiring, and dynamic model resolution, which matches the Model Hub proposal without needing core changes.
  - file: [docs/plugins/sdk-provider-plugins.md:11](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/docs/plugins/sdk-provider-plugins.md#L11)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Onboarding choices already come from provider contributions:** Current main builds auth-choice options and CLI values from `resolveProviderSetupFlowContributions(...)`, so provider onboarding surfaces are already plugin/setup driven rather than hard-coded per provider in core.
  - file: [src/commands/auth-choice-options.ts:31](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/commands/auth-choice-options.ts#L31)
  - sha: [9577de2da7df](https://github.com/openclaw/openclaw/commit/9577de2da7dfb6234cf30e3b879940067804e26e)
- **Provider env-var detection already comes from plugin metadata:** Current main collects provider auth env vars from plugin manifests and setup metadata before merging core fallbacks, so a provider plugin can expose `MODEL_HUB_API_KEY` without adding new core env-var tables.
  - file: [src/secrets/provider-env-vars.ts:85](https://github.com/openclaw/openclaw/blob/9577de2da7dfb6234cf30e3b879940067804e26e/src/secrets/provider-env-vars.ts#L85)
