## Summary

Keep open. Current `main` does not contain an equivalent resume-loop breaker, and the reported Discord 1005/1006 reconnect-loop remains plausible in the refactored lifecycle. The PR itself also has an unresolved correctness concern from review, so there is no strong basis for conservative auto-close.

## Evidence

- **Current lifecycle still clears the READY watchdog on every close:** `createGatewayStatusObserver()` starts a 30s READY watch after `Gateway websocket opened`, but `Gateway websocket closed` immediately clears that watch and only records disconnected status. In a rapid open->1005->close loop, current main has no resume-state reset or alternate breaker in this path.

  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **Lifecycle stop still depends on Carbon reporting reconnect exhaustion:** `classifyDiscordGatewayEvent()` only turns reconnect failures into a lifecycle-stopping `reconnect-exhausted` event when the error text already contains `Max reconnect attempts`. There is no OpenClaw-side handling here for repeated pre-READY 1005/1006 resume failures that never trip Carbon's counter.
