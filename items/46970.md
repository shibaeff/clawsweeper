## Summary

Current `main` already implements MCP/plugin tool approval gating through the existing plugin-approval flow, including gateway request/wait/resolve RPCs, chat/native approval resolution, MCP bridge enforcement, and Codex harness routing. This PR's separate `approvals.toolPolicy` / `tool.approval.*` stack is redundant against the implementation already present on `main`.

## Close Comment

Closing based on a Codex review of current `main`. The underlying request is already covered by the plugin-approval path on `main`: tool calls can be paused for approval in `src/agents/pi-tools.before-tool-call.ts:300`, the gateway already exposes `plugin.approval.request` / `plugin.approval.waitDecision` / `plugin.approval.resolve` in `src/gateway/server-methods/plugin-approval.ts:26`, the MCP bridge applies that same boundary in `src/mcp/plugin-tools-handlers.ts:21`, and there is an MCP-specific regression test in `src/mcp/plugin-tools-serve.test.ts:92`. Current docs also describe the shipped flow, including `approvals.plugin` forwarding and Codex MCP approval routing in `docs/plugins/building-plugins.md:186`, `docs/tools/exec-approvals-advanced.md:207`, and `docs/plugins/codex-harness.md:551`. Because `main` already has working MCP/plugin approval gating through the existing plugin approval system, this parallel `tool.approval.*` / `approvals.toolPolicy` implementation is no longer needed as a separate open PR.
