## Summary

Issue #50391 proposes a new compile-and-execute/FSM core architecture, but current main scope explicitly pushes optional execution models into plugins/ClawHub and rejects new heavy orchestration layers in core. OpenClaw already exposes plugin seams for alternate low-level executors through agent harness plugins, so this belongs as an external plugin/harness proposal rather than a core issue.

## Risks / Open Questions

- The agent-harness surface is documented as experimental and intended for bundled or trusted native plugins, so a third-party FSM runner may still need follow-on SDK discussion.
- This review is scope-based rather than a proof that every desired enterprise control in the RFC is already achievable with today's plugin API.

## Close Comment

Closing this as `clawhub` after a Codex review of current `main`. The request is for a new core compile-and-execute/FSM architecture, but the current project scope explicitly says optional capability should usually ship as plugins and that new skills should go to ClawHub first (`VISION.md:54-70`). The same vision doc also lists “heavy orchestration layers that duplicate existing agent and tool infrastructure” as out of scope for core (`VISION.md:99-107`). Current main already exposes plugin seams for alternate executors through agent harness plugins (`docs/tools/plugin.md:11-15`, `docs/plugins/sdk-agent-harness.md:11-46`, `src/plugins/types.ts:2120-2121`), so an FSM runner fits better as a ClawHub/trusted plugin proposal than as an OpenClaw core issue. If this is still interesting, the next concrete step would be a plugin/harness design that uses those seams rather than adding a second core architecture.
