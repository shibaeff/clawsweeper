## Summary

Keep open: current `main` still accepts any non-JSON stdout verbatim for single-id exec secret providers when `jsonOnly` is false, so noisy multi-line output can still be treated as the secret value. The latest release `v2026.4.23` shows the same behavior, the current tests only cover the clean single-line case, and docs still recommend `jsonOnly: false` for common exec-provider integrations. This PR appears to address a real unshipped bug/security hardening gap, so no conservative auto-close reason applies.

## Evidence

- **Current main still trusts full non-JSON stdout:** `parseExecValues()` on current `main` tries `JSON.parse(trimmed)` for single-id exec providers with `jsonOnly: false`, and on parse failure immediately returns `{ [id]: trimmed }`. That means multi-line log noise plus a value is accepted wholesale instead of being rejected.
