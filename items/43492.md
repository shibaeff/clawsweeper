## Summary

Current `main` still appears to have the reported gap. Fallback retry strips explicit `images`/`imageOrder`, but it preserves the original prompt text, then unconditionally re-scans that prompt for image paths and sends any loaded prompt-detected images to the model. I did not find a current-main guard that suppresses prompt-detected images on cross-provider fallback retry, so this should stay open.

## Evidence

- **Fallback retry only strips explicit image payloads:** `runAgentAttempt` passes `images` and `imageOrder` as `undefined` only when `isFallbackRetry` is true, which addresses explicit attachments but nothing else.
  - file: [src/agents/command/attempt-execution.ts:332](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/command/attempt-execution.ts#L332)
  - command: `nl -ba src/agents/command/attempt-execution.ts | sed -n '300,450p'`
  - sha: [28fc03c386e1](https://github.com/openclaw/openclaw/commit/28fc03c386e14db502429bbe7e062cbe90439ca1)
- **Fallback retry keeps the original prompt body:** `resolveFallbackRetryPrompt` returns the original body on fallback retry, and with session history it prepends a retry notice to the same body instead of replacing it. Any image file paths in the user's text therefore remain present on retry.
  - file: [src/agents/command/attempt-execution.helpers.ts:104](https://github.com/openclaw/openclaw/blob/28fc03c386e14db502429bbe7e062cbe90439ca1/src/agents/command/attempt-execution.helpers.ts#L104)
