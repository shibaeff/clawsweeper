## Summary

Keep open. Current `main` still has the pre-fix TUI flow: submit clears the editor before acceptance, no active-run guard is wired into the submit handler, and `sendMessage` still sends while another run can already be active. The linked fix PR for this issue was closed unmerged.

## Evidence

- **Submit still clears input before any busy-state check:** `createEditorSubmitHandler` calls `params.editor.setText("")` immediately, then routes to `sendMessage` with no `isRunActive`/busy guard in the function signature or body.
  - file: [src/tui/tui-submit.ts:15](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/tui/tui-submit.ts#L15)
  - command: `nl -ba src/tui/tui-submit.ts | sed -n '1,80p'`

- **runTui does not pass an active-run blocker:** The TUI wires `createEditorSubmitHandler` with only `editor`, `handleCommand`, `sendMessage`, and `handleBangLine`; there is no callback to block normal submits while `activeChatRunId` is set.
  - file: [src/tui/tui.ts:953](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/tui/tui.ts#L953)

- **sendMessage still accepts submits while a run may already be active:** `sendMessage` checks only `state.isConnected`. It then adds the user message, marks `pendingOptimisticUserMessage`, and sends `chat.send`; there is no early return when `state.activeChatRunId !== null`.
