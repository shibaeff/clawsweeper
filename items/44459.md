## Summary

Current main already implements the original Telegram TTS voice-note request: `/tts audio` marks compatible synthesized audio as `audioAsVoice`, and the Telegram send path routes compatible audio with that flag through `sendVoice` rather than `sendAudio`. The implementation is present on the `v2026.4.22` release tag.

- The April 12 follow-up comment appears to broaden the scope to generic outbound `--media <mp3>` delivery. Current docs still describe plain audio as the default unless `asVoice` / `[[audio_as_voice]]` is set, so that broader UX question is separate from the original built-in TTS report.

## Close Comment

Closing from a Codex review because the original built-in Telegram TTS issue is implemented on current `main` and in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). `/tts audio` now sets `audioAsVoice` on voice-compatible syntheses (`src/auto-reply/reply/commands-tts.ts:167`), and the Telegram outbound path routes compatible audio through `sendVoice` (`extensions/telegram/src/send.ts:924`). The behavior is also covered by Telegram send tests for OGG and MP3 voice sends (`extensions/telegram/src/send.test.ts:1332`). The April 12 follow-up looks like it was testing the broader generic outbound-media surface, which still defaults to normal audio unless `asVoice` / `[[audio_as_voice]]` is requested (`docs/channels/telegram.md:525`). If that broader media-send UX still needs work, please open a fresh issue scoped to the outbound media path.
