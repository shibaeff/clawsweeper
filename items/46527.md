## Summary

Keep open. Current `main` still uses a config/probe-only health snapshot path, while `channels status` uses live runtime snapshots, and the PR discussion leaves a valid multi-account cache-staleness concern unresolved.

## Evidence

- **Health snapshot on main still ignores live runtime state:** `getHealthSnapshot` builds channel summaries from resolved config accounts plus optional probe results, then selects the top-level summary from `preferredAccountId`; there is no `runtimeSnapshot` parameter or runtime-backed builder in this path on current `main`. That means the PR's stated fix is not already implemented.
  - file: [src/commands/health.ts:263](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/commands/health.ts#L263)
  - command: `nl -ba src/commands/health.ts | sed -n '250,360p'`
  - sha: [0bef73d151bd](https://github.com/openclaw/openclaw/commit/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f)
- **`channels status` already uses runtime snapshots through a different code path:** The gateway `channels.status` handler reads `context.getRuntimeSnapshot()` and passes per-account runtime into `buildChannelAccountSnapshot`, confirming the mismatch described in the PR: `channels status` and `health` do not currently share the same runtime-backed summary path.
  - file: [src/gateway/server-methods/channels.ts:161](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/gateway/server-methods/channels.ts#L161)

- **Gateway health cache on main has no runtime-aware invalidation:** The health RPC fast path returns cached data based only on cache age, and the health-state refresh path calls `getHealthSnapshot({ probe })` with no runtime provider. The runtime-aware cache invalidation and provider wiring described in PR #46527 are absent from current `main`.
  - file: [src/gateway/server-methods/health.ts:16](https://github.com/openclaw/openclaw/blob/0bef73d151bd0a8d547b28bf16c3aa2e21078e0f/src/gateway/server-methods/health.ts#L16)

Separately, health chooses a `preferredAccountId` that can differ from the default when an agent is bound to another account.
