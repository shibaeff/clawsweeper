## Summary

Current `main` does not match the report's claimed root cause. `chat.send` uses session store updates for conversation state, ordinary runs resolve a deterministic per-agent workspace, and per-session workspace overrides are explicitly restricted to spawned `subagent:*` / `acp:*` sessions.

## Evidence

- **Docs separate session creation from chat execution:** Gateway protocol docs say `sessions.create` creates a session entry, while chat execution uses `chat.send`/`chat.history`/`chat.abort`/`chat.inject`. That does not match the report's claim that `chat.send` silently owns a separate session+workspace lifecycle.
  - file: [docs/gateway/protocol.md:342](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/docs/gateway/protocol.md#L342)
  - command: `nl -ba docs/gateway/protocol.md | sed -n '340,350p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Session model is sessionKey -> sessionId, not sessionKey -> workspace dir:** Session-management docs describe `sessionKey` as the conversation bucket and `sessionId` as the current transcript file for that bucket. The documented lifecycle is transcript/session oriented, not implicit per-session workspace allocation.
  - file: [docs/reference/session-management-compaction.md:110](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/docs/reference/session-management-compaction.md#L110)
  - command: `nl -ba docs/reference/session-management-compaction.md | sed -n '108,131p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Workspace overrides are restricted to spawned sessions only:** `spawnedWorkspaceDir` is rejected unless the store key is `subagent:*` or `acp:*`. Ordinary sessions cannot carry an arbitrary per-session workspace dir through the session store.
  - file: [src/gateway/sessions-patch.ts:134](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/gateway/sessions-patch.ts#L134)
  - command: `nl -ba src/gateway/sessions-patch.ts | sed -n '120,155p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Ordinary agent runs resolve workspace from agentId:** Workspace resolution is deterministic: default agent uses the default workspace, non-default agents use a stable `workspace-<agentId>` path. This is per-agent, not per-session or per-run.
  - file: [src/agents/agent-scope-config.ts:149](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/agent-scope-config.ts#L149)
  - command: `nl -ba src/agents/agent-scope-config.ts | sed -n '149,172p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Workspace creation happens through agent workspace setup, not chat.send-specific session scaffolding:** When an agent run starts, OpenClaw calls `ensureAgentWorkspace` with the resolved agent workspace dir. That setup path is tied to the resolved agent workspace, not an ad hoc `chat.send` session directory.
  - file: [src/agents/agent-command.ts:352](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/agent-command.ts#L352)
  - command: `nl -ba src/agents/agent-command.ts | sed -n '351,359p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Session persistence updates the session store entry only:** Session initialization persists `sessionEntry` back into the session store with `updateSessionStore(...)`; this code path does not mint a session-scoped workspace directory.
  - file: [src/auto-reply/reply/session.ts:750](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/auto-reply/reply/session.ts#L750)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '744,758p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Current chat.send rejects unknown agent session targets:** `chat.send` loads the target session, then rejects `agent:<id>:...` keys whose agent no longer exists in config. That prevents arbitrary unknown agent ids from creating new `workspace-<id>` directories via chat sends.
  - file: [src/gateway/server-methods/chat.ts:2123](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/gateway/server-methods/chat.ts#L2123)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '2123,2141p'`

## Risks / Open Questions

- This was a read-only code/docs review; I did not launch a live gateway process because the sweep instructions forbid artifact-producing repro runs.
- If the reporter observed leftover `workspace-*` dirs, they may have come from a different flow such as spawned subagents/ACP sessions or from an older/non-main build, but that is not supported by current `main` for plain `chat.send`.
