## Summary

Keep open. Current `main` still rejects empty or whitespace-only `write.content`, so this is not implemented on `main` and the PR addresses a real behavior gap in OpenClaw’s write-tool wrapper.

## Evidence

- **Current validator still rejects empty strings for `write.content`:** `REQUIRED_PARAM_GROUPS.write` on current `main` requires `content` without `allowEmpty`, and `assertRequiredParams` only satisfies string groups when `value.trim().length > 0` unless `allowEmpty` is set. That means `""`, `"\n"`, and whitespace-only strings fail validation before the write tool runs.
  - file: [src/agents/pi-tools.params.ts:75](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/pi-tools.params.ts#L75)
  - command: `nl -ba src/agents/pi-tools.params.ts | sed -n '1,140p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Existing tests on `main` encode the rejecting behavior:** The current test suite explicitly expects whitespace-only `content` to throw `Missing required parameter`, which matches the reported bug and shows the behavior is intentional in current code, not already fixed elsewhere.
  - file: [src/agents/pi-tools.params.test.ts:98](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/pi-tools.params.test.ts#L98)

Inspecting the published npm tarball for that version shows the upstream write tool uses `Type.String()` for `content` and writes `content` directly with `ops.writeFile(...)`, returning `content.length` in the success message.
