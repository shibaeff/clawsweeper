## Summary

Keep open. Current `main` still does not carry actual provider/model identity into `message_sending` delivery hooks, while `llm_output` receives that identity separately as an observation-only hook. That means the reported provenance/enforcement gap is still present rather than implemented away.

## Evidence

- **Outbound delivery hook still omits model/provider metadata:** `applyMessageSendingHook` calls `runMessageSending(...)` with `to`, `content`, `replyToId`, `threadId`, and `metadata: { channel, accountId, mediaUrls }`. There is no provider, model, or resolved runtime ref in the event passed to delivery hooks.
  - file: [src/infra/outbound/deliver.ts:620](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/infra/outbound/deliver.ts#L620)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '580,940p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Message hook contract has no model identity fields:** `PluginHookMessageContext` exposes channel/account/conversation/run/message/trace correlation, and `PluginHookMessageSendingEvent` exposes only `to`, `content`, `replyToId`, `threadId`, and opaque `metadata`. No typed provider/model/resolved-ref field exists for outbound delivery hooks.
  - file: [src/plugins/hook-message.types.ts:4](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/plugins/hook-message.types.ts#L4)

  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **LLM output hook does receive truthful runtime identity, but only as observation:** `PluginHookLlmOutputEvent` includes `provider`, `model`, and optional `resolvedRef`, and the embedded runner populates those fields when emitting `llm_output`. `runLlmOutput` is a void/observation hook, not the outbound delivery rewrite hook.
