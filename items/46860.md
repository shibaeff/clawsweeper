## Summary

Current `main` has partial restart-resume handling for session-scoped restarts, but the issue is not fully implemented for generic `openclaw gateway restart` or system reboot scenarios. Startup recovery only runs when a restart sentinel file exists, and current sentinel writers are limited to chat/tool/config/update paths rather than the service restart path described in the issue.

## Evidence

- **Startup recovery is sentinel-gated:** Gateway startup only schedules restart recovery when `hasRestartSentinel()` returns true, then imports `server-restart-sentinel` to process it.
  - file: [src/gateway/server-startup-post-attach.ts:333](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/server-startup-post-attach.ts#L333)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Sentinel processing does proactive notify and continuation:** When a sentinel exists, OpenClaw enqueues a restart notice, requests a heartbeat wake, attempts outbound delivery back to the original route, and can dispatch an `agentTurn` continuation to resume/respond after reboot.
  - file: [src/gateway/server-restart-sentinel.ts:310](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/server-restart-sentinel.ts#L310)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Chat `/restart` writes a routed sentinel:** The session restart command captures `sessionKey`, `deliveryContext`, and `threadId`, then writes a restart sentinel with a default continuation message before restarting.
  - file: [src/auto-reply/reply/commands-session.ts:45](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/auto-reply/reply/commands-session.ts#L45)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Gateway tool restart writes a routed sentinel:** The agent-facing gateway tool also writes a restart sentinel with delivery context and optional `continuationMessage`, covering session-triggered restarts from chat/tool flows.
  - file: [src/agents/tools/gateway-tool.ts:357](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/agents/tools/gateway-tool.ts#L357)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Generic service restart path does not write a sentinel:** Repository-wide search for `writeRestartSentinel(` only finds chat/tool/config/update/auto-recovery writers, not the daemon service restart command path. The CLI `restart` command delegates to `runDaemonRestart` without capturing any per-agent route to notify.
  - file: [src/cli/daemon-cli/register-service-commands.ts:114](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/cli/daemon-cli/register-service-commands.ts#L114)

## Risks / Open Questions

- The issue is narrower on current main than when filed: chat-initiated restarts appear substantially improved, so future triage should distinguish session-scoped restart recovery from generic service reboot behavior.
- A maintainer could still decide to split this into two issues: one for already-landed session-scoped restart continuation and one for operator/system reboot notifications.
