## Summary

Keep open. Current `main` still sends Claude CLI system prompts through argv on Windows and has no guard in the execution path, while the PR discussion contains a fresh shipped-release repro showing the bug is still real.

## Evidence

- **Claude backend still uses stdin for prompt but argv for system prompt:** The built-in Anthropic CLI backend on current `main` is configured with `input: "stdin"` and `systemPromptArg: "--append-system-prompt"`, so user prompt and system prompt still travel through different channels exactly as described in the PR.
  - file: [extensions/anthropic/cli-backend.ts:57](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/anthropic/cli-backend.ts#L57)
  - command: `nl -ba extensions/anthropic/cli-backend.ts | sed -n '20,90p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Current helper still appends system prompt directly into argv:** `buildCliArgs` on current `main` still pushes `params.backend.systemPromptArg` plus the full system prompt text into the argv array when no file-config key is present; there is no length check in this helper.
  - file: [src/agents/cli-runner/helpers.ts:382](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/cli-runner/helpers.ts#L382)
  - command: `nl -ba src/agents/cli-runner/helpers.ts | sed -n '360,395p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Execution path has no Windows argv-length mitigation:** `executePreparedCliRun` still derives `stdinPayload`, builds `args`, and proceeds toward spawn without any Windows-specific argv guard or fallback between those steps on current `main`. That means the reported failure mode is not already fixed in tree.
  - file: [src/agents/cli-runner/execute.ts:230](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/cli-runner/execute.ts#L230)
  - command: `nl -ba src/agents/cli-runner/execute.ts | sed -n '228,330p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **File-based system-prompt path exists, but only for Codex backend:** The built-in Codex backend uses `systemPromptFileConfigKey: "model_instructions_file"`, which avoids large system prompts in argv. That alternative exists on current `main`, but Claude CLI is not using it.
  - file: [extensions/openai/cli-backend.ts:52](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/extensions/openai/cli-backend.ts#L52)
  - command: `nl -ba extensions/openai/cli-backend.ts | sed -n '35,60p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **PR discussion adds fresh reproduction and narrows the correct fix surface:** In the PR comments, `Stache73` reports a clean repro on OpenClaw 2026.4.22 on Windows 11 with `spawn ENAMETOOLONG`, notes bootstrap injection at 51,507 chars, and points out prior PR #69179 moved the user prompt to stdin instead of the system prompt. That strengthens the case that this PR addresses a still-open bug rather than obsolete behavior.
