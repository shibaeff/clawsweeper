## Summary

Keep open. The PR addresses a real Feishu docs gap, but the proposed wording is not safe to auto-close or treat as already handled. Current main still omits Feishu-specific `accountId` routing guidance, and the runtime evidence supports the review comment that omitted `match.accountId` matches the literal `default` account key, not an arbitrary configured `channels.feishu.defaultAccount` for inbound routing.

## Evidence

- **Binding matcher treats omitted accountId as literal default only:** `resolveAccountMatchPriority()` returns an exact match only when the actual account id equals `DEFAULT_ACCOUNT_ID` if `match.accountId` is omitted; `"*"` is the only channel-wide fallback.
  - file: [src/channels/plugins/configured-binding-match.ts:17](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/channels/plugins/configured-binding-match.ts#L17)
  - command: `nl -ba src/channels/plugins/configured-binding-match.ts | sed -n '10,28p'`
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **Feishu docs currently leave out accountId routing semantics:** The Feishu multi-agent section documents `match.channel`, `match.peer.kind`, and `match.peer.id`, but does not explain `match.accountId` at all on current `main`. That means the PR is still addressing a live docs gap.
  - file: [docs/channels/feishu.md:331](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/docs/channels/feishu.md#L331)

  - file: [docs/channels/feishu.md:229](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/docs/channels/feishu.md#L229)
