## Summary

Current `main` does not support the reported root cause. `session_status` still omits an explicit `contextTokens` argument, but the status renderer now derives context usage from the passed `sessionEntry` and transcript/session usage data, so the missing parameter is no longer what drives the `Context:` line.

## Evidence

- **`session_status` forwards the full session entry and transcript-usage flag:** The tool calls `buildStatusText` with `sessionEntry: statusSessionEntry` and `includeTranscriptUsage: true`, so the renderer has access to persisted session usage data even without a separate `contextTokens` field.
  - file: [src/agents/tools/session-status-tool.ts:547](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/agents/tools/session-status-tool.ts#L547)
  - command: `nl -ba src/agents/tools/session-status-tool.ts | sed -n '547,599p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **`buildStatusText` passes `sessionEntry` through to the status renderer:** `buildStatusText` constructs the status payload with `sessionEntry` and only conditionally adds `agent.contextTokens`, so the explicit parameter is optional rather than the sole source of context-window data.
  - file: [src/status/status-text.ts:297](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/status/status-text.ts#L297)
  - command: `nl -ba src/status/status-text.ts | sed -n '293,331p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Current renderer reads persisted context tokens from the session entry:** `buildStatusMessage` first resolves `persistedContextTokens` from `entry.contextTokens` and uses that before falling back to `args.agent?.contextTokens`, so the missing tool argument is not required when the session entry already has context data.
  - file: [src/status/status-message.ts:620](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/status/status-message.ts#L620)
  - command: `nl -ba src/status/status-message.ts | sed -n '620,686p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Current renderer reads token usage from persisted totals and transcript logs:** The same code path computes `totalTokens` from `entry.totalTokens` or `inputTokens + outputTokens`, and with `includeTranscriptUsage` enabled it can replace that with larger transcript-log usage. That no longer matches the issue's claim that the tool display depends on provider API usage being null/zero.
  - file: [src/status/status-message.ts:551](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/status/status-message.ts#L551)
  - command: `nl -ba src/status/status-message.ts | sed -n '551,566p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **Existing tests expect persisted session context tokens to drive the rendered context line:** Status tests assert `Context: 49k/128k` and explicitly reject `Context: 49k/32k` when the session entry carries `contextTokens: 128_000`, which matches the current renderer behavior the tool now relies on.
  - file: [src/auto-reply/status.test.ts:1842](https://github.com/openclaw/openclaw/blob/7f6452897e25543c9263b67aabf140379d1c0b0c/src/auto-reply/status.test.ts#L1842)
  - command: `nl -ba src/auto-reply/status.test.ts | sed -n '1842,1855p'`
  - sha: [7f6452897e25](https://github.com/openclaw/openclaw/commit/7f6452897e25543c9263b67aabf140379d1c0b0c)
- **The linked PR targeted the old suspected call-site gap:** The issue timeline cross-references PR #69177 (`fix(agents): pass contextTokens to buildStatusText in session_status tool`), which was useful for isolating the suspected omission, but current `main` already resolves the status line from `sessionEntry`/transcript data instead.
