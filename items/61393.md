## Summary

Current `main` already refreshes Control UI chat history for active-session transcript updates from other clients via `session.message`, with deferral while a local run is active. That supersedes this PR's lifecycle-end reload approach and avoids the extra reload/error-state risks raised in review.

## Evidence

- **UI reloads on active-session transcript events:** `handleSessionMessageGatewayEvent` reloads chat history when the gateway sends `session.message` for the active session, and defers that reload if `host.chatRunId` is still active to avoid resetting in-flight stream state.
