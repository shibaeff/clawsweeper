## Summary

Keep open. Current `main` clearly routes new `dmScope="main"` DMs back to the canonical main session, and Telegram tests cover not reusing shared-thread current-session state, but I did not find any dmScope-aware reconciliation that archives, marks inactive, or removes already-existing peer-keyed DM sessions after switching back to `main`. The documented cleanup path is generic stale/missing/count-budget maintenance, not a targeted collapse of legacy peer-keyed DM identities.

## Evidence

- **Direct-message routing under `dmScope="main"` stays canonical:** `buildAgentPeerSessionKey` returns `buildAgentMainSessionKey(...)` for direct chats unless dmScope is one of the isolated modes, so new direct traffic on current main resolves to the canonical main session key rather than a peer-keyed DM session.
  - file: [src/routing/session-key.ts:143](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/routing/session-key.ts#L143)
  - command: `nl -ba src/routing/session-key.ts | sed -n '138,172p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Telegram shared-DM test avoids recovering thread/current-session state:** The Telegram route test explicitly asserts that shared `dmScope "main"` DMs do not recover `currentSessionKey` thread state and instead use `agent:main:main`. That supports that stale keyed sessions are less likely to be reused for routing now, but it is not reconciliation of the stale rows themselves.
  - file: [extensions/telegram/src/session-route.test.ts:34](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/telegram/src/session-route.test.ts#L34)

  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Public cleanup command is generic maintenance only:** The cleanup implementation only classifies actions as `keep`, `prune-missing`, `prune-stale`, `cap-overflow`, or `evict-budget`. There is no dmScope-change-specific action to retire or archive peer-keyed DM sessions when switching back to `main`.
  - file: [src/commands/sessions-cleanup.ts:48](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/commands/sessions-cleanup.ts#L48)

- **CLI docs do not document dmScope reconciliation:** The user-facing `openclaw sessions cleanup` docs only describe dry-run/enforce/fix-missing/active-key/store maintenance. They do not describe any supported command for collapsing stale per-peer/per-channel DM identities after a dmScope switch.
  - file: [docs/cli/sessions.md:61](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/docs/cli/sessions.md#L61)

But this is not automatic on dmScope change, and I found no corresponding documented CLI flow for operators in this issue’s scenario.
