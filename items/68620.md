## Summary

Keep open. Current `main` adds an LLM idle-timeout guard for normal embedded runs, but the specific stuck-session report is not clearly resolved end to end: the diagnostic heartbeat still only emits warnings, and cron-triggered runs explicitly disable the default idle watchdog and can still sit behind a much longer outer timeout, which matches the follow-up comment about shared-session blockage across Telegram and cron.

## Evidence

- **Diagnostic heartbeat still warns only:** `logSessionStuck` only logs a warning and emits a `session.stuck` diagnostic event; it does not abort the run or clear the session. The heartbeat loop still just calls `logSessionStuck(...)` when a processing session exceeds the threshold.
  - file: [src/logging/diagnostic.ts:296](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/logging/diagnostic.ts#L296)

## Risks / Open Questions

- Closing as implemented would overstate the current fix: normal interactive runs have an idle-timeout path, but the diagnostic heartbeat still is not a corrective auto-kill mechanism.
- The follow-up discussion specifically mentions cron sharing the same session lane, and current cron defaults still allow much longer hangs than the requested 2-5 minute kill window.
