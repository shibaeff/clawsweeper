## Summary

Keep open. Current `main` has session rollover by daily boundary or idle inactivity, plus thread-bound max-age and pre-compaction memory flush, but it does not implement a general agent-session max lifetime/TTL or max-turn auto-rotation like #45390 requests.

## Evidence

- **User-facing session lifecycle only exposes daily, idle, and manual reset:** Docs describe session reuse until expiry via daily reset, idle reset, or manual `/new`/`/reset`; no general `sessionMaxAge` or turn-count rotation is documented.
  - file: [docs/concepts/session.md:57](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/docs/concepts/session.md#L57)
  - command: `nl -ba docs/concepts/session.md | sed -n '56,76p'`
  - sha: [028b6c9b13c9](https://github.com/openclaw/openclaw/commit/028b6c9b13c960b6f59a7643aff4f6de04e07f71)
- **Rollover happens on the next inbound message after reset conditions:** Deep-dive docs say daily reset creates a new `sessionId` on the next message after the boundary, and idle expiry does so when a message arrives after the idle window. That is not an active-session elapsed-age TTL.
  - file: [docs/reference/session-management-compaction.md:130](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/docs/reference/session-management-compaction.md#L130)

- **Session config schema has only daily/idle reset controls:** The runtime config schema accepts `mode: daily|idle`, `atHour`, and `idleMinutes` for session reset. There is no config field for session max age duration from creation or session max turns.
  - file: [src/config/zod-schema.session.ts:17](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/config/zod-schema.session.ts#L17)

- **Hard max-age exists only for thread-bound sessions:** Type definitions expose `threadBindings.maxAgeHours` as an optional hard max age for thread-bound sessions, not for general agent sessions. General `SessionResetConfig` still only carries `mode`, `atHour`, and `idleMinutes`.
  - file: [src/config/types.base.ts:124](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/config/types.base.ts#L124)

- **Freshness logic is time-boundary and inactivity based:** `evaluateSessionFreshness()` marks a session stale only when `updatedAt` falls before the daily reset boundary or past `updatedAt + idleMinutes`. There is no turn counter or elapsed-lifetime check from session creation.
  - file: [src/config/sessions/reset-policy.ts:33](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/config/sessions/reset-policy.ts#L33)

This supports inbound rollover, not automatic rotation after N turns or fixed lifetime during an active session.

## Risks / Open Questions

- There is partial overlap with existing features: operators can approximate some hygiene using `session.reset` daily/idle rollover and `agents.defaults.compaction.memoryFlush`, but that is not the requested active-session TTL or max-turn rotation.
- The repo already has a thread-only hard max age (`session.threadBindings.maxAgeHours`), which could make the issue look solved at a glance even though general agent sessions still lack the requested control.
