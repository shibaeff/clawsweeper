## Summary

Current main already solves the reported hot-path SecretRef re-resolution problem by resolving secrets eagerly into a process-local runtime snapshot at startup/reload and serving runtime reads from that in-memory snapshot instead of re-reading providers on each request. The implementation shipped by v2026.4.22, so the requested TTL cache is no longer needed for the gateway self-call case described in the issue.

## Risks / Open Questions

- There is no dedicated user-configurable TTL cache knob; the issue is only closable because the underlying repeated-resolution behavior is already addressed by the stronger runtime-snapshot design.
- Some one-off CLI/helper paths can still resolve SecretRefs directly when they are operating outside an active gateway runtime snapshot, but that is not the long-lived gateway self-call case described here.

## Close Comment

Closing from a Codex review because current `main` already covers the underlying problem well enough. OpenClaw now resolves SecretRefs eagerly into an in-memory runtime snapshot and serves runtime reads from that snapshot instead of re-resolving providers on hot paths: see `docs/gateway/secrets.md:16`, `src/config/io.ts:1969`, `src/secrets/runtime.ts:337`, and `src/gateway/server-startup-config.ts:263`. There is also direct regression coverage showing `gateway.auth.token` stays available from the last-known-good runtime snapshot even after the backing exec resolver starts failing (`src/gateway/server.reload.test.ts:809`, `src/secrets/runtime.gateway-auth.integration.test.ts:83`). This behavior is already present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The specific TTL-cache config proposed here was not added, but the reported repeated in-process SecretRef re-resolution path is already addressed by the runtime-snapshot design.
