## Summary

Current main no longer routes `channels.telegram.*` config edits through a gateway restart path. Telegram config changes are classified as hot channel reloads and handled with in-process `stopChannel`/`startChannel`, and the macOS launchd restart path on main also avoids the old bootout-only behavior.

## Evidence

- **Telegram config is a hot-reload prefix:** The Telegram plugin registers `channels.telegram` under `reload.configPrefixes`, so edits to `channels.telegram.groups.*` are treated as channel reload inputs, not generic gateway restart inputs.
  - file: [extensions/telegram/src/shared.ts:161](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/extensions/telegram/src/shared.ts#L161)

  - file: [src/gateway/config-reload-plan.ts:127](https://github.com/openclaw/openclaw/blob/028b6c9b13c960b6f59a7643aff4f6de04e07f71/src/gateway/config-reload-plan.ts#L127)

## Close Comment

Closing from a Codex review because current `main` already avoids the reported failure mode. Telegram config changes are registered as hot-reload prefixes in `extensions/telegram/src/shared.ts:161`, the reload planner maps those paths to `restart-channel:telegram` instead of `restartGateway` in `src/gateway/config-reload-plan.ts:127`, and the hot-reload handler only does in-process `stopChannel`/`startChannel` work in `src/gateway/server-reload-handlers.ts:166`. The channel stop path itself is local task shutdown logic in `src/gateway/server-channels.ts:528`, not a launchd unload/restart path. Main also contains the launchd-side fix: `src/infra/process-respawn.ts:30` now leaves macOS launchd restarts to the supervisor, and `src/infra/restart.ts:409` bootstraps the plist before retrying `kickstart` if a service was previously booted out. I could not tie that to a shipped release tag yet, so `fixedRelease` is left unset.
