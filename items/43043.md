## Summary

Keep open. Current `main` still leaves `openclaw channels logs` on the old code path: it reads `getResolvedLoggerSettings().file` directly, which resolves to today's rolling filename, while the cross-date fallback now exists only in the shared Gateway/log-tail path. The underlying bug described by this PR therefore still appears real on `main`, although the PR's proposed implementation is stale relative to the newer shared `src/logging/log-tail.ts` utility.

## Evidence

- **`channels logs` still bypasses rolling-log resolution:** `src/commands/channels/logs.ts:88-89` reads `const file = getResolvedLoggerSettings().file;` and immediately tails that file via `readTailLines(...)`. There is no call into `src/logging/log-tail.ts` or any fallback to the most recent rolling log.
  - file: [src/commands/channels/logs.ts:88](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/commands/channels/logs.ts#L88)
  - command: `nl -ba src/commands/channels/logs.ts | sed -n '1,180p'`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Logger path is date-dependent per process:** `src/logging/logger.ts:127` sets `file = cfg?.file ?? defaultRollingPathForToday()`, and `src/logging/logger.ts:337-339` builds that default from `new Date()`. A fresh CLI process on the next day therefore points at the new day's filename, matching the PR's reported failure mode.
