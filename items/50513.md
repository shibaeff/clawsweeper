## Summary

The requested post-response GitHub issue work-loop is a good automation idea, but current main already provides the needed post-turn hook and follow-up run APIs for plugins, and project scope says this kind of optional workflow should live as a plugin/ClawHub skill rather than new core behavior.

## Risks / Open Questions

- There is not a bundled example for this exact GitHub issue-resolution workflow yet; someone still has to package the behavior as a plugin/ClawHub skill.
- A plugin implementation would still need explicit operator setup for GitHub auth/repo targeting and safe close/comment policy.

## Close Comment

Closing after Codex review. The post-response seam requested here is already present on current `main` via `agent_end` (`src/agents/pi-embedded-runner/run/attempt.ts:2655`, documented at `docs/concepts/agent-loop.md:97`). Current plugin APIs also already support follow-up agent work from that seam through `api.runtime.agent.runEmbeddedAgent(...)` / `runEmbeddedPiAgent(...)` and subagent helpers (`docs/plugins/sdk-runtime.md:28`, `docs/plugins/sdk-runtime.md:87`). The bundled `skill-workshop` plugin is already using this pattern from `extensions/skill-workshop/index.ts:48` and `extensions/skill-workshop/src/reviewer.ts:225`. Given `VISION.md:54` and `VISION.md:68`, this kind of repo-specific GitHub maintenance loop belongs as a plugin or ClawHub skill rather than additional OpenClaw core behavior.
