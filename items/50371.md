## Summary

Current `main` already implements gateway-bindable plugin subagent runtime for `before_prompt_build` and other gateway hook paths, and the same implementation is already present in the shipped `v2026.4.22` release tag.

## Evidence

- **Gateway plugin loads opt into gateway-bindable runtime:** Gateway plugin bootstrap passes `runtimeOptions.allowGatewaySubagentBinding: true`, so plugin hooks register against a gateway-bindable runtime rather than the default unavailable subagent stub. I verified this on current `main` and in the `v2026.4.22` tag.
  - file: [src/gateway/server-plugins.ts:503](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/server-plugins.ts#L503)
  - command: `nl -ba src/gateway/server-plugins.ts | sed -n '498,505p' && git show v2026.4.22:src/gateway/server-plugins.ts | nl -ba | sed -n '461,473p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Plugin runtime late-binds `api.runtime.subagent` when enabled:** `createPluginRuntime()` resolves `api.runtime.subagent` through a late-binding proxy when `allowGatewaySubagentBinding` is enabled, so gateway-loaded plugins do not hit the default `RequestScopedSubagentRuntimeError` path.
  - file: [src/plugins/runtime/index.ts:159](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/plugins/runtime/index.ts#L159)
  - command: `nl -ba src/plugins/runtime/index.ts | sed -n '153,218p' && git show v2026.4.22:src/plugins/runtime/index.ts | nl -ba | sed -n '176,218p'`

- **Gateway request scope and fallback context are wired for plugin subagent dispatch:** All gateway handlers run inside `withPluginRuntimeGatewayRequestScope(...)`, and plugin subagent dispatch also falls back to the process gateway context when no per-request scope is present. That is the wiring needed for hook-triggered `api.runtime.subagent.run()` calls, including fire-and-forget hook paths.
  - file: [src/gateway/server-methods.ts:171](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/gateway/server-methods.ts#L171)

## Risks / Open Questions

- `message_received` still exposes `PluginHookMessageContext` (`channelId` / `accountId` / `conversationId`) rather than a `sessionKey`, so plugins spawning there must choose their own target session key instead of reading one from hook context.
- This review relied on code, tag inspection, and existing tests; I did not run a live gateway reproduction because the task required a byte-for-byte read-only checkout.

## Close Comment

Closing from a Codex review because this is already implemented on current `main`, and the shipped `v2026.4.22` tag (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) already contains the needed path. Gateway plugin bootstrap opts into `allowGatewaySubagentBinding` in `src/gateway/server-plugins.ts:503`, `createPluginRuntime()` late-binds `api.runtime.subagent` to the active gateway runtime in `src/plugins/runtime/index.ts:159`, and all gateway handlers run inside `withPluginRuntimeGatewayRequestScope(...)` in `src/gateway/server-methods.ts:171`. `before_prompt_build` already runs with `ctx.sessionKey` in `src/agents/pi-embedded-runner/run/attempt.ts:2005`, and the current tests in `src/plugins/runtime/index.test.ts:258` plus `src/gateway/server-plugins.test.ts:565` and `src/gateway/server-plugins.test.ts:587` cover both request-scoped and fallback subagent dispatch. If you still hit a concrete repro on a newer build, please open a fresh issue with the exact plugin hook snippet and OpenClaw version.
