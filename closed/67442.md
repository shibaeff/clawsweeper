## Summary

Current `main` and the shipped `v2026.4.23` release already block the reported transport-metadata -> dreaming -> `MEMORY.md` path through transcript-envelope stripping, non-inline default dreaming storage, and promotion exclusion of dreaming artifacts.

## Evidence

- **Session transcript ingestion strips inbound metadata before normalization:** `sanitizeSessionText` removes inbound metadata envelopes for user-role messages before normalization and drops generated `System (untrusted)` wrapper messages, preventing those transport/session blocks from entering the dreaming session corpus.

  - file: [src/memory-host-sdk/host/session-files.test.ts:179](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/memory-host-sdk/host/session-files.test.ts#L179)

## Risks / Open Questions

- There is no single generic deep-phase heuristic that rejects every possible metadata-looking daily-note snippet; this close is based on the reported live path already being blocked earlier in the pipeline and on dreaming artifacts no longer being promotable input.
