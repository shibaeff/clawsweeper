## Summary

Current `main` already contains the Discord gateway restart hardening needed for this report: late `Max reconnect attempts ... code 1005` errors are classified and suppressed during dispose/teardown instead of surfacing as uncaught exceptions, provider startup cleanup explicitly disconnects and disposes the shared gateway, and stale-socket health checks now key off transport activity rather than idle app-event gaps.

## Evidence

- **Late reconnect-exhausted gateway errors are suppressed after dispose/teardown:** `classifyDiscordGatewayEvent` maps `Max reconnect attempts` errors to `reconnect-exhausted`, and `createDiscordGatewaySupervisor` logs and suppresses those events in `disposed`/`teardown` phases instead of letting the emitter crash the process.
  - file: [extensions/discord/src/monitor/gateway-supervisor.ts:76](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/gateway-supervisor.ts#L76)
  - command: `nl -ba extensions/discord/src/monitor/gateway-supervisor.ts | sed -n '1,220p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Provider startup cleanup disconnects and disposes the gateway before lifecycle start:** When Discord monitor startup fails before lifecycle begins, the provider now calls `lifecycleGateway?.disconnect()` and then `gatewaySupervisor?.dispose()` in the `finally` cleanup path, which matches the failure mode described in the issue.
  - file: [extensions/discord/src/monitor/provider.ts:1143](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/provider.ts#L1143)

  - file: [extensions/discord/src/monitor/provider.test.ts:281](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/provider.test.ts#L281)
