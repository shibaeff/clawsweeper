## Summary

Current `main` already includes a macOS Talk Mode fix that retries failed local ElevenLabs playback through gateway `talk.speak`, which matches the reported case where server-side TTS exists but the macOS app stays silent.

## Evidence

- **macOS Talk Mode now retries gateway playback:** `playAssistant` no longer drops straight from local ElevenLabs failure to system voice. It now retries via gateway `talk.speak` first, directly matching a server-generated-audio/local-playback-drop scenario.
  - file: [apps/macos/Sources/OpenClaw/TalkModeRuntime.swift:471](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/apps/macos/Sources/OpenClaw/TalkModeRuntime.swift#L471)
  - command: `nl -ba apps/macos/Sources/OpenClaw/TalkModeRuntime.swift | sed -n '468,770p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Gateway `talk.speak` audio is decoded and played locally:** The new fallback path calls `GatewayConnection.shared.requestDecoded(method: .talkSpeak, ...)`, decodes `audioBase64`, and plays that audio on macOS.
  - file: [apps/macos/Sources/OpenClaw/TalkModeRuntime.swift:733](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/apps/macos/Sources/OpenClaw/TalkModeRuntime.swift#L733)
