## Summary

Current `main` still appears to drop model-registry/provider headers on the image-understanding runtime path. The code resolves only an API key for `imageModel` requests and calls `complete()` without forwarding any resolved headers, and the only related fix work I found is still in an open PR rather than on `main`.

## Evidence

- **Image runtime resolves only API key from model auth:** `resolveImageRuntime` looks up the model, calls `getApiKeyForModel(...)`, stores the runtime API key, and returns only `{ apiKey, model }`. There is no request-header lookup or header return path here.
  - file: [src/media-understanding/image.ts:138](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/media-understanding/image.ts#L138)
  - command: `nl -ba src/media-understanding/image.ts | sed -n '130,161p'`
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **Image completion call omits headers:** The non-MiniMax image path calls `complete(model, context, { apiKey, maxTokens, signal, ... })` and does not pass any `headers` option, so model-registry/provider headers are not preserved on this path.
  - file: [src/media-understanding/image.ts:320](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/media-understanding/image.ts#L320)
