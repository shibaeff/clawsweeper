## Summary

Current `main` already supports provider API-key rotation for OpenRouter through the shared auth-profile rotation path, with OpenRouter-specific coverage and docs now present.

## Evidence

- **OpenRouter uses the standard API-key auth/profile path:** The bundled OpenRouter provider registers a normal API-key auth method, so its credentials participate in the shared provider auth machinery rather than a one-off path.
  - file: [extensions/openrouter/index.ts:72](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/openrouter/index.ts#L72)

  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Runtime resolves provider credentials by iterating auth-profile order:** `resolveApiKeyForProvider` loads the provider's ordered auth profiles and returns the first usable candidate, which is the core rotation mechanism used before env/config fallback.
  - file: [src/agents/model-auth.ts:496](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/agents/model-auth.ts#L496)

  - file: [src/agents/auth-profiles/order.ts:66](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/agents/auth-profiles/order.ts#L66)
