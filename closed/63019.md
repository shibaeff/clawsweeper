## Summary

Current `main` already includes a targeted WhatsApp QR/relink synchronization fix that matches this report: Control UI now sends its current QR into `web.login.wait`, the gateway validates and forwards that field, and the WhatsApp login waiter returns refreshed QR payloads instead of leaving the UI stuck on stale state.

## Evidence

- **Control UI now tracks QR rotation during wait:** `waitWhatsAppLogin` sends `currentQrDataUrl` with `web.login.wait` and updates `whatsappLoginQrDataUrl` when the backend returns a refreshed QR, directly addressing the reported stuck relink flow.
  - file: [ui/src/ui/controllers/channels.ts:64](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/ui/src/ui/controllers/channels.ts#L64)

  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Gateway protocol and handler now carry the caller's current QR:** `WebLoginWaitParamsSchema` accepts `currentQrDataUrl`, and `web.login.wait` forwards it to the provider wait path, so the UI/backend handoff described in the issue is now part of the public request flow.
  - file: [src/gateway/protocol/schema/channels.ts:233](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/protocol/schema/channels.ts#L233)
