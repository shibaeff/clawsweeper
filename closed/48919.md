## Summary

Current `main` already has the reported fix: plugin runtime registry state is process-global via `globalThis` + `Symbol.for("openclaw.pluginRegistryState")`, and agent/plugin tool resolution reuses the active gateway-bindable registry instead of loading a separate isolated copy. The same implementation is present in release `v2026.4.22`, so the reported Rollup split-state/tool-exposure bug is already addressed.

## Evidence

- **Process-global plugin registry state:** `src/plugins/runtime.ts` stores registry state on `globalThis[Symbol.for("openclaw.pluginRegistryState")]` and all `set/getActivePluginRegistry` calls read/write that shared holder, which is the exact architectural fix the issue asked for.
