## Close Comment

Closing this as implemented after Codex review.

Current main already ships the requested plugin extension point for custom web search providers via the renamed public API `registerWebSearchProvider(...)`, with registry wiring, `web_search` runtime integration, docs, tests, and bundled provider examples.

What I checked:

- **Public plugin API already exposes web-search provider registration:** `OpenClawPluginApi` includes `registerWebSearchProvider: (provider: WebSearchProviderPlugin) => void`. The v2026.4.22 tag contains the same line, so this shipped in a release. ([`src/plugins/types.ts:1999`](https://github.com/openclaw/openclaw/blob/00bd2cf7a376/src/plugins/types.ts#L1999), [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376))
- **Plugin registry wires registered web-search providers into core:** `createPluginRegistry` defines `registerWebSearchProvider(...)` and stores entries in `registry.webSearchProviders`, preventing this from being a docs-only/API-only stub. ([`src/plugins/registry.ts:931`](https://github.com/openclaw/openclaw/blob/a53fea39050b/src/plugins/registry.ts#L931), [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050b))
- **Captured registration path supports the API during plugin loading:** Plugin registration capture records `api.registerWebSearchProvider(...)` calls into `webSearchProviders`, showing the loader/runtime path already supports this capability. - **`web_search` now resolves provider-owned tool definitions from plugin/runtime registries:** The current tool implementation delegates to `resolveWebSearchDefinition(...)` and `runWebSearch(...)`, which resolve plugin web-search providers and build the active `web_search` tool from provider-owned definitions.
