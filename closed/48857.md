## Summary

Current main already handles `message --action send --filePath` for Feishu: the shared message-action runner normalizes `filePath`/`path` into media for `send`, and the Feishu channel routes media sends through its media outbound path instead of the text-only send path. Regression tests cover both the Feishu dispatch and local-path media loading, and the same implementation is present in release `v2026.4.22`.

## Evidence

- **Core send path accepts `filePath` for `send`:** `runMessageAction` explicitly reads `media`, `mediaUrl`, `path`, `filePath`, and `fileUrl` for `send`, then assigns the resolved attachment back to `params.media` before channel dispatch.

## Risks / Open Questions

- Feishu outbound still has a fallback that sends plain text when `sendMediaFeishu` itself throws (`extensions/feishu/src/outbound.ts:264`), so a new bug with upload/auth/local-root failures could still produce link-like fallback behavior even though the reported `filePath` routing gap is implemented.
