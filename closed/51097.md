## Summary

Current main already bounds `sessions.json` growth and prunes isolated cron-run sessions, which addresses the unbounded-growth path described in the issue even though the store remains JSON-backed.

## Risks / Open Questions

- The session store is still loaded from and rewritten to a single JSON file, so this is a mitigation-by-bounding/pruning rather than a directory-backed or lazy-loaded redesign.
- Docs still describe `session.maintenance.mode` as `warn` by default in some places, while code currently defaults to `enforce`; that documentation drift does not change the current runtime behavior but it is a source of ambiguity.
