## Summary

Current `main` already keeps device identity attached for authenticated loopback gateway calls and probes, matching the reported fix, and the same behavior is present in the shipped `v2026.4.22` release.

## Evidence

- **Gateway call path now always resolves device identity by default:** `resolveDeviceIdentityForGatewayCall()` unconditionally loads device identity for gateway calls, with only a fallback to `null` if persistence/loading fails; the client wiring uses that default unless the caller explicitly passes `deviceIdentity`.
  - file: [src/gateway/call.ts:196](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/gateway/call.ts#L196)

  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Probe/status path keeps identity for authenticated loopback probes:** `probeGateway()` skips device identity only for unauthenticated loopback probes. Authenticated loopback probes load and attach device identity specifically so read/detail RPCs are not scope-limited.
  - file: [src/gateway/probe.ts:148](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/gateway/probe.ts#L148)
