## Summary

Keep open. Current `main` does not implement Feishu `asVoice`/`audioAsVoice` handling for generic outbound media. The Feishu adapter routes by filename/MIME only, only treats Opus/Ogg as native `audio`, and existing tests explicitly expect MP3 to fall back to a generic file attachment.

## Evidence

- **Generic outbound voice flag exists:** Channel outbound context exposes `audioAsVoice?: boolean`, so plugins can receive the intent.
  - file: [src/channels/plugins/outbound.types.ts:14](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/channels/plugins/outbound.types.ts#L14)
  - command: `nl -ba src/channels/plugins/outbound.types.ts | sed -n '10,24p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Feishu adapter drops the voice flag:** `extensions/feishu/src/outbound.ts` destructures `cfg,to,text,mediaUrl,accountId,mediaLocalRoots,replyToId,threadId` in `sendMedia`, but not `audioAsVoice`, and calls `sendMediaFeishu` without any voice/intention parameter.
  - file: [extensions/feishu/src/outbound.ts:230](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/feishu/src/outbound.ts#L230)
  - command: `nl -ba extensions/feishu/src/outbound.ts | sed -n '220,275p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Feishu media routing is extension/MIME based only:** `resolveFeishuOutboundMediaKind` maps only `.opus`/`.ogg` or `audio/ogg`/`audio/opus` to `msgType: "audio"`; unsupported audio falls through to `file`. `sendMediaFeishu` has no `audioAsVoice` parameter and uses that routing result directly.
  - file: [extensions/feishu/src/media.ts:524](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/feishu/src/media.ts#L524)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '520,660p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Current test suite expects MP3 to become a file attachment:** The Feishu test `falls back to generic file for unsupported audio formats` loads `song.mp3` / `audio/mpeg` and asserts `file_type: "stream"` and `msg_type: "file"`.
  - file: [extensions/feishu/src/media.test.ts:239](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/feishu/src/media.test.ts#L239)
  - command: `nl -ba extensions/feishu/src/media.test.ts | sed -n '239,258p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Docs only promise Feishu voice delivery for Opus-capable TTS paths:** `docs/tools/tts.md` says Feishu voice messages are Opus-based and notes MiniMax MP3 does not natively support voice-note format, which matches the current code path rather than the requested generic `asVoice` behavior.
  - file: [docs/tools/tts.md:428](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/docs/tools/tts.md#L428)
  - command: `nl -ba docs/tools/tts.md | sed -n '424,432p;504,510p'`

## Risks / Open Questions

- This review is code-and-docs based; I did not perform a live Feishu API send against current `main`.
- Opus/Ogg media may already produce native Feishu voice bubbles, so the gap is specifically the generic `asVoice` intent and non-Opus audio such as MP3.
