## Close Comment

Closing this as implemented after Codex review.

Current main already strips `[[reply_to...]]` directives before outbound delivery in the shared reply pipeline, so the iMessage bug this PR targets is already handled. The PR's separate LanceDB rejected-promise retry change is also already present on main in the newer runtime loader.

What I checked:

- **Final reply pipeline strips reply tags before delivery:** `buildReplyPayloads()` always normalizes reply directives for outbound payloads with `parseMode: "always"` before final delivery, so `[[reply_to...]]` text is removed upstream of iMessage transport. ([`src/auto-reply/reply/agent-runner-payloads.ts:146`](https://github.com/openclaw/openclaw/blob/a53fea39050b/src/auto-reply/reply/agent-runner-payloads.ts#L146), [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050b))
- **Directive parser removes reply tags while preserving reply metadata:** `parseReplyDirectives()` calls `parseInlineDirectives(... stripReplyTags: true)` and returns cleaned text plus `replyToId`/`replyToCurrent`, which is the behavior this PR was trying to add locally in the iMessage sanitizer. ([`src/auto-reply/reply/reply-directives.ts:23`](https://github.com/openclaw/openclaw/blob/a53fea39050b/src/auto-reply/reply/reply-directives.ts#L23), [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050b))
- **Streamed/block replies use the same normalization path:** `createBlockReplyDeliveryHandler()` also runs `normalizeReplyPayloadDirectives()` before emitting block payloads, so streamed iMessage replies do not depend on [`extensions/imessage/src/monitor/sanitize-outbound.ts`](https://github.com/openclaw/openclaw/blob/a53fea39050b/extensions/imessage/src/monitor/sanitize-outbound.ts) for reply-tag stripping either. ([`src/auto-reply/reply/reply-delivery.ts:119`](https://github.com/openclaw/openclaw/blob/a53fea39050b/src/auto-reply/reply/reply-delivery.ts#L119), [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050b))
- **iMessage delivery uses structured replyToId metadata:** The iMessage sender passes `payload.replyToId` separately to `sendMessageIMessage()` and only sends the normalized text body, so reply threading no longer depends on visible inline tags. ([`extensions/imessage/src/monitor/deliver.ts:41`](https://github.com/openclaw/openclaw/blob/a53fea39050b/extensions/imessage/src/monitor/deliver.ts#L41), [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050b))
- **The PR's extra LanceDB retry fix is already on main elsewhere:** Current main already clears the cached LanceDB load promise on failure in the newer runtime loader, making the PR's [`extensions/memory-lancedb/index.ts`](https://github.com/openclaw/openclaw/blob/a53fea39050b/extensions/memory-lancedb/index.ts) hunk obsolete. ([`extensions/memory-lancedb/lancedb-runtime.ts:320`](https://github.com/openclaw/openclaw/blob/a53fea39050b/extensions/memory-lancedb/lancedb-runtime.ts#L320), [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050b))

So I’m closing this as already implemented rather than keeping a duplicate issue open.
