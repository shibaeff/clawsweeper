## Summary

Current `main` already addresses this Discord startup/restart race by polling `gateway.isConnected` during lifecycle startup instead of relying on the post-start debug listener to catch the initial websocket-open event. The fix is covered by lifecycle tests and documented in the `2026.4.1` changelog.

## Evidence

- **Lifecycle now waits for READY by polling connection state:** `waitForGatewayReady()` checks `gateway.isConnected` every 250ms, pushes a connected status when READY is observed, and restarts the gateway after 15s if READY never arrives. This removes the original dependency on the lifecycle debug listener seeing the first websocket-open event.
