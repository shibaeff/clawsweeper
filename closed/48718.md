## Summary

Current main already contains the remote/attachOnly CDP health-check and fail-fast handling that addresses this report, and the fix shipped in v2026.4.22.

## Evidence

- **Status now requires a real CDP command-health probe:** The browser status route computes `cdpReady`/`running` from `isReachable()`, and `isChromeCdpReady()` returns true only when `diagnoseChromeCdp()` succeeds rather than on HTTP reachability alone.
  - file: [extensions/browser/src/browser/routes/basic.ts:82](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/browser/src/browser/routes/basic.ts#L82)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Stale websocket command channels are explicitly covered:** Regression tests lock the case where the websocket opens but the CDP command channel is stale, asserting `cdpReady` is false and the diagnostic code is `websocket_health_command_timeout`.
  - file: [extensions/browser/src/browser/chrome.test.ts:395](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/browser/src/browser/chrome.test.ts#L395)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Attach-only and remote CDP no longer fall through to generic timeouts:** For attachOnly or remote profiles, when HTTP responds but the CDP websocket is not healthy, `ensureBrowserAvailable()` now throws `BrowserProfileUnavailableError` with explicit remote/attachOnly websocket messaging instead of hanging until the browser tool times out.
  - file: [extensions/browser/src/browser/server-context.availability.ts:264](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/browser/src/browser/server-context.availability.ts#L264)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Agent-side timeout diagnostics include CDP health state:** The node-host browser proxy formats timeout errors with `running`, `cdpHttp`, `cdpReady`, transport, and redacted `cdpUrl`, and tests cover the ws-backed timeout path.
  - file: [extensions/browser/src/node-host/invoke-browser.ts:190](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/browser/src/node-host/invoke-browser.ts#L190)
