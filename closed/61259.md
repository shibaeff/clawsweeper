## Summary

Current `main` has moved bundled-plugin Jiti loading onto module-relative paths, keeps bundled channel startup on compiled `dist/extensions` entries with native-load preference where possible, and carries regression/smoke coverage for the loader path. This appears fixed on `main`, but I could not tie it to a released tag yet.

## Evidence

- **Unreleased changelog entry matches the reported failure mode:** `CHANGELOG.md` explicitly records `Plugins/startup: resolve bundled plugin Jiti loads relative to the target plugin module instead of the central loader` for `#70073`, which aligns with this issue's Jiti recursion/stack-overflow report.
  - file: [CHANGELOG.md:120](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/CHANGELOG.md#L120)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Bundled plugin loader now anchors Jiti to the target module:** The live bundled plugin loader passes `jitiFilename: modulePath` when constructing the Jiti loader, instead of anchoring loads to the central loader file.
  - file: [src/plugins/loader.ts:451](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/plugins/loader.ts#L451)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Bundled channel runtime path stays on compiled dist entries and prefers native JS loading:** Bundled channel startup resolves generated entries and loads them through `loadChannelPluginModule`, with a `shouldTryNativeRequire` gate for compiled `dist/**` JavaScript modules to avoid unnecessary Jiti bridging on that hot path.
  - file: [src/channels/plugins/bundled.ts:219](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/channels/plugins/bundled.ts#L219)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Regression test covers the module-relative Jiti filename behavior:** There is a loader regression test asserting that bundled plugin loads use the bundled plugin module path as the Jiti filename.
  - file: [src/plugins/loader.jiti-filename.test.ts:51](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/plugins/loader.jiti-filename.test.ts#L51)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Smoke guidance exists for the packaged/global-install loader path:** The testing guide documents a Bun global-install smoke that verifies bundled provider discovery returns successfully instead of hanging, which is the same packaged-loader class of failure referenced in the issue timeline and `#70073`.
  - file: [docs/help/testing.md:548](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/docs/help/testing.md#L548)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Temporary env workaround from the issue is absent on current main:** A repo-wide search found no current `OPENCLAW_SKIP_CHANNEL_PLUGINS` usage, so the mainline resolution is the loader-path fix rather than preserving the temporary bypass described in the report.
  - command: `rg -n "OPENCLAW_SKIP_CHANNEL_PLUGINS" -S .`
