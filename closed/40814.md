## Summary

Current `main` and release `v2026.4.23` already handle Telegram `/cmd@ThisBot` input by normalizing the current bot suffix into `CommandBody` before inline directive parsing. This PR is superseded, and its permissive regex approach would also accept `@OtherBot`, which conflicts with the now-tested bot-specific behavior.

## Evidence

- **Telegram inbound context normalizes current-bot command suffixes:** `buildTelegramInboundContextPayload` derives `commandBody` with `normalizeCommandBody(rawBody, { botUsername })` before storing `CommandBody`, so Telegram `/model@ThisBot ...` style input is normalized on the channel path before directive parsing.
  - file: [extensions/telegram/src/bot-message-context.session.ts:309](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/extensions/telegram/src/bot-message-context.session.ts#L309)
  - command: `nl -ba extensions/telegram/src/bot-message-context.session.ts | sed -n '304,330p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Directive parsing prefers normalized CommandBody:** `resolveDirectiveCommandText` reads `sessionCtx.CommandBody` before `RawBody`, and `continueReplyWithDirectives` passes that `commandText` into `parseInlineDirectives()`. That means Telegram's normalized command text is the directive-parsing input on current `main`.
  - file: [src/auto-reply/reply/get-reply-directives.ts:203](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/auto-reply/reply/get-reply-directives.ts#L203)

- **Normalization is bot-specific, not wildcard:** `normalizeCommandBody` strips `@suffix` only when the suffix matches the active bot username. Other bots' suffixes are preserved, which is the safer multi-bot behavior.
  - file: [src/auto-reply/commands-registry-normalize.ts:70](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/auto-reply/commands-registry-normalize.ts#L70)

- **Tests cover current-bot and other-bot cases:** Current tests assert `/help@openclaw` normalizes to `/help`, while `/help@otherbot` stays unchanged. That matches the desired Telegram disambiguation contract and makes the PR's wildcard suffix matcher unnecessary.
