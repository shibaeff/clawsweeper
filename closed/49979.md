## Summary

Current `main` does not show the reported nested-property-loss bug for supported cron inputs, and the PR’s added examples rely on unsupported extra fields that the cron API rejects.

## Evidence

- **Normalization only rewrites top-level cron fields:** `coerceSchedule`, `coercePayload`, `coerceDelivery`, and `normalizeCronJobInput` all start from shallow copies and then trim/delete or rewrite top-level keys only; they do not traverse or strip nested payload/delivery objects, so nested values are preserved by reference on current `main`.

- **Supported cron schemas reject the PR test shapes:** The public cron contract sets `additionalProperties: false` for schedule, payload, and delivery objects. Extra fields used by this PR such as `payload.options`, `payload.config`, and `delivery.headers` are outside the supported API surface.
