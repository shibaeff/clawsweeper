## Summary

Current `main` already separates gateway stop vs restart, routes normal control-plane restarts through authorized `SIGUSR1`, suppresses noop config-write restarts, coalesces repeated restart requests with a cooldown, and surfaces planned restarts in the UI instead of a generic disconnect. The same implementation is present in release `v2026.4.23`.

## Evidence

- **Gateway restart is no longer SIGTERM-driven:** The run loop treats `SIGTERM` as `request("stop", "SIGTERM")`, while authorized `SIGUSR1` becomes `request("restart", "SIGUSR1")`; the process stays alive for in-process restarts.
  - file: [src/cli/gateway-cli/run-loop.ts:222](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/cli/gateway-cli/run-loop.ts#L222)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Config writes avoid unnecessary restart churn:** `config.patch` short-circuits as `noop` when there are no changed paths, and both `config.patch` and `config.apply` schedule `scheduleGatewaySigusr1Restart(...)` instead of issuing SIGTERM directly.
  - file: [src/gateway/server-methods/config.ts:549](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server-methods/config.ts#L549)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Restart requests are single-flight with cooldown:** The restart scheduler defines a 30s cooldown and returns `coalesced: true` when another restart is already in flight or already scheduled, which directly addresses repeated restart churn.
  - file: [src/infra/restart.ts:466](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/infra/restart.ts#L466)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Update-triggered restarts use the same guarded path:** `update.run` only schedules a restart after a successful update, uses `scheduleGatewaySigusr1Restart(...)`, and explicitly avoids restarting after failed updates because that can cause crash-loop behavior.
  - file: [src/gateway/server-methods/update.ts:104](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server-methods/update.ts#L104)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Control UI now presents planned restarts as restarts:** Gateway `shutdown` events with `restartExpectedMs` are rendered as `Restarting: ...`, and tests cover both restart-reason shutdown events and `1012 service restart` closes so planned restarts are not shown as a generic crash/disconnect.
  - file: [ui/src/ui/app-gateway.ts:554](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/ui/src/ui/app-gateway.ts#L554)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **This shipped in the latest release:** The relevant gateway restart files have no diff between release `v2026.4.23` (`a9797214338ba31c52c796adbb75afb16e0684a9`) and current `main`, so the implementation is already in the released build.
  - command: `git diff --stat a9797214338ba31c52c796adbb75afb16e0684a9..245451b6a9c529464e177afa65013adca0d63eb5 -- src/cli/gateway-cli/run-loop.ts src/gateway/server-methods/config.ts src/gateway/server-methods/update.ts src/infra/restart.ts ui/src/ui/app-gateway.ts`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
