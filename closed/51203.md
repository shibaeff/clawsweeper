## Summary

Current `main` already has an out-of-band approval layer that pauses execution, routes approval requests through gateway/operator channels, and binds approved exec requests to a canonical stored plan so later agent-side mutations cannot bypass the decision. The shipped release `v2026.4.22` also contains this approval system, including iOS approval delivery and UI review flows.

## Evidence

- **Exec approvals are externalized through the gateway and replayable to operator clients:** The docs describe a separate approval lifecycle: the gateway broadcasts `exec.approval.requested`, operator clients resolve with `exec.approval.resolve`, and approved node exec reuses a stored canonical `systemRunPlan` instead of trusting later caller edits.

- **Agent execution blocks pending approval:** The exec approval requester uses two-phase registration and then waits on `exec.approval.waitDecision`; the comments explicitly call out registering server-side first and waiting for the final decision before proceeding.
  - file: [src/agents/bash-tools.exec-approval-request.ts:89](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/bash-tools.exec-approval-request.ts#L89)

## Risks / Open Questions

- The request argues for broader policy coverage across all destructive actions; OpenClaw already has the approval architecture, but non-exec/non-plugin side effects still depend on the relevant tool or plugin opting into that approval path.
