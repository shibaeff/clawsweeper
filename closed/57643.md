## Summary

Current `main` routes unresolved-model misses to structured `Unknown model` / `model_not_found` errors before `normalizeResolvedModel()` is reached, so the reported crash path does not match the present code.

## Evidence

- **Guarded normalizeResolvedModel call sites:** All direct callers in `model.ts` prove a concrete model exists before normalizing: inline matches require `?.api`, registry matches require `if (model)`, dynamic runtime models bail out on `if (!pluginDynamicModel) return undefined`, and the fallback path constructs a full model object before calling `normalizeResolvedModel()`.
  - file: [src/agents/pi-embedded-runner/model.ts:425](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/pi-embedded-runner/model.ts#L425)

  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Resolution misses return errors, not undefined models:** `resolveModelWithRegistry()` returns `undefined` on a miss, and both `resolveModel()` and `resolveModelAsync()` convert that into `error: buildUnknownModelError(...)` rather than passing an undefined model deeper into normalization.
  - file: [src/agents/pi-embedded-runner/model.ts:648](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/pi-embedded-runner/model.ts#L648)
