## Summary

The proposal is coherent and potentially useful, but current OpenClaw scope docs route optional integrations, community-maintained capabilities, and discoverability to external plugins/skills on ClawHub rather than OpenClaw core or a new org-owned core project.

## Risks / Open Questions

- Maintainers could still choose to make a policy exception and incubate an org-owned protocol project later, but that would be a roadmap decision against the current documented plugin/ClawHub direction.
- The proposal also asks for a docs/community-standard link; if maintainers want that kind of recognition in the future, they may need a dedicated policy for external protocol/spec listings beyond the current ClawHub/plugin discovery flow.

## Close Comment

Thanks for the RFC. I reviewed this against current `main` as a Codex triage pass. The proposal looks like a real ecosystem idea, but it fits the repo’s external-extension path rather than OpenClaw core/org adoption right now: `VISION.md:54-70` says optional capability should usually ship as plugins, plugin authors should host and maintain them in their own repo, and new skills should go to ClawHub first; `docs/plugins/community.md:9-16` says third-party capabilities belong on ClawHub/npm and specifically says not to use OpenClaw docs PRs just for discoverability; `docs/tools/clawhub.md:10-16` defines ClawHub as the public registry for OpenClaw skills and plugins. Current multi-agent support is already documented as narrow and opt-in (`docs/concepts/multi-agent.md:409-414`), and there is no LCP integration on `main` at `34896839ba22f3a71a3d5a8fb4ca5f4c6989a534`. On that basis, this is better treated as a community plugin/skill or external protocol project published via ClawHub, not as a core OpenClaw issue.
