## Summary

Current `main` already routes Slack native `/stop` to the active chat session and the generic stop handler aborts that target session instead of treating the Slack command as a separate prompt.

## Evidence

- **Slack native commands carry the active target session:** The Slack slash handler resolves both a slash-session key and a `commandTargetSessionKey` from the routed chat session, then writes `CommandTargetSessionKey` into the inbound context for native commands.
  - file: [extensions/slack/src/monitor/slash.ts:560](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/extensions/slack/src/monitor/slash.ts#L560)

  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **`/stop` aborts the target session:** `handleStopCommand` resolves the abort target from `CommandTargetSessionKey`, clears queued followups for that target, applies the abort, and returns the abort reply instead of continuing as a normal prompt.
  - file: [src/auto-reply/reply/commands-session-abort.ts:129](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/src/auto-reply/reply/commands-session-abort.ts#L129)
