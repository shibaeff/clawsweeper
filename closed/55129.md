## Summary

Current `main` does not match the report’s cited `200_000` extraction cap. Feishu inbound file attachments still flow into media-understanding, but extracted file text is clamped to `60_000` chars before file blocks are appended to the prompt, and the latest release already contains the same cap. The specific reported reproduction path is therefore obsolete on current code.

## Close Comment

Thanks for the report. I reviewed this with Codex against current `main` and could not match the reported reproduction path. Feishu inbound file attachments still flow through `extensions/feishu/src/bot-content.ts:389` into media understanding, but extracted file text is clamped to `DEFAULT_INPUT_FILE_MAX_CHARS = 60_000` in `src/media/input-files.ts:112`, enforced by `src/media/input-files.ts:225` and applied before prompt assembly at `src/media/input-files.ts:385` and `src/media/input-files.ts:393`. The resulting file blocks are then appended in `src/media-understanding/apply.ts:565`.

The issue body specifically cites a current-main `200_000` cap in `src/media/input-files.ts`, but that is not what current `main` contains, and the same `60_000` cap is already present in the latest release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`v2026.4.22`). On that basis, this Codex review is closing the item as not reproducible on current code.

If you can still reproduce this on a current build, please open a fresh issue with the exact model, the effective `gateway.http.endpoints.responses.files.maxChars` setting, and the request/body sizes involved.
