## Summary

Current main already uses on-demand skill loading: the system prompt injects only an `<available_skills>` catalog and tells the model to `read` a selected skill’s `SKILL.md` only after matching it.

## Evidence

- **System prompt requires selecting a skill before reading it:** The Skills section says to scan `<available_skills>` first, then read exactly one `SKILL.md` at `<location>` only if a skill clearly applies.
  - file: [src/agents/system-prompt.ts:156](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/system-prompt.ts#L156)

  - file: [src/agents/skills/skill-contract.ts:44](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/skills/skill-contract.ts#L44)

## Risks / Open Questions

- The current catalog still includes per-skill descriptions, so there is remaining token overhead proportional to the number of visible skills; the request seems satisfied on the core lazy-load point, not necessarily on eliminating catalog cost entirely.
