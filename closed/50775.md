## Summary

Current `main` already persists Control UI image uploads to inbound media storage and injects the saved paths into the user turn metadata via `MediaPath`/`MediaPaths`; the reported missing-path behavior is no longer the current implementation.

## Evidence

- **Control UI still sends pasted and file-selected images through `chat.send`:** Pasted images are captured in `handlePaste`, file-picker images in `handleFileSelect`, and both are converted to attachment payloads for the `chat.send` RPC.
  - file: [ui/src/ui/views/chat.ts:206](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/ui/src/ui/views/chat.ts#L206)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Control UI serializes image attachments into RPC payloads:** `buildApiAttachments()` turns UI image data URLs into `{ type: "image", mimeType, content }` objects and `requestChatSend()` passes them as `attachments` on `chat.send`.
  - file: [ui/src/ui/controllers/chat.ts:213](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/ui/src/ui/controllers/chat.ts#L213)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Gateway persists inbound chat attachments and records media paths:** `persistChatSendImages()` saves inbound images to the `inbound` media store, and `resolveChatSendTranscriptMediaFields()` builds `MediaPath`, `MediaPaths`, `MediaType`, and `MediaTypes` from the saved files. `emitUserTranscriptUpdate()` and `rewriteChatSendUserTurnMediaPaths()` attach those fields to the user turn metadata.
  - file: [src/gateway/server-methods/chat.ts:676](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/server-methods/chat.ts#L676)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Regression tests assert the fixed behavior:** `chat.directive-tags.test.ts` checks that chat.send attachment sends save files under the inbound media store and that the emitted user transcript update contains `MediaPath`/`MediaPaths` for both inline and offloaded images.
  - file: [src/gateway/server-methods/chat.directive-tags.test.ts:1679](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/server-methods/chat.directive-tags.test.ts#L1679)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **The fix shipped in the latest release tag:** The same attachment-persistence and transcript-media-path code is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this is not just unreleased `main` behavior.
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/gateway/server-methods/chat.ts | rg -n "persistChatSendImages|resolveChatSendTranscriptMediaFields|rewriteChatSendUserTurnMediaPaths|MediaPath|MediaPaths"`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Risks / Open Questions

- The current implementation exposes attachment paths as `MediaPath`/`MediaPaths` on the user turn metadata, not as an `attachments` array inside the untrusted sender metadata block shown in the issue body.
