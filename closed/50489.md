## Summary

Current OpenClaw already routes proactive WhatsApp sends through the same controller-backed listener used by the inbound monitor, and it has regression coverage for the duplicate-module/listener-registry split described in the issue.

## Evidence

- **Inbound monitor creates one send-capable listener:** `attachWebInboxToSocket()` returns the inbound monitor listener with the outbound IPC surface (`sendMessage`, `sendPoll`, `sendReaction`, `sendComposingTo`) on the same object, so inbound handling and proactive sends are no longer separate listener implementations.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:754](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/whatsapp/src/inbound/monitor.ts#L754)
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Controller registers that live listener for outbound resolution:** After the listener is created, the connection controller stores it as the active connection and registers the controller for the account. Outbound send resolution reads from that controller-backed registry.
  - file: [extensions/whatsapp/src/connection-controller.ts:367](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/whatsapp/src/connection-controller.ts#L367)
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Outbound send path uses the registered controller:** `requireOutboundActiveWebListener()` resolves outbound WhatsApp sends from `getRegisteredWhatsAppConnectionController(...).getActiveListener()`, matching the controller registration path instead of a separate ad hoc listener map.
  - file: [extensions/whatsapp/src/send.ts:39](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/whatsapp/src/send.ts#L39)
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Regression test covers duplicate-module registry split:** There is a dedicated test asserting that registered controllers are shared across duplicate module instances, which is the exact failure mode behind 'inbound works but proactive send sees no active listener' after module duplication/cache splits.
  - file: [extensions/whatsapp/src/connection-controller-registry.test.ts:11](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/whatsapp/src/connection-controller-registry.test.ts#L11)

  - file: [extensions/whatsapp/src/connection-controller.test.ts:96](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/whatsapp/src/connection-controller.test.ts#L96)

## Close Comment

Closing this as a Codex review because the listener-registry split described here is already implemented away on current OpenClaw. On current `main`, the inbound WhatsApp monitor returns a single send-capable listener object in `extensions/whatsapp/src/inbound/monitor.ts:754`, the connection controller registers that live listener in `extensions/whatsapp/src/connection-controller.ts:367`, and proactive sends resolve that same registered controller in `extensions/whatsapp/src/send.ts:39`. There is explicit regression coverage for duplicate module instances in `extensions/whatsapp/src/connection-controller-registry.test.ts:11` and reconnect handoff in `extensions/whatsapp/src/connection-controller.test.ts:96`. I also compared the latest release snapshot to current `main`; the controller-registry behavior is already present there as well. If you still see a WhatsApp send failure on a newer build, please open a fresh issue with the exact version, restart path, and gateway logs, because that would be a different bug than the listener-registry divergence reported here.
