## Summary

Current main already resolves a `sessions_spawn` model choice through the gateway `sessions.patch` path, which canonicalizes aliases via `resolveAllowedModelRef(...)` before storing the child session override. That means alias inputs no longer silently fall back to the default model as described in the report.

## Evidence

- **sessions_spawn writes the child model through sessions.patch:** The subagent spawn flow builds `initialSessionPatch` from the requested model and sends it with `callSubagentGateway({ method: "sessions.patch", ... })`, so alias handling is delegated to the gateway patch path rather than used raw at run start.
  - file: [src/agents/subagent-spawn.ts:730](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/subagent-spawn.ts#L730)

- **gateway patch resolves aliases to canonical model refs:** `applySessionsPatchToStore()` loads the model catalog and calls `resolveAllowedModelRef({ raw: trimmed, ... })` for `patch.model`, which is the shared resolver that uses the configured alias index from `agents.defaults.models`.
  - file: [src/gateway/sessions-patch.ts:411](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/sessions-patch.ts#L411)

  - file: [src/sessions/model-overrides.ts:40](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/sessions/model-overrides.ts#L40)
