## Summary

Current `main` already fixes the Feishu dual-module runtime-store failure by making plugin runtime stores global and plugin-keyed, rather than by removing the Windows native-Jiti guard. The Feishu plugin now uses that keyed store, and `src/plugin-sdk/runtime-store.test.ts` adds a regression test that simulates duplicate module instances and proves the runtime is shared across them.

## Evidence

- **Global runtime-store registry on current main:** `createPluginRuntimeStore` now uses a `globalThis` registry keyed by plugin id / explicit key, so duplicate module instances converge on the same runtime slot instead of each keeping a private closure.

## Risks / Open Questions

- This was a read-only code review; I did not run a Windows gateway repro in this environment. Confidence comes from the current implementation and the targeted duplicate-module regression test.
- The review only supports closing this Feishu crash report. Similar plugins would need the plugin-id/global-store path as well; legacy string-key runtime stores remain module-local by design.
