## Summary

Keep open. Current `main` does not already implement Azure Speech TTS, so `implemented_on_main` is not supported. The PR is also not incoherent or obsolete enough for the other close reasons: the feature is still plausible, but its patch shape no longer matches current main because speech providers are now plugin-owned and configured through the generic `messages.tts.providers` map rather than new core-specific `messages.tts.azure` wiring.

## Evidence

- **Speech providers are resolved from plugin capability registrations on main:** `src/tts/provider-registry.ts` no longer hard-codes built-in provider builders; it resolves `speechProviders` from the plugin capability registry, so adding `src/tts/providers/azure.ts` and importing it there no longer matches the current architecture.
  - file: [src/tts/provider-registry.ts:16](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/tts/provider-registry.ts#L16)
  - command: `nl -ba src/tts/provider-registry.ts | sed -n '1,120p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **TTS config is now a generic provider map:** `TtsConfig` uses `providers?: Record<string, Record<string, unknown>>` for provider-specific settings. Current main does not expose a dedicated top-level `messages.tts.azure` shape like this PR adds.
  - file: [src/config/types.tts.ts:26](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/config/types.tts.ts#L26)
