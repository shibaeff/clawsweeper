## Summary

Keep open. Current `main` added the Google video provider on April 25, 2026, and its download path still assumes `generatedVideos[*].video` can be handed to `@google/genai` `files.download`. Upstream SDK 1.50.1 resolves a `Video.uri` into `files/${uri}:download` instead of fetching the URI directly, which matches the reported 404 failure mode for MLDev Veo responses. OpenClaw also lacks a regression test for the `video.uri` case.

## Evidence

- **Current OpenClaw download path:** `generateVideo()` reads `operation.response?.generatedVideos`, returns inline bytes when `videoBytes` exists, and otherwise passes `entry.video` to `downloadGeneratedVideo()`, which calls `client.files.download({ file: params.file, downloadPath })` with no special handling for direct `video.uri` responses.
  - file: [extensions/google/video-generation-provider.ts:128](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/extensions/google/video-generation-provider.ts#L128)
  - command: `nl -ba extensions/google/video-generation-provider.ts | sed -n '120,250p'`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Upstream SDK contract:** OpenClaw pins `@google/genai` `^1.50.1`. The SDK types say `DownloadFileParameters.file` accepts `string | File | GeneratedVideo | Video`, so passing a `Video` object is intentional, not a caller typo.
  - file: [extensions/google/package.json:8](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/extensions/google/package.json#L8)
  - command: `rg -n "@google/genai" extensions/google/package.json pnpm-lock.yaml`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Upstream SDK implementation mismatch:** In `@google/genai` 1.50.1, `tFileName()` turns a `Video` into `video.uri`, and `downloadFile()` then requests `files/${name}:download?alt=media`. For a full MLDev download URL, that becomes an invalid Files API path instead of a direct fetch, which explains a 404. The fallback branches for `Video`/`GeneratedVideo` only return inline `videoBytes`; they do not download from `uri`.
  - command: `curl -L --silent https://registry.npmjs.org/@google/genai/-/genai-1.50.1.tgz | tar -xzOf - package/dist/node/index.mjs | sed -n '19400,19435p'`
  - sha: [v1.50.1](https://github.com/openclaw/openclaw/commit/v1.50.1)
- **Missing regression coverage:** The Google provider tests cover inline `videoBytes`, mixed input rejection, and duration rounding, but there is no test where the completed operation returns `generatedVideos[*].video.uri` and requires a download.
  - file: [extensions/google/video-generation-provider.test.ts:42](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/extensions/google/video-generation-provider.test.ts#L42)
  - command: `nl -ba extensions/google/video-generation-provider.test.ts | sed -n '1,220p'`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Issue discussion adds concrete workaround:** The reporter supplied a focused follow-up patch that keeps the existing file-handle path and adds a direct-fetch branch for `inline.uri`. That comment strengthens the report because it isolates the failure to the MLDev direct-URI case instead of a generic auth or polling problem.
- **User-facing surface is real on main:** Docs already advertise `Google (Gemini / Veo)` under `video_generate`, so this is an exposed current-main capability rather than a speculative future integration.
  - file: [docs/tools/video-generation.md:294](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/docs/tools/video-generation.md#L294)

## Risks / Open Questions

- The exact 404 may originate inside `@google/genai`, but OpenClaw currently relies on that behavior without compensating for MLDev direct-download URIs.
- The Google video provider was introduced after the latest release, so the bug may affect current `main` more than the last shipped build; still not safe to close because the reported code path remains present.
