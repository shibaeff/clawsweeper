## Summary

Current main shows this report is caused by using the root CLI `--profile` flag, which intentionally switches OpenClaw to a profile-specific state/config dir like `~/.openclaw-<profile>`. The browser-specific selector is `--browser-profile`, and current docs/tests make that split explicit.

## Evidence

- **Root `--profile` intentionally rewrites state/config paths:** `parseCliProfileArgs` treats `--profile` as a root CLI option, and `applyCliProfileEnv` sets `OPENCLAW_PROFILE`, then defaults `OPENCLAW_STATE_DIR` to `~/.openclaw-<profile>` and `OPENCLAW_CONFIG_PATH` to `<stateDir>/openclaw.json`. Passing `--profile openclaw` therefore intentionally resolves `~/.openclaw-openclaw/openclaw.json`.
  - file: [src/cli/profile.ts:35](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cli/profile.ts#L35)
  - command: `nl -ba src/cli/profile.ts | sed -n '1,140p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Profile env is applied before command dispatch:** `runCli` parses root profile args and calls `applyCliProfileEnv` before dotenv loading and before the CLI routes subcommands, so `openclaw browser ... --profile openclaw` will already be running against the profile-specific env/config by design.
  - file: [src/cli/run-main.ts:156](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cli/run-main.ts#L156)
  - command: `nl -ba src/cli/run-main.ts | sed -n '148,190p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Browser command uses `--browser-profile`, not `--profile`:** The browser CLI registers `--browser-profile <name>` as the browser profile selector. The public browser docs also use `openclaw browser --browser-profile openclaw start` in examples.
  - file: [extensions/browser/src/cli/browser-cli.ts:20](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/browser/src/cli/browser-cli.ts#L20)
  - command: `nl -ba extensions/browser/src/cli/browser-cli.ts | sed -n '1,120p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Current main has an explicit no-conflict test:** The browser CLI test asserts that the global `--profile` flag and the browser `--browser-profile` flag are separate and do not conflict, reinforcing that the observed `.openclaw-openclaw` path comes from the root profile feature rather than browser-profile resolution.
  - file: [extensions/browser/src/cli/browser-cli.test.ts:44](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/browser/src/cli/browser-cli.test.ts#L44)
  - command: `nl -ba extensions/browser/src/cli/browser-cli.test.ts | sed -n '1,90p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Public docs show the intended flag:** The browser CLI reference lists `--browser-profile <name>` under common flags and uses it in quick-start commands.
  - file: [docs/cli/browser.md:18](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/docs/cli/browser.md#L18)
  - command: `nl -ba docs/cli/browser.md | sed -n '1,80p'`

## Risks / Open Questions

- The CLI UX is still easy to misread because `--profile` and `--browser-profile` are both valid, so users can still make the same mistake even though current behavior is intentional.
