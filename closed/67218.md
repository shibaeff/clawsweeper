## Summary

Current `main` already applies configured HTTP headers to remote SSE MCP connections, including the EventSource fetch path, and the same transport code is present in shipped release `v2026.4.23`. The reported hardcoded-`Accept` behavior does not match current OpenClaw.

## Evidence

- **SSE EventSource fetch overlays user headers last:** `buildSseEventSourceFetch` copies SDK-provided EventSource headers, then spreads configured server headers after them before calling fetch, so a configured `Accept` header is preserved on the SSE stream request.
  - file: [src/agents/mcp-transport.ts:59](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/src/agents/mcp-transport.ts#L59)
  - sha: [c7f18a6b9d76](https://github.com/openclaw/openclaw/commit/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac)
- **SSE transport uses the custom EventSource fetch:** Remote SSE MCP connections are created with both `requestInit.headers` and `eventSourceInit.fetch: buildSseEventSourceFetch(headers)`, so configured headers apply to both HTTP request paths.
  - file: [src/agents/mcp-transport.ts:117](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/src/agents/mcp-transport.ts#L117)
  - sha: [c7f18a6b9d76](https://github.com/openclaw/openclaw/commit/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac)
- **HTTP MCP config keeps arbitrary headers:** HTTP MCP server config accepts a `headers` object and carries it through unchanged into the resolved transport config.
  - file: [src/agents/mcp-http.ts:19](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/src/agents/mcp-http.ts#L19)
  - sha: [c7f18a6b9d76](https://github.com/openclaw/openclaw/commit/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac)

  - file: [src/agents/mcp-transport-config.test.ts:99](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/src/agents/mcp-transport-config.test.ts#L99)
