## Summary

Current main already avoids the reported Ollama catalog bug: Ollama models are built from live discovery with no static `deepseek-v3.2`/`supportsTools: false` entry, and provider config overrides now preserve `compat` metadata. The remaining `deepseek-v3.2 supportsTools: false` entry is Venice-specific and documented as such.

## Evidence

- **Ollama uses dynamic discovery, not a static `deepseek-v3.2` catalog flag:** `buildOllamaProvider()` fetches `/api/tags`, enriches models with `/api/show`, and maps them through `buildOllamaModelDefinition()`. That definition sets `id`, `name`, `reasoning`, `input`, `cost`, `contextWindow`, and `maxTokens`, but does not set `supportsTools: false` or any equivalent compat flag.
  - file: [extensions/ollama/src/provider-models.ts:214](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/extensions/ollama/src/provider-models.ts#L214)

  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **No Ollama-side `deepseek-v3.2` or `supportsTools` entry remains:** A repo search under `extensions/ollama` returned no matches for `deepseek-v3.2` or `supportsTools`, so the issue's reported static Ollama catalog entry is not present on current main.
  - command: `rg -n "deepseek-v3\.2|supportsTools" extensions/ollama`
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
