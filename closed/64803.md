## Summary

Current `main` does not rely on the Discord `<media:audio>` placeholder to trigger inbound audio transcription. The Discord monitor now carries downloaded attachment MIME into `MediaType`/`MediaTypes`, reply processing runs media understanding whenever any inbound media is present, and media-understanding can also classify audio by filename extension. The reported `requireMention: false` failure mode therefore does not match current `main`, even though the placeholder helper still keys off `attachment.content_type`.

## Evidence

- **Discord attachment ingest preserves detected MIME, not just Discord metadata:** Inbound Discord attachments are fetched with `filePathHint: attachment.filename ?? attachment.url`, then saved with `fetched.contentType ?? attachment.content_type`, and the resulting media entry carries `saved.contentType` forward. That means a voice note with a usable filename or detectable body can still become `audio/ogg` even if `attachment.content_type` is missing or wrong.
  - file: [extensions/discord/src/monitor/message-utils.ts:385](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/discord/src/monitor/message-utils.ts#L385)
  - command: `nl -ba extensions/discord/src/monitor/message-utils.ts | sed -n '384,417p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Remote media MIME is inferred from buffer plus filename hint:** `fetchRemoteMedia` derives `contentType` via `detectMime({ buffer, headerMime, filePath: filePathForMime })`, where `filePathForMime` prefers the filename hint. This is the current main-path mechanism that recovers audio MIME from `.ogg` filenames or sniffed bytes.
  - file: [src/media/fetch.ts:258](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/media/fetch.ts#L258)
  - command: `nl -ba src/media/fetch.ts | sed -n '254,276p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Discord inbound context always includes media payload:** After resolving attachments, the Discord monitor spreads `...mediaPayload` into the finalized inbound context. Audio transcription therefore depends on the media payload, not on the raw placeholder text alone.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:488](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/discord/src/monitor/message-handler.process.ts#L488)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.process.ts | sed -n '488,529p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Reply pipeline runs media understanding for any inbound media:** `getReplyFromConfig` calls media understanding whenever `hasInboundMedia(ctx)` is true. That gate checks for media paths/URLs/types, so transcription is not contingent on the body being exactly `<media:audio>`.
  - file: [src/auto-reply/reply/get-reply.ts:131](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/auto-reply/reply/get-reply.ts#L131)
  - command: `nl -ba src/auto-reply/reply/get-reply.ts | sed -n '131,143p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Media-understanding still falls back to audio filename extensions:** When MIME is absent, attachment classification falls back to file extension and treats `.ogg`/`.opus` names as audio. This provides a second current-main path for Discord voice notes to reach audio transcription.
  - file: [src/media-understanding/attachments.normalize.ts:79](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/media-understanding/attachments.normalize.ts#L79)

## Risks / Open Questions

- `extensions/discord/src/monitor/preflight-audio.ts:18-25` still filters preflight candidates only by `attachment.content_type`, so mention-gated voice-message detection may still have a separate edge case when Discord omits MIME.
- I did not run a live Discord repro in this read-only review, and there is no Discord-specific regression test in current main that explicitly covers missing `attachment.content_type` on inbound voice messages.
