## Summary

Current main already resolves the freshest matching session-store row and reads session history from the resolved transcript path, which covers the reported symptom where a transcript exists on disk but `sessions_history`/`sessions_list(...messageLimit=...)` surface empty history.

## Evidence

- **Freshest duplicate session row wins:** `loadSessionEntry()` resolves all candidate store keys for a logical session and returns the freshest matching entry by `updatedAt`, instead of blindly using one stale row.
  - file: [src/gateway/session-utils.ts:417](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/gateway/session-utils.ts#L417)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **History reads from resolved transcript entry:** `chat.history` loads the resolved session entry and reads transcript messages from `readSessionMessages(sessionId, storePath, entry?.sessionFile)`, so history is sourced from the selected transcript path.
  - file: [src/gateway/server-methods/chat.ts:1891](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/gateway/server-methods/chat.ts#L1891)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Transcript fallback scans candidate paths:** `readSessionMessages()` resolves transcript candidates from the session id plus the persisted `sessionFile` and returns the first existing transcript, rather than requiring one single fixed location.
  - file: [src/gateway/session-utils.fs.ts:93](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/gateway/session-utils.fs.ts#L93)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **`sessions_list(messageLimit=...)` uses the same history path:** When `messageLimit > 0`, `sessions_list` fetches per-row messages by calling `chat.history`, so the history fix applies to both APIs named in the report.
  - file: [src/agents/tools/sessions-list-tool.ts:338](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/agents/tools/sessions-list-tool.ts#L338)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Direct regression coverage exists:** There is an explicit regression test proving direct history reads prefer a fresh transcript over a stale duplicate row and return `fresh history` for the logical session key.
  - file: [src/gateway/sessions-history-http.test.ts:310](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/gateway/sessions-history-http.test.ts#L310)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Shipped release contains the regression test:** The same direct-history regression test is present in release tag `v2026.4.22`, so this behavior is not only on unreleased main.
  - command: `git grep -n "prefers the freshest duplicate row for direct history reads" v2026.4.22 -- src/gateway/sessions-history-http.test.ts`
