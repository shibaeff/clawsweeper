## Summary

Current `main` does not match the reported Telegram startup race. Telegram token resolution is now account-aware, prefers configured `botToken`/`tokenFile` over env, treats configured-but-unavailable credentials as not configured, and throws an explicit missing-token error instead of silently skipping startup. The issue discussion requested a current reproduction with `openclaw channels status --probe`, token source, and exact version/commit, but no follow-up data was provided.

## Evidence

- **Issue discussion already narrowed the mismatch:** A maintainer comment on the issue states the reported late-env root cause does not match current `main`, notes that Telegram now prefers configured account/channel tokens before env fallback, and asks for `openclaw channels status --probe`, credential source, and exact reproducing release/commit. No follow-up reproduction data was added afterward.
- **Current token resolution is config-first and env-last:** `resolveTelegramToken` checks per-account `tokenFile` and `botToken`, then channel-level `tokenFile` and `botToken`, and only then falls back to `TELEGRAM_BOT_TOKEN`, with env fallback limited to the default account.
