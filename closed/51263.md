Current main already resolves message-tool channel SecretRefs through the command secret resolver with channel/account scoping, and doctor now uses read-only account inspection instead of strict token resolution on this path.

## Close Comment

Closing from a Codex review because current `main` already implements this. The message CLI now scopes channel/account SecretRef resolution before dispatch (`src/commands/message.ts:34`), and the agent `message` tool does the same before invoking outbound actions (`src/agents/tools/message-tool.ts:738`). Both paths go through the command-secret resolver, which calls gateway `secrets.resolve` and applies the active runtime snapshot into a cloned config (`src/cli/command-secret-gateway.ts:654`). The scope narrowing is explicit in `src/cli/command-secret-targets.ts:174`, and there is regression coverage asserting that a Discord run only resolves `channels.discord.*` plus the selected account path (`src/agents/tools/message-tool.test.ts:256`). The user-facing docs now describe that behavior too (`docs/cli/message.md:54`). On the doctor side, the current code uses read-only account inspection instead of strict token resolution for this path (`src/commands/doctor-security.ts:311`), and Discord inspection marks configured SecretRefs as `configured_unavailable` rather than throwing (`extensions/discord/src/account-inspect.ts:27`). Verified against current main `608c08fc54da2f86338366f8eeafd7e2a02713aa`.
