## Close Comment

Closing this as implemented after Codex review.

Current main already avoids the reported manual isolated cron self-deadlock by remapping nested cron-triggered agent work off the `cron` lane, so this PR’s separate `cron-dispatch` lane is obsolete.

What I checked:

- **Manual cron runs still enter the cron lane:** `enqueueRun` queues manual `cron.run` work onto `CommandLane.Cron`, so the only remaining question is whether isolated execution re-enters that same lane. (`src/cron/service/ops.ts:697`)
- **Gateway still tags isolated cron runs as cron-lane work:** `runIsolatedAgentJob` passes `lane: "cron"` into `runCronIsolatedAgentTurn`, matching the path described in the PR. (`src/gateway/server-cron.ts:333`)
- **Current main remaps nested cron work away from the cron lane:** `resolveNestedAgentLane` explicitly maps `cron` to `nested` with a comment explaining this avoids cron-lane self-deadlock while cron jobs dispatch inner work. (`src/agents/lanes.ts:8`)
- **Isolated cron executor applies the nested-lane remap:** `runCronIsolatedAgentTurn` calls `runEmbeddedPiAgent` with `lane: resolveNestedAgentLane(params.lane)`, so inner isolated execution no longer re-enters the outer cron lane. (`src/cron/isolated-agent/run-executor.ts:153`)
- **Shipped release note matches this fix:** The changelog entry for `2026.3.13` says: `Cron/isolated sessions: route nested cron-triggered embedded runner work onto the nested lane so isolated cron jobs no longer deadlock when compaction or other queued inner work runs.`. (`CHANGELOG.md:2555`)

So I’m closing this as already implemented rather than keeping a duplicate issue open.
