## Summary

Keep open. Current `main` still allows `cron.update` callers to send a cron schedule patch without `tz`, and `applyJobPatch` replaces the stored cron schedule instead of merging the existing timezone. When that happens, later schedule evaluation falls back to the system timezone, which is still consistent with the reported restart catch-up misfires.

## Evidence

- **Cron update schema makes `tz` optional on cron schedule patches:** The gateway protocol accepts `patch.schedule` objects with `kind: "cron"` and `expr`, while `tz` remains optional, so callers can legally update a cron schedule without resending the timezone.
  - file: [src/gateway/protocol/schema/cron.ts:119](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/gateway/protocol/schema/cron.ts#L119)
  - command: `nl -ba src/gateway/protocol/schema/cron.ts | sed -n '103,128p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **`cron.update` forwards the normalized patch directly into the cron service:** The gateway handler normalizes `params.patch`, validates it, and then calls `context.cron.update(jobId, patch)` without reconstructing the existing schedule or backfilling a missing `tz`.
  - file: [src/gateway/server-methods/cron.ts:250](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/gateway/server-methods/cron.ts#L250)
  - command: `nl -ba src/gateway/server-methods/cron.ts | sed -n '250,325p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **`applyJobPatch` replaces cron schedules instead of preserving the previous timezone:** When `patch.schedule.kind === "cron"`, the service assigns `job.schedule = { ...patch.schedule, ... }` or `patch.schedule` directly. It preserves `staggerMs`, but it does not merge or retain the existing `job.schedule.tz`.
  - file: [src/cron/service/jobs.ts:627](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cron/service/jobs.ts#L627)
  - command: `nl -ba src/cron/service/jobs.ts | sed -n '627,639p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Missing `tz` still falls back to the local/system timezone:** Cron evaluation resolves the timezone from `schedule.tz`, but if it is missing it falls back to `Intl.DateTimeFormat().resolvedOptions().timeZone`. That means a dropped `tz` changes future next/previous-run math.
  - file: [src/cron/schedule.ts:9](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cron/schedule.ts#L9)
  - command: `nl -ba src/cron/schedule.ts | sed -n '9,15p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Restart catch-up replay logic depends on the stored schedule:** Startup catch-up uses `computeJobPreviousRunAtMs(job, nowMs)` to decide whether a cron slot was missed. If the stored schedule lost its `tz`, this path will evaluate against the fallback timezone instead of the job's original timezone.
  - file: [src/cron/service/timer.ts:903](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cron/service/timer.ts#L903)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '903,920p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Issue discussion already pointed at the right fix area:** The issue timeline cross-references closed PR #61086 titled `fix(cron): preserve tz in applyJobPatch to prevent spurious restart catch-up fires`, which matches the still-unfixed merge gap in the current `applyJobPatch` implementation.
