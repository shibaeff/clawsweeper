## Close Comment

Closing this as implemented after Codex review.

Current `main` already has a broader Telegram polling-stall hardening implementation, so PR #41011 is superseded. The code now uses a default 120s watchdog threshold with bounded config overrides, tracks `getUpdates` plus other in-flight API activity before declaring a stall, and has focused tests/docs covering the false-positive restart cases this PR targets.

What I checked:

- **Current stall threshold + watchdog path on main:** `extensions/telegram/src/polling-session.ts:25-28` defines a 120s default threshold and 30s watchdog interval; `extensions/telegram/src/polling-session.ts:55-63` clamps configurable overrides; `extensions/telegram/src/polling-session.ts:246-340` routes all Telegram API calls through a liveness tracker and only restarts when `detectStall` reports a real stall. ([`extensions/telegram/src/polling-session.ts:25`](https://github.com/openclaw/openclaw/blob/b0d2003fe65b/extensions/telegram/src/polling-session.ts#L25), [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65b))
