## Summary

Current `main` already implements unique-prefix exec approval lookup in a safer form than this PR: prefix matching is limited to approval lookup paths, resolved events use the canonical full id, duplicate explicit-id checks stay exact, and `system.run` replay checks still require the exact run id. The same behavior and tests are present in shipped release `v2026.4.23`.

## Close Comment

Closing this as implemented after Codex review.

Current `main` already implements unique-prefix approval resolution, uses the canonical matched id when resolving/broadcasting, preserves exact matching for duplicate-id and `system.run` replay checks, and includes tests for both unique and ambiguous prefixes. The behavior is also present in the shipped `v2026.4.22` release.

What I checked:

- **Prefix lookup already exists on main:** `ExecApprovalManager.lookupPendingId()` resolves pending approval ids by exact match first, then unique prefix, and returns an explicit ambiguous result when multiple pending ids share the prefix. ([`src/gateway/exec-approval-manager.ts:185`](https://github.com/openclaw/openclaw/blob/350d3221800a/src/gateway/exec-approval-manager.ts#L185), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Resolve path uses canonical full id:** `resolvePendingApprovalRecord()` converts user input into the canonical approval id, and `handleApprovalResolve()` passes that canonical id into `manager.resolve()` and the emitted resolved event payload. This covers the PR’s intended UX without leaving stale pending entries behind. ([`src/gateway/server-methods/approval-shared.ts:69`](https://github.com/openclaw/openclaw/blob/350d3221800a/src/gateway/server-methods/approval-shared.ts#L69), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Exact-only checks were intentionally preserved:** Creation-time duplicate-id validation still checks `manager.getSnapshot(explicitId)` exactly, and `system.run` approval replay still uses exact `getSnapshot(runId)` / `consumeAllowOnce(runId)` instead of prefix matching. ([`src/gateway/server-methods/exec-approval.ts:217`](https://github.com/openclaw/openclaw/blob/350d3221800a/src/gateway/server-methods/exec-approval.ts#L217), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **System.run replay remains exact:** `sanitizeSystemRunParamsForForwarding()` looks up approvals by exact `runId` and consumes allow-once approvals by the same exact `runId`, avoiding the regression called out in PR review. ([`src/gateway/node-invoke-system-run-approval.ts:137`](https://github.com/openclaw/openclaw/blob/350d3221800a/src/gateway/node-invoke-system-run-approval.ts#L137), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Tests cover the shipped behavior:** Gateway tests assert unique short prefixes resolve successfully and ambiguous prefixes are rejected; there is also a regression test ensuring a resolved exact id is not later reused as another approval’s prefix. ([`src/gateway/server-methods/server-methods.test.ts:929`](https://github.com/openclaw/openclaw/blob/350d3221800a/src/gateway/server-methods/server-methods.test.ts#L929), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Behavior is present in the latest release:** The `v2026.4.22` release commit contains the same `lookupPendingId()` implementation and prefix-resolution tests, so this is already shipped rather than only present on unreleased `main`. ([00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376))

So I’m closing this as already implemented rather than keeping a duplicate issue open.
