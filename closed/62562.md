## Summary

Current `main` removes the stale-resume path described in #62562: `sessions_spawn` now creates a fresh UUID-backed child session key, persists the requested runtime model/provider onto that child session before starting the run, and only reports `modelApplied: true` after that persistence succeeds. Regression tests cover both the fresh-session shape and the persist-before-run ordering.

## Evidence

- **Fresh child sessions are UUID-backed, not resumed by bare agentId:** `spawnSubagentDirect` constructs `childSessionKey` as `agent:${targetAgentId}:subagent:${crypto.randomUUID()}`, which makes each native subagent spawn a fresh session instead of reusing a prior agent-scoped session entry.
  - file: [src/agents/subagent-spawn.ts:686](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/subagent-spawn.ts#L686)
  - command: `nl -ba src/agents/subagent-spawn.ts | sed -n '560,860p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Requested model is persisted before the child run starts:** After the provisional session patch, the spawn path calls `persistInitialChildSessionRuntimeModel(...)`; if that write fails it deletes the provisional session and returns an error. Only after this succeeds does it set `modelApplied = true`, and the later `agent` RPC uses that same fresh `childSessionKey` to start the run.
