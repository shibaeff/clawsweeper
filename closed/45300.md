## Summary

Current `main` appears to cover the reported Discord mention-gating failure: multi-account startup still resolves a separate `botUserId` per account, and the preflight path now hydrates raw Discord messages from REST when a payload contains mention syntax but missing mention metadata, with a regression test for that exact shape. I do not see this shipped in the latest tagged release yet, so this is an `implemented_on_main` close with no fixed release tag.

## Evidence

- **Per-account bot identity is still threaded through the Discord monitor:** `main` fetches the bot identity for each configured Discord account and passes that account's `botUserId` into the message handler, so mention checks are not using one shared global bot id across accounts.
  - file: [extensions/discord/src/monitor/provider.ts:998](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/discord/src/monitor/provider.ts#L998)
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Preflight now hydrates messages when mention metadata is missing:** The Discord preflight path now calls `hydrateDiscordMessageIfNeeded()` before mention gating. It explicitly hydrates when a message contains raw mention tokens like `<@...>` / `@everyone` but the inbound payload has no populated mention metadata, then re-runs gating on the hydrated message.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:327](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/discord/src/monitor/message-handler.preflight.ts#L327)
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Regression test covers the raw-mention/no-metadata failure shape:** Current tests include a case where a guild message contains `<@botId>` while `mentionedUsers` is empty; the mocked REST hydrate returns the mention metadata and preflight is expected to accept the message.
  - file: [extensions/discord/src/monitor/message-handler.preflight.test.ts:598](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/discord/src/monitor/message-handler.preflight.test.ts#L598)
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Change landed after the latest release:** Diffing the latest release against current `main` shows the relevant change from `hydrateDiscordMessageIfEmpty` to the broader `shouldHydrateDiscordMessage` / `hydrateDiscordMessageIfNeeded` logic plus the matching regression test.
  - command: `git diff --unified=20 a9797214338ba31c52c796adbb75afb16e0684a9..0e9156d2053022d208c3eea1d0fa1e377ed0a154 -- extensions/discord/src/monitor/message-handler.preflight.ts extensions/discord/src/monitor/message-handler.preflight.test.ts`

## Risks / Open Questions

- There is still no dedicated multi-account regression test in `extensions/discord/src/monitor/message-handler.preflight.test.ts`; the close is based on the per-account bot-id wiring plus the new raw-mention hydration regression that matches the reported failure mode.
