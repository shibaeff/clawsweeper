## Summary

Current `main` already routes HTTP 400/422 invalid-request/model-resolution failures into the failover pipeline and advances to configured fallback candidates instead of treating them as non-fallback explicit aborts. The reported `2026.3.28` behavior does not match the current code/docs/tests on `fd65caf4b0bad1b0ac27a82d349c22e92167e7bc`.

## Evidence

- **Docs say invalid-request errors are failover-worthy:** The current failover docs explicitly say format/invalid-request errors are treated as failover-worthy, and fallback continues on remaining candidates except true explicit aborts/context-overflow cases.
