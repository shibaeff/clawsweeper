## Summary

PR #44892 does not reproduce a current-main command-injection bug on the path it describes. `src/link-understanding/runner.ts` passes the templated URL as argv into `runExec`, and `src/process/exec.ts` hard-disables shell execution and uses `execFile`, so `$(...)`-style shell expansion is not evaluated for the PR's `command: "curl", args: ["{{LinkUrl}}"]` reproduction. The real risky case is an explicit operator-chosen shell wrapper (`sh -c ...`), which this PR does not model well and which the proposed URL filter would only partially address while also rejecting legitimate URLs.

## Evidence

- **Link-understanding uses argv, not a concatenated shell command:** `runCliEntry` builds `argv` from `command` plus templated args, then calls `runExec(argv[0], argv.slice(1), ...)`. That means the reported `curl {{LinkUrl}}` path is passed as discrete argv entries, not interpolated into a shell command string.

  - file: [src/process/exec.test.ts:94](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/process/exec.test.ts#L94)
