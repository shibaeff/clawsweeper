## Summary

Current `main` no longer depends on a stale one-time cron object capture for memory-core dreaming reconciliation. The plugin now keeps a live cron resolver from `gateway_start`/`ctx.getCron()`, reloads runtime config during heartbeat reconciliation, and has regression tests for the startup-missing-cron and cron-service-swap cases described in #67362. The fix is present on `main` in commit `355c92d69b3803644a6a56247e5e9273bc1d17b9`, after the latest tagged release `v2026.4.23`, so `fixedRelease` is unknown/not yet shipped.

## Evidence

- **Memory-core now uses a live cron accessor on each reconcile:** `registerShortTermPromotionDreaming` stores `startupCron` as a function and calls `resolveStartupCron?.()` during reconciliation, rather than holding a one-time cron service object. Startup now seeds that resolver from `gateway_start` context via `startupCron: () => resolveCronServiceFromGatewayContext(ctx)`.
