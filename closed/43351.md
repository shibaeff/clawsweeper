## Summary

Keep open. Current `main` still has the reported failure mode: the doctor/store migration does not reconstruct flat legacy cron fields into the canonical nested `schedule` shape, while runtime, CLI, and Control UI code still dereference `job.schedule.kind` / `job.payload.kind` without a guard. That matches the issue’s reported `TypeError ... reading 'kind'` and leaves the cron subsystem vulnerable on migrated legacy stores.

## Evidence

- **Doctor migration misses flat legacy schedule/session schema:** `src/commands/doctor-cron-store-migration.ts:319-389` only infers/migrates payload and top-level delivery fields. `src/commands/doctor-cron-store-migration.ts:391-473` only normalizes `raw.schedule` when a nested schedule object already exists; it does not turn top-level legacy fields like `kind` / `cron` / `tz` into `schedule: { kind, expr, tz }`. `src/commands/doctor-cron-store-migration.ts:497-555` infers `sessionTarget`/delivery, but leaves unrelated flat legacy fields in place.
