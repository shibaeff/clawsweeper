## Summary

Issue #48676 does not map to an OpenClaw code path. The report centers on a `cloudcache://` scheme and cloud object storage assumptions that do not exist in this repository; current main handles local media via direct local paths / `file://` and uses `media://inbound/...` only as an internal pseudo-URL for inbound offload.

## Evidence

- **No `cloudcache` implementation in repo or history:** `rg -n "cloudcache://|cloudcache|CloudCache" src extensions docs ui test package.json pnpm-lock.yaml` returned no matches on current checkout, and `git log --all --oneline -S'cloudcache' -- .` returned no commits introducing that term.
  - command: `rg -n "cloudcache://|cloudcache|CloudCache" src extensions docs ui test package.json pnpm-lock.yaml ; git log --all --oneline -S'cloudcache' -- .`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Current main reads local media paths directly:** The outbound/inbound media loader strips `MEDIA:`, accepts `file://`, resolves local filesystem paths, enforces allowed roots, and reads the file directly. There is no conversion to a cloud-backed URL scheme.
  - file: [src/media/web-media.ts:398](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/media/web-media.ts#L398)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **OpenClaw's only media pseudo-URL is `media://inbound/...`:** Large/text-only inbound attachments are offloaded to disk and referenced as `media://inbound/<id>`, explicitly described as an opaque URI decoupled from filesystem layout.
  - file: [src/gateway/chat-attachments.ts:419](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/gateway/chat-attachments.ts#L419)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **`media://inbound/...` is a supported and tested read path:** Tests cover hydrating `media://inbound/<id>` back into readable media and rejecting malformed variants, confirming the internal pseudo-URL behavior on current main.
  - file: [src/media/web-media.test.ts:442](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/media/web-media.test.ts#L442)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Docs describe local media support, not cloud-only storage:** Docs say inbound media exposes `{{MediaPath}}` and `{{MediaUrl}}` and outbound attachments use `MEDIA:<path-or-url>`; local paths are supported under the file-read policy.
  - file: [docs/start/openclaw.md:181](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/docs/start/openclaw.md#L181)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Tests explicitly cover host-local outbound attachments:** `runMessageAction` tests allow host-local image attachment paths when fs root expansion is enabled, which contradicts the report's claim that local media is fundamentally unsupported.
  - file: [src/infra/outbound/message-action-runner.media.test.ts:359](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/infra/outbound/message-action-runner.media.test.ts#L359)
