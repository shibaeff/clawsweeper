## Summary

Current `main` now models WhatsApp auth/link state separately from live listener health, and `channels status`/doctor warnings surface dead or stale listeners instead of only showing a linked session. That addresses the reported mismatch where sends failed with `No active WhatsApp Web listener` while status still looked healthy.

## Evidence

- **Dead-listener detection is now part of the WhatsApp monitor state:** The WhatsApp monitor marks the channel `stale` on watchdog timeout, records reconnect/close health states, and emits runtime status updates instead of leaving the channel looking simply linked.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:299](https://github.com/openclaw/openclaw/blob/bcdacfa1/extensions/whatsapp/src/auto-reply/monitor.ts#L299)

  - sha: [bcdacfa1](https://github.com/openclaw/openclaw/commit/bcdacfa1)
- **Runtime status now carries healthState/lastError/reconnect metadata:** WhatsApp channel status summaries now include `connected`, `lastDisconnect`, `lastInboundAt`, `lastEventAt`, `lastError`, and `healthState`, so a linked auth session no longer implies a healthy live listener.
  - file: [extensions/whatsapp/src/channel.ts:187](https://github.com/openclaw/openclaw/blob/bcdacfa1/extensions/whatsapp/src/channel.ts#L187)

  - sha: [bcdacfa1](https://github.com/openclaw/openclaw/commit/bcdacfa1)

  - file: [extensions/whatsapp/src/status-issues.ts:43](https://github.com/openclaw/openclaw/blob/bcdacfa1/extensions/whatsapp/src/status-issues.ts#L43)
