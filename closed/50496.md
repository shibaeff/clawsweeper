## Summary

Current `main` already prevents this restart replay class at the delivery-queue layer, and the same implementation is present in release `v2026.4.22`. Successful deliveries are durably acknowledged before cleanup, stale delivered markers are ignored on startup, and tests cover both stale-marker and stale-startup-snapshot duplicate-delivery cases.

## Evidence

- **Successful deliveries are durably marked complete before cleanup:** `ackDelivery()` now uses a two-phase ack: rename `{id}.json` to `{id}.delivered`, then unlink the marker. The comment and implementation explicitly state this prevents replay after a crash between delivery and cleanup.
  - file: [src/infra/outbound/delivery-queue-storage.ts:157](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/infra/outbound/delivery-queue-storage.ts#L157)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Startup recovery skips delivered markers instead of replaying them:** `loadPendingDeliveries()` removes any stale `.delivered` marker files and only loads `.json` entries as pending work, so a successfully acked delivery is not retried on gateway restart.
  - file: [src/infra/outbound/delivery-queue-storage.ts:219](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/infra/outbound/delivery-queue-storage.ts#L219)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Regression tests cover the no-replay behavior:** Storage tests assert that acked entries and stale `.delivered` markers do not replay, and reconnect/startup tests assert a stale startup snapshot is not delivered twice after another path already acked it.
  - file: [src/infra/outbound/delivery-queue.storage.test.ts:80](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/infra/outbound/delivery-queue.storage.test.ts#L80)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Reconnect/startup duplicate-delivery case is explicitly tested:** The reconnect-drain test verifies only one delivery occurs for the target entry and that startup recovery skips the already-acked delivery.
  - file: [src/infra/outbound/delivery-queue.reconnect-drain.test.ts:278](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/infra/outbound/delivery-queue.reconnect-drain.test.ts#L278)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Risks / Open Questions

- This was a read-only code/history review; I did not run a live gateway restart reproduction against a real Feishu/Lark setup.
- The fix is implemented at the delivery-queue durability/recovery layer rather than by explicitly checking whether a session file was moved to `.trash/`; if a different duplicate-delivery path still exists, it would need a fresh report.
