## Summary

Current main already shipped a different fix for the reported config-triggered macOS gateway restart failure: launchd-supervised SIGUSR1 restarts now exit cleanly and let launchd KeepAlive relaunch the process, instead of relying on the detached `start-after-exit` handoff that this PR hardens. That behavior is documented in the 2026.4.22 changelog and implemented in the current restart path, so this PR's targeted handoff changes are obsolete on main.

## Evidence

- **Current macOS supervised restart path no longer uses detached handoff:** `restartGatewayProcessWithFreshPid` returns `mode: "supervised"` for launchd and explicitly says to avoid detached kickstart/start handoffs, letting KeepAlive relaunch the service instead.
  - file: [src/infra/process-respawn.ts:32](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/infra/process-respawn.ts#L32)
  - command: `nl -ba src/infra/process-respawn.ts | sed -n '1,120p'`
- **SIGUSR1/config restart exits cleanly after a short launchd delay:** The gateway run loop treats a launchd-supervised restart as a full process restart, waits briefly to avoid launchd crash-loop throttling, then exits 0.
  - file: [src/cli/gateway-cli/run-loop.ts:92](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/cli/gateway-cli/run-loop.ts#L92)
