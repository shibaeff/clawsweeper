## Summary

Current `main` already handles `/model list` and `/model status` as `/model` actions instead of parsing them as model refs, so the reported `openai-codex/list` and `openai-codex/status` failure is obsolete on `main`.

## Evidence

- **`status`/`list` are recognized as `/model` actions:** `maybeHandleModelDirectiveInfo()` explicitly treats `status` and `list` as special `/model` directives and returns early unless the directive is a real model-selection input.
  - file: [src/auto-reply/reply/directive-handling.model.ts:214](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/auto-reply/reply/directive-handling.model.ts#L214)
  - command: `nl -ba src/auto-reply/reply/directive-handling.model.ts | sed -n '206,255p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **`/model list` is routed to the listing path:** When the directive is `list`, current `main` calls `resolveModelsCommandReply({ commandBodyNormalized: "/models", ... })`, so `/model list` goes through the models-list reply path rather than the model-set path that produced `model not allowed`.
  - file: [src/auto-reply/reply/directive-handling.model.ts:233](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/auto-reply/reply/directive-handling.model.ts#L233)

- **Directive-only `/model` commands hit info handling before normal selection flow:** `handleDirectiveOnly()` calls `maybeHandleModelDirectiveInfo()` before continuing into `resolveModelSelectionFromDirective()`, so `/model status` and `/model list` are answered as info/list commands instead of being forced through model selection.
