## Summary

Current `main` already hardens the Discord gateway teardown path so reconnect-exhausted errors are classified and suppressed during teardown/dispose instead of escaping as uncaught exceptions, and the regression is covered by focused tests.

## Evidence

- **Reconnect exhaustion is explicitly classified:** The Discord gateway supervisor now treats `Max reconnect attempts ...` errors as a `reconnect-exhausted` lifecycle event instead of leaving them as raw emitter errors.
  - file: [extensions/discord/src/monitor/gateway-supervisor.ts:89](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/gateway-supervisor.ts#L89)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Late teardown/dispose errors are suppressed:** During `teardown` and `disposed` phases, late gateway errors are logged as suppressed rather than re-thrown, which directly covers the stale-socket restart shutdown window described in the issue.
  - file: [extensions/discord/src/monitor/gateway-supervisor.ts:140](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/gateway-supervisor.ts#L140)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Startup cleanup disposes the supervisor after disconnect:** Provider cleanup calls `lifecycleGateway?.disconnect()` and then `gatewaySupervisor?.dispose()`, so any follow-on reconnect exhaustion from intentional shutdown is routed through the suppression path instead of crashing the process.
  - file: [extensions/discord/src/monitor/provider.ts:1143](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/provider.ts#L1143)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Regression tests cover the exact late-error path:** Tests assert that late `Max reconnect attempts` errors emitted during teardown or after dispose do not throw and are logged as suppressed.
  - file: [extensions/discord/src/monitor/gateway-supervisor.test.ts:59](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/gateway-supervisor.test.ts#L59)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Provider-level test verifies no crash after startup cleanup disconnect:** The Discord provider test emits `Max reconnect attempts (0) reached after code 1005` after cleanup and asserts that it does not throw.
  - file: [extensions/discord/src/monitor/provider.test.ts:293](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/provider.test.ts#L293)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
