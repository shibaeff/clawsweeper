## Summary

Current main already handles corrupted/invalid `openclaw.json` via automatic last-known-good recovery during gateway startup and hot reload, with shipped docs and tests. That makes this PR’s separate `openclaw config restore` command redundant.

## Close Comment

Closing this as implemented after Codex review.

Current `main` already handles the reported corrupted-config recovery path: gateway startup and `openclaw gateway run` auto-restore invalid configs from the promoted last-known-good backup, the rejected payload is preserved as `.clobbered.*`, and the behavior is documented and tested. The PR's separate `openclaw config restore` command is therefore superseded by existing `main` behavior.

What I checked:

- **Gateway startup now auto-recovers invalid config:** `loadGatewayStartupConfigSnapshot()` calls `recoverConfigFromLastKnownGood()` when the config snapshot is invalid, logs `gateway: invalid config was restored from last-known-good backup`, reloads the config, and continues startup. ([`src/gateway/server-startup-config.ts:71`](https://github.com/openclaw/openclaw/blob/350d3221800a/src/gateway/server-startup-config.ts#L71), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **CLI gateway run path also performs the same recovery:** `readGatewayStartupConfig()` in the gateway CLI restores invalid effective config from the last-known-good backup before continuing, covering the 'gateway fails to start' case from the PR description. ([`src/cli/gateway-cli/run.ts:261`](https://github.com/openclaw/openclaw/blob/350d3221800a/src/cli/gateway-cli/run.ts#L261), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Shared recovery helper restores the file in place:** `recoverConfigFromLastKnownGood()` reads the promoted last-known-good copy, saves the broken payload as a `.clobbered.*` artifact, copies the good config back into place, and logs `Config auto-restored from last-known-good`. ([`src/config/io.observe-recovery.ts:1008`](https://github.com/openclaw/openclaw/blob/a50aef82ba32/src/config/io.observe-recovery.ts#L1008), [a50aef82ba32](https://github.com/openclaw/openclaw/commit/a50aef82ba32))
- **Main now promotes a last-known-good config during normal startup:** After successful server startup, `server.impl.ts` promotes the startup snapshot to the last-known-good backup used by the recovery helper. ([`src/gateway/server.impl.ts:898`](https://github.com/openclaw/openclaw/blob/350d3221800a/src/gateway/server.impl.ts#L898), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Docs and tests cover the automatic recovery flow:** Docs state that invalid direct edits can be restored from the last-known-good backup during startup or hot reload, troubleshooting explains the `.clobbered.*` workflow, and tests verify promotion plus restore after an invalid edit. Public docs: [`docs/cli/config.md`](https://docs.openclaw.ai/cli/config). ([`docs/cli/config.md:383`](https://docs.openclaw.ai/cli/config), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))

So I’m closing this as already implemented rather than keeping a duplicate issue open.
