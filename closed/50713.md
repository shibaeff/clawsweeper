## Summary

Current main already normalizes replayed Ollama tool-call arguments from JSON strings back into objects before sending them to Ollama, and the shipped v2026.4.22 release contains the same fix plus regression coverage for the reported round-trip failure.

## Evidence

- **Replay args are normalized before Ollama requests:** `ensureArgsObject()` parses stringified JSON into an object, and `extractToolCalls()` now applies it to both `toolCall.arguments` and `tool_use.input` before `convertToOllamaMessages()` emits `tool_calls` for Ollama.
  - file: [extensions/ollama/src/stream.ts:388](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/ollama/src/stream.ts#L388)
  - command: `nl -ba extensions/ollama/src/stream.ts | sed -n '388,545p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **OpenAI-compat payloads are normalized too:** `normalizeOllamaCompatMessageToolArgs()` rewrites `function_call.arguments` and `tool_calls[].function.arguments` in outgoing payload records, covering OpenAI-compatible Ollama transport as well.
  - file: [extensions/ollama/src/stream.ts:392](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/ollama/src/stream.ts#L392)
  - command: `nl -ba extensions/ollama/src/stream.ts | sed -n '388,545p'`
