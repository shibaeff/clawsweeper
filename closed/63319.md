## Summary

Keep open. Current `main` still lacks OAuth-aware MiniMax TTS on the speech path, and the proposed PR was correctly flagged in review for marking `minimax-portal` configured without giving `synthesize` access to those credentials.

## Evidence

- **Current speech provider still requires direct API key:** On `main`, MiniMax speech `isConfigured` only checks `providerConfig.apiKey` or `MINIMAX_API_KEY`, and `synthesize` throws `MiniMax API key missing` if neither is present. There is no auth-profile lookup here.
  - file: [extensions/minimax/speech-provider.ts:206](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/minimax/speech-provider.ts#L206)
  - command: `nl -ba extensions/minimax/speech-provider.ts | sed -n '180,240p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **`minimax-portal` is not registered as a speech provider on main:** The MiniMax plugin registers only `buildMinimaxSpeechProvider()`, and the plugin manifest advertises only `"minimax"` under `contracts.speechProviders`. The portal provider is still absent from TTS registration.
  - file: [extensions/minimax/index.ts:28](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/minimax/index.ts#L28)
