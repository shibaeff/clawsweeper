## Summary

Current `main` already contains the duplicate-plugin protections that match this report: plugin manifest discovery now collapses same-physical-root duplicates, and repeated startup loads reuse a cached plugin registry instead of rerunning plugin registration.

## Evidence

- **Manifest registry deduplicates same physical plugin roots:** When the same plugin id is discovered again, the manifest registry now compares root paths and realpaths, silently skips aliases of the same physical plugin directory, and only keeps the higher-precedence candidate.
  - file: [src/plugins/manifest-registry.ts:625](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/plugins/manifest-registry.ts#L625)
  - command: `nl -ba src/plugins/manifest-registry.ts | sed -n '620,675p'`
- **Plugin loader returns cached registry before register hooks rerun:** `loadOpenClawPlugins()` checks the cache first and returns the cached registry immediately; only cache misses proceed into a fresh load/registration path. This prevents repeated startup calls with the same cache key from invoking plugin registration again.
  - file: [src/plugins/loader.ts:1858](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/plugins/loader.ts#L1858)
  - command: `nl -ba src/plugins/loader.ts | sed -n '1856,1889p'`
- **Startup reload collapsing is intentional:** Both discovery and manifest loading carry a short cache window explicitly documented as being there to collapse bursty startup reloads.
  - file: [src/plugins/manifest-registry.ts:159](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/plugins/manifest-registry.ts#L159)
  - command: `nl -ba src/plugins/manifest-registry.ts | sed -n '154,162p'`
- **Fix was shipped in the changelog:** The changelog for release `2026.4.20` says OpenClaw now keeps only the highest-precedence manifest when distinct discovered plugins share an id, so lower-precedence duplicates no longer load beside the selected plugin.
  - file: [CHANGELOG.md:353](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/CHANGELOG.md#L353)
  - command: `nl -ba CHANGELOG.md | sed -n '350,410p'`

## Risks / Open Questions

- If an operator truly has multiple distinct physical copies of the same plugin id installed, current main still treats that as a real duplicate and only one wins by precedence; this close only covers the repeated-load report for the same plugin being rediscovered through multiple paths/startup passes.
- An external plugin that logs from top-level module evaluation under genuinely different loader cache keys could still produce repeated noise, but that is a different failure mode than the report's same-plugin startup re-registration.

## Close Comment

Closing from a Codex review because current `main` already has the protections this report was asking for. OpenClaw now collapses duplicate discoveries of the same physical plugin root, including symlink/path-alias cases, in `src/plugins/manifest-registry.ts:625`, and repeated startup loads reuse the cached plugin registry before rerunning plugin registration in `src/plugins/loader.ts:1858`. The changelog shipped this duplicate-manifest fix in `2026.4.20` (`CHANGELOG.md:353`, `CHANGELOG.md:407`). If you still see repeated registration on a current build, please open a new issue with the exact plugin install paths and startup logs, because that would likely be a different code path.
