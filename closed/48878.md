## Summary

Current `main` already preserves fully qualified `provider/model` refs in the webchat/Control UI model picker and sends that qualified value through `sessions.patch`, with regression coverage for the exact failure mode described in #48878. The fix is on `main` but not in the latest tagged release `v2026.4.22`.

## Evidence

- **Picker option values are provider-qualified:** `buildChatModelOptionFromLookup()` constructs each select option `value` with `buildQualifiedChatModelValue(entry.id, provider)`, so dropdown selections carry full `provider/model` refs instead of bare model ids.
  - file: [ui/src/ui/chat-model-ref.ts:266](https://github.com/openclaw/openclaw/blob/8fade9df27a9a25cd017db1154409b3ad9bd3741/ui/src/ui/chat-model-ref.ts#L266)

  - sha: [8fade9df27a9](https://github.com/openclaw/openclaw/commit/8fade9df27a9a25cd017db1154409b3ad9bd3741)
- **Dropdown sends selected value unchanged to sessions.patch:** The chat header model select reads `e.target.value.trim()` and passes it directly as `model` in the `sessions.patch` RPC, so a qualified option stays qualified on the wire.
