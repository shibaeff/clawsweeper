## Summary

Current `main` already fixes late plugin HTTP route matching after reload via a pinned HTTP-route registry and per-request route-registry resolution, with targeted regression tests. The bug this PR addresses is therefore already implemented on `main` and was shipped in `v2026.4.22`.

## Close Comment

Closing from a Codex review because current `main` already fixes this behavior. The gateway now pins a dedicated HTTP-route registry at startup (`src/plugins/runtime.ts:109-149`, `src/gateway/server-runtime-state.ts:108-113`) and the HTTP handler re-resolves against that live route registry on each request (`src/gateway/server/plugins-http.ts:63-83`), so late `registerPluginHttpRoute(...)` calls remain visible after reloads instead of 404ing against a stale registry snapshot. The regression is covered in `src/gateway/server/plugins-http.test.ts:300-367` and `src/plugins/http-registry.test.ts:208-229`. This implementation is already present in release `v2026.4.22` at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, so this PR is no longer needed.
