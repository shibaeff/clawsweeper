## Summary

Current `main` already has the restart/catch-up guards needed for this report: non-schedule edits preserve a future `nextRunAtMs`, startup catch-up only replays provably missed cron slots, and regression tests cover both an `Asia/Shanghai` daily cron and the no-duplicate-on-restart case. The same behavior is present in the shipped `v2026.4.23` release.

## Evidence

- **Non-schedule edits preserve future nextRunAtMs:** `update()` only recomputes `nextRunAtMs` when the schedule or enabled state changes; otherwise it leaves an existing scheduled future run intact unless the stored value is missing/invalid.

  - file: [src/cron/service/ops.test.ts:260](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cron/service/ops.test.ts#L260)

- **Restart recompute avoids advancing still-future jobs:** `recomputeNextRuns()` explicitly preserves a still-future `nextRunAtMs` during restart recovery instead of recomputing it.
  - file: [src/cron/service/jobs.ts:477](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cron/service/jobs.ts#L477)

- **Startup catch-up only replays true missed cron slots:** Restart catch-up first runs due jobs, then only replays a cron slot when `computeJobPreviousRunAtMs(job, nowMs)` proves the previous scheduled slot is later than `lastRunAtMs`; a future same-day slot does not qualify.
  - file: [src/cron/service/timer.ts:894](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cron/service/timer.ts#L894)
