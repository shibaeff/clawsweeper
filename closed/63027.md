## Summary

Current `main` already has Discord inbound replay dedupe in the message handler path, with regression tests that drop duplicate inbound deliveries before preflight and changelog evidence that the replay-dedupe fix shipped in `v2026.4.12`.

## Evidence

- **Discord handler now claims a per-message replay key before debounce/preflight:** `createDiscordMessageHandler()` builds `accountId:channelId:messageId` replay keys and returns early when the same inbound Discord delivery was already claimed, which directly addresses one inbound message being processed twice.
