## Summary

Current `main` and the shipped `v2026.4.23` tree no longer route `api: "google-generative-ai"` through an OpenAI-compatible payload path. The bundled Google provider now uses a dedicated Google transport that builds Gemini-native request bodies, so the reported `store` / `service_tier` / `metadata` collision does not reproduce on the supported core path, while Codex/OpenAI-specific request shaping remains isolated to the OpenAI/Codex wrappers.

## Evidence

  - file: [extensions/google/provider-registration.ts:52](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/extensions/google/provider-registration.ts#L52)

## Risks / Open Questions

- This close is scoped to the bundled OpenClaw core/provider path. The exported helper `createGoogleThinkingPayloadWrapper()` still only sanitizes thinking config, so a third-party plugin that reuses that helper against Google's OpenAI-compatible endpoint may still need its own provider-local field stripping.
