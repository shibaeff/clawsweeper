## Summary

Current main already supports multimodal image input on the gateway `agent` method through `attachments`; the exact proposed `message: ContentBlock[]` union is not the public contract, but the reported use case is already implemented and shipped.

## Evidence

- **Public agent schema:** `AgentParamsSchema` still keeps `message` as a string, but it also exposes an `attachments` field on the public gateway `agent` params.
  - file: [src/gateway/protocol/schema/agent.ts:131](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/src/gateway/protocol/schema/agent.ts#L131)
  - sha: [a2221d6b4709](https://github.com/openclaw/openclaw/commit/a2221d6b47094cd34c12025ce809aba02c5e1c1c)
- **Nested base64 image shape accepted:** Gateway attachment normalization accepts dashboard-style nested image input with `source.type = "base64"`, `source.media_type`, and `source.data`, matching the issue's image payload shape.
  - file: [src/gateway/server-methods/attachment-normalize.ts:24](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/src/gateway/server-methods/attachment-normalize.ts#L24)
  - sha: [a2221d6b4709](https://github.com/openclaw/openclaw/commit/a2221d6b47094cd34c12025ce809aba02c5e1c1c)
- **Agent handler builds multimodal payloads:** The `agent` handler normalizes `request.attachments`, parses them with `parseMessageWithAttachments`, and produces `images[]` for the model request.
  - file: [src/gateway/server-methods/agent.ts:394](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/src/gateway/server-methods/agent.ts#L394)
