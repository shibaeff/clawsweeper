## Summary

Current main already carries the gateway handshake-timeout fixes that match this 2026.3.13 regression: the pre-auth handshake timeout was raised for busy local CLI connections, and the gateway call path now yields before opening WS so `connect.challenge` processing is not starved.

## Evidence

- **Shipped fix note:** The 2026.3.22 changelog explicitly says the Gateway/WS pre-auth handshake timeout was raised to 10 seconds so busy local gateways stop dropping healthy CLI connections.
  - file: [CHANGELOG.md:2446](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/CHANGELOG.md#L2446)
  - command: `nl -ba CHANGELOG.md | sed -n '2428,2460p'`
- **Current timeout default:** Current main sets `DEFAULT_PREAUTH_HANDSHAKE_TIMEOUT_MS = 10_000`, and connect-challenge timeouts clamp to that shared pre-auth timeout.
  - file: [src/gateway/handshake-timeouts.ts:1](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/handshake-timeouts.ts#L1)
  - command: `nl -ba src/gateway/handshake-timeouts.ts | sed -n '1,220p'`
- **Current CLI mitigation:** The gateway call path now yields to the event loop before starting the WebSocket connection, with an inline note that heavy synchronous loading could otherwise starve `connect.challenge` handling and cause handshake timeouts.
  - file: [src/gateway/call.ts:479](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/call.ts#L479)
  - command: `nl -ba src/gateway/call.ts | sed -n '430,560p'`
- **Failure mode match:** Server-side handshake timeout handling still applies only before a client successfully sends `connect`, which matches the reporter's `connect challenge timeout` / closed-before-connect symptom.
  - file: [src/gateway/server/ws-connection.ts:366](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/server/ws-connection.ts#L366)
  - command: `nl -ba src/gateway/server/ws-connection.ts | sed -n '320,430p'`
