## Summary

Issue #51171 is best closed as `cannot_reproduce`: the current Telegram voice path on `main` is single-turn, and the reporter later confirmed the duplicate replies were caused by two OpenClaw instances long-polling the same Telegram bot token, not by Telegram voice/audio transcription creating a second inbound event.

## Evidence

- **Reporter confirmed config root cause and stable workaround:** In the issue thread, the reporter identified that two OpenClaw instances were running on the same host with the same Telegram bot token, then reported that switching to a separate token was stable. That makes the original bug report obsolete as a product bug and points to operator configuration as the cause.
- **Current voice-message body path is single-delivery:** `resolveTelegramBody` replaces the audio placeholder with the preflight transcript in the same message body instead of emitting a second event. The code sets `bodyText = preflightTranscript` when audio is present and the body is `<media:audio>`, then falls back to one body string for the message.

  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Test coverage matches one-turn DM voice behavior:** The Telegram tests explicitly cover DM voice-note preflight transcription and assert one resulting `bodyText` containing the transcript, with no remaining `<media:audio>` placeholder.

The existing test proves duplicate `update_id` values are deduped within one bot instance. Separate processes using the same token would not share this state, so both would process the same update.

- **Docs already steer multi-gateway users to separate Telegram bots/tokens:** The multi-gateway guide says the recommended setup is to use a completely separate Telegram bot for the rescue/extra gateway and lists the Telegram bot token as part of the per-gateway isolation boundary.
  - file: [docs/gateway/multiple-gateways.md:14](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/docs/gateway/multiple-gateways.md#L14)

## Risks / Open Questions

- Separate OpenClaw gateway processes can still duplicate Telegram updates if an operator intentionally reuses one bot token across them; this review only supports closing this report because the reporter confirmed that exact setup and confirmed the issue stopped after separating tokens.
- There may still be room for better warnings or diagnostics around cross-process token reuse, but that is not evidence that this reported voice-message bug remains reproducible on current `main`.
