## Summary

Current main has moved TTS provider ownership to the plugin capability system, and the proposed local CLI TTS bridge fits the repo's documented 'optional capability as plugin' direction better than a core change. This PR's core-file approach is stale against main and is better pursued as a community/ClawHub plugin.

## Close Comment

Closing after Codex review. The idea is useful, but current `main` routes TTS providers through the plugin capability system (`src/tts/provider-registry.ts:16`) and the SDK already exposes `api.registerSpeechProvider(...)` for exactly this kind of integration (`docs/plugins/sdk-provider-plugins.md:495`). The project vision also says optional capability should usually ship as plugins and that the bar for adding optional plugins to core is intentionally high (`VISION.md:54`). Given that, an arbitrary local CLI TTS bridge for tools like Kokoro/Piper is a better fit as a community/ClawHub plugin than a core PR touching `src/tts/*`. If you want to continue this, the right path is to rework it as a plugin using the speech-provider SDK surface rather than landing it in core.
