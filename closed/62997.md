## Summary

Current `main` already closes the reported ACP zombie path: ACP child failures are recorded as terminal failures, terminal delivery is single-shot instead of replaying forever, stale ACP bindings are auto-unbound on missing-metadata init failures, and Telegram restart cleanup for bindings to failed/missing ACP sessions shipped in `v2026.4.18`.

## Evidence

- **ACP child errors are terminal failures:** `AcpSessionManager.runTurn()` catches runtime errors and marks the background task terminal with `status: resolveBackgroundTaskFailureStatus(acpError)`, so provider billing/auth failures land as `failed` or `timed_out` instead of a successful completion.
  - file: [src/acp/control-plane/manager.core.ts:879](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/acp/control-plane/manager.core.ts#L879)
  - command: `nl -ba src/acp/control-plane/manager.core.ts | sed -n '879,913p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Terminal delivery is not replayed indefinitely:** `maybeDeliverTaskTerminalUpdate()` only auto-delivers while `deliveryStatus === pending`; after the first attempt it transitions to `session_queued`, `delivered`, or `failed`, which prevents the same terminal notice from being re-announced on every later interaction.
  - file: [src/tasks/task-registry.ts:1018](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/tasks/task-registry.ts#L1018)

  - file: [src/tasks/task-executor-policy.ts:29](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/tasks/task-executor-policy.ts#L29)
