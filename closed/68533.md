## Summary

The reported `/api/projects/<name>?companyId=...` PostgreSQL UUID failure does not map to any current OpenClaw main codepath. Current main only references Paperclip as opaque wake metadata; the linked timeline already points to an external Paperclip PR that appears to own the actual route bug.

## Evidence

- **No matching OpenClaw route or query handling:** Repository search on current main found no `/api/projects` route, no `companyId` handling, and no matching UUID/PostgreSQL error path in OpenClaw code. The only `api/projects` hit is an unrelated Azure Foundry test URL under `extensions/microsoft-foundry/index.test.ts`.
  - command: `rg -n "api/projects|/api/projects|companyId|invalid input syntax for type uuid|board-issues-and-requests|hermes-paperclip-adapter|project name" . --glob '!extensions/diffs/**' --glob '!skills/**'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Paperclip surface is opaque wake metadata only:** The Gateway wake schema only defines `mode` and `text`, with `additionalProperties: true` so external senders can attach metadata. There is no project lookup or database-routing logic here.
  - file: [src/gateway/protocol/schema/agent.ts:197](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/gateway/protocol/schema/agent.ts#L197)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Current tests use Paperclip only as forward-compatible metadata:** `validateWakeParams` explicitly accepts a nested `paperclip` object as unknown metadata for forward compatibility; this is the current tested behavior on main.
  - file: [src/gateway/protocol/index.test.ts:159](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/gateway/protocol/index.test.ts#L159)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Changelog describes Paperclip as an external sender, not an owned projects API:** The changelog entry for `#68355` says wake payloads now allow unknown properties so external senders like Paperclip can attach opaque metadata.
  - file: [CHANGELOG.md:747](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/CHANGELOG.md#L747)
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Issue timeline already points to the owning upstream fix discussion:** The issue timeline cross-references `paperclipai/paperclip#3956` (`fix: throw 404 when project name cannot be resolved in API route`), which matches the reported API-route behavior and indicates the bug lives in Paperclip rather than OpenClaw.
