## Summary

Current main already prevents Telegram `sendPayload` oversized-message failures through the extension-owned Telegram transport. The branch’s adapter-level fix is obsolete: long HTML text is chunked in `sendMessageTelegram`, long captions are converted into follow-up text sends, HTML chunking is tag/entity-aware, and empty text without media throws instead of silently fabricating success.

## Close Comment

Closing this as implemented after Codex review.

Current `main` already fixes the Telegram oversized `sendPayload` crash through the newer extension-based outbound path, so this PR is superseded by code now on `main` rather than needing to land as-is.

What I checked:

- **Telegram sendPayload now goes through the new outbound adapter:** `sendPayload` delegates to `sendTelegramPayloadMessages`, which preserves Telegram-specific buttons/media sequencing and forwards text to the transport send path instead of emitting one oversized raw payload. - **Transport-layer HTML chunking fixes the >4096-char failure:** `sendMessageTelegram` builds a chunk plan with `splitTelegramHtmlChunks(rawText, 4000)` and sends the chunks sequentially for HTML-mode text, so oversized Telegram text no longer goes out as a single request.
