## Close Comment

Closing this as implemented after Codex review.

Current main already ships the supported shared rich-message path through `ReplyPayload.presentation` / `MessagePresentation`, with native channel renderers for Slack, Discord, Telegram, and Teams. That supersedes this PR’s older `<!--adaptive-card-->` marker-parsing approach.

What I checked:

- **Shared contract exists on main:** Docs define `MessagePresentation` as the shared rich outbound UI contract and explicitly say channel plugins should own provider-native outputs instead of adding native card/block/component fields to the shared path. Public docs: [`docs/plugins/message-presentation.md`](https://docs.openclaw.ai/plugins/message-presentation). ([`docs/plugins/message-presentation.md:11`](https://docs.openclaw.ai/plugins/message-presentation), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Architecture directs new work to presentation, not native escape hatches:** Architecture docs say shared send presentation should use the generic `MessagePresentation` contract and should not expose provider-native UI escape hatches from the generic message tool. Public docs: [`docs/plugins/architecture-internals.md`](https://docs.openclaw.ai/plugins/architecture-internals). ([`docs/plugins/architecture-internals.md:615`](https://docs.openclaw.ai/plugins/architecture-internals), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Reply payload already carries channel-agnostic rich presentation:** `ReplyPayload` on main includes a `presentation?: MessagePresentation` field documented as channel-agnostic rich presentation that core can degrade or map through channel renderers. ([`src/auto-reply/reply-payload.ts:7`](https://github.com/openclaw/openclaw/blob/350d3221800a/src/auto-reply/reply-payload.ts#L7), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Slack, Discord, and Telegram already render shared presentation natively:** Slack outbound declares `presentationCapabilities` and builds Block Kit from `presentation`; Discord outbound does the same for component payloads; Telegram maps `presentation` to fallback text plus interactive controls. ([`extensions/slack/src/outbound-adapter.ts:105`](https://github.com/openclaw/openclaw/blob/350d3221800a/extensions/slack/src/outbound-adapter.ts#L105), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Discord renderer implementation is present:** `buildDiscordPresentationComponents(...)` converts shared presentation blocks into Discord-native component messages. ([`extensions/discord/src/shared-interactive.ts:87`](https://github.com/openclaw/openclaw/blob/350d3221800a/extensions/discord/src/shared-interactive.ts#L87), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))
- **Teams already maps shared presentation to Adaptive Cards:** Teams builds an Adaptive Card from `MessagePresentation` and sends it through `sendAdaptiveCardMSTeams(...)` when a message action includes `presentation`. ([`extensions/msteams/src/presentation.ts:4`](https://github.com/openclaw/openclaw/blob/350d3221800a/extensions/msteams/src/presentation.ts#L4), [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a))

So I’m closing this as already implemented rather than keeping a duplicate issue open.
