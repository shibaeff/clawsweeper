## Summary

Current main already implements host-local scheduling for cron jobs when no timezone is provided, and the shipped release `v2026.4.22` contains the same behavior. The Control UI keeps timezone optional, passes `tz` only when explicitly set, and the scheduler resolves omitted `tz` to the host system timezone.

## Evidence

- **Scheduler default is host-local timezone:** `resolveCronTimezone()` returns the explicit cron timezone when present, otherwise `Intl.DateTimeFormat().resolvedOptions().timeZone`, so recurring cron schedules do not default to UTC when `tz` is omitted.
