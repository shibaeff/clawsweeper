## Summary

Current `main` already implements the requested Slack thread opt-out through the shipped `channels.slack.thread.requireExplicitMention` setting, with config/schema wiring, runtime enforcement, docs, tests, and a changelog entry in `v2026.4.10`. This PR's alternate `thread.autoReplyOnParticipation` knob is redundant on current `main`.

## Close Comment

Closing this as implemented after Codex review.

Current `main` already ships the requested Slack thread opt-out behavior under `channels.slack.thread.requireExplicitMention` instead of the PR's proposed `thread.autoReplyOnParticipation`. The implementation is wired through config schema, monitor context/provider plumbing, mention gating, docs, and targeted tests, and it is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

What I checked:

- **Config type documents equivalent setting:** `SlackThreadConfig` includes `requireExplicitMention?: boolean` with docs stating it suppresses implicit thread mentions so only explicit `@bot` mentions trigger replies in threads. ([`src/config/types.slack.ts:92`](https://github.com/openclaw/openclaw/blob/7ac35b4f6900/src/config/types.slack.ts#L92), [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f6900))
- **Schema exposes the setting:** `SlackThreadSchema` accepts `requireExplicitMention` as an optional boolean. ([`src/config/zod-schema.providers-core.ts:880`](https://github.com/openclaw/openclaw/blob/7ac35b4f6900/src/config/zod-schema.providers-core.ts#L880), [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f6900))
- **Provider reads and propagates the setting:** Slack monitor provider resolves `slackCfg.thread?.requireExplicitMention ?? false` and passes it into monitor context. ([`extensions/slack/src/monitor/provider.ts:177`](https://github.com/openclaw/openclaw/blob/7ac35b4f6900/extensions/slack/src/monitor/provider.ts#L177), [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f6900))
- **Mention gating enforces explicit-thread-only behavior:** Inbound mention resolution disables all implicit thread mention kinds when `threadRequireExplicitMention` is true, which restores explicit-mention-only behavior in threads. ([`extensions/slack/src/monitor/message-handler/prepare.ts:438`](https://github.com/openclaw/openclaw/blob/7ac35b4f6900/extensions/slack/src/monitor/message-handler/prepare.ts#L438), [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f6900))
- **Tests cover the requested behavior:** Tests verify that thread replies without an explicit mention are dropped when the setting is enabled, while explicit `@mention` replies still work and the default remains permissive. ([`extensions/slack/src/monitor/message-handler/prepare.test.ts:902`](https://github.com/openclaw/openclaw/blob/7ac35b4f6900/extensions/slack/src/monitor/message-handler/prepare.test.ts#L902), [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f6900))
- **Docs describe the user-facing config:** Slack docs state that `channels.slack.thread.requireExplicitMention` suppresses implicit thread mentions and prevents bot-participated threads from bypassing `requireMention` gating. Public docs: [`docs/channels/slack.md`](https://docs.openclaw.ai/channels/slack). ([`docs/channels/slack.md:520`](https://docs.openclaw.ai/channels/slack), [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f6900))

So I’m closing this as already implemented rather than keeping a duplicate issue open.
