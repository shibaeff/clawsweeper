## Summary

Current `main` no longer matches the PR's reported breakage: Google Chat webhook handling already returns a valid 200/JSON response for interaction events, Google Chat chunking now happens inside one delivery call so all text chunks share the same thread id, and the proposed `ctxPayload.ReplyToId` fallback would conflict with the current reply-threading contract.

## Evidence

- **Webhook handler already returns successful JSON responses:** The current Google Chat webhook handler always ends successful requests with `200`, `Content-Type: application/json`, and `{}` after dispatching the event.
  - file: [extensions/googlechat/src/monitor-webhook.ts:278](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/googlechat/src/monitor-webhook.ts#L278)
  - command: `nl -ba extensions/googlechat/src/monitor-webhook.ts | sed -n '180,320p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Non-message events are intentionally treated as no-op:** Current event processing returns early for any event whose type is not `MESSAGE`, so `ADDED_TO_SPACE` is intentionally acknowledged without extra message delivery on `main`.
  - file: [extensions/googlechat/src/monitor.ts:67](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/googlechat/src/monitor.ts#L67)
  - command: `nl -ba extensions/googlechat/src/monitor.ts | sed -n '40,110p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Current Google Chat reply delivery keeps one thread across text chunks:** `deliverGoogleChatReply()` performs chunking inside a single delivery call and every chunk send uses the same `payload.replyToId` when calling `sendGoogleChatMessage`, so chunk 2+ do not lose thread context in the current code path.
  - file: [extensions/googlechat/src/monitor.ts:389](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/googlechat/src/monitor.ts#L389)
  - command: `nl -ba extensions/googlechat/src/monitor.ts | sed -n '388,460p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Chunk loop is inside one payload delivery:** The shared helper `deliverTextOrMediaReply()` loops over chunks by repeatedly calling `sendText(chunk)` for the same payload, which matches the current Google Chat delivery structure above.
  - file: [src/plugin-sdk/reply-payload.ts:393](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/plugin-sdk/reply-payload.ts#L393)
  - command: `nl -ba src/plugin-sdk/reply-payload.ts | sed -n '388,470p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Proposed fallback conflicts with current reply-threading policy:** Google Chat threading resolves per-account `replyToMode` with fallback `off`, and shared outbound tests document that single-use reply ids are intentionally cleared after the first send. The PR's unconditional `ctxPayload.ReplyToId` backfill would override that policy on later sends, matching the useful concern already raised in prior PR review.
  - file: [extensions/googlechat/src/channel.adapters.ts:104](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/googlechat/src/channel.adapters.ts#L104)
  - command: `nl -ba extensions/googlechat/src/channel.adapters.ts | sed -n '90,130p' && nl -ba src/infra/outbound/deliver.test.ts | sed -n '481,492p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Upstream Google Chat contract allows async handling:** Google's interaction-event docs say apps may respond asynchronously, and only require a populated `Message` object when the app wants to respond synchronously in real time. The same doc's Java sample uses an empty `new Message()` in the default branch, which aligns with OpenClaw's `{}` acknowledgement path.
  - command: `web search: developers.google.com/workspace/chat/receive-respond-interactions`
