## Summary

Current main does not exhibit the race described in PR #67302: lifecycle `end` does not clear `activeChatRunId`, active-run lifecycle end already resets status to `idle`, and main now also has a streaming watchdog for late/missing finals. The proposed branch would change non-active abort/error status handling without a regression test for a reproducible current-main failure.

## Evidence

- **`wasActiveRun` is computed before any finalize-path clear:** `handleChatEvent` captures `wasActiveRun` from `state.activeChatRunId === evt.runId` before calling `finalizeRun`, and the helper that clears `activeChatRunId` lives later in the finalize/terminate path. That means the PR body's claimed `lifecycle end -> clear activeChatRunId -> final sees wasActiveRun=false` flow is not present in current main.
