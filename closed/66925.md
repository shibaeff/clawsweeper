## Summary

Current main does not match the reported overwrite path: `active-memory` is not a memory-kind plugin and does not register a memory capability, so it cannot override `memory-core`'s `publicArtifacts` on `main`.

## Evidence

- **Memory capability is a single stored registration:** `registerMemoryCapability` still overwrites the single `memoryPluginState.capability` slot, and `listActiveMemoryPublicArtifacts` reads only that stored capability.
  - file: [src/plugins/memory-state.ts:170](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/plugins/memory-state.ts#L170)
  - command: `nl -ba src/plugins/memory-state.ts | sed -n '160,305p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Only memory-kind plugins may register that capability:** The plugin registry rejects `registerMemoryCapability(...)` from plugins that do not have `kind: "memory"`.
  - file: [src/plugins/registry.ts:1553](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/plugins/registry.ts#L1553)
  - command: `nl -ba src/plugins/registry.ts | sed -n '1550,1579p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **memory-core is the bundled memory capability owner:** `memory-core` declares `kind: "memory"` and registers `publicArtifacts.listArtifacts` through `api.registerMemoryCapability(...)`.
  - file: [extensions/memory-core/index.ts:33](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/memory-core/index.ts#L33)
  - command: `nl -ba extensions/memory-core/openclaw.plugin.json | sed -n '1,20p'; nl -ba extensions/memory-core/index.ts | sed -n '1,120p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **active-memory is not a memory plugin on current main:** The `active-memory` manifest has no `kind: "memory"` declaration, so it is not eligible to register the exclusive memory capability.
  - file: [extensions/active-memory/openclaw.plugin.json:1](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/active-memory/openclaw.plugin.json#L1)
  - command: `nl -ba extensions/active-memory/openclaw.plugin.json | sed -n '1,40p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **No memory capability registration exists in active-memory:** A repo search found no `registerMemoryCapability`, `registerMemoryPromptSection`, `registerMemoryFlushPlanResolver`, or `registerMemoryRuntime` calls anywhere under `extensions/active-memory`.
  - command: `rg -n "registerMemoryCapability|registerMemoryPromptSection|registerMemoryFlushPlanResolver|registerMemoryRuntime" extensions/active-memory`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Bridge import still consumes exported public artifacts manually:** `wiki bridge import` still pulls from `listActiveMemoryPublicArtifacts(...)`, and the CLI docs still describe bridge import as an explicit/manual sync step.
  - file: [extensions/memory-wiki/src/bridge.ts:224](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/memory-wiki/src/bridge.ts#L224)
  - command: `nl -ba extensions/memory-wiki/src/bridge.ts | sed -n '216,260p'; nl -ba docs/cli/wiki.md | sed -n '160,210p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)

## Risks / Open Questions

- `registerMemoryCapability` is still singleton/overwrite-only, so a future second memory-kind plugin could clobber `publicArtifacts`; this review only establishes that `active-memory` is not that registrant on current `main`.
