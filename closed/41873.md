## Close Comment

Closing this as implemented after Codex review.

Current `main` already avoids the reported manual-cron deadlock by redirecting nested embedded-agent work away from the `cron` lane, so this PR's behavior change is already present through a different implementation.

What I checked:

- **Outer manual run still uses cron lane:** `enqueueRun` still queues manual runs on `CommandLane.Cron`, matching the PR's reported outer enqueue point. ([`src/cron/service/ops.ts:704`](https://github.com/openclaw/openclaw/blob/7ac35b4f6900/src/cron/service/ops.ts#L704), [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f6900))
- **Inner isolated agent run is remapped off cron:** The isolated cron executor now passes `resolveNestedAgentLane(params.lane)` into `runEmbeddedPiAgent`, so a caller lane of `cron` is not reused for nested agent work. ([`src/cron/isolated-agent/run-executor.ts:179`](https://github.com/openclaw/openclaw/blob/7ac35b4f6900/src/cron/isolated-agent/run-executor.ts#L179), [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f6900))
- **Cron lane explicitly maps to nested lane:** `resolveNestedAgentLane` documents and implements the deadlock avoidance directly: if the incoming lane is `cron`, it returns `CommandLane.Nested` instead. ([`src/agents/lanes.ts:10`](https://github.com/openclaw/openclaw/blob/29d0e930bb3d/src/agents/lanes.ts#L10), [29d0e930bb3d](https://github.com/openclaw/openclaw/commit/29d0e930bb3d))
- **Regression coverage for cron remapping exists:** Unit coverage asserts that `resolveNestedAgentLane("cron")` and whitespace-trimmed `cron` both resolve to the nested lane. ([`src/agents/lanes.test.ts:14`](https://github.com/openclaw/openclaw/blob/29d0e930bb3d/src/agents/lanes.test.ts#L14), [29d0e930bb3d](https://github.com/openclaw/openclaw/commit/29d0e930bb3d))
- **Fix is on current main but not in latest release:** The lane-remapping code is blamed to commit `29d0e930bb3de4886b2e799781eb5620d973d1be`, while the provided latest release is `v2026.4.22` at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, so there is no released tag to attribute yet.
