## Summary

Current main already implements the reported behavior: Control UI token sessions in the `gateway.controlUi.dangerouslyDisableDeviceAuth=true` path keep `operator.read` instead of dropping to chat-only access, and that behavior shipped in `v2026.4.22`.

## Evidence

- **Control UI requests operator scopes by default:** The browser client always builds Control UI connects with the operator scope set including `operator.read`, `operator.write`, `operator.admin`, `operator.approvals`, and `operator.pairing`.
  - file: [ui/src/ui/gateway.ts:403](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/ui/src/ui/gateway.ts#L403)

## Close Comment

Closing after Codex review. Current OpenClaw already preserves `operator.read` for the Control UI path you described when `gateway.controlUi.dangerouslyDisableDeviceAuth=true` is used with token auth. The Control UI client requests `operator.read` by default (`ui/src/ui/gateway.ts:403`), the gateway bypass path does not clear those scopes in this configuration (`src/gateway/server/ws-connection/connect-policy.ts:84`, `src/gateway/server/ws-connection/message-handler.ts:621`), and the shipped regression test explicitly asserts that this exact mode returns `helloOk.auth.scopes = ["operator.read"]` and allows read calls (`src/gateway/server.auth.control-ui.suite.ts:427`). This behavior is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this issue appears to be already implemented on current main and in the latest release.
