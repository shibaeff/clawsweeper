## Summary

Close as `cannot_reproduce`: current main still formats Discord voice-note attachments as `<media:document>` in the initial placeholder text, but the actual inbound agent path rewrites any standalone `<media:...>` placeholder to the audio transcript before agent execution, so the reported DM behavior no longer matches current code.

## Evidence

- **Discord placeholder still falls back to document:** `resolveDiscordMessageText()` still uses `buildDiscordAttachmentPlaceholder()`, which collapses any non-image attachment to `<media:document> (N files)` even though per-attachment media inference recognizes `audio/*` as `<media:audio>`. This matches the reporter's local observation, but it is only the initial placeholder stage.
  - file: [extensions/discord/src/monitor/message-utils.ts:571](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/discord/src/monitor/message-utils.ts#L571)
  - command: `nl -ba extensions/discord/src/monitor/message-utils.ts | sed -n '545,605p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Discord still preserves audio MIME for downstream media understanding:** When Discord attachments are fetched or fall back to URL-only handling, the saved media payload keeps the real `contentType` from the attachment/fetch result. That means DM voice notes still reach shared media-understanding as audio, regardless of the placeholder tag.
  - file: [extensions/discord/src/monitor/message-utils.ts:400](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/discord/src/monitor/message-utils.ts#L400)
  - command: `nl -ba extensions/discord/src/monitor/message-utils.ts | sed -n '400,430p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Agent path always applies media understanding before reply generation:** `getReplyFromConfig()` finalizes inbound context and then calls `applyMediaUnderstandingIfNeeded()` for any inbound media before agent execution. This is the path that matters for DM voice-note transcripts reaching the agent.
  - file: [src/auto-reply/reply/get-reply.ts:244](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/auto-reply/reply/get-reply.ts#L244)
  - command: `nl -ba src/auto-reply/reply/get-reply.ts | sed -n '244,252p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Shared formatter strips any standalone media placeholder, not just audio:** `extractMediaUserText()` treats any standalone `<media:...>` token, including `<media:document> (1 file)`, as a removable placeholder. So a Discord DM voice note reaching media-understanding with that body shape will still be reformatted around the transcript.
  - file: [src/media-understanding/format.ts:3](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/media-understanding/format.ts#L3)
  - command: `nl -ba src/media-understanding/format.ts | sed -n '1,16p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **Successful audio transcription rewrites Body and command/raw text:** `applyMediaUnderstanding()` rewrites `ctx.Body` from the media outputs, sets `Transcript`, updates `CommandBody`/`RawBody`, and then re-finalizes the inbound context for the agent. The test suite explicitly covers that successful audio transcription replaces the body with `[Audio]\nTranscript:\n...`.
  - file: [src/media-understanding/apply.ts:531](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/src/media-understanding/apply.ts#L531)
  - command: `nl -ba src/media-understanding/apply.ts | sed -n '531,578p' ; nl -ba src/media-understanding/apply.test.ts | sed -n '332,347p'`
  - sha: [7b5307acfc34](https://github.com/openclaw/openclaw/commit/7b5307acfc347a4e6d6647156600ee4b8d6eaa37)
- **DM preflight transcription is not the agent-body path:** Discord preflight audio transcription is gated to non-DM mention detection (`!params.isDirectMessage`). That means the issue's DM diagnosis around preflight does not match the current architecture; DM transcript-to-agent behavior comes from the shared media-understanding path above, not guild mention preflight.
  - file: [extensions/discord/src/monitor/preflight-audio.ts:45](https://github.com/openclaw/openclaw/blob/7b5307acfc347a4e6d6647156600ee4b8d6eaa37/extensions/discord/src/monitor/preflight-audio.ts#L45)
