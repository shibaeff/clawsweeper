## Summary

Current `main` already preserves original basenames for reply-media outbound staging. Local `MEDIA:` paths keep their basename through `loadWebMedia`, `resolveOutboundAttachmentFromUrl` now passes that filename into `saveMediaBuffer`, and `saveMediaBuffer` encodes the friendly name into the staged outbound path. A focused unit test covers the exact seam reported in #68536.

## Evidence

- **Reply-media staging forwards original filename:** `resolveOutboundAttachmentFromUrl` passes `media.fileName` as the `originalFilename` argument to `saveMediaBuffer`, which is the missing link described in the issue.
  - file: [src/media/outbound-attachment.ts:23](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/media/outbound-attachment.ts#L23)
  - command: `nl -ba src/media/outbound-attachment.ts | sed -n '1,120p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Local MEDIA paths keep basename before staging:** For local files, `loadWebMedia` derives `fileName` from `path.basename(mediaUrl)` and carries it forward, so `MEDIA:./Friendly Name.docx` produces `fileName: "Friendly Name.docx"` before staging.
  - file: [src/media/web-media.ts:555](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/media/web-media.ts#L555)

- **Media store preserves friendly name in staged ID/path:** `saveMediaBuffer` uses `originalFilename` to build IDs like `<sanitized-base>---<uuid>.<ext>`, so outbound staging no longer has to be UUID-only when a source filename is available.
  - file: [src/media/store.ts:271](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/media/store.ts#L271)

- **Regression test covers exact reported seam:** The dedicated test asserts that staging `./report.pdf` calls `saveMediaBuffer(..., "report.pdf")`, directly checking basename preservation through outbound staging.
