## Summary

Current main already contains a Telegram reasoning/answer lane coordinator and delivery logic that preserves partial preview streaming instead of overwriting or stalling the answer lane. The same implementation is present in release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` for `v2026.4.22`, alongside targeted Telegram regression tests for separate preview lanes and final-answer buffering.

## Evidence

- **Reasoning and answer are split into separate Telegram lanes on current main:** `splitTelegramReasoningText` extracts reasoning vs answer text, and `splitTextIntoLaneSegments` routes them onto `reasoning` and `answer` lanes instead of treating the whole stream as one preview.
  - file: [extensions/telegram/src/reasoning-lane-coordinator.ts:65](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/telegram/src/reasoning-lane-coordinator.ts#L65)
  - command: `nl -ba extensions/telegram/src/reasoning-lane-coordinator.ts | sed -n '1,240p'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Final answers are buffered until the reasoning lane is delivered:** Current dispatch logic buffers a final answer when reasoning was hinted first, then flushes the answer only after the reasoning lane is delivered, which directly addresses the reported 'no live preview / final reply only' failure mode when reasoning and answer streams interleave.
