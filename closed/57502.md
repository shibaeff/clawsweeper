## Summary

Current `main` does not contain this PR's exact patch, but the reviewed code path already accepts explicit Discord `user:<id>` targets in isolated cron delivery and sends them through the DM-channel creation flow, so the reported failure does not hold up against current `main`.

## Evidence

- **Core resolver already falls back to normalized explicit targets:** `resolveMessagingTarget()` treats id-like inputs as resolved even when a plugin has no `messaging.targetResolver.resolveTarget`; after `maybeResolveIdLikeTarget()` returns nothing, it returns `buildNormalizedResolveResult({ normalized, kind })`, and `detectTargetKind()` infers `user` for `user:` and `<@...>` targets.
  - file: [src/infra/outbound/target-resolver.ts:144](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/infra/outbound/target-resolver.ts#L144)
  - command: `nl -ba src/infra/outbound/target-resolver.ts | sed -n '144,175p' && nl -ba src/infra/outbound/target-resolver.ts | sed -n '304,381p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Plugin resolver is optional in the normalization layer:** `maybeResolvePluginMessagingTarget()` returns `undefined` when `resolver?.resolveTarget` is absent, which means the core fallback path above is the intended behavior rather than an error state.
  - file: [src/infra/outbound/target-normalization.ts:120](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/infra/outbound/target-normalization.ts#L120)

- **Current Discord plugin still lacks the PR's added messaging resolver:** `extensions/discord/src/channel.ts` defines `messaging.targetResolver` with `looksLikeId` and `hint` only; the proposed `resolveTarget: resolveDiscordTargetForMessaging` wiring is not present on `main`. That means the exact PR is not implemented, but the repo is relying on the existing core fallback instead.
