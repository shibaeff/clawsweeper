## Summary

Current `main` already documents cron `NO_REPLY` suppression and the runtime now suppresses silent replies across the direct-delivery path that originally made this PR inaccurate/stale.

## Evidence

- **Cron docs already mention silent-token suppression:** The current cron docs state that if an isolated run returns only `NO_REPLY` / `no_reply`, OpenClaw suppresses direct outbound delivery and the fallback queued summary path.
  - file: [docs/automation/cron-jobs.md:416](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/docs/automation/cron-jobs.md#L416)

- **CLI docs also document the same behavior:** The cron CLI docs include the same note about suppressing both direct outbound delivery and the fallback queued summary path for isolated silent replies.
  - file: [docs/cli/cron.md:51](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/docs/cli/cron.md#L51)
