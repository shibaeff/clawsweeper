## Summary

Current main already implements a cross-channel reply-routing guard for shared sessions and threads the inbound channel metadata through the delivery path; the same code is present in release `v2026.4.22`, so this `2026.3.13` regression appears fixed on current builds.

## Evidence

- **Product contract says replies go back to the source channel:** The routing docs state that OpenClaw routes replies back to the channel where a message came from, and that `webchat` is internal-only rather than a normal outbound channel.
  - file: [docs/channels/channel-routing.md:10](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/docs/channels/channel-routing.md#L10)
  - command: `nl -ba docs/channels/channel-routing.md | sed -n '1,80p'`

- **Core outbound resolver has an explicit cross-channel guard:** `resolveSessionDeliveryTarget` now prefers `turnSourceChannel`/`turnSourceTo` over the mutable session `lastChannel`/`lastTo`, with an inline comment that this prevents cross-channel reply routing when multiple channels share one session.
  - file: [src/infra/outbound/targets-session.ts:56](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/infra/outbound/targets-session.ts#L56)

- **Gateway agent delivery passes inbound channel context into that guard:** The gateway agent server method forwards the request's `channel`, `to`, and `accountId` as `turnSource*` fields when building the delivery plan, so the guard is used on real agent replies.
  - file: [src/gateway/server-methods/agent.ts:766](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/gateway/server-methods/agent.ts#L766)
