## Summary

Current main’s shared media-understanding path preserves provider and model as separate values for image handling, so the reported QQ image-processing failure mode of splitting `Pro/MiniMaxAI/...` into provider=`Pro` no longer matches the code on `main`. The qqbot extension does not contain its own model-ref parsing for this path.

## Evidence

- **Image auto-selection keeps provider separate from slash-containing model ids:** `resolveAutoImageModelId` returns the configured or explicit image model id as a plain model string, and both image-selection call sites return `{ provider: providerId, model }` without reparsing the model string.
  - file: [src/media-understanding/runner.ts:146](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/media-understanding/runner.ts#L146)
  - command: `nl -ba src/media-understanding/runner.ts | sed -n '146,170p;468,476p;660,675p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Image execution resolves provider/model as separate inputs:** The image runtime receives `provider` and `model` separately and calls `normalizeModelRef(params.provider, params.model)`, so a model id like `Pro/MiniMaxAI/MiniMax-M2.5` stays in the model slot instead of becoming the provider.
  - file: [src/media-understanding/image.ts:98](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/media-understanding/image.ts#L98)
