## Summary

Current `main` routes heartbeat turns through the normal session initialization path, which regenerates a missing `sessionId` and persists a resolved `sessionFile` before history is read. I could not support the reported `heartbeat` -> `sessionId/sessionFile = null forever` behavior from the current code.

## Evidence

- **Heartbeat runs still enter reply/session initialization:** `runHeartbeatOnce()` invokes `getReplyFromConfig(ctx, replyOpts, cfg)` for heartbeat turns, so heartbeat-origin sessions do not bypass the standard session setup path.
  - file: [src/infra/heartbeat-runner.ts:1062](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/infra/heartbeat-runner.ts#L1062)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '1040,1088p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Reply handling always initializes session state:** `getReplyFromConfig()` calls `initSessionState(...)` on the normal path, including heartbeat runs.
  - file: [src/auto-reply/reply/get-reply.ts:265](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/auto-reply/reply/get-reply.ts#L265)
  - command: `nl -ba src/auto-reply/reply/get-reply.ts | sed -n '264,286p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Missing sessionId is not reused:** `initSessionState()` only reuses an entry when `Boolean(entry?.sessionId)` is true; otherwise it generates a new UUID and builds a fresh `sessionEntry` with that `sessionId`. That directly contradicts a 'heartbeat-origin sessions stay null' failure mode on current main.
  - file: [src/auto-reply/reply/session.ts:427](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/auto-reply/reply/session.ts#L427)

  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Session file is persisted during init:** After session init chooses the session id, it calls `resolveAndPersistSessionFile(...)`, which writes `sessionId` and `sessionFile` back into the store when either differs or is missing.
  - file: [src/config/sessions/session-file.ts:18](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/config/sessions/session-file.ts#L18)
  - command: `nl -ba src/config/sessions/session-file.ts | sed -n '1,120p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)

Since heartbeat runs on current main go through the repairing init path above, I could not reproduce the reported permanent empty-history state from the current code path.
