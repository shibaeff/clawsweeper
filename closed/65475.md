## Summary

Keep open. Current `main` still passes a normalized Google Gemini base URL ending in `/v1beta` into `@google/genai` for music generation, and the SDK still appends its own `apiVersion` when building request URLs. The reported double-version 404 path remains plausible on `main`, and the music provider tests do not cover the configured `models.providers.google.baseUrl` case.

## Evidence

- **Music provider still forwards normalized `/v1beta` baseUrl into `GoogleGenAI`:** `resolveConfiguredGoogleMusicBaseUrl()` normalizes `cfg.models.providers.google.baseUrl` with `normalizeGoogleApiBaseUrl()`, and `generateMusic()` passes that value directly as `httpOptions.baseUrl` to `createGoogleGenAI(...)`. That preserves a `/v1beta` suffix on the SDK base URL instead of stripping it to the API origin.
  - file: [extensions/google/music-generation-provider.ts:38](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/google/music-generation-provider.ts#L38)
  - command: `nl -ba extensions/google/music-generation-provider.ts | sed -n '1,220p'`
  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Repo already has the correct helper for Gemini API origin normalization:** `resolveGoogleGenerativeAiApiOrigin()` explicitly removes a trailing `/v1beta`, and the existing test asserts that `https://generativelanguage.googleapis.com/v1beta` resolves to `https://generativelanguage.googleapis.com`. The music provider does not use this helper.
  - file: [extensions/google/provider-policy.ts:93](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/google/provider-policy.ts#L93)

  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Upstream SDK still builds Gemini URLs as `baseUrl` + `apiVersion`:** In `@google/genai` v1.50.1, the Gemini client defaults `apiVersion` to `v1beta` and defaults `baseUrl` to `https://generativelanguage.googleapis.com/`; URL construction trims `baseUrl` and appends `apiVersion` when present. Passing a custom `baseUrl` that already ends with `/v1beta` still yields a doubled version segment.
  - command: `curl -L --silent https://raw.githubusercontent.com/googleapis/js-genai/v1.50.1/src/_api_client.ts | rg -n "apiVersion|baseUrl|v1beta" -n -C 2`
- **Music provider tests miss the configured-baseUrl path:** Current tests cover basic generation success and format validation, but they do not assert behavior when `cfg.models.providers.google.baseUrl` is set, nor do they verify that `httpOptions.baseUrl` avoids a duplicated API version.
  - file: [extensions/google/music-generation-provider.test.ts:34](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/extensions/google/music-generation-provider.test.ts#L34)
  - command: `nl -ba extensions/google/music-generation-provider.test.ts | sed -n '1,220p'`

## Risks / Open Questions

- Users who configure `models.providers.google.baseUrl` for Gemini routing can still hit `.../v1beta/v1beta/...` 404s via `music_generate` on current `main`.
- Because the music provider lacks a configured-baseUrl regression test, this bug can persist or recur without detection.
