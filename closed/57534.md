## Summary

Current `main` already avoids reparsing `sessions.json` on repeated `sessions.list` calls via the session-store cache, and the list RPC returns lightweight session rows that do not include `skillsSnapshot` or `systemPromptReport`. The same implementation is present in the shipped `v2026.4.23` tree.

## Evidence

- **Session store cache short-circuits rereads:** `loadSessionStore()` checks the mtime/size keyed cache before any disk read or JSON parse, and writes the cache back after loading.
  - file: [src/config/sessions/store-load.ts:80](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/config/sessions/store-load.ts#L80)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **sessions.list uses cached loader path:** `sessions.list` calls `loadCombinedSessionStoreForGateway()`, which loads backing stores through `loadSessionStore()` for each session store.
  - file: [src/gateway/server-methods/sessions.ts:579](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/server-methods/sessions.ts#L579)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Combined-store loader uses loadSessionStore:** The combined gateway store path merges entries from each store returned by `loadSessionStore()`, so repeated list calls benefit from the cache rather than reparsing files.
  - file: [src/gateway/session-utils.ts:1011](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/session-utils.ts#L1011)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **List rows exclude bulky snapshot/report fields:** `GatewaySessionRow` includes summary/session metadata only; it has no `skillsSnapshot` or `systemPromptReport` fields.
  - file: [src/gateway/session-utils.types.ts:18](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/session-utils.types.ts#L18)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Regression test covers no-disk-read repeat loads:** The cache test asserts repeated `loadSessionStore()` calls are served from cache with zero `fs.readFileSync` calls, including stores containing `skillsSnapshot` data.
  - file: [src/config/sessions.cache.test.ts:68](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/config/sessions.cache.test.ts#L68)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Shipped release contains the implementation:** The `v2026.4.23` tag already contains `src/config/sessions/store-load.ts` with the same cache-first logic, so this is not just on unreleased `main`.
  - command: `git show 9b18ae877018a34933e27bb7b2986cb97e9602aa:src/config/sessions/store-load.ts`
  - sha: [9b18ae877018](https://github.com/openclaw/openclaw/commit/9b18ae877018a34933e27bb7b2986cb97e9602aa)

## Risks / Open Questions

- The store still persists `skillsSnapshot`/`systemPromptReport` inline, so very large stores can still increase clone/merge cost on cache hits; this review only establishes that the reported per-call disk parse path is already fixed.
