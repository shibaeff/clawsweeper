## Summary

Keep open: current `main` still sends Feishu MP3 TTS audio as `msg_type=file`, not `msg_type=audio`, so the reported behavior is not implemented.

## Evidence

- **Feishu routing only treats OGG/Opus as audio:** `extensions/feishu/src/media.ts:539-567` maps `.opus`/`.ogg` to `msgType: "audio"`; other audio falls through to detected file type and usually `msgType: "file"`.
  - file: [extensions/feishu/src/media.ts:539](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/feishu/src/media.ts#L539)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '530,585p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Feishu tests still expect MP3 to be sent as file:** `extensions/feishu/src/media.test.ts:239-261` has an explicit test named `falls back to generic file for unsupported audio formats` using `audio/mpeg` and asserting `msg_type: "file"`.
  - file: [extensions/feishu/src/media.test.ts:239](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/feishu/src/media.test.ts#L239)
  - command: `nl -ba extensions/feishu/src/media.test.ts | sed -n '168,262p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Microsoft TTS still defaults to MP3-compatible output:** `extensions/microsoft/speech-provider.ts:233-274` uses the configured Microsoft output format and falls back to the default `audio-24khz-48kbitrate-mono-mp3`; it marks compatibility from the produced filename but does not transcode to Feishu-native audio.
  - file: [extensions/microsoft/speech-provider.ts:233](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/microsoft/speech-provider.ts#L233)
  - command: `nl -ba extensions/microsoft/speech-provider.ts | sed -n '228,278p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **TTS sets voice intent, but Feishu outbound does not consume it:** `src/auto-reply/reply/commands-tts.ts:167-170` sets `audioAsVoice` when TTS is voice-compatible, but `extensions/feishu/src/outbound.ts:264-274` calls `sendMediaFeishu(...)` without any `audioAsVoice` handling, so Feishu delivery still depends on file extension/MIME routing.
  - file: [extensions/feishu/src/outbound.ts:264](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/extensions/feishu/src/outbound.ts#L264)
  - command: `nl -ba src/auto-reply/reply/commands-tts.ts | sed -n '154,171p' && nl -ba extensions/feishu/src/outbound.ts | sed -n '225,290p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Docs claim behavior that code does not meet:** `docs/tools/tts.md:507-509` says Feishu TTS audio is delivered as a voice message rather than a file attachment, which conflicts with the current Feishu MP3 routing and test expectations.
  - file: [docs/tools/tts.md:507](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/docs/tools/tts.md#L507)
  - command: `nl -ba docs/tools/tts.md | sed -n '505,511p'`
  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
